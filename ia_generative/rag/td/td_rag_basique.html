TD Pratique - RAG Basique

Mon Premier RAG -- De la theorie a l'implementation complete

1. Recapitulatif -- Qu'est-ce que le RAG ?

RAG (Retrieval-Augmented Generation) est une technique qui enrichit les reponses d'un LLM en lui fournissant des documents pertinents extraits d'une base de connaissances externe. Au lieu de se fier uniquement a sa memoire parametrique, le modele genere sa reponse en s'appuyant sur des preuves recuperees dynamiquement.

Un LLM seul a deux limites majeures :

  • Hallucination : il invente des faits avec assurance
  • Connaissances figees : il ne sait rien de ce qui s'est passe apres son entrainement

Le RAG resout ces deux problemes en injectant des informations factuelles et a jour directement dans le prompt.

Les 2 phases fondamentales du RAG

Phase 1 : Indexation (hors ligne)

  1. Charger les documents sources (PDF, texte, web...)
  2. Decouper les documents en chunks (morceaux) de taille geerable
  3. Vectoriser chaque chunk grace a un modele d'embeddings
  4. Stocker les vecteurs dans une base vectorielle (FAISS, Chroma...)

Phase 2 : Requete (en ligne)

  1. Recevoir la question de l'utilisateur
  2. Vectoriser la question avec le meme modele d'embeddings
  3. Rechercher les chunks les plus similaires dans la base vectorielle
  4. Construire un prompt enrichi = contexte + question
  5. Generer la reponse avec le LLM
La qualite d'un RAG depend de chaque maillon de la chaine. Un mauvais decoupage ou un mauvais modele d'embeddings peut ruiner la qualite finale, meme avec le meilleur LLM du monde.

4. Exercice 2 -- Decoupage en chunks (Text Splitting)

Les documents sont souvent trop longs pour etre passes entierement au LLM. Il faut les decouper en morceaux (chunks) de taille optimale.

Le chunking consiste a decouper un document en segments plus petits, chacun suffisamment court pour etre indexe et recupere efficacement, mais suffisamment long pour conserver le contexte semantique.

4.1 Decoupage avec RecursiveCharacterTextSplitter

Plus le chunk_size est petit, plus on a de chunks (recherche precise) mais moins de contexte par chunk. Plus il est grand, plus chaque chunk est riche mais la recherche est moins precise. En pratique, 300-500 caracteres est un bon point de depart.

5. Exercice 3 -- Embeddings et base vectorielle FAISS

Chaque chunk doit etre transforme en un vecteur numerique (embedding) pour permettre la recherche par similarite.

Un embedding est une representation vectorielle dense d'un texte dans un espace a haute dimension (ex: 1536 dimensions pour OpenAI). Des textes semantiquement proches produisent des vecteurs proches dans cet espace.

5.1 Creer les embeddings

from langchain_openai import OpenAIEmbeddings from langchain_community.vectorstores import FAISS import os # Configuration de la cle API os.environ["OPENAI_API_KEY"] = "votre-cle-api-ici" # Creer le modele d'embeddings embeddings_model = OpenAIEmbeddings( model="text-embedding-3-small" # Modele rapide et economique ) # Tester sur un texte test_embedding = embeddings_model.embed_query("Bonjour, ceci est un test") print(f"Dimension du vecteur : {len(test_embedding)}") print(f"Premiers elements : {test_embedding[:5]}")
Dimension du vecteur : 1536 Premiers elements : [0.0234, -0.0156, 0.0412, -0.0089, 0.0567]

5.2 Creer la base vectorielle FAISS

# Creer la base FAISS a partir des chunks vectorstore = FAISS.from_documents( documents=chunks, # Les chunks issus du splitting embedding=embeddings_model # Le modele d'embeddings ) print(f"Base vectorielle creee avec succes !") print(f"Nombre de vecteurs stockes : {vectorstore.index.ntotal}")
Base vectorielle creee avec succes ! Nombre de vecteurs stockes : 12
FAISS (Facebook AI Similarity Search) stocke les 12 chunks sous forme vectorielle. Chaque chunk est desormais un point dans un espace a 1536 dimensions. La recherche par similarite trouvera les chunks les plus proches de la question.

5.3 Visualiser la structure de la base

Ne jamais oublier l'instruction "reponds uniquement sur la base du contexte". Sans cette contrainte, le LLM melangera ses connaissances generales avec le contexte fourni et vous ne saurez plus ce qui est factuel.

8. Exercice 6 -- Construire la chaine RAG avec LCEL

Nous allons maintenant assembler toutes les pieces avec LCEL (LangChain Expression Language) pour creer une chaine RAG fonctionnelle.

LCEL (LangChain Expression Language) est un langage declaratif pour composer des chaines de traitement. L'operateur | (pipe) enchaine les composants : la sortie de chacun devient l'entree du suivant.

8.1 Les composants de la chaine

Architecture de la chaine LCEL

{"context": retriever, "question": RunnablePassthrough()} | prompt | llm | parser

  • retriever : recupere les chunks pertinents depuis FAISS
  • RunnablePassthrough() : passe la question telle quelle
  • prompt : assemble le contexte et la question
  • llm : genere la reponse (ChatOpenAI)
  • parser : extrait le texte brut de la reponse

8.2 Construction de la chaine

from langchain_openai import ChatOpenAI from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough # 1. Creer le retriever a partir du vector store retriever = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 3} # Retourner les 3 meilleurs chunks ) # 2. Creer le LLM llm = ChatOpenAI( model="gpt-4o-mini", temperature=0 # Reponses deterministes ) # 3. Fonction pour formater les documents recuperes def format_docs(docs): return "\n\n".join(doc.page_content for doc in docs) # 4. Construire la chaine RAG avec LCEL rag_chain = ( { "context": retriever | format_docs, "question": RunnablePassthrough() } | rag_prompt_v2 | llm | StrOutputParser() ) print("Chaine RAG construite avec succes !") print(f"Type : {type(rag_chain)}")
Chaine RAG construite avec succes ! Type : <class 'langchain_core.runnables.base.RunnableSequence'>

8.3 Tester la chaine

# Premiere question reponse = rag_chain.invoke("Quel est le prix de l'offre MedIA Pro ?") print(f"Q: Quel est le prix de l'offre MedIA Pro ?") print(f"R: {reponse}")

10. Exercice 8 -- Attribution des sources

En production, il est essentiel d'indiquer les sources des reponses. Modifions notre chaine pour inclure les references.

10.1 Chaine avec sources

from langchain_core.runnables import RunnableParallel # Prompt avec instruction de citation rag_prompt_sources = ChatPromptTemplate.from_template( """Tu es un assistant expert de NovaTech Solutions. 11. Erreurs Courantes

Chunks trop grands : Si vos chunks font plus de 1000 caracteres, la recherche par similarite sera imprecise. Le chunk pertinent sera "noye" dans du texte non pertinent, et le LLM recevra trop de bruit. Commencez avec 300-500 caracteres.
Pas de chunk_overlap : Sans chevauchement, une information coupee entre deux chunks sera perdue. Utilisez un overlap de 10 a 20% de la taille du chunk (ex: chunk_size=500, overlap=50-100).
Oublier temperature=0 : Pour un RAG factuel, utilisez toujours temperature=0. Une temperature elevee introduit de l'aleatoire et donc des risques d'hallucination, meme avec un bon contexte.
Prompt trop permissif : Si votre prompt ne dit pas explicitement "reponds UNIQUEMENT sur la base du contexte", le LLM completera avec ses connaissances generales. Vous perdez alors tout l'interet du RAG.
k trop petit ou trop grand : Avec k=1, vous risquez de manquer des informations reparties sur plusieurs chunks. Avec k=10, vous noyez le LLM sous trop de contexte. k=3 a 5 est generalement optimal.
Ne pas tester les limites : Testez toujours avec des questions hors contexte. Si votre RAG invente des reponses, c'est un probleme critique a corriger avant la mise en production.
