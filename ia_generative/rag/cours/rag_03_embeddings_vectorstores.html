Embeddings et Vector Stores pour le RAG

Cours 3/4 -- Choisir les bons modeles d'embedding et bases vectorielles pour un RAG performant

1. Embeddings pour le RAG - Rappel et importance

Un embedding est une representation vectorielle d'un texte dans un espace mathematique de haute dimension. Deux textes semantiquement proches (meme sens, meme sujet) auront des vecteurs proches dans cet espace, meme si les mots utilises sont differents. Dans un systeme RAG, la qualite des embeddings determine directement la qualite du retrieval : si les embeddings ne capturent pas correctement le sens, le systeme ne retrouvera pas les bons documents.

L'embedding est le pont entre le texte humain et les mathematiques. C'est ce qui permet de transformer une question comme "Comment faire un remboursement ?" en un vecteur [0.023, -0.145, 0.892, ...] qui sera compare aux vecteurs des chunks de votre base documentaire pour trouver les plus pertinents.

Pourquoi la qualite des embeddings est cruciale pour le RAG ?

Le systeme RAG suit la chaine : Question -> Embedding -> Similarite -> Chunks recuperes -> LLM -> Reponse. Si l'embedding de la question ne capture pas correctement son sens, les chunks recuperes seront non pertinents, et le LLM generera une reponse basee sur du mauvais contexte. C'est le principe "Garbage In, Garbage Out" applique au RAG.

  • Bon embedding : "politique de retour" est proche de "comment renvoyer un article" -> chunk pertinent recupere -> bonne reponse
  • Mauvais embedding : "politique de retour" est plus proche de "politique de l'entreprise" (meme mot "politique") -> chunk non pertinent -> mauvaise reponse
# Demonstration : embeddings et similarite from langchain_openai import OpenAIEmbeddings import numpy as np embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Textes a comparer textes = [ "Comment faire un remboursement ?", # Question "Politique de retour et remboursement", # Semantiquement proche "Renvoyer un produit et recuperer l'argent",# Meme sens, mots differents "La meteo de demain sera ensoleillee", # Completement different "Les politiques de l'entreprise en 2024", # Piege : mot "politique" ] # Calculer les embeddings vecteurs = embeddings.embed_documents(textes) # Calculer la similarite cosinus avec la question (index 0) def cosine_sim(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) question_vec = vecteurs[0] print("Similarite avec 'Comment faire un remboursement ?':\n") for i in range(1, len(textes)): sim = cosine_sim(question_vec, vecteurs[i]) print(f" {sim:.4f} | {textes[i]}")
Similarite avec 'Comment faire un remboursement ?': 0.9234 | Politique de retour et remboursement 0.8876 | Renvoyer un produit et recuperer l'argent 0.1245 | La meteo de demain sera ensoleillee 0.3421 | Les politiques de l'entreprise en 2024
Le modele d'embedding comprend que "remboursement" et "renvoyer un produit et recuperer l'argent" sont semantiquement proches (0.89) meme sans mots en commun. Il distingue aussi "politique de retour" (pertinent, 0.92) de "politiques de l'entreprise" (non pertinent, 0.34) malgre le mot "politique" en commun.

2. Choisir le bon modele d'embeddings

Le choix du modele d'embedding est une decision strategique qui impacte la qualite, le cout et la performance de votre systeme RAG. Voici une comparaison des principaux modeles disponibles en 2025-2026 :

2.1 Comparaison des modeles d'embedding

Modele Fournisseur Dimensions MTEB Score Prix (1M tokens) Cas d'usage
text-embedding-3-small OpenAI 1536 62.3 $0.02 Prototype, cout reduit
text-embedding-3-large OpenAI 3072 64.6 $0.13 Production, haute qualite
embed-v4 Cohere 1024 65.2 $0.10 Multilingue, leader MTEB
voyage-3 Voyage AI 1024 64.8 $0.06 RAG optimise, code
all-MiniLM-L6-v2 Open Source 384 56.3 Gratuit Local, rapide, prototype
BGE-M3 BAAI (Open Source) 1024 63.0 Gratuit Multilingue (100+ langues)
NV-Embed-v2 NVIDIA 4096 62.7 (retrieval) Gratuit Retrieval haute precision
e5-mistral-7b-instruct Microsoft (Open Source) 4096 56.9 (retrieval) Gratuit (GPU requis) Haute qualite, auto-heberge
Le score MTEB (Massive Text Embedding Benchmark) est le benchmark de reference pour comparer les modeles d'embedding. Cependant, ne choisissez pas votre modele uniquement sur ce score. Un modele avec un score MTEB plus bas peut etre meilleur pour VOTRE cas d'usage specifique. Testez toujours avec vos propres donnees et questions. Cohere embed-v4 est actuellement le leader global (65.2), mais OpenAI text-embedding-3-large offre le meilleur rapport qualite/prix pour la plupart des projets.

2.2 Criteres de choix

Comment choisir votre modele d'embedding ?

  • Budget limite / prototype : OpenAI text-embedding-3-small ($0.02/1M tokens) ou all-MiniLM-L6-v2 (gratuit)
  • Production, meilleure qualite : OpenAI text-embedding-3-large ou Cohere embed-v4
  • Corpus multilingue : BGE-M3 (100+ langues) ou Cohere embed-v4
  • Code / technique : Voyage code-2 ou voyage-3
  • Auto-heberge / confidentialite : BGE-M3 ou e5-mistral-7b-instruct
  • Latence minimale : all-MiniLM-L6-v2 (384 dims, ultra rapide)

3. Embedding Dimensions - Impact sur qualite et cout

Les dimensions de l'embedding representent la taille du vecteur genere. Plus de dimensions = plus de nuances capturees, mais aussi plus de stockage et de calcul.

Dimensions Stockage par vecteur Qualite relative Vitesse de recherche Cas d'usage
256 1 KB Acceptable Tres rapide Recherche rapide, IoT, edge
384 1.5 KB Bonne Rapide Prototype, petits corpus
768 3 KB Tres bonne Moyenne Production standard
1536 6 KB Excellente Plus lente RAG haute qualite
3072 12 KB Maximum Lente Precision critique, grands budgets
# OpenAI text-embedding-3 supporte la reduction de dimensions from langchain_openai import OpenAIEmbeddings # Version complete (1536 dims pour small, 3072 pour large) emb_full = OpenAIEmbeddings(model="text-embedding-3-small") # Version reduite (256 dims) - plus rapide, moins cher en stockage emb_reduced = OpenAIEmbeddings( model="text-embedding-3-small", dimensions=256 # Reduire a 256 dimensions ) texte = "Comment fonctionne la politique de retour ?" vec_full = emb_full.embed_query(texte) vec_reduced = emb_reduced.embed_query(texte) print(f"Dimensions completes : {len(vec_full)}") print(f"Dimensions reduites : {len(vec_reduced)}") print(f"Ratio stockage : {len(vec_reduced)/len(vec_full):.1%}")
Dimensions completes : 1536 Dimensions reduites : 256 Ratio stockage : 16.7%
La reduction de dimensions (Matryoshka embeddings) est une fonctionnalite puissante d'OpenAI text-embedding-3. Vous pouvez reduire a 256 ou 512 dimensions avec une perte de qualite minime (environ 2-5% de degradation sur les benchmarks) tout en divisant le stockage par 3 a 6x. Ideal pour les gros corpus ou la vitesse est critique.

4. Vector Stores pour le RAG

Un vector store est une base de donnees specialisee pour stocker et rechercher des vecteurs de maniere efficace. C'est le "coeur" du retrieval dans un systeme RAG. Le choix du vector store depend de votre echelle, de vos besoins en production et de votre budget.

4.1 Comparaison des Vector Stores

Vector Store Type Persistance Scalabilite Cas d'usage
FAISS Librairie (local) Fichier local Millions de vecteurs Prototype rapide, recherche, benchmarks
Chroma DB locale/serveur SQLite / fichier Centaines de milliers Dev local, projets petits/moyens
Pinecone Cloud (managed) Cloud natif Milliards (serverless) Production SaaS, zero ops
Qdrant OSS + Cloud Disque / Cloud Milliards Production, filtrage avance
Weaviate OSS + Cloud Disque / Cloud Milliards Hybrid search, GraphQL
Milvus OSS + Cloud Disque / Cloud Milliards+ Echelle industrielle, billion-scale
pgvector Extension PostgreSQL PostgreSQL Millions Si deja sur PostgreSQL

Recommandation par situation

  • Prototype / apprentissage : FAISS (zero config, en memoire) ou Chroma (persistant, simple)
  • Projet perso / startup : Chroma ou Qdrant (gratuit, facile)
  • Production SaaS (zero ops) : Pinecone (cloud managed, serverless)
  • Production auto-heberge : Qdrant ou Weaviate (OSS, performant, filtrage riche)
  • Echelle massive (milliards) : Milvus (concu pour le billion-scale)
  • Stack PostgreSQL existant : pgvector (extension, pas de nouvelle infra)

6. Similarity Search - Les metriques de distance

La recherche par similarite est l'operation fondamentale du retrieval RAG. Trois metriques de distance principales sont utilisees pour comparer les vecteurs :

Pour le RAG, utilisez la cosine similarity dans 95% des cas. C'est la metrique par defaut de la plupart des vector stores et modeles d'embedding. Elle est insensible a la longueur des textes (magnitude du vecteur) et compare uniquement le "sens" (direction du vecteur). Si vos embeddings sont normalises (comme c'est souvent le cas), le dot product donne les memes resultats et est plus rapide a calculer.
# Comparaison des metriques de distance import numpy as np from numpy.linalg import norm def cosine_similarity(a, b): return np.dot(a, b) / (norm(a) * norm(b)) def euclidean_distance(a, b): return norm(np.array(a) - np.array(b)) def dot_product(a, b): return np.dot(a, b) # Exemple avec des embeddings reels embeddings = OpenAIEmbeddings(model="text-embedding-3-small") vec_q = embeddings.embed_query("politique de retour") vec_a = embeddings.embed_query("remboursement produit") vec_b = embeddings.embed_query("meteo de demain") print("Question vs 'remboursement produit' (pertinent) :") print(f" Cosine sim : {cosine_similarity(vec_q, vec_a):.4f}") print(f" Euclidean : {euclidean_distance(vec_q, vec_a):.4f}") print(f" Dot product : {dot_product(vec_q, vec_a):.4f}") print(f"\nQuestion vs 'meteo de demain' (non pertinent) :") print(f" Cosine sim : {cosine_similarity(vec_q, vec_b):.4f}") print(f" Euclidean : {euclidean_distance(vec_q, vec_b):.4f}") print(f" Dot product : {dot_product(vec_q, vec_b):.4f}")
Question vs 'remboursement produit' (pertinent) : Cosine sim : 0.9234 Euclidean : 0.3912 Dot product : 0.9234 Question vs 'meteo de demain' (non pertinent) : Cosine sim : 0.1245 Euclidean : 1.3218 Dot product : 0.1245

7. Search Parameters - Parametres de recherche

Les parametres de recherche controlent combien et quels documents sont retournes par le retriever. Les bien configurer est essentiel.

7.1 Le parametre k (nombre de resultats)

k - Nombre de documents retournes

Valeur par defaut : 4

Plage recommandee : 3 a 8

Impact : k trop petit = risque de manquer des infos pertinentes. k trop grand = bruit, dilution du contexte, cout LLM accru.

7.2 score_threshold - Filtrage par score minimum

# Retourner uniquement les resultats au-dessus d'un seuil de similarite retriever = vectorstore.as_retriever( search_type="similarity_score_threshold", search_kwargs={ "score_threshold": 0.75, # Minimum 75% de similarite "k": 10 # Maximum 10 resultats } ) # Avec une question pertinente results = retriever.invoke("Comment configurer le Wi-Fi ?") print(f"Resultats au-dessus du seuil : {len(results)}") # Avec une question hors-sujet results = retriever.invoke("Quelle est la capitale du Japon ?") print(f"Resultats au-dessus du seuil : {len(results)}")
Resultats au-dessus du seuil : 4 Resultats au-dessus du seuil : 0
Le score_threshold empeche de retourner des chunks non pertinents quand la question est hors-sujet. Sans ce seuil, le systeme retournerait toujours k resultats, meme si aucun n'est pertinent, ce qui pourrait conduire le LLM a halluciner une reponse basee sur du contexte non pertinent.

7.3 MMR (Maximum Marginal Relevance)

# MMR : diversifier les resultats en evitant la redondance retriever = vectorstore.as_retriever( search_type="mmr", search_kwargs={ "k": 4, # Retourner 4 resultats finaux "fetch_k": 20, # Chercher dans les 20 plus proches "lambda_mult": 0.7 # 0=diversite max, 1=similarite pure } ) results = retriever.invoke("Comment configurer le produit ?") print(f"Resultats MMR (diversifies) : {len(results)}") for i, doc in enumerate(results): print(f" {i+1}. {doc.page_content[:80]}...")
Le MMR est crucial quand vos chunks contiennent du contenu redondant (par exemple, la meme information reperee dans differentes pages ou documents). Sans MMR, vous pourriez recuperer 4 chunks quasi-identiques qui n'ajoutent rien au contexte. Le fetch_k=20 signifie qu'on cherche d'abord les 20 plus proches, puis on selectionne les 4 les plus diversifies parmi eux. Nous detaillerons le MMR dans le cours 4.

8. Metadata Filtering - Filtrage par metadonnees

Le filtrage par metadonnees est une technique puissante qui combine la recherche semantique avec des filtres structures. Cela permet de restreindre la recherche a un sous-ensemble de documents avant la recherche vectorielle.

# Filtrage par metadonnees avec Chroma from langchain_community.vectorstores import Chroma # Recherche avec filtre sur la source results = vectorstore.similarity_search( "politique de conges", k=4, filter={"source": "guide_rh_2024.pdf"} # Uniquement dans ce fichier ) print(f"Resultats filtres par source : {len(results)}") # Filtre sur la categorie results = vectorstore.similarity_search( "delai de livraison", k=4, filter={"category": "logistique"} # Uniquement les docs logistique ) # Filtres multiples (AND) results = vectorstore.similarity_search( "procedure de validation", k=4, filter={ "$and": [ {"category": "RH"}, {"department": "juridique"} ] } ) # Filtrage avec Pinecone (syntaxe differente) # results = vectorstore.similarity_search( # "politique de conges", # k=4, # filter={"category": {"$eq": "RH"}, "year": {"$gte": 2024}} # )
# Cas d'usage : RAG multi-source avec routage par metadonnees def smart_retrieval(question, vectorstore, department=None, year_min=None): """Recherche intelligente avec filtrage contextuel.""" # Construire le filtre dynamiquement filters = [] if department: filters.append({"department": department}) if year_min: filters.append({"year": {"$gte": year_min}}) filter_dict = None if len(filters) == 1: filter_dict = filters[0] elif len(filters) > 1: filter_dict = {"$and": filters} # Recherche avec filtres results = vectorstore.similarity_search( question, k=4, filter=filter_dict ) print(f"Question: {question}") print(f"Filtres: department={department}, year_min={year_min}") print(f"Resultats: {len(results)} chunks trouves") return results # Exemples d'utilisation smart_retrieval("politique de conges", vectorstore, department="RH") smart_retrieval("procedure de deploiement", vectorstore, department="IT", year_min=2024)
Question: politique de conges Filtres: department=RH, year_min=None Resultats: 4 chunks trouves Question: procedure de deploiement Filtres: department=IT, year_min=2024 Resultats: 3 chunks trouves
Le filtrage par metadonnees est ESSENTIEL pour les systemes RAG multi-sources en production. Sans filtrage, une question RH pourrait retourner des chunks de documentation technique. Investissez du temps dans l'enrichissement des metadonnees (cours 2) -- c'est ici que cet investissement porte ses fruits. Les filtres sont appliques AVANT la recherche vectorielle, ce qui accelere aussi la recherche.

9.3 Reindexation complete

# Quand reindexer completement ? # - Changement de modele d'embedding # - Changement de chunk_size/overlap # - Mise a jour majeure du corpus # - Migration de vector store import shutil def full_reindex(docs_directory, persist_dir, embeddings, chunk_size=500): """Reindexation complete du vector store.""" # 1. Supprimer l'ancien index if os.path.exists(persist_dir): shutil.rmtree(persist_dir) print("Ancien index supprime.") # 2. Recharger et re-decouper tous les documents loader = DirectoryLoader(docs_directory, glob="**/*.pdf", loader_cls=PyPDFLoader) docs = loader.load() splitter = RecursiveCharacterTextSplitter( chunk_size=chunk_size, chunk_overlap=int(chunk_size * 0.1) ) chunks = splitter.split_documents(docs) print(f"Documents recharges : {len(docs)}, Chunks : {len(chunks)}") # 3. Creer le nouvel index vectorstore = Chroma.from_documents( chunks, embeddings, persist_directory=persist_dir ) print(f"Nouvel index cree avec {vectorstore._collection.count()} vecteurs") return vectorstore # Utilisation lors d'un changement de modele d'embedding new_embeddings = OpenAIEmbeddings(model="text-embedding-3-large") # Nouveau modele vectorstore = full_reindex("./documents/", "./chroma_db_v2", new_embeddings)
Changer de modele d'embedding OBLIGE a reindexer tous les documents. Les vecteurs de l'ancien modele ne sont pas compatibles avec le nouveau. Prevoyez cette operation (temps + cout API) dans votre planning. Pour un corpus de 10 000 chunks avec text-embedding-3-small, la reindexation coute environ $0.01 et prend quelques minutes.

Checklist des bonnes pratiques

  1. Meme modele d'embedding pour l'indexation et la requete -- OBLIGATOIRE
  2. Testez avec vos donnees -- ne vous fiez pas uniquement aux scores MTEB
  3. Commencez simple -- FAISS + text-embedding-3-small, puis iterez
  4. Enrichissez les metadonnees -- pour le filtrage et la tracabilite
  5. Utilisez le score_threshold -- evitez les resultats non pertinents
  6. Activez MMR -- pour la diversite des resultats
  7. Prevoyez la reindexation -- en cas de changement de modele ou parametres
  8. Sauvegardez votre index -- sur disque ou dans le cloud
  9. Surveillez les couts -- embeddings API + stockage vectoriel
  10. Mesurez la qualite -- avec des metriques de retrieval (precision, recall)
Ce cours vous a fourni les bases solides pour choisir et configurer les embeddings et vector stores de votre systeme RAG. Dans le cours suivant, nous plongerons dans les strategies de retrieval avancees (Multi-Query, HyDE, Hybrid Search, Reranking) qui sont le coeur de l'optimisation RAG. C'est la ou les vrais gains de performance se trouvent.
