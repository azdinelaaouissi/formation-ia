Ingestion des Documents - Preparer les Donnees pour le RAG

Cours 2/4 -- Le pipeline d'ingestion : charger, decouper, enrichir et stocker

1. Le Pipeline d'Ingestion

L'ingestion est la phase hors-ligne du RAG ou l'on transforme des documents bruts (PDF, TXT, HTML, CSV, bases de donnees) en vecteurs stockes dans une base vectorielle. Cette phase determine directement la qualite du retrieval : si les chunks sont mal decoupes ou les metadonnees absentes, meme le meilleur retriever ne pourra pas trouver l'information pertinente.

Le pipeline d'ingestion comporte 5 etapes sequentielles. Chaque etape doit etre soigneusement configuree car les erreurs se propagent en cascade : un mauvais decoupage entraine un mauvais retrieval, qui entraine une mauvaise generation.

2. Document Loading - Charger les Documents

La premiere etape consiste a charger les documents bruts et a les convertir en objets Document exploitables par le pipeline. Chaque type de fichier necessite un loader specifique. LangChain fournit des dizaines de loaders pour couvrir tous les formats courants.

3. Chunking Strategy - La cle du RAG performant

Le chunking (decoupage) est l'etape la plus critique du pipeline d'ingestion. Un mauvais chunking est la cause numero 1 des mauvaises performances RAG. Il s'agit de trouver le bon equilibre entre des chunks trop petits (manque de contexte) et trop grands (trop de bruit).

3.1 Regles de base pour le chunking

Parametre Recommandation Explication
chunk_size 300-600 caracteres (ou 100-200 tokens) Assez grand pour contenir une idee complete, assez petit pour etre precis lors de la recherche
chunk_overlap 10-20% de chunk_size (50-100 caracteres) Evite de couper une idee a la frontiere entre deux chunks. Cree de la redondance benefique
Separateurs ["\n\n", "\n", ". ", " ", ""] Coupe d'abord sur les paragraphes, puis les lignes, puis les phrases, puis les mots
Test Toujours verifier visuellement les chunks Inspecter 20-30 chunks pour verifier la coherence semantique
Ne vous fiez pas aux benchmarks generiques pour choisir votre chunk_size. Un benchmark recent (Vecta, fevrier 2026) a montre que les chunks de 512 tokens avec RecursiveCharacterTextSplitter obtiennent 69% de precision, mais cela depend enormement de vos donnees. Testez TOUJOURS avec vos propres documents et questions.

4. RecursiveCharacterTextSplitter - Le splitter de reference

Le RecursiveCharacterTextSplitter est le text splitter le plus utilise et recommande pour la majorite des cas d'usage RAG. Il tente de decouper le texte en respectant les limites naturelles (paragraphes, phrases) tout en maintenant la taille des chunks sous un seuil configure.

Le RecursiveCharacterTextSplitter fonctionne de maniere "recursive" : il essaie d'abord de couper sur le premier separateur (ex: "\n\n" pour les paragraphes). Si un morceau est encore trop grand, il essaie le separateur suivant (ex: "\n" pour les lignes), puis le suivant (ex: ". " pour les phrases), et ainsi de suite jusqu'a obtenir des chunks de la bonne taille.

5. Semantic Chunking - Decoupage par le sens

Le chunking semantique est une approche plus sophistiquee qui decoupe le texte en fonction du sens plutot que du nombre de caracteres. L'idee : quand le sujet change significativement dans le texte, on cree un nouveau chunk.

Le Semantic Chunking utilise un modele d'embedding pour calculer la similarite entre phrases consecutives. Quand la similarite chute en dessous d'un seuil (indiquant un changement de sujet), le splitter cree une frontiere de chunk. Chaque chunk contient ainsi un sujet coherent.
from langchain_experimental.text_splitter import SemanticChunker from langchain_openai import OpenAIEmbeddings # Creer le semantic chunker embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Methode 1 : Seuil base sur le percentile chunker = SemanticChunker( embeddings, breakpoint_threshold_type="percentile", breakpoint_threshold_amount=70 # Couper quand la similarite est dans le 30% le plus bas ) texte = """L'intelligence artificielle (IA) est un domaine de l'informatique qui vise a creer des systemes capables de realiser des taches necessitant normalement l'intelligence humaine. Les reseaux de neurones sont une approche fondamentale de l'IA moderne. Le jardinage biologique gagne en popularite. Les jardiniers preferent eviter les pesticides chimiques et utilisent des methodes naturelles comme le compostage et les insectes benefiques. L'apprentissage profond (deep learning) utilise des reseaux de neurones a plusieurs couches pour apprendre des representations de donnees. GPT et BERT sont des exemples de modeles de deep learning pour le NLP.""" chunks = chunker.split_text(texte) for i, chunk in enumerate(chunks): print(f"\n--- Chunk semantique {i+1} ---") print(chunk)
--- Chunk semantique 1 --- L'intelligence artificielle (IA) est un domaine de l'informatique qui vise a creer des systemes capables de realiser des taches necessitant normalement l'intelligence humaine. Les reseaux de neurones sont une approche fondamentale de l'IA moderne. --- Chunk semantique 2 --- Le jardinage biologique gagne en popularite. Les jardiniers preferent eviter les pesticides chimiques et utilisent des methodes naturelles comme le compostage et les insectes benefiques. --- Chunk semantique 3 --- L'apprentissage profond (deep learning) utilise des reseaux de neurones a plusieurs couches pour apprendre des representations de donnees. GPT et BERT sont des exemples de modeles de deep learning pour le NLP.
Le chunking semantique a correctement identifie 3 sujets distincts (IA, jardinage, deep learning) et a cree 3 chunks coherents, meme si le paragraphe sur le jardinage est au milieu de deux paragraphes sur l'IA. Un RecursiveCharacterTextSplitter aurait pu melanger ces sujets dans un meme chunk.
Un article de NAACL 2025 Findings a montre que le semantic chunking ne justifie pas toujours son cout supplementaire : des chunks fixes de 200 mots egalent ou surpassent le chunking semantique sur de nombreux benchmarks. Le cout en embeddings est aussi significatif (il faut embedder chaque phrase pour calculer les points de coupure). Evaluez toujours si le gain de precision justifie le cout pour votre cas d'usage specifique.

6. Hierarchical Chunking - Chunks parents-enfants

Le chunking hierarchique est une strategie avancee qui cree des chunks a deux niveaux : des petits chunks pour la recherche precise, et des grands chunks (parents) pour fournir un contexte complet au LLM.

# Implementer le chunking hierarchique avec LangChain from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.storage import InMemoryStore from langchain.retrievers import ParentDocumentRetriever from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings # Splitter pour les chunks enfants (petits = recherche precise) child_splitter = RecursiveCharacterTextSplitter( chunk_size=200, chunk_overlap=20 ) # Splitter pour les chunks parents (grands = contexte complet) parent_splitter = RecursiveCharacterTextSplitter( chunk_size=1000, chunk_overlap=100 ) # Vector store pour les enfants (utilisee pour la recherche) vectorstore = Chroma( collection_name="children", embedding_function=OpenAIEmbeddings(model="text-embedding-3-small") ) # Store pour les parents (en memoire ou persistent) docstore = InMemoryStore() # Le ParentDocumentRetriever lie les deux retriever = ParentDocumentRetriever( vectorstore=vectorstore, docstore=docstore, child_splitter=child_splitter, parent_splitter=parent_splitter, ) # Ajouter les documents retriever.add_documents(documents) # Recherche : cherche dans les enfants, retourne les parents results = retriever.invoke("Comment faire un retour ?") print(f"Nombre de parents retournes : {len(results)}") print(f"Taille du contexte : {len(results[0].page_content)} caracteres")
Nombre de parents retournes : 2 Taille du contexte : 956 caracteres
Le chunking hierarchique est particulierement efficace pour les documents longs et structures (manuels techniques, documentation API, contrats). Il combine la precision de la recherche (petits chunks) avec la richesse du contexte (grands chunks). C'est une des techniques les plus impactantes du Advanced RAG.

Il n'existe pas de configuration universelle. La strategie de chunking optimale depend fortement du type de document que vous indexez :

8. Metadata Enrichment - Enrichir les metadonnees

Les metadonnees sont des informations structurees attachees a chaque chunk. Elles sont cruciales pour le filtrage lors du retrieval et pour la tracabilite des sources. Un chunk sans metadonnees est comme un livre sans table des matieres.

8.1 Types de metadonnees essentielles

Metadonnee Utilite Exemple
source Savoir d'ou vient l'information "guide_rh_2024.pdf"
page Localiser l'information dans le document original 3
section / titre Filtrer par section, comprendre le contexte "Politique de conges"
date Filtrer par date, privilegier les docs recents "2024-11-15"
categorie / type Router vers le bon sous-ensemble de documents "RH", "technique", "commercial"
langue Filtrer par langue pour les corpus multilingues "fr", "en"
auteur Tracabilite, credibilite de la source "Service juridique"

8.2 Enrichir les metadonnees programmatiquement

from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain_community.document_loaders import PyPDFLoader import os from datetime import datetime def enrich_metadata(docs, category, department): """Enrichir les metadonnees de chaque document/chunk.""" for doc in docs: # Metadonnees existantes source = doc.metadata.get("source", "inconnu") # Enrichissement doc.metadata["category"] = category doc.metadata["department"] = department doc.metadata["indexed_at"] = datetime.now().isoformat() doc.metadata["file_name"] = os.path.basename(source) doc.metadata["file_type"] = os.path.splitext(source)[1] # Ajouter la taille du chunk (utile pour le debug) doc.metadata["chunk_size"] = len(doc.page_content) # Extraire le titre (premiere ligne si c'est un titre) first_line = doc.page_content.split("\n")[0].strip() if len(first_line) < 100: doc.metadata["title_hint"] = first_line return docs # Utilisation loader = PyPDFLoader("politique_rh.pdf") splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50) pages = loader.load() chunks = splitter.split_documents(pages) chunks = enrich_metadata(chunks, category="RH", department="Ressources Humaines") print(f"Metadonnees enrichies du chunk 3 :") for key, value in chunks[3].metadata.items(): print(f" {key}: {value}")

9. Deduplication - Eviter les doublons

Quand on indexe de nombreux documents, il est frequent d'avoir du contenu en double (memes paragraphes dans differents documents, versions multiples du meme document, headers/footers repetes). La deduplication ameliore la qualite du retrieval en evitant de recuperer le meme contenu plusieurs fois.

import hashlib from typing import List from langchain.schema import Document def deduplicate_chunks(chunks: List[Document], similarity_threshold: float = 0.95) -> List[Document]: """Supprimer les chunks en double base sur le hash du contenu.""" seen_hashes = set() unique_chunks = [] for chunk in chunks: # Normaliser le texte (minuscules, espaces) normalized = chunk.page_content.lower().strip() normalized = " ".join(normalized.split()) # Calculer le hash content_hash = hashlib.md5(normalized.encode()).hexdigest() if content_hash not in seen_hashes: seen_hashes.add(content_hash) unique_chunks.append(chunk) print(f"Chunks avant deduplication : {len(chunks)}") print(f"Chunks apres deduplication : {len(unique_chunks)}") print(f"Doublons supprimes : {len(chunks) - len(unique_chunks)}") return unique_chunks # Utilisation unique_chunks = deduplicate_chunks(all_chunks)

# Deduplication plus avancee avec MinHash pour la similarite approximative from datasketch import MinHash, MinHashLSH def deduplicate_fuzzy(chunks, threshold=0.85): """Deduplication floue : detecte les quasi-doublons.""" lsh = MinHashLSH(threshold=threshold, num_perm=128) unique = [] for i, chunk in enumerate(chunks): # Creer le MinHash du contenu m = MinHash(num_perm=128) for word in chunk.page_content.lower().split(): m.update(word.encode('utf-8')) # Verifier s'il existe un quasi-doublon similar = lsh.query(m) if not similar: lsh.insert(f"chunk_{i}", m) unique.append(chunk) print(f"Quasi-doublons detectes : {len(chunks) - len(unique)}") return unique

10. Batch Processing - Traitement en lots

En production, vous devrez souvent indexer des centaines ou des milliers de documents. Le traitement en lots (batch processing) est essentiel pour gerer efficacement cette charge.

import os from pathlib import Path from langchain_community.document_loaders import PyPDFLoader, TextLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain_openai import OpenAIEmbeddings from langchain_community.vectorstores import Chroma from tqdm import tqdm import logging logging.basicConfig(level=logging.INFO) logger = logging.getLogger("ingestion") class IngestionPipeline: """Pipeline d'ingestion complet et robuste pour le RAG.""" def __init__(self, persist_dir="./chroma_db", chunk_size=500, chunk_overlap=50): self.splitter = RecursiveCharacterTextSplitter( chunk_size=chunk_size, chunk_overlap=chunk_overlap ) self.embeddings = OpenAIEmbeddings(model="text-embedding-3-small") self.persist_dir = persist_dir self.stats = {"loaded": 0, "chunks": 0, "errors": 0} def _get_loader(self, file_path): """Selectionner le bon loader selon l'extension.""" ext = Path(file_path).suffix.lower() loaders = { ".pdf": PyPDFLoader, ".txt": TextLoader, ".md": TextLoader, } loader_cls = loaders.get(ext) if not loader_cls: raise ValueError(f"Extension non supportee : {ext}") return loader_cls(file_path) def process_directory(self, directory, batch_size=50): """Traiter tous les fichiers d'un repertoire par lots.""" files = list(Path(directory).rglob("*")) files = [f for f in files if f.suffix.lower() in [".pdf", ".txt", ".md"]] logger.info(f"Fichiers trouves : {len(files)}") all_chunks = [] for file_path in tqdm(files, desc="Chargement"): try: loader = self._get_loader(str(file_path)) docs = loader.load() chunks = self.splitter.split_documents(docs) # Enrichir les metadonnees for chunk in chunks: chunk.metadata["file_type"] = file_path.suffix chunk.metadata["file_name"] = file_path.name all_chunks.extend(chunks) self.stats["loaded"] += 1 except Exception as e: logger.error(f"Erreur {file_path}: {e}") self.stats["errors"] += 1 # Deduplication all_chunks = deduplicate_chunks(all_chunks) self.stats["chunks"] = len(all_chunks) # Indexation par lots logger.info(f"Indexation de {len(all_chunks)} chunks par lots de {batch_size}...") vectorstore = None for i in tqdm(range(0, len(all_chunks), batch_size), desc="Indexation"): batch = all_chunks[i:i + batch_size] if vectorstore is None: vectorstore = Chroma.from_documents( batch, self.embeddings, persist_directory=self.persist_dir ) else: vectorstore.add_documents(batch) logger.info(f"Ingestion terminee ! Stats : {self.stats}") return vectorstore # Utilisation pipeline = IngestionPipeline(chunk_size=500, chunk_overlap=50) vectorstore = pipeline.process_directory("./documents/")

11. Resume - Strategie de chunking par cas d'usage

Decision : quelle strategie de chunking choisir ?

Situation Strategie recommandee chunk_size overlap
Debut / prototype rapide RecursiveCharacterTextSplitter 500 50
FAQ / Q&A structures Separateurs personnalises 200-300 0-20
Documentation technique (MD) MarkdownHeaderTextSplitter 500-800 50-80

Code source Language-specific splitter 1000-2000 100-200
Precision maximale requise Semantic Chunking Dynamique N/A
Corpus heterogene (multi-types) Pipeline avec detection de type Variable Variable

Checklist d'ingestion

  1. Identifiez les types de documents dans votre corpus
  2. Choisissez les bons loaders pour chaque type
  3. Definissez votre strategie de chunking selon le tableau ci-dessus
  4. Enrichissez les metadonnees (source, page, date, categorie)
  5. Effectuez la deduplication avant l'indexation
  6. Traitez en lots pour les gros corpus
  7. Inspectez visuellement 20-30 chunks pour valider la qualite
  8. Testez le retrieval avec 10 questions representatives
  9. Iterez sur les parametres en mesurant l'impact
L'ingestion est la fondation de votre systeme RAG. 80% des problemes de qualite en production viennent d'une mauvaise ingestion (mauvais chunking, metadonnees manquantes, doublons).
