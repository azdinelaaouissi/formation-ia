Génération et Prompts RAG

Cours 05 - Produire des Réponses de Qualité à partir du Contexte Récupéré

1. Le Rôle de la Génération dans le Pipeline RAG

La génération est l'étape finale du pipeline RAG où un modèle de langage (LLM) prend en entrée la question de l'utilisateur et les documents récupérés (contexte) pour produire une réponse précise, cohérente et fidèle aux sources.

Dans un système RAG complet, la génération ne se limite pas à "demander au LLM de répondre". C'est une étape critique qui détermine la qualité finale de la réponse. Un excellent système de retrieval est inutile si la génération est mal configurée : le LLM peut halluciner, ignorer le contexte, ou produire des réponses hors sujet.

Les 3 composantes de la génération RAG

  • Le prompt système : Instructions qui définissent le comportement du LLM (rester fidèle au contexte, citer les sources, etc.)
  • Le contexte : Les documents/chunks récupérés par le système de retrieval
  • La question : La requête originale de l'utilisateur (parfois reformulée)
La génération est l'étape où la magie opère, mais aussi où les erreurs surviennent. Un prompt mal conçu peut conduire le LLM à ignorer le contexte et à halluciner. La qualité du prompt est au moins aussi importante que la qualité du retrieval.

2. Le Prompt RAG Standard

Le prompt RAG est structuré en trois parties distinctes qui guident le LLM pour produire une réponse fidèle au contexte récupéré.

2.1 Structure de base

# Template de prompt RAG standard RAG_PROMPT_TEMPLATE = """Tu es un assistant expert qui répond aux questions en se basant UNIQUEMENT sur le contexte fourni ci-dessous. Règles : - Réponds UNIQUEMENT à partir des informations du contexte - Si le contexte ne contient pas la réponse, dis "Je ne trouve pas cette information dans les documents disponibles" - Cite les sources pertinentes dans ta réponse - Sois précis et concis --- CONTEXTE --- {context} --- FIN DU CONTEXTE --- Question : {question} Réponse :"""

2.2 Implémentation Python complète

from openai import OpenAI from typing import List, Dict class RAGGenerator: """Générateur de réponses RAG avec prompt structuré.""" def __init__(self, model: str = "gpt-4o"): self.client = OpenAI() self.model = model def format_context(self, documents: List[Dict]) -> str: """Formate les documents récupérés en contexte textuel.""" context_parts = [] for i, doc in enumerate(documents, 1): source = doc.get("metadata", {}).get("source", "Inconnu") context_parts.append( f"[Document {i}] (Source: {source})\n{doc['content']}" ) return "\n\n".join(context_parts) def generate(self, question: str, documents: List[Dict]) -> str: """Génère une réponse à partir de la question et des documents.""" context = self.format_context(documents) system_prompt = """Tu es un assistant expert qui répond aux questions en se basant UNIQUEMENT sur le contexte fourni. Cite toujours les numéros de documents entre crochets [Document X] pour chaque affirmation. Si l'information n'est pas dans le contexte, dis-le clairement.""" user_prompt = f"""Contexte : {context} Question : {question} Réponds de manière précise en citant les sources.""" response = self.client.chat.completions.create( model=self.model, messages=[ {"role": "system", "content": system_prompt}, {"role": "user", "content": user_prompt} ], temperature=0.1, # Basse pour plus de fidélité max_tokens=1024 ) return response.choices[0].message.content # Utilisation generator = RAGGenerator() documents = [ {"content": "Python a été créé par Guido van Rossum en 1991.", "metadata": {"source": "wiki_python.pdf"}}, {"content": "Python 3.12 introduit les f-strings améliorées.", "metadata": {"source": "changelog_3.12.md"}} ] answer = generator.generate("Qui a créé Python ?", documents) print(answer)
D'après les documents disponibles, Python a été créé par Guido van Rossum en 1991 [Document 1].
La température basse (0.1) force le LLM à rester proche du contexte fourni, réduisant les hallucinations. Pour des tâches créatives, on augmenterait cette valeur.

3. Prompt Engineering pour RAG

Le prompt engineering pour RAG est fondamentalement différent du prompt engineering général. L'objectif n'est pas de stimuler la créativité du LLM mais de le contraindre à extraire et synthétiser l'information du contexte fourni.

3.1 Instructions pour citer les sources

# Prompt avec instructions de citation détaillées CITATION_PROMPT = """Tu es un assistant de recherche documentaire. INSTRUCTIONS DE CITATION : 1. Chaque affirmation doit être suivie d'une référence [Doc X] 2. Si plusieurs documents confirment la même info, cite-les tous [Doc 1, Doc 3] 3. Ne fais JAMAIS d'affirmation sans référence au contexte 4. Si tu synthétises plusieurs documents, indique lesquels Contexte : {context} Question : {question} Réponse (avec citations) :"""

3.2 Instructions pour gérer l'absence d'information

# Prompt qui gère gracieusement le "je ne sais pas" HONEST_PROMPT = """Tu es un assistant honnête et précis. RÈGLES FONDAMENTALES : - Réponds UNIQUEMENT à partir du contexte fourni - Si le contexte ne contient PAS l'information demandée, réponds exactement : "Cette information n'est pas disponible dans les documents consultés." - Ne complète JAMAIS avec tes connaissances générales - Si le contexte est partiellement pertinent, précise ce que tu as trouvé ET ce qui manque CONTEXTE : {context} QUESTION : {question} RÉPONSE :"""

3.3 Gestion des contradictions entre sources

# Prompt pour gérer les contradictions entre documents CONTRADICTION_PROMPT = """Tu es un assistant d'analyse documentaire. Si tu détectes des CONTRADICTIONS entre les documents : 1. Signale explicitement la contradiction 2. Présente les deux versions avec leurs sources 3. Ne tranche PAS en faveur d'une version 4. Suggère de vérifier les sources originales Format de réponse si contradiction : "Attention : Les documents présentent des informations contradictoires sur ce point. - Selon [Doc X] : ... - Selon [Doc Y] : ... Il est recommandé de vérifier les sources originales." CONTEXTE : {context} QUESTION : {question} RÉPONSE :"""

Bibliothèque de directives de prompt RAG

  • Fidélité : "Base ta réponse EXCLUSIVEMENT sur le contexte fourni"
  • Honnêteté : "Si l'information n'est pas dans le contexte, dis-le"
  • Attribution : "Cite les sources pour chaque affirmation"
  • Précision : "Utilise les termes exacts des documents"
  • Concision : "Réponds de manière concise et structurée"
  • Contradictions : "Signale les incohérences entre documents"
  • Limites : "N'extrapole pas au-delà du contexte"

3.4 Prompt structuré avancé avec Few-Shot

# Prompt avec exemples (few-shot) pour guider le format FEW_SHOT_RAG_PROMPT = """Tu es un assistant expert. Voici un exemple du format de réponse attendu : EXEMPLE : Question : Quelle est la population de Paris ? Contexte : [Doc 1] Paris compte 2,1 millions d'habitants intra-muros. [Doc 2] L'agglomération parisienne regroupe 12 millions de personnes. Réponse : Paris compte 2,1 millions d'habitants dans sa zone intra-muros [Doc 1]. Si l'on considère l'agglomération élargie, ce chiffre atteint 12 millions de personnes [Doc 2]. Maintenant réponds à cette question : CONTEXTE : {context} QUESTION : {question} RÉPONSE :"""
Le few-shot est particulièrement utile pour imposer un format de réponse précis. En montrant un exemple, le LLM comprend exactement comment structurer sa réponse avec les citations et le niveau de détail attendu.

4. Stuffing vs Map-Reduce vs Refine

Lorsque le système de retrieval renvoie de nombreux documents, il existe trois stratégies principales pour les passer au LLM. Le choix de la stratégie dépend du nombre de documents, de la taille de la fenêtre de contexte du LLM, et du type de réponse attendu.

4.1 Stratégie Stuffing (Bourrage)

def stuffing_strategy(question: str, documents: list, llm) -> str: """Tous les documents sont concaténés dans un seul prompt.""" context = "\n\n".join([ f"[Doc {i+1}] {doc['content']}" for i, doc in enumerate(documents) ]) prompt = f"""Contexte : {context} Question : {question} Réponse :""" return llm.generate(prompt) # Avantage : 1 seul appel LLM, simple et rapide # Inconvénient : Limité par la fenêtre de contexte # Utiliser quand : Peu de documents (< 10), context window large

4.2 Stratégie Map-Reduce

def map_reduce_strategy(question: str, documents: list, llm) -> str: """Phase MAP : chaque document traité séparément. Phase REDUCE : les résumés sont combinés.""" # Phase MAP : extraire l'info pertinente de chaque document summaries = [] for i, doc in enumerate(documents): map_prompt = f"""Extrais les informations pertinentes pour répondre à cette question. Document : {doc['content']} Question : {question} Informations pertinentes (ou "Aucune info pertinente") :""" summary = llm.generate(map_prompt) if "aucune" not in summary.lower(): summaries.append(f"[Doc {i+1}] {summary}") # Phase REDUCE : combiner les résumés en réponse finale reduce_prompt = f"""Voici des extraits pertinents de différents documents. Synthétise-les en une réponse complète et cohérente. Extraits : {chr(10).join(summaries)} Question : {question} Réponse synthétisée :""" return llm.generate(reduce_prompt) # Avantage : Scalable, parallélisable, gère beaucoup de docs # Inconvénient : Plus d'appels LLM (coût + latence) # Utiliser quand : Nombreux documents (> 20), résumés longs

4.3 Stratégie Refine (Itérative)

def refine_strategy(question: str, documents: list, llm) -> str: """Raffine itérativement la réponse avec chaque nouveau document.""" # Première réponse avec le premier document initial_prompt = f"""Contexte : {documents[0]['content']} Question : {question} Réponse initiale :""" current_answer = llm.generate(initial_prompt) # Raffiner avec chaque document suivant for doc in documents[1:]: refine_prompt = f"""Voici une réponse existante et un nouveau document. Améliore la réponse en intégrant les nouvelles informations. Réponse actuelle : {current_answer} Nouveau document : {doc['content']} Question originale : {question} Réponse améliorée (intègre les nouvelles infos si pertinentes) :""" current_answer = llm.generate(refine_prompt) return current_answer # Avantage : Réponse progressive, pas de perte d'info # Inconvénient : Séquentiel (pas parallélisable), N appels # Utiliser quand : Documents complémentaires, besoin de précision

4.4 Tableau comparatif

CritèreStuffingMap-ReduceRefine
Appels LLM1N + 1N
LatenceFaibleMoyenne (parallélisable)Haute (séquentiel)
CoûtFaibleÉlevéÉlevé
Max documents~10-15Illimité~20-30
QualitéBonne si peu de docsBonne pour synthèseExcellente, précise
Cas d'usageRAG standardRésumé de corpusAnalyse détaillée
Avec les modèles modernes ayant des fenêtres de contexte très larges (128K tokens pour GPT-4o, 200K pour Claude), le stuffing est devenu la stratégie par défaut dans la plupart des cas. Map-Reduce et Refine restent utiles pour des corpus vraiment massifs ou quand on veut maîtriser les coûts.

5. Attribution des Sources

L'attribution des sources est essentielle pour la crédibilité d'un système RAG. Elle permet aux utilisateurs de vérifier les informations et de remonter aux documents originaux.

5.1 Sortie structurée avec citations

from pydantic import BaseModel from typing import List from openai import OpenAI import json class Citation(BaseModel): """Une citation avec sa source.""" text: str # Le texte affirmé source_doc: int # Numéro du document source source_page: str # Page/section dans le document confidence: float # Confiance dans la citation (0-1) class RAGResponse(BaseModel): """Réponse RAG structurée avec citations.""" answer: str # La réponse complète citations: List[Citation] # Liste des citations confidence: float # Confiance globale (0-1) has_sufficient_context: bool # Le contexte suffit-il ? def generate_with_citations(question: str, documents: list) -> RAGResponse: """Génère une réponse structurée avec citations vérifiables.""" client = OpenAI() context = "\n\n".join([ f"[Document {i+1}] (Source: {d['source']})\n{d['content']}" for i, d in enumerate(documents) ]) response = client.beta.chat.completions.parse( model="gpt-4o", messages=[ {"role": "system", "content": """Réponds à la question en citant tes sources. Retourne une réponse structurée JSON avec : - answer: la réponse complète - citations: liste de {text, source_doc, source_page, confidence} - confidence: score global de confiance - has_sufficient_context: true/false"""}, {"role": "user", "content": f"Contexte:\n{context}\n\nQuestion: {question}"} ], response_format=RAGResponse, temperature=0.1 ) return response.choices[0].message.parsed # Utilisation docs = [ {"content": "Le chiffre d'affaires 2024 est de 5,2M EUR.", "source": "rapport_annuel_2024.pdf, page 12"}, {"content": "La croissance prévue pour 2025 est de 15%.", "source": "prévisions_2025.pdf, page 3"} ] result = generate_with_citations("Quel est le CA 2024 ?", docs) print(f"Réponse: {result.answer}") print(f"Confiance: {result.confidence}") for c in result.citations: print(f" - '{c.text}' [Doc {c.source_doc}] ({c.confidence:.0%})")
Réponse: Le chiffre d'affaires 2024 s'élève à 5,2 millions d'euros. Confiance: 0.95 - 'Le chiffre d'affaires 2024 est de 5,2M EUR' [Doc 1] (95%)
L'utilisation de la sortie structurée (structured output) via Pydantic est une pratique de production essentielle. Elle garantit que la réponse est toujours parseable et que les citations sont exploitables programmatiquement pour l'affichage dans l'interface utilisateur.

6. Gérer le "Je ne sais pas"

Un des problèmes les plus critiques du RAG est quand le contexte récupéré ne contient pas la réponse à la question. Le LLM a tendance à halluciner plutôt qu'à admettre son ignorance. Il faut explicitement programmer ce comportement.

6.1 Stratégie de détection d'absence d'information

class SafeRAGGenerator: """Générateur RAG qui sait dire 'je ne sais pas'.""" SYSTEM_PROMPT = """Tu es un assistant documentaire strict. RÈGLE ABSOLUE : Tu ne dois JAMAIS inventer d'information. Processus de réponse : 1. Lis attentivement TOUT le contexte fourni 2. Identifie les passages pertinents pour la question 3. Si des passages pertinents existent : réponds en les citant 4. Si AUCUN passage n'est pertinent : utilise la réponse type ci-dessous RÉPONSE TYPE (absence d'info) : "Je n'ai pas trouvé d'information sur [sujet] dans les documents disponibles. Les documents consultés portent sur [thèmes des docs]. Vous pourriez reformuler votre question ou consulter d'autres sources." RÉPONSE TYPE (info partielle) : "J'ai trouvé des informations partielles : [info trouvée]. Cependant, je n'ai pas trouvé [ce qui manque]." """ def generate(self, question: str, documents: list) -> dict: context = self._format_docs(documents) response = self.client.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": self.SYSTEM_PROMPT}, {"role": "user", "content": f"""Contexte : {context} Question : {question} AVANT de répondre, vérifie que le contexte contient l'information nécessaire."""} ], temperature=0.0 # Température à 0 pour max fidélité ) answer = response.choices[0].message.content # Détection automatique du "je ne sais pas" no_info_markers = [ "pas trouvé", "pas d'information", "ne contient pas", "n'ai pas trouvé", "information partielle", "pas disponible" ] has_answer = not any(m in answer.lower() for m in no_info_markers) return { "answer": answer, "has_answer": has_answer, "confidence": "high" if has_answer else "low" }

6.2 Vérification post-génération

def verify_grounding(answer: str, context: str, llm) -> dict: """Vérifie si la réponse est bien ancrée dans le contexte.""" verification_prompt = f"""Analyse cette réponse et détermine si CHAQUE affirmation est supportée par le contexte. CONTEXTE : {context} RÉPONSE À VÉRIFIER : {answer} Pour chaque affirmation de la réponse, indique : - "SUPPORTÉ" si elle vient directement du contexte - "INFÉRÉ" si c'est une déduction logique du contexte - "NON SUPPORTÉ" si elle n'est pas dans le contexte Retourne aussi un score de fidélité global (0-1).""" result = llm.generate(verification_prompt) return result
Ne faites JAMAIS confiance au LLM à 100% pour dire "je ne sais pas". Même avec des instructions explicites, il peut parfois halluciner. La vérification post-génération (grounding check) est une couche de sécurité essentielle en production.

from sentence_transformers import SentenceTransformer # Modèle multilingue : comprend 100+ langues model = SentenceTransformer("intfloat/multilingual-e5-large") # Documents en anglais docs_en = [ "The Eiffel Tower was completed in 1889.", "Python was created by Guido van Rossum in 1991." ] # Embeddings des documents anglais doc_embeddings = model.encode( [f"passage: {d}" for d in docs_en] ) # Question en français (cross-lingue !) question_fr = "Quand a été construite la Tour Eiffel ?" query_embedding = model.encode(f"query: {question_fr}") # La recherche sémantique fonctionne entre les langues from sklearn.metrics.pairwise import cosine_similarity import numpy as np similarities = cosine_similarity( [query_embedding], doc_embeddings )[0] best_idx = np.argmax(similarities) print(f"Meilleur document : {docs_en[best_idx]}") print(f"Score : {similarities[best_idx]:.3f}")
Meilleur document : The Eiffel Tower was completed in 1889. Score : 0.847

7.3 Prompt multilingue pour la génération

MULTILINGUAL_PROMPT = """Tu es un assistant multilingue. RÈGLES : - Les documents de contexte peuvent être dans n'importe quelle langue - Réponds TOUJOURS dans la langue de la question de l'utilisateur - Si un document est dans une autre langue, traduis l'information pertinente dans ta réponse - Cite les sources même si elles sont dans une autre langue Contexte (potentiellement multilangue) : {context} Question : {question} Réponse (dans la langue de la question) :"""
Les modèles d'embeddings multilingues comme multilingual-e5 ou Cohere multilingual permettent de faire du retrieval cross-lingue sans aucune étape de traduction, ce qui est plus rapide et évite les erreurs de traduction.

8. RAG Conversationnel

Le RAG conversationnel ajoute la dimension de la mémoire de conversation. Le défi principal : comment reformuler une question qui dépend du contexte de la conversation pour en faire une question autonome utilisable par le système de retrieval.

Le RAG conversationnel est un système qui maintient un historique de conversation et reformule automatiquement les questions contextuelles en questions autonomes avant d'effectuer le retrieval.

8.1 Condenseur de questions

class QuestionCondenser: """Reformule les questions contextuelles en questions autonomes.""" CONDENSE_PROMPT = """Étant donné l'historique de conversation et la nouvelle question, reformule la question pour qu'elle soit autonome (compréhensible sans l'historique). Historique de conversation : {chat_history} Nouvelle question : {question} Question autonome reformulée :""" def condense(self, question: str, history: list) -> str: if not history: return question # Pas besoin de reformuler formatted_history = "\n".join([ f"{'Utilisateur' if m['role']=='user' else 'Assistant'}: {m['content']}" for m in history[-6:] # Derniers 6 messages max ]) prompt = self.CONDENSE_PROMPT.format( chat_history=formatted_history, question=question ) return self.llm.generate(prompt)

8.2 Pipeline conversationnel complet

class ConversationalRAG: """RAG avec mémoire de conversation complète.""" def __init__(self, retriever, generator, condenser): self.retriever = retriever self.generator = generator self.condenser = condenser self.history = [] def chat(self, user_message: str) -> str: # Étape 1 : Reformuler la question avec le contexte standalone_question = self.condenser.condense( question=user_message, history=self.history ) print(f"Question reformulée: {standalone_question}") # Étape 2 : Retrieval avec la question autonome documents = self.retriever.search(standalone_question, k=5) # Étape 3 : Génération de la réponse answer = self.generator.generate( question=standalone_question, documents=documents, chat_history=self.history[-4:] # Contexte récent ) # Étape 4 : Mise à jour de l'historique self.history.append({"role": "user", "content": user_message}) self.history.append({"role": "assistant", "content": answer}) # Limiter la taille de l'historique if len(self.history) > 20: self.history = self.history[-20:] return answer def reset(self): """Réinitialise la conversation.""" self.history = [] # Exemple d'utilisation rag = ConversationalRAG(retriever, generator, condenser) # Tour 1 r1 = rag.chat("Parle-moi de Python") # Question reformulée: "Parle-moi de Python" # → Pas de reformulation nécessaire # Tour 2 r2 = rag.chat("Et qui l'a créé ?") # Question reformulée: "Qui a créé Python ?" # → "l'" a été résolu en "Python" grâce à l'historique # Tour 3 r3 = rag.chat("En quelle année ?") # Question reformulée: "En quelle année Python a-t-il été créé ?"
Tour 1 - Question reformulée: Parle-moi de Python Tour 2 - Question reformulée: Qui a créé le langage de programmation Python ? Tour 3 - Question reformulée: En quelle année le langage Python a-t-il été créé ?

8.3 Gestion avancée de la mémoire

class SmartMemory: """Mémoire intelligente qui résume les anciennes conversations.""" def __init__(self, max_messages=10, llm=None): self.messages = [] self.summary = "" self.max_messages = max_messages self.llm = llm def add(self, role: str, content: str): self.messages.append({"role": role, "content": content}) # Si trop de messages, résumer les anciens if len(self.messages) > self.max_messages: old_msgs = self.messages[:self.max_messages // 2] self.summary = self._summarize(old_msgs) self.messages = self.messages[self.max_messages // 2:] def _summarize(self, messages: list) -> str: text = "\n".join([f"{m['role']}: {m['content']}" for m in messages]) prompt = f"""Résume cette conversation en 2-3 phrases : {text} Résumé :""" return self.llm.generate(prompt) def get_context(self) -> str: parts = [] if self.summary: parts.append(f"Résumé des échanges précédents : {self.summary}") for m in self.messages: parts.append(f"{m['role']}: {m['content']}") return "\n".join(parts)
La gestion de la mémoire est un compromis entre contexte disponible et coût/latence. La technique de résumé progressif est utilisée par ChatGPT et d'autres chatbots pour maintenir une conversation cohérente sur de longues sessions sans dépasser la fenêtre de contexte.

9.3 Checklist de la génération RAG

  • Le prompt contient des instructions claires de fidélité au contexte
  • Le LLM sait dire "je ne sais pas" quand l'info manque
  • Les sources sont citées dans chaque réponse
  • La température est basse (0.0 - 0.2) pour maximiser la fidélité
  • Les contradictions entre sources sont signalées
  • La sortie est structurée (JSON/Pydantic) pour le parsing
  • La stratégie multi-documents est adaptée au cas d'usage
  • Le système gère les conversations multi-tours
  • Le prompt est testé avec des cas limites (pas d'info, contradictions)
  • Une vérification post-génération détecte les hallucinations
La génération est la partie la plus visible du RAG mais aussi la plus délicate. Un bon pipeline RAG investit autant dans le prompt engineering et la vérification que dans le retrieval.
