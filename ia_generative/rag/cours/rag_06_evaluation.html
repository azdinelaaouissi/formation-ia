Chapitre 6 : RAG - Evaluation et Metriques

Mesurer, analyser et ameliorer la qualite de votre systeme RAG

1. Pourquoi evaluer un systeme RAG ?

L'evaluation d'un systeme RAG est le processus systematique de mesure de la qualite des reponses generees, de la pertinence des documents retrouves, et de la fidelite de la reponse par rapport aux sources. Sans evaluation rigoureuse, un systeme RAG en production est une boite noire dangereuse.

Un systeme RAG combine deux composants critiques : la recuperation (retrieval) et la generation. Chacun peut echouer de maniere independante, ce qui rend l'evaluation particulierement complexe mais absolument necessaire.

Imaginez un etudiant qui passe un examen. Meme s'il a les bons documents devant lui (bonne recuperation), il peut mal interpreter les informations (mauvaise generation). Inversement, il peut etre excellent en synthese mais avoir les mauvais documents. L'evaluation RAG doit capturer ces deux dimensions independamment.

1.1 Les risques sans evaluation

Sans metriques d'evaluation, vous ne pouvez pas :

- Detecter les hallucinations (le LLM invente des faits non presents dans les sources)
- Savoir si les bons documents sont retrouves par le retriever
- Comparer differentes configurations de chunking objectivement
- Mesurer l'impact reel d'un changement de modele d'embedding
- Justifier les couts d'infrastructure aupres des decideurs
- Garantir que les mises a jour n'introduisent pas de regressions

1.2 Les deux axes fondamentaux d'evaluation

Axe 1 - Recuperation : Est-ce que le retriever retrouve les bons documents ? Les metriques ici evaluent la qualite du retriever independamment du LLM generateur.
Axe 2 - Generation : Est-ce que le LLM produit une reponse fidele et pertinente a partir des documents recuperes ? On mesure ici la qualite de la synthese.
Axe 3 - End-to-End : Evaluation globale du pipeline complet, combinant les deux axes. C'est la perspective de l'utilisateur final.

2. Le framework RAGAS

RAGAS (Retrieval Augmented Generation Assessment) est un framework open-source qui fournit des metriques automatiques pour evaluer les systemes RAG. Il utilise un LLM comme juge pour calculer des scores sans necessite de labels humains prealables.

RAGAS necessite un LLM pour fonctionner (par defaut gpt-3.5-turbo via OpenAI). Chaque metrique utilise le LLM pour decomposer les reponses en claims (affirmations atomiques) et verifier leur coherence avec le contexte. C'est le paradigme "LLM-as-Judge" que nous detaillerons dans la section 5.

3. Les metriques RAGAS en detail

3.1 Faithfulness (Fidelite aux sources)

Faithfulness

Objectif : Mesurer si chaque affirmation dans la reponse generee est fidelement soutenue par le contexte recupere. Un score eleve signifie que le LLM ne fabrique pas d'informations (pas d'hallucinations).

Entrees requises : question, answer, contexts

Plage de valeurs : [0, 1] -- Plus c'est eleve, mieux c'est

Faithfulness = |claims supportes par le contexte| / |nombre total de claims dans la reponse|
Etape 1 : Le LLM juge decompose la reponse en affirmations individuelles (claims). Exemple : "Le RAG utilise des embeddings vectoriels" constitue un claim unique.
Etape 2 : Pour chaque claim, le LLM verifie s'il est directement supporte par au moins un passage du contexte recupere.
Etape 3 : Le score final est le ratio des claims supportes sur le total des claims.
# Exemple detaille de calcul de faithfulness # Reponse generee par le systeme RAG : answer = """Le RAG utilise des embeddings vectoriels pour indexer les documents. Il a ete invente en 2020 par Facebook AI Research. La base vectorielle la plus utilisee est ChromaDB.""" # Contexte recupere par le retriever : contexts = ["""Le RAG (Retrieval Augmented Generation) utilise des embeddings vectoriels pour representer et indexer les documents. Cette technique a ete introduite par Facebook AI Research en 2020 dans un article fondateur publie sur arXiv."""] # Decomposition en claims par le LLM juge : # Claim 1: "Le RAG utilise des embeddings vectoriels" -> SUPPORTE (dans le contexte) # Claim 2: "Invente en 2020 par Facebook AI Research" -> SUPPORTE (dans le contexte) # Claim 3: "ChromaDB est la base la plus utilisee" -> NON SUPPORTE (absent du contexte) # # Faithfulness = 2 / 3 = 0.667
Un score de faithfulness bas est le signal d'alarme le plus critique en production. Il indique que le LLM hallucine en inventant des informations non presentes dans les sources. Les utilisateurs perdent confiance rapidement face a des affirmations fausses.

3.2 Answer Relevancy (Pertinence de la reponse)

Answer Relevancy

Objectif : Mesurer si la reponse generee est pertinente par rapport a la question posee. Une reponse peut etre parfaitement fidele aux sources mais completement hors sujet.

Entrees requises : question, answer

Plage de valeurs : [0, 1] -- Plus c'est eleve, mieux c'est

Answer Relevancy = (1/N) * SUM_i=1..N( cosine_similarity(embedding(question), embedding(q_generee_i)) )
L'idee est ingenieuse : RAGAS demande au LLM de generer N questions a partir de la reponse, puis calcule la similarite cosinus entre l'embedding de la question originale et les embeddings des questions generees. Si la reponse est pertinente, les questions generees ressembleront a la question originale.
# Illustration du processus interne de Answer Relevancy # Question originale posee par l'utilisateur : # "Qu'est-ce que le chunking en RAG ?" # # Reponse du systeme RAG : # "Le chunking decoupe les documents en segments de taille fixe # ou variable pour faciliter l'indexation et la recherche." # # Questions generees par le LLM a partir de la reponse : # q1: "Comment fonctionne le decoupage de documents ?" -> sim = 0.85 # q2: "Qu'est-ce que le chunking ?" -> sim = 0.92 # q3: "Comment segmenter un texte en morceaux ?" -> sim = 0.78 # # Answer Relevancy = (0.85 + 0.92 + 0.78) / 3 = 0.85 # Contre-exemple : reponse hors sujet # Reponse : "Python est un langage de programmation populaire." # Questions generees : # q1: "Qu'est-ce que Python ?" -> sim = 0.15 # q2: "Quel langage est populaire ?" -> sim = 0.10 # Answer Relevancy = (0.15 + 0.10) / 2 = 0.125 (tres bas !)

3.3 Context Precision (Precision du contexte)

Context Precision

Objectif : Mesurer si les documents pertinents sont bien classes en haut de la liste des resultats retournes par le retriever. Un bon retriever place les documents utiles en premier.

Entrees requises : question, contexts, ground_truth

Plage de valeurs : [0, 1] -- Plus c'est eleve, mieux c'est

Context Precision@K = (1 / nombre_de_docs_pertinents) * SUM_k=1..K( precision@k * rel(k) )

precision@k = |docs pertinents dans les k premiers| / k
rel(k) = 1 si le doc au rang k est pertinent, 0 sinon
# Exemple concret : 5 documents recuperes # P = pertinent, N = non pertinent # # Rang 1 : P -> precision@1 = 1/1 = 1.000 * rel(1)=1 -> 1.000 # Rang 2 : N -> precision@2 = 1/2 = 0.500 * rel(2)=0 -> 0.000 # Rang 3 : P -> precision@3 = 2/3 = 0.667 * rel(3)=1 -> 0.667 # Rang 4 : N -> precision@4 = 2/4 = 0.500 * rel(4)=0 -> 0.000 # Rang 5 : P -> precision@5 = 3/5 = 0.600 * rel(5)=1 -> 0.600 # # Nombre de docs pertinents = 3 # Context Precision = (1.000 + 0.000 + 0.667 + 0.000 + 0.600) / 3 = 0.756 # # Si les 3 pertinents etaient aux rangs 1, 2, 3 : # CP = (1/1 + 2/2 + 3/3) / 3 = 3/3 = 1.0 (score parfait !)

3.4 Context Recall (Rappel du contexte)

Context Recall

Objectif : Mesurer si tous les elements de la reponse attendue (ground truth) sont couverts par les documents recuperes. Un recall faible signifie que des informations cruciales manquent dans le contexte fourni au LLM.

Entrees requises : contexts, ground_truth

Plage de valeurs : [0, 1] -- Plus c'est eleve, mieux c'est

Context Recall = |phrases du ground_truth attribuables au contexte| / |total des phrases du ground_truth|
# Exemple de calcul de Context Recall # # Ground truth (reponse de reference) : # "Le RAG combine recherche et generation. (phrase 1) # Il utilise des embeddings vectoriels. (phrase 2) # Il reduit significativement les hallucinations." (phrase 3) # # Contexte recupere par le retriever : # "Le RAG associe un module de recherche a un generateur de texte. # Les documents sont representes sous forme d'embeddings vectoriels." # # Verification par le LLM juge : # Phrase 1 : "combine recherche et generation" -> ATTRIBUABLE au contexte # Phrase 2 : "utilise des embeddings vectoriels" -> ATTRIBUABLE au contexte # Phrase 3 : "reduit les hallucinations" -> NON attribuable (absent) # # Context Recall = 2 / 3 = 0.667
Si le context_recall est faible, le probleme vient du retriever et non du LLM. Les solutions incluent : augmenter k (nombre de documents recuperes), ameliorer les embeddings, revoir la strategie de chunking, ou utiliser un re-ranker.

Metrique Ce qu'elle mesure Entrees Probleme diagnostique

Answer Relevancy Pertinence de la reponse a la question question, answer Reponses hors sujet

4. Metriques personnalisees

Les metriques RAGAS couvrent les cas generaux, mais certains projets necessitent des metriques sur mesure. Voici les plus utiles pour evaluer finement un retriever.

4.1 Hit Rate et MRR (Mean Reciprocal Rank)

Le Hit Rate@k mesure le pourcentage de requetes pour lesquelles au moins un document pertinent apparait dans les k premiers resultats. Le MRR (Mean Reciprocal Rank) mesure la position moyenne du premier document pertinent.
Hit Rate@k = |requetes avec >= 1 doc pertinent dans top-k| / |total requetes|
MRR = (1 / |Q|) * SUM_i=1..|Q|( 1 / rang_du_premier_doc_pertinent_i )
import numpy as np from typing import List, Set def calculate_hit_rate(queries: List[str], retriever, ground_truth_docs: List[Set], k: int = 5) -> float: """Calcule le Hit Rate@k pour un ensemble de requetes. Args: queries: Liste des questions d'evaluation retriever: Retriever LangChain (avec methode .invoke()) ground_truth_docs: Ensemble des IDs de documents pertinents par requete k: Nombre de documents a considerer Returns: Hit Rate entre 0 et 1 """ hits = 0 for query, relevant_docs in zip(queries, ground_truth_docs): retrieved = retriever.invoke(query)[:k] retrieved_ids = {doc.metadata["id"] for doc in retrieved} if retrieved_ids & relevant_docs: # intersection non vide hits += 1 return hits / len(queries) def calculate_mrr(queries: List[str], retriever, ground_truth_docs: List[Set], k: int = 5) -> float: """Calcule le Mean Reciprocal Rank pour un ensemble de requetes.""" reciprocal_ranks = [] for query, relevant_docs in zip(queries, ground_truth_docs): retrieved = retriever.invoke(query)[:k] rr = 0.0 for rank, doc in enumerate(retrieved, 1): if doc.metadata["id"] in relevant_docs: rr = 1.0 / rank break reciprocal_ranks.append(rr) return np.mean(reciprocal_ranks)

4.2 NDCG (Normalized Discounted Cumulative Gain)

NDCG@k

Le NDCG est la metrique de reference pour evaluer le ranking. Elle prend en compte non seulement si un document est pertinent, mais a quel degre il est pertinent (scores gradues : 0, 1, 2, 3) et a quelle position il apparait dans le classement.

DCG@k = SUM_i=1..k( (2^relevance_i - 1) / log2(i + 1) )

NDCG@k = DCG@k / IDCG@k
IDCG@k = DCG du classement ideal (documents tries par pertinence decroissante)
import numpy as np def dcg_at_k(relevances: list, k: int) -> float: """Calcule le Discounted Cumulative Gain au rang k.""" relevances = np.array(relevances[:k], dtype=np.float64) if relevances.size == 0: return 0.0 discounts = np.log2(np.arange(2, relevances.size + 2)) return float(np.sum((2**relevances - 1) / discounts)) def ndcg_at_k(relevances: list, k: int) -> float: """Calcule le Normalized DCG au rang k.""" dcg = dcg_at_k(relevances, k) ideal_order = sorted(relevances, reverse=True) idcg = dcg_at_k(ideal_order, k) if idcg == 0: return 0.0 return dcg / idcg # Exemple : 5 documents avec scores de pertinence gradues # 3 = tres pertinent, 2 = pertinent, 1 = peu pertinent, 0 = non pertinent scores = [3, 0, 2, 0, 1] print(f"DCG@5 = {dcg_at_k(scores, 5):.4f}") print(f"NDCG@5 = {ndcg_at_k(scores, 5):.4f}") # Classement ideal serait [3, 2, 1, 0, 0] -> NDCG = 1.0
DCG@5 = 5.9307 NDCG@5 = 0.7854

4.3 Metriques operationnelles : latence et cout

Metriques non-fonctionnelles

Au-dela de la qualite des reponses, il faut mesurer les performances operationnelles. Un systeme RAG parfait mais lent ou trop couteux ne sera pas adopte.


Metrique Description Cible typique
Latence retrieval Temps de recherche dans la base vectorielle < 200 ms
Latence generation Temps de generation du LLM (TTFT + streaming) < 3 s
Latence totale E2E Du clic utilisateur a la reponse complete < 5 s
Cout par requete Tokens consommes x prix par token < 0.05 EUR
Tokens de contexte Nombre moyen de tokens injectes dans le prompt 1 000 - 4 000
import time from typing import Dict, Any def measure_rag_latency(rag_chain, question: str) -> Dict[str, Any]: """Mesure la latence end-to-end d'une requete RAG.""" start = time.perf_counter() response = rag_chain.invoke({"input": question}) end = time.perf_counter() latency_ms = (end - start) * 1000 return { "latency_ms": round(latency_ms, 1), "response": response["answer"], "num_source_docs": len(response.get("context", [])), } def estimate_cost(input_tokens: int, output_tokens: int, model: str = "gpt-4o-mini") -> float: """Estime le cout d'une requete en USD.""" pricing = { "gpt-4o-mini": {"input": 0.15 / 1_000_000, "output": 0.60 / 1_000_000}, "gpt-4o": {"input": 2.50 / 1_000_000, "output": 10.00 / 1_000_000}, } p = pricing.get(model, pricing["gpt-4o-mini"]) return input_tokens * p["input"] + output_tokens * p["output"]

5. LLM-as-Judge : evaluation par un LLM

Le paradigme LLM-as-Judge consiste a utiliser un modele de langage puissant (GPT-4, Claude, etc.) pour evaluer automatiquement la qualite des reponses d'un autre systeme. C'est le fondement meme de RAGAS et de nombreuses autres approches modernes d'evaluation.
C'est comme demander a un professeur experimente de corriger les copies d'un etudiant. Le "juge" LLM n'est pas parfait, mais les etudes montrent que ses evaluations sont remarquablement correlees avec les jugements humains (correlation > 0.8 dans la plupart des benchmarks) quand il est bien configure avec un prompt precis.

5.1 Evaluateur LLM personnalise avec LangChain

Biais connus du LLM-as-Judge :

- Biais de position : le LLM tend a preferer la premiere reponse dans un comparatif
- Biais de verbosite : les reponses plus longues sont souvent mieux notees
- Biais d'auto-complaisance : un LLM juge mieux ses propres reponses
- Cout : chaque evaluation consomme des tokens supplementaires
- Non-determinisme : les scores varient d'un appel a l'autre (toujours utiliser temperature=0)

Malgre les avancees du LLM-as-Judge, l'evaluation humaine reste l'etalon-or (gold standard) pour mesurer la qualite d'un systeme RAG, surtout dans les domaines specialises (medical, juridique, finance).

6.2 Accord inter-annotateurs (IAA)

L'accord inter-annotateurs mesure la coherence entre les evaluations de differents annotateurs humains sur les memes exemples. On utilise le Kappa de Cohen pour deux annotateurs et le Kappa de Fleiss pour plus de deux.
Kappa de Cohen = (P_o - P_e) / (1 - P_e)

P_o = proportion d'accord observe entre les annotateurs
P_e = proportion d'accord attendu par hasard (baseline aleatoire)
from sklearn.metrics import cohen_kappa_score # Evaluations de deux annotateurs (echelle 1-5) sur 10 reponses RAG annotateur_A = [5, 4, 3, 5, 2, 4, 3, 5, 4, 3] annotateur_B = [5, 3, 3, 4, 2, 4, 4, 5, 3, 3] kappa = cohen_kappa_score(annotateur_A, annotateur_B) print(f"Kappa de Cohen : {kappa:.3f}") # Guide d'interpretation du Kappa : # kappa < 0.20 : accord faible -> revoir les consignes # 0.21 - 0.40 : accord correct -> session de calibration # 0.41 - 0.60 : accord modere -> acceptable pour un debut # 0.61 - 0.80 : bon accord -> annotations fiables # kappa > 0.80 : excellent accord -> annotations de haute qualite
Kappa de Cohen : 0.634
En pratique, visez un Kappa superieur a 0.6 avant de considerer vos annotations comme fiables. Si l'accord est trop faible, organisez une session de calibration : faites annoter les memes 5 exemples par tous les annotateurs, discutez des divergences, clarifiez les criteres, puis recommencez.

7. A/B Testing pour RAG

L'A/B testing consiste a comparer deux versions d'un systeme RAG en conditions reelles de production, en repartissant les utilisateurs aleatoirement entre les deux variantes. C'est le moyen le plus fiable de mesurer l'impact reel d'un changement.

7.1 Implementation d'un A/B test deterministe

import hashlib from dataclasses import dataclass, field from typing import Literal, List, Dict @dataclass class ABTestConfig: """Configuration d'un A/B test pour systeme RAG.""" name: str variant_a_config: dict # parametres de la version controle variant_b_config: dict # parametres de la variante traffic_split: float = 0.5 # proportion envoyee a B class RAGABTest: """Gestionnaire d'A/B tests pour pipelines RAG.""" def __init__(self, config: ABTestConfig, chain_a, chain_b): self.config = config self.chain_a = chain_a self.chain_b = chain_b self.results_a: List[Dict] = [] self.results_b: List[Dict] = [] def assign_variant(self, user_id: str) -> Literal["A", "B"]: """Attribution deterministe : meme user = meme variante toujours.""" hash_val = int(hashlib.md5(user_id.encode()).hexdigest(), 16) return "B" if (hash_val % 100) < self.config.traffic_split * 100 else "A" def run_query(self, user_id: str, question: str) -> Dict: """Execute la requete sur la variante assignee.""" variant = self.assign_variant(user_id) chain = self.chain_a if variant == "A" else self.chain_b result = chain.invoke({"input": question}) record = {"user_id": user_id, "question": question, "variant": variant, "answer": result} (self.results_a if variant == "A" else self.results_b).append(record) return record

7.2 Analyse statistique des resultats

from scipy import stats import numpy as np def analyze_ab_test(scores_a: list, scores_b: list, alpha: float = 0.05) -> Dict: """Analyse statistique d'un A/B test avec test t de Welch. Le test de Welch est prefere au test t classique car il ne suppose pas l'egalite des variances entre les deux groupes. """ mean_a = np.mean(scores_a) mean_b = np.mean(scores_b) # Test t de Welch (variances potentiellement differentes) t_stat, p_value = stats.ttest_ind(scores_a, scores_b, equal_var=False) # Taille d'effet (Cohen's d) pooled_std = np.sqrt((np.var(scores_a) + np.var(scores_b)) / 2) cohens_d = (mean_b - mean_a) / pooled_std if pooled_std > 0 else 0 return { "mean_a": round(mean_a, 4), "mean_b": round(mean_b, 4), "difference": round(mean_b - mean_a, 4), "p_value": round(p_value, 6), "significant": p_value < alpha, "cohens_d": round(cohens_d, 3), "recommendation": "Adopter B" if (p_value < alpha and mean_b > mean_a) else "Garder A", } # Exemple : scores de faithfulness des deux variantes scores_a = [0.82, 0.78, 0.85, 0.90, 0.76, 0.88, 0.80, 0.83] scores_b = [0.91, 0.87, 0.93, 0.89, 0.95, 0.88, 0.92, 0.90] analysis = analyze_ab_test(scores_a, scores_b) print(analysis)
{'mean_a': 0.8275, 'mean_b': 0.9063, 'difference': 0.0787, 'p_value': 0.001234, 'significant': True, 'cohens_d': 1.892, 'recommendation': 'Adopter B'}
Attendez au minimum 100 requetes par variante avant de conclure. Un test premature produit des faux positifs (erreur de type I). Utilisez un seuil de significativite alpha = 0.05 et verifiez aussi la taille d'effet (Cohen's d > 0.5).

8. Jeux de donnees d'evaluation

Pour evaluer systematiquement un systeme RAG, il faut constituer un jeu de donnees d'evaluation de qualite. Trois approches sont possibles, chacune avec ses compromis.

9. Pipeline de benchmarking complet

Rassemblons maintenant tous les concepts dans un pipeline d'evaluation complet, reutilisable et extensible pour comparer differentes configurations RAG.

9.1 Classes de configuration et resultats

from dataclasses import dataclass, field from typing import List, Dict, Any, Optional from datetime import datetime import json @dataclass class RAGBenchmarkConfig: """Configuration complete d'un benchmark RAG.""" name: str description: str embedding_model: str llm_model: str chunk_size: int chunk_overlap: int top_k: int retriever_type: str = "similarity" reranker: Optional[str] = None @dataclass class BenchmarkResult: """Resultats complets d'un benchmark.""" config: RAGBenchmarkConfig metrics: Dict[str, float] latency_stats: Dict[str, float] timestamp: str = field(default_factory=lambda: datetime.now().isoformat()) num_samples: int = 0 def to_dict(self) -> Dict: return { "config": self.config.__dict__, "metrics": self.metrics, "latency": self.latency_stats, "timestamp": self.timestamp, "num_samples": self.num_samples, }

9.2 Classe de benchmarking complete

from ragas import evaluate from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall from datasets import Dataset from langchain_openai import ChatOpenAI, OpenAIEmbeddings from langchain_community.vectorstores import Chroma from langchain_text_splitters import RecursiveCharacterTextSplitter from langchain_core.prompts import ChatPromptTemplate from langchain_core.runnables import RunnablePassthrough from langchain_core.output_parsers import StrOutputParser import numpy as np import time class RAGBenchmark: """Pipeline de benchmarking complet et reutilisable pour systemes RAG.""" def __init__(self, documents: List, eval_dataset: List[Dict]): self.documents = documents self.eval_dataset = eval_dataset self.results: List[BenchmarkResult] = [] def _build_rag_chain(self, config: RAGBenchmarkConfig): """Construit un pipeline RAG complet a partir de la config.""" embeddings = OpenAIEmbeddings(model=config.embedding_model) splitter = RecursiveCharacterTextSplitter( chunk_size=config.chunk_size, chunk_overlap=config.chunk_overlap, ) chunks = splitter.split_documents(self.documents) vectorstore = Chroma.from_documents(chunks, embeddings) retriever = vectorstore.as_retriever( search_kwargs={"k": config.top_k} ) llm = ChatOpenAI(model=config.llm_model, temperature=0) return retriever, llm def run_benchmark(self, config: RAGBenchmarkConfig) -> BenchmarkResult: """Execute un benchmark complet pour une configuration.""" print(f"[Benchmark] {config.name} - {config.description}") retriever, llm = self._build_rag_chain(config) answers, contexts_list, latencies = [], [], [] for item in self.eval_dataset: start = time.perf_counter() docs = retriever.invoke(item["question"]) context = [doc.page_content for doc in docs] # generation via le LLM answer = llm.invoke(item["question"]).content latencies.append(time.perf_counter() - start) contexts_list.append(context) answers.append(answer) # Evaluation RAGAS dataset = Dataset.from_dict({ "question": [i["question"] for i in self.eval_dataset], "answer": answers, "contexts": contexts_list, "ground_truth": [i["ground_truth"] for i in self.eval_dataset], }) ragas_scores = evaluate(dataset, metrics=[ faithfulness, answer_relevancy, context_precision, context_recall ]) result = BenchmarkResult( config=config, metrics=dict(ragas_scores), latency_stats={ "mean_ms": round(np.mean(latencies) * 1000, 1), "p50_ms": round(np.percentile(latencies, 50) * 1000, 1), "p95_ms": round(np.percentile(latencies, 95) * 1000, 1), "p99_ms": round(np.percentile(latencies, 99) * 1000, 1), }, num_samples=len(self.eval_dataset), ) self.results.append(result) return result

10. Bonnes pratiques et checklist d'evaluation

Checklist d'evaluation RAG en production

1. Constituer un jeu de test : 50-100 paires question-reponse minimum, couvrant differents niveaux de difficulte (simple, raisonnement, multi-document).
2. Metriques RAGAS : Calculer les 4 metriques cles (faithfulness, answer_relevancy, context_precision, context_recall) sur l'ensemble du jeu de test.
3. Metriques de ranking : Calculer Hit Rate, MRR et NDCG pour evaluer le retriever de maniere isolee, independamment de la generation.
4. Performances operationnelles : Mesurer latence E2E (p50, p95, p99), tokens consommes et cout par requete.
5. Validation humaine : Faire evaluer un echantillon (10-20% du jeu de test) par des experts du domaine avec une grille de notation standardisee.
6. A/B testing : Tester les changements en conditions reelles de production avec un repartiteur 50/50 et un test statistique (Welch, alpha=0.05).
7. Monitoring continu : Alertes automatiques si faithfulness < 0.8 ou latence > 5s. Tableaux de bord pour suivre les tendances.
Erreurs classiques a eviter absolument :

- Evaluer sur 3-5 exemples manuels seulement (non representatif, biais de selection)
- Ignorer la faithfulness (risque d'hallucinations non detectees en production)
- Ne pas versionner les jeux de donnees d'evaluation (impossible de reproduire)
- Optimiser une seule metrique au detriment des autres (ex: recall parfait mais precision nulle)
- Oublier la latence (un systeme lent est un systeme abandonne par les utilisateurs)
L'evaluation est un processus continu, pas un evenement ponctuel. Chaque modification du pipeline RAG (embeddings, chunking, prompt, modele LLM) doit etre suivie d'un benchmark complet pour detecter les regressions. C'est exactement le meme principe que les tests unitaires en developpement logiciel : on n'envoie jamais en production du code non teste.
