Introduction au RAG - Retrieval-Augmented Generation

Cours 1/4 -- Comprendre l'architecture qui revolutionne l'utilisation des LLMs

1. Qu'est-ce que le RAG ?

RAG (Retrieval-Augmented Generation) est une architecture d'intelligence artificielle qui ameliore les reponses d'un grand modele de langage (LLM) en lui donnant acces a des sources de connaissances externes. Au lieu de se fier uniquement a sa memoire parametrique (les poids appris pendant l'entrainement), le modele recherche d'abord des documents pertinents dans une base de connaissances, puis les injecte dans le contexte du prompt avant de generer sa reponse.

Le terme RAG a ete introduit par Patrick Lewis et al. (Meta AI) dans leur article fondateur de 2020 : "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks". L'idee centrale est simple mais puissante : plutot que de demander a un LLM de "tout savoir", on lui donne la capacite de "chercher avant de repondre".

Pensez-y comme un etudiant qui passe un examen. Un LLM classique est un etudiant qui doit tout memoriser et repondre de memoire. Un systeme RAG est un etudiant qui a le droit d'ouvrir ses livres et ses notes pendant l'examen -- il peut chercher l'information exacte dont il a besoin avant de formuler sa reponse.

Le RAG est devenu en 2024-2025 la technique la plus utilisee en entreprise pour deployer des LLMs sur des donnees privees. C'est LE pattern architectural que vous devez maitriser absolument. Selon les derniers rapports, plus de 80% des applications LLM en production utilisent une forme de RAG.

1.1 Le flux fondamental du RAG

Le schema ci-dessus illustre les trois mots qui composent RAG :

  • Retrieval (Recherche) : On cherche les documents les plus pertinents par rapport a la question de l'utilisateur dans une base vectorielle.
  • Augmented (Augmentation) : On injecte ces documents dans le prompt du LLM comme contexte supplementaire.
  • Generation : Le LLM genere une reponse en se basant sur ce contexte, produisant une reponse fondee et sourcee.

1.2 Exemple concret minimal

# Exemple simplifie d'un pipeline RAG avec LangChain from langchain_community.document_loaders import TextLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain_community.vectorstores import FAISS from langchain.chains import RetrievalQA # 1. Charger le document loader = TextLoader("documentation_produit.txt") documents = loader.load() # 2. Decouper en chunks splitter = RecursiveCharacterTextSplitter( chunk_size=500, chunk_overlap=50 ) chunks = splitter.split_documents(documents) print(f"Nombre de chunks : {len(chunks)}") # 3. Creer les embeddings et le vector store embeddings = OpenAIEmbeddings(model="text-embedding-3-small") vectorstore = FAISS.from_documents(chunks, embeddings) # 4. Creer la chaine RAG llm = ChatOpenAI(model="gpt-4o", temperature=0) qa_chain = RetrievalQA.from_chain_type( llm=llm, retriever=vectorstore.as_retriever(search_kwargs={"k": 4}), return_source_documents=True ) # 5. Poser une question result = qa_chain.invoke({"query": "Quelle est la politique de retour ?"}) print(result["result"]) print("Sources :", [doc.metadata["source"] for doc in result["source_documents"]])
Nombre de chunks : 47 La politique de retour permet un remboursement complet dans les 30 jours suivant l'achat. Le produit doit etre retourne dans son emballage d'origine. Les frais de retour sont a la charge du client sauf en cas de defaut. Sources : ['documentation_produit.txt', 'documentation_produit.txt']
En seulement 15 lignes de code, nous avons un systeme RAG fonctionnel qui peut repondre a des questions sur n'importe quel document texte, avec des sources citees. C'est la puissance du pattern RAG.

2. Pourquoi le RAG ? Les problemes des LLMs

Les grands modeles de langage (GPT-4, Claude, Gemini, Llama) sont remarquablement puissants, mais ils souffrent de plusieurs limitations fondamentales qui rendent le RAG indispensable pour les applications en production :

2.1 Les 4 problemes majeurs des LLMs

Probleme 1 : Hallucinations

Les LLMs generent parfois des informations completement fausses mais qui semblent plausibles et confiantes. C'est le probleme le plus critique pour les applications d'entreprise. Un LLM peut inventer des faits, des chiffres, voire des references bibliographiques qui n'existent pas.

Exemple : "Selon l'article 42-B du code du travail..." -- cet article n'existe pas, mais le LLM le presente comme un fait.

Probleme 2 : Connaissances obsoletes

Un LLM est entraine sur des donnees avec une date de coupure (knowledge cutoff). GPT-4 a ete entraine sur des donnees jusqu'a avril 2024, Claude sur des donnees jusqu'en 2025. Toute information apparue apres cette date est inconnue du modele.

Exemple : Un LLM ne peut pas connaitre les resultats financiers du dernier trimestre ou les dernieres mises a jour de votre produit.

Probleme 3 : Pas de connaissances specifiques au domaine

Les LLMs sont entraines sur des donnees publiques (Internet). Ils ne connaissent pas vos documents internes, votre base de connaissances client, votre documentation technique privee, vos contrats, vos procedures.

Exemple : "Quelle est la procedure de validation des commandes chez nous ?" -- le LLM ne peut pas repondre car cette info est interne.

Probleme 4 : Pas de sources citables

Un LLM genere du texte "de memoire" sans pouvoir citer ses sources. En entreprise, la tracabilite est essentielle : on doit savoir d'ou vient l'information pour pouvoir la verifier.

Exemple : Un chatbot juridique doit citer les articles de loi exacts sur lesquels il base son avis.

2.2 Comment le RAG resout chaque probleme

Probleme LLM Solution RAG Mecanisme
Hallucinations Grounding (ancrage factuel) Le LLM base sa reponse sur les documents fournis, pas sur sa memoire. On peut instruire : "Reponds uniquement a partir du contexte fourni"
Connaissances obsoletes Donnees toujours a jour La base vectorielle peut etre mise a jour en continu avec les derniers documents, sans reentrainer le modele
Pas de donnees privees Acces aux documents internes On indexe les documents proprietaires dans le vector store. Le LLM y accede au moment de la requete sans jamais stocker ces donnees dans ses poids
Pas de sources Citations et tracabilite Chaque chunk recupere porte des metadonnees (source, page, date). Le systeme peut citer exactement d'ou vient chaque information
Le RAG ne supprime pas totalement les hallucinations, il les reduit considerablement. Un LLM peut encore halluciner meme avec du contexte pertinent, par exemple en melangeant des informations de differents chunks. C'est pourquoi les strategies de retrieval avancees (cours 4) et l'evaluation (metriques de faithfulness) sont cruciales.

3. Architecture globale du RAG

L'architecture RAG se decompose en deux phases distinctes qui operent a des moments differents : la phase d'indexation (hors-ligne, une seule fois) et la phase de requete (en ligne, a chaque question).

4. Les 2 Phases du RAG en detail

4.1 Phase 1 : Indexation (Hors-ligne)

La phase d'indexation est executee une seule fois (ou periodiquement pour mettre a jour la base). Elle prepare les documents pour une recherche rapide et semantique.

Etape 1 - Chargement des documents (Document Loading)
On charge les documents bruts depuis differentes sources : fichiers PDF, pages web, bases de donnees, fichiers CSV, documents Word. Chaque type de source a son propre loader. Les metadonnees sont extraites automatiquement (nom du fichier, chemin, date de modification).
Etape 2 - Decoupage en chunks (Text Splitting)
Les documents sont decoupes en morceaux de taille geree (typiquement 200 a 1000 tokens). Le decoupage est critique : trop petit = pas assez de contexte, trop grand = trop de bruit. On utilise generalement le RecursiveCharacterTextSplitter avec un overlap pour eviter de couper au milieu d'une idee. Les strategies avancees incluent le chunking semantique et hierarchique (voir cours 2).
Etape 3 - Creation des embeddings
Chaque chunk de texte est transforme en vecteur numerique (embedding) grace a un modele d'embedding (ex: text-embedding-3-small). Ce vecteur capture le sens semantique du texte. Deux textes ayant un sens similaire auront des vecteurs proches dans l'espace vectoriel, meme si les mots utilises sont differents.
Etape 4 - Stockage dans le Vector Store
Les vecteurs sont stockes dans une base de donnees vectorielle (FAISS, Chroma, Pinecone, Qdrant, Weaviate) qui permet des recherches par similarite tres rapides, meme sur des millions de vecteurs. L'index vectoriel est optimise pour les requetes de type "plus proche voisin" (k-NN).

4.2 Phase 2 : Requete (En ligne)

La phase de requete est executee a chaque question de l'utilisateur. Elle est optimisee pour etre rapide (typiquement < 2 secondes).

Etape 1 - Embedding de la question
La question de l'utilisateur est transformee en vecteur avec le meme modele d'embedding que celui utilise pour l'indexation. C'est essentiel : le modele d'embedding pour les questions et pour les documents doit etre identique.
Etape 2 - Recherche par similarite
Le vecteur de la question est compare aux vecteurs des chunks dans le vector store. Les k chunks les plus similaires (typiquement k=4) sont recuperes. Des strategies avancees (MMR, filtrage par metadonnees, hybrid search) ameliorent cette etape (voir cours 4).
Etape 3 - Construction du prompt augmente
Les chunks recuperes sont injectes dans le prompt du LLM en tant que contexte. Le prompt contient typiquement : (1) des instructions systeme ("Reponds en te basant uniquement sur le contexte"), (2) le contexte (les chunks), (3) la question de l'utilisateur.
Etape 4 - Generation de la reponse
Le LLM genere une reponse en se basant sur le contexte fourni. Il peut citer ses sources, admettre qu'il ne sait pas si l'information n'est pas dans le contexte, et fournir une reponse precise et fondee.
L'erreur la plus frequente des debutants est d'utiliser un modele d'embedding different entre l'indexation et la requete. Les vecteurs ne seront pas dans le meme espace et la recherche par similarite ne fonctionnera pas du tout.
# Illustration : le prompt augmente envoye au LLM prompt_template = """Tu es un assistant qui repond aux questions en te basant UNIQUEMENT sur le contexte fourni ci-dessous. Si l'information n'est pas dans le contexte, dis "Je ne dispose pas de cette information dans les documents fournis." Contexte : --- {context} --- Question : {question} Reponse (avec references aux sources) :""" # Exemple de contexte injecte context = """ [Source: politique_retour.pdf, page 3] Les retours sont acceptes dans un delai de 30 jours apres reception. 5. RAG vs Fine-tuning : quand utiliser quoi ?

Le fine-tuning et le RAG sont deux approches differentes pour adapter un LLM a un domaine specifique. Ils ne sont pas mutuellement exclusifs mais repondent a des besoins differents.

Fine-tuning

On modifie les poids du modele en l'entrainant sur des donnees specifiques. Le modele "apprend" de nouvelles connaissances et/ou un nouveau style de reponse. Cela necessite des donnees d'entrainement, du calcul GPU, et du temps.

RAG

On ne modifie pas le modele. On lui fournit dynamiquement les informations pertinentes au moment de la requete. Pas d'entrainement necessaire, mise a jour instantanee en modifiant la base documentaire.

Critere RAG Fine-tuning
Cout initial Faible (embeddings + vector store) Eleve (GPU, donnees, expertise)
Fraicheur des donnees Temps reel (mise a jour du vector store) Statique (necessite re-entrainement)
Connaissances factuelles Excellent (documents exacts fournis) Moyen (peut "oublier" ou melanger)
Style / format de reponse Moyen (depend du prompt) Excellent (appris dans les poids)
Reduction des hallucinations Forte (ancrage factuel) Moyenne (ameliore mais n'elimine pas)
Tracabilite / Sources Oui (les sources sont connues) Non (melange dans les poids)
Volume de donnees necessaire N'importe quelle quantite Milliers d'exemples minimum
Temps de mise en place Heures a jours Jours a semaines
Latence en inference Legerement plus lente (retrieval + generation) Standard (generation seule)
Cas d'usage ideal Q&A sur documents, chatbot entreprise, support client Style specifique, taches specialisees, classification
En pratique, la combinaison RAG + Fine-tuning est souvent la meilleure approche pour les cas d'usage exigeants. Le fine-tuning donne au modele le bon "style" et le bon format de reponse, tandis que le RAG lui fournit les connaissances factuelles actualisees. Mais commencez TOUJOURS par le RAG seul -- c'est plus rapide, moins cher, et souvent suffisant.

6. RAG vs Long Context : pourquoi le RAG reste pertinent

Avec l'emergence de fenetres de contexte de plus en plus grandes (Claude 200K tokens, Gemini 1M tokens, GPT-4o 128K tokens), une question legitime se pose : pourquoi ne pas simplement injecter tous les documents dans le contexte du LLM sans passer par le retrieval ?

Critere RAG (Retrieval) Long Context (Stuffing)
Cout par requete Faible (seulement k chunks) Tres eleve (tous les tokens factures)
Latence Rapide (petit prompt) Lente (prompt enorme a traiter)
Precision Forte (chunks pertinents cibles) Variable (info noyee dans le bruit)
Scalabilite Millions de documents Limitee par la fenetre de contexte
Probleme "Lost in the Middle" Evite (peu de contexte, bien cible) Oui (LLMs ignorent le milieu du contexte)
Mise en place Plus complexe (pipeline a construire) Simple (copier-coller dans le prompt)

Le probleme "Lost in the Middle"

Des recherches (Liu et al., 2023) ont montre que les LLMs ont tendance a mieux traiter les informations au debut et a la fin du contexte, mais oublient ou ignorent les informations au milieu. Avec un contexte de 200K tokens, une information cruciale situee au milieu peut etre ignoree. Le RAG, en ne fournissant que les chunks les plus pertinents (typiquement 4-8), evite ce probleme.

En 2025-2026, l'approche hybride "RAG + Long Context" est la tendance : on utilise le RAG pour selectionner les chunks pertinents, puis on peut en inclure plus (20-50) grace aux grandes fenetres de contexte. C'est le concept de "Context Engineering" -- l'art d'assembler le contexte optimal pour chaque requete. Cela dit, pour la grande majorite des applications, le RAG classique avec k=4 a 8 chunks reste la meilleure approche en termes de rapport qualite/cout.

7. Les composants du RAG

Un systeme RAG est compose de plusieurs briques technologiques qui s'assemblent en pipeline. Chaque composant a un role precis et peut etre interchangeable.

Composant Role Exemples Cours detaille
Document Loaders Charger les documents bruts depuis differentes sources PyPDFLoader, WebBaseLoader, CSVLoader, UnstructuredLoader Cours 2
Text Splitters Decouper les documents en chunks de taille geree RecursiveCharacterTextSplitter, SemanticChunker, TokenTextSplitter Cours 2
Embedding Models Transformer le texte en vecteurs numeriques OpenAI text-embedding-3, Cohere embed-v4, sentence-transformers Cours 3
Vector Stores Stocker et rechercher les vecteurs efficacement FAISS, Chroma, Pinecone, Qdrant, Weaviate, Milvus Cours 3
Retrievers Strategies de recherche des documents pertinents Similarity, MMR, Multi-Query, HyDE, Self-Query, Hybrid Cours 4
LLM (Generateur) Generer la reponse a partir du contexte recupere GPT-4o, Claude 3.5, Llama 3, Mistral, Gemini Tous les cours
Prompt Templates Structurer le prompt avec contexte et question LangChain PromptTemplate, ChatPromptTemplate Tous les cours

8. Types de RAG : du Naive au Modular

L'evolution du RAG a donne naissance a trois generations d'architectures, chacune adressant les limitations de la precedente. Comprendre ces differences est essentiel pour choisir la bonne approche selon votre cas d'usage.

8.1 Naive RAG (Generation 1)

L'architecture originale et la plus simple. Pipeline lineaire : Indexation -> Retrieval -> Generation. Pas d'optimisation du retrieval, pas de post-traitement des resultats.

Limites :

  • Chunks non pertinents souvent recuperes (precision faible)
  • Pas de diversite dans les resultats (redundance)
  • La question de l'utilisateur est utilisee telle quelle pour la recherche
  • Pas de reranking ni de filtrage intelligent

8.2 Advanced RAG (Generation 2)

Ajoute des optimisations avant et apres le retrieval :

  • Pre-retrieval : Reformulation de la question (Multi-Query, HyDE), enrichissement de la requete, routing vers le bon index
  • Retrieval : Hybrid search (BM25 + vecteurs), MMR pour la diversite, filtrage par metadonnees
  • Post-retrieval : Reranking des resultats, compression contextuelle, filtrage des chunks non pertinents

Gains : Amelioration de 10 a 40% de la precision par rapport au Naive RAG selon les benchmarks.

8.3 Modular RAG (Generation 3)

Architecture modulaire et flexible ou chaque composant peut etre interchangeable et configurable independamment :

  • Routing : Diriger la question vers le bon pipeline selon le type de question
  • Agentic RAG : Un agent decide quand et comment utiliser le retrieval, peut faire plusieurs iterations, utiliser des outils
  • Self-RAG : Le modele decide lui-meme s'il a besoin de retrieval et evalue la qualite de ses propres reponses
  • Graph RAG : Utilise un graphe de connaissances en plus des vecteurs pour capturer les relations entre entites
  • Adaptive RAG : Ajuste dynamiquement la strategie de retrieval selon la complexite de la question
En pratique, commencez TOUJOURS par un Naive RAG pour valider votre cas d'usage, puis evoluez vers un Advanced RAG en ajoutant des optimisations une par une. Le Modular RAG est reserve aux cas complexes (multi-sources, questions complexes, exigences de production elevees). Mesurez l'impact de chaque amelioration avec des metriques (RAGAS : faithfulness, answer relevancy, context precision).

9. Metriques d'evaluation du RAG

Evaluer un systeme RAG est essentiel pour mesurer sa qualite et guider les ameliorations. Le framework RAGAS (Retrieval Augmented Generation Assessment) est la reference pour l'evaluation automatique.

9.1 Les metriques cles

Metrique Ce qu'elle mesure Formule simplifiee Score ideal
Faithfulness La reponse est-elle fidele au contexte recupere ? (pas d'hallucination) Claims supportes par le contexte / Total claims 1.0
Answer Relevancy La reponse est-elle pertinente par rapport a la question ? Similarite semantique entre la reponse et la question 1.0
Context Precision Les chunks recuperes sont-ils pertinents ? (pas de bruit) Chunks pertinents / Total chunks recuperes 1.0
Context Recall Tous les chunks necessaires ont-ils ete recuperes ? Infos necessaires trouvees / Infos necessaires totales 1.0

10. Resume et checklist

Checklist de l'architecture RAG

  1. Documents source -- Identifie les documents a indexer (PDF, web, DB, etc.)
  2. Strategie de chunking -- Choisi la taille et le chevauchement des chunks
  3. Modele d'embedding -- Selectionne le bon modele (qualite vs cout vs dimensions)
  4. Vector store -- Choisi la base vectorielle (local vs cloud, prototype vs production)
  5. Strategie de retrieval -- Commence par similarity simple, evolue vers MMR/Hybrid/Reranking
  6. Prompt template -- Concu le prompt avec instructions, contexte et question
  7. LLM -- Choisi le modele generateur (GPT-4o, Claude, Mistral, etc.)
  8. Evaluation -- Mis en place les metriques RAGAS pour mesurer la qualite
  9. Iteration -- Ameliore chaque composant en mesurant l'impact

Arbre de decision : quand utiliser le RAG ?

  • Besoin d'acceder a des donnees privees/internes ? --> RAG
  • Besoin de donnees a jour (apres la date de coupure du LLM) ? --> RAG
  • Besoin de citer les sources des reponses ? --> RAG
  • Besoin de reduire les hallucinations sur des faits precis ? --> RAG
  • Besoin d'un style/format de reponse specifique ? --> Fine-tuning (ou few-shot prompting)
  • Corpus tres petit (< 50 pages) et questions simples ? --> Long context stuffing (peut suffire)
  • Questions complexes sur un large corpus ? --> RAG avance (Advanced/Modular)
