Strategies de Retrieval - Retrouver l'Information Pertinente

Cours 4/4 -- Les techniques avancees de recherche : le coeur de l'optimisation RAG

Ce cours est le plus important de la serie RAG. La qualite du retrieval est le facteur numero 1 de la performance d'un systeme RAG. Un LLM, meme le meilleur, ne peut pas generer une bonne reponse si les chunks fournis ne sont pas pertinents. Toutes les strategies presentees ici visent a ameliorer la qualite, la diversite et la pertinence des documents recuperes.

1. Naive Retrieval - La ligne de base

Le Naive Retrieval (retrieval naif) est l'approche la plus simple : on transforme la question en embedding, on cherche les k vecteurs les plus proches dans le vector store par similarite cosinus, et on retourne les chunks correspondants. C'est la strategie par defaut de tout systeme RAG.
# Naive Retrieval : la base de reference from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings(model="text-embedding-3-small") vectorstore = Chroma( persist_directory="./chroma_db", embedding_function=embeddings ) # Retrieval naif : similarity search avec k=4 retriever = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 4} ) results = retriever.invoke("Quels sont les delais de livraison ?") for i, doc in enumerate(results): print(f"\n--- Resultat {i+1} ---") print(f"Source: {doc.metadata.get('source', 'N/A')}") print(doc.page_content[:120])

1.1 Les problemes du Naive Retrieval

Pourquoi le Naive Retrieval ne suffit pas ?

  • Redondance : Les resultats 1 et 3 ci-dessus parlent du meme sujet (delais standard). On gaspille 2 slots sur la meme information au lieu d'avoir des infos complementaires.
  • Mismatch question-document : L'utilisateur formule sa question differemment des documents. "Comment renvoyer un article ?" peut ne pas matcher "Politique de retour" car les embeddings ne capturent pas parfaitement la relation.
  • Pas de filtrage intelligent : La question "articles Python publies apres 2024" ne peut pas etre traitee car il n'y a pas d'extraction de filtres structures.
  • Precision limitee : Les chunks retournes contiennent souvent du contenu partiellement pertinent avec du bruit.
Le Naive Retrieval est votre baseline -- le point de reference. Chaque technique avancee que nous verrons dans ce cours doit etre evaluee par rapport a cette baseline. Si une technique avancee ne fait pas mieux que le Naive Retrieval sur vos donnees, elle n'est pas utile pour votre cas. Mesurez TOUJOURS.

2. MMR (Maximum Marginal Relevance) - Diversite

Le MMR (Maximum Marginal Relevance) est une strategie de retrieval qui equilibre la pertinence par rapport a la question et la diversite entre les resultats. Au lieu de retourner les k resultats les plus similaires a la question (qui peuvent etre redondants entre eux), MMR selectionne iterativement des resultats qui sont a la fois pertinents ET differents des resultats deja selectionnes.

2.1 La formule MMR

MMR selectionne le prochain document Di qui maximise :

MMR(Di) = lambda * Sim(Di, Q) - (1 - lambda) * max(Sim(Di, Dj))

  • Sim(Di, Q) : similarite entre le document candidat et la question (pertinence)
  • max(Sim(Di, Dj)) : similarite maximale avec les documents deja selectionnes (redondance)
  • lambda (lambda_mult) : controle l'equilibre entre pertinence et diversite

lambda_mult = 1.0 : pure similarite (comme le naive retrieval)

lambda_mult = 0.0 : pure diversite (maximise les differences)

lambda_mult = 0.5 a 0.7 : equilibre recommande pour le RAG

# MMR avec LangChain # fetch_k = combien de candidats initiaux a considerer # k = combien de resultats finaux a retourner # lambda_mult = equilibre pertinence/diversite retriever_mmr = vectorstore.as_retriever( search_type="mmr", search_kwargs={ "k": 4, # 4 resultats finaux "fetch_k": 20, # Considerer les 20 plus proches "lambda_mult": 0.7 # 70% pertinence, 30% diversite } ) # Comparaison : Similarity vs MMR question = "Quels sont les delais de livraison ?" print("=== Similarity Search ===") sim_results = vectorstore.similarity_search(question, k=4) for i, doc in enumerate(sim_results): print(f" {i+1}. {doc.page_content[:80]}...") print("\n=== MMR ===") mmr_results = retriever_mmr.invoke(question) for i, doc in enumerate(mmr_results): print(f" {i+1}. {doc.page_content[:80]}...")

MMR est la premiere optimisation a ajouter a votre systeme RAG. C'est simple, sans cout supplementaire, et l'amelioration est immediate. Mes recommandations : fetch_k=20 (ou 4x votre k), lambda_mult=0.7 pour commencer, puis ajustez selon vos resultats.

3. Multi-Query Retrieval - Reformulation de la question

Le Multi-Query Retrieval utilise un LLM pour generer plusieurs reformulations de la question originale. Chaque reformulation est ensuite utilisee pour une recherche independante, et les resultats sont fusionnes (union) et dedupliques. Cela augmente considerablement le recall en capturant differentes "facettes" de la question.
# Multi-Query Retrieval avec LangChain from langchain.retrievers.multi_query import MultiQueryRetriever from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3) # Creer le multi-query retriever multi_retriever = MultiQueryRetriever.from_llm( retriever=vectorstore.as_retriever(search_kwargs={"k": 4}), llm=llm ) # Activer le logging pour voir les requetes generees import logging logging.getLogger("langchain.retrievers.multi_query").setLevel(logging.INFO) # Recherche question = "Comment fonctionne le remboursement ?" results = multi_retriever.invoke(question) print(f"\nNombre de chunks uniques recuperes : {len(results)}") for i, doc in enumerate(results): print(f" {i+1}. [{doc.metadata.get('source', 'N/A')}] {doc.page_content[:80]}...")

Le Multi-Query ajoute un appel LLM supplementaire (pour generer les variantes de la question). Cela augmente la latence (typiquement +500ms a 1s) et le cout. Pour les applications temps reel, evaluez si le gain en recall justifie la latence supplementaire.

4. HyDE (Hypothetical Document Embeddings)

HyDE (Hypothetical Document Embeddings) est une technique ingenieuse : au lieu de chercher des documents similaires a la question, on demande au LLM de generer une reponse hypothetique, puis on cherche des documents similaires a cette reponse. L'intuition : une reponse hypothetique ressemble plus aux documents reels qu'une question.
# HyDE avec LangChain from langchain.chains import HypotheticalDocumentEmbedder, LLMChain from langchain.prompts import PromptTemplate from langchain_openai import ChatOpenAI, OpenAIEmbeddings llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.5) base_embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Prompt pour generer le document hypothetique prompt = PromptTemplate( input_variables=["question"], template="""Ecris un court paragraphe qui pourrait etre la reponse a cette question dans un document d'entreprise : Question : {question} Reponse hypothetique (1 paragraphe) :""" ) # Creer l'embedder HyDE hyde_embeddings = HypotheticalDocumentEmbedder.from_llm( llm=llm, base_embeddings=base_embeddings, prompt_key="question" ) # Utiliser HyDE comme embeddings dans le retriever retriever_hyde = vectorstore.as_retriever( search_kwargs={"k": 4} ) # La recherche utilise maintenant l'embedding du doc hypothetique # au lieu de l'embedding de la question directement question = "Quelle est la politique de teletravail ?" # Generer le document hypothetique hypothetical_vec = hyde_embeddings.embed_query(question) print(f"Vecteur HyDE genere : {len(hypothetical_vec)} dimensions") # Chercher avec ce vecteur results = vectorstore.similarity_search_by_vector(hypothetical_vec, k=4) for i, doc in enumerate(results): print(f"\n{i+1}. {doc.page_content[:100]}...")
Vecteur HyDE genere : 1536 dimensions 1. La politique de teletravail de l'entreprise permet aux collaborateurs de travailler a distance jus... 2. Les conditions du teletravail : l'employe doit avoir au moins 6 mois d'anciennete et disposer d'u... 3. Equipement teletravail : l'entreprise fournit un ordinateur portable, un ecran supplementaire et ... 4. Les jours de teletravail doivent etre declares a l'avance via le portail RH et valides par le man...
HyDE est particulierement efficace quand il y a un "gap semantique" entre la facon dont les utilisateurs posent des questions et la facon dont l'information est ecrite dans les documents. Par exemple, une question courte et informelle vs des documents formels et techniques. Cependant, HyDE ajoute un appel LLM (latence + cout). A utiliser quand le naive retrieval montre des problemes de recall.

5. Self-Query Retrieval - Extraction de filtres

Le Self-Query Retrieval utilise un LLM pour analyser la question et extraire automatiquement des filtres structures (metadonnees) en plus de la requete semantique. Cela permet de traiter des questions comme "les articles sur Python publies apres 2023" en separant la partie semantique ("articles sur Python") de la partie filtre ("publies apres 2023").
# Self-Query Retrieval avec LangChain from langchain.retrievers.self_query.base import SelfQueryRetriever from langchain.chains.query_constructor.base import AttributeInfo from langchain_openai import ChatOpenAI # Decrire les metadonnees disponibles dans votre vector store metadata_field_info = [ AttributeInfo( name="source", description="Le nom du fichier source du document", type="string" ), AttributeInfo( name="category", description="La categorie du document : 'RH', 'technique', 'commercial', 'juridique'", type="string" ), AttributeInfo( name="year", description="L'annee de publication du document", type="integer" ), AttributeInfo( name="department", description="Le departement concerne : 'IT', 'RH', 'Finance', 'Marketing'", type="string" ), ] llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Creer le Self-Query Retriever self_query_retriever = SelfQueryRetriever.from_llm( llm=llm, vectorstore=vectorstore, document_contents="Documentation interne de l'entreprise couvrant RH, technique, commercial et juridique", metadata_field_info=metadata_field_info, verbose=True ) # Exemple 1 : Question avec filtre implicite sur la categorie results = self_query_retriever.invoke( "Politique de conges dans le departement RH" ) print(f"Resultats : {len(results)}") # Exemple 2 : Question avec filtre sur l'annee results = self_query_retriever.invoke( "Documents techniques publies apres 2024" ) print(f"Resultats : {len(results)}") # Exemple 3 : Question avec filtres multiples results = self_query_retriever.invoke( "Procedures de securite IT mises a jour en 2025" ) print(f"Resultats : {len(results)}")
Query: "Politique de conges" | Filter: department = "RH" Resultats : 4 Query: "Documents techniques" | Filter: category = "technique" AND year > 2024 Resultats : 3 Query: "Procedures de securite" | Filter: department = "IT" AND year = 2025 Resultats : 2
Le LLM a automatiquement extrait les filtres structures de chaque question en langage naturel. "Documents techniques publies apres 2024" a ete decompose en une requete semantique ("Documents techniques") + un filtre structure (category="technique" AND year>2024). Cela combine la puissance de la recherche semantique avec la precision du filtrage structure.

6. Parent Document Retrieval

Le Parent Document Retrieval utilise le chunking hierarchique (cours 2) au moment du retrieval : on indexe des petits chunks (enfants) pour une recherche precise, mais on retourne les grands chunks (parents) au LLM pour un contexte riche. C'est le meilleur des deux mondes : precision de recherche + richesse de contexte.
# Parent Document Retriever avec LangChain from langchain.retrievers import ParentDocumentRetriever from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.storage import InMemoryStore from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings # Petits chunks pour la recherche (precise) child_splitter = RecursiveCharacterTextSplitter(chunk_size=200, chunk_overlap=20) # Grands chunks pour le contexte (riche) parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=100) # Stores vectorstore = Chroma( collection_name="child_chunks", embedding_function=OpenAIEmbeddings(model="text-embedding-3-small") ) docstore = InMemoryStore() # Stocke les parents # Le retriever parent-enfant retriever = ParentDocumentRetriever( vectorstore=vectorstore, docstore=docstore, child_splitter=child_splitter, parent_splitter=parent_splitter, ) # Indexer les documents retriever.add_documents(documents) # Recherche : cherche dans les enfants, retourne les parents results = retriever.invoke("Comment configurer le Wi-Fi ?") print(f"Parents retournes : {len(results)}") for i, doc in enumerate(results): print(f"\n--- Parent {i+1} ({len(doc.page_content)} caracteres) ---") print(doc.page_content[:200], "...")

Le Parent Document Retrieval est l'une des techniques les plus efficaces du Advanced RAG. La recherche se fait sur des petits chunks de 200 caracteres (tres precis), mais le LLM recoit des blocs de 1000 caracteres (contexte riche). En pratique, cette technique ameliore de 15 a 25% la qualite des reponses sur les documents longs.

7. Contextual Compression

La Contextual Compression post-traite les chunks recuperes pour ne garder que les parties reellement pertinentes par rapport a la question. Au lieu de donner tout le chunk (500 caracteres) au LLM, on extrait uniquement les phrases pertinentes (100-200 caracteres), ce qui reduit le bruit et le cout LLM.
# Contextual Compression avec LangChain from langchain.retrievers import ContextualCompressionRetriever from langchain.retrievers.document_compressors import LLMChainExtractor from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Le compresseur : extrait les parties pertinentes compressor = LLMChainExtractor.from_llm(llm) # Le retriever de base (naive ou MMR) base_retriever = vectorstore.as_retriever(search_kwargs={"k": 6}) # Combiner : retrieval + compression compression_retriever = ContextualCompressionRetriever( base_compressor=compressor, base_retriever=base_retriever ) # Recherche avec compression question = "Quel est le delai de livraison express ?" compressed_results = compression_retriever.invoke(question) print(f"Chunks apres compression : {len(compressed_results)}") for i, doc in enumerate(compressed_results): print(f"\n{i+1}. ({len(doc.page_content)} car.) {doc.page_content}")

Sur les 6 chunks recuperes initialement, 3 ont ete juges non pertinents et supprimes. Les 3 restants ont ete comprimes pour ne garder que les phrases repondant directement a la question. Le contexte est passe de ~3000 caracteres a ~277 caracteres -- beaucoup plus precis et moins cher a traiter par le LLM generateur.
La compression contextuelle ajoute un appel LLM pour CHAQUE chunk recupere. Pour k=6, cela represente 6 appels LLM supplementaires, ce qui augmente significativement la latence et le cout. A utiliser uniquement quand la precision est critique et que le budget le permet. Alternative plus legere : les filtres par score minimum (score_threshold).

8. Ensemble/Hybrid Retrieval - BM25 + Vecteurs

Le Hybrid Retrieval combine la recherche lexicale (BM25, basee sur les mots-cles) et la recherche semantique (vecteurs, basee sur le sens). Cette combinaison est souvent superieure a chaque methode seule car elles sont complementaires : BM25 excelle pour les termes exacts (noms, acronymes, numeros), tandis que les vecteurs excellent pour comprendre le sens global.

Pourquoi l'hybrid search bat les deux approches seules ?

Type de requete BM25 (lexical) Vecteurs (semantique) Hybrid (les deux)
"erreur 404 nginx" Excellent (termes exacts) Moyen (confondrait avec d'autres erreurs) Excellent
"le site ne marche pas" Mauvais (pas de mots-cles techniques) Bon (comprend le sens) Bon
"article 42 du reglement" Excellent (numero exact) Moyen (ignore l'importance du numero) Excellent
"comment reduire les couts" Moyen (cherche "couts" litteralement) Excellent (comprend economiser, optimiser) Excellent
# Hybrid Retrieval avec EnsembleRetriever (BM25 + Vecteurs) from langchain.retrievers import EnsembleRetriever from langchain_community.retrievers import BM25Retriever from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings # Retriever 1 : BM25 (recherche lexicale) bm25_retriever = BM25Retriever.from_documents(chunks) bm25_retriever.k = 4 # Retriever 2 : Vector store (recherche semantique) embeddings = OpenAIEmbeddings(model="text-embedding-3-small") vectorstore = Chroma.from_documents(chunks, embeddings) vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 4}) # Combiner avec EnsembleRetriever (Reciprocal Rank Fusion) ensemble_retriever = EnsembleRetriever( retrievers=[bm25_retriever, vector_retriever], weights=[0.4, 0.6] # 40% BM25, 60% semantique ) # Test 1 : Question avec termes techniques (BM25 aide) results = ensemble_retriever.invoke("erreur 503 serveur nginx") print(f"Hybrid - termes techniques : {len(results)} resultats") for i, doc in enumerate(results[:3]): print(f" {i+1}. {doc.page_content[:80]}...") # Test 2 : Question en langage naturel (vecteurs aident) results = ensemble_retriever.invoke("le site web ne repond plus depuis ce matin") print(f"\nHybrid - langage naturel : {len(results)} resultats") for i, doc in enumerate(results[:3]): print(f" {i+1}. {doc.page_content[:80]}...")

Les poids recommandes sont 0.3-0.4 pour BM25 et 0.6-0.7 pour les vecteurs. Ajustez selon vos donnees : si votre corpus contient beaucoup de termes techniques, augmentez le poids BM25.

9. Reranking - Reordonner les resultats

Le Reranking est une etape post-retrieval ou un modele specialise (cross-encoder) re-evalue et reordonne les resultats du retrieval initial. Contrairement aux bi-encoders (embeddings), un cross-encoder prend en entree la PAIRE (question, document) et calcule un score de pertinence beaucoup plus precis, mais aussi plus couteux.

L'approche en deux etapes est la suivante : (1) le retrieval initial (rapide, embeddings) recupere les top-k candidats (ex: 20), puis (2) le reranker (lent mais precis, cross-encoder) reordonne ces candidats et ne garde que les top-n (ex: 4).

# Reranking avec Cohere Reranker from langchain.retrievers import ContextualCompressionRetriever from langchain_cohere import CohereRerank # Retrieval initial : top-20 candidats base_retriever = vectorstore.as_retriever( search_kwargs={"k": 20} # Recuperer plus de candidats ) # Reranker : reordonne et ne garde que les top-4 reranker = CohereRerank( model="rerank-v3.5", top_n=4 # Ne garder que les 4 meilleurs apres reranking ) # Pipeline : retrieval + reranking rerank_retriever = ContextualCompressionRetriever( base_compressor=reranker, base_retriever=base_retriever ) # Recherche avec reranking question = "Comment obtenir un remboursement pour un produit defectueux ?" results = rerank_retriever.invoke(question) print(f"Resultats apres reranking : {len(results)}") for i, doc in enumerate(results): score = doc.metadata.get("relevance_score", "N/A") print(f"\n{i+1}. Score: {score} | {doc.page_content[:100]}...")

Le reranking est souvent la technique qui apporte le gain le plus significatif en production (gain de 5 a 15% en precision). Le principe est simple : le retrieval initial est rapide mais approximatif (bi-encoder), le reranking est lent mais precis (cross-encoder). On combine les deux pour avoir rapidite ET precision. Cohere rerank-v3.5 est le meilleur reranker commercial, mais le cross-encoder open source ms-marco-MiniLM est excellent pour les budgets limites.

Decision : quelle strategie pour quel probleme ?

  • Les resultats sont redondants --> Ajouter MMR (lambda_mult=0.7)
  • Les bons docs ne sont pas trouves (recall faible) --> Ajouter Multi-Query ou Hybrid Search
  • La question est differente du vocabulaire des docs --> Essayer HyDE
  • Besoin de filtrer par metadonnees naturellement --> Ajouter Self-Query
  • Les chunks sont trop courts pour le contexte LLM --> Utiliser Parent Document Retrieval
  • Les resultats contiennent trop de bruit --> Ajouter Reranking ou Contextual Compression
  • Termes techniques/acronymes importants --> Ajouter BM25 (Hybrid Search)

Pipeline de production recommande (Advanced RAG)

Voici l'architecture recommandee pour un systeme RAG performant en production :

  1. Ingestion : RecursiveCharacterTextSplitter (500/50) + metadata enrichment + deduplication
  2. Embeddings : text-embedding-3-large (production) ou embed-v4 (multilingue)
  3. Vector Store : Qdrant ou Pinecone (production), Chroma (dev)
  4. Pre-retrieval : Multi-Query pour les questions complexes
  5. Retrieval : Hybrid Search (BM25 + vecteurs) avec MMR + metadata filtering
  6. Post-retrieval : Reranking (Cohere rerank-v3.5 ou cross-encoder)
  7. Generation : GPT-4o ou Claude avec prompt structure
  8. Evaluation : RAGAS (faithfulness, answer_relevancy, context_precision, context_recall)

Ce pipeline combine 3 techniques complementaires : Hybrid Search (recall), MMR (diversite), et Reranking (precision). En production, ce pipeline offre typiquement 30 a 50% d'amelioration par rapport au Naive Retrieval. N'ajoutez pas toutes les techniques d'un coup -- commencez par Hybrid Search + MMR, mesurez l'amelioration, puis ajoutez le Reranking si necessaire.
