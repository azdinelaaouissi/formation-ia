Chapitre 7 : RAG - Techniques Avancees

HyDE, CRAG, Self-RAG, Agentic RAG, RAG Fusion et architectures de production

1. Au-dela du RAG classique

Le RAG classique (naive RAG) suit un pipeline lineaire simple : question -> retrieval -> generation. Les techniques avancees introduisent des boucles de retroaction, des transformations de requetes, des agents decisionnels et des mecanismes d'auto-correction pour depasser les limites du paradigme lineaire.

Le RAG classique souffre de plusieurs limitations fondamentales que les techniques avancees cherchent a resoudre :

Limites du RAG classique

Limitation Description Technique avancee
Ecart semantique La question et les documents n'utilisent pas les memes termes HyDE, Step-Back Prompting
Documents non pertinents Le retriever ramene du bruit qui pollue la generation CRAG, Self-RAG
Raisonnement limité Questions complexes necessitant plusieurs etapes Agentic RAG, RAG Fusion
Donnees multimodales Images, tableaux, graphiques ignores Multi-modal RAG
Relations complexes Liens entre entites difficiles a capturer par embeddings Knowledge Graphs + RAG
Pensez au RAG classique comme une recherche Google basique : vous tapez des mots-cles et vous obtenez des resultats. Les techniques avancees, c'est comme avoir un assistant de recherche intelligent qui reformule votre question, verifie la qualite des sources, et synthetise les informations de maniere critique. C'est un changement de paradigme.

2. HyDE - Hypothetical Document Embeddings

HyDE (Hypothetical Document Embeddings) est une technique qui comble l'ecart semantique entre la question de l'utilisateur et les documents en demandant au LLM de generer d'abord un document hypothetique qui repondrait a la question, puis utilise ce document comme requete de recherche.

2.1 Le probleme de l'ecart semantique

Probleme du RAG classique : la question "Quels sont les effets secondaires de l'aspirine ?" est semantiquement differente d'un document medical qui dit "L'acide acetylsalicylique peut provoquer des irritations gastriques, des saignements..." Les embeddings de la question et du document peuvent etre eloignes dans l'espace vectoriel.

from langchain_openai import ChatOpenAI, OpenAIEmbeddings from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Etape 1 : Prompt pour generer le document hypothetique hyde_prompt = ChatPromptTemplate.from_template(""" Ecris un court passage de document technique (3-5 phrases) qui repondrait parfaitement a la question suivante. N'invente pas de chiffres precis, reste factuel et general. Question : {question} Passage :""") # Etape 2 : Chaine HyDE hyde_chain = hyde_prompt | llm | StrOutputParser() # Etape 3 : Utiliser le document hypothetique comme requete def hyde_retrieval(question: str, retriever, k: int = 4): """Retrieval augmente par HyDE.""" # Generer le document hypothetique hypothetical_doc = hyde_chain.invoke({"question": question}) print(f"[HyDE] Doc hypothetique : {hypothetical_doc[:100]}...") # Utiliser le doc hypothetique comme requete de recherche results = retriever.invoke(hypothetical_doc) return results[:k] # Exemple d'utilisation docs = hyde_retrieval("Quels sont les effets secondaires de l'aspirine ?", retriever)
HyDE ameliore typiquement le recall de 10-25% sur des requetes complexes ou techniques. C'est particulierement efficace pour les domaines specialises (medical, juridique, scientifique) ou l'ecart de vocabulaire entre la question et les documents est important.
HyDE ajoute un appel LLM supplementaire avant la recherche, ce qui augmente la latence d'environ 500-1500ms et le cout. De plus, si le LLM genere un document hypothetique incorrect, la recherche peut partir dans la mauvaise direction.

3. CRAG - Corrective RAG

CRAG (Corrective Retrieval Augmented Generation) ajoute un mecanisme d'auto-correction apres le retrieval. Un evaluateur de pertinence note chaque document recupere et decide s'il faut les garder, les affiner ou effectuer une nouvelle recherche (par exemple sur le web).

3.1 Le flux CRAG en trois etapes

Etape 1 - Retrieval : Le retriever classique recupere k documents depuis la base vectorielle.
Etape 2 - Evaluation : Un evaluateur de pertinence (LLM ou modele leger) classe chaque document comme "Correct", "Ambigu" ou "Incorrect".
Etape 3 - Action corrective :
- Si tous les documents sont "Correct" : generer directement la reponse.
- Si certains sont "Ambigu" : extraire les passages pertinents et completer.
- Si tous sont "Incorrect" : lancer une recherche web complementaire.
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from enum import Enum class RelevanceGrade(Enum): CORRECT = "correct" AMBIGUOUS = "ambiguous" INCORRECT = "incorrect" llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Evaluateur de pertinence des documents relevance_prompt = ChatPromptTemplate.from_template(""" Evalue si le document suivant est pertinent pour repondre a la question. Question : {question} Document : {document} Reponds par un seul mot : CORRECT, AMBIGUOUS ou INCORRECT """) relevance_chain = relevance_prompt | llm | StrOutputParser() def crag_pipeline(question: str, retriever, web_search_tool=None): """Pipeline CRAG complet avec correction automatique.""" # Etape 1 : Retrieval classique docs = retriever.invoke(question) # Etape 2 : Evaluer chaque document graded_docs = [] for doc in docs: grade = relevance_chain.invoke({ "question": question, "document": doc.page_content, }).strip().upper() graded_docs.append((doc, grade)) # Etape 3 : Decider l'action corrective correct_docs = [d for d, g in graded_docs if g == "CORRECT"] ambiguous_docs = [d for d, g in graded_docs if g == "AMBIGUOUS"] if len(correct_docs) >= 2: # Suffisamment de documents pertinents context_docs = correct_docs print("[CRAG] Action : utiliser les documents corrects") elif correct_docs or ambiguous_docs: # Extraire les passages pertinents + completer context_docs = correct_docs + ambiguous_docs print("[CRAG] Action : extraire passages + completer") else: # Tous incorrects : recherche web de secours print("[CRAG] Action : recherche web complementaire") if web_search_tool: web_results = web_search_tool.invoke(question) context_docs = web_results else: context_docs = docs # fallback return context_docs
CRAG est une avancee majeure car il introduit la notion de "doute intelligent". Au lieu de faire aveuglément confiance au retriever, le systeme se pose la question : "Ces documents sont-ils vraiment utiles pour repondre a cette question ?" C'est exactement ce qu'un bon chercheur fait instinctivement.

4. Self-RAG : le RAG auto-reflexif

Self-RAG va plus loin que CRAG en ajoutant des tokens speciaux de reflexion que le LLM genere pour decider : (1) s'il a besoin de retrieval, (2) si les documents recuperes sont pertinents, (3) si la reponse generee est fidelement soutenue par le contexte.

4.1 Les quatre tokens de reflexion

Token de reflexion Question posee Valeurs possibles
[Retrieve] Faut-il chercher des documents ? yes / no / continue
[IsRelevant] Ce document est-il pertinent ? relevant / irrelevant
[IsSupported] La reponse est-elle soutenue ? fully / partially / no_support
[IsUseful] La reponse est-elle utile ? 5 / 4 / 3 / 2 / 1
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import JsonOutputParser llm = ChatOpenAI(model="gpt-4o", temperature=0) # Etape 1 : Decide si le retrieval est necessaire need_retrieval_prompt = ChatPromptTemplate.from_template(""" Analyse la question suivante et decide si une recherche documentaire est necessaire pour y repondre correctement. Question : {question} Reponds en JSON : {{"need_retrieval": true/false, "reason": "explication breve"}} """) # Etape 2 : Evalue la pertinence du contexte support_check_prompt = ChatPromptTemplate.from_template(""" Evalue si la reponse generee est soutenue par le contexte fourni. Contexte : {context} Reponse : {answer} Reponds en JSON : {{ "support_level": "fully|partially|no_support", "unsupported_claims": ["liste des affirmations non soutenues"], "confidence": 0.0 }} """) class SelfRAG: """Implementation simplifiee de Self-RAG.""" def __init__(self, retriever, llm): self.retriever = retriever self.llm = llm self.retrieval_checker = need_retrieval_prompt | llm | JsonOutputParser() self.support_checker = support_check_prompt | llm | JsonOutputParser() def invoke(self, question: str) -> dict: # Token [Retrieve] : faut-il chercher ? retrieval_decision = self.retrieval_checker.invoke({"question": question}) if not retrieval_decision["need_retrieval"]: # Repondre directement sans retrieval answer = self.llm.invoke(question).content return {"answer": answer, "used_retrieval": False} # Retrieval + generation docs = self.retriever.invoke(question) context = "\n".join(d.page_content for d in docs) answer = self.llm.invoke(f"Contexte: {context}\nQuestion: {question}").content # Token [IsSupported] : la reponse est-elle soutenue ? support = self.support_checker.invoke({ "context": context, "answer": answer }) if support["support_level"] == "no_support": # Regenerer avec un prompt plus strict answer = self.llm.invoke( f"Reponds UNIQUEMENT avec les infos du contexte.\n{context}\n{question}" ).content return {"answer": answer, "support": support, "used_retrieval": True}
Self-RAG reduit les hallucinations de 30-50% par rapport au RAG classique en introduisant une boucle d'auto-verification. Le cout est un nombre accru d'appels au LLM (2 a 4 appels au lieu de 1).

5. Agentic RAG avec LangGraph

L'Agentic RAG transforme le pipeline RAG en un agent autonome capable de planifier, executer et corriger ses actions. En utilisant LangGraph, on modelise le RAG comme un graphe d'etats avec des noeuds decisionnels et des boucles de retroaction conditionnelles.

5.2 Implementation avec LangGraph

from langgraph.graph import StateGraph, END from typing import TypedDict, List, Annotated from langchain_core.documents import Document import operator # Definir l'etat du graphe class AgenticRAGState(TypedDict): question: str documents: List[Document] generation: str web_search_needed: bool retries: int # Noeud 1 : Retrieve def retrieve(state: AgenticRAGState) -> AgenticRAGState: """Recupere les documents depuis la base vectorielle.""" docs = retriever.invoke(state["question"]) return {"documents": docs} # Noeud 2 : Grade documents def grade_documents(state: AgenticRAGState) -> AgenticRAGState: """Evalue la pertinence de chaque document recupere.""" question = state["question"] docs = state["documents"] relevant_docs = [] for doc in docs: grade = relevance_chain.invoke({ "question": question, "document": doc.page_content, }) if "CORRECT" in grade.upper(): relevant_docs.append(doc) web_needed = len(relevant_docs) < 2 return {"documents": relevant_docs, "web_search_needed": web_needed} # Noeud 3 : Web search def web_search(state: AgenticRAGState) -> AgenticRAGState: """Recherche web complementaire si les docs locaux sont insuffisants.""" web_results = web_tool.invoke(state["question"]) web_docs = [Document(page_content=r) for r in web_results] return {"documents": state["documents"] + web_docs} # Noeud 4 : Generate def generate(state: AgenticRAGState) -> AgenticRAGState: """Genere la reponse a partir des documents valides.""" context = "\n\n".join(d.page_content for d in state["documents"]) answer = llm.invoke( f"Contexte:\n{context}\n\nQuestion: {state['question']}" ).content return {"generation": answer} # Construire le graphe LangGraph workflow = StateGraph(AgenticRAGState) workflow.add_node("retrieve", retrieve) workflow.add_node("grade_documents", grade_documents) workflow.add_node("web_search", web_search) workflow.add_node("generate", generate) # Definir les transitions workflow.set_entry_point("retrieve") workflow.add_edge("retrieve", "grade_documents") workflow.add_conditional_edges( "grade_documents", lambda s: "web_search" if s["web_search_needed"] else "generate", {"web_search": "web_search", "generate": "generate"}, ) workflow.add_edge("web_search", "generate") workflow.add_edge("generate", END) # Compiler et executer app = workflow.compile() result = app.invoke({"question": "Qu'est-ce que le CRAG ?", "retries": 0})
LangGraph est un framework de LangChain qui permet de modeliser des workflows complexes sous forme de graphes d'etats. Chaque noeud est une fonction, et les aretes peuvent etre conditionnelles. C'est ideal pour l'Agentic RAG car on peut facilement ajouter des boucles de retroaction et des embranchements decisionnels.

6. RAG Fusion

RAG Fusion combine la generation de requetes multiples avec le Reciprocal Rank Fusion (RRF) pour ameliorer la qualite du retrieval. Le LLM genere plusieurs variantes de la question, chacune est utilisee pour chercher, et les resultats sont fusionnes par RRF.
RRF(document) = SUM_q( 1 / (k + rang_du_doc_dans_la_recherche_q) )

k = constante de lissage (typiquement k = 60)
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from collections import defaultdict llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) # Etape 1 : Generer des variantes de la question query_gen_prompt = ChatPromptTemplate.from_template(""" Genere 4 reformulations differentes de la question suivante. Chaque reformulation doit aborder la question sous un angle different ou utiliser un vocabulaire different. Question originale : {question} Reponds avec une question par ligne, sans numerotation. """) query_gen_chain = query_gen_prompt | llm | StrOutputParser() # Etape 2 : Reciprocal Rank Fusion def reciprocal_rank_fusion(results_lists: list, k: int = 60) -> list: """Fusionne plusieurs listes de resultats par RRF. Args: results_lists: Liste de listes de documents (une par requete) k: Constante de lissage (defaut=60, standard) Returns: Liste de documents tries par score RRF decroissant """ rrf_scores = defaultdict(float) doc_map = {} for results in results_lists: for rank, doc in enumerate(results, 1): doc_id = doc.page_content[:100] # cle de deduplication rrf_scores[doc_id] += 1.0 / (k + rank) doc_map[doc_id] = doc # Trier par score RRF decroissant sorted_ids = sorted(rrf_scores, key=rrf_scores.get, reverse=True) return [doc_map[doc_id] for doc_id in sorted_ids] # Pipeline RAG Fusion complet def rag_fusion(question: str, retriever, top_k: int = 5) -> list: """Pipeline RAG Fusion complet.""" # Generer les variantes variants_text = query_gen_chain.invoke({"question": question}) queries = [question] + [q.strip() for q in variants_text.split("\n") if q.strip()] print(f"[RAG Fusion] {len(queries)} requetes generees") # Rechercher pour chaque variante all_results = [] for q in queries: results = retriever.invoke(q) all_results.append(results) # Fusionner par RRF fused = reciprocal_rank_fusion(all_results) return fused[:top_k]
RAG Fusion ameliore la diversite des documents retrouves de 20-40% par rapport a une seule requete. En generant des formulations differentes, on couvre un espace semantique plus large et on reduit le risque de manquer un document pertinent.

7. Step-Back Prompting

Le Step-Back Prompting consiste a generer une question plus generale et abstraite a partir de la question specifique de l'utilisateur, puis a utiliser les deux questions (specifique + abstraite) pour la recherche. Cela aide a recuperer des documents de contexte plus large.
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Prompt Step-Back : generer une question plus abstraite stepback_prompt = ChatPromptTemplate.from_template(""" A partir de la question specifique suivante, genere une question plus generale et abstraite qui aiderait a comprendre le contexte plus large. Exemple : - Specifique : "Quel est le taux de cholesterol normal pour un homme de 40 ans ?" - Step-back : "Quels sont les facteurs de risque cardiovasculaire et les normes biologiques ?" Question specifique : {question} Question step-back :""") stepback_chain = stepback_prompt | llm | StrOutputParser() def stepback_retrieval(question: str, retriever) -> list: """Retrieval avec Step-Back Prompting.""" # Generer la question abstraite stepback_question = stepback_chain.invoke({"question": question}) print(f"[Step-Back] Question abstraite : {stepback_question}") # Chercher avec les deux questions docs_specific = retriever.invoke(question) docs_abstract = retriever.invoke(stepback_question) # Combiner et dedupliquer seen = set() combined = [] for doc in docs_specific + docs_abstract: key = doc.page_content[:100] if key not in seen: seen.add(key) combined.append(doc) return combined # Exemple docs = stepback_retrieval( "Quelle est la complexite temporelle de l'algorithme de Dijkstra ?", retriever, )
[Step-Back] Question abstraite : Quels sont les principaux algorithmes de plus court chemin en theorie des graphes et leurs caracteristiques ?
Le Step-Back Prompting est particulierement efficace pour les questions tres specifiques ou techniques. En prenant du recul, on recupere des documents de contexte qui aident le LLM a mieux comprendre le domaine avant de repondre a la question precise.

8. Multi-modal RAG

Le Multi-modal RAG etend le paradigme RAG aux donnees non textuelles : images, tableaux, graphiques, diagrammes. Les documents contenant des figures, des schemas ou des photos peuvent etre indexes et recuperes grace a des modeles multi-modaux comme GPT-4V ou des embeddings multi-modaux (CLIP).

8.1 Strategies d'indexation multi-modale

Trois approches pour le Multi-modal RAG


Approche Methode Avantages Inconvenients
Description textuelle LLM multi-modal decrit l'image en texte, puis indexe le texte Simple, compatible avec tous les retrievers Perte d'information visuelle
Embeddings multi-modaux CLIP ou similaire pour embedder images et texte dans le meme espace Recherche cross-modale native Qualite des embeddings variable
Stockage brut + LLM vision Stocker les images brutes, les passer au LLM multi-modal au moment de la generation Pas de perte d'information Cout et latence eleves

9. Knowledge Graphs + RAG

Le GraphRAG combine les graphes de connaissances (Knowledge Graphs) avec le RAG vectoriel pour capturer les relations structurelles entre entites que les embeddings seuls peinent a representer. Un graphe de connaissances stocke des triplets (sujet, relation, objet) qui enrichissent le contexte.

9.1 Pourquoi combiner graphes et vecteurs ?

Embeddings vectoriels : Excellents pour la similarite semantique ("ce texte parle de la meme chose que cette requete"), mais ne capturent pas explicitement les relations entre concepts.
Graphes de connaissances : Capturent les relations explicites ("A est auteur de B", "C est un type de D"), mais ne gerent pas la recherche semantique floue.
Combinaison : Le meilleur des deux mondes. On cherche par similarite dans les vecteurs ET par traversee de graphe pour les relations.
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import JsonOutputParser llm = ChatOpenAI(model="gpt-4o", temperature=0) # Etape 1 : Extraire les triplets d'un texte kg_extraction_prompt = ChatPromptTemplate.from_template(""" Extrait les relations entre entites du texte suivant sous forme de triplets. Texte : {text} Reponds en JSON : [ {{"subject": "...", "relation": "...", "object": "..."}}, ... ] Exemples de relations : "est_auteur_de", "est_type_de", "fait_partie_de", "a_ete_cree_en", "utilise", "est_composant_de" """) kg_chain = kg_extraction_prompt | llm | JsonOutputParser() # Extraire les triplets d'un document text = """Le RAG a ete propose par Lewis et al. en 2020 chez Meta AI Research. Il combine FAISS pour la recherche vectorielle et BART pour la generation. LangChain est un framework Python qui facilite l'implementation du RAG.""" triplets = kg_chain.invoke({"text": text}) print(triplets)
[ {"subject": "RAG", "relation": "a_ete_propose_par", "object": "Lewis et al."}, {"subject": "RAG", "relation": "a_ete_cree_en", "object": "2020"}, {"subject": "RAG", "relation": "utilise", "object": "FAISS"}, {"subject": "RAG", "relation": "utilise", "object": "BART"}, {"subject": "LangChain", "relation": "facilite_implementation_de", "object": "RAG"} ]
# Etape 2 : Stocker dans un graphe (NetworkX pour la simplicite) import networkx as nx graph = nx.DiGraph() for triplet in triplets: graph.add_edge( triplet["subject"], triplet["object"], relation=triplet["relation"], ) # Etape 3 : Retrieval hybride (vecteurs + graphe) def graph_augmented_retrieval(question: str, retriever, graph: nx.DiGraph) -> str: """Combine retrieval vectoriel et navigation dans le graphe.""" # Retrieval vectoriel classique vector_docs = retriever.invoke(question) # Extraire les entites mentionnees dans les documents entities_in_docs = set() for doc in vector_docs: for node in graph.nodes(): if node.lower() in doc.page_content.lower(): entities_in_docs.add(node) # Naviguer dans le graphe pour trouver les relations graph_context = [] for entity in entities_in_docs: for _, target, data in graph.edges(entity, data=True): graph_context.append(f"{entity} {data['relation']} {target}") for source, _, data in graph.in_edges(entity, data=True): graph_context.append(f"{source} {data['relation']} {entity}") # Combiner le contexte vectoriel et le contexte graphe full_context = "\n".join(d.page_content for d in vector_docs) full_context += "\n\nRelations connues :\n" + "\n".join(graph_context) return full_context

10. Fine-tuning des embeddings pour RAG

Le fine-tuning des embeddings consiste a adapter un modele d'embedding pre-entraine a votre domaine specifique. Les embeddings generiques (comme text-embedding-3) fonctionnent bien en general, mais un fine-tuning sur votre corpus peut ameliorer le recall du retriever de 5-15% sur des domaines specialises.
from sentence_transformers import SentenceTransformer, InputExample, losses from torch.utils.data import DataLoader # Charger un modele d'embedding de base model = SentenceTransformer("all-MiniLM-L6-v2") # Preparer les donnees d'entrainement : paires (question, passage pertinent) train_examples = [ InputExample( texts=["Qu'est-ce que le RAG ?", "Le RAG combine recherche documentaire et generation par LLM"], label=1.0, # score de similarite (1.0 = tres pertinent) ), InputExample( texts=["Qu'est-ce que le RAG ?", "La cuisine italienne est reputee pour ses pates"], label=0.0, # score de similarite (0.0 = non pertinent) ), # ... des centaines d'exemples supplementaires ... ] # Configurer l'entrainement train_dataloader = DataLoader(train_examples, shuffle=True, batch_size=16) train_loss = losses.CosineSimilarityLoss(model) # Fine-tuner le modele model.fit( train_objectives=[(train_dataloader, train_loss)], epochs=3, warmup_steps=100, output_path="./models/rag_embeddings_finetuned", ) # Utiliser le modele fine-tune dans LangChain from langchain_huggingface import HuggingFaceEmbeddings custom_embeddings = HuggingFaceEmbeddings( model_name="./models/rag_embeddings_finetuned" )
Le fine-tuning est particulierement efficace pour les domaines avec un vocabulaire specifique (medical, juridique, technique). Les embeddings generiques ne comprennent pas toujours que "ACE inhibiteur" et "inhibiteur de l'enzyme de conversion" sont synonymes. Apres fine-tuning, le modele apprend ces equivalences.
Le fine-tuning necessite au minimum 500-1000 paires d'entrainement de qualite. Il faut aussi un jeu de validation separe pour eviter le sur-apprentissage. Si vous avez moins de 500 exemples, les embeddings generiques seront probablement meilleurs qu'un modele fine-tune sur trop peu de donnees.

11. Bonnes pratiques de production

Passer un systeme RAG du prototype a la production necessite d'adresser trois defis majeurs : le caching, le monitoring et le scaling.

11.1 Caching intelligent

Le caching dans un systeme RAG consiste a stocker les resultats de requetes frequentes (embeddings, documents recuperes, reponses) pour eviter les appels redondants au modele d'embedding et au LLM, reduisant ainsi la latence et les couts.
import hashlib import json from functools import lru_cache from typing import Optional, Dict class RAGCache: """Cache multi-niveaux pour systeme RAG en production.""" def __init__(self, redis_client=None, ttl_seconds: int = 3600): self.redis = redis_client self.ttl = ttl_seconds self.local_cache: Dict[str, dict] = {} def _hash_query(self, query: str) -> str: """Genere un hash deterministe de la requete.""" return hashlib.sha256(query.strip().lower().encode()).hexdigest() def get(self, query: str) -> Optional[dict]: """Cherche dans le cache (local d'abord, puis Redis).""" key = self._hash_query(query) # Niveau 1 : cache local (en memoire) if key in self.local_cache: return self.local_cache[key] # Niveau 2 : cache distribue (Redis) if self.redis: cached = self.redis.get(f"rag:{key}") if cached: result = json.loads(cached) self.local_cache[key] = result # propager au cache local return result return None def set(self, query: str, result: dict): """Stocke le resultat dans les deux niveaux de cache.""" key = self._hash_query(query) self.local_cache[key] = result if self.redis: self.redis.setex(f"rag:{key}", self.ttl, json.dumps(result)) # Utilisation dans le pipeline RAG cache = RAGCache(ttl_seconds=1800) # cache de 30 minutes def cached_rag_query(question: str, rag_chain) -> dict: """Pipeline RAG avec caching transparent.""" cached = cache.get(question) if cached: print("[Cache] Hit !") return cached print("[Cache] Miss - execution du pipeline") result = rag_chain.invoke({"input": question}) cache.set(question, result) return result

11.2 Monitoring et observabilite

11.3 Scaling et architecture de production

Architecture de production recommandee

Base vectorielle : Utiliser une solution managee (Pinecone, Weaviate Cloud, Qdrant Cloud) ou auto-hebergee (Milvus, Qdrant) avec replication pour la disponibilite.
Cache : Redis ou Memcached pour le caching des requetes frequentes. Caching semantique possible avec des seuils de similarite.
Load Balancer : Distribuer les requetes entre plusieurs instances du pipeline RAG. Le retrieval et la generation peuvent etre scales independamment.
File d'attente : RabbitMQ ou Kafka pour gerer les pics de charge. Les requetes non urgentes peuvent etre traitees en mode asynchrone.
Monitoring : Prometheus + Grafana pour les metriques techniques. LangSmith ou Langfuse pour le tracing des chaines LangChain.
CI/CD de l'evaluation : A chaque mise a jour du corpus ou du pipeline, le benchmark d'evaluation tourne automatiquement et bloque le deploiement si les metriques descendent sous les seuils definis.

12. Recapitulatif des techniques avancees

Technique Probleme resolu Complexite Gain typique
HyDE Ecart semantique question/document Faible +10-25% recall
CRAG Documents non pertinents recuperes Moyenne +15-20% precision
Self-RAG Hallucinations, auto-correction Elevee -30-50% hallucinations
Agentic RAG Questions complexes multi-etapes Elevee +20-35% correctness
RAG Fusion Couverture insuffisante d'une requete unique Faible +20-40% diversite
Step-Back Questions trop specifiques Faible +10-20% contexte
Multi-modal Images, graphiques non indexes Moyenne Couverture +100%
GraphRAG Relations entre entites Elevee +15-30% sur Q relationnelles
Fine-tuning embed. Embeddings generiques inadaptes Moyenne +5-15% recall domaine
Ne tentez pas d'implementer toutes ces techniques en meme temps. Commencez par un RAG classique bien optimise (bon chunking, bons embeddings, bon prompt). Puis identifiez votre probleme principal (hallucinations ? recall faible ? questions complexes ?) et choisissez la technique appropriee. Mesurez l'impact avec les metriques du chapitre 6 avant de continuer. L'ingenierie RAG est un processus iteratif d'amelioration continue.
Chaque technique avancee ajoute de la complexite, de la latence et des couts. Un pipeline RAG trop complexe est difficile a debugger, a maintenir et a faire evoluer. La simplicite est une vertu en ingenierie : ne complexifiez que quand les metriques montrent un besoin reel.
