TD Pratique : Utilisation des APIs LLM

Maitriser les appels API, les parametres de generation et l'integration en production

1. Recapitulatif des APIs LLM

Une API LLM (Application Programming Interface) est un point d'acces HTTP qui permet d'envoyer du texte a un grand modele de langage et de recevoir une reponse generee. Au lieu d'installer le modele localement (ce qui necesite des GPUs couteux), vous envoyez une requete a un serveur distant qui execute le modele et vous retourne le resultat.

Fiche d'identite

ProprieteDetail
ProtocoleAPI REST (requetes HTTP POST avec JSON)
AuthentificationCle API (API key) a inclure dans chaque requete
EntreeTableau de messages (system, user, assistant) + parametres
SortieTexte genere, nombre de tokens utilises, metadata
FacturationPar token (generalement : X$ / 1 million de tokens)
FournisseursOpenAI (GPT), Anthropic (Claude), Google (Gemini), Mistral, etc.

Pourquoi utiliser une API plutot qu'un modele local ?

  • Pas d'infrastructure GPU : GPT-4 necessite ~1.8 To de VRAM. Via API, vous n'avez besoin que d'une connexion internet.
  • Toujours a jour : le fournisseur met a jour le modele sans action de votre part
  • Scalabilite instantanee : 1 requete ou 10 000 requetes/minute, l'API gere la charge
  • Pay-per-use : vous ne payez que ce que vous consommez (pas de cout fixe de serveur)

2. Principe de Fonctionnement

L'interaction avec une API LLM suit un cycle requete/reponse HTTP classique. Voici les etapes detaillees :

Etape 1 - Authentification : Vous obtenez une cle API (API key) depuis le tableau de bord du fournisseur. Cette cle identifie votre compte et autorise les appels. Elle doit rester secrete et ne jamais apparaitre dans le code source public.
Etape 2 - Construction de la requete : Vous creez un objet JSON avec le modele choisi, le tableau de messages (system/user/assistant) et les parametres de generation (temperature, max_tokens, etc.).
Etape 3 - Envoi HTTP POST : La requete est envoyee au endpoint de l'API (ex: https://api.openai.com/v1/chat/completions). Le header contient votre cle API.
Etape 4 - Traitement serveur : Le serveur recoit la requete, tokenise le texte, execute le modele, et genere la reponse token par token.
Etape 5 - Reponse JSON : Le serveur retourne un JSON contenant le texte genere, le nombre de tokens consommes (input + output), et des metadonnees (model, id, finish_reason).
from openai import OpenAI # Initialiser le client (la cle API est lue depuis la variable d'environnement OPENAI_API_KEY) client = OpenAI() # Appel API basique response = client.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": "Tu es un assistant concis."}, {"role": "user", "content": "Qu'est-ce qu'un token en NLP ?"} ], temperature=0.3, max_tokens=200 ) # Extraire la reponse texte = response.choices[0].message.content tokens_in = response.usage.prompt_tokens tokens_out = response.usage.completion_tokens cout_total = response.usage.total_tokens print(f"Reponse : {texte}") print(f"\n--- Statistiques ---") print(f"Tokens entree : {tokens_in}") print(f"Tokens sortie : {tokens_out}") print(f"Tokens total : {cout_total}") print(f"Cout estime : ${cout_total / 1_000_000 * 10:.6f}")
Reponse : Un token est l'unite de base qu'un modele de langage manipule. Ce n'est pas exactement un mot : c'est un fragment de texte issu de la tokenisation (souvent BPE). Par exemple, "bonjour" est 1 token, mais "anticonstitutionnellement" est decompose en 4-5 tokens. En moyenne, 1 token = 0.75 mot en francais et 1 token = 4 caracteres en anglais. --- Statistiques --- Tokens entree : 28 Tokens sortie : 87 Tokens total : 115 Cout estime : $0.001150
Remarquez que le francais consomme plus de tokens que l'anglais pour exprimer la meme idee (environ 30% de plus). C'est parce que les tokenizers des LLMs sont majoritairement entraines sur du texte anglais. Ce surcoute est a prendre en compte dans vos estimations budgetaires.

Comprendre les tokens

Un token est l'unite de texte minimale traitee par le modele. Ce n'est ni un mot, ni un caractere, mais un fragment intermediaire issu de l'algorithme BPE (Byte Pair Encoding). Le nombre de tokens determine le cout et la limite de la fenetre de contexte.
import tiktoken # Charger le tokenizer de GPT-4o enc = tiktoken.encoding_for_model("gpt-4o") textes = [ "Hello world", "Bonjour le monde", "anticonstitutionnellement", "L'intelligence artificielle transforme le monde.", "{'sentiment': 'positif', 'score': 0.95}", ] print(f"{'Texte':<50} | {'Tokens':>6} | {'Ratio c/t':>9}") print("-" * 75) for texte in textes: tokens = enc.encode(texte) ratio = len(texte) / len(tokens) print(f"{texte:<50} | {len(tokens):>6} | {ratio:>7.1f} c/t") print(f" Tokens detailles : {[enc.decode([t]) for t in tokens]}")

Points cles a retenir : (1) Le francais produit ~30% plus de tokens que l'anglais pour le meme sens. (2) Le JSON est tres couteux en tokens a cause de la ponctuation (accolades, guillemets, deux-points). (3) Les mots courants anglais ("Hello", "world") sont souvent 1 seul token. (4) Les mots francais longs sont decomposes en sous-tokens.

4. Les Parametres de l'API (en detail)

L'API LLM offre plusieurs parametres qui controlent le comportement de la generation. Chaque parametre influence la qualite, la creativite et le cout de la reponse. Voici les 5 parametres essentiels :

Parametre 1 : temperature - LE plus important

Role : Controle le degre d'aleatoire dans la generation. A chaque token, le modele calcule une distribution de probabilites sur tous les tokens possibles. La temperature modifie cette distribution avant l'echantillonnage.

Plage : 0.0 a 2.0 (defaut : 1.0)

Impact :

  • temperature = 0.0 : Le modele choisit TOUJOURS le token le plus probable. Reponses deterministes et reproductibles. Ideal pour les taches factuelles et le JSON.
  • temperature = 0.3 : Presque deterministe avec une legere variation. Bon pour les taches structurees qui tolerent un peu de creativite.
  • temperature = 0.7 : Bon equilibre creativite/coherence. Recommande pour la generation de texte general, les chatbots conversationnels.
  • temperature = 1.0 : Distribution originale du modele. Plus creatif mais risque d'incoherences. Bon pour le brainstorming.
  • temperature = 1.5+ : Tres aleatoire. Les reponses deviennent souvent incoherentes ou absurdes. A eviter en production.

Regle pratique : temperature = 0.0 pour les taches deterministes (classification, extraction, JSON). temperature = 0.7 pour les taches creatives (redaction, dialogue).

# Tester differentes temperatures sur le meme prompt prompt = "Ecris une metaphore originale pour decrire l'intelligence artificielle." print("=== Impact de la Temperature ===") print() for temp in [0.0, 0.3, 0.7, 1.0, 1.5]: print(f"--- temperature = {temp} ---") # Executer 3 fois pour voir la variabilite for run in range(3): response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=temp, max_tokens=60 ) texte = response.choices[0].message.content.strip() print(f" Run {run+1}: {texte[:80]}...") print()

Parametre 2 : max_tokens - Controle de la longueur

Role : Fixe le nombre maximum de tokens que le modele peut generer dans sa reponse. Le modele s'arrete des qu'il atteint cette limite, meme au milieu d'un mot ou d'une phrase.

Plage : 1 a la limite du modele (ex: 4096 pour GPT-4o par defaut)

Impact :

  • max_tokens = 50 : Reponses tres courtes. Risque de troncature en plein milieu d'une phrase.
  • max_tokens = 200 : Suffisant pour un paragraphe. Bon pour les classifications, extractions et reponses courtes.
  • max_tokens = 1000 : Un texte complet d'une page. Suffisant pour la plupart des taches.
  • max_tokens non defini : Le modele genere jusqu'a sa limite par defaut ou jusqu'au token de fin.

Regle pratique : Toujours definir max_tokens pour controler le cout et eviter les reponses trop longues. Estimez la longueur attendue et ajoutez 20% de marge.

# Demonstration de l'effet de max_tokens prompt = "Explique le fonctionnement d'un reseau de neurones en 5 points." for max_tok in [50, 200, 1000]: response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=0.0, max_tokens=max_tok ) texte = response.choices[0].message.content finish = response.choices[0].finish_reason tokens_out = response.usage.completion_tokens print(f"=== max_tokens = {max_tok} ===") print(f"Tokens generes : {tokens_out} | Raison d'arret : {finish}") print(f"Reponse : {texte[:150]}...") print()

Parametre 3 : top_p (Nucleus Sampling)

Role : A chaque etape de generation, le modele trie les tokens par probabilite decroissante et ne considere que le sous-ensemble minimal dont la somme des probabilites atteint le seuil top_p. Les tokens en dehors de ce "noyau" sont exclus.

Plage : 0.0 a 1.0 (defaut : 1.0)

Impact :

  • top_p = 0.1 : Seuls les tokens les plus probables (couvrant 10% de la masse de probabilites) sont consideres. Tres deterministe.
  • top_p = 0.5 : On considere les tokens couvrant 50% de la masse. Equilibre entre diversite et coherence.
  • top_p = 0.9 : On considere 90% de la masse. Plus de diversite, mais on exclut les tokens tres improbables ("queue" de distribution).
  • top_p = 1.0 : Aucun filtrage. Tous les tokens sont consideres (la temperature seule controle l'aleatoire).

Regle pratique : Modifier temperature OU top_p, mais pas les deux en meme temps. En production, la plupart des praticiens utilisent temperature et laissent top_p a 1.0.

# Comparaison de top_p avec temperature fixe a 1.0 prompt = "Complete cette phrase de maniere creative : 'Le futur de la programmation est'" print("=== Comparaison de top_p (temperature fixe a 1.0) ===\n") for tp in [0.1, 0.5, 0.9]: print(f"--- top_p = {tp} ---") for run in range(3): response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=1.0, top_p=tp, max_tokens=40 ) print(f" Run {run+1}: {response.choices[0].message.content.strip()[:75]}") print()

Parametre 4 : frequency_penalty et presence_penalty

Role : Ces deux parametres penalisent la repetition de tokens deja generes.

frequency_penalty (0.0 a 2.0, defaut 0.0) : Penalise un token proportionnellement au nombre de fois qu'il apparait deja dans la reponse. Plus un mot est repete, plus il est penalise.

presence_penalty (0.0 a 2.0, defaut 0.0) : Penalise un token des sa premiere apparition (presence binaire : present ou absent, peu importe le nombre d'occurrences).

Difference cle :

  • frequency_penalty : "le mot X est apparu 5 fois, je le penalise fortement" (penalite croissante)
  • presence_penalty : "le mot X est deja apparu, je le penalise un peu" (penalite fixe)
# Impact des penalties sur la diversite lexicale prompt = "Liste 10 synonymes du mot 'intelligent'." configs = [ ("Sans penalty", 0.0, 0.0), ("frequency_penalty=1.5", 1.5, 0.0), ("presence_penalty=1.5", 0.0, 1.5), ("Les deux a 1.0", 1.0, 1.0), ] for nom, freq_p, pres_p in configs: response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=0.7, frequency_penalty=freq_p, presence_penalty=pres_p, max_tokens=100 ) print(f"=== {nom} ===") print(response.choices[0].message.content.strip()[:120]) print()

Avec les penalties, le modele evite les repetitions et explore un vocabulaire plus large. Sans penalty, il peut repeter des structures syntaxiques similaires. Les penalties sont surtout utiles pour la generation longue (articles, histoires) ou le brainstorming.

Parametre 5 : stop - Sequences d'arret

Role : Definit une ou plusieurs chaines de caracteres qui, si generees par le modele, provoquent l'arret immediat de la generation. Le texte d'arret n'est PAS inclus dans la reponse.

Plage : Liste de 1 a 4 strings

Cas d'usage :

  • Empecher le modele de generer du texte apres un delimiteur (ex: "---")
  • Limiter la reponse a un seul element (arreter apres "\n")
  • Forcer le modele a s'arreter apres avoir genere un bloc specifique (ex: "```")
# Utiliser stop pour extraire uniquement la premiere ligne response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Donne 5 noms de villes francaises."}], temperature=0.0, stop=["\n\n"] # Arreter apres le premier paragraphe ) print("Avec stop=['\\n\\n'] :") print(response.choices[0].message.content) print(f"Finish reason: {response.choices[0].finish_reason}") print() # Arreter apres le 3eme element d'une liste numerotee response2 = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Donne 5 noms de villes francaises, une par ligne."}], temperature=0.0, stop=["4."] # Arreter avant le 4eme element ) print("Avec stop=['4.'] (seulement 3 villes) :") print(response2.choices[0].message.content) print(f"Finish reason: {response2.choices[0].finish_reason}")

Les stop sequences sont un outil puissant mais sous-utilise. Elles permettent de controler precisement ou le modele s'arrete, sans gaspiller de tokens. Par exemple, dans un chatbot, vous pouvez utiliser stop=["User:", "Human:"] pour empecher le modele de generer des faux messages utilisateur.

5. Les Metriques de Performance de l'API

Pour optimiser vos appels API en production, vous devez suivre 4 metriques cles :

Metrique 1 : Comptage de Tokens

Le nombre total de tokens (input + output) determine directement votre facture. Chaque appel API retourne les compteurs dans response.usage.

Formule : Tokens total = prompt_tokens + completion_tokens

Optimiser : Reduire la longueur du system prompt, limiter max_tokens, utiliser des modeles moins chers pour les taches simples.

Metrique 2 : Latence (temps de reponse)

Le temps entre l'envoi de la requete et la reception de la reponse complete. La latence depend principalement du nombre de tokens generes (output).

Typiquement : 50-100ms pour le premier token (TTFT), puis ~30-50 tokens/seconde pour le reste.

Optimiser : Utiliser le streaming pour afficher les premiers tokens immediatement, reduire max_tokens.

Metrique 3 : Cout par Requete

Le cout monetaire de chaque appel API. Varie selon le modele, le nombre de tokens input et le nombre de tokens output.

Formule : Cout = (prompt_tokens * prix_input / 1M) + (completion_tokens * prix_output / 1M)

Optimiser : Utiliser GPT-4o-mini pour les taches simples ($0.15/$0.60 par 1M) au lieu de GPT-4o ($2.50/$10.00 par 1M).

Metrique 4 : Taux de Succes

Le pourcentage de requetes qui retournent une reponse valide (pas d'erreur HTTP, JSON parsable, reponse pertinente).

Cible : >= 99.5% en production. En dessous, il faut ameliorer le retry logic et le prompt.

import time import json def benchmark_api(client, messages, model="gpt-4o", n_runs=5, prix_in=2.50, prix_out=10.00): """Benchmark complet d'un appel API : tokens, latence, cout.""" resultats = [] for i in range(n_runs): start = time.time() try: resp = client.chat.completions.create( model=model, messages=messages, temperature=0.0, max_tokens=200 ) latence = time.time() - start resultats.append({ "tokens_in": resp.usage.prompt_tokens, "tokens_out": resp.usage.completion_tokens, "latence": latence, "succes": True, "finish": resp.choices[0].finish_reason }) except Exception as e: resultats.append({"succes": False, "erreur": str(e)}) # Calculer les statistiques succes = [r for r in resultats if r["succes"]] taux_succes = len(succes) / len(resultats) * 100 if succes: avg_in = sum(r["tokens_in"] for r in succes) / len(succes) avg_out = sum(r["tokens_out"] for r in succes) / len(succes) avg_latence = sum(r["latence"] for r in succes) / len(succes) cout_moyen = (avg_in * prix_in + avg_out * prix_out) / 1_000_000 print("=== Benchmark API ===") print(f"Modele : {model}") print(f"Runs : {n_runs}") print(f"Taux de succes : {taux_succes:.1f}%") print(f"Tokens moyens : {avg_in:.0f} in + {avg_out:.0f} out = {avg_in+avg_out:.0f} total") print(f"Latence moy. : {avg_latence:.2f}s") print(f"Cout moyen : ${cout_moyen:.6f} par requete") print(f"Cout pour 10K : ${cout_moyen * 10000:.2f}") # Benchmark msgs = [ {"role": "system", "content": "Classifie le sentiment en JSON : {\"sentiment\": \"...\", \"score\": ...}"}, {"role": "user", "content": "Texte : 'Ce produit est fantastique, je le recommande !'"} ] benchmark_api(client, msgs)

6. Implementation Complete : API en Production

Voici une implementation robuste couvrant les 4 aspects essentiels d'une utilisation en production : setup securise, appel basique, streaming, et gestion des erreurs avec retry logic.

6.3 Streaming (reponse en temps reel)

# Streaming : recevoir la reponse token par token (pour l'affichage en temps reel) import sys def appel_streaming(messages, model="gpt-4o", temperature=0.7, max_tokens=300): """Appel API en streaming : affiche les tokens au fur et a mesure.""" stream = client.chat.completions.create( model=model, messages=messages, temperature=temperature, max_tokens=max_tokens, stream=True # Active le streaming ) texte_complet = "" print("Reponse (streaming) : ", end="") for chunk in stream: # Chaque chunk contient un ou plusieurs tokens if chunk.choices[0].delta.content is not None: token = chunk.choices[0].delta.content print(token, end="", flush=True) # Affichage immediat texte_complet += token print() # Nouvelle ligne a la fin return texte_complet # Utilisation : l'utilisateur voit les mots apparaitre progressivement resultat = appel_streaming([ {"role": "user", "content": "Explique en 3 phrases pourquoi le streaming est utile pour les APIs LLM."} ])
Reponse (streaming) : Le streaming permet d'afficher les premiers mots de la reponse en quelques millisecondes au lieu d'attendre parfois plusieurs secondes pour la reponse complete. Cela ameliore considerablement l'experience utilisateur car l'attente percue est reduite : l'utilisateur commence a lire immediatement. En production, c'est essentiel pour les chatbots et les interfaces conversationnelles ou la reactivite est primordiale.

6.4 Gestion des erreurs et retry logic

import time import random from openai import ( APIError, APIConnectionError, RateLimitError, APITimeoutError, ) def appel_robuste( messages, model="gpt-4o", max_retries=5, base_delay=1.0, temperature=0.0, max_tokens=500, ): """Appel API avec retry exponential backoff + jitter.""" for attempt in range(max_retries): try: response = client.chat.completions.create( model=model, messages=messages, temperature=temperature, max_tokens=max_tokens, ) return response # Succes ! except RateLimitError as e: # HTTP 429 : trop de requetes. Attendre et reessayer. delay = base_delay * (2 ** attempt) + random.uniform(0, 1) print(f"[429] Rate limit. Retry {attempt+1}/{max_retries} dans {delay:.1f}s") time.sleep(delay) except APITimeoutError as e: # Timeout : le serveur n'a pas repondu a temps delay = base_delay * (2 ** attempt) print(f"[Timeout] Retry {attempt+1}/{max_retries} dans {delay:.1f}s") time.sleep(delay) except APIConnectionError as e: # Erreur reseau : pas de connexion au serveur delay = base_delay * (2 ** attempt) print(f"[Connexion] Retry {attempt+1}/{max_retries} dans {delay:.1f}s") time.sleep(delay) except APIError as e: # Erreur serveur (500, 503) : reessayer if e.status_code in [500, 502, 503]: delay = base_delay * (2 ** attempt) print(f"[{e.status_code}] Erreur serveur. Retry {attempt+1}/{max_retries} dans {delay:.1f}s") time.sleep(delay) else: # Erreur client (400, 401, 403) : ne PAS reessayer print(f"[{e.status_code}] Erreur client non retriable : {e.message}") raise # Toutes les tentatives ont echoue raise Exception(f"Echec apres {max_retries} tentatives") # Utilisation response = appel_robuste( messages=[ {"role": "system", "content": "Sois concis."}, {"role": "user", "content": "Qu'est-ce que le backoff exponentiel ?"} ] ) print(response.choices[0].message.content)
Le backoff exponentiel est une strategie de retry ou le delai entre chaque tentative double a chaque echec : 1s, 2s, 4s, 8s, 16s... On ajoute souvent du "jitter" (delai aleatoire) pour eviter que tous les clients retentent au meme moment. Cela evite de surcharger un serveur deja en difficulte.
Les etudes montrent que 70-80% des erreurs transitoires (rate limit, timeout, erreur serveur) se resolvent en quelques secondes. Le retry avec backoff exponentiel est la strategie standard en production. Ne jamais reessayer les erreurs 400 (bad request) ou 401 (authentification) car elles indiquent un probleme dans votre code, pas un probleme temporaire du serveur.

8. Erreurs Courantes de Debutant

Erreur 1 : Cle API exposee dans le code source. Si votre cle API est dans un fichier Python commite sur GitHub, n'importe qui peut l'utiliser et vous facturer des milliers d'euros. Toujours utiliser des variables d'environnement ou un fichier .env (ajoute au .gitignore). Si votre cle est compromise, revoquez-la immediatement dans le dashboard du fournisseur.
Erreur 2 : Pas de gestion d'erreurs. Les APIs LLM echouent regulierement : rate limits (429), timeouts, erreurs serveur (500/503). Sans retry logic, votre application plante a la premiere erreur. Implementez toujours un backoff exponentiel avec au moins 3-5 retries. 70-80% des erreurs transitoires se resolvent en quelques secondes.
Erreur 3 : Ne pas definir max_tokens. Sans limite, le modele peut generer des milliers de tokens pour une question simple, explosant votre facture. Pour une classification qui devrait retourner un mot, le modele peut generer un paragraphe d'explication. Toujours fixer max_tokens a la longueur attendue + 20% de marge.
Erreur 4 : Utiliser GPT-4o pour tout. GPT-4o coute ~16x plus que GPT-4o-mini. Pour des taches simples (classification, extraction de mots-cles, reformulation), GPT-4o-mini suffit largement. Reservez les modeles premium aux taches complexes (raisonnement multi-etapes, generation longue, analyse fine). Cette seule optimisation peut diviser votre facture par 10.
Erreur 5 : Ignorer le finish_reason. Quand finish_reason = "length", la reponse a ete tronquee. Si vous parsez cette reponse tronquee comme du JSON, json.loads() va echouer. Toujours verifier que finish_reason == "stop" avant de traiter la reponse. Si la reponse est tronquee, augmentez max_tokens ou reduisez le prompt.
Erreur 6 : Modifier temperature ET top_p en meme temps. Ces deux parametres controlent l'aleatoire de maniere differente. Les modifier simultanement produit des effets imprevisibles et difficiles a debugger. Choisissez un parametre (generalement temperature) et laissez l'autre a sa valeur par defaut. OpenAI recommande explicitement cette approche.

Checklist de Deploiement en Production

  1. Securite : Cle API dans les variables d'environnement, jamais dans le code
  2. Resilience : Retry avec backoff exponentiel (3-5 tentatives)
  3. Cout : Modele adapte a la tache, max_tokens defini, monitoring des depenses
  4. Performance : Streaming pour les interfaces utilisateur, batch pour le traitement de masse
  5. Qualite : Verification du finish_reason, validation du JSON, tests automatises
  6. Monitoring : Logger les tokens, la latence, le taux d'erreur et le cout par requete
  7. Fallback : Si le modele principal echoue apres tous les retries, basculer sur un modele de secours
  8. Rate limiting : Implementer votre propre rate limiter cote client pour rester sous les limites de l'API
Ce TD vous a appris a maitriser les APIs LLM de bout en bout : de l'appel basique a l'implementation robuste en production. Les deux competences cles a retenir sont : (1) choisir les bons parametres (temperature, max_tokens) pour chaque tache, et (2) implementer une gestion d'erreurs robuste (retry, fallback, monitoring).
