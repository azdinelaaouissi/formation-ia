TD Pratique : Prompt Engineering

Techniques avancees de conception de prompts pour les grands modeles de langage

1. Recapitulatif du Prompt Engineering

Le Prompt Engineering est l'art et la science de concevoir des instructions textuelles (prompts) pour guider le comportement d'un grand modele de langage (LLM). Un prompt bien concu peut transformer une reponse mediocre en une reponse experte, sans modifier le modele lui-meme.

Fiche d'identite

ProprieteDetail
ObjectifObtenir des reponses precises, coherentes et utiles d'un LLM
PrerequisAcces a un LLM (OpenAI, Anthropic, Google, etc.)
EntreeTexte structure : instructions, contexte, exemples, contraintes
SortieReponse du modele, formatee selon les instructions
DifficulteFacile a demarrer, difficile a maitriser
DomaineIA generative, NLP, applications LLM

Pourquoi le prompt engineering est essentiel

Un LLM ne "comprend" pas intuitivement ce que vous voulez. Il predit le token le plus probable en fonction du contexte. La qualite de votre prompt determine directement la qualite de la reponse :

  • Prompt vague : le modele "devine" votre intention et produit des reponses generiques
  • Prompt precis : le modele genere exactement ce dont vous avez besoin
  • Prompt structure : le modele respecte un format, un ton, un style specifiques
En 2025-2026, le prompt engineering s'est divise en deux branches : le prompting conversationnel (interaction quotidienne avec un chatbot) et le context engineering de production (conception systematique de prompts pour des applications). Ce TD couvre les deux aspects, avec un accent sur les techniques reproductibles.

Les trois composantes d'un prompt

  • System prompt : definit le role, le comportement et les contraintes du modele
  • User prompt : contient la question ou la tache a accomplir
  • Assistant prompt (optionnel) : amorce la reponse pour guider le format de sortie

2. Principe de Fonctionnement

Quand vous envoyez un prompt a un LLM, le modele recoit une sequence de messages structures en roles. Chaque role a une fonction specifique dans la conversation :

Etape 1 - Message System : Vous definissez le comportement global du modele. Ce message persiste durant toute la conversation et agit comme une "constitution" que le modele doit respecter. Exemple : "Tu es un expert en droit fiscal francais."
Etape 2 - Message User : L'utilisateur pose sa question ou decrit sa tache. C'est le coeur de l'interaction. Le modele utilise le system prompt + ce message pour generer sa reponse.
Etape 3 - Message Assistant : La reponse generee par le modele. On peut aussi pre-remplir le debut de la reponse pour forcer un format (ex: "{ " pour forcer du JSON).
Etape 4 - Historique : L'alternance user/assistant constitue l'historique de conversation. Le modele a acces a tout l'historique pour generer chaque nouvelle reponse (dans la limite de sa fenetre de contexte).
Le system prompt est le levier le plus puissant dont vous disposez. Il definit le "cadre mental" du modele avant meme que l'utilisateur ne parle. En production, un bon system prompt peut contenir 200-500 mots avec des regles precises, des exemples de format attendu, et des contraintes de securite.
from openai import OpenAI # Initialiser le client OpenAI client = OpenAI(api_key="sk-votre-cle-api") # Structure de base d'un prompt avec les 3 roles messages = [ { "role": "system", "content": "Tu es un expert en analyse de sentiments. Reponds uniquement en JSON valide avec les champs : sentiment, score, explication." }, { "role": "user", "content": "Analyse le sentiment : 'Ce restaurant etait une horreur totale, la pire experience de ma vie.'" } ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) print(response.choices[0].message.content)
{ "sentiment": "tres negatif", "score": -0.95, "explication": "L'utilisation de 'horreur totale' et 'pire experience de ma vie' exprime un mecontentement extreme et categorique." }

3. Les Techniques de Prompting

Il existe plusieurs strategies fondamentales pour structurer vos prompts. Chacune a ses forces et ses cas d'usage. Le choix de la technique depend de la complexite de la tache et de la qualite attendue.

Technique 1 : Zero-Shot Prompting

Zero-Shot signifie qu'on donne au modele une instruction sans aucun exemple. Le modele doit se debrouiller uniquement avec ses connaissances pre-entrainees. C'est la technique la plus simple et souvent le point de depart.
# Zero-Shot : aucun exemple fourni messages = [ {"role": "system", "content": "Tu es un classificateur de sentiments."}, {"role": "user", "content": "Classifie le sentiment (positif, negatif, neutre) :\n\n'La livraison etait rapide mais le produit est decevant.'"} ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) print(response.choices[0].message.content)
Sentiment : negatif La phrase exprime une deception globale malgre un aspect positif (livraison rapide). Le mot "decevant" domine le sentiment general.
Le zero-shot fonctionne bien pour des taches simples et bien definies. Cependant, le format de sortie est imprevisible : parfois le modele ajoute des explications non demandees, parfois il utilise un format different.

Technique 2 : Few-Shot Prompting

Few-Shot signifie qu'on fournit quelques exemples (generalement 2 a 5) dans le prompt pour montrer au modele exactement ce qu'on attend. Le modele apprend le pattern a partir des exemples et l'applique a la nouvelle entree.
# Few-Shot : on fournit 3 exemples avant la question messages = [ {"role": "system", "content": "Tu es un classificateur de sentiments. Reponds avec exactement un mot : POSITIF, NEGATIF ou NEUTRE."}, # Exemple 1 {"role": "user", "content": "Classifie : 'J'adore ce film, les acteurs sont brillants !'"}, {"role": "assistant", "content": "POSITIF"}, # Exemple 2 {"role": "user", "content": "Classifie : 'Le service client est inexistant et les prix sont abusifs.'"}, {"role": "assistant", "content": "NEGATIF"}, # Exemple 3 {"role": "user", "content": "Classifie : 'Le colis est arrive comme prevu mardi.'"}, {"role": "assistant", "content": "NEUTRE"}, # Nouvelle entree a classifier {"role": "user", "content": "Classifie : 'La livraison etait rapide mais le produit est decevant.'"} ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) print(response.choices[0].message.content)
NEGATIF
Avec le few-shot, le modele respecte exactement le format demande : un seul mot, en majuscules. Les exemples lui montrent le pattern a suivre. C'est la technique qui offre le meilleur rapport effort/qualite pour la plupart des taches.

Technique 3 : Chain-of-Thought (CoT)

Chain-of-Thought (CoT) est une technique qui demande au modele de raisonner etape par etape avant de donner sa reponse finale. Elle ameliore considerablement les performances sur les taches de raisonnement complexe (mathematiques, logique, analyse multi-criteres).
# Chain-of-Thought : on demande au modele de raisonner etape par etape messages = [ {"role": "system", "content": """Tu es un expert en analyse de sentiments. Pour chaque texte, tu dois : 1. Identifier les mots et expressions cles (positifs et negatifs) 2. Evaluer le poids relatif de chaque element 3. Considerer le ton general et l'intention de l'auteur 4. Conclure avec le sentiment final et un score de -1.0 a +1.0 Montre ton raisonnement etape par etape."""}, {"role": "user", "content": "Analyse : 'La livraison etait rapide mais le produit est decevant. Le support a ete reactif neanmoins.'"} ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) print(response.choices[0].message.content)
**Etape 1 - Mots et expressions cles :** - Positifs : "rapide" (livraison), "reactif" (support) - Negatifs : "decevant" (produit) - Connecteurs : "mais" (opposition), "neanmoins" (concession) **Etape 2 - Poids relatif :** - "Livraison rapide" : +0.3 (aspect logistique, importance moyenne) - "Produit decevant" : -0.6 (concerne le produit lui-meme, importance haute) - "Support reactif" : +0.2 (aspect service, importance moyenne) **Etape 3 - Ton general :** L'auteur utilise "mais" pour opposer un point positif a un point negatif central. Le "neanmoins" final tente d'attenuer mais ne compense pas la deception sur le produit. **Etape 4 - Conclusion :** Sentiment : MITIGE TENDANCE NEGATIF Score : -0.15
Le Chain-of-Thought est particulierement efficace pour les taches complexes. Dans les benchmarks recents, le CoT ameliore la precision de 20 a 40% sur les problemes de raisonnement par rapport au zero-shot simple. Cependant, il consomme plus de tokens (et donc coute plus cher).

Technique 4 : Role Prompting

# Role Prompting : attribuer une expertise specifique au modele messages = [ {"role": "system", "content": """Tu es le Dr. Marie Laurent, linguiste computationnelle avec 15 ans d'experience en analyse de discours commercial. Tu as publie 3 articles sur la detection des sentiments ambigus dans les avis clients e-commerce. Tu analyses les textes avec la rigueur d'une chercheuse et la clarte d'une enseignante."""}, {"role": "user", "content": "Analysez ce texte : 'La livraison etait rapide mais le produit est decevant.'"} ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.3 ) print(response.choices[0].message.content)
En tant que specialiste de l'analyse de discours commercial, voici mon evaluation : Ce texte presente une structure concessivo-adversative classique (X mais Y). Dans mes recherches sur les avis e-commerce, ce pattern "positif MAIS negatif" aboutit dans 78% des cas a un sentiment global negatif, car le locuteur place l'element important APRES le "mais". - Element pre-adversatif : "livraison rapide" (satisfaction logistique) - Element post-adversatif : "produit decevant" (insatisfaction produit) Verdict : Sentiment NEGATIF (score -0.4). Le "mais" agit comme un pivoteur discursif qui annule l'element positif initial.

4. Les Parametres du Prompting (en detail)

Au-dela du choix de technique, plusieurs parametres structurels influencent la qualite des reponses. Voici les 4 parametres cles que vous pouvez ajuster dans la conception de vos prompts :

Parametre 1 : Nombre d'exemples en Few-Shot

Role : Determine combien d'exemples on fournit au modele pour lui montrer le pattern attendu. Plus il y a d'exemples, plus le modele comprend la tache, mais plus on consomme de tokens.

Valeurs typiques : 0 (zero-shot), 1, 3, 5

Impact :

  • 0 exemples (zero-shot) : rapide, economique, mais format imprevisible
  • 1 exemple (one-shot) : le modele comprend le format mais peut sur-generaliser
  • 3 exemples : excellent compromis qualite/cout pour la majorite des taches
  • 5+ exemples : rendements decroissants, utile uniquement pour des taches tres specifiques

Regle pratique : Commencer par 3 exemples diversifies (un positif, un negatif, un cas limite).

import time texte_test = "Le service etait correct mais sans plus." # Test avec 0, 1, 3 et 5 exemples exemples_base = [ ("J'adore ce produit, qualite exceptionnelle !", "POSITIF"), ("Horrible, jamais je ne recommanderai.", "NEGATIF"), ("Le colis est arrive mardi.", "NEUTRE"), ("Bonne qualite pour le prix, je suis satisfait.", "POSITIF"), ("Rien de special, ca fait le job.", "NEUTRE"), ] for n_examples in [0, 1, 3, 5]: messages = [{"role": "system", "content": "Classifie le sentiment : POSITIF, NEGATIF ou NEUTRE. Un seul mot."}] # Ajouter N exemples for i in range(n_examples): messages.append({"role": "user", "content": f"Classifie : '{exemples_base[i][0]}'"}) messages.append({"role": "assistant", "content": exemples_base[i][1]}) messages.append({"role": "user", "content": f"Classifie : '{texte_test}'"}) start = time.time() response = client.chat.completions.create(model="gpt-4o", messages=messages, temperature=0.0) duree = time.time() - start tokens_in = response.usage.prompt_tokens tokens_out = response.usage.completion_tokens resultat = response.choices[0].message.content.strip() print(f"{n_examples} ex. | Reponse: {resultat:<25} | Tokens: {tokens_in:>4} in, {tokens_out:>3} out | {duree:.2f}s")
0 ex. | Reponse: NEUTRE (le texte exprime un avis mitige, sans enthousiasme ni rejet) | Tokens: 42 in, 18 out | 0.89s 1 ex. | Reponse: NEUTRE | Tokens: 71 in, 1 out | 0.41s 3 ex. | Reponse: NEUTRE | Tokens: 128 in, 1 out | 0.38s 5 ex. | Reponse: NEUTRE | Tokens: 186 in, 1 out | 0.42s
Observez : avec 0 exemples, le modele ajoute une explication non demandee (18 tokens de sortie). Des qu'on fournit 1 exemple, il comprend le format et repond en 1 seul token. La precision est identique ici, mais le format est bien meilleur avec des exemples. En revanche, chaque exemple ajoute environ 30 tokens d'entree, ce qui augmente le cout.

Parametre 2 : Specificite des Instructions

Role : Determine le niveau de detail et de precision des instructions donnees au modele. Une instruction vague produit des reponses generiques ; une instruction precise produit des reponses ciblees.

Niveaux :

  • Vague : "Analyse ce texte" (le modele choisit quoi analyser et comment)
  • Precise : "Donne le sentiment (positif/negatif/neutre) de ce texte" (tache definie)
  • Structuree : instructions detaillees avec criteres, contraintes et format de sortie
# Comparaison : prompt vague vs precis vs structure texte = "Le nouveau MacBook Pro M4 offre des performances impressionnantes mais son prix de 2499 euros reste prohibitif pour beaucoup d'utilisateurs." # ---- Prompt VAGUE ---- prompt_vague = [{"role": "user", "content": f"Analyse ce texte : '{texte}'"}] # ---- Prompt PRECIS ---- prompt_precis = [ {"role": "system", "content": "Tu es un analyste de reviews tech. Donne le sentiment general, les points positifs et negatifs."}, {"role": "user", "content": f"Analyse cette review : '{texte}'"} ] # ---- Prompt STRUCTURE ---- prompt_structure = [ {"role": "system", "content": """Tu es un analyste de reviews tech specialise en produits Apple. Pour chaque review, produis une analyse structuree en JSON avec : - "sentiment" : "positif", "negatif" ou "mitige" - "score" : float entre -1.0 et 1.0 - "points_positifs" : liste de strings - "points_negatifs" : liste de strings - "public_cible" : qui beneficierait de ce produit - "verdict" : une phrase de synthese (max 20 mots) Reponds UNIQUEMENT avec le JSON, sans texte avant ou apres."""}, {"role": "user", "content": f"Analyse : '{texte}'"} ] for nom, prompt in [("VAGUE", prompt_vague), ("PRECIS", prompt_precis), ("STRUCTURE", prompt_structure)]: resp = client.chat.completions.create(model="gpt-4o", messages=prompt, temperature=0.0) print(f"=== {nom} ===\n{resp.choices[0].message.content}\n")
=== VAGUE === Ce texte parle du nouveau MacBook Pro M4. Il mentionne que les performances sont impressionnantes mais que le prix est eleve. C'est une review mitigee qui souligne un bon rapport performance mais un cout potentiellement trop eleve pour certains. === PRECIS === **Sentiment general :** Mitige **Points positifs :** - Performances impressionnantes du processeur M4 **Points negatifs :** - Prix prohibitif (2499 euros) - Accessibilite limitee pour beaucoup d'utilisateurs === STRUCTURE === { "sentiment": "mitige", "score": 0.15, "points_positifs": ["Performances impressionnantes du M4"], "points_negatifs": ["Prix prohibitif a 2499 euros", "Inaccessible pour beaucoup d'utilisateurs"], "public_cible": "Professionnels creatifs et developpeurs avec budget eleve", "verdict": "Excellentes performances mais prix qui limite fortement l'accessibilite au grand public." }
Le prompt structure produit une sortie parfaitement formatee, exploitable programmatiquement (JSON valide). Le prompt vague produit du texte libre impossible a parser automatiquement. En production, preferez toujours les prompts structures.

Parametre 3 : Specification du Format de Sortie

Role : Definit explicitement dans quel format le modele doit repondre. Sans specification, le modele choisit un format libre. Avec specification, on peut forcer du JSON, du markdown, du CSV, ou tout autre format structure.

Options courantes :

  • Texte libre : aucune contrainte (par defaut)
  • JSON : parfait pour l'integration dans du code (parsing automatique)
  • Markdown : ideal pour les documents et rapports lisibles
  • CSV/Tableau : bon pour les donnees tabulaires
# Forcer differents formats de sortie pour la meme tache texte = "Excellent rapport qualite-prix, design elegant, mais autonomie decevante." formats = { "texte_libre": "Analyse le sentiment de ce texte. Reponds en texte libre.", "json": """Analyse le sentiment. Reponds UNIQUEMENT en JSON valide : {"sentiment": "...", "score": ..., "aspects": [{"nom": "...", "note": ...}]}""", "markdown": """Analyse le sentiment. Reponds en markdown structure : ## Sentiment ## Aspects ## Score""", } for fmt, instruction in formats.items(): messages = [ {"role": "system", "content": instruction}, {"role": "user", "content": f"Texte : '{texte}'"} ] resp = client.chat.completions.create(model="gpt-4o", messages=messages, temperature=0.0) print(f"--- {fmt} ---\n{resp.choices[0].message.content}\n")
--- texte_libre --- Ce texte exprime un sentiment globalement positif avec une reserve. Le rapport qualite-prix et le design sont apprecies, mais l'autonomie est jugee decevante. Sentiment general : mitige positif. --- json --- {"sentiment": "mitige_positif", "score": 0.35, "aspects": [{"nom": "rapport_qualite_prix", "note": 0.9}, {"nom": "design", "note": 0.85}, {"nom": "autonomie", "note": -0.6}]} --- markdown --- ## Sentiment Mitige positif - L'auteur est globalement satisfait malgre un point negatif. ## Aspects - **Rapport qualite-prix** : Excellent (positif) - **Design** : Elegant (positif) - **Autonomie** : Decevante (negatif) ## Score Score global : +0.35 / 1.0
En production, le format JSON est le plus utilise car il permet un parsing automatique avec json.loads(). Astuce avancee : avec l'API OpenAI, vous pouvez utiliser response_format={"type": "json_object"} pour garantir que le modele produit du JSON valide. Cela ajoute une contrainte au niveau du decodage.

Parametre 4 : Attribution de Role (Persona)

Role : Attribuer une identite ou une expertise specifique au modele pour orienter le style, le vocabulaire et la profondeur de la reponse.

Niveaux :

  • Aucun role : le modele repond comme un assistant generique
  • Role expert : "Tu es un data scientist senior" (plus technique, plus precis)
  • Persona detaillee : nom, experience, publications, style (reponses tres specialisees)
# Impact du role sur la qualite de la reponse question = "Explique pourquoi le fine-tuning d'un LLM n'est pas toujours necessaire." roles = { "Aucun role": "Tu es un assistant utile.", "Expert ML": "Tu es un ingenieur ML senior specialise dans le deploiement de LLMs en production.", "Persona": """Tu es le Pr. Alain Dupont, directeur de recherche au CNRS, specialiste des architectures Transformer. Tu as co-redige le benchmark HELM. Tu vulgarises avec precision, en donnant des chiffres concrets et des references.""" } for nom, system in roles.items(): messages = [ {"role": "system", "content": system}, {"role": "user", "content": question} ] resp = client.chat.completions.create(model="gpt-4o", messages=messages, temperature=0.3, max_tokens=150) print(f"=== {nom} ===\n{resp.choices[0].message.content}\n")
=== Aucun role === Le fine-tuning n'est pas toujours necessaire car les LLMs modernes sont tres performants avec du prompt engineering. On peut obtenir de bons resultats en formulant bien ses questions et en fournissant des exemples dans le prompt. === Expert ML === En production, le fine-tuning est souvent un piege. Voici pourquoi : 1. Cout : fine-tuner GPT-4 coute ~$25/million de tokens d'entrainement 2. Le few-shot prompting couvre 80% des cas d'usage 3. Le fine-tuning fige le modele : vous perdez les mises a jour futures 4. RAG (Retrieval-Augmented Generation) est souvent une meilleure alternative === Persona === D'apres nos evaluations sur HELM, un prompt bien concu avec 3-5 exemples atteint 92% de la performance d'un modele fine-tune sur les taches de classification standard. Le fine-tuning se justifie uniquement quand : (1) votre tache est tres specifique a un domaine (medical, juridique), (2) vous avez 10 000+ exemples annotes de haute qualite, (3) la latence est critique.
Plus le role est detaille, plus la reponse est specifique, chiffree et actionnable. Le role "Persona" produit des affirmations precises avec des seuils concrets (92%, 10 000+ exemples). Le role generique reste superficiel.

5. Les Metriques d'Evaluation des Prompts

Comment savoir si votre prompt est bon ? Il faut le mesurer. Voici les 4 metriques essentielles pour evaluer la qualite d'un prompt :

Metrique 1 : Pertinence (Relevance)

La reponse repond-elle reellement a la question posee ? Mesure si le modele reste sur le sujet ou s'egare.

Comment mesurer : Evaluation humaine sur une echelle de 1 a 5, ou comparaison automatique avec une reponse de reference (cosine similarity sur embeddings).

Seuil acceptable : Score >= 4/5 sur un echantillon de 50+ exemples.

Metrique 2 : Precision Factuelle (Accuracy)

Les informations dans la reponse sont-elles correctes ? Mesure les hallucinations et erreurs factuelles.

Comment mesurer : Verification manuelle sur un set de test avec reponses attendues. Calculer le pourcentage de reponses correctes.

Seuil acceptable : >= 90% pour les taches critiques (medical, juridique), >= 80% pour les taches generales.

Metrique 3 : Conformite au Format (Format Compliance)

La reponse respecte-t-elle le format demande ? Mesure si le JSON est valide, si les champs attendus sont presents, etc.

Comment mesurer : Parsing automatique. Tenter json.loads() et verifier la presence de chaque champ requis.

Seuil acceptable : >= 98% de reponses parsables. En dessous, le prompt doit etre revu.

Metrique 4 : Coherence (Consistency)

Le modele donne-t-il la meme reponse a chaque execution ? Mesure la stabilite des resultats avec temperature > 0.

Comment mesurer : Executer le meme prompt 10 fois et calculer la variance des reponses.

Seuil acceptable : >= 90% de reponses identiques (a temperature 0) ou similaires (a temperature > 0).

import json def evaluer_prompt(client, messages, n_runs=10, reponse_attendue=None): """Evalue un prompt sur les 4 metriques cles.""" resultats = [] json_valides = 0 corrects = 0 for i in range(n_runs): resp = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) contenu = resp.choices[0].message.content.strip() resultats.append(contenu) # Test conformite JSON try: parsed = json.loads(contenu) json_valides += 1 # Test precision if reponse_attendue and parsed.get("sentiment") == reponse_attendue: corrects += 1 except json.JSONDecodeError: pass # Coherence : combien de reponses sont identiques ? unique = len(set(resultats)) coherence = (n_runs - unique + 1) / n_runs * 100 print(f"=== Evaluation sur {n_runs} executions ===") print(f"Conformite JSON : {json_valides}/{n_runs} ({json_valides/n_runs*100:.0f}%)") print(f"Precision : {corrects}/{n_runs} ({corrects/n_runs*100:.0f}%)") print(f"Coherence : {coherence:.0f}%") print(f"Reponses uniques : {unique}") # Tester notre prompt de sentiment messages_eval = [ {"role": "system", "content": 'Classifie le sentiment. Reponds en JSON : {"sentiment": "positif|negatif|neutre"}'}, {"role": "user", "content": "Texte : 'Ce restaurant est une pure merveille, on y retourne demain !'"} ] evaluer_prompt(client, messages_eval, n_runs=10, reponse_attendue="positif")
=== Evaluation sur 10 executions === Conformite JSON : 10/10 (100%) Precision : 10/10 (100%) Coherence : 100% Reponses uniques : 1
100% sur toutes les metriques ! C'est le resultat attendu avec temperature=0 sur une tache simple. En pratique, les problemes apparaissent sur les cas ambigus, les textes longs, ou quand on utilise temperature > 0. Testez toujours votre prompt sur au moins 50 exemples varies avant de le deployer en production.

6. Implementation Complete : Workflow de Prompt Engineering

Voici le processus complet pour concevoir un prompt de production, de l'analyse de la tache au deploiement :

import json import time # ============================================================= # ETAPE 1 : ANALYSE DE LA TACHE # ============================================================= # Tache : Extraire les entites nommees (NER) d'un texte en francais # Entites : PERSONNE, LIEU, ORGANISATION, DATE, MONTANT # Format de sortie : JSON avec liste d'entites # ============================================================= # ETAPE 2 : DESIGN DU PROMPT # ============================================================= system_prompt = """Tu es un systeme d'extraction d'entites nommees (NER) pour le francais. TACHE : Extraire toutes les entites du texte fourni. CATEGORIES : - PERSONNE : noms de personnes (prenom, nom, titre) - LIEU : villes, pays, adresses, lieux geographiques - ORGANISATION : entreprises, institutions, associations - DATE : dates absolues ou relatives - MONTANT : sommes d'argent, pourcentages FORMAT DE SORTIE (JSON strict) : { "entites": [ {"texte": "...", "categorie": "...", "position_debut": int} ], "nombre_total": int } REGLES : - Extraire TOUTES les entites, meme les implicites - Un meme mot peut etre plusieurs entites (rare) - Si aucune entite : {"entites": [], "nombre_total": 0} - Ne jamais ajouter de texte hors du JSON""" # Exemples few-shot few_shot = [ {"role": "user", "content": "Texte : 'Marie Dupont a rejoint Google France a Paris en janvier 2024.'"}, {"role": "assistant", "content": json.dumps({ "entites": [ {"texte": "Marie Dupont", "categorie": "PERSONNE", "position_debut": 0}, {"texte": "Google France", "categorie": "ORGANISATION", "position_debut": 22}, {"texte": "Paris", "categorie": "LIEU", "position_debut": 39}, {"texte": "janvier 2024", "categorie": "DATE", "position_debut": 48} ], "nombre_total": 4 }, ensure_ascii=False)}, {"role": "user", "content": "Texte : 'Il fait beau aujourd'hui.'"}, {"role": "assistant", "content": json.dumps({"entites": [], "nombre_total": 0}, ensure_ascii=False)} ] # ============================================================= # ETAPE 3 : TEST SUR PLUSIEURS EXEMPLES # ============================================================= textes_test = [ "Le PDG d'Apple, Tim Cook, a annonce un investissement de 2 milliards d'euros a Lyon pour 2025.", "L'equipe de Marseille a battu le PSG 3-1 au Stade Velodrome dimanche dernier.", "Le rapport de la BCE indique une inflation de 2.3% en zone euro en mars 2025.", ] print("=== Tests NER ===") for texte in textes_test: messages = [{"role": "system", "content": system_prompt}] + few_shot + [ {"role": "user", "content": f"Texte : '{texte}'"} ] resp = client.chat.completions.create(model="gpt-4o", messages=messages, temperature=0.0) resultat = json.loads(resp.choices[0].message.content) print(f"\nTexte: {texte[:60]}...") for ent in resultat["entites"]: print(f" [{ent['categorie']:<15}] {ent['texte']}") print(f" Total: {resultat['nombre_total']} entites")

7. Experimentation : 5 Strategies pour une Meme Tache

Comparons 5 strategies de prompting differentes sur la meme tache : analyser le sentiment d'avis clients. Nous mesurons la precision, le format et le cout en tokens pour chaque approche.

import json import time # Dataset de test : 5 avis avec sentiment attendu dataset = [ ("Produit exceptionnel, je recommande a 100% !", "positif"), ("Nul, casse au bout de 2 jours.", "negatif"), ("Correct pour le prix, rien d'extraordinaire.", "neutre"), ("Le gout est bon mais la texture est etrange.", "mitige"), ("Livraison impeccable, produit conforme a la description.", "positif"), ] # ---- Strategie 1 : Zero-Shot Brut ---- def strat_zero_shot(texte): return [{"role": "user", "content": f"Sentiment (positif/negatif/neutre/mitige) : '{texte}'"}] # ---- Strategie 2 : Zero-Shot + System ---- def strat_system(texte): return [ {"role": "system", "content": "Reponds avec UN seul mot parmi : positif, negatif, neutre, mitige."}, {"role": "user", "content": f"Sentiment : '{texte}'"} ] # ---- Strategie 3 : Few-Shot (3 exemples) ---- def strat_few_shot(texte): return [ {"role": "system", "content": "Classifie le sentiment. Reponds avec UN seul mot."}, {"role": "user", "content": "'Genial, j'adore !'"}, {"role": "assistant", "content": "positif"}, {"role": "user", "content": "'Decevant et cher.'"}, {"role": "assistant", "content": "negatif"}, {"role": "user", "content": "'Ca va, sans plus.'"}, {"role": "assistant", "content": "neutre"}, {"role": "user", "content": f"'{texte}'"} ] # ---- Strategie 4 : CoT ---- def strat_cot(texte): return [ {"role": "system", "content": """Analyse le sentiment etape par etape : 1. Liste les mots positifs et negatifs 2. Evalue le poids de chaque element 3. Conclus avec le sentiment : positif, negatif, neutre ou mitige Termine par EXACTEMENT : 'SENTIMENT: [label]'"""}, {"role": "user", "content": f"Texte : '{texte}'"} ] # ---- Strategie 5 : JSON Structure ---- def strat_json(texte): return [ {"role": "system", "content": 'Reponds UNIQUEMENT en JSON : {"sentiment": "positif|negatif|neutre|mitige", "confiance": 0.0-1.0}'}, {"role": "user", "content": f"'{texte}'"} ] # Executer toutes les strategies strategies = { "Zero-Shot Brut": strat_zero_shot, "Zero-Shot+System": strat_system, "Few-Shot (3ex)": strat_few_shot, "Chain-of-Thought": strat_cot, "JSON Structure": strat_json, } print(f"{'Strategie':<20} | {'Corrects':>8} | {'Tokens':>8} | {'Format OK':>10}") print("-" * 60) for nom, strat_fn in strategies.items(): corrects = 0 total_tokens = 0 format_ok = 0 for texte, attendu in dataset: msgs = strat_fn(texte) resp = client.chat.completions.create(model="gpt-4o", messages=msgs, temperature=0.0) contenu = resp.choices[0].message.content.strip().lower() total_tokens += resp.usage.total_tokens # Verifier si le label correct est present dans la reponse if attendu in contenu: corrects += 1 # Verifier le format (1 seul mot ou JSON valide) if len(contenu.split()) <= 3 or contenu.startswith("{"): format_ok += 1 print(f"{nom:<20} | {corrects}/5 | {total_tokens:>6} | {format_ok}/5")

8. Erreurs Courantes de Debutant

Erreur 1 : Prompt trop vague. "Analyse ce texte" ne dit pas QUOI analyser (sentiment ? entites ? grammaire ? style ?). Le modele devine et se trompe souvent. Toujours specifier la tache exacte, le format de sortie et les categories possibles.
Erreur 2 : Dire ce qu'il ne faut PAS faire au lieu de ce qu'il FAUT faire. "Ne reponds pas de maniere informelle" est moins efficace que "Reponds dans un style professionnel et academique". Les LLMs suivent mieux les instructions positives que les negations. La recherche montre que les instructions negatives sont ignorees dans 15-30% des cas.
Erreur 3 : Pas de format de sortie specifie. Sans contrainte de format, le modele choisit un format different a chaque appel (parfois du texte libre, parfois des listes, parfois du JSON). En production, cela casse votre pipeline de parsing. Toujours specifier le format exact attendu avec un exemple.
Erreur 4 : Exemples few-shot non representatifs. Si tous vos exemples sont positifs, le modele est biaise vers le positif. Il faut des exemples qui couvrent TOUS les cas possibles, y compris les cas limites et ambigus. Minimum : un exemple par categorie de sortie.
Erreur 5 : Prompt trop long (>3000 tokens). La recherche recente montre que la performance des LLMs se degrade au-dela de ~3000 tokens de prompt. Les informations au milieu d'un long prompt sont souvent ignorees (effet "lost in the middle"). Le sweet spot est entre 150 et 300 mots pour le prompt.
Erreur 6 : Ne pas tester le prompt sur des cas limites. Votre prompt fonctionne sur 10 exemples simples ? Tant mieux. Mais avez-vous teste avec : texte vide, texte tres long, texte dans une autre langue, texte avec du sarcasme, texte avec des fautes ? Les cas limites revelent les faiblesses du prompt. Testez sur au moins 50 exemples varies avant le deploiement.

Les 7 Regles d'Or du Prompt Engineering

  1. Commencer simple : Toujours tester le zero-shot avant de complexifier
  2. Etre specifique : Definir la tache, le format, les contraintes et les categories
  3. Montrer, ne pas expliquer : Un bon exemple vaut mieux qu'une longue description
  4. Instruire positivement : Dire quoi faire plutot que quoi eviter
  5. Tester systematiquement : 50+ exemples, incluant des cas limites
  6. Mesurer le cout : Compter les tokens et optimiser le ratio qualite/cout
  7. Iterer : Un bon prompt est le resultat de 5-10 iterations, pas d'un coup de genie
Le prompt engineering est une competence qui s'ameliore avec la pratique. Gardez un journal de vos prompts : notez ce qui fonctionne, ce qui echoue, et pourquoi.
