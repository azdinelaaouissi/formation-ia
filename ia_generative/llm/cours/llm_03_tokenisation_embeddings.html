Tokenisation et Embeddings - Comment les LLMs "Lisent"

Cours 3/3 - Le pont entre le langage humain et les mathematiques

1. Pourquoi tokeniser ?

Le probleme fondamental

Les ordinateurs ne comprennent pas le texte. Ils ne comprennent que des nombres. Pour qu'un LLM puisse "lire" la phrase "Bonjour, comment allez-vous ?", il faut la convertir en une sequence de nombres.

Ce processus de conversion se fait en plusieurs etapes :

Analogie : la traduction pour un extra-terrestre

Imaginez que vous devez expliquer un poeme francais a un extra-terrestre qui ne comprend que les nombres :

  1. Tokenisation : Decouper le poeme en morceaux digestibles (pas forcement des mots entiers)
  2. Numerotation : Donner un numero unique a chaque morceau (son ID dans le dictionnaire)
  3. Embedding : Convertir chaque numero en une "coordonnee GPS" dans un espace mathematique ou les morceaux similaires sont proches

2. Methodes de tokenisation

Il existe plusieurs facons de decouper du texte en tokens. Voici les principales :

Approche 1 : Par caractere (Character-level)

Chaque caractere est un token.

Exemple : "Chat" = ["C", "h", "a", "t"] = 4 tokens

  • Avantage : Vocabulaire tres petit (~256 caracteres). Pas de mots inconnus.
  • Inconvenient : Sequences tres longues. "Bonjour" = 7 tokens au lieu de 1-2. Le modele doit apprendre a reconstituer les mots, ce qui est inefficace.

Approche 2 : Par mot (Word-level)

Chaque mot est un token.

Exemple : "Le chat mange" = ["Le", "chat", "mange"] = 3 tokens

  • Avantage : Intuitif, sequences courtes.
  • Inconvenient : Vocabulaire enorme (>500K mots en francais). Mots rares ou nouveaux = inconnus. "anticonstitutionnellement" prend 1 seul slot, ce qui est du gachis.

Approche 3 : Par sous-mot (Subword) - la methode utilisee

C'est le compromis optimal et la methode utilisee par TOUS les LLMs modernes. Les mots frequents restent entiers, les mots rares sont decomposes en sous-parties connues.

Exemple : "unbelievable" = ["un", "believ", "able"] = 3 tokens

Exemple en francais : "anticonstitutionnellement" = ["anti", "constitu", "tion", "nelle", "ment"] = 5 tokens

Les algorithmes de tokenisation subword

Algorithme Utilise par Principe
BPE (Byte Pair Encoding) GPT-2, GPT-3, GPT-4, Llama Part des caracteres, fusionne iterativement les paires les plus frequentes. "l" + "e" = "le" si tres frequent.
WordPiece BERT, DistilBERT Similaire a BPE mais maximise la vraisemblance du corpus plutot que la frequence.
SentencePiece T5, Llama, Mistral Traite le texte comme un flux de bytes/caracteres sans pre-tokenisation par espaces. Ideal pour le multilingue.
Tiktoken (BPE) GPT-3.5, GPT-4, GPT-4o Implementation rapide en Rust de BPE par OpenAI. Le standard de facto pour compter les tokens.

Code : compter les tokens avec tiktoken

# Installation : pip install tiktoken import tiktoken # Charger le tokenizer de GPT-4 enc = tiktoken.encoding_for_model("gpt-4") # Tokeniser une phrase texte = "Bonjour, comment allez-vous ?" tokens = enc.encode(texte) print(f"Texte : '{texte}'") print(f"Nombre de tokens : {len(tokens)}") print(f"Token IDs : {tokens}") # Decoder chaque token pour voir le decoupage for token_id in tokens: print(f" ID {token_id:>6} -> '{enc.decode([token_id])}'") # Comparer : francais vs anglais fr = "Le chat mange la souris dans le jardin." en = "The cat eats the mouse in the garden." tokens_fr = enc.encode(fr) tokens_en = enc.encode(en) print(f"\nFrancais : {len(tokens_fr)} tokens") print(f"Anglais : {len(tokens_en)} tokens") print(f"Ratio FR/EN : {len(tokens_fr)/len(tokens_en):.2f}x")

Le francais consomme generalement 10 a 40% plus de tokens que l'anglais pour le meme contenu. C'est parce que les tokenizers sont principalement entraines sur des corpus anglophones. Cela signifie que les requetes en francais coutent un peu plus cher en API et prennent un peu plus de place dans la fenetre de contexte.
Tiktoken est l'outil officiel d'OpenAI pour compter les tokens. Pour les modeles Anthropic (Claude), le comptage est similaire mais utilise un tokenizer different. Pour estimer, la regle approximative est : 1 token = environ 4 caracteres en anglais, 2-3 caracteres en francais.

3. Le vocabulaire

Le vocabulaire d'un tokenizer est l'ensemble de tous les tokens connus, chacun associe a un identifiant numerique unique (Token ID). C'est le "dictionnaire" du modele.

Taille du vocabulaire par modele

ModeleTokenizerTaille du vocabulaire
GPT-2BPE50 257 tokens
GPT-3.5 / GPT-4Tiktoken (cl100k_base)100 256 tokens
GPT-4oTiktoken (o200k_base)200 019 tokens
BERTWordPiece30 522 tokens
Llama 2SentencePiece BPE32 000 tokens
Llama 3/4Tiktoken BPE128 256 tokens
MistralSentencePiece32 768 tokens
La tendance est a l'augmentation du vocabulaire. GPT-4o est passe a 200K tokens, ce qui permet une meilleure couverture du multilingue et reduit le nombre de tokens necessaires pour encoder le meme texte (donc reduit les couts et augmente la vitesse).

Tokens speciaux

Comment BPE construit le vocabulaire

Etape 1 : Commencer avec tous les caracteres individuels comme vocabulaire initial : {a, b, c, ..., z, A, B, ..., Z, 0, ..., 9, espace, ponctuation...}
Etape 2 : Compter toutes les paires de tokens adjacentes dans le corpus d'entrainement.
Etape 3 : Fusionner la paire la plus frequente en un nouveau token. Par exemple, si "t" + "h" est la paire la plus frequente, creer le token "th".
Etape 4 : Repeter les etapes 2-3 jusqu'a atteindre la taille de vocabulaire desiree (ex: 100 000 tokens).
# Exemple simplifie de BPE corpus = "le chat et le chien" # Etape 0 : decoupage en caracteres # ['l','e',' ','c','h','a','t',' ','e','t',' ','l','e',' ','c','h','i','e','n'] # Compter les paires : # ('l','e') : 2 fois <- la plus frequente ! # ('c','h') : 2 fois # ('e',' ') : 2 fois # ('h','a') : 1 fois # ... # Fusion 1 : ('l','e') -> 'le' # ['le',' ','c','h','a','t',' ','e','t',' ','le',' ','c','h','i','e','n'] # Fusion 2 : ('c','h') -> 'ch' # ['le',' ','ch','a','t',' ','e','t',' ','le',' ','ch','i','e','n'] # Et ainsi de suite... jusqu'a avoir 100K tokens print("Vocabulaire final : mots frequents en entier, mots rares en morceaux")

4. Embeddings : les mots comme vecteurs

Un embedding est une representation numerique d'un token (ou d'un texte) sous forme de vecteur dans un espace de haute dimension. Les tokens semantiquement proches ont des vecteurs proches dans cet espace.

L'idee cle : la geometrie du sens

Les mots vivent dans un espace mathematique

Imaginez un espace en 3D (en realite, c'est 768 a 12288 dimensions, mais le principe est le meme). Dans cet espace :

  • "chat" et "chien" sont proches (ce sont des animaux domestiques)
  • "chat" et "avion" sont eloignes (aucun rapport semantique)
  • "roi" et "reine" ont la meme direction que "homme" et "femme"

La decouverte la plus celebre de Word2Vec (2013) :

roi - homme + femme = reine

Dimensions des embeddings par modele

ModeleDimensionsType

Plus il y a de dimensions, plus le modele peut capturer de nuances semantiques. Mais cela coute plus de memoire et de calcul. Les modeles d'embeddings comme text-embedding-3-small d'OpenAI permettent de reduire les dimensions apres coup (par exemple de 1536 a 256) sans trop perdre en qualite, grace a la technique de "Matryoshka".

Embeddings statiques vs contextuels

Statiques (Word2Vec, GloVe) : Chaque mot a UN seul vecteur, quel que soit le contexte. Le mot "banque" a le meme embedding qu'il parle de finance ou de riviere.

Contextuels (BERT, GPT) : Le vecteur d'un mot change selon son contexte. "banque" dans "je vais a la banque" (finance) aura un embedding different de "la banque de la riviere" (geographie). C'est un progres majeur.

Code : creer des embeddings avec OpenAI

"La voiture roule sur l'autoroute", # Tres different "Python est un langage de programmation", "Le serpent python vit en Afrique", # Meme mot, sens different ] embeddings = [get_embedding(p) for p in phrases] print(f"Dimensions de chaque embedding : {len(embeddings[0])}") print(f"Premiers 5 valeurs du 1er embedding : {embeddings[0][:5].round(4)}")

6. Similarite cosinus : mesurer la proximite semantique

La similarite cosinus mesure l'angle entre deux vecteurs. Elle donne un score entre -1 (opposes) et 1 (identiques). C'est la metrique standard pour comparer des embeddings.
cosine_similarity(A, B) = (A . B) / (||A|| * ||B||)

Ou A . B est le produit scalaire et ||A|| est la norme du vecteur.

Interpretation des scores

ScoreInterpretationExemple
0.90 - 1.00Quasi-identiques / paraphrases"Le chat dort" vs "Le felin sommeille"
0.70 - 0.89Tres similaires / meme sujet"Le chat dort" vs "Mon chat se repose"
0.50 - 0.69Assez lies / meme domaine"Le chat dort" vs "J'ai un animal de compagnie"
0.20 - 0.49Peu lies"Le chat dort" vs "La voiture roule vite"
0.00 - 0.19Aucun rapport"Le chat dort" vs "Les equations differentielles"

Code : calculer la similarite cosinus

import numpy as np from openai import OpenAI client = OpenAI() def get_embedding(text, model="text-embedding-3-small"): response = client.embeddings.create(input=text, model=model) return np.array(response.data[0].embedding) def cosine_similarity(a, b): """Calcul de la similarite cosinus entre deux vecteurs.""" return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) # Creer des embeddings phrases = { "A": "Le chat dort sur le canape", "B": "Le felin sommeille sur le sofa", "C": "La voiture roule sur l'autoroute", "D": "Python est un langage de programmation", "E": "Le serpent python vit en Afrique", } embeddings = {k: get_embedding(v) for k, v in phrases.items()} # Matrice de similarite print("Matrice de similarite cosinus :") print(f"{'':>5}", end="") for k in phrases: print(f"{k:>8}", end="") print() for k1, e1 in embeddings.items(): print(f"{k1:>5}", end="") for k2, e2 in embeddings.items(): sim = cosine_similarity(e1, e2) print(f"{sim:>8.3f}", end="") print()
Matrice de similarite cosinus : A B C D E A 1.000 0.921 0.312 0.198 0.287 B 0.921 1.000 0.298 0.185 0.271 C 0.312 0.298 1.000 0.234 0.201 D 0.198 0.185 0.234 1.000 0.456 E 0.287 0.271 0.201 0.456 1.000 A et B (chat/felin) : 0.921 = quasi-identiques ! D et E (python code/serpent) : 0.456 = lies par le mot "python" A et C (chat/voiture) : 0.312 = peu de rapport
On observe que les phrases semantiquement proches (A et B) ont un score tres eleve (0.92), tandis que des phrases sans rapport (A et D) ont un score faible (0.20). C'est exactement ce comportement qu'on exploite dans le RAG pour trouver les documents pertinents.

7. Word Embeddings vs Sentence Embeddings

Il est crucial de distinguer deux types d'embeddings qui servent a des fins differentes :

Word Embeddings

Un vecteur par mot

Exemples : Word2Vec, GloVe, couches internes de BERT/GPT

Usage : analogies, NER, POS tagging

Sentence Embeddings

Un vecteur pour une phrase entiere

Exemples : text-embedding-3, sentence-transformers, BGE-M3

Usage : RAG, recherche semantique, clustering

Evolution historique

GenerationModeleAnneeTypeLimite
1ere gen Word2Vec, GloVe 2013-2014 Word (statique) Un seul vecteur par mot, pas de contexte
2eme gen ELMo 2018 Word (contextuel) Base sur LSTM, lent
3eme gen BERT 2018 Word/sentence (contextuel) Pas optimise pour la similarite de phrases
4eme gen Sentence-BERT, DPR 2019-2020 Sentence (optimise) Entrainement specifique pour la similarite
5eme gen OpenAI text-embedding-3, BGE-M3 2023-2025 Sentence (SOTA) Multilingue, multi-tache, Matryoshka
Pour le RAG (Retrieval-Augmented Generation), on utilise toujours des sentence embeddings. L'idee est de transformer la question de l'utilisateur ET les documents en vecteurs, puis de trouver les documents dont le vecteur est le plus proche de la question. C'est la base de la recherche semantique.

Code : sentence-transformers (gratuit et local)

# Installation : pip install sentence-transformers from sentence_transformers import SentenceTransformer # Charger un modele d'embeddings (gratuit, tourne en local) model = SentenceTransformer('all-MiniLM-L6-v2') # Creer des embeddings pour des phrases phrases = [ "Le machine learning est une branche de l'IA", "L'apprentissage automatique fait partie de l'intelligence artificielle", "La recette du gateau au chocolat est simple", ] embeddings = model.encode(phrases) print(f"Shape des embeddings : {embeddings.shape}") # (3, 384) = 3 phrases, 384 dimensions chacune # Calculer la similarite from sklearn.metrics.pairwise import cosine_similarity sim_matrix = cosine_similarity(embeddings) print("\nMatrice de similarite :") print(f"Phrase 1 vs 2 : {sim_matrix[0][1]:.3f}") # Elevee ! (meme sens) print(f"Phrase 1 vs 3 : {sim_matrix[0][2]:.3f}") # Basse (aucun rapport)

8. Application pratique : recherche semantique

Mettons tout ensemble dans un mini-systeme de recherche semantique, le coeur du RAG :

Code complet : mini recherche semantique

from sentence_transformers import SentenceTransformer import numpy as np # 1. Charger le modele d'embeddings model = SentenceTransformer('all-MiniLM-L6-v2') # 2. Notre "base de connaissances" (dans un vrai RAG, ce sont des documents) documents = [ "Le machine learning est une branche de l'intelligence artificielle.", "Les reseaux de neurones sont inspires du cerveau humain.", "Le deep learning utilise des reseaux de neurones profonds.", "Python est le langage le plus utilise en data science.", "Le RAG combine la recherche documentaire avec la generation par LLM.", "Les embeddings representent les mots sous forme de vecteurs numeriques.", "GPT-4 est un modele de langage cree par OpenAI.", "La tokenisation decoupe le texte en unites appelees tokens.", "La descente de gradient est l'algorithme d'optimisation principal du ML.", "Le Transformer utilise le mecanisme d'attention pour traiter le texte.", ] # 3. Creer les embeddings des documents (indexation) doc_embeddings = model.encode(documents) print(f"Base indexee : {len(documents)} documents, {doc_embeddings.shape[1]} dimensions") # 4. Fonction de recherche def search(query, top_k=3): """Recherche semantique : trouver les documents les plus similaires.""" # Embedder la question query_embedding = model.encode([query]) # Calculer la similarite avec tous les documents similarities = np.dot(doc_embeddings, query_embedding.T).flatten() # Trier par similarite decroissante top_indices = similarities.argsort()[::-1][:top_k] print(f"\nQuestion : '{query}'") print(f"Top {top_k} resultats :") for i, idx in enumerate(top_indices, 1): print(f" {i}. [{similarities[idx]:.3f}] {documents[idx]}") # 5. Tester la recherche search("Comment fonctionne l'attention dans les modeles de langage ?") search("Qu'est-ce que le RAG ?") search("Comment representer les mots en nombres ?")

La recherche semantique fonctionne parfaitement : meme si les mots exacts de la question ne sont pas dans le document, le systeme trouve le bon resultat grace a la proximite des embeddings. C'est la magie de la similarite semantique.
Ce code est une version simplifiee de ce qu'on appelle un pipeline RAG. En production, on utilise des bases vectorielles comme ChromaDB, FAISS, Pinecone ou Weaviate pour stocker et rechercher efficacement des millions de documents.

Resume complet du cours

Ce qu'il faut retenir

  • La tokenisation decoupe le texte en sous-unites (tokens) car les modeles ne comprennent que des nombres
  • L'algorithme BPE (Byte Pair Encoding) est le standard : mots frequents = 1 token, mots rares = plusieurs sous-tokens
  • Le vocabulaire est l'ensemble des tokens connus (50K a 200K selon le modele)
  • Les embeddings transforment les tokens en vecteurs dans un espace semantique ou les mots proches sont voisins
  • La similarite cosinus mesure la proximite semantique entre deux vecteurs (de 0 a 1)
  • Les sentence embeddings (OpenAI, sentence-transformers) sont la base du RAG et de la recherche semantique
  • La formule cle : roi - homme + femme = reine illustre la geometrie du sens dans l'espace des embeddings
  • Le pipeline complet : texte -> tokens -> IDs -> embeddings -> Transformer -> probabilites -> texte genere
