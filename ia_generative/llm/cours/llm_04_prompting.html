Prompt Engineering - L'Art de Parler aux LLMs

Cours 04 - Maîtriser les techniques de formulation pour obtenir des résultats optimaux

1. Qu'est-ce que le Prompt Engineering ?

Le Prompt Engineering (ingénierie de prompt) est la discipline qui consiste à concevoir, structurer et optimiser les instructions textuelles envoyées à un modèle de langage (LLM) afin d'obtenir des réponses précises, pertinentes et de haute qualité.

Pourquoi est-ce si important ? Un même modèle peut produire des résultats radicalement différents selon la façon dont on lui pose une question. Considérez les deux prompts suivants :

# Prompt vague prompt_v1 = "Parle-moi de Python"
Python est un langage de programmation... (réponse générique, peu structurée, 3 paragraphes vagues)
# Prompt bien structuré prompt_v2 = """Tu es un professeur d'informatique. Explique les 5 avantages principaux de Python pour un débutant. Format : liste numérotée avec un exemple de code pour chaque point. Longueur : 300 mots maximum."""
1. Syntaxe lisible : x = 5 (pas de point-virgule) 2. Typage dynamique : nom = "Alice" (pas besoin de déclarer le type) 3. Bibliothèques riches : import pandas as pd ... (réponse structurée, précise, avec exemples)
Le prompt engineering n'est pas de la programmation classique, mais c'est devenu une compétence professionnelle à part entière en 2025-2026. De nombreuses entreprises recrutent désormais des "Prompt Engineers" spécialisés.

Principe fondamental

La qualité de la sortie d'un LLM est directement proportionnelle à la qualité de l'entrée (le prompt). Un prompt bien conçu peut transformer un modèle moyen en outil puissant, tandis qu'un mauvais prompt peut rendre inutile même le modèle le plus avancé.

2. Anatomie d'un bon prompt

Un prompt efficace se compose de plusieurs éléments clés, chacun jouant un rôle spécifique dans la qualité de la réponse obtenue :

# Prompt complet intégrant tous les éléments prompt_complet = { "system": "Tu es un nutritionniste diplômé avec 10 ans d'expérience.", # Rôle "user": """ Crée un plan alimentaire hebdomadaire. # Instruction Contexte : Patient de 30 ans, sportif, intolérant au gluten. # Contexte Exemple de format pour un jour : # Exemple - Lundi : Petit-déj: Smoothie banane-avoine | Déj: Salade quinoa | Dîner: Saumon grillé Règles : # Contraintes - 2000-2500 kcal/jour - Pas de gluten - Format : tableau avec colonnes Jour|Repas|Calories # Format """ }
Tous les éléments ne sont pas toujours nécessaires. Pour une question simple, une instruction claire peut suffire. Mais pour des tâches complexes, chaque composant améliore significativement la qualité du résultat.

3. Techniques de base

3.1 Zero-shot Prompting

Le zero-shot prompting consiste à demander au modèle d'exécuter une tâche sans lui fournir aucun exemple. On s'appuie entièrement sur les connaissances déjà acquises par le modèle lors de son entraînement.
# Zero-shot : pas d'exemple fourni prompt = "Classifie le sentiment de cette phrase comme positif, négatif ou neutre : 'Ce restaurant a une cuisine exceptionnelle mais le service est déplorable.'" response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}] )
Sentiment : Mitigé/Neutre. La phrase exprime à la fois un avis très positif sur la cuisine ("exceptionnelle") et un avis très négatif sur le service ("déplorable").

3.2 Few-shot Prompting

Le few-shot prompting fournit quelques exemples (généralement 2 à 5) de la tâche attendue avant de poser la question réelle. Cela guide le modèle vers le format et le type de réponse souhaité.
# Few-shot : on donne des exemples pour guider le modèle prompt = """Classifie le sentiment de chaque phrase. Phrase : "J'adore ce film, il est magnifique !" Sentiment : positif Phrase : "Le temps est couvert aujourd'hui." Sentiment : neutre Phrase : "Cette application plante constamment, c'est inutilisable." Sentiment : négatif Phrase : "Le nouveau campus universitaire est vraiment bien conçu." Sentiment : """
positif
Avec le few-shot, le modèle comprend exactement le format attendu (un seul mot) et la granularité de classification (positif/négatif/neutre). Le taux de précision augmente de 15 à 30% par rapport au zero-shot pour les tâches de classification.

3.3 Instruction-following

Cette technique repose sur des instructions explicites et détaillées qui décrivent précisément ce que le modèle doit faire :

# Instruction-following : instructions très détaillées prompt = """Analyse le texte suivant en effectuant ces étapes : 1. Identifie la langue du texte 2. Extrais les entités nommées (personnes, lieux, organisations) 3. Résume le texte en une phrase 4. Attribue un score de complexité de 1 à 10 Texte : 'Le président Emmanuel Macron a inauguré le nouveau siège de l'INRIA à Saclay lors d'une cérémonie officielle.' Format de réponse : - Langue : [langue] - Entités : [liste] - Résumé : [phrase] - Complexité : [score]/10"""
- Langue : Français - Entités : Emmanuel Macron (personne), INRIA (organisation), Saclay (lieu) - Résumé : Macron a inauguré le nouveau siège de l'INRIA à Saclay. - Complexité : 3/10

4. Techniques avancées

4.1 Chain-of-Thought (CoT) - Chaîne de Pensée

Le Chain-of-Thought (CoT) est une technique qui encourage le modèle à raisonner étape par étape plutôt que de sauter directement à la réponse finale. Cette approche améliore considérablement les performances sur les tâches de raisonnement logique et mathématique.
# SANS Chain-of-Thought prompt_sans_cot = "Si un magasin vend 3 chemises à 25 euros chacune avec une réduction de 15%, quel est le montant total ?"
Le montant total est de 63,75 euros. (réponse directe, parfois incorrecte)
# AVEC Chain-of-Thought prompt_avec_cot = """Si un magasin vend 3 chemises à 25 euros chacune avec une réduction de 15%, quel est le montant total ? Réfléchis étape par étape avant de donner ta réponse finale."""
Étape 1 : Prix d'une chemise = 25 euros Étape 2 : Prix de 3 chemises = 3 x 25 = 75 euros Étape 3 : Réduction de 15% = 75 x 0.15 = 11.25 euros Étape 4 : Total après réduction = 75 - 11.25 = 63.75 euros Réponse finale : Le montant total est de 63,75 euros.
Le CoT augmente la précision des calculs mathématiques de 20 à 40% selon les études. La simple phrase "Réfléchis étape par étape" active le raisonnement séquentiel du modèle.

4.2 Step-by-Step structuré

Variante plus dirigée du CoT où l'on impose explicitement les étapes de raisonnement :

# Step-by-step avec étapes imposées prompt = """Détermine si l'argument suivant est valide. Argument : "Tous les développeurs connaissent Python. Marie est développeuse. Donc Marie connaît Python." Suis ces étapes : Étape 1 - Identifie les prémisses Étape 2 - Identifie la conclusion Étape 3 - Vérifie la structure logique (syllogisme) Étape 4 - Évalue la validité des prémisses dans le monde réel Étape 5 - Conclusion finale"""
Étape 1 - Prémisses : (P1) Tous les développeurs connaissent Python. (P2) Marie est développeuse. Étape 2 - Conclusion : Marie connaît Python. Étape 3 - Structure : Syllogisme valide (Barbara) : Tout A est B, C est A, donc C est B. Étape 4 - Validité réelle : P1 est FAUSSE (tous les développeurs ne connaissent pas Python). Étape 5 - L'argument est logiquement valide mais factuellement incorrect.

4.3 Self-Consistency (Auto-cohérence)

La Self-Consistency consiste à générer plusieurs raisonnements indépendants pour la même question, puis à choisir la réponse la plus fréquente (vote majoritaire). Cela réduit l'impact des erreurs aléatoires.
import anthropic from collections import Counter client = anthropic.Anthropic() def self_consistency(question, n=5): """Génère n raisonnements et retourne la réponse majoritaire.""" reponses = [] for i in range(n): response = client.messages.create( model="claude-sonnet-4-5-20250514", max_tokens=500, temperature=0.7, # Température > 0 pour diversité messages=[{ "role": "user", "content": question + "\nRéfléchis étape par étape. Termine par 'Réponse: [X]'" }] ) # Extraire la réponse finale texte = response.content[0].text reponse_finale = texte.split("Réponse:")[-1].strip() reponses.append(reponse_finale) # Vote majoritaire compteur = Counter(reponses) return compteur.most_common(1)[0][0] resultat = self_consistency( "Un fermier a 17 moutons. Tous sauf 9 meurent. Combien en reste-t-il ?" ) print(f"Réponse finale : {resultat}")
Réponse finale : 9 (obtenue par vote majoritaire sur 5 raisonnements indépendants)

4.4 Tree of Thoughts (ToT) - Arbre de Pensées

Le Tree of Thoughts (ToT) généralise le Chain-of-Thought en explorant plusieurs chemins de raisonnement simultanément, construisant un arbre où chaque noeud représente une étape intermédiaire et chaque branche explore une approche alternative. Le modèle évalue chaque branche et peut abandonner les pistes non prometteuses.
# Tree of Thoughts - implémentation simplifiée prompt_tot = """Résous ce problème en explorant 3 approches différentes. Problème : Comment réduire le temps de réponse d'une API de 2s à 200ms ? Pour chaque approche : 1. Décris l'approche en une phrase 2. Liste les étapes de mise en oeuvre 3. Évalue la faisabilité (facile/moyen/difficile) 4. Estime le gain potentiel Ensuite, choisis la meilleure approche et justifie ton choix."""
Approche 1 : Mise en cache Redis - Étapes : Installer Redis, identifier les requêtes fréquentes, implémenter le cache - Faisabilité : Facile | Gain : -80% (400ms) Approche 2 : Optimisation des requêtes SQL - Étapes : Analyser les requêtes lentes, ajouter des index, réécrire les JOIN - Faisabilité : Moyen | Gain : -60% (800ms) Approche 3 : Architecture microservices + CDN - Étapes : Découper le monolithe, déployer sur CDN, load balancer - Faisabilité : Difficile | Gain : -90% (200ms) Meilleur choix : Combinaison des approches 1 et 2 (cache + SQL) Justification : Gain cumulé de ~90% avec une faisabilité modérée.
Le Tree of Thoughts est particulièrement efficace pour les problèmes créatifs, la planification stratégique et les scénarios de décision complexes. Il imite le processus de réflexion humain où l'on considère plusieurs options avant de s'engager.

5. System Prompts

Le system prompt (message système) est une instruction spéciale envoyée au modèle qui définit son comportement global, son rôle et ses contraintes avant toute interaction avec l'utilisateur.

5.1 Attribution de rôle (Role-based prompting)

# Différents rôles pour différents résultats # Rôle 1 : Expert technique system_dev = "Tu es un architecte logiciel senior avec 15 ans d'expérience en Python et systèmes distribués. Tu réponds de manière technique et précise, en citant les design patterns pertinents." # Rôle 2 : Pédagogue system_prof = "Tu es un professeur d'informatique qui enseigne à des étudiants de première année. Tu utilises des analogies simples, des exemples concrets et tu évites le jargon technique non expliqué." # Rôle 3 : Reviewer de code system_review = "Tu es un reviewer de code strict mais bienveillant. Pour chaque code soumis, tu identifies : - Les bugs potentiels - Les problèmes de performance - Les violations des bonnes pratiques Tu proposes toujours une version corrigée."

5.2 Contraintes et garde-fous

# System prompt avec contraintes system_contraint = """Tu es un assistant juridique spécialisé en droit français. CONTRAINTES : - Réponds UNIQUEMENT sur le droit français - Cite toujours les articles de loi pertinents - Précise que tu ne remplaces pas un avocat - Ne donne JAMAIS de conseil médical - Limite tes réponses à 500 mots maximum - Si la question sort de ton domaine, indique-le clairement FORMAT : - Utilise des puces pour la clarté - Termine par une section "Références légales" """ response = client.messages.create( model="claude-sonnet-4-5-20250514", max_tokens=1024, system=system_contraint, messages=[{ "role": "user", "content": "Quels sont mes droits en cas de licenciement abusif ?" }] )
Le system prompt est l'outil le plus puissant pour contrôler le comportement du modèle. Il est traité avec une priorité plus élevée que les messages utilisateur par la plupart des APIs (OpenAI, Anthropic).

5.3 Persona et style de communication

# Persona : Créer un style unique system_persona = """Tu es "DataBot", un assistant d'analyse de données. Personnalité : - Précis et factuel (cite toujours les chiffres) - Utilise des métaphores liées à la cuisine pour expliquer les concepts - Termine chaque réponse par un "Conseil du chef" lié aux données Style : - Phrases courtes et directes - Utilise des tableaux quand possible - Inclut toujours une recommandation actionnable """
Ne confondez pas le system prompt avec les instructions utilisateur. Le system prompt définit le cadre général (qui est le modèle), tandis que le message utilisateur contient la tâche spécifique à exécuter.

6. Formatage de la sortie

Contrôler le format de sortie est essentiel pour intégrer les réponses des LLMs dans des pipelines de traitement automatisés.

6.1 Sortie JSON

# Demander une sortie JSON structurée prompt_json = """Analyse le produit suivant et retourne un JSON. Produit : "iPhone 15 Pro Max 256Go - 1399 euros - Apple Store" Retourne UNIQUEMENT un JSON valide avec cette structure : { "nom": "string", "marque": "string", "stockage_go": number, "prix_euros": number, "categorie": "string", "gamme": "entree|milieu|haut" } Pas de texte avant ou après le JSON."""
{ "nom": "iPhone 15 Pro Max", "marque": "Apple", "stockage_go": 256, "prix_euros": 1399, "categorie": "smartphone", "gamme": "haut" }

6.2 Sortie XML

# Sortie XML pour intégration avec des systèmes legacy prompt_xml = """Convertis cette information en XML valide : "Marie Dupont, 28 ans, ingénieure logiciel chez TechCorp, Paris" Utilise la structure : <employe> <identite>...</identite> <poste>...</poste> <entreprise>...</entreprise> </employe>"""
<employe> <identite> <nom>Dupont</nom> <prenom>Marie</prenom> <age>28</age> </identite> <poste>Ingénieure logiciel</poste> <entreprise> <nom>TechCorp</nom> <ville>Paris</ville> </entreprise> </employe>

6.3 Sortie Markdown structurée

# Sortie Markdown pour documentation prompt_md = """Génère la documentation de cette fonction Python en Markdown : def calculer_imc(poids_kg, taille_m): return poids_kg / (taille_m ** 2) Format attendu : ## Nom de la fonction ### Description ### Paramètres | Paramètre | Type | Description | ### Valeur de retour ### Exemple d'utilisation"""

6.4 Sortie structurée avec schéma (Structured Outputs)

from openai import OpenAI from pydantic import BaseModel # Utilisation de Structured Outputs (OpenAI, 2025+) class AnalyseSentiment(BaseModel): texte: str sentiment: str # positif, négatif, neutre confiance: float # 0.0 à 1.0 mots_cles: list[str] client = OpenAI() completion = client.beta.chat.completions.parse( model="gpt-4o", messages=[ {"role": "system", "content": "Analyse le sentiment du texte."}, {"role": "user", "content": "J'adore cette bibliothèque, elle simplifie tout !"} ], response_format=AnalyseSentiment ) resultat = completion.choices[0].message.parsed print(resultat.model_dump_json(indent=2))
{ "texte": "J'adore cette bibliothèque, elle simplifie tout !", "sentiment": "positif", "confiance": 0.95, "mots_cles": ["adore", "simplifie"] }
Les Structured Outputs (sorties structurées) disponibles via l'API OpenAI garantissent que la réponse respecte exactement le schéma Pydantic défini. C'est beaucoup plus fiable que de demander du JSON dans le prompt texte.

7. Prompt Patterns - Modèles réutilisables

Voici une collection de patterns de prompts éprouvés pour les tâches les plus courantes :

Tâche Modèle de prompt Astuce
Extraction "Extrais les [entités] du texte suivant. Retourne un JSON avec les clés : [clés]." Spécifier le format exact de sortie
Résumé "Résume ce texte en [N] phrases/points. Public cible : [audience]. Focus : [aspect]." Préciser la longueur et l'audience
Traduction "Traduis de [source] vers [cible]. Conserve le ton [formel/informel]. Domaine : [spécialité]." Indiquer le domaine spécialisé
Classification "Classifie dans les catégories suivantes : [A, B, C]. Retourne uniquement la catégorie." Lister les catégories possibles
Génération de code "Écris une fonction [langage] qui [action]. Inclus : docstring, types hints, tests unitaires." Demander les tests et la doc
Analyse de sentiment "Analyse le sentiment : positif/négatif/neutre. Score de confiance 0-1. Mots-clés détectés." Demander le score numérique
Q&A sur document "En te basant UNIQUEMENT sur le contexte ci-dessous, réponds à la question. Si non trouvé, dis 'Non trouvé'." Empêcher l'hallucination
Réécriture "Réécris ce texte en conservant le sens mais en changeant le style vers [style cible]." Donner un exemple du style visé

7.2 Pattern de génération de code

# Pattern pour générer du code de qualité production prompt_code = """Écris une fonction Python qui respecte ces spécifications : Nom : valider_email But : Vérifie si une chaîne est un email valide Entrée : email (str) Sortie : bool Exigences : - Utilise une regex complète - Inclus une docstring Google-style - Ajoute les type hints - Écris 5 tests unitaires avec pytest - Gère les cas limites (None, chaîne vide, espaces) """
En spécifiant clairement les exigences de qualité (docstring, type hints, tests), le code généré est directement utilisable en production, sans retouche manuelle.

8. Erreurs courantes en Prompt Engineering

Les 8 péchés capitaux du prompting

1. Le prompt vague - "Parle-moi d'IA" au lieu de "Explique les 3 types d'apprentissage automatique avec un exemple concret pour chacun." Un prompt sans direction donne une réponse sans valeur.
2. Le prompt fleuve - Un prompt de 2000 mots avec des détails inutiles. Le modèle se perd dans le bruit et rate l'essentiel. Soyez concis mais précis.
3. L'absence d'exemples - Ne pas fournir d'exemples quand le format de sortie est spécifique. Le modèle doit deviner ce que vous voulez. Le few-shot résout ce problème.
4. Aucun format demandé - Ne pas spécifier si vous voulez du JSON, une liste, un paragraphe. Le modèle choisit arbitrairement et le résultat est imprévisible.
5. Ignorer le system prompt - Ne pas définir de rôle. C'est comme demander à un généraliste quand vous avez besoin d'un spécialiste.
6. Instructions contradictoires - "Sois bref mais détaille tout" ou "Réponds en JSON et en texte libre". Le modèle ne peut pas satisfaire deux exigences opposées.
7. Ne pas itérer - Envoyer un seul prompt et accepter le résultat tel quel. Les meilleurs résultats viennent d'un processus itératif d'amélioration.
8. Ignorer les limites du modèle - Demander des calculs mathématiques complexes ou des données en temps réel à un modèle qui n'y a pas accès.

8.1 Exemples de corrections

Mauvais prompt Bon prompt Pourquoi
"Fais-moi un site web" "Crée une page HTML/CSS pour un portfolio. 3 sections : à propos, projets, contact. Style minimaliste, palette bleue." Spécifications concrètes
"Corrige mon code" "Ce code Python lève une TypeError à la ligne 15. Identifie le bug, explique la cause, et propose la correction." Contexte de l'erreur fourni
"Résume ce texte" "Résume ce texte en 3 bullet points pour un public non-technique. Maximum 50 mots." Longueur et audience définies
"Traduis en anglais" "Traduis en anglais américain professionnel. Domaine : juridique. Conserve les termes techniques en français entre parenthèses." Registre et domaine précisés
Le prompt engineering est un processus itératif. Ne cherchez pas le prompt parfait du premier coup. Commencez simple, testez, analysez les erreurs, et affinez progressivement. La règle d'or : si le modèle ne donne pas le résultat attendu, c'est probablement le prompt qui est à améliorer, pas le modèle.

9. Résumé des techniques

Tableau récapitulatif des techniques de prompting

Technique Description Quand l'utiliser Complexité
Zero-shot Question directe sans exemple Tâches simples et génériques Faible
Few-shot Inclure 2-5 exemples dans le prompt Classification, format spécifique, tâches inhabituelles Faible
Instruction-following Étapes et contraintes explicites Tâches multi-étapes, pipelines Moyenne
Chain-of-Thought Raisonnement étape par étape Math, logique, raisonnement complexe Moyenne
Self-Consistency Multiple raisonnements + vote majoritaire Décisions critiques, précision maximale Élevée
Tree of Thoughts Exploration de chemins multiples en arbre Problèmes créatifs, planification, décision Élevée
Role-based Assigner un rôle d'expert via system prompt Toute tâche spécialisée Faible
Structured Output Imposer un format JSON/XML/schéma Intégration API, pipelines automatisés Moyenne

Les 5 règles d'or du Prompt Engineering

  1. Soyez spécifique : Plus le prompt est précis, meilleure est la réponse.
  2. Donnez du contexte : Le modèle ne peut pas deviner ce que vous savez déjà.
  3. Spécifiez le format : Dites exactement comment vous voulez la réponse.
  4. Utilisez des exemples : Le few-shot est votre meilleur allié pour les tâches spécifiques.
  5. Itérez : Testez, analysez, améliorez. Le premier prompt n'est jamais le dernier.
En 2025-2026, le prompt engineering évolue rapidement avec l'arrivée de modèles de raisonnement (comme o1 et o3 d'OpenAI, ou le mode "extended thinking" de Claude) qui intègrent le Chain-of-Thought automatiquement. Le métier se déplace vers l'orchestration de prompts complexes, le "prompt blending" (combinaison de techniques) et la conception d'agents autonomes.
