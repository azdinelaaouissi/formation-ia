APIs LLM et Paramètres - Contrôler la Génération

Cours 05 - Maîtriser les APIs d'OpenAI, Anthropic et les paramètres qui contrôlent le comportement des LLMs

1. Les APIs LLM

Une API LLM (Application Programming Interface) est une interface qui permet à votre code d'envoyer des requêtes textuelles à un modèle de langage hébergé dans le cloud et de recevoir des réponses générées. Vous n'avez pas besoin d'héberger le modèle vous-même : vous payez à l'utilisation (par token).

1.1 Les principaux fournisseurs (2025-2026)

Fournisseur Modèles phares Point fort
OpenAI GPT-4o, GPT-4o-mini, o3, o4-mini Écosystème mature, Structured Outputs
Anthropic Claude Opus 4.5, Claude Sonnet 4.5, Claude Haiku 4.5 Sécurité, fenêtre de contexte 200K-1M tokens
Google Gemini 2.5 Pro, Gemini 2.5 Flash Multimodal natif, longue fenêtre contexte
Meta (open source) Llama 4 Scout, Llama 4 Maverick Gratuit, auto-hébergeable
Mistral AI Mistral Large, Mistral Medium Entreprise française, souveraineté des données

1.2 Structure d'un appel API - OpenAI

from openai import OpenAI # Initialisation du client client = OpenAI(api_key="sk-votre-cle-api") # Appel API de base response = client.chat.completions.create( model="gpt-4o", # Modèle à utiliser messages=[ # Historique de conversation { "role": "system", # Message système (rôle) "content": "Tu es un assistant utile et concis." }, { "role": "user", # Message de l'utilisateur "content": "Qu'est-ce qu'une API REST ?" } ], temperature=0.7, # Créativité (0.0 à 2.0) max_tokens=500, # Longueur max de la réponse top_p=1.0, # Nucleus sampling ) # Extraire la réponse reponse_texte = response.choices[0].message.content tokens_utilises = response.usage.total_tokens print(f"Réponse : {reponse_texte}") print(f"Tokens utilisés : {tokens_utilises}")
Réponse : Une API REST (Representational State Transfer) est une interface qui permet à des applications de communiquer via HTTP en utilisant des méthodes standard (GET, POST, PUT, DELETE) pour manipuler des ressources identifiées par des URLs. Tokens utilisés : 127

1.3 Structure d'un appel API - Anthropic (Claude)

import anthropic # Initialisation du client Anthropic client = anthropic.Anthropic(api_key="sk-ant-votre-cle-api") # Appel API Claude response = client.messages.create( model="claude-sonnet-4-5-20250514", # Modèle Claude max_tokens=1024, # Longueur max system="Tu es un assistant technique expert en Python.", # System prompt messages=[ { "role": "user", "content": "Explique les list comprehensions en Python." } ], temperature=0.5, # Créativité (0.0 à 1.0) top_p=0.9, # Nucleus sampling ) # Extraire la réponse reponse_texte = response.content[0].text tokens_entree = response.usage.input_tokens tokens_sortie = response.usage.output_tokens print(f"Réponse : {reponse_texte}") print(f"Tokens entrée : {tokens_entree} | Tokens sortie : {tokens_sortie}")
Réponse : Une list comprehension est une syntaxe concise pour créer des listes en Python. Au lieu d'utiliser une boucle for classique, vous écrivez : [expression for item in iterable if condition]. Tokens entrée : 42 | Tokens sortie : 89
Notez les différences entre les deux APIs : chez OpenAI, le system prompt est un message dans la liste messages. Chez Anthropic, c'est un paramètre system séparé. La structure de la réponse diffère aussi : response.choices[0].message.content (OpenAI) vs response.content[0].text (Anthropic).

2. Le paramètre Temperature

La temperature contrôle le degré d'aléatoire (ou de "créativité") dans la génération de texte. Elle modifie la distribution de probabilité des tokens suivants : une température basse concentre les probabilités sur les tokens les plus probables, tandis qu'une température haute aplatit la distribution, rendant les tokens moins probables plus accessibles.

Temperature : plage et comportement

  • 0.0 : Déterministe - Toujours le même token le plus probable. Idéal pour les tâches factuelles.
  • 0.3 - 0.5 : Peu créatif - Réponses cohérentes, légèrement variées. Bon pour le code et l'analyse.
  • 0.7 : Équilibré (valeur par défaut courante) - Bon équilibre entre cohérence et créativité.
  • 1.0 : Créatif - Distribution naturelle. Bon pour l'écriture créative.
  • 1.5 - 2.0 : Très aléatoire - Risque d'incohérence. Rarement utile en production.

2.1 Comparaison pratique des températures

from openai import OpenAI client = OpenAI() def generer_avec_temperature(prompt, temp): """Génère une réponse avec une température donnée.""" response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=temp, max_tokens=100 ) return response.choices[0].message.content prompt = "Invente un nom pour une startup d'IA" # Température 0.0 - Toujours le même résultat for i in range(3): print(f"Temp 0.0, essai {i+1} :", generer_avec_temperature(prompt, 0.0)) # Température 0.7 - Varié mais cohérent for i in range(3): print(f"Temp 0.7, essai {i+1} :", generer_avec_temperature(prompt, 0.7)) # Température 1.5 - Très créatif, parfois incohérent for i in range(3): print(f"Temp 1.5, essai {i+1} :", generer_avec_temperature(prompt, 1.5))
Temp 0.0, essai 1 : NeuralForge Temp 0.0, essai 2 : NeuralForge Temp 0.0, essai 3 : NeuralForge Temp 0.7, essai 1 : CogniWave Temp 0.7, essai 2 : SynapTech Temp 0.7, essai 3 : MindBridge AI Temp 1.5, essai 1 : Zephyra Cognitêl Temp 1.5, essai 2 : QuantumDream Nexus++ Temp 1.5, essai 3 : Blüfyre Synth_Minds
À temperature 0.0, le résultat est toujours identique. À 0.7, les noms sont créatifs et utilisables. À 1.5, on obtient des résultats originaux mais parfois bizarres ou mal formés.
Chez Anthropic (Claude), la temperature va de 0.0 à 1.0 (pas 2.0 comme chez OpenAI). La valeur par défaut pour GPT-4o est 1.0. Pour les tâches de code ou d'extraction de données, utilisez toujours 0.0 à 0.3.

3. top_p (Nucleus Sampling)

Le paramètre top_p (aussi appelé "nucleus sampling") définit un seuil de probabilité cumulée. Le modèle ne considère que les tokens dont la probabilité cumulée atteint p. Par exemple, top_p=0.9 signifie que le modèle choisit parmi les tokens qui représentent 90% de la masse de probabilité, ignorant les 10% les plus improbables.

3.1 Temperature vs top_p

Critère Temperature top_p
Mécanisme Modifie les probabilités de TOUS les tokens Coupe les tokens les moins probables
Plage 0.0 à 2.0 (OpenAI) / 0.0 à 1.0 (Anthropic) 0.0 à 1.0
Valeur pour déterminisme 0.0 Non applicable (utiliser temperature 0.0)
Valeur par défaut 1.0 (OpenAI) / 1.0 (Anthropic) 1.0
Usage recommandé Contrôle général de la créativité Éliminer les tokens improbables/aberrants
OpenAI recommande de modifier soit la temperature, soit top_p, mais pas les deux en même temps. Modifier les deux peut créer des interactions imprévisibles. La bonne pratique : fixez top_p à 1.0 et ajustez uniquement la temperature (ou inversement).
# Bonne pratique : ajuster un seul paramètre à la fois # Option 1 : Utiliser temperature (plus intuitif) response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Écris un poème"}], temperature=0.9, # Ajusté top_p=1.0, # Défaut ) # Option 2 : Utiliser top_p (coupe les extrêmes) response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Analyse ce contrat"}], temperature=1.0, # Défaut top_p=0.85, # Ajusté : ignore les 15% les moins probables )

4. max_tokens - Contrôle de la longueur

Le paramètre max_tokens définit le nombre maximum de tokens que le modèle peut générer dans sa réponse. Si la réponse atteint cette limite, elle est tronquée brutalement (même en plein milieu d'une phrase).
# Démonstration de l'impact de max_tokens # Réponse courte (50 tokens ~ 35 mots) response_courte = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Explique la gravité"}], max_tokens=50 ) # Réponse complète (500 tokens ~ 375 mots) response_complete = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Explique la gravité"}], max_tokens=500 ) # Vérifier si la réponse a été tronquée finish_reason = response_courte.choices[0].finish_reason print(f"Raison d'arrêt : {finish_reason}") # "length" = tronquée, "stop" = terminée naturellement
50 tokens : "La gravité est une force fondamentale de la nature qui attire les objets ayant une masse les uns vers les autres. Décrite par Isaac Newton puis raffinée par Albert Einstein dans sa théorie de la relativité" Raison d'arrêt : length (TRONQUÉE)

Estimation tokens vs mots

Langue Ratio moyen 100 tokens ≈
Anglais 1 token ≈ 0.75 mot ~75 mots
Français 1 token ≈ 0.60 mot ~60 mots
Code Python 1 token ≈ 0.40 mot/symbole ~3-4 lignes de code

En français, les mots sont souvent plus longs qu'en anglais, donc chaque mot consomme plus de tokens. Prévoyez environ 30% de tokens supplémentaires par rapport à l'anglais.

Le paramètre max_tokens ne contrôle que la sortie. Les tokens d'entrée (prompt) ne sont pas affectés. Le total (entrée + sortie) ne doit pas dépasser la fenêtre de contexte du modèle (128K pour GPT-4o, 200K pour Claude Sonnet 4.5).

5. Stop Sequences (Séquences d'arrêt)

Les stop sequences sont des chaînes de caractères qui, lorsqu'elles sont générées par le modèle, interrompent immédiatement la génération. La séquence d'arrêt elle-même n'est pas incluse dans la réponse.
# Utilisation de stop sequences # Exemple 1 : Arrêter après la première réponse dans un format Q&A response = client.chat.completions.create( model="gpt-4o", messages=[{ "role": "user", "content": """Réponds aux questions suivantes : Q1: Quelle est la capitale de la France ? A1:""" }], stop=["\nQ2", "\n\n"], # Arrêter avant Q2 ou double saut de ligne max_tokens=100 ) print(response.choices[0].message.content)
Paris est la capitale de la France.
# Exemple 2 : Générer uniquement du code (arrêter à l'explication) response = client.chat.completions.create( model="gpt-4o", messages=[{ "role": "user", "content": "Écris une fonction Python pour calculer le PGCD" }], stop=["Explication", "```\n\n", "Note:"], # Arrêter avant les explications max_tokens=200 ) # Exemple 3 : Anthropic - même concept, paramètre "stop_sequences" response_claude = client_anthropic.messages.create( model="claude-sonnet-4-5-20250514", max_tokens=200, stop_sequences=["---FIN---", "\n\nQuestion"], messages=[{ "role": "user", "content": "Décris Python en un paragraphe, puis écris ---FIN---" }] )
Les stop sequences sont particulièrement utiles pour le parsing structuré : si vous demandez au modèle de générer un JSON, utilisez stop=["}"] et ajoutez manuellement le "}" final pour éviter du texte supplémentaire après le JSON. OpenAI accepte jusqu'à 4 stop sequences.

6. frequency_penalty et presence_penalty

Ces deux paramètres modifient les probabilités des tokens pour réduire la répétition et encourager la diversité du vocabulaire.

frequency_penalty (pénalité de fréquence)

Plage : -2.0 à 2.0 (défaut : 0.0)

Pénalise les tokens proportionnellement à leur nombre d'apparitions dans le texte généré. Plus un token a été utilisé, plus sa probabilité future diminue.

  • 0.0 : Aucune pénalité (défaut)
  • 0.5 - 1.0 : Réduit les répétitions modérément
  • 1.5 - 2.0 : Forte pénalité, le modèle évite fortement de réutiliser des mots

presence_penalty (pénalité de présence)

Plage : -2.0 à 2.0 (défaut : 0.0)

Pénalise les tokens qui ont déjà apparu au moins une fois, indépendamment de leur fréquence. C'est un signal binaire : présent ou absent.

  • 0.0 : Aucune pénalité (défaut)
  • 0.5 - 1.0 : Encourage le modèle à introduire de nouveaux sujets
  • 1.5 - 2.0 : Pousse fortement vers de nouveaux thèmes à chaque phrase
# Comparaison sans et avec pénalités # Sans pénalité (tend à répéter) response_sans = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Écris un texte de 100 mots sur l'océan"}], frequency_penalty=0.0, presence_penalty=0.0, max_tokens=200 ) # Avec pénalités (vocabulaire plus riche) response_avec = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Écris un texte de 100 mots sur l'océan"}], frequency_penalty=0.8, # Réduit les répétitions de mots presence_penalty=0.6, # Encourage de nouveaux sujets max_tokens=200 )
Sans pénalité : "L'océan est vaste. L'océan est profond. L'océan abrite de nombreuses espèces. L'océan est essentiel pour l'océan..." Avec pénalité : "L'océan s'étend à perte de vue. Ses profondeurs abyssales recellent une biodiversité insoupçonnée. Des récifs coralliens aux fosses marines, chaque strate révèle un écosystème unique..."
La combinaison frequency_penalty=0.8 + presence_penalty=0.6 produit un texte plus riche et varié, évitant les répétitions tout en explorant différents aspects du sujet.
Ces paramètres ne sont pas disponibles dans l'API Anthropic (Claude). Anthropic utilise un mécanisme interne différent pour gérer la répétition. Si vous avez besoin de réduire la répétition avec Claude, utilisez des instructions explicites dans le prompt.

7. System message vs User message

Les APIs LLM utilisent un système de rôles pour différencier les types de messages :

# Conversation multi-tours avec rôles messages = [ # Rôle défini une seule fois {"role": "system", "content": "Tu es un tuteur Python. Réponds en français, avec des exemples de code. Niveau : débutant."}, # Tour 1 : question de l'utilisateur {"role": "user", "content": "C'est quoi une boucle for ?"}, # Tour 1 : réponse du modèle (historique) {"role": "assistant", "content": "Une boucle for permet de répéter une action pour chaque élément d'une collection..."}, # Tour 2 : question de suivi {"role": "user", "content": "Et une boucle while, c'est quoi la différence ?"} ] response = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.3 )
L'historique de conversation (messages user + assistant précédents) permet au modèle de maintenir le contexte. Mais attention : chaque message dans l'historique consomme des tokens d'entrée, donc des coûts. En production, il faut souvent tronquer l'historique pour les longues conversations.

8. Streaming - Réponses en temps réel

Le streaming permet de recevoir la réponse du modèle token par token (en temps réel) au lieu d'attendre la génération complète. Cela améliore considérablement l'expérience utilisateur en réduisant le temps d'attente perçu.

8.1 Streaming avec OpenAI

from openai import OpenAI client = OpenAI() # Activer le streaming avec stream=True stream = client.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": "Tu es un conteur."}, {"role": "user", "content": "Raconte une histoire courte sur un robot."} ], stream=True, # Active le streaming temperature=0.8, max_tokens=300 ) # Itérer sur les chunks reçus texte_complet = "" for chunk in stream: # Chaque chunk contient un delta (fragment de texte) delta = chunk.choices[0].delta if delta.content is not None: print(delta.content, end="", flush=True) # Affichage en temps réel texte_complet += delta.content print() # Saut de ligne final print(f"\nTexte complet ({len(texte_complet)} caractères) généré.")

8.2 Streaming avec Anthropic (Claude)

import anthropic client = anthropic.Anthropic() # Streaming avec l'API Anthropic with client.messages.stream( model="claude-sonnet-4-5-20250514", max_tokens=300, system="Tu es un conteur créatif.", messages=[{ "role": "user", "content": "Raconte une histoire courte sur un robot." }] ) as stream: for text in stream.text_stream: print(text, end="", flush=True) # Accéder aux statistiques après le streaming message = stream.get_final_message() print(f"\nTokens utilisés : {message.usage.input_tokens} (in) + {message.usage.output_tokens} (out)")
Il était une fois, dans un laboratoire oublié, un petit robot nommé Pixel. Contrairement à ses frères métalliques, Pixel avait une particularité : il rêvait. Chaque nuit, quand les lumières s'éteignaient... [texte apparaissant progressivement, mot par mot] Tokens utilisés : 35 (in) + 187 (out)
Le streaming est essentiel pour les applications interactives (chatbots, assistants). Sans streaming, l'utilisateur attend parfois 5-10 secondes sans feedback. Avec le streaming, le premier mot apparaît en ~200ms, créant une expérience fluide. Notez que le coût en tokens est identique avec ou sans streaming.

9. Gestion des coûts

Les APIs LLM facturent par token. Comprendre la structure de coûts est essentiel pour contrôler votre budget.

9.1 Tarification par modèle (mars 2026)

Modèle Input ($/1M tokens) Output ($/1M tokens) Contexte max
GPT-4o $2.50 $10.00 128K tokens
GPT-4o-mini $0.15 $0.60 128K tokens
Claude Opus 4.5 $15.00 $75.00 200K tokens
Claude Sonnet 4.5 $3.00 $15.00 200K tokens
Claude Haiku 4.5 $0.80 $4.00 200K tokens
Gemini 2.5 Pro $1.25 $10.00 1M tokens
Llama 4 (self-hosted) Coût GPU uniquement Coût GPU uniquement Variable
Les tokens d'output coûtent 3 à 5 fois plus cher que les tokens d'input. Réduire la longueur des réponses (via max_tokens et des prompts concis) est le levier d'économie le plus efficace.

9.2 Calcul de coût

def calculer_cout(input_tokens, output_tokens, modele="gpt-4o"): """Calcule le coût d'un appel API en dollars.""" # Tarifs par million de tokens (mars 2026) tarifs = { "gpt-4o": {"input": 2.50, "output": 10.00}, "gpt-4o-mini": {"input": 0.15, "output": 0.60}, "claude-sonnet-4.5": {"input": 3.00, "output": 15.00}, "claude-haiku-4.5": {"input": 0.80, "output": 4.00}, } tarif = tarifs[modele] cout_input = (input_tokens / 1_000_000) * tarif["input"] cout_output = (output_tokens / 1_000_000) * tarif["output"] cout_total = cout_input + cout_output return { "cout_input": round(cout_input, 6), "cout_output": round(cout_output, 6), "cout_total": round(cout_total, 6), } # Exemple : 500 tokens input, 200 tokens output avec GPT-4o cout = calculer_cout(500, 200, "gpt-4o") print(f"Coût total : ${cout['cout_total']:.6f}") print(f" - Input : ${cout['cout_input']:.6f}") print(f" - Output : ${cout['cout_output']:.6f}") # Estimation pour 10 000 requêtes/jour cout_journalier = cout["cout_total"] * 10000 print(f"\nCoût estimé pour 10K requêtes/jour : ${cout_journalier:.2f}")
Coût total : $0.003250 - Input : $0.001250 - Output : $0.002000 Coût estimé pour 10K requêtes/jour : $32.50

9.3 Stratégies d'optimisation des coûts

7 stratégies pour réduire vos coûts API

1. Choisir le bon modèle : Utilisez GPT-4o-mini ou Claude Haiku pour les tâches simples (classification, extraction). Réservez les modèles premium (GPT-4o, Claude Sonnet) pour le raisonnement complexe.
2. Limiter max_tokens : Fixez un max_tokens adapté à la tâche. Pour une classification, 10 tokens suffisent. Ne mettez pas 4096 par défaut.
3. Optimiser les prompts : Un prompt concis coûte moins cher. Éliminez les instructions redondantes. Chaque mot du prompt consomme des tokens d'input.
4. Utiliser le cache (Anthropic) : L'API Anthropic propose le "prompt caching" qui réduit le coût des tokens répétés à 10% du prix normal (cache read).
5. Batch API : Pour le traitement en masse (non temps réel), utilisez les APIs batch d'OpenAI et Anthropic pour obtenir 50% de réduction.
6. Tronquer l'historique : Dans les conversations longues, gardez uniquement les N derniers tours. Un historique de 50 messages consomme énormément de tokens d'input à chaque requête.
7. Mise en cache locale : Cachez les réponses fréquentes avec Redis ou un cache mémoire. Si 100 utilisateurs posent la même question, un seul appel API suffit.
# Exemple : Routeur de modèles pour optimiser les coûts def choisir_modele(tache, complexite): """Sélectionne le modèle le plus économique pour la tâche.""" # Tâches simples -> modèle économique if tache in ["classification", "extraction", "sentiment"]: return "gpt-4o-mini" # ~17x moins cher que GPT-4o # Tâches moyennes -> modèle intermédiaire if complexite == "moyenne": return "claude-sonnet-4-5-20250514" # Tâches complexes -> modèle premium if complexite == "haute" or tache in ["raisonnement", "code_complexe"]: return "gpt-4o" return "gpt-4o-mini" # Défaut économique # Utilisation modele = choisir_modele("classification", "basse") print(f"Modèle sélectionné : {modele}") # gpt-4o-mini
En production, les entreprises utilisent souvent un "routeur de modèles" qui sélectionne automatiquement le modèle le plus économique capable de traiter chaque requête. Cette approche peut réduire les coûts de 60-80% sans impact significatif sur la qualité.

Points clés à retenir

  1. Les APIs LLM sont des interfaces HTTP qui permettent d'envoyer des prompts et recevoir des réponses de modèles hébergés dans le cloud.
  2. Temperature contrôle la créativité : 0.0 = déterministe, 1.0+ = créatif.
  3. top_p filtre les tokens improbables. Ne pas modifier en même temps que la temperature.
  4. max_tokens limite la longueur de la réponse. Une réponse tronquée a un finish_reason: "length".
  5. Stop sequences permettent un arrêt conditionnel précis de la génération.
  6. Penalties réduisent la répétition et encouragent la diversité (OpenAI uniquement).
  7. Streaming améliore l'UX en affichant les tokens au fur et à mesure.
  8. Les coûts sont proportionnels aux tokens. Les tokens de sortie coûtent 3-5x plus cher que les tokens d'entrée. Optimisez avec le routage de modèles, le caching et les API batch.
En 2025-2026, l'écosystème des APIs LLM évolue rapidement. De nouvelles fonctionnalités apparaissent régulièrement : structured outputs, tool use / function calling, vision (images), audio, et même vidéo. Maîtriser les paramètres fondamentaux vus dans ce cours est le prérequis essentiel avant d'explorer ces fonctionnalités avancées.
