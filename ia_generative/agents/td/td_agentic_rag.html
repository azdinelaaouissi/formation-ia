TD Pratique - Pipeline RAG Agentique Complet

Agents IA - TD 4/5 : Implémenter un pipeline CRAG (Corrective RAG) avec grading, web search fallback et hallucination check via LangGraph

1. Récapitulatif : CRAG, Self-RAG, Adaptive RAG

Le RAG classique (Retrieve-then-Generate) a un problème fondamental : il ne vérifie jamais la qualité des documents récupérés ni la véracité de la réponse générée. Les approches agentiques corrigent ces faiblesses.

Le RAG Agentique désigne les architectures RAG qui intègrent des boucles de contrôle qualité : évaluation de la pertinence des documents récupérés, recherche web de secours, vérification des hallucinations, et régénération conditionnelle. Les variantes principales sont CRAG, Self-RAG et Adaptive RAG.

Comparaison des approches RAG

Approche Mécanisme Avantage
RAG Naif Retrieve -> Generate (sans vérification) Simple, rapide
CRAG Retrieve -> Grade -> Si mauvais: Web Search -> Generate Corrige les retrieval de mauvaise qualité
Self-RAG Retrieve -> Generate -> Grade hallucinations -> Si mauvais: Regenerate Détecte et corrige les hallucinations
Adaptive RAG Analyse la question -> Route vers RAG ou Web ou LLM direct Choisit la meilleure stratégie par question
Dans ce TD, nous combinons CRAG et Self-RAG dans un seul pipeline. C'est l'approche la plus robuste : on vérifie à la fois la qualité du retrieval ET la qualité de la génération.

Le pipeline que nous allons construire

  1. Retrieval : Chercher les documents pertinents dans ChromaDB
  2. Grading : Évaluer la pertinence de chaque document avec un LLM
  3. Web Search Fallback : Si les documents ne sont pas pertinents, chercher sur le web
  4. Generation : Générer la réponse à partir des documents validés
  5. Hallucination Check : Vérifier que la réponse est fidèle aux documents
  6. Output : Retourner la réponse validée ou régénérer
Le RAG agentique consomme plus de tokens qu'un RAG simple (3-5x) car chaque étape de vérification nécessite un appel LLM supplémentaire. C'est un compromis qualité vs coût.

Voici l'architecture détaillée de notre pipeline CRAG avec tous les noeuds et les edges conditionnels.

class CRAGState(TypedDict): """État du pipeline CRAG.""" question: str # Question de l'utilisateur documents: List[Document] # Documents récupérés filtered_documents: List[Document] # Documents après grading web_results: str # Résultats de recherche web generation: str # Réponse générée is_relevant: bool # Documents pertinents ? is_grounded: bool # Réponse fidèle aux sources ? is_useful: bool # Réponse utile ? retry_count: int # Compteur de retry
Le State contient toutes les informations nécessaires au pipeline : la question, les documents à chaque étape de filtrage, la réponse générée, et les flags de qualité.

5. Exercice : Implémenter le Grading Node

Le Grading Node évalue la pertinence de chaque document récupéré par rapport à la question. C'est le coeur du "Corrective" dans CRAG.

Étape 5.1 : Créer le grader de pertinence des documents.
GRADER_PROMPT = """Tu es un grader qui évalue si un document est pertinent pour répondre à une question. Question : {question} Document : {document} Évalue si ce document contient des informations utiles pour répondre à la question. Réponds UNIQUEMENT par "yes" ou "no". Un document est pertinent s'il contient des faits, données ou explications qui aident directement à répondre à la question, même partiellement.""" def grade_documents_node(state: CRAGState) -> dict: """Évalue la pertinence de chaque document récupéré.""" question = state["question"] documents = state["documents"] print(f" [Grader] Évaluation de {len(documents)} documents...") filtered_docs = [] for i, doc in enumerate(documents): prompt = GRADER_PROMPT.format( question=question, document=doc.page_content[:500] ) response = llm.invoke([HumanMessage(content=prompt)]) grade = response.content.strip().lower() is_relevant = "yes" in grade print(f" Doc {i+1}: {'PERTINENT' if is_relevant else 'NON PERTINENT'}") if is_relevant: filtered_docs.append(doc) # Décider si on a assez de documents pertinents has_relevant = len(filtered_docs) > 0 print(f" [Grader] {len(filtered_docs)}/{len(documents)} documents pertinents") print(f" [Grader] → {'Génération directe' if has_relevant else 'Fallback Web Search'}") return { "filtered_documents": filtered_docs, "is_relevant": has_relevant, }
Étape 5.2 : Créer la fonction de routing après le grading.
def route_after_grading(state: CRAGState) -> str: """Route vers generate ou web_search selon la pertinence des documents.""" if state["is_relevant"]: return "generate" else: return "web_search"
Le grader élimine les documents non pertinents. Si aucun document pertinent n'est trouvé, le pipeline bascule automatiquement sur la recherche web. C'est le mécanisme "correctif" de CRAG.

6. Exercice : Implémenter le Web Search Fallback

Quand les documents locaux ne sont pas pertinents, le pipeline bascule sur Tavily pour chercher l'information sur le web.

Étape 6.1 : Créer le noeud de recherche web de secours.
from tavily import TavilyClient tavily = TavilyClient(api_key=os.getenv("TAVILY_API_KEY")) def web_search_node(state: CRAGState) -> dict: """Recherche web quand les documents locaux ne sont pas pertinents.""" question = state["question"] print(f" [Web Search] Recherche web pour : '{question[:50]}...'") try: response = tavily.search( query=question, max_results=3, search_depth="advanced", include_answer=True ) # Formater les résultats web_results_parts = [] if response.get("answer"): web_results_parts.append(f"Réponse: {response['answer']}") for r in response["results"]: web_results_parts.append( f"Source: {r['title']} ({r['url']})\n{r['content'][:300]}" ) web_results = "\n\n".join(web_results_parts) # Convertir en documents pour la génération web_docs = [] for r in response["results"]: web_docs.append(Document( page_content=r["content"], metadata={"source": r["url"], "title": r["title"]} )) print(f" [Web Search] {len(web_docs)} résultats trouvés") return { "web_results": web_results, "filtered_documents": web_docs, } except Exception as e: print(f" [Web Search] Erreur : {str(e)}") return { "web_results": f"Erreur de recherche : {str(e)}", "filtered_documents": [], }
Le web search fallback est crucial pour la robustesse du système. Si la base de données ne contient pas l'information, le pipeline ne donne pas une mauvaise réponse : il va la chercher sur le web.

7. Exercice : Génération avec Hallucination Check

Le noeud de génération produit la réponse, puis le hallucination check vérifie que la réponse est fidèle aux documents sources.

Étape 7.1 : Créer le noeud de génération.
GENERATE_PROMPT = """Tu es un assistant qui répond aux questions en utilisant UNIQUEMENT les documents fournis comme contexte. ## Question : {question} ## Documents sources : {context} ## Instructions : 1. Réponds UNIQUEMENT à partir des informations des documents 2. Si les documents ne contiennent pas assez d'info, dis-le clairement 3. Cite les sources quand c'est pertinent 4. Sois précis et structuré dans ta réponse 5. Réponds en français""" def generate_node(state: CRAGState) -> dict: """Génère la réponse à partir des documents filtrés.""" question = state["question"] docs = state.get("filtered_documents", []) print(f" [Generate] Génération à partir de {len(docs)} documents") # Construire le contexte context_parts = [] for i, doc in enumerate(docs): source = doc.metadata.get("source", f"doc_{i+1}") context_parts.append( f"[Document {i+1} - Source: {source}]\n{doc.page_content}" ) context = "\n\n".join(context_parts) prompt = GENERATE_PROMPT.format( question=question, context=context ) response = llm.invoke([HumanMessage(content=prompt)]) generation = response.content print(f" [Generate] Réponse générée : {len(generation)} caractères") return {"generation": generation}
Étape 7.2 : Créer le noeud de vérification des hallucinations.
HALLUCINATION_PROMPT = """Tu es un grader qui évalue si une réponse est fidèle aux documents sources. ## Documents sources : {documents} ## Réponse à évaluer : {generation} ## Critères : - La réponse contient-elle des informations qui NE SONT PAS dans les documents ? - La réponse déforme-t-elle les informations des documents ? - La réponse invente-t-elle des chiffres ou des faits ? Réponds UNIQUEMENT par un JSON : {{"is_grounded": true/false, "explanation": "..."}}""" def hallucination_check_node(state: CRAGState) -> dict: """Vérifie que la réponse est fidèle aux documents sources.""" docs = state.get("filtered_documents", []) generation = state.get("generation", "") print(f" [Hallucination Check] Vérification de la réponse...") docs_text = "\n\n".join([d.page_content for d in docs]) prompt = HALLUCINATION_PROMPT.format( documents=docs_text[:3000], generation=generation ) response = llm.invoke([HumanMessage(content=prompt)]) import json try: content = response.content json_start = content.find("{") json_end = content.rfind("}") + 1 result = json.loads(content[json_start:json_end]) is_grounded = result.get("is_grounded", True) explanation = result.get("explanation", "") except: is_grounded = True explanation = "Parsing error, assume grounded" print(f" [Hallucination Check] Grounded: {is_grounded}") if not is_grounded: print(f" [Hallucination Check] Raison: {explanation}") return { "is_grounded": is_grounded, "retry_count": state.get("retry_count", 0) + 1, }
Étape 7.3 : Créer le noeud de vérification d'utilité et le routing.
USEFULNESS_PROMPT = """La réponse suivante répond-elle bien à la question posée ? Question : {question} Réponse : {generation} Réponds UNIQUEMENT par "yes" ou "no".""" def answer_useful_node(state: CRAGState) -> dict: """Vérifie que la réponse est utile pour la question.""" question = state["question"] generation = state.get("generation", "") prompt = USEFULNESS_PROMPT.format( question=question, generation=generation ) response = llm.invoke([HumanMessage(content=prompt)]) is_useful = "yes" in response.content.strip().lower() print(f" [Useful Check] Réponse utile: {is_useful}") return {"is_useful": is_useful} def route_after_hallucination(state: CRAGState) -> str: """Route selon le résultat du hallucination check.""" if state.get("retry_count", 0) > 2: return "end" # Max 2 retries if state.get("is_grounded", False): return "useful_check" else: return "regenerate" def route_after_useful(state: CRAGState) -> str: """Route selon l'utilité de la réponse.""" if state.get("is_useful", False): return "end" else: return "web_search" # Chercher plus d'info
Le retry_count est essentiel pour éviter les boucles infinies. Si après 2 tentatives la réponse n'est toujours pas fidèle aux sources, on retourne quand même le meilleur résultat avec un avertissement.

8. Exercice : Assembler le pipeline CRAG complet

Assemblons tous les noeuds dans un StateGraph LangGraph avec les edges conditionnels.

Étape 8.1 : Construire et compiler le graphe CRAG.
# Construire le graphe CRAG crag_workflow = StateGraph(CRAGState) # Ajouter les noeuds crag_workflow.add_node("retrieve", retrieve_node) crag_workflow.add_node("grade", grade_documents_node) crag_workflow.add_node("web_search", web_search_node) crag_workflow.add_node("generate", generate_node) crag_workflow.add_node("hallucination_check", hallucination_check_node) crag_workflow.add_node("useful_check", answer_useful_node) # Point d'entrée crag_workflow.set_entry_point("retrieve") # Edges fixes crag_workflow.add_edge("retrieve", "grade") crag_workflow.add_edge("web_search", "generate") crag_workflow.add_edge("generate", "hallucination_check") # Edges conditionnels crag_workflow.add_conditional_edges( "grade", route_after_grading, {"generate": "generate", "web_search": "web_search"} ) crag_workflow.add_conditional_edges( "hallucination_check", route_after_hallucination, {"useful_check": "useful_check", "regenerate": "generate", "end": END} ) crag_workflow.add_conditional_edges( "useful_check", route_after_useful, {"end": END, "web_search": "web_search"} ) # Compiler crag_graph = crag_workflow.compile() print("Pipeline CRAG compilé avec succès !")

9. Points clés à retenir

Résumé du TD - Pipeline RAG Agentique

  • CRAG ajoute un grading entre le retrieval et la génération. Les documents non pertinents sont éliminés et un web search prend le relais si nécessaire.
  • Le hallucination check vérifie que la réponse est fidèle aux documents sources. Si la réponse "invente" des informations, elle est régénérée.
  • Le web search fallback est le filet de sécurité : quand la base vectorielle ne contient pas l'info, le système va la chercher sur internet.
  • Le grading avec LLM est plus fiable qu'un simple seuil de similarité cosinus. Le LLM comprend le sens, pas juste la proximité vectorielle.
  • Chaque étape de vérification coûte un appel LLM supplémentaire. En production, optimisez avec des modèles plus petits pour le grading (GPT-4o-mini).
  • Les retry limits sont essentiels : max 2 régénérations pour éviter les boucles infinies et le gaspillage de tokens.
  • LangGraph est idéal pour ce type de pipeline : les conditional edges permettent de router dynamiquement selon la qualité du contenu.
  • En production, ajoutez un cache sur les résultats de grading et de recherche web pour réduire les coûts et la latence.

Quand utiliser quel type de RAG ?

Situation RAG recommandé
Données fiables, questions simples RAG Naif (retrieve + generate)
Base documentaire parfois incomplète CRAG (grading + web fallback)
Applications critiques (médical, juridique) Self-RAG (hallucination check)
Mix de questions factuelles et complexes Adaptive RAG (routing par type)
Production avec exigence qualité maximale CRAG + Self-RAG combiné (ce TD)
