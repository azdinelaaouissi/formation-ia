TD Pratique - Système Multi-Agents Collaboratifs

Agents IA - TD 3/5 : Concevoir un système où 3 agents spécialisés (Researcher, Writer, Reviewer) collaborent sous la supervision d'un Supervisor

1. Récapitulatif : Systèmes Multi-Agents

Jusqu'ici, nous avons travaillé avec un seul agent. Mais pour les tâches complexes, un seul agent atteint vite ses limites. Les systèmes multi-agents permettent de décomposer un problème en sous-tâches, chacune gérée par un agent spécialisé.

Un système multi-agents est une architecture où plusieurs agents IA autonomes collaborent pour résoudre un problème complexe. Chaque agent a un rôle spécifique, un ensemble d'outils dédié, et un système de communication pour échanger des informations avec les autres agents.

Patterns de communication multi-agents

Pattern Description Cas d'usage
Supervisor Un agent central coordonne les autres, décide qui travaille et quand Projets complexes avec workflow défini
Peer-to-peer Les agents communiquent directement entre eux sans hiérarchie Brainstorming, débats, consensus
Sequential Les agents travaillent en chaîne : la sortie de l'un est l'entrée du suivant Pipelines de production de contenu
Hierarchical Arbre de superviseurs et sous-agents avec délégation Organisations complexes, entreprises virtuelles
Dans ce TD, nous utilisons le pattern Supervisor car c'est le plus courant et le plus facile à contrôler. Le Supervisor décide quel agent doit travailler à chaque étape, et peut renvoyer le travail pour révision si le résultat n'est pas satisfaisant.

Notre système : 3 agents + 1 Supervisor

  • Researcher : Recherche d'informations sur le web, synthèse de données, collecte de faits
  • Writer : Rédaction structurée d'articles, rapports, synthèses
  • Reviewer : Critique du travail produit, scoring de qualité, suggestions d'amélioration
  • Supervisor : Orchestre le workflow, décide qui travaille, détecte quand le résultat est suffisant
Les systèmes multi-agents sont plus puissants mais aussi plus complexes à debugger. Chaque interaction agent-agent ajoute de la latence et du coût. Utilisez-les uniquement quand un seul agent ne suffit pas.

2. Architecture : Supervisor avec 3 Agents

Voici l'architecture de notre système multi-agents. Le Supervisor est au centre et orchestre le flux de travail entre les trois agents spécialisés.

Le workflow typique est : Supervisor envoie la tâche au Researcher, celui-ci collecte les informations. Le Supervisor envoie ensuite au Writer qui rédige. Puis le Reviewer critique le résultat. Si le score est insuffisant, le Supervisor peut renvoyer au Writer pour amélioration.

3. Exercice : Définir le State partagé et le Supervisor Node

Commençons par définir la structure de données partagée entre tous les agents et le noeud Supervisor qui orchestre le workflow.

Étape 3.1 : Installer les dépendances et créer les imports.
# Installation pip install langchain langgraph langchain-openai tavily-python # Imports import os import json from typing import Annotated, TypedDict, Literal from dotenv import load_dotenv from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage, AIMessage from langgraph.graph import StateGraph, END from langgraph.graph.message import add_messages load_dotenv() llm = ChatOpenAI(model="gpt-4o", temperature=0)
Étape 3.2 : Définir le State partagé entre tous les agents.
class MultiAgentState(TypedDict): """État partagé entre tous les agents du système.""" # Historique des messages (accumulé) messages: Annotated[list, add_messages] # Le sujet de recherche initial topic: str # Données collectées par le Researcher research_data: str # Article rédigé par le Writer draft: str # Feedback du Reviewer review_feedback: str review_score: int # Prochain agent à activer (décidé par le Supervisor) next_agent: str # Compteur d'itérations iteration: int
Le State centralise toutes les informations. Chaque agent lit ce dont il a besoin et écrit ses résultats. Le Supervisor utilise next_agent pour router vers le bon agent.
Étape 3.3 : Implémenter le Supervisor Node.
SUPERVISOR_PROMPT = """Tu es le Supervisor d'une équipe de 3 agents IA. Tu coordonnes leur travail pour produire un article de qualité. ## Tes agents : - **researcher** : Collecte des informations sur le sujet - **writer** : Rédige l'article à partir des recherches - **reviewer** : Évalue la qualité et donne un feedback ## Workflow standard : 1. researcher → collecte les données 2. writer → rédige un brouillon 3. reviewer → évalue et note (score /10) 4. Si score < 7, renvoie au writer avec le feedback 5. Si score >= 7, terminer (FINISH) ## État actuel : - Sujet: {topic} - Recherche effectuée: {has_research} - Brouillon rédigé: {has_draft} - Dernier score review: {review_score} - Feedback: {review_feedback} - Itération: {iteration} Réponds UNIQUEMENT avec le nom du prochain agent : researcher, writer, reviewer, ou FINISH.""" def supervisor_node(state: MultiAgentState) -> dict: """Le Supervisor décide quel agent doit travailler ensuite.""" # Préparer le contexte pour le Supervisor prompt = SUPERVISOR_PROMPT.format( topic=state.get("topic", "Non défini"), has_research="Oui" if state.get("research_data") else "Non", has_draft="Oui" if state.get("draft") else "Non", review_score=state.get("review_score", "Pas encore évalué"), review_feedback=state.get("review_feedback", "Aucun")[:200], iteration=state.get("iteration", 0), ) response = llm.invoke([ SystemMessage(content=prompt), HumanMessage(content="Quel agent doit travailler maintenant ?") ]) next_agent = response.content.strip().lower() # Sécurité : forcer FINISH après 5 itérations current_iter = state.get("iteration", 0) + 1 if current_iter > 5: next_agent = "finish" print(f" [Supervisor] Itération {current_iter} → Prochain: {next_agent}") return { "next_agent": next_agent, "iteration": current_iter, "messages": [AIMessage( content=f"[Supervisor] Itération {current_iter}: envoi à {next_agent}" )] }
Le compteur d'itérations et la limite à 5 sont essentiels. Sans cela, un cycle reviewer->writer->reviewer pourrait tourner indéfiniment si le score n'atteint jamais 7/10.

4. Exercice : Implémenter le Researcher Agent

Le Researcher est chargé de collecter des informations pertinentes sur le sujet demandé via la recherche web.

Étape 4.1 : Créer le Researcher Agent avec recherche web et synthèse.
from tavily import TavilyClient tavily = TavilyClient(api_key=os.getenv("TAVILY_API_KEY")) RESEARCHER_PROMPT = """Tu es un agent Researcher spécialisé dans la collecte d'informations. ## Ta mission : Recherche des informations complètes et fiables sur le sujet suivant : {topic} ## Informations de recherche web : {search_results} ## Instructions : 1. Synthétise les informations trouvées 2. Organise-les par thème/sous-sujet 3. Note les sources et les chiffres clés 4. Identifie les points qui nécessiteraient plus de recherche 5. Fournis un résumé structuré exploitable par un rédacteur Rédige ta synthèse de recherche en français, de manière factuelle et structurée.""" def researcher_node(state: MultiAgentState) -> dict: """Le Researcher collecte et synthétise les informations.""" topic = state["topic"] print(f" [Researcher] Recherche sur : {topic}") # Effectuer 2 recherches complémentaires search1 = tavily.search( query=topic, max_results=3, search_depth="advanced", include_answer=True ) search2 = tavily.search( query=f"{topic} statistiques chiffres données récentes", max_results=2, search_depth="basic" ) # Combiner les résultats all_results = [] if search1.get("answer"): all_results.append(f"Réponse directe: {search1['answer']}") for r in search1["results"] + search2["results"]: all_results.append( f"[{r['title']}] ({r['url']})\n{r['content'][:300]}" ) search_text = "\n\n---\n\n".join(all_results) # Synthèse par le LLM prompt = RESEARCHER_PROMPT.format( topic=topic, search_results=search_text ) response = llm.invoke([HumanMessage(content=prompt)]) research_data = response.content print(f" [Researcher] Synthèse produite : {len(research_data)} caractères") return { "research_data": research_data, "messages": [AIMessage( content=f"[Researcher] Recherche terminée. {len(all_results)} sources analysées." )] }
Le Researcher effectue deux recherches complémentaires (générale + données chiffrées), combine les résultats, puis utilise le LLM pour produire une synthèse structurée. Ce travail sera passé au Writer.

5. Exercice : Implémenter le Writer Agent

Le Writer transforme les données brutes du Researcher en un article structuré et bien rédigé.

Étape 5.1 : Créer le Writer Agent avec gestion du feedback.
WRITER_PROMPT = """Tu es un agent Writer spécialisé dans la rédaction d'articles. ## Sujet : {topic} ## Données de recherche : {research_data} {feedback_section} ## Instructions : 1. Rédige un article complet et structuré en français 2. Structure avec : Introduction, 2-3 sections principales, Conclusion 3. Utilise les données factuelles de la recherche 4. Style professionnel mais accessible 5. Inclus des chiffres et exemples concrets 6. L'article doit faire 400-600 mots Rédige l'article maintenant.""" WRITER_REVISION_SECTION = """ ## IMPORTANT - Feedback du Reviewer (score: {score}/10) : {feedback} Tu dois améliorer l'article en tenant compte de ce feedback. Voici le brouillon précédent à améliorer : --- {previous_draft} --- """ def writer_node(state: MultiAgentState) -> dict: """Le Writer rédige ou améliore l'article.""" topic = state["topic"] research = state.get("research_data", "Pas de données de recherche.") feedback = state.get("review_feedback", "") previous_draft = state.get("draft", "") score = state.get("review_score", 0) # Si on a du feedback, c'est une révision if feedback and previous_draft: print(f" [Writer] Révision de l'article (score précédent: {score}/10)") feedback_section = WRITER_REVISION_SECTION.format( score=score, feedback=feedback, previous_draft=previous_draft ) else: print(f" [Writer] Première rédaction sur : {topic}") feedback_section = "" prompt = WRITER_PROMPT.format( topic=topic, research_data=research[:3000], # Limiter la taille feedback_section=feedback_section ) response = llm.invoke([HumanMessage(content=prompt)]) draft = response.content print(f" [Writer] Article rédigé : {len(draft)} caractères, ~{len(draft.split())} mots") return { "draft": draft, "messages": [AIMessage( content=f"[Writer] Article {'révisé' if feedback else 'rédigé'}. {len(draft.split())} mots." )] }
Le Writer gère deux cas : la première rédaction (à partir des données brutes) et la révision (à partir du feedback du Reviewer). Cette dualité est essentielle pour le cycle d'amélioration itératif.

6. Exercice : Implémenter le Reviewer Agent

Le Reviewer est l'agent de contrôle qualité. Il évalue l'article, attribue un score, et fournit des suggestions d'amélioration concrètes.

Étape 6.1 : Créer le Reviewer Agent avec scoring structuré.
REVIEWER_PROMPT = """Tu es un agent Reviewer expert en évaluation de contenu. ## Sujet de l'article : {topic} ## Article à évaluer : {draft} ## Grille d'évaluation (note chaque critère sur 10) : 1. **Exactitude factuelle** : Les informations sont-elles correctes ? 2. **Complétude** : Le sujet est-il couvert de manière complète ? 3. **Structure** : L'article est-il bien organisé ? 4. **Clarté** : Le style est-il clair et accessible ? 5. **Sources** : Les données sont-elles sourcées et vérifiables ? ## Format de réponse OBLIGATOIRE (JSON) : {{ "scores": {{ "exactitude": X, "completude": X, "structure": X, "clarte": X, "sources": X }}, "score_global": X, "points_forts": ["...", "..."], "points_faibles": ["...", "..."], "suggestions": ["suggestion concrète 1", "suggestion concrète 2", "..."] }} Évalue l'article et retourne UNIQUEMENT le JSON.""" def reviewer_node(state: MultiAgentState) -> dict: """Le Reviewer évalue l'article et produit un feedback structuré.""" topic = state["topic"] draft = state.get("draft", "") if not draft: return { "review_feedback": "Aucun article à évaluer.", "review_score": 0, } print(f" [Reviewer] Évaluation de l'article ({len(draft.split())} mots)") prompt = REVIEWER_PROMPT.format( topic=topic, draft=draft ) response = llm.invoke([HumanMessage(content=prompt)]) # Parser le JSON de review try: # Extraire le JSON de la réponse content = response.content # Chercher le JSON dans la réponse json_start = content.find("{") json_end = content.rfind("}") + 1 json_str = content[json_start:json_end] review = json.loads(json_str) score = review.get("score_global", 5) feedback_parts = [] feedback_parts.append(f"Score global: {score}/10") feedback_parts.append(f"\nScores détaillés:") for critere, note in review.get("scores", {}).items(): feedback_parts.append(f" - {critere}: {note}/10") feedback_parts.append(f"\nPoints forts:") for p in review.get("points_forts", []): feedback_parts.append(f" + {p}") feedback_parts.append(f"\nÀ améliorer:") for p in review.get("suggestions", []): feedback_parts.append(f" → {p}") feedback = "\n".join(feedback_parts) except (json.JSONDecodeError, ValueError): score = 5 feedback = response.content print(f" [Reviewer] Score: {score}/10") return { "review_feedback": feedback, "review_score": score, "messages": [AIMessage( content=f"[Reviewer] Évaluation terminée. Score: {score}/10" )] }
Le Reviewer produit un feedback structuré avec des scores détaillés par critère. Si le score global est inférieur à 7, le Supervisor renverra l'article au Writer pour amélioration.

7. Exercice : Orchestration avec LangGraph

Assemblons maintenant tous les noeuds dans un StateGraph LangGraph avec le routing conditionnel du Supervisor.

Étape 7.1 : Définir la fonction de routing basée sur la décision du Supervisor.
def route_from_supervisor(state: MultiAgentState) -> str: """Route vers le bon agent selon la décision du Supervisor.""" next_agent = state.get("next_agent", "finish").strip().lower() if "researcher" in next_agent: return "researcher" elif "writer" in next_agent: return "writer" elif "reviewer" in next_agent: return "reviewer" else: return "end"
Étape 7.2 : Construire et compiler le graphe multi-agents.
# Construire le graphe workflow = StateGraph(MultiAgentState) # Ajouter les noeuds workflow.add_node("supervisor", supervisor_node) workflow.add_node("researcher", researcher_node) workflow.add_node("writer", writer_node) workflow.add_node("reviewer", reviewer_node) # Point d'entrée : toujours le Supervisor workflow.set_entry_point("supervisor") # Routing conditionnel depuis le Supervisor workflow.add_conditional_edges( "supervisor", route_from_supervisor, { "researcher": "researcher", "writer": "writer", "reviewer": "reviewer", "end": END, } ) # Après chaque agent, retour au Supervisor workflow.add_edge("researcher", "supervisor") workflow.add_edge("writer", "supervisor") workflow.add_edge("reviewer", "supervisor") # Compiler multi_agent_graph = workflow.compile() print("Graphe multi-agents compilé avec succès !")

9. Points clés à retenir

Résumé du TD - Système Multi-Agents

  • Le pattern Supervisor est le plus courant : un agent central coordonne les spécialistes, décide du workflow, et détecte la fin.
  • Le State partagé est le medium de communication entre agents. Chaque agent lit ce dont il a besoin et écrit ses résultats.
  • Spécialisation des agents : chaque agent a un rôle unique et un prompt spécialisé. Le Researcher ne rédige pas, le Writer ne critique pas.
  • Le cycle de feedback (Writer -> Reviewer -> Writer) permet une amélioration itérative de la qualité, comme dans une vraie rédaction humaine.
  • Toujours limiter les itérations : sans garde-fou, le cycle peut tourner indéfiniment. Fixez un max_iterations ET un score seuil.
  • Le scoring structuré (JSON avec critères multiples) donne un feedback actionnable au Writer, pas juste "c'est bien" ou "c'est mauvais".
  • LangGraph rend l'orchestration élégante : les noeuds sont des fonctions, les edges définissent le flux, et les conditional_edges gèrent le routing dynamique.
  • Le coût est multiplicatif : 3 agents x N itérations = beaucoup d'appels LLM. Optimisez les prompts et limitez les tokens pour contrôler les coûts.

Quand utiliser un système multi-agents ?

Situation Single Agent Multi-Agents
Question simple / lookup Oui Non (overkill)
Tâche multi-étapes avec outils Oui (ReAct) Possible
Production de contenu de qualité Résultat moyen Oui (feedback loop)
Analyse complexe multi-perspective Limité Oui (agents spécialisés)
Workflow d'entreprise Insuffisant Oui (hiérarchique)
