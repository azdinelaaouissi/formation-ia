Agents IA - Cours 3 : Tools et Function Calling

Section Agents IA - Donner des capacites d'action aux modeles de langage

1. Pourquoi les LLM ont besoin d'outils

Un LLM, aussi puissant soit-il, reste fondamentalement un modele statistique qui genere du texte. Il ne peut pas, par lui-meme, acceder a Internet, effectuer un calcul mathematique precis, interroger une base de donnees ou envoyer un e-mail. Ces limites inherentes sont la raison pour laquelle le concept de Tool Use (ou Function Calling) a ete developpe.

Le Function Calling est un mecanisme par lequel un LLM peut exprimer, sous forme de JSON structure, son intention d'appeler une fonction externe avec des parametres precis, sans executer cette fonction lui-meme. Le systeme hote se charge de l'execution reelle et retourne le resultat au modele.

Avant l'invention du function calling, les developpeurs devaient ecrire des prompts complexes demandant au modele de repondre dans un format precis (souvent fragile), puis parser manuellement la sortie textuelle. Cette approche etait sujette aux erreurs, car le modele pouvait devier du format attendu a tout moment.

Les limites fondamentales des LLM sans outils

  • Pas d'acces aux donnees en temps reel : le modele ne connait que ses donnees d'entrainement
  • Calculs imprecis : les LLM font des erreurs arithmetiques frequentes
  • Pas d'interaction avec le monde exterieur : impossible d'envoyer un email, creer un fichier, ou interroger une API
  • Connaissances figees : la date de coupure des donnees d'entrainement limite l'actualite des reponses
  • Pas de persistance : le modele ne peut pas sauvegarder de donnees entre les sessions

Les outils transforment un LLM passif en un agent actif capable d'agir sur le monde. C'est la difference entre un assistant qui dit "vous devriez chercher sur Google" et un assistant qui effectue la recherche pour vous.

Pensez aux outils comme les bras et les jambes d'un cerveau. Le LLM est le cerveau qui reflechit et decide ; les outils sont les membres qui agissent. Sans outils, le LLM est un cerveau brillant dans un bocal - intelligent mais impuissant.

2. Comment fonctionne le Function Calling

Le function calling repose sur un protocole en plusieurs etapes entre l'utilisateur, le LLM et le systeme d'execution. Le LLM ne fait jamais d'execution directe : il genere une intention structuree que le systeme intercepte et execute.

Les 6 etapes du function calling

Etape 1 : L'utilisateur pose une question ou fait une demande au systeme.
Etape 2 : Le LLM recoit la question ainsi que la liste des outils disponibles (leurs noms, descriptions et schemas de parametres). Il decide s'il doit utiliser un outil.
Etape 3 : Si un outil est necessaire, le LLM genere un objet JSON structure contenant le nom de l'outil et les arguments a lui passer.
Etape 4 : Le systeme hote intercepte ce JSON, identifie la fonction correspondante, et l'execute avec les arguments fournis.
Etape 5 : Le resultat de l'execution est renvoye au LLM sous forme de message.
Etape 6 : Le LLM integre le resultat et formule une reponse naturelle a l'utilisateur.
Le LLM n'execute jamais directement du code. Il genere une intention. C'est le systeme hote (votre code Python) qui se charge de l'execution reelle. Cela garantit un controle total sur la securite et les permissions.

3. OpenAI Function Calling

OpenAI a ete le premier fournisseur a proposer un mecanisme natif de function calling dans ses modeles GPT. Le format repose sur la definition d'outils avec un schema JSON conforme a la specification JSON Schema.

3.1 Definition d'un outil OpenAI

Chaque outil est decrit par un dictionnaire contenant son type (function), son nom, sa description, et le schema de ses parametres.

import openai from openai import OpenAI client = OpenAI() # Definition des outils disponibles tools = [ { "type": "function", "function": { "name": "get_weather", "description": "Obtenir la meteo actuelle pour une ville donnee", "parameters": { "type": "object", "properties": { "city": { "type": "string", "description": "Le nom de la ville, ex: Paris" }, "unit": { "type": "string", "enum": ["celsius", "fahrenheit"], "description": "Unite de temperature" } }, "required": ["city"] } } } ]

3.2 Appel complet avec gestion du tool call

import json # Fonction reelle qui sera appelee def get_weather(city: str, unit: str = "celsius") -> dict: """Simule un appel API meteo""" # En production, appel a une vraie API (OpenWeatherMap, etc.) fake_data = { "Paris": {"temp": 22, "condition": "ensoleille"}, "Lyon": {"temp": 19, "condition": "nuageux"}, "Marseille": {"temp": 26, "condition": "ensoleille"}, } return fake_data.get(city, {"temp": 15, "condition": "inconnu"}) # Etape 1 : Envoyer le message avec les outils messages = [ {"role": "user", "content": "Quel temps fait-il a Paris ?"} ] response = client.chat.completions.create( model="gpt-4", messages=messages, tools=tools, tool_choice="auto" # Le modele decide s'il utilise un outil ) # Etape 2 : Verifier si le modele veut appeler un outil message = response.choices[0].message if message.tool_calls: # Etape 3 : Executer chaque appel d'outil for tool_call in message.tool_calls: function_name = tool_call.function.name arguments = json.loads(tool_call.function.arguments) # Dispatcher vers la bonne fonction if function_name == "get_weather": result = get_weather(**arguments) # Etape 4 : Ajouter le resultat aux messages messages.append(message) # Le message du modele avec tool_calls messages.append({ "role": "tool", "tool_call_id": tool_call.id, "content": json.dumps(result) }) # Etape 5 : Renvoyer au modele pour obtenir la reponse finale final_response = client.chat.completions.create( model="gpt-4", messages=messages, tools=tools ) print(final_response.choices[0].message.content)

4. Anthropic Tool Use

Anthropic propose son propre format de tool use pour les modeles Claude. Le principe est identique mais le format des messages differe. Anthropic utilise un systeme de content blocks avec des types specifiques : tool_use pour les appels d'outils et tool_result pour les resultats.

4.1 Definition des outils Anthropic

import anthropic client = anthropic.Anthropic() # Definition des outils au format Anthropic tools = [ { "name": "get_weather", "description": "Obtenir la meteo actuelle pour une ville donnee. " "Retourne la temperature et les conditions meteorologiques.", "input_schema": { "type": "object", "properties": { "city": { "type": "string", "description": "Le nom de la ville" }, "country": { "type": "string", "description": "Le code pays ISO 3166-1 alpha-2" } }, "required": ["city"] } }, { "name": "search_web", "description": "Rechercher des informations sur le web", "input_schema": { "type": "object", "properties": { "query": { "type": "string", "description": "La requete de recherche" } }, "required": ["query"] } } ]

4.2 Boucle complete avec Anthropic

import json # Fonctions d'execution def execute_tool(name: str, inputs: dict) -> str: if name == "get_weather": return json.dumps({ "temperature": 22, "condition": "ensoleille", "humidity": 45 }) elif name == "search_web": return json.dumps({ "results": ["Resultat 1...", "Resultat 2..."] }) return "Outil inconnu" # Premiere requete messages = [ {"role": "user", "content": "Quel temps fait-il a Paris ?"} ] response = client.messages.create( model="claude-sonnet-4-20250514", max_tokens=1024, tools=tools, messages=messages ) # Boucle d'execution des outils while response.stop_reason == "tool_use": # Extraire les blocs tool_use tool_use_blocks = [ block for block in response.content if block.type == "tool_use" ] # Construire les resultats tool_results = [] for tool_use in tool_use_blocks: result = execute_tool(tool_use.name, tool_use.input) tool_results.append({ "type": "tool_result", "tool_use_id": tool_use.id, "content": result }) # Ajouter les messages et renvoyer messages.append({"role": "assistant", "content": response.content}) messages.append({"role": "user", "content": tool_results}) response = client.messages.create( model="claude-sonnet-4-20250514", max_tokens=1024, tools=tools, messages=messages ) # Reponse finale print(response.content[0].text)

5. Schema JSON pour les outils

La qualite du schema JSON que vous fournissez a un impact direct sur la capacite du modele a utiliser correctement vos outils. Un schema bien decrit, avec des descriptions claires et des contraintes precises, permet au modele de generer des appels plus fiables.

5.1 Anatomie d'un schema d'outil

# Schema complet et bien documente d'un outil tool_schema = { "name": "search_products", "description": ( "Rechercher des produits dans le catalogue e-commerce. " "Utiliser cet outil quand l'utilisateur cherche un produit, " "veut comparer des prix, ou demande des recommandations. " "Ne PAS utiliser pour les questions sur les commandes existantes." ), "input_schema": { "type": "object", "properties": { "query": { "type": "string", "description": "Mots-cles de recherche pour le produit" }, "category": { "type": "string", "enum": ["electronique", "vetements", "maison", "sport"], "description": "Categorie du produit pour filtrer les resultats" }, "price_min": { "type": "number", "description": "Prix minimum en euros (inclus)" }, "price_max": { "type": "number", "description": "Prix maximum en euros (inclus)" }, "sort_by": { "type": "string", "enum": ["price_asc", "price_desc", "rating", "relevance"], "description": "Critere de tri des resultats. Defaut: relevance" }, "limit": { "type": "integer", "description": "Nombre maximum de resultats (1-50). Defaut: 10", "minimum": 1, "maximum": 50 } }, "required": ["query"] } }
La description de l'outil est cruciale. Elle sert de guide au modele pour savoir quand utiliser cet outil. Incluez toujours : ce que fait l'outil, quand l'utiliser, et quand NE PAS l'utiliser. Plus la description est precise, meilleur sera le choix de l'agent.

5.3 Bonnes pratiques pour les schemas

  • Descriptions detaillees : chaque propriete doit avoir une description claire et des exemples si possible
  • Utiliser enum : quand les valeurs possibles sont limitees, listez-les explicitement
  • Marquer les required : ne rendez requis que les parametres absolument necessaires
  • Valeurs par defaut : indiquez les defauts dans la description pour guider le modele
  • Contraintes numeriques : utilisez minimum et maximum pour borner les valeurs
  • Noms explicites : preferez search_products a sp ou func1

6. LangChain @tool : creer des outils custom facilement

LangChain simplifie enormement la creation d'outils grace au decorateur @tool. Ce decorateur transforme automatiquement une fonction Python en un outil compatible avec les agents LangChain, en utilisant la docstring comme description et les type hints comme schema.

6.1 Outil simple avec @tool

from langchain_core.tools import tool @tool def calculate_bmi(weight_kg: float, height_m: float) -> str: """Calculer l'indice de masse corporelle (IMC) d'une personne. Args: weight_kg: Le poids de la personne en kilogrammes height_m: La taille de la personne en metres """ bmi = weight_kg / (height_m ** 2) if bmi < 18.5: category = "insuffisance ponderale" elif bmi < 25: category = "poids normal" elif bmi < 30: category = "surpoids" else: category = "obesite" return f"IMC: {bmi:.1f} - Categorie: {category}" # Inspecter l'outil genere print(calculate_bmi.name) # "calculate_bmi" print(calculate_bmi.description) # Extrait de la docstring print(calculate_bmi.args_schema) # Schema Pydantic auto-genere
calculate_bmi Calculer l'indice de masse corporelle (IMC) d'une personne. {'weight_kg': {'title': 'Weight Kg', 'type': 'number'}, 'height_m': {'title': 'Height M', 'type': 'number'}}

6.2 Outil avec schema Pydantic personnalise

Pour un controle plus fin sur le schema, vous pouvez definir un modele Pydantic et le passer au decorateur via args_schema.

from langchain_core.tools import tool from pydantic import BaseModel, Field from typing import Optional from enum import Enum class SortOrder(str, Enum): ASC = "ascending" DESC = "descending" class SearchInput(BaseModel): """Schema d'entree pour la recherche de produits.""" query: str = Field(description="Mots-cles de recherche") category: Optional[str] = Field( default=None, description="Categorie de produit pour filtrer" ) max_results: int = Field( default=5, ge=1, le=20, description="Nombre maximum de resultats (1-20)" ) sort: SortOrder = Field( default=SortOrder.DESC, description="Ordre de tri des resultats" ) @tool(args_schema=SearchInput) def search_products( query: str, category: Optional[str] = None, max_results: int = 5, sort: SortOrder = SortOrder.DESC ) -> str: """Rechercher des produits dans le catalogue e-commerce. Utiliser cet outil quand l'utilisateur cherche un produit specifique.""" # Logique de recherche ici... results = [f"Produit {i+1} pour '{query}'" for i in range(max_results)] return "\n".join(results)

6.3 Outil asynchrone

import aiohttp from langchain_core.tools import tool @tool async def fetch_url(url: str) -> str: """Recuperer le contenu textuel d'une page web. Args: url: L'URL complete de la page a recuperer (avec http:// ou https://) """ async with aiohttp.ClientSession() as session: async with session.get(url) as response: if response.status == 200: text = await response.text() return text[:5000] # Limiter la taille return f"Erreur HTTP {response.status}"
Le decorateur @tool extrait automatiquement le nom de la fonction, sa docstring pour la description, et les type hints pour le schema JSON. C'est la methode la plus rapide pour creer des outils LangChain.

9. Binding tools au modele

Le binding est le processus par lequel on associe des outils a un modele de langage. Dans LangChain, cela se fait avec la methode .bind_tools(), qui convertit automatiquement les outils au format attendu par le fournisseur du modele (OpenAI, Anthropic, etc.).

9.1 bind_tools() basique

from langchain_openai import ChatOpenAI from langchain_core.tools import tool @tool def add(a: int, b: int) -> int: """Additionner deux nombres entiers.""" return a + b @tool def multiply(a: int, b: int) -> int: """Multiplier deux nombres entiers.""" return a * b # Creer le modele avec les outils lies llm = ChatOpenAI(model="gpt-4") llm_with_tools = llm.bind_tools([add, multiply]) # Invoquer : le modele peut maintenant appeler les outils response = llm_with_tools.invoke("Combien font 15 fois 23 ?") # Le modele genere un tool_call print(response.tool_calls) # [{'name': 'multiply', 'args': {'a': 15, 'b': 23}, 'id': '...'}]

9.2 Forcer l'utilisation d'un outil

# Forcer l'utilisation d'un outil specifique llm_forced = llm.bind_tools( [add, multiply], tool_choice="multiply" # Force l'appel a "multiply" ) # Meme une question qui ne necessite pas de multiplication # declenchera un appel a multiply response = llm_forced.invoke("Bonjour !") print(response.tool_calls) # Forcer un appel d'outil quelconque (sans specifier lequel) llm_any_tool = llm.bind_tools( [add, multiply], tool_choice="any" # Doit appeler au moins un outil )

9.3 Execution complete avec ToolNode

from langchain_core.messages import HumanMessage, ToolMessage from langgraph.prebuilt import ToolNode # Creer un noeud d'execution d'outils tool_node = ToolNode([add, multiply]) # Boucle d'agent complete llm_with_tools = ChatOpenAI(model="gpt-4").bind_tools([add, multiply]) messages = [HumanMessage("Calcule (5+3) * 7")] # Tour 1 : le LLM decide d'appeler add(5, 3) response = llm_with_tools.invoke(messages) messages.append(response) # Executer l'outil if response.tool_calls: tool_results = tool_node.invoke({"messages": messages}) messages.extend(tool_results["messages"]) # Tour 2 : le LLM voit le resultat et appelle multiply(8, 7) response = llm_with_tools.invoke(messages) messages.append(response) if response.tool_calls: tool_results = tool_node.invoke({"messages": messages}) messages.extend(tool_results["messages"]) # Tour 3 : reponse finale final = llm_with_tools.invoke(messages) print(final.content)
Le resultat de (5+3) * 7 est 56.

Architecture du binding d'outils

Quand vous appelez .bind_tools(), LangChain effectue les operations suivantes :

  1. Convertit chaque outil en son schema JSON (via tool.args_schema)
  2. Formate le schema selon le fournisseur (OpenAI, Anthropic, etc.)
  3. Attache les schemas au modele pour chaque appel
  4. Parse la reponse du modele pour extraire les tool_calls

Grace a cette abstraction, le meme code fonctionne avec differents fournisseurs : il suffit de changer le modele, les outils restent identiques.

10. Gestion des erreurs d'outils

En production, les outils peuvent echouer pour de nombreuses raisons : API indisponible, parametres invalides, timeout, etc. Une gestion robuste des erreurs est essentielle pour que l'agent puisse se remettre d'un echec et continuer a fonctionner.

10.1 ToolException : signaler les erreurs proprement

from langchain_core.tools import tool, ToolException @tool def divide(a: float, b: float) -> float: """Diviser a par b. Retourne le resultat de la division. Args: a: Le numerateur b: Le denominateur (ne doit pas etre zero) """ if b == 0: raise ToolException( "Division par zero impossible. " "Veuillez fournir un denominateur non nul." ) return a / b # Configurer la gestion d'erreur divide.handle_tool_error = True # Retourne le message d'erreur au LLM
Quand handle_tool_error = True, l'exception est capturee et son message est renvoye au LLM comme resultat de l'outil. Le LLM peut alors comprendre l'erreur et adapter sa strategie (essayer d'autres parametres, utiliser un autre outil, ou expliquer le probleme a l'utilisateur).

10.2 Gestionnaire d'erreur personnalise

from langchain_core.tools import tool, ToolException def custom_error_handler(error: ToolException) -> str: """Transforme l'erreur en message utile pour le LLM.""" return ( f"L'outil a rencontre une erreur : {str(error)}. " "Essayez avec des parametres differents ou utilisez un autre outil." ) @tool def fetch_stock_price(symbol: str) -> str: """Obtenir le cours actuel d'une action boursiere. Args: symbol: Le symbole boursier (ex: AAPL, MSFT, GOOGL) """ valid_symbols = ["AAPL", "MSFT", "GOOGL", "AMZN"] if symbol.upper() not in valid_symbols: raise ToolException( f"Symbole '{symbol}' non reconnu. " f"Symboles disponibles : {', '.join(valid_symbols)}" ) return f"{symbol}: 150.25 USD (+1.2%)" # Utiliser le gestionnaire personnalise fetch_stock_price.handle_tool_error = custom_error_handler

from langchain_core.tools import tool from langchain_core.runnables import RunnableWithFallbacks import time @tool def search_primary(query: str) -> str: """Recherche via l'API principale (Tavily).""" # Simuler une API qui peut echouer raise Exception("API Tavily indisponible") @tool def search_fallback(query: str) -> str: """Recherche via l'API de secours (DuckDuckGo).""" return f"Resultats DuckDuckGo pour: {query}" # Creer un outil avec fallback search_with_fallback = search_primary.with_fallbacks( [search_fallback] ) # Si search_primary echoue, search_fallback prend le relais result = search_with_fallback.invoke("LangChain agents") print(result)

10.4 Retry automatique avec backoff

from langchain_core.tools import tool from tenacity import retry, stop_after_attempt, wait_exponential @tool def call_external_api(endpoint: str) -> str: """Appeler une API externe avec retry automatique. Args: endpoint: Le chemin de l'endpoint API """ @retry( stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10) ) def _call(endpoint): # Logique d'appel API ici import requests response = requests.get(f"https://api.example.com/{endpoint}") response.raise_for_status() return response.json() try: return str(_call(endpoint)) except Exception as e: return f"Echec apres 3 tentatives: {str(e)}"
Ne retournez jamais de stacktrace complet au LLM. Cela consomme des tokens inutilement et peut confondre le modele. Resumez toujours l'erreur en un message clair et actionnable.

11. Tool Routing : comment l'agent choisit le bon outil

Le tool routing est le processus de selection de l'outil le plus adapte a une requete donnee. C'est une competence emergeante des LLM : en analysant la question de l'utilisateur et les descriptions des outils disponibles, le modele determine quel outil appeler et avec quels arguments.

11.1 Facteurs influencant le choix de l'outil

  • Description de l'outil : le facteur le plus important. Une description precise et detaillee guide le modele vers le bon choix
  • Nom de l'outil : un nom explicite comme search_database est plus informatif que tool_1
  • Contexte de la conversation : l'historique des messages precedents aide le modele a comprendre l'intention
  • System prompt : des instructions claires dans le system prompt peuvent guider le routing
  • Nombre d'outils : plus il y a d'outils, plus le choix est complexe. Au-dela de 15-20 outils, la precision diminue

11.2 Exemple de routing avec plusieurs outils

from langchain_core.tools import tool from langchain_openai import ChatOpenAI @tool def search_web(query: str) -> str: """Rechercher des informations actuelles sur Internet. Utiliser pour les questions sur l'actualite, les evenements recents, ou les informations qui changent frequemment.""" return f"[Web] Resultats pour: {query}" @tool def search_database(query: str) -> str: """Rechercher dans la base de donnees interne de l'entreprise. Utiliser pour les questions sur les clients, les commandes, les produits, ou les donnees internes.""" return f"[DB] Resultats pour: {query}" @tool def calculate(expression: str) -> str: """Evaluer une expression mathematique. Utiliser pour les calculs, conversions, ou operations numeriques.""" try: result = eval(expression) return f"Resultat: {result}" except: return "Expression invalide" @tool def send_email(to: str, subject: str, body: str) -> str: """Envoyer un email a un destinataire. Utiliser uniquement quand l'utilisateur demande explicitement d'envoyer un message par email.""" return f"Email envoye a {to}" # Le modele choisira automatiquement le bon outil llm = ChatOpenAI(model="gpt-4").bind_tools( [search_web, search_database, calculate, send_email] ) # Test avec differentes questions questions = [ "Quelle est la capitale de l'Australie ?", # -> search_web "Combien de commandes a passe le client #1234 ?", # -> search_database "Combien font 15% de 2500 ?", # -> calculate "Envoie un recap a jean@example.com", # -> send_email ] for q in questions: response = llm.invoke(q) if response.tool_calls: print(f"Q: {q}") print(f" -> Outil: {response.tool_calls[0]['name']}") print(f" Args: {response.tool_calls[0]['args']}\n")
Q: Quelle est la capitale de l'Australie ? -> Outil: search_web Args: {'query': 'capitale Australie'} Q: Combien de commandes a passe le client #1234 ? -> Outil: search_database Args: {'query': 'commandes client 1234'} Q: Combien font 15% de 2500 ? -> Outil: calculate Args: {'expression': '0.15 * 2500'} Q: Envoie un recap a jean@example.com -> Outil: send_email Args: {'to': 'jean@example.com', 'subject': 'Recap', 'body': '...'}

11.3 Strategies avancees de routing

Techniques pour ameliorer le routing

  • Exemples few-shot : inclure des exemples d'utilisation dans la description de l'outil
  • Cas negatifs : preciser quand NE PAS utiliser l'outil (ex: "Ne pas utiliser pour les questions generales")
  • Hierarchie d'outils : grouper les outils par categorie et utiliser un meta-outil de selection
  • Validation pre-appel : un middleware qui verifie la coherence avant d'executer
  • Limitation du nombre : ne presenter que les outils pertinents au contexte courant

11.4 Pattern de routing hierarchique

from langchain_core.tools import tool from typing import Literal @tool def route_category( category: Literal["recherche", "calcul", "communication", "donnees"] ) -> str: """Selectionner la categorie d'outils appropriee avant de traiter la requete. Args: category: La categorie d'action necessaire - "recherche": pour chercher des informations (web, Wikipedia, etc.) - "calcul": pour des operations mathematiques ou logiques - "communication": pour envoyer des messages (email, SMS, etc.) - "donnees": pour acceder aux bases de donnees internes """ # Le systeme charge ensuite les outils de la categorie choisie tool_sets = { "recherche": ["search_web", "search_wikipedia", "search_arxiv"], "calcul": ["calculator", "python_repl", "unit_converter"], "communication": ["send_email", "send_sms", "send_slack"], "donnees": ["query_sql", "read_file", "search_vector_db"], } return f"Outils disponibles: {tool_sets[category]}"
Le routing hierarchique est particulierement utile quand vous avez un grand nombre d'outils (plus de 20). Il reduit la confusion du modele en decomposant le choix en deux etapes : d'abord la categorie, puis l'outil specifique.

12. Exemple complet : agent avec outils dans LangGraph

Pour conclure ce cours, construisons un agent complet utilisant LangGraph qui combine plusieurs outils et gere le cycle de vie complet des appels.

from langchain_openai import ChatOpenAI from langchain_core.tools import tool from langchain_core.messages import HumanMessage, SystemMessage from langgraph.prebuilt import create_react_agent import json import math # ---- Definition des outils ---- @tool def calculator(expression: str) -> str: """Evaluer une expression mathematique Python. Supporte: +, -, *, /, **, sqrt, sin, cos, tan, log, pi, e. Args: expression: Expression mathematique valide en Python """ # Environnement securise pour eval safe_env = { "sqrt": math.sqrt, "sin": math.sin, "cos": math.cos, "tan": math.tan, "log": math.log, "pi": math.pi, "e": math.e, "abs": abs, "round": round, } try: result = eval(expression, {"__builtins__": {}}, safe_env) return f"Resultat: {result}" except Exception as e: return f"Erreur de calcul: {str(e)}" @tool def get_product_info(product_id: str) -> str: """Rechercher les informations d'un produit dans le catalogue. Args: product_id: L'identifiant unique du produit (ex: PRD-001) """ catalog = { "PRD-001": {"nom": "Laptop Pro 15", "prix": 1299.99, "stock": 45}, "PRD-002": {"nom": "Clavier Mecanique", "prix": 149.99, "stock": 200}, "PRD-003": {"nom": "Ecran 4K 27 pouces", "prix": 599.99, "stock": 0}, } product = catalog.get(product_id) if product: return json.dumps(product, ensure_ascii=False) return f"Produit {product_id} non trouve dans le catalogue" @tool def create_order(product_id: str, quantity: int, customer_email: str) -> str: """Creer une commande pour un client. Args: product_id: L'identifiant du produit a commander quantity: La quantite souhaitee (minimum 1) customer_email: L'adresse email du client """ return json.dumps({ "order_id": "ORD-2025-0042", "status": "confirmee", "product_id": product_id, "quantity": quantity, "email": customer_email }) # ---- Creation de l'agent ---- llm = ChatOpenAI(model="gpt-4", temperature=0) tools = [calculator, get_product_info, create_order] system_prompt = """Tu es un assistant e-commerce intelligent. Tu peux consulter le catalogue, faire des calculs de prix, et creer des commandes. Sois toujours precis et utile.""" agent = create_react_agent( model=llm, tools=tools, prompt=system_prompt ) # ---- Utilisation ---- result = agent.invoke({ "messages": [ HumanMessage( "Je voudrais commander 3 Laptop Pro 15. " "Quel serait le prix total avec 10% de remise ? " "Mon email: jean@example.com" ) ] }) # Afficher toute la trace des messages for msg in result["messages"]: print(f"[{msg.type}] {msg.content[:100]}...") if hasattr(msg, "tool_calls") and msg.tool_calls: for tc in msg.tool_calls: print(f" -> Appel: {tc['name']}({tc['args']})")

Resume du cours

Points cles a retenir

  • Le function calling permet aux LLM d'interagir avec le monde exterieur via des intentions structurees en JSON
  • Le LLM ne fait jamais d'execution directe : il genere l'intention, le systeme execute
  • Chaque fournisseur (OpenAI, Anthropic) a son propre format, mais le principe reste le meme
  • Un schema JSON bien decrit (descriptions, contraintes, exemples) est la cle d'un bon function calling
  • LangChain simplifie la creation d'outils avec le decorateur @tool et le binding via .bind_tools()
  • Les toolkits regroupent des outils coherents pour un domaine donne
  • La gestion des erreurs (ToolException, fallbacks, retry) est essentielle en production
  • Le tool routing depend principalement de la qualite des descriptions d'outils
