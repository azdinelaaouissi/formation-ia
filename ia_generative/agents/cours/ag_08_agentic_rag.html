Cours 8 : RAG Agentique

Quand l'agent prend le controle du processus de retrieval

1. Limites du RAG Naif

Le RAG (Retrieval-Augmented Generation) classique fonctionne selon un pipeline rigide : l'utilisateur pose une question, le systeme recupere des documents, et le LLM genere une reponse. Ce processus lineaire presente des limites importantes.

Le RAG naif (ou Naive RAG) est le pipeline standard question -> retrieval -> generation, sans aucune intelligence dans le processus de recuperation. Chaque question declenche automatiquement une recherche vectorielle, sans evaluation de la pertinence des resultats.

1.1 Les problemes du RAG naif

Probleme 1 : Pas de raisonnement sur la recherche

Le RAG naif cherche toujours, meme quand ce n'est pas necessaire. Si l'utilisateur demande "Bonjour, comment ca va ?", le systeme va quand meme interroger la base vectorielle. Il ne sait pas distinguer les questions qui necessitent une recherche de celles qui n'en ont pas besoin.

Probleme 2 : Retrieval aveugle

Les documents recuperes ne sont jamais evalues. Le systeme fait confiance au score de similarite vectorielle, mais un document peut etre semantiquement proche sans etre pertinent pour la question. Il n'y a aucun mecanisme de grading (evaluation de la pertinence).

Probleme 3 : Pas de self-correction

Si la reponse generee est de mauvaise qualite ou contient des hallucinations, le RAG naif ne s'en rend pas compte. Il n'y a aucune boucle de feedback : la premiere reponse est la reponse finale, qu'elle soit bonne ou mauvaise.

Probleme 4 : Source unique

Le RAG naif interroge une seule source (typiquement un vector store). Mais certaines questions necessitent des donnees provenant de multiples sources : base vectorielle, recherche web, base SQL, API externes.

Le RAG naif fonctionne bien pour les cas simples : questions factuelles sur un corpus bien defini. Mais des que les questions deviennent complexes, ambigues ou multi-sources, il montre ses limites. C'est la que le RAG agentique entre en jeu.

2. CRAG : Corrective RAG

Le Corrective RAG (CRAG) ajoute une etape d'evaluation apres le retrieval. Un LLM "grade" chaque document recupere pour determiner s'il est pertinent. Si les documents ne sont pas pertinents, le systeme tente une recherche web comme fallback.

2.1 Principe du CRAG

Le CRAG introduit trois innovations majeures par rapport au RAG naif :

  1. Document Grading : chaque document recupere est evalue par un LLM qui decide s'il est pertinent ("yes") ou non ("no") par rapport a la question.
  2. Web Search Fallback : si aucun document n'est pertinent, le systeme effectue une recherche web pour trouver des informations complementaires.
  3. Knowledge Refinement : les documents non pertinents sont filtres avant la generation, ne gardant que les passages utiles.

2.2 Workflow CRAG

2.3 Implementation CRAG avec LangGraph

# === CRAG COMPLET AVEC LANGGRAPH === from typing import TypedDict, List, Literal from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI, OpenAIEmbeddings from langchain_community.vectorstores import Chroma from langchain_community.tools import TavilySearchResults from langchain_core.messages import HumanMessage, SystemMessage from langchain_core.documents import Document # --- Etat du CRAG --- class CRAGState(TypedDict): question: str documents: List[Document] relevant_docs: List[Document] web_results: str generation: str need_web_search: bool # --- Initialisation --- llm = ChatOpenAI(model="gpt-4", temperature=0) embeddings = OpenAIEmbeddings() vectorstore = Chroma(embedding_function=embeddings, persist_directory="./chroma_db") retriever = vectorstore.as_retriever(search_kwargs={"k": 4}) web_search = TavilySearchResults(max_results=3) # --- Noeud 1 : Retrieval --- def retrieve(state: CRAGState) -> dict: """Recupere les documents depuis le vector store.""" question = state["question"] documents = retriever.invoke(question) return {"documents": documents} # --- Noeud 2 : Grade Documents --- def grade_documents(state: CRAGState) -> dict: """Evalue la pertinence de chaque document.""" question = state["question"] documents = state["documents"] relevant_docs = [] need_web = False for doc in documents: response = llm.invoke([ SystemMessage(content="""Tu es un evaluateur de pertinence. Evalue si le document est pertinent pour repondre a la question. Reponds UNIQUEMENT par 'yes' ou 'no'."""), HumanMessage(content=f"""Question : {question} Document : {doc.page_content} Le document est-il pertinent pour repondre a la question ?""") ]) grade = response.content.strip().lower() if grade == "yes": relevant_docs.append(doc) # Si moins de 2 docs pertinents, declencher web search if len(relevant_docs) < 2: need_web = True return { "relevant_docs": relevant_docs, "need_web_search": need_web } # --- Noeud 3 : Web Search (fallback) --- def web_search_node(state: CRAGState) -> dict: """Recherche web quand les documents locaux sont insuffisants.""" question = state["question"] results = web_search.invoke(question) web_content = "\n\n".join([r["content"] for r in results]) return {"web_results": web_content} # --- Noeud 4 : Generation --- def generate(state: CRAGState) -> dict: """Genere la reponse finale.""" question = state["question"] docs = state.get("relevant_docs", []) web = state.get("web_results", "") context = "\n\n".join([d.page_content for d in docs]) if web: context += f"\n\nResultats web complementaires :\n{web}" response = llm.invoke([ SystemMessage(content="""Genere une reponse precise et complete en te basant UNIQUEMENT sur le contexte fourni. Si tu ne peux pas repondre, dis-le clairement."""), HumanMessage(content=f"Question : {question}\n\nContexte :\n{context}") ]) return {"generation": response.content} # --- Routage conditionnel --- def route_after_grading(state: CRAGState) -> Literal["web_search", "generate"]: if state.get("need_web_search", False): return "web_search" return "generate" # --- Construction du graphe --- workflow = StateGraph(CRAGState) workflow.add_node("retrieve", retrieve) workflow.add_node("grade_documents", grade_documents) workflow.add_node("web_search", web_search_node) workflow.add_node("generate", generate) workflow.set_entry_point("retrieve") workflow.add_edge("retrieve", "grade_documents") workflow.add_conditional_edges("grade_documents", route_after_grading) workflow.add_edge("web_search", "generate") workflow.add_edge("generate", END) app = workflow.compile() # Execution result = app.invoke({"question": "Quels sont les avantages du RAG agentique ?"}) print(result["generation"])
Le CRAG ameliore significativement la qualite des reponses en filtrant les documents non pertinents et en utilisant la recherche web comme filet de securite. C'est la premiere etape vers un RAG intelligent.

3. Self-RAG : L'Agent qui se Critique

Le Self-RAG va plus loin que le CRAG en ajoutant une capacite d'auto-critique. L'agent decide quand chercher, evalue la pertinence des documents, genere une reponse, puis critique sa propre reponse. Si la reponse n'est pas satisfaisante, il relance le processus.

3.1 Les tokens de reflexion du Self-RAG

Le Self-RAG original utilise des "tokens de reflexion" speciaux que le modele genere pour guider le processus :

Token Valeurs possibles Role
Retrieve yes / no Faut-il chercher des documents ?
ISREL relevant / irrelevant Le document est-il pertinent ?
ISSUP fully / partially / no support La reponse est-elle supportee par les documents ?
ISUSE 1 a 5 La reponse est-elle utile pour l'utilisateur ?

3.3 Implementation Self-RAG

# === SELF-RAG AVEC LANGGRAPH === from typing import TypedDict, List, Literal from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage class SelfRAGState(TypedDict): question: str documents: list generation: str need_retrieval: bool is_relevant: bool is_supported: bool is_useful: bool iteration: int llm = ChatOpenAI(model="gpt-4", temperature=0) # --- Decide si retrieval est necessaire --- def decide_retrieval(state: SelfRAGState) -> dict: question = state["question"] response = llm.invoke([ SystemMessage(content="""Determine si cette question necessite une recherche dans une base de connaissances. Reponds UNIQUEMENT par 'yes' ou 'no'. - Questions factuelles specifiques -> yes - Salutations, questions generales -> no"""), HumanMessage(content=question) ]) need = response.content.strip().lower() == "yes" return {"need_retrieval": need} # --- Retrieval --- def retrieve_docs(state: SelfRAGState) -> dict: question = state["question"] # Simulation de retrieval docs = retriever.invoke(question) return {"documents": docs} # --- Evaluation de la pertinence --- def check_relevance(state: SelfRAGState) -> dict: question = state["question"] docs = state["documents"] context = "\n".join([d.page_content for d in docs]) response = llm.invoke([ SystemMessage(content="""Les documents fournis sont-ils pertinents pour repondre a la question ? Reponds par 'yes' ou 'no'."""), HumanMessage(content=f"Question: {question}\nDocuments: {context}") ]) return {"is_relevant": response.content.strip().lower() == "yes"} # --- Generation --- def generate_answer(state: SelfRAGState) -> dict: question = state["question"] docs = state.get("documents", []) context = "\n".join([d.page_content for d in docs]) if docs else "Aucun contexte." response = llm.invoke([ SystemMessage(content="Reponds a la question en te basant sur le contexte."), HumanMessage(content=f"Question: {question}\nContexte: {context}") ]) return {"generation": response.content} # --- Auto-critique --- def self_critique(state: SelfRAGState) -> dict: question = state["question"] generation = state["generation"] docs = state.get("documents", []) context = "\n".join([d.page_content for d in docs]) # Verifier si la reponse est supportee par les documents sup_response = llm.invoke([ SystemMessage(content="""La reponse est-elle supportee par les documents ? Reponds par 'yes' ou 'no'."""), HumanMessage(content=f"Reponse: {generation}\nDocuments: {context}") ]) # Verifier si la reponse est utile use_response = llm.invoke([ SystemMessage(content="""La reponse repond-elle correctement et completement a la question ? Reponds par 'yes' ou 'no'."""), HumanMessage(content=f"Question: {question}\nReponse: {generation}") ]) return { "is_supported": sup_response.content.strip().lower() == "yes", "is_useful": use_response.content.strip().lower() == "yes", "iteration": state.get("iteration", 0) + 1 } # --- Routages --- def route_retrieval(state) -> Literal["retrieve", "generate"]: return "retrieve" if state["need_retrieval"] else "generate" def route_after_critique(state) -> Literal["retrieve", "end"]: if state["is_supported"] and state["is_useful"]: return "end" if state.get("iteration", 0) >= 2: return "end" return "retrieve" # Retenter avec une nouvelle recherche # --- Graphe --- graph = StateGraph(SelfRAGState) graph.add_node("decide", decide_retrieval) graph.add_node("retrieve", retrieve_docs) graph.add_node("check_relevance", check_relevance) graph.add_node("generate", generate_answer) graph.add_node("critique", self_critique) graph.set_entry_point("decide") graph.add_conditional_edges("decide", route_retrieval) graph.add_edge("retrieve", "check_relevance") graph.add_edge("check_relevance", "generate") graph.add_edge("generate", "critique") graph.add_conditional_edges("critique", route_after_critique, {"retrieve": "retrieve", "end": END}) app = graph.compile()

4. Adaptive RAG : Routing Dynamique

L'Adaptive RAG combine le CRAG et le Self-RAG en ajoutant un routeur intelligent en entree. Avant toute action, le systeme classifie la question pour choisir la strategie optimale : pas de retrieval, RAG simple, ou RAG agentique complet.

4.1 Les trois strategies

Strategie A : No Retrieval

Pour les questions simples, les salutations, ou les connaissances generales que le LLM maitrise deja. Pas besoin de chercher dans la base de donnees.

Exemple : "Qu'est-ce que le machine learning ?"

Strategie B : Simple RAG

Pour les questions factuelles directes ou une seule recherche vectorielle suffit. Pipeline classique retrieve -> generate.

Exemple : "Quel est le prix du produit X dans notre catalogue ?"

Strategie C : Agentic RAG

Pour les questions complexes qui necessitent plusieurs sources, du raisonnement, ou de l'auto-correction. Pipeline complet avec grading, web search et self-critique.

Exemple : "Compare les performances de notre produit X avec les concurrents et recommande des ameliorations"

4.2 Le Routeur de Requetes

# === ROUTEUR ADAPTATIF === def query_router(state) -> dict: """Classifie la question pour choisir la strategie RAG.""" question = state["question"] response = llm.invoke([ SystemMessage(content="""Tu es un routeur de requetes intelligent. Classifie la question dans l'une de ces 3 categories : - "no_retrieval" : salutations, questions generales, connaissances communes que tu maitrises deja - "simple_rag" : questions factuelles directes necessitant une recherche dans la base de donnees - "agentic_rag" : questions complexes necessitant plusieurs sources, du raisonnement, ou une analyse approfondie Reponds UNIQUEMENT par le nom de la categorie."""), HumanMessage(content=question) ]) route = response.content.strip().lower() if route not in ["no_retrieval", "simple_rag", "agentic_rag"]: route = "simple_rag" # Defaut securise return {"route": route} # Routage conditionnel def adaptive_route(state) -> Literal["direct_llm", "simple_rag", "agentic_rag"]: route = state["route"] if route == "no_retrieval": return "direct_llm" elif route == "simple_rag": return "simple_rag" else: return "agentic_rag"

4.3 Implementation Adaptive RAG complete

# === ADAPTIVE RAG COMPLET === from typing import TypedDict, Literal from langgraph.graph import StateGraph, END class AdaptiveState(TypedDict): question: str route: str documents: list relevant_docs: list web_results: str generation: str is_quality_ok: bool iteration: int # --- Routeur --- def router_node(state: AdaptiveState) -> dict: question = state["question"] response = llm.invoke([ SystemMessage(content="""Classifie la question : - "no_retrieval" : question generale, salutation - "simple_rag" : question factuelle simple - "agentic_rag" : question complexe, multi-source Reponds UNIQUEMENT par le nom de la categorie."""), HumanMessage(content=question) ]) return {"route": response.content.strip().lower()} # --- LLM Direct (pas de retrieval) --- def direct_llm_node(state: AdaptiveState) -> dict: response = llm.invoke([ HumanMessage(content=state["question"]) ]) return {"generation": response.content} # --- Simple RAG --- def simple_rag_node(state: AdaptiveState) -> dict: docs = retriever.invoke(state["question"]) context = "\n".join([d.page_content for d in docs]) response = llm.invoke([ SystemMessage(content="Reponds en te basant sur le contexte fourni."), HumanMessage(content=f"Question: {state['question']}\nContexte: {context}") ]) return {"generation": response.content, "documents": docs} # --- Agentic RAG (retrieve + grade + generate + critique) --- def agentic_retrieve(state: AdaptiveState) -> dict: docs = retriever.invoke(state["question"]) return {"documents": docs} def agentic_grade(state: AdaptiveState) -> dict: relevant = [] for doc in state["documents"]: resp = llm.invoke([ SystemMessage(content="Ce document est-il pertinent ? Reponds 'yes' ou 'no'."), HumanMessage(content=f"Q: {state['question']}\nDoc: {doc.page_content}") ]) if resp.content.strip().lower() == "yes": relevant.append(doc) return {"relevant_docs": relevant} def agentic_generate(state: AdaptiveState) -> dict: docs = state.get("relevant_docs", []) web = state.get("web_results", "") context = "\n".join([d.page_content for d in docs]) if web: context += f"\nWeb: {web}" response = llm.invoke([ SystemMessage(content="Reponds avec precision en te basant sur le contexte."), HumanMessage(content=f"Q: {state['question']}\nContexte: {context}") ]) return {"generation": response.content} def agentic_critique(state: AdaptiveState) -> dict: resp = llm.invoke([ SystemMessage(content="La reponse est-elle complete et correcte ? Reponds 'yes' ou 'no'."), HumanMessage(content=f"Q: {state['question']}\nR: {state['generation']}") ]) ok = resp.content.strip().lower() == "yes" return {"is_quality_ok": ok, "iteration": state.get("iteration", 0) + 1} # --- Construction du graphe --- def adaptive_route(state) -> Literal["direct_llm", "simple_rag", "agentic_retrieve"]: r = state["route"] if r == "no_retrieval": return "direct_llm" if r == "simple_rag": return "simple_rag" return "agentic_retrieve" def critique_route(state) -> Literal["agentic_retrieve", "end"]: if state["is_quality_ok"] or state.get("iteration", 0) >= 2: return "end" return "agentic_retrieve" wf = StateGraph(AdaptiveState) wf.add_node("router", router_node) wf.add_node("direct_llm", direct_llm_node) wf.add_node("simple_rag", simple_rag_node) wf.add_node("agentic_retrieve", agentic_retrieve) wf.add_node("agentic_grade", agentic_grade) wf.add_node("agentic_generate", agentic_generate) wf.add_node("agentic_critique", agentic_critique) wf.set_entry_point("router") wf.add_conditional_edges("router", adaptive_route) wf.add_edge("direct_llm", END) wf.add_edge("simple_rag", END) wf.add_edge("agentic_retrieve", "agentic_grade") wf.add_edge("agentic_grade", "agentic_generate") wf.add_edge("agentic_generate", "agentic_critique") wf.add_conditional_edges("agentic_critique", critique_route, {"agentic_retrieve": "agentic_retrieve", "end": END}) app = wf.compile()

5. Router Query : Classifier la Question

Le routeur de requetes est le composant cle de l'Adaptive RAG. Il existe plusieurs approches pour implementer un routeur performant.

5.1 Routeur base sur le LLM (prompt engineering)

L'approche la plus simple : on demande directement au LLM de classifier la question. C'est ce que nous avons vu dans la section precedente.

5.2 Routeur base sur un classificateur entraine

# Routeur avec un classificateur zero-shot from transformers import pipeline classifier = pipeline("zero-shot-classification", model="facebook/bart-large-mnli") def classify_query(question: str) -> str: """Classifie la question avec un modele zero-shot.""" categories = [ "question generale ne necessitant pas de recherche", "question factuelle simple necessitant une recherche", "question complexe necessitant une analyse approfondie" ] result = classifier(question, categories) top_label = result["labels"][0] mapping = { categories[0]: "no_retrieval", categories[1]: "simple_rag", categories[2]: "agentic_rag" } return mapping[top_label] # Utilisation route = classify_query("Quels sont les KPI de vente du Q3 compares au Q2 ?") print(route) # -> "agentic_rag"

5.3 Routeur semantique (embeddings)

# Routeur base sur la similarite semantique import numpy as np from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings() # Exemples de reference pour chaque categorie route_examples = { "no_retrieval": [ "Bonjour, comment ca va ?", "Qu'est-ce que le machine learning ?", "Explique-moi le concept de recursivite" ], "simple_rag": [ "Quel est le prix du produit A ?", "Quelle est la politique de retour ?", "Comment contacter le support ?" ], "agentic_rag": [ "Compare les ventes Q1 et Q2 et propose des recommandations", "Analyse les retours clients et identifie les tendances", "Fais une etude de marche sur nos concurrents" ] } # Pre-calculer les embeddings de reference route_embeddings = {} for route, examples in route_examples.items(): vecs = embeddings.embed_documents(examples) route_embeddings[route] = np.mean(vecs, axis=0) def semantic_router(question: str) -> str: q_vec = embeddings.embed_query(question) similarities = {} for route, ref_vec in route_embeddings.items(): sim = np.dot(q_vec, ref_vec) / ( np.linalg.norm(q_vec) * np.linalg.norm(ref_vec) ) similarities[route] = sim return max(similarities, key=similarities.get)
Le routeur semantique est plus rapide et moins couteux que le routeur LLM car il n'effectue qu'un calcul d'embedding au lieu d'un appel LLM complet. En production, combinez les deux : routeur semantique en premier filtre, routeur LLM pour les cas ambigus.

6. Agent qui Decide Quand Chercher

Une approche elegante consiste a donner le retrieval comme un outil a l'agent. L'agent decide lui-meme quand et comment chercher, comme il utiliserait n'importe quel autre outil.

# Agent avec retrieval comme outil from langchain_core.tools import tool from langchain_openai import ChatOpenAI from langgraph.prebuilt import create_react_agent @tool def search_knowledge_base(query: str) -> str: """Recherche dans la base de connaissances interne. Utilise cet outil pour les questions specifiques sur nos produits, nos politiques ou nos donnees internes.""" docs = retriever.invoke(query) return "\n\n".join([d.page_content for d in docs]) @tool def search_web(query: str) -> str: """Recherche sur le web pour des informations recentes. Utilise cet outil pour les questions sur l'actualite ou les sujets non couverts par la base interne.""" results = TavilySearchResults(max_results=3).invoke(query) return "\n".join([r["content"] for r in results]) @tool def query_database(sql_query: str) -> str: """Execute une requete SQL sur la base de donnees des ventes. Utilise pour les questions necessitant des donnees chiffrees.""" # Connection a la BDD et execution return "Resultats de la requete SQL..." # L'agent decide quand et quel outil utiliser llm = ChatOpenAI(model="gpt-4") tools = [search_knowledge_base, search_web, query_database] agent = create_react_agent( model=llm, tools=tools, state_modifier="""Tu es un assistant intelligent. Tu as acces a plusieurs sources d'information. Choisis la source la plus appropriee pour chaque question. Tu peux aussi repondre directement si tu connais la reponse.""" ) result = agent.invoke({"messages": [("user", "Quelles sont nos ventes du mois dernier ?")]})
Avec cette approche, l'agent a un controle total sur le processus de retrieval. Il peut choisir de ne pas chercher, de combiner plusieurs sources, ou de reformuler sa requete si les premiers resultats sont insuffisants.

8. Comparaison : Naive RAG vs CRAG vs Self-RAG vs Adaptive RAG

Critere Naive RAG CRAG Self-RAG Adaptive RAG
Retrieval Toujours Toujours + grading Conditionnel Route dynamique
Grading docs Non Oui Oui Oui (mode agentic)
Web fallback Non Oui Non (natif) Oui
Self-critique Non Non Oui Oui (mode agentic)
Routing Non Non Partiel Oui (3 strategies)
Qualite Basse-Moyenne Moyenne-Haute Haute Tres haute
Cout Faible Moyen (grading) Eleve (multi-LLM calls) Variable (adaptatif)
Latence Faible Moyenne Elevee Variable
Complexite impl. Tres faible Moyenne Elevee Elevee

Quand utiliser quelle approche ?

  • Naive RAG : prototype rapide, corpus simple et bien defini, questions factuelles directes
  • CRAG : quand la qualite des documents recuperes est variable et qu'un fallback web est utile
  • Self-RAG : quand la fiabilite de la reponse est critique (domaine medical, juridique, financier)
  • Adaptive RAG : applications en production avec des types de questions tres varies
Le RAG agentique multiplie les appels LLM (grading, routing, critique). Un workflow Adaptive RAG complet peut necessiter 5 a 10 appels LLM par question, contre 1 seul pour le Naive RAG. Surveillez attentivement les couts et la latence en production.

Resume du cours

Dans ce cours, nous avons parcouru l'evolution du RAG, du pipeline naif au RAG agentique :

  • Les limites du RAG naif : pas de raisonnement, retrieval aveugle, pas de self-correction
  • CRAG : document grading + web search fallback
  • Self-RAG : l'agent decide quand chercher et critique sa propre reponse
  • Adaptive RAG : routing dynamique entre 3 strategies selon la complexite de la question
  • Query routing : LLM, classificateur zero-shot, ou routeur semantique
  • Multi-source retrieval : combiner vector store, web, SQL, APIs
