Cours 10 : Deployer des Agents en Production

Du prototype fonctionnel a l'agent robuste, scalable et observable

1. Du Prototype a la Production : Les Defis

Vous avez construit un agent qui fonctionne parfaitement dans votre notebook Jupyter. Il repond correctement, utilise ses outils avec precision, et votre demo impressionne tout le monde. Maintenant, il faut le deployer en production. C'est la que les vrais defis commencent.

Les differences fondamentales entre prototype et production

Aspect Prototype Production
Utilisateurs Vous seul, requetes controlees Centaines/milliers d'utilisateurs imprevisibles
Disponibilite Acceptable si ca plante 99.9% uptime requis
Couts Quelques dollars de test Budget a maitriser au centime pres
Latence 30 secondes ? Pas grave L'utilisateur s'impatiente apres 5s
Erreurs Stacktrace dans le terminal Message d'erreur user-friendly + alerte equipe
Securite API key en dur Secrets Manager, authentification, guardrails
Observabilite print() partout Traces structurees, metriques, dashboards

1.1 Les risques specifiques aux agents en production

  • Non-determinisme : le meme input peut produire des outputs differents a chaque execution
  • Couts imprevisibles : un agent qui boucle peut consommer des centaines de dollars en minutes
  • Latence variable : un agent peut repondre en 2s ou en 30s selon la complexite
  • Dependances externes : APIs LLM, outils externes, bases de donnees... chaque dependance est un point de panne
  • Prompt injection : les utilisateurs malveillants testeront les limites de votre agent

3. Streaming : Server-Sent Events (SSE)

Les agents IA sont lents par nature : ils effectuent plusieurs appels LLM, utilisent des outils, raisonnent. Sans streaming, l'utilisateur attend 10-30 secondes sans aucun retour, ce qui est inacceptable pour l'experience utilisateur.

Le streaming SSE (Server-Sent Events) permet d'envoyer des donnees du serveur vers le client de maniere incrementale. L'utilisateur voit la reponse se construire en temps reel, mot par mot, etape par etape.

3.1 Types de streaming pour un agent

Type Ce qu'on streame Interet
Token streaming Les tokens du LLM un par un L'utilisateur voit la reponse s'ecrire en direct
Event streaming Les evenements de l'agent (tool call, reasoning step) L'utilisateur voit ce que l'agent fait
Hybrid streaming Les deux combines Experience optimale mais plus complexe

3.2 Implementation SSE avec FastAPI

# Endpoint de streaming SSE from fastapi.responses import StreamingResponse import json import asyncio @app.post("/agent/stream") async def stream_agent(request: AgentRequest): """Endpoint SSE pour streamer les reponses de l'agent.""" async def event_generator(): session_id = request.session_id or str(uuid.uuid4()) # Streamer les evenements de l'agent async for event in agent.astream_events( {"messages": [("user", request.message)]}, version="v2" ): kind = event["event"] # Evenement : debut d'appel outil if kind == "on_tool_start": data = { "type": "tool_start", "tool": event["name"], "input": str(event.get("data", {}).get("input", "")) } yield f"data: {json.dumps(data)}\n\n" # Evenement : resultat d'outil elif kind == "on_tool_end": data = { "type": "tool_end", "tool": event["name"], "output": str(event.get("data", {}).get("output", ""))[:200] } yield f"data: {json.dumps(data)}\n\n" # Evenement : tokens du LLM elif kind == "on_chat_model_stream": chunk = event.get("data", {}).get("chunk") if chunk and chunk.content: data = {"type": "token", "content": chunk.content} yield f"data: {json.dumps(data)}\n\n" # Signal de fin yield f"data: {json.dumps({'type': 'done'})}\n\n" return StreamingResponse( event_generator(), media_type="text/event-stream", headers={ "Cache-Control": "no-cache", "Connection": "keep-alive" } )

4. Observabilite : LangSmith

LangSmith est la plateforme d'observabilite de LangChain. Elle capture automatiquement les traces d'execution de vos agents : chaque appel LLM, chaque outil utilise, chaque decision de routage. C'est l'equivalent de Datadog/New Relic pour les applications LLM.

4.1 Configuration de LangSmith

# Configuration via variables d'environnement import os os.environ["LANGCHAIN_TRACING_V2"] = "true" os.environ["LANGCHAIN_API_KEY"] = "lsv2_..." os.environ["LANGCHAIN_PROJECT"] = "agent-production" os.environ["LANGCHAIN_ENDPOINT"] = "https://api.smith.langchain.com" # C'est tout ! LangSmith capture automatiquement toutes les traces # de vos appels LangChain/LangGraph

4.2 Ce que LangSmith capture

Informations tracees automatiquement

  • Runs : chaque execution de l'agent, avec entree et sortie completes
  • Spans : chaque etape interne (appel LLM, appel outil, routage)
  • Latence : temps de chaque etape et temps total
  • Tokens : nombre de tokens en entree et sortie pour chaque appel LLM
  • Cout : cout estime de chaque execution
  • Erreurs : stacktrace complete en cas d'erreur
  • Arbre d'execution : visualisation hierarchique de toutes les etapes

4.3 Feedback utilisateur avec LangSmith

# Envoyer du feedback utilisateur a LangSmith from langsmith import Client ls_client = Client() # Apres l'execution de l'agent, enregistrer le feedback def record_feedback(run_id: str, score: float, comment: str = ""): """Enregistre le feedback utilisateur pour une execution.""" ls_client.create_feedback( run_id=run_id, key="user_satisfaction", score=score, # 0.0 a 1.0 comment=comment ) # Endpoint FastAPI pour recevoir le feedback @app.post("/feedback") async def submit_feedback(run_id: str, thumbs_up: bool, comment: str = ""): score = 1.0 if thumbs_up else 0.0 record_feedback(run_id, score, comment) return {"status": "feedback recorded"}
LangSmith est gratuit jusqu'a 5000 traces par mois, ce qui suffit largement pour le developpement et les petits projets. Pour la production a grande echelle, les plans payants commencent a 39$/mois.

7. Retry et Fallback : Strategies de Resilience

Les agents dependent de services externes (APIs LLM, outils, bases de donnees) qui peuvent echouer. Une strategie de resilience robuste est essentielle.

7.1 Exponential Backoff

L'exponential backoff est une strategie de retry ou le delai entre les tentatives augmente de maniere exponentielle : 1s, 2s, 4s, 8s, 16s... Cela evite de surcharger un service deja en difficulte.
# Retry avec exponential backoff import asyncio import random from functools import wraps def retry_with_backoff(max_retries=3, base_delay=1.0, max_delay=60.0, exceptions=(Exception,)): """Decorateur de retry avec exponential backoff et jitter.""" def decorator(func): @wraps(func) async def wrapper(*args, **kwargs): for attempt in range(max_retries + 1): try: return await func(*args, **kwargs) except exceptions as e: if attempt == max_retries: raise # Calcul du delai avec jitter delay = min(base_delay * (2 ** attempt), max_delay) jitter = random.uniform(0, delay * 0.1) wait_time = delay + jitter logger.warning( f"Tentative {attempt + 1}/{max_retries} echouee: {e}. " f"Retry dans {wait_time:.1f}s" ) await asyncio.sleep(wait_time) return wrapper return decorator # Utilisation @retry_with_backoff(max_retries=3, base_delay=1.0) async def call_llm(prompt): return await llm.ainvoke(prompt)

7.2 Model Fallback

# Fallback entre modeles LLM from langchain_openai import ChatOpenAI from langchain_anthropic import ChatAnthropic class LLMWithFallback: def __init__(self): self.models = [ {"name": "gpt-4", "llm": ChatOpenAI(model="gpt-4")}, {"name": "gpt-3.5-turbo", "llm": ChatOpenAI(model="gpt-3.5-turbo")}, {"name": "claude-3-haiku", "llm": ChatAnthropic(model="claude-3-haiku-20240307")}, ] async def invoke(self, messages) -> str: """Essaie chaque modele en sequence jusqu'a ce qu'un fonctionne.""" last_error = None for model_config in self.models: try: logger.info(f"Essai du modele: {model_config['name']}") result = await model_config["llm"].ainvoke(messages) return result except Exception as e: last_error = e logger.warning( f"Modele {model_config['name']} echoue: {e}. Fallback..." ) raise Exception(f"Tous les modeles ont echoue. Derniere erreur: {last_error}") # Ou plus simplement avec LangChain from langchain_core.runnables import RunnableWithFallbacks llm_with_fallback = ChatOpenAI(model="gpt-4").with_fallbacks([ ChatOpenAI(model="gpt-3.5-turbo"), ChatAnthropic(model="claude-3-haiku-20240307") ])
Le model fallback est une strategie puissante pour maintenir la disponibilite. Si GPT-4 est en panne, l'agent bascule automatiquement sur GPT-3.5, puis sur Claude. La qualite peut baisser, mais le service reste disponible.

8. Gestion des Couts

Les couts des API LLM peuvent exploser rapidement. Un agent non maitrise peut consommer des centaines de dollars par jour. La gestion des couts est une responsabilite de production critique.

8.1 Token Counting

# Comptage precis des tokens import tiktoken class CostTracker: # Prix par 1000 tokens (USD) - mise a jour selon le modele PRICING = { "gpt-4": {"input": 0.03, "output": 0.06}, "gpt-4-turbo": {"input": 0.01, "output": 0.03}, "gpt-3.5-turbo": {"input": 0.0005, "output": 0.0015}, "gpt-4o": {"input": 0.005, "output": 0.015}, "gpt-4o-mini": {"input": 0.00015, "output": 0.0006}, } def __init__(self, model: str = "gpt-4"): self.model = model self.encoder = tiktoken.encoding_for_model(model) self.total_input_tokens = 0 self.total_output_tokens = 0 def count_tokens(self, text: str) -> int: return len(self.encoder.encode(text)) def track(self, input_text: str, output_text: str): self.total_input_tokens += self.count_tokens(input_text) self.total_output_tokens += self.count_tokens(output_text) def get_cost(self) -> float: pricing = self.PRICING.get(self.model, self.PRICING["gpt-4"]) input_cost = (self.total_input_tokens / 1000) * pricing["input"] output_cost = (self.total_output_tokens / 1000) * pricing["output"] return input_cost + output_cost def get_summary(self) -> dict: return { "model": self.model, "input_tokens": self.total_input_tokens, "output_tokens": self.total_output_tokens, "total_cost_usd": round(self.get_cost(), 4) }

8.2 Strategies de reduction des couts

Strategie Description Economies
Caching semantique Mettre en cache les reponses aux questions similaires 30-70% selon le taux de repetition
Model routing Utiliser GPT-4 pour les questions complexes, GPT-4o-mini pour les simples 50-80% sur les requetes simples
Prompt compression Reduire la taille des prompts (historique, contexte) 20-40% de tokens en moins
Truncation intelligente Limiter l'historique de conversation aux N derniers messages Variable mais significatif
Modeles locaux Utiliser Llama, Mistral pour les taches simples Cout token nul (cout infra uniquement)
# Caching semantique avec Redis import hashlib import json import redis class SemanticCache: def __init__(self, redis_url: str, ttl: int = 3600): self.redis = redis.Redis.from_url(redis_url) self.ttl = ttl def _make_key(self, query: str) -> str: # Hash de la question normalisee normalized = query.strip().lower() return f"agent_cache:{hashlib.md5(normalized.encode()).hexdigest()}" def get(self, query: str): key = self._make_key(query) cached = self.redis.get(key) if cached: return json.loads(cached) return None def set(self, query: str, response: dict): key = self._make_key(query) self.redis.setex(key, self.ttl, json.dumps(response)) # Utilisation dans l'agent cache = SemanticCache(redis_url="redis://localhost:6379") async def agent_with_cache(query: str) -> str: # Verifier le cache cached = cache.get(query) if cached: logger.info("Cache hit !") return cached # Executer l'agent result = await agent.ainvoke({"messages": [("user", query)]}) # Mettre en cache cache.set(query, result) return result

9. Rate Limiting

Le rate limiting protege a la fois votre agent (pour eviter les abus) et les APIs externes (pour ne pas depasser les quotas).

# Rate limiter avec fenetre glissante import time from collections import defaultdict class RateLimiter: def __init__(self, max_requests: int = 60, window_seconds: int = 60): self.max_requests = max_requests self.window = window_seconds self.requests = defaultdict(list) # user_id -> [timestamps] def allow(self, user_id: str = "default") -> bool: """Verifie si la requete est autorisee.""" now = time.time() window_start = now - self.window # Nettoyer les anciennes requetes self.requests[user_id] = [ ts for ts in self.requests[user_id] if ts > window_start ] # Verifier la limite if len(self.requests[user_id]) >= self.max_requests: return False self.requests[user_id].append(now) return True def remaining(self, user_id: str = "default") -> int: """Nombre de requetes restantes dans la fenetre.""" now = time.time() window_start = now - self.window current = [ts for ts in self.requests[user_id] if ts > window_start] return max(0, self.max_requests - len(current)) # Middleware FastAPI from fastapi import Request rate_limiter = RateLimiter(max_requests=60, window_seconds=60) @app.middleware("http") async def rate_limit_middleware(request: Request, call_next): # Identifier l'utilisateur (IP, API key, etc.) client_ip = request.client.host if not rate_limiter.allow(client_ip): return JSONResponse( status_code=429, content={"detail": "Trop de requetes. Reessayez plus tard."}, headers={"Retry-After": "60"} ) response = await call_next(request) response.headers["X-RateLimit-Remaining"] = str(rate_limiter.remaining(client_ip)) return response

10. Human-in-the-Loop en Production

Certaines actions d'agent sont trop critiques pour etre executees sans validation humaine. Le Human-in-the-Loop (HITL) introduit des points d'arret ou un humain doit approuver avant de continuer.

10.1 Quand exiger une validation humaine

  • Envoi d'emails ou de notifications
  • Transactions financieres (paiements, remboursements)
  • Modifications de base de donnees (DELETE, UPDATE en masse)
  • Actions irreversibles (suppression de comptes, envoi de donnees)
  • Decisions impactant des personnes (recrutement, notation, diagnostic)

10.2 Implementation avec LangGraph

# Human-in-the-loop avec LangGraph interrupt from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from langgraph.prebuilt import ToolNode from langgraph.types import interrupt, Command def sensitive_action_node(state): """Noeud qui demande une approbation humaine.""" action = state["pending_action"] # Interruption : demander l'approbation humaine approval = interrupt({ "action": action["type"], "details": action["details"], "message": f"L'agent veut executer: {action['type']}. Approuvez-vous ?" }) if approval == "approve": # Executer l'action result = execute_action(action) return {"action_result": result} else: return {"action_result": "Action refusee par l'utilisateur"} # Construire le graphe avec checkpointing memory = MemorySaver() graph = StateGraph(AgentState) # ... ajouter les noeuds ... app = graph.compile(checkpointer=memory) # Execution avec interruption config = {"configurable": {"thread_id": "user_123"}} # Premier appel : l'agent s'execute jusqu'a l'interruption result = app.invoke(input_data, config) # -> L'agent s'arrete et attend l'approbation # L'humain approuve via l'API result = app.invoke( Command(resume="approve"), config ) # -> L'agent reprend et execute l'action

11. Monitoring Dashboards

Un dashboard de monitoring centralise les metriques cles de votre agent en production. Il permet de detecter les problemes rapidement et de suivre la sante du systeme.

12. Checklist Production Complete

Checklist avant mise en production

Securite

  • Secrets dans un Secrets Manager (pas dans le code ni les .env commites)
  • Input sanitization et detection d'injection active
  • Output validation (PII, toxicite)
  • Authentification et autorisation sur les endpoints
  • HTTPS uniquement
  • Principe du moindre privilege pour les outils
  • Sandboxing pour l'execution de code

Performance

  • Streaming SSE implemente
  • Caching semantique actif (Redis)
  • Rate limiting par utilisateur
  • Timeout global par requete (30-60s max)
  • Limite d'iterations de l'agent (15-20 max)
  • Model fallback configure
  • Retry avec exponential backoff

Couts

  • Token counting actif
  • Budget par requete defini
  • Budget quotidien avec alerte a 80%
  • Model routing (modele leger pour les requetes simples)
  • Dashboard de suivi des couts en temps reel

Monitoring

  • LangSmith ou LangFuse integre
  • Logging structure (structlog/JSON)
  • Metriques Prometheus exposees
  • Dashboard Grafana configure
  • Alertes sur les metriques critiques
  • Health check endpoint (/health)

Operations

  • Dockerfile optimise (multi-stage build)
  • CI/CD pipeline avec tests automatiques
  • Rollback rapide possible
  • Documentation API (OpenAPI/Swagger)
  • Runbook pour les incidents courants
  • Plan de capacite (combien de requetes par seconde ?)

Qualite

  • Suite de tests end-to-end executee
  • Tests de regression sur les cas critiques
  • Evaluation LLM-as-Judge validee
  • Feedback utilisateur collecte et analyse
  • A/B testing possible
