Cours 7 : Systemes Multi-Agents

Orchestrer plusieurs agents pour resoudre des problemes complexes

1. Introduction aux Systemes Multi-Agents (MAS)

Un systeme multi-agents (MAS) est un ensemble d'agents autonomes qui interagissent entre eux pour accomplir des taches qu'un agent seul ne pourrait pas realiser efficacement. Chaque agent possede ses propres competences, son propre role, et communique avec les autres pour coordonner l'action collective.

Dans les cours precedents, nous avons etudie un agent unique equipe d'outils. Mais de nombreux problemes reels exigent la collaboration de plusieurs specialistes. Pensez a une equipe de developpement logiciel : un architecte, un developpeur, un testeur et un chef de projet travaillent ensemble. Le meme principe s'applique aux agents IA.

1.1 Pourquoi plusieurs agents ?

Il existe plusieurs raisons fondamentales pour adopter une approche multi-agents :

  • Specialisation : chaque agent maitrise un domaine precis (recherche, analyse, code, redaction). Un prompt systeme plus cible produit de meilleurs resultats.
  • Decomposition de la complexite : un probleme complexe est decoupe en sous-taches assignees a des agents specialises.
  • Robustesse : si un agent echoue, les autres peuvent compenser ou relancer la tache.
  • Scalabilite : on ajoute de nouveaux agents sans modifier les existants.
  • Parallelisme : certaines sous-taches peuvent etre executees en parallele par differents agents.

Analogie : L'equipe de redaction

Imaginez la production d'un article de blog technique :

  • Agent Chercheur : collecte les sources et donnees pertinentes
  • Agent Redacteur : ecrit le brouillon a partir des recherches
  • Agent Editeur : corrige, ameliore le style, verifie la coherence
  • Agent SEO : optimise les mots-cles, les titres, la meta-description

Chaque agent a un role clair, des competences specifiques, et le resultat final est meilleur que si un seul agent faisait tout.

1.2 Topologies multi-agents

Les agents peuvent etre organises selon differentes topologies :

Topologie Description Cas d'usage
Sequentielle Agent A passe a B, B passe a C (pipeline) Chaine de traitement, workflow lineaire
Hierarchique Un manager delegue aux agents subalternes Gestion de projet, decomposition de taches
Collaborative Les agents discutent librement pour arriver a un consensus Brainstorming, debat, revision par les pairs
Competitive Plusieurs agents proposent des solutions, le meilleur gagne Optimisation, encheres, selection de strategie
Hub-and-Spoke Un agent central orchestre tous les echanges Chatbot avec sous-agents specialises

2. Communication Inter-Agents

La communication est le pilier fondamental de tout systeme multi-agents. Les agents doivent echanger des informations pour coordonner leurs actions. Il existe trois paradigmes principaux.

2.1 Message Passing (Passage de messages)

Le message passing est un mode de communication ou les agents s'envoient directement des messages structures. Chaque agent possede une "boite de reception" et traite les messages a leur arrivee.

C'est le mode le plus courant dans les frameworks modernes. Les messages contiennent generalement :

  • L'identite de l'emetteur et du destinataire
  • Le type de message (requete, reponse, notification)
  • Le contenu (texte, donnees structurees)
  • Des metadonnees (timestamp, priorite)
# Exemple conceptuel de message passing class Message: def __init__(self, sender, receiver, content, msg_type="request"): self.sender = sender self.receiver = receiver self.content = content self.msg_type = msg_type self.timestamp = time.time() class Agent: def __init__(self, name): self.name = name self.inbox = [] def send(self, receiver, content): msg = Message(self.name, receiver.name, content) receiver.inbox.append(msg) def process_messages(self): for msg in self.inbox: print(f"[{self.name}] Message de {msg.sender}: {msg.content}") self.inbox.clear()

2.2 Shared State (Etat partage)

Dans ce paradigme, tous les agents lisent et ecrivent dans un espace memoire commun. C'est l'approche utilisee par LangGraph avec son State partage entre les noeuds.

# Etat partage dans LangGraph from typing import TypedDict, Annotated from langgraph.graph import StateGraph import operator class SharedState(TypedDict): messages: Annotated[list, operator.add] # Chaque agent ajoute ses messages research_data: str # Donnees de recherche draft: str # Brouillon de l'article feedback: str # Retour de l'editeur final_output: str # Resultat final
L'avantage du shared state est la simplicite : chaque agent lit l'etat global et y ecrit ses contributions. L'inconvenient est le risque de conflits d'ecriture si plusieurs agents modifient la meme cle simultanement.

2.3 Blackboard (Tableau noir)

Le blackboard est un espace de travail commun structure. Les agents surveillent le tableau et interviennent quand ils detectent une information relevant de leur competence. C'est un modele reactif.

3. CrewAI : Equipes d'Agents Specialises

CrewAI est un framework multi-agents qui organise les agents en "equipes" (crews). Chaque agent a un role, un objectif et un backstory. Les taches sont assignees aux agents et executees selon un processus defini (sequentiel ou hierarchique).

3.1 Les composants de CrewAI

Agent

Un agent CrewAI est defini par :

  • role : le titre du poste (ex: "Analyste de donnees senior")
  • goal : l'objectif principal de l'agent
  • backstory : le contexte qui donne de la personnalite a l'agent
  • tools : les outils a disposition
  • llm : le modele de langage utilise
  • allow_delegation : si l'agent peut deleguer a d'autres

Task

Une tache CrewAI contient :

  • description : ce que l'agent doit faire
  • agent : l'agent assigne
  • expected_output : le format attendu du resultat
  • context : les taches dont le resultat sert d'input

Process

  • sequential : les taches s'executent dans l'ordre, chaque resultat alimente la suivante
  • hierarchical : un agent manager decompose et delegue les taches automatiquement

3.2 Exemple complet avec CrewAI

# Installation: pip install crewai crewai-tools from crewai import Agent, Task, Crew, Process from crewai_tools import SerperDevTool, ScrapeWebsiteTool # --- Outils --- search_tool = SerperDevTool() scrape_tool = ScrapeWebsiteTool() # --- Agents --- researcher = Agent( role="Chercheur Senior en IA", goal="Trouver les informations les plus recentes et pertinentes sur le sujet", backstory="""Tu es un chercheur experimenté avec 15 ans d'experience en intelligence artificielle. Tu excelles dans la recherche d'informations fiables et leur synthese.""", tools=[search_tool, scrape_tool], verbose=True, allow_delegation=False ) writer = Agent( role="Redacteur Technique", goal="Rediger un article clair, engage et pedagogique", backstory="""Tu es un redacteur technique reconnu pour ta capacite a vulgariser des concepts complexes. Tu ecris avec precision et clarte, en utilisant des exemples concrets.""", verbose=True, allow_delegation=False ) editor = Agent( role="Editeur en Chef", goal="Garantir la qualite, la coherence et l'exactitude de l'article", backstory="""Tu es un editeur meticuleux qui s'assure que chaque article est impeccable. Tu verifies les faits, ameliores le style et t'assures de la coherence globale.""", verbose=True, allow_delegation=True ) # --- Taches --- research_task = Task( description="""Recherche approfondie sur les systemes multi-agents en IA. Trouve les frameworks principaux, les cas d'usage recents, et les tendances pour 2025. Fournis des sources fiables.""", agent=researcher, expected_output="Un rapport de recherche detaille avec sources et statistiques" ) writing_task = Task( description="""A partir des recherches fournies, redige un article de blog complet sur les systemes multi-agents. L'article doit etre structure avec des sections claires, des exemples concrets et un ton accessible.""", agent=writer, expected_output="Un article de blog de 1500 mots minimum, bien structure", context=[research_task] # Utilise le resultat de la recherche ) editing_task = Task( description="""Relis et ameliore l'article. Verifie les faits, corrige les erreurs, ameliore le style. Assure-toi que le ton est professionnel mais accessible.""", agent=editor, expected_output="L'article final, pret a publier, avec notes de revision", context=[writing_task] ) # --- Crew --- crew = Crew( agents=[researcher, writer, editor], tasks=[research_task, writing_task, editing_task], process=Process.sequential, verbose=True ) # Execution result = crew.kickoff() print(result)
CrewAI est le framework le plus simple pour debuter avec le multi-agents. Son API haut niveau (Agent, Task, Crew) est intuitive et sa documentation est excellente. Cependant, il offre moins de controle fin que LangGraph sur le flux d'execution.

4. AutoGen / AG2 : Conversations Multi-Agents

AutoGen (maintenant AG2) est un framework de Microsoft qui modelise les interactions multi-agents sous forme de conversations. Les agents communiquent par message passing dans des patterns definis : two-agent chat, group chat, nested chat.

4.1 Types d'agents AutoGen

  • AssistantAgent : un agent IA qui repond aux messages en utilisant un LLM
  • UserProxyAgent : represente l'utilisateur humain, peut executer du code et valider
  • GroupChatManager : orchestre une discussion de groupe entre plusieurs agents

4.2 Conversation a deux agents

# Installation: pip install pyautogen import autogen # Configuration du modele config_list = [{ "model": "gpt-4", "api_key": "sk-..." }] llm_config = {"config_list": config_list} # Agent assistant assistant = autogen.AssistantAgent( name="assistant_python", system_message="""Tu es un expert Python. Tu ecris du code propre, bien commente et tu expliques tes choix.""", llm_config=llm_config ) # Agent utilisateur (peut executer du code) user_proxy = autogen.UserProxyAgent( name="utilisateur", human_input_mode="NEVER", # Pas d'input humain code_execution_config={ "work_dir": "coding", "use_docker": False }, max_consecutive_auto_reply=5 ) # Lancer la conversation user_proxy.initiate_chat( assistant, message="Ecris un script Python qui analyse un fichier CSV de ventes et genere un graphique des tendances mensuelles." )
AutoGen excelle dans les scenarios ou l'assistant ecrit du code et le UserProxy l'execute automatiquement, creant un cycle d'iteration rapide : ecrire, executer, corriger.

4.3 Group Chat (Discussion de groupe)

# Plusieurs agents en discussion de groupe import autogen config_list = [{"model": "gpt-4", "api_key": "sk-..."}] llm_config = {"config_list": config_list} # Agent architecte architect = autogen.AssistantAgent( name="architecte", system_message="""Tu es un architecte logiciel senior. Tu concois les architectures et definis les interfaces.""", llm_config=llm_config ) # Agent developpeur developer = autogen.AssistantAgent( name="developpeur", system_message="""Tu es un developpeur Python senior. Tu implementes le code a partir des specifications de l'architecte.""", llm_config=llm_config ) # Agent testeur tester = autogen.AssistantAgent( name="testeur", system_message="""Tu es un ingenieur QA. Tu ecris des tests unitaires et d'integration pour le code du developpeur.""", llm_config=llm_config ) # Agent utilisateur proxy user_proxy = autogen.UserProxyAgent( name="chef_projet", human_input_mode="TERMINATE", code_execution_config={"work_dir": "project"} ) # Configuration du group chat groupchat = autogen.GroupChat( agents=[user_proxy, architect, developer, tester], messages=[], max_round=15, speaker_selection_method="auto" # Le LLM choisit qui parle ) manager = autogen.GroupChatManager( groupchat=groupchat, llm_config=llm_config ) # Lancer le groupe de discussion user_proxy.initiate_chat( manager, message="Construisez une API REST pour gerer une todo-list avec FastAPI." )
Avec speaker_selection_method="auto", le LLM decide quel agent parle a chaque tour. Cela peut etre imprevisible. Pour un controle strict, utilisez "round_robin" ou definissez un speaker_selection_func personnalise.

5. LangGraph Multi-Agents

LangGraph est le framework le plus flexible pour construire des systemes multi-agents. Chaque agent est un noeud dans un graphe d'etat, et les aretes definissent les transitions entre agents.

5.1 Architecture : Agents comme Noeuds

Principe fondamental

Dans LangGraph multi-agents, chaque agent est encapsule dans une fonction (noeud) qui :

  1. Recoit l'etat global du graphe en entree
  2. Execute sa logique (appel LLM, outils, raisonnement)
  3. Retourne une mise a jour partielle de l'etat

Le StateGraph orchestre les transitions entre agents selon des conditions definies par le developpeur.

5.2 Exemple : Systeme de recherche et redaction

# LangGraph avec plusieurs agents comme noeuds from typing import TypedDict, Annotated, Literal from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage import operator # --- Etat partage --- class MultiAgentState(TypedDict): messages: Annotated[list, operator.add] query: str research: str draft: str review: str final_article: str iteration: int llm = ChatOpenAI(model="gpt-4", temperature=0.7) # --- Agent Chercheur --- def researcher_agent(state: MultiAgentState) -> dict: query = state["query"] response = llm.invoke([ SystemMessage(content="""Tu es un chercheur expert. Fournis une analyse detaillee et structuree du sujet demande. Inclus des faits, des chiffres et des sources."""), HumanMessage(content=f"Recherche sur : {query}") ]) return { "research": response.content, "messages": [("researcher", response.content)] } # --- Agent Redacteur --- def writer_agent(state: MultiAgentState) -> dict: research = state["research"] review = state.get("review", "") prompt = f"Recherches :\n{research}" if review: prompt += f"\n\nRetour de l'editeur :\n{review}" response = llm.invoke([ SystemMessage(content="""Tu es un redacteur technique expert. Ecris un article clair et engage a partir des recherches. Si un feedback de l'editeur est fourni, integre-le."""), HumanMessage(content=prompt) ]) return { "draft": response.content, "messages": [("writer", response.content)] } # --- Agent Editeur --- def editor_agent(state: MultiAgentState) -> dict: draft = state["draft"] response = llm.invoke([ SystemMessage(content="""Tu es un editeur en chef exigeant. Evalue l'article : qualite, clarte, exactitude. Reponds UNIQUEMENT par : - "APPROUVE" si l'article est pret - Sinon, donne tes corrections detaillees"""), HumanMessage(content=f"Article a evaluer :\n{draft}") ]) return { "review": response.content, "messages": [("editor", response.content)], "iteration": state.get("iteration", 0) + 1 } # --- Routage conditionnel --- def should_continue(state: MultiAgentState) -> Literal["writer", "finalize"]: if "APPROUVE" in state["review"].upper(): return "finalize" if state.get("iteration", 0) >= 3: return "finalize" # Max 3 iterations return "writer" # --- Noeud de finalisation --- def finalize_agent(state: MultiAgentState) -> dict: return {"final_article": state["draft"]} # --- Construction du graphe --- graph = StateGraph(MultiAgentState) graph.add_node("researcher", researcher_agent) graph.add_node("writer", writer_agent) graph.add_node("editor", editor_agent) graph.add_node("finalize", finalize_agent) graph.set_entry_point("researcher") graph.add_edge("researcher", "writer") graph.add_edge("writer", "editor") graph.add_conditional_edges("editor", should_continue) graph.add_edge("finalize", END) # Compilation et execution app = graph.compile() result = app.invoke({"query": "Les tendances de l'IA generative en 2025", "iteration": 0}) print(result["final_article"])

6. OpenAI Swarm : Orchestration Legere

Swarm est un framework experimental d'OpenAI qui propose une approche minimaliste du multi-agents, basee sur deux concepts : les routines (instructions + outils d'un agent) et les handoffs (transfert de la conversation d'un agent a un autre).

6.1 Concepts cles

Routine

Une routine est la combinaison d'instructions systeme et de fonctions (outils) pour un agent. C'est ce qui definit le "comportement" de l'agent.

Handoff

Un handoff est le mecanisme par lequel un agent transfere la conversation a un autre agent. L'agent source retourne simplement un objet Agent comme resultat d'une fonction outil, et Swarm redirige automatiquement la conversation.

6.2 Exemple complet avec Swarm

# Installation: pip install openai-swarm from swarm import Swarm, Agent client = Swarm() # --- Fonctions de handoff --- def transfer_to_sales(): """Transferer au service commercial.""" return sales_agent def transfer_to_support(): """Transferer au support technique.""" return support_agent def transfer_to_billing(): """Transferer au service facturation.""" return billing_agent # --- Fonctions outils --- def check_order_status(order_id: str): """Verifier le statut d'une commande.""" # Simulation de base de donnees statuses = { "ORD-001": "En cours de livraison", "ORD-002": "Livree", "ORD-003": "En preparation" } return statuses.get(order_id, "Commande non trouvee") def process_refund(order_id: str, reason: str): """Traiter un remboursement.""" return f"Remboursement initie pour {order_id}. Raison: {reason}. Delai: 5-7 jours." # --- Agents --- triage_agent = Agent( name="Agent de Triage", instructions="""Tu es l'agent d'accueil. Determine le besoin du client et redirige vers le bon service : - Questions commerciales -> service commercial - Problemes techniques -> support technique - Questions de facturation/remboursement -> facturation""", functions=[transfer_to_sales, transfer_to_support, transfer_to_billing] ) sales_agent = Agent( name="Agent Commercial", instructions="""Tu es un agent commercial. Tu reponds aux questions sur les produits, les prix et les promotions. Sois enthousiaste et convaincant.""" ) support_agent = Agent( name="Agent Support", instructions="""Tu es un agent de support technique. Tu aides les clients avec leurs problemes techniques.""", functions=[check_order_status] ) billing_agent = Agent( name="Agent Facturation", instructions="""Tu es un agent de facturation. Tu geres les questions de facturation et les remboursements.""", functions=[process_refund, check_order_status] ) # --- Execution --- response = client.run( agent=triage_agent, messages=[{"role": "user", "content": "Je voudrais un remboursement pour ma commande ORD-001"}] ) print(f"Agent actif: {response.agent.name}") print(f"Reponse: {response.messages[-1]['content']}")
Agent actif: Agent Facturation Reponse: J'ai bien pris en compte votre demande de remboursement pour la commande ORD-001. Le remboursement a ete initie et sera traite sous 5 a 7 jours ouvrables.
Swarm est un framework "educatif" d'OpenAI, pas destine a la production. Son interet est de montrer les principes du multi-agents avec un minimum de code. Pour la production, preferez LangGraph ou CrewAI.

7. Patterns de Collaboration

Au-dela des frameworks, il existe des patterns de collaboration generiques que vous pouvez implementer dans n'importe quel systeme multi-agents.

7.1 Delegation

Un agent manager decompose une tache complexe en sous-taches et les delegue a des agents specialises. Le manager synthetise ensuite les resultats.

  • Quand l'utiliser : problemes decomposables, equipes avec des roles clairs
  • Framework ideal : CrewAI (mode hierarchique), LangGraph

7.2 Voting (Vote)

Plusieurs agents produisent chacun une reponse, puis un mecanisme de vote determine la meilleure. C'est similaire au "self-consistency" mais avec des agents differents.

  • Quand l'utiliser : decisions critiques necessitant de la fiabilite
  • Implementation : 3+ agents repondent, un agent juge vote
# Pattern de vote avec LangGraph def voting_node(state): """Trois agents votent sur la meilleure approche.""" proposals = state["proposals"] # Liste de 3 propositions judge_prompt = f"""Voici 3 propositions pour resoudre le probleme : Proposition 1: {proposals[0]} Proposition 2: {proposals[1]} Proposition 3: {proposals[2]} Evalue chaque proposition et choisis la meilleure. Reponds avec le numero de la proposition gagnante (1, 2 ou 3) et justifie ton choix.""" response = llm.invoke([ SystemMessage(content="Tu es un juge impartial et expert."), HumanMessage(content=judge_prompt) ]) return {"decision": response.content}

7.3 Consensus

Les agents discutent jusqu'a atteindre un accord. Chaque agent argumente sa position, ecoute les autres et ajuste son point de vue. Le processus s'arrete quand tous convergent.

  • Quand l'utiliser : decisions complexes avec de multiples perspectives
  • Risque : boucles infinies si les agents ne convergent pas

7.4 Encheres (Auction)

Chaque agent "encherit" sur une tache en indiquant sa capacite a la realiser (score de confiance). La tache est attribuee a l'agent avec la meilleure enchere.

  • Quand l'utiliser : allocation dynamique de taches, optimisation de ressources
  • Implementation : chaque agent evalue sa pertinence pour la tache (0-100)
# Pattern d'encheres def auction_router(state): """Chaque agent bid sur sa capacite a traiter la requete.""" task = state["task"] agents_config = { "data_analyst": "Expert en analyse de donnees et statistiques", "code_expert": "Expert en programmation Python", "writer": "Expert en redaction technique" } bids = {} for agent_name, expertise in agents_config.items(): response = llm.invoke([ SystemMessage(content=f"Tu es un {expertise}."), HumanMessage(content=f"""Evalue ta capacite a traiter cette tache : '{task}' Reponds uniquement par un score de 0 a 100.""") ]) bids[agent_name] = int(response.content.strip()) winner = max(bids, key=bids.get) return {"assigned_agent": winner, "bids": bids}

8. Implementation Complete : Systeme 3 Agents

Voici une implementation complete et fonctionnelle d'un systeme multi-agents avec LangGraph. Ce systeme analyse une question technique avec trois agents specialises : un planificateur, un executeur et un reviseur.

# === SYSTEME MULTI-AGENTS COMPLET AVEC LANGGRAPH === # Systeme de resolution de problemes techniques from typing import TypedDict, Annotated, Literal from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage import operator import json # ---- Etat global du systeme ---- class TeamState(TypedDict): problem: str # Le probleme a resoudre plan: str # Plan de l'architecte solution: str # Solution du developpeur review_feedback: str # Feedback du reviseur is_approved: bool # Approuve par le reviseur ? iteration: int # Numero d'iteration max_iterations: int # Maximum d'iterations messages: Annotated[list, operator.add] # Historique complet # ---- Initialisation du LLM ---- llm = ChatOpenAI(model="gpt-4", temperature=0.3) # ---- AGENT 1 : PLANIFICATEUR ---- def planner_agent(state: TeamState) -> dict: """L'architecte analyse le probleme et cree un plan.""" problem = state["problem"] feedback = state.get("review_feedback", "") context = f"Probleme : {problem}" if feedback: context += f"\n\nFeedback du reviseur sur l'iteration precedente :\n{feedback}" response = llm.invoke([ SystemMessage(content="""Tu es un architecte logiciel senior. Analyse le probleme et produis un plan structure : 1. Decomposition du probleme 2. Choix technologiques justifies 3. Etapes d'implementation numerotees 4. Points d'attention et risques Si un feedback est fourni, ajuste ton plan en consequence."""), HumanMessage(content=context) ]) return { "plan": response.content, "messages": [{"agent": "planner", "content": response.content}] } # ---- AGENT 2 : EXECUTEUR ---- def executor_agent(state: TeamState) -> dict: """Le developpeur implemente la solution selon le plan.""" plan = state["plan"] problem = state["problem"] feedback = state.get("review_feedback", "") context = f"Probleme : {problem}\n\nPlan de l'architecte :\n{plan}" if feedback: context += f"\n\nCorrections demandees :\n{feedback}" response = llm.invoke([ SystemMessage(content="""Tu es un developpeur Python senior. A partir du plan fourni, ecris la solution complete : - Code Python fonctionnel et commente - Gestion des erreurs - Tests unitaires basiques - Explication de chaque partie cle"""), HumanMessage(content=context) ]) return { "solution": response.content, "messages": [{"agent": "executor", "content": response.content}] } # ---- AGENT 3 : REVISEUR ---- def reviewer_agent(state: TeamState) -> dict: """Le reviseur evalue la solution et decide si elle est acceptable.""" solution = state["solution"] plan = state["plan"] problem = state["problem"] response = llm.invoke([ SystemMessage(content="""Tu es un ingenieur QA senior et reviseur de code. Evalue la solution par rapport au plan et au probleme initial. Criteres d'evaluation : - Correctness : le code resout-il le probleme ? - Qualite : le code est-il propre et maintenable ? - Completude : toutes les fonctionnalites sont-elles implementees ? - Tests : les tests sont-ils suffisants ? Reponds EXACTEMENT dans ce format JSON : { "approved": true/false, "score": 0-100, "feedback": "tes commentaires detailles" }"""), HumanMessage(content=f"""Probleme : {problem} Plan : {plan} Solution : {solution}""") ]) try: review = json.loads(response.content) except json.JSONDecodeError: review = {"approved": False, "score": 0, "feedback": response.content} return { "review_feedback": review.get("feedback", ""), "is_approved": review.get("approved", False), "iteration": state.get("iteration", 0) + 1, "messages": [{"agent": "reviewer", "content": json.dumps(review, indent=2)}] } # ---- ROUTAGE ---- def route_after_review(state: TeamState) -> Literal["planner", "end"]: """Decide si on continue ou si c'est termine.""" if state.get("is_approved", False): return "end" if state.get("iteration", 0) >= state.get("max_iterations", 3): return "end" # Securite anti-boucle infinie return "planner" # Retour au planificateur pour correction # ---- CONSTRUCTION DU GRAPHE ---- workflow = StateGraph(TeamState) # Ajout des noeuds workflow.add_node("planner", planner_agent) workflow.add_node("executor", executor_agent) workflow.add_node("reviewer", reviewer_agent) # Flux de base workflow.set_entry_point("planner") workflow.add_edge("planner", "executor") workflow.add_edge("executor", "reviewer") # Routage conditionnel apres review workflow.add_conditional_edges( "reviewer", route_after_review, { "planner": "planner", "end": END } ) # Compilation app = workflow.compile() # ---- EXECUTION ---- result = app.invoke({ "problem": """Creer une API REST avec FastAPI qui gere un systeme de gestion de bibliotheque : livres, emprunts, utilisateurs. Inclure l'authentification JWT et la pagination.""", "iteration": 0, "max_iterations": 3, "is_approved": False }) # Affichage des resultats print("=== PLAN ===") print(result["plan"]) print("\n=== SOLUTION ===") print(result["solution"]) print(f"\nApprouve: {result['is_approved']}") print(f"Iterations: {result['iteration']}")

10. Gestion des Conflits entre Agents

Quand plusieurs agents travaillent ensemble, des conflits peuvent apparaitre : recommandations contradictoires, boucles infinies, desaccords sur la strategie. Voici comment les gerer.

10.1 Types de conflits

Type de conflit Description Solution
Contradiction Deux agents donnent des reponses opposees Agent arbitre / vote majoritaire
Boucle infinie Deux agents se corrigent mutuellement sans fin Compteur d'iterations + timeout
Deadlock Chaque agent attend l'output de l'autre Valeurs par defaut + timeouts
Surcharge Un agent recoit trop de taches Load balancing + file d'attente

10.2 Strategies de resolution

Strategie 1 : Limite d'iterations - Toujours definir un nombre maximum de tours de discussion. Si le consensus n'est pas atteint, un agent superviseur tranche.
Strategie 2 : Agent arbitre - Un agent special qui n'intervient que pour resoudre les conflits. Il a acces aux arguments des deux parties et prend une decision finale.
Strategie 3 : Priorite hierarchique - Definir une hierarchie de priorite : l'agent manager a toujours le dernier mot en cas de desaccord.
Strategie 4 : Scoring - Chaque proposition recoit un score de confiance. La proposition avec le meilleur score est selectionnee automatiquement.
# Gestionnaire de conflits class ConflictResolver: def __init__(self, llm, max_iterations=3): self.llm = llm self.max_iterations = max_iterations def resolve(self, agent_outputs: dict) -> str: """Resout un conflit entre agents.""" conflict_prompt = "Les agents suivants ont des avis divergents :\n\n" for agent, output in agent_outputs.items(): conflict_prompt += f"Agent '{agent}': {output}\n\n" conflict_prompt += "Analyse les arguments de chaque agent et fournis une decision finale justifiee." response = self.llm.invoke([ SystemMessage(content="Tu es un arbitre impartial et expert."), HumanMessage(content=conflict_prompt) ]) return response.content
Les boucles infinies sont le risque numero 1 des systemes multi-agents. Implementez TOUJOURS un compteur d'iterations et un timeout global. Sans ces garde-fous, vos couts API peuvent exploser en quelques minutes.

11. Bonnes Pratiques pour les Systemes Multi-Agents

Checklist de conception multi-agents

  1. Commencez simple : ne creez pas 10 agents quand 2 suffisent. Chaque agent ajoute de la complexite et du cout.
  2. Roles clairs : chaque agent doit avoir un role precis et non-ambigu. Evitez le chevauchement des responsabilites.
  3. Prompts systeme detailles : plus le prompt systeme est precis, mieux l'agent remplit son role.
  4. Limites strictes : max_iterations, timeouts, budgets tokens. Toujours.
  5. Logging complet : enregistrez chaque echange entre agents pour le debug.
  6. Tests unitaires par agent : testez chaque agent individuellement avant de les assembler.
  7. Graceful degradation : si un agent echoue, le systeme doit continuer avec un resultat partiel plutot que planter.
  8. Monitoring des couts : chaque agent consomme des tokens. Surveillez les couts par execution.
