Agents IA - Cours 4 : Memoire et Gestion du Contexte

Section Agents IA - Doter les agents d'une memoire persistante et efficace

1. Pourquoi la memoire est cruciale pour les agents

Imaginez un assistant qui oublie tout ce que vous lui avez dit des que vous changez de sujet. Vous devriez repeter votre nom, votre probleme, vos preferences a chaque message. C'est exactement ce qui se passe avec un LLM sans memoire : chaque appel est completement independant, sans aucune notion de continuite.

La memoire d'un agent IA designe l'ensemble des mecanismes permettant de stocker, organiser et retrouver des informations issues des interactions passees, afin de maintenir une coherence conversationnelle et d'ameliorer la pertinence des reponses futures.

La memoire remplit trois fonctions essentielles pour un agent :

Les trois piliers de la memoire agent

  • Continuite : maintenir le fil de la conversation, se souvenir de ce qui a ete dit, eviter les repetitions et les contradictions
  • Apprentissage : capitaliser sur les interactions passees pour ameliorer les reponses futures, apprendre les preferences de l'utilisateur
  • Personnalisation : adapter le comportement de l'agent a chaque utilisateur en fonction de son historique, ses habitudes et ses besoins specifiques

Sans memoire, un agent ne peut pas :

  • Suivre une conversation multi-tours (ex: "comme je disais tout a l'heure...")
  • Se souvenir des preferences utilisateur (ex: "tu preferes les reponses courtes")
  • Accumuler des connaissances au fil du temps
  • Resoudre des taches complexes qui necessitent plusieurs etapes avec des resultats intermediaires
  • Detecter et eviter de repeter les memes erreurs
La memoire est ce qui transforme un simple chatbot en un veritable assistant personnel. C'est la difference entre parler a un inconnu a chaque message et avoir un collaborateur qui vous connait et s'adapte a vous au fil du temps.

2. Les 4 types de memoire cognitive

Pour concevoir un systeme de memoire efficace pour les agents IA, nous pouvons nous inspirer de la psychologie cognitive, qui distingue quatre types de memoire dans le cerveau humain. Chaque type a un equivalent direct dans le monde de l'IA.

2.1 Memoire de travail = Context Window

La memoire de travail humaine retient temporairement 7 elements (plus ou moins 2) pour les manipuler. Chez un LLM, la context window joue exactement ce role : c'est l'ensemble des tokens actuellement "visibles" par le modele pour generer sa reponse.

Tailles de context window par modele

ModeleContext WindowEquivalent texte
GPT-3.5 Turbo16K tokens~12 000 mots
GPT-4128K tokens~96 000 mots
Claude 3.5 Sonnet200K tokens~150 000 mots
Gemini 1.5 Pro1M tokens~750 000 mots
Llama 38K tokens~6 000 mots

2.2 Memoire episodique = Historique de conversation

La memoire episodique stocke les souvenirs personnels lies au temps et au lieu. Pour un agent, cela correspond a l'historique des conversations : qui a dit quoi et quand. C'est la forme de memoire la plus directe et la plus couramment implementee.

2.3 Memoire semantique = Knowledge Base / Vector Store

La memoire semantique contient nos connaissances generales, independamment du contexte ou elles ont ete acquises. Pour un agent, cela se traduit par une base de connaissances interrogeable via des embeddings et une recherche vectorielle (RAG).

2.4 Memoire procedurale = Skills et comportements appris

La memoire procedurale concerne les savoir-faire automatises (faire du velo, taper au clavier). Pour un agent, cela correspond aux prompts systeme, aux comportements appris par fine-tuning, et aux regles internes codees en dur.

En combinant ces quatre types de memoire, un agent peut maintenir une conversation coherente (episodique), acceder a des connaissances profondes (semantique), gerer efficacement son espace de reflexion (travail), et appliquer des competences acquises (procedurale).

3. Context Window : limites et strategies

La context window est la contrainte fondamentale de tout systeme base sur un LLM. Tout ce que le modele "sait" a un instant donne doit tenir dans cette fenetre : le system prompt, l'historique, les resultats d'outils, et la question actuelle.

3.2 Estimer la consommation de tokens

def estimate_conversation_tokens(messages: list, model: str = "gpt-4") -> dict: """Estimer le nombre de tokens dans une conversation.""" encoder = tiktoken.encoding_for_model(model) total_tokens = 0 breakdown = [] for msg in messages: # Overhead par message (role, separateurs) msg_tokens = 4 # Overhead fixe par message msg_tokens += len(encoder.encode(msg["role"])) msg_tokens += len(encoder.encode(msg["content"])) breakdown.append({ "role": msg["role"], "tokens": msg_tokens, "preview": msg["content"][:50] + "..." }) total_tokens += msg_tokens total_tokens += 2 # Tokens de fin de conversation return { "total_tokens": total_tokens, "breakdown": breakdown, "pct_of_128k": f"{total_tokens / 128000 * 100:.2f}%" } # Exemple d'utilisation messages = [ {"role": "system", "content": "Tu es un assistant utile et precis."}, {"role": "user", "content": "Explique-moi le machine learning."}, {"role": "assistant", "content": "Le machine learning est une branche..."}, {"role": "user", "content": "Et le deep learning ?"}, ] result = estimate_conversation_tokens(messages) print(f"Total: {result['total_tokens']} tokens ({result['pct_of_128k']} de 128K)")
Le depassement de la context window provoque soit une erreur (chez OpenAI), soit une troncature silencieuse des messages les plus anciens (chez certains fournisseurs). Il est crucial de surveiller et gerer proactivement la consommation de tokens.

4. ConversationBufferMemory : le buffer complet

ConversationBufferMemory est la forme de memoire la plus simple de LangChain. Elle stocke tous les messages de la conversation dans une liste et les injecte integralement dans chaque appel au modele.

5. ConversationBufferWindowMemory : la fenetre glissante

ConversationBufferWindowMemory resout le probleme de croissance infinie en ne conservant que les k derniers echanges. C'est comme une fenetre qui glisse sur la conversation, ne gardant que les messages les plus recents.

from langchain.memory import ConversationBufferWindowMemory # Garder seulement les 5 derniers echanges memory = ConversationBufferWindowMemory( k=5, # Nombre de paires (human, ai) a garder return_messages=True, memory_key="history" ) # Simuler une longue conversation for i in range(10): memory.save_context( {"input": f"Question numero {i+1}"}, {"output": f"Reponse numero {i+1}"} ) # Verifier : seuls les 5 derniers echanges sont gardes messages = memory.load_memory_variables({})["history"] print(f"Nombre de messages en memoire: {len(messages)}") for msg in messages: print(f" [{msg.type}] {msg.content}")
Nombre de messages en memoire: 10 [human] Question numero 6 [ai] Reponse numero 6 [human] Question numero 7 [ai] Reponse numero 7 [human] Question numero 8 [ai] Reponse numero 8 [human] Question numero 9 [ai] Reponse numero 9 [human] Question numero 10 [ai] Reponse numero 10
Avec k=5, la memoire conserve les 5 derniers echanges (10 messages : 5 humain + 5 assistant). Les messages plus anciens sont definitivement supprimes. Le nombre de tokens reste ainsi borne, quelle que soit la duree de la conversation.
Le choix de la valeur k est un compromis : une valeur trop petite (k=2) fait perdre le contexte rapidement, une valeur trop grande (k=50) consomme beaucoup de tokens. En pratique, k=5 a k=10 est un bon point de depart pour la plupart des cas d'usage conversationnels.

6. ConversationSummaryMemory : le resume progressif

ConversationSummaryMemory adopte une approche radicalement differente : au lieu de stocker les messages bruts, elle utilise le LLM pour resumer progressivement la conversation. A chaque nouvel echange, le resume existant est mis a jour avec les nouvelles informations.

from langchain.memory import ConversationSummaryMemory from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4", temperature=0) # Creer la memoire avec resume memory = ConversationSummaryMemory( llm=llm, # LLM utilise pour generer les resumes return_messages=False, # Retourne du texte (pas des Messages) memory_key="history" ) # Simuler une conversation memory.save_context( {"input": "Bonjour, je suis Pierre. Je travaille chez TechCorp comme ingenieur ML."}, {"output": "Bonjour Pierre ! Bienvenue. Comment puis-je vous aider avec le ML ?"} ) memory.save_context( {"input": "Je cherche a ameliorer notre pipeline de RAG. On utilise ChromaDB."}, {"output": "ChromaDB est un bon choix. Quels problemes rencontrez-vous exactement ?"} ) memory.save_context( {"input": "Les resultats de recherche ne sont pas assez pertinents. Notre recall est de 60%."}, {"output": "Un recall de 60% peut etre ameliore. Avez-vous essaye le re-ranking ?"} ) # Voir le resume genere summary = memory.load_memory_variables({})["history"] print("Resume de la conversation :") print(summary)
Resume de la conversation : Pierre, ingenieur ML chez TechCorp, cherche a ameliorer leur pipeline RAG qui utilise ChromaDB. Le recall actuel est de 60%, ce qui n'est pas satisfaisant. L'assistant a suggere d'essayer le re-ranking pour ameliorer la pertinence des resultats de recherche.

Comment fonctionne le resume progressif

A chaque nouvel echange, le systeme envoie au LLM :

  1. Le resume precedent de la conversation
  2. Les nouveaux messages (question + reponse)
  3. L'instruction : "Mets a jour ce resume avec les nouvelles informations"

Le LLM produit alors un nouveau resume qui integre les informations precedentes et nouvelles. Ce processus consomme des tokens a chaque echange (pour generer le resume), mais le resume final reste compact.

Le resume est genere par un LLM, ce qui signifie qu'il peut omettre ou deformer certaines informations. Les details precis (nombres, dates, noms techniques) peuvent etre perdus ou alteres. Ce n'est pas adapte aux cas ou la precision factuelle est critique.

7. ConversationSummaryBufferMemory : l'approche hybride

ConversationSummaryBufferMemory combine le meilleur des deux mondes : elle conserve les messages recents en entier (comme le buffer) et resume les messages plus anciens (comme le summary). La bascule se fait quand le nombre total de tokens depasse un seuil configurable.

GPT-4 est polyvalent, Claude excelle en analyse."), ("Notre budget est de 5000 euros par mois pour les API.", "Avec 5000 euros/mois, vous avez de la marge. Avantages de la memoire vectorielle

  • Scalabilite : peut stocker des milliers de messages sans surcharger la context window
  • Pertinence : retrouve les informations semantiquement proches de la question, meme si elles datent de loin
  • Efficience : ne charge que les k messages les plus utiles, optimisant la consommation de tokens
  • Persistance : les embeddings sont stockes sur disque, survivant aux redemarrages

9. RAG comme memoire long terme

Le Retrieval-Augmented Generation (RAG) peut etre utilise comme un systeme de memoire long terme pour les agents. L'idee est de stocker toutes les interactions et connaissances dans une base vectorielle, puis de les retrouver au moment opportun pour enrichir le contexte de l'agent.

from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain_community.vectorstores import Chroma from langchain.schema import Document from datetime import datetime import json class RAGMemory: """Systeme de memoire long terme base sur le RAG.""" def __init__(self, collection_name: str = "agent_memory"): self.embeddings = OpenAIEmbeddings( model="text-embedding-3-small" ) self.vectorstore = Chroma( collection_name=collection_name, embedding_function=self.embeddings, persist_directory="./long_term_memory" ) self.llm = ChatOpenAI(model="gpt-4", temperature=0) def store_interaction(self, user_msg: str, ai_msg: str, metadata: dict = None): """Stocker une interaction dans la memoire long terme.""" timestamp = datetime.now().isoformat() # Combiner question et reponse en un seul document content = f"Utilisateur: {user_msg}\nAssistant: {ai_msg}" doc_metadata = { "timestamp": timestamp, "type": "conversation", "user_query": user_msg[:200], # Pour le filtrage } if metadata: doc_metadata.update(metadata) doc = Document( page_content=content, metadata=doc_metadata ) self.vectorstore.add_documents([doc]) def recall(self, query: str, k: int = 3) -> list: """Retrouver les souvenirs pertinents pour une requete.""" results = self.vectorstore.similarity_search_with_score( query, k=k ) memories = [] for doc, score in results: memories.append({ "content": doc.page_content, "score": round(score, 3), "timestamp": doc.metadata.get("timestamp", "inconnu") }) return memories def format_memories_for_prompt(self, query: str, k: int = 3) -> str: """Formater les souvenirs pour injection dans le prompt.""" memories = self.recall(query, k) if not memories: return "Aucun souvenir pertinent trouve." formatted = "Souvenirs pertinents de conversations passees:\n" for i, mem in enumerate(memories, 1): formatted += f"\n--- Souvenir {i} (pertinence: {mem['score']}) ---\n" formatted += mem["content"] + "\n" return formatted # Utilisation memory = RAGMemory() # Stocker des interactions memory.store_interaction( "Comment configurer un pipeline RAG ?", "Voici les etapes : 1) Charger les documents, 2) Creer les embeddings...", {"topic": "RAG"} ) # Plus tard, retrouver l'information context = memory.format_memories_for_prompt("Comment ameliorer mon RAG ?") print(context)
Le RAG comme memoire long terme est particulierement puissant pour les assistants qui interagissent avec le meme utilisateur sur des semaines ou des mois. Les souvenirs sont stockes de maniere permanente et retrouves par pertinence semantique, pas par ordre chronologique.

10. Entity Memory : extraction et suivi d'entites

ConversationEntityMemory utilise le LLM pour extraire et mettre a jour automatiquement des informations sur les entites mentionnees dans la conversation : personnes, entreprises, projets, lieux, etc. Chaque entite a sa propre fiche qui est mise a jour au fil des echanges.

from langchain.memory import ConversationEntityMemory from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4", temperature=0) # Creer la memoire d'entites memory = ConversationEntityMemory( llm=llm, return_messages=True ) # Premier echange : introduction d'entites memory.save_context( {"input": "Je travaille avec Alice sur le projet Phoenix chez TechCorp."}, {"output": "Interessant ! Parlez-moi du projet Phoenix."} ) # Deuxieme echange : enrichissement des entites memory.save_context( {"input": "Phoenix est un chatbot IA pour le service client. Alice est la lead developer."}, {"output": "Alice a un role cle dans Phoenix en tant que lead developer."} ) # Troisieme echange : nouvelles informations memory.save_context( {"input": "On utilise LangChain et GPT-4. Le budget est de 50K euros."}, {"output": "Un bon budget pour un projet de cette envergure."} ) # Consulter les entites extraites print("=== Entites en memoire ===") for entity, description in memory.entity_store.store.items(): print(f"\n[{entity}]") print(f" {description}")

11. Systeme de memoire multi-niveaux

En production, un agent performant combine plusieurs types de memoire. Construisons un systeme complet qui integre la memoire de conversation recente, le resume, la recherche vectorielle et le suivi d'entites.

from langchain_openai import ChatOpenAI, OpenAIEmbeddings from langchain_community.vectorstores import Chroma from langchain.schema import Document, SystemMessage, HumanMessage from typing import List, Dict from datetime import datetime import json class MultiLevelMemory: """Systeme de memoire multi-niveaux pour agents IA. Niveau 1 : Buffer recent (derniers k echanges, haute fidelite) Niveau 2 : Resume compresse (conversation entiere resumee) Niveau 3 : Memoire vectorielle (recherche semantique, long terme) Niveau 4 : Entites (fiches structurees sur les sujets cles) """ def __init__(self, k: int = 5): self.llm = ChatOpenAI(model="gpt-4", temperature=0) self.embeddings = OpenAIEmbeddings(model="text-embedding-3-small") self.k = k # Niveau 1 : Buffer recent self.recent_buffer: List[Dict] = [] # Niveau 2 : Resume self.summary: str = "" # Niveau 3 : Vector store self.vectorstore = Chroma( collection_name="multi_level_memory", embedding_function=self.embeddings, ) # Niveau 4 : Entites self.entities: Dict[str, str] = {} def add_interaction(self, user_msg: str, ai_msg: str): """Ajouter un echange a tous les niveaux de memoire.""" interaction = { "human": user_msg, "ai": ai_msg, "timestamp": datetime.now().isoformat() } # Niveau 1 : Ajouter au buffer (garder k derniers) self.recent_buffer.append(interaction) if len(self.recent_buffer) > self.k: self.recent_buffer = self.recent_buffer[-self.k:] # Niveau 2 : Mettre a jour le resume self._update_summary(user_msg, ai_msg) # Niveau 3 : Stocker dans le vector store doc = Document( page_content=f"Q: {user_msg}\nR: {ai_msg}", metadata={"timestamp": interaction["timestamp"]} ) self.vectorstore.add_documents([doc]) # Niveau 4 : Extraire les entites self._extract_entities(user_msg, ai_msg) def _update_summary(self, user_msg: str, ai_msg: str): """Mettre a jour le resume de la conversation.""" prompt = f"""Resume actuel de la conversation: {self.summary if self.summary else '(Debut de conversation)'} Nouveau message utilisateur: {user_msg} Nouvelle reponse assistant: {ai_msg} Mets a jour le resume pour inclure les nouvelles informations. Garde le resume concis (max 200 mots). Conserve les faits importants.""" response = self.llm.invoke(prompt) self.summary = response.content def _extract_entities(self, user_msg: str, ai_msg: str): """Extraire et mettre a jour les entites.""" existing = json.dumps(self.entities, ensure_ascii=False) prompt = f"""Entites connues: {existing} Nouveau message: {user_msg} Reponse: {ai_msg} Extrais les entites (personnes, projets, entreprises, technologies) mentionnees. Retourne un JSON avec les entites mises a jour. Format: {{"nom_entite": "description mise a jour"}}""" response = self.llm.invoke(prompt) try: new_entities = json.loads(response.content) self.entities.update(new_entities) except json.JSONDecodeError: pass # Ignorer si le format n'est pas valide def get_context(self, current_query: str) -> str: """Construire le contexte memoire complet pour le prompt.""" sections = [] # Resume global if self.summary: sections.append(f"## Resume de la conversation\n{self.summary}") # Entites pertinentes if self.entities: entity_text = "\n".join( f"- {k}: {v}" for k, v in self.entities.items() ) sections.append(f"## Entites connues\n{entity_text}") # Souvenirs pertinents (vector search) relevant = self.vectorstore.similarity_search( current_query, k=2 ) if relevant: mem_text = "\n".join( doc.page_content for doc in relevant ) sections.append(f"## Souvenirs pertinents\n{mem_text}") # Messages recents (buffer) if self.recent_buffer: recent = "\n".join( f"User: {m['human']}\nAI: {m['ai']}" for m in self.recent_buffer[-3:] ) sections.append(f"## Echanges recents\n{recent}") return "\n\n".join(sections) # ---- Utilisation ---- memory = MultiLevelMemory(k=5) # Conversation simulee memory.add_interaction( "Je suis Marc, CTO de StartupAI. On lance un projet de RAG.", "Bonjour Marc ! Un projet RAG pour StartupAI, c'est un excellent choix." ) memory.add_interaction( "On utilise LangChain avec Pinecone et GPT-4.", "LangChain + Pinecone + GPT-4 est un stack tres solide pour le RAG." ) # Recuperer le contexte pour une nouvelle question context = memory.get_context("Quelles ameliorations pour notre RAG ?") print(context)

Le projet vient de demarrer. ## Entites connues - Marc: CTO de StartupAI, responsable du projet RAG - StartupAI: Startup qui developpe un systeme RAG - Pinecone: Base vectorielle utilisee pour le projet RAG ## Souvenirs pertinents Q: On utilise LangChain avec Pinecone et GPT-4. 12. Memory dans LangGraph : State et Persistence

LangGraph, le framework de graphes pour agents de LangChain, a une approche differente de la memoire. Au lieu des classes Memory de LangChain legacy, LangGraph utilise le concept de State (etat) qui est persiste via un systeme de checkpointing.

12.1 State avec messages et memoire

from langgraph.graph import StateGraph, MessagesState, START, END from langgraph.checkpoint.memory import MemorySaver from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage from typing import Annotated, TypedDict from langgraph.graph.message import add_messages # State avec messages et memoire supplementaire class AgentState(TypedDict): messages: Annotated[list, add_messages] summary: str # Resume de la conversation user_profile: dict # Informations sur l'utilisateur llm = ChatOpenAI(model="gpt-4", temperature=0) def chatbot(state: AgentState) -> AgentState: """Noeud principal du chatbot avec gestion de memoire.""" # Construire le contexte memoire system_parts = ["Tu es un assistant utile et attentif."] if state.get("summary"): system_parts.append( f"Resume des echanges precedents: {state['summary']}" ) if state.get("user_profile"): profile = state["user_profile"] system_parts.append( f"Profil utilisateur: {profile}" ) system_msg = SystemMessage(content="\n".join(system_parts)) messages = [system_msg] + state["messages"] response = llm.invoke(messages) return {"messages": [response]} def should_summarize(state: AgentState) -> str: """Decider si un resume est necessaire.""" if len(state["messages"]) > 10: return "summarize" return END def summarize_conversation(state: AgentState) -> AgentState: """Resumer la conversation et tronquer les messages.""" messages = state["messages"] old_summary = state.get("summary", "") # Resumer les anciens messages summary_prompt = f"""Resume precedent: {old_summary} Messages a resumer: {chr(10).join(f'{m.type}: {m.content[:100]}' for m in messages[:-4])} Produis un resume concis incluant les informations importantes.""" summary_response = llm.invoke(summary_prompt) # Garder seulement les 4 derniers messages return { "summary": summary_response.content, "messages": messages[-4:] } # Construire le graphe graph = StateGraph(AgentState) graph.add_node("chatbot", chatbot) graph.add_node("summarize", summarize_conversation) graph.add_edge(START, "chatbot") graph.add_conditional_edges("chatbot", should_summarize) graph.add_edge("summarize", END) # Checkpointer pour la persistence checkpointer = MemorySaver() app = graph.compile(checkpointer=checkpointer)

12.2 Utilisation avec persistence de session

# Configuration de la session (thread_id identifie la conversation) config = {"configurable": {"thread_id": "session_user_42"}} # Premier message result = app.invoke( { "messages": [HumanMessage("Bonjour, je suis Julie, developpeuse Python.")], "summary": "", "user_profile": {} }, config=config ) print(result["messages"][-1].content) # Deuxieme message - meme session, le contexte est conserve result = app.invoke( {"messages": [HumanMessage("Quel est mon prenom ?")]}, config=config ) print(result["messages"][-1].content)

12.3 Persistence avec SQLite (production)

from langgraph.checkpoint.sqlite import SqliteSaver # Persistence SQLite pour la production with SqliteSaver.from_conn_string("checkpoints.db") as checkpointer: app = graph.compile(checkpointer=checkpointer) # Les conversations survivent aux redemarrages # Chaque thread_id represente une conversation independante config_user_1 = {"configurable": {"thread_id": "user_alice_001"}} config_user_2 = {"configurable": {"thread_id": "user_bob_002"}} # Alice et Bob ont des conversations completement separees app.invoke( {"messages": [HumanMessage("Je suis Alice.")]}, config=config_user_1 ) app.invoke( {"messages": [HumanMessage("Je suis Bob.")]}, config=config_user_2 ) # Recuperer l'historique d'un thread state = app.get_state(config_user_1) print(f"Messages d'Alice: {len(state.values['messages'])}")
En production, preferez toujours SqliteSaver ou PostgresSaver a MemorySaver. Le MemorySaver stocke tout en RAM et perd les donnees au redemarrage. Pour les applications a grande echelle, PostgresSaver avec une base de donnees dediee est la solution la plus robuste.

Points cles a retenir

  • La memoire est essentielle pour la continuite, l'apprentissage et la personnalisation des agents
  • Les 4 types de memoire cognitive (travail, episodique, semantique, procedurale) trouvent leurs equivalents dans les systemes IA
  • La context window est la contrainte fondamentale : tout doit tenir dans cette fenetre de tokens
  • ConversationBufferMemory est simple mais ne passe pas a l'echelle
  • ConversationBufferWindowMemory resout le probleme de taille avec une fenetre glissante
  • ConversationSummaryMemory compresse l'historique en resume, ideal pour les longues conversations
  • SummaryBufferMemory offre le meilleur compromis resume + messages recents
  • VectorStoreMemory permet la recherche semantique dans tout l'historique
  • Le RAG sert de memoire long terme persistante et scalable
  • L'Entity Memory extrait automatiquement des fiches sur les entites mentionnees
  • En production, combinez plusieurs niveaux de memoire pour un systeme robuste
  • LangGraph offre une approche moderne avec State, checkpointing et persistence SQL
