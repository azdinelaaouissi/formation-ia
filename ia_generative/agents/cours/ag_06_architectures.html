Cours 6 : Architectures d'Agents

Section Agents IA - Comprendre et choisir la bonne architecture pour chaque cas d'usage

1. Vue d'ensemble des architectures d'agents IA

Un agent IA n'est pas un monolithe. Il existe de nombreuses manières de structurer un système d'agents, chacune avec ses forces, ses faiblesses et ses cas d'usage privilégiés. Ce cours explore les principales architectures, des plus simples aux plus sophistiquées.

Une architecture d'agent est le schéma structurel qui définit comment un ou plusieurs agents sont organisés, comment ils communiquent, comment les tâches sont distribuées et comment les résultats sont agrégés.

Le choix d'architecture dépend de plusieurs facteurs : la complexité de la tâche, le besoin de spécialisation, les contraintes de latence et de coût, et le niveau de fiabilité requis. Voici un aperçu des architectures que nous allons étudier :

Les 8 architectures d'agents

  1. Single Agent Loop - Un seul agent en boucle réactive (ReAct)
  2. Router Agent - Un agent qui dispatche vers des spécialistes
  3. Supervisor - Un agent superviseur qui coordonne des workers
  4. Hierarchical - Architecture hiérarchique multi-niveaux
  5. Swarm - Agents autonomes sans superviseur central
  6. Map-Reduce - Parallélisation puis agrégation
  7. Debate - Agents qui argumentent pour converger
  8. Pipeline - Agents en chaîne séquentielle
En pratique, les architectures réelles combinent souvent plusieurs patterns. Un système de production pourrait utiliser un Router pour le premier dispatch, un Supervisor pour les tâches complexes, et un Pipeline pour le post-traitement. Ne vous enfermez pas dans une seule architecture.

2. Single Agent Loop (Boucle Agent Simple)

L'architecture la plus fondamentale est la boucle agent simple, souvent implémentée via le pattern ReAct (Reasoning + Acting). Un seul agent, équipé d'outils, raisonne et agit en boucle jusqu'à produire une réponse finale.

Caractéristiques du Single Agent Loop

PropriétéValeur
ComplexitéFaible
CoûtFaible à modéré (selon le nombre d'itérations)
LatenceVariable (dépend du nombre de boucles)
FiabilitéModérée (un seul point de défaillance)
Cas d'usageTâches simples à modérément complexes
C'est l'architecture par défaut de la plupart des frameworks (LangChain AgentExecutor, OpenAI Assistants). Commencez toujours par cette architecture et n'évoluez vers des architectures plus complexes que si elle ne suffit pas.

3. Router Agent

Le Router Agent est un agent "aiguilleur" dont le seul rôle est d'analyser la requête de l'utilisateur et de la diriger vers l'agent spécialisé le plus approprié. Il ne traite pas la tâche lui-même, il la dispatche.

Un Router Agent est un agent qui classifie les requêtes entrantes et les redirige vers des agents spécialisés. Il agit comme un point d'entrée unique qui détermine quel spécialiste est le mieux qualifié pour traiter chaque demande.

Caractéristiques du Router Agent

PropriétéValeur
ComplexitéFaible à modérée
CoûtFaible (1 appel router + 1 appel spécialiste)
LatenceFaible (2 appels séquentiels)
FiabilitéBonne (spécialisation = meilleure qualité)
Cas d'usageAssistants polyvalents, support client

4. Multi-Agent Supervisor

L'architecture Supervisor (superviseur) place un agent central en position de chef d'orchestre. Contrairement au Router qui dispatche une seule fois, le Supervisor coordonne activement plusieurs workers tout au long de l'exécution. Il décide quel worker appeler à chaque étape, collecte les résultats, et orchestre la collaboration.

Différence Router vs Supervisor

AspectRouterSupervisor
DispatchingUne seule foisItératif, multiple
CoordinationAucuneActive, continue
CollaborationAgents indépendantsWorkers coordonnés
RésultatRéponse d'un seul agentSynthèse de plusieurs workers
ComplexitéFaibleModérée à élevée

5. Hierarchical Agents (Architecture Hiérarchique)

L'architecture hiérarchique étend le concept de Supervisor sur plusieurs niveaux. Un superviseur de haut niveau délègue à des superviseurs intermédiaires, qui eux-mêmes coordonnent des workers. C'est l'équivalent d'un organigramme d'entreprise.

L'architecture hiérarchique ajoute de la latence à chaque niveau. Avec 3 niveaux, un message doit traverser au minimum 6 appels LLM (descente + remontée). Limitez-vous à 2-3 niveaux maximum et n'utilisez cette architecture que pour des tâches véritablement complexes nécessitant une coordination fine entre de nombreux spécialistes.

6. Swarm Architecture

L'architecture Swarm (essaim) est radicalement différente des approches centralisées. Il n'y a pas de superviseur central. Les agents sont autonomes et se passent le contrôle les uns aux autres via des "handoffs" (transferts). Chaque agent décide quand il a terminé et à qui passer le relais.

L'architecture Swarm est un modèle décentralisé où des agents autonomes se transfèrent le contrôle mutuellement. Chaque agent connaît ses compétences et sait à quels autres agents il peut déléguer. Popularisée par le framework Swarm d'OpenAI.

Avantages et inconvénients du Swarm

AvantagesInconvénients
Pas de point unique de défaillanceDifficile à déboguer
Facile d'ajouter de nouveaux agentsRisque de boucles infinies entre agents
Chaque agent est simple et testablePas de vision globale de la tâche
Modèle naturel pour le support clientOrdre d'exécution imprévisible

7. Map-Reduce Agents

L'architecture Map-Reduce, inspirée du paradigme de calcul distribué, divise une tâche en sous-tâches identiques qui sont exécutées en parallèle (phase Map), puis les résultats sont agrégés en un résultat final (phase Reduce). C'est idéal pour traiter de grandes quantités de données.

Map-Reduce est particulièrement efficace pour : résumer de nombreux documents, analyser un grand dataset en morceaux, évaluer de multiples candidats, ou toute tâche où la même opération doit être appliquée à de nombreuses entrées indépendantes.

8. Debate Pattern (Agents en Débat)

Le Debate Pattern met en scène plusieurs agents qui argumentent entre eux pour arriver à une meilleure réponse. Un agent "Pour" et un agent "Contre" confrontent leurs arguments, et un agent "Juge" tranche. Cette approche est inspirée de la méthode socratique et du débat contradictoire.

Le Debate Pattern est particulièrement utile pour les décisions à fort enjeu où les biais du LLM pourraient mener à une réponse unilatérale. En forçant un agent à argumenter contre, on explore des angles morts que l'agent "Pour" n'aurait pas considérés.

9. Pipeline Architecture

L'architecture Pipeline organise les agents en chaîne séquentielle. La sortie de chaque agent devient l'entrée du suivant. C'est l'architecture la plus simple pour les workflows où l'ordre est fixe et prédéfini.

Exemples de pipelines courants

  • Pipeline de contenu : Recherche -> Brouillon -> Révision -> Formatage -> Publication
  • Pipeline de code : Spécification -> Génération -> Test -> Revue -> Documentation
  • Pipeline d'analyse : Extraction -> Nettoyage -> Analyse -> Visualisation -> Rapport
  • Pipeline de traduction : Analyse source -> Traduction -> Post-édition -> Contrôle qualité

Caractéristiques du Pipeline

PropriétéValeur
ComplexitéTrès faible
CoûtPrévisible (N agents x 1 appel)
LatenceSomme des latences de chaque étape
FiabilitéFragile (une étape échoue = tout échoue)
FlexibilitéFaible (ordre fixe)
Cas d'usageWorkflows prévisibles, chaînes de traitement

Router Faible Faible Faible Bonne Assistants polyvalents

Hierarchical Élevée Élevé Élevée Variable Projets très complexes

11. Implémentation d'un Supervisor Agent avec LangGraph

Voici une implémentation complète d'un agent superviseur avec LangGraph. Le superviseur coordonne trois workers spécialisés : un chercheur, un analyste et un rédacteur.

11.1 Définition des workers

from typing import List, Optional, Literal, Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage, AIMessage from pydantic import BaseModel, Field import operator # --- Modèle pour les décisions du supervisor --- class DecisionSupervisor(BaseModel): prochain_worker: Literal["chercheur", "analyste", "redacteur", "TERMINER"] = Field( description="Le prochain worker à appeler, ou TERMINER" ) instruction: str = Field( description="Instruction spécifique pour le worker choisi" ) raison: str = Field( description="Pourquoi ce worker est choisi" ) # --- État du graphe --- class SupervisorState(TypedDict): tache: str messages: Annotated[List[dict], operator.add] prochain: Optional[str] instruction_worker: Optional[str] resultat_final: Optional[str] iterations: int # --- LLMs --- llm_supervisor = ChatOpenAI(model="gpt-4o", temperature=0) llm_worker = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)

11.2 Définition des noeuds

12. Implémentation d'un Router Agent

L'implémentation d'un Router Agent est plus simple que celle du Supervisor car il n'y a pas de boucle : le router classifie la requête une seule fois, puis l'agent spécialisé traite et retourne le résultat.

12.1 Router avec classification structurée

from typing import Literal, Optional from typing_extensions import TypedDict from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field # --- Classification du Router --- class ClassificationRoute(BaseModel): route: Literal["code", "recherche", "redaction", "conversation"] = Field( description="Le type d'agent spécialisé à utiliser" ) confiance: float = Field( description="Score de confiance de la classification (0 à 1)" ) justification: str = Field( description="Pourquoi cette route est choisie" ) # --- État --- class RouterState(TypedDict): requete: str route: Optional[str] confiance: Optional[float] reponse: Optional[str] # --- LLMs --- llm_router = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Rapide et pas cher llm_specialiste = ChatOpenAI(model="gpt-4o", temperature=0.3) # --- Noeud Router --- def noeud_router(state: RouterState) -> dict: """Classifie la requête et détermine la route.""" requete = state["requete"] prompt = f"""Classifie cette requête utilisateur dans une des catégories : - code : questions de programmation, debugging, génération de code - recherche : questions factuelles, recherche d'information - redaction : rédaction de textes, emails, rapports, résumés - conversation : discussion générale, questions sur l'IA Requête : {requete}""" router_llm = llm_router.with_structured_output(ClassificationRoute) classification = router_llm.invoke(prompt) print(f"Route : {classification.route} (confiance: {classification.confiance})") print(f"Raison : {classification.justification}") return { "route": classification.route, "confiance": classification.confiance } # --- Agents Spécialisés --- def agent_code(state: RouterState) -> dict: prompt = f"""Tu es un expert en programmation Python. Aide l'utilisateur avec sa question de code. Fournis des exemples concrets et explique ton raisonnement. Question : {state['requete']}""" reponse = llm_specialiste.invoke(prompt) return {"reponse": reponse.content} def agent_recherche(state: RouterState) -> dict: prompt = f"""Tu es un chercheur expert. Fournis une réponse détaillée, factuelle et structurée. Cite tes sources quand possible. Question : {state['requete']}""" reponse = llm_specialiste.invoke(prompt) return {"reponse": reponse.content} def agent_redaction(state: RouterState) -> dict: prompt = f"""Tu es un rédacteur professionnel. Produis un texte de haute qualité, bien structuré et engageant. Demande : {state['requete']}""" reponse = llm_specialiste.invoke(prompt) return {"reponse": reponse.content} def agent_conversation(state: RouterState) -> dict: prompt = f"""Tu es un assistant conversationnel amical et compétent. Réponds de manière naturelle et utile. Message : {state['requete']}""" reponse = llm_specialiste.invoke(prompt) return {"reponse": reponse.content} # --- Routage conditionnel --- def choisir_route(state: RouterState) -> str: return state["route"] # --- Construction du graphe --- graphe = StateGraph(RouterState) graphe.add_node("router", noeud_router) graphe.add_node("code", agent_code) graphe.add_node("recherche", agent_recherche) graphe.add_node("redaction", agent_redaction) graphe.add_node("conversation", agent_conversation) graphe.set_entry_point("router") graphe.add_conditional_edges( "router", choisir_route, { "code": "code", "recherche": "recherche", "redaction": "redaction", "conversation": "conversation" } ) # Toutes les routes mènent à END graphe.add_edge("code", END) graphe.add_edge("recherche", END) graphe.add_edge("redaction", END) graphe.add_edge("conversation", END) app = graphe.compile() # --- Test avec plusieurs requêtes --- requetes = [ "Comment implémenter un décorateur en Python ?", "Quels sont les derniers progrès en IA générative ?", "Rédige un email professionnel pour proposer un partenariat", "Bonjour, comment ça va ?" ] for requete in requetes: print(f"\n--- Requête : {requete[:50]}... ---") resultat = app.invoke({"requete": requete}) print(f"Réponse : {resultat['reponse'][:100]}...")

13. Design Patterns et Anti-Patterns

Au-delà des architectures, certains patterns de conception récurrents et anti-patterns sont essentiels à connaître pour construire des systèmes d'agents robustes.

13.1 Design Patterns recommandés

Pattern 1 : Fallback Chain

Si un agent échoue ou donne une réponse de basse qualité, basculer automatiquement sur un agent de secours (plus puissant, plus lent, ou avec une approche différente).

class FallbackChain: """Chaîne de fallback : essaie chaque agent jusqu'au succès.""" def __init__(self, agents: list, evaluateur): self.agents = agents self.evaluateur = evaluateur def executer(self, tache: str) -> str: for i, agent in enumerate(self.agents): print(f"Tentative {i+1}/{len(self.agents)}: {agent.nom}") try: resultat = agent.executer(tache) score = self.evaluateur.evaluer(tache, resultat) if score >= 0.7: print(f" Succès (score: {score:.2f})") return resultat print(f" Score insuffisant ({score:.2f}), fallback...") except Exception as e: print(f" Erreur : {e}, fallback...") return "Tous les agents ont échoué."

Pattern 2 : Guard Rails

Encadrer chaque agent avec des vérifications d'entrée (pré-conditions) et de sortie (post-conditions) pour garantir la qualité et la sécurité.

class GuardedAgent: """Agent encadré par des garde-fous.""" def __init__(self, agent, pre_checks=None, post_checks=None): self.agent = agent self.pre_checks = pre_checks or [] self.post_checks = post_checks or [] def executer(self, tache: str) -> str: # Vérifications pré-exécution for check in self.pre_checks: ok, raison = check(tache) if not ok: return f"Refusé (pré-condition) : {raison}" # Exécution resultat = self.agent.executer(tache) # Vérifications post-exécution for check in self.post_checks: ok, raison = check(resultat) if not ok: return f"Filtré (post-condition) : {raison}" return resultat # Exemple de checks def check_longueur(texte): if len(texte) > 10000: return False, "Requête trop longue" return True, "" def check_contenu_inapproprie(resultat): mots_interdits = ["confidentiel", "secret"] for mot in mots_interdits: if mot in resultat.lower(): return False, f"Contenu potentiellement sensible détecté" return True, ""

13.2 Anti-Patterns à éviter

Anti-Pattern 1 : Le Supervisor Omniscient - Un supervisor qui essaie de tout faire lui-même au lieu de déléguer. Si votre supervisor a un prompt de 2000 tokens décrivant toutes les tâches possibles, c'est un signe qu'il faut décomposer.
Anti-Pattern 2 : La Boucle Infinie - Des agents qui se renvoient indéfiniment des messages sans converger. Toujours mettre un compteur maximum d'itérations et un timeout. Un bon défaut est 10 itérations maximum.
Anti-Pattern 3 : L'Over-Engineering - Utiliser une architecture hiérarchique à 4 niveaux pour une tâche qui pourrait être résolue par un seul agent. Commencez simple, mesurez, et ajoutez de la complexité uniquement quand c'est justifié.
Anti-Pattern 4 : Le Contexte Perdu - Des workers qui n'ont pas accès aux résultats des workers précédents. Assurez-vous que l'état partagé contient toutes les informations nécessaires et que chaque worker y a accès.
Anti-Pattern 5 : Pas de Monitoring - Déployer un système multi-agents sans traces, logs ou métriques. En production, vous devez pouvoir observer chaque décision du supervisor, chaque appel de worker, et le temps passé à chaque étape.

13.3 Bonnes pratiques d'architecture

1. Commencer par un Single Agent : résolvez d'abord le problème avec un seul agent. Ne passez au multi-agents que si les résultats sont insuffisants.
2. Utiliser des modèles adaptés : un modèle puissant (GPT-4o, Claude) pour le supervisor/router, et des modèles rapides (GPT-4o-mini) pour les workers.
3. Définir des interfaces claires : chaque agent doit avoir un format d'entrée et de sortie bien défini. Utilisez Pydantic pour la validation structurée.
4. Implémenter des circuit breakers : si un agent échoue N fois, arrêtez de l'appeler et basculez sur une alternative ou retournez une erreur.
5. Tester chaque agent isolément : avant de tester l'architecture complète, testez chaque agent individuellement avec des cas représentatifs.
6. Monitorer en production : utilisez LangSmith, Phoenix ou un outil similaire pour tracer chaque exécution et identifier les goulots d'étranglement.

Arbre de décision : quelle architecture choisir ?

  1. La tâche est simple et un seul LLM suffit ? -> Single Agent
  2. Il y a plusieurs types de requêtes distinctes ? -> Router Agent
  3. La tâche nécessite plusieurs étapes coordonnées ? -> Supervisor
  4. Il y a des dizaines d'agents à coordonner ? -> Hierarchical
  5. Les agents doivent se passer le contrôle librement ? -> Swarm
  6. La même opération doit être appliquée à N entrées ? -> Map-Reduce
  7. La décision est critique et nécessite des perspectives opposées ? -> Debate
  8. Le workflow est fixe et séquentiel ? -> Pipeline
