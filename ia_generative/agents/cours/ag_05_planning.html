Cours 5 : Planification et Décomposition de Tâches

Section Agents IA - Maîtriser les stratégies de planification pour des agents autonomes efficaces

1. Pourquoi la planification est essentielle

Imaginez qu'on vous demande de rédiger un rapport complet sur l'impact de l'IA dans la santé. Un humain ne commence pas à écrire immédiatement. Il réfléchit d'abord : quels sujets couvrir, dans quel ordre, quelles sources consulter, comment structurer l'ensemble. C'est exactement ce que la planification apporte à un agent IA.

La planification dans le contexte des agents IA est le processus par lequel un agent décompose un objectif complexe en une séquence structurée de sous-tâches exécutables, en anticipant les dépendances, les ressources nécessaires et les points de vérification.

1.1 Le problème des tâches complexes

Les LLM sont remarquablement performants pour des tâches atomiques : répondre à une question, résumer un texte, traduire une phrase. Mais dès qu'une tâche implique plusieurs étapes interdépendantes, les erreurs s'accumulent rapidement sans planification.

Exemple : recherche et rédaction d'un article

Sans planification, un agent pourrait :

  • Commencer à rédiger sans avoir toutes les informations
  • Oublier de vérifier ses sources
  • Produire un texte incohérent car il n'a pas de plan global
  • Répéter des étapes déjà accomplies
  • Ne pas savoir quand il a terminé

Avec planification, l'agent :

  1. Identifie les sujets clés à couvrir
  2. Recherche les informations pour chaque sujet
  3. Organise les informations en plan structuré
  4. Rédige chaque section en suivant le plan
  5. Relit et vérifie la cohérence globale

1.2 Les limites du raisonnement pas-à-pas

Le prompting Chain-of-Thought (CoT) aide le LLM à raisonner étape par étape, mais il reste linéaire. Le modèle ne peut pas revenir en arrière, explorer des alternatives, ou restructurer son approche en cours de route. La planification explicite résout ces limitations.

Approche Avantage Limitation
Appel LLM unique Rapide, simple Échoue sur les tâches complexes
Chain-of-Thought Meilleur raisonnement Linéaire, pas de retour arrière
ReAct (Cours 3) Utilise des outils Réactif, pas de vision globale
Planification explicite Vision globale, adaptative Plus complexe à implémenter
La planification n'est pas toujours nécessaire. Pour une tâche simple comme "traduis cette phrase", un appel direct est suffisant. Réservez la planification aux tâches qui nécessitent réellement plusieurs étapes coordonnées. Surplanifier est aussi un anti-pattern courant.

2. Décomposition de tâches

La décomposition de tâches est le coeur de la planification. Il s'agit de transformer un objectif de haut niveau en une liste de sous-tâches concrètes et exécutables. Chaque sous-tâche doit être suffisamment petite pour être réalisée par un seul appel LLM ou outil.

2.1 Principes de décomposition

Critères d'une bonne sous-tâche

  • Atomique : réalisable en une seule action
  • Vérifiable : on peut déterminer si elle est accomplie
  • Indépendante (autant que possible) : minimiser les dépendances
  • Ordonnée : respecter l'ordre logique quand des dépendances existent
  • Claire : description non ambiguë de ce qui doit être fait

2.2 Implémentation d'un décomposeur de tâches

from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from pydantic import BaseModel, Field from typing import List # Modèle structuré pour les sous-tâches class SousTache(BaseModel): id: int = Field(description="Identifiant unique de la sous-tâche") description: str = Field(description="Description claire de la sous-tâche") dependances: List[int] = Field( default_factory=list, description="Liste des IDs des sous-tâches pré-requises" ) outil: str = Field( default="llm", description="Outil nécessaire : llm, search, code, file" ) class PlanDeTaches(BaseModel): objectif: str = Field(description="L'objectif global à atteindre") taches: List[SousTache] = Field(description="Liste ordonnée des sous-tâches") # Création du décomposeur llm = ChatOpenAI(model="gpt-4o", temperature=0) structured_llm = llm.with_structured_output(PlanDeTaches) prompt = ChatPromptTemplate.from_messages([ ("system", """Tu es un planificateur expert. Décompose l'objectif donné en sous-tâches concrètes et exécutables. Règles : - Chaque sous-tâche doit être réalisable en une seule action - Indique les dépendances entre les tâches - Utilise les outils disponibles : llm, search, code, file - Ordonne les tâches logiquement"""), ("human", "{objectif}") ]) chain_decomposition = prompt | structured_llm # Utilisation objectif = "Créer un rapport comparatif sur les frameworks d'agents IA" plan = chain_decomposition.invoke({"objectif": objectif}) for tache in plan.taches: deps = f" (après: {tache.dependances})" if tache.dependances else "" print(f"[{tache.id}] {tache.description} [{tache.outil}]{deps}")

3. Le pattern Plan-and-Execute

Le pattern Plan-and-Execute sépare explicitement deux rôles : un planificateur qui crée le plan global, et un exécuteur qui réalise chaque étape. Cette séparation des préoccupations permet à chaque composant de se concentrer sur sa spécialité.

Le pattern Plan-and-Execute est une architecture d'agent où un module de planification génère d'abord un plan complet, puis un module d'exécution réalise chaque étape du plan séquentiellement, avec la possibilité de replanifier si nécessaire.
La séparation planificateur/exécuteur est inspirée de la dualité "System 1 / System 2" en psychologie cognitive. Le planificateur est la réflexion lente et délibérée (System 2), tandis que l'exécuteur est l'action rapide et ciblée (System 1).

4. Implémentation Plan-and-Execute avec LangGraph

Voici une implémentation complète du pattern Plan-and-Execute en utilisant LangGraph. Nous allons construire un agent capable de planifier et exécuter une recherche complexe.

4.1 Définition de l'état

from typing import List, Optional, Annotated from typing_extensions import TypedDict from pydantic import BaseModel, Field from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate import operator # --- Modèles de données --- class Etape(BaseModel): description: str = Field(description="Description de l'étape à réaliser") outil: str = Field(default="llm", description="Outil à utiliser") class Plan(BaseModel): etapes: List[Etape] = Field(description="Liste ordonnée des étapes du plan") # --- État du graphe --- class PlanExecuteState(TypedDict): objectif: str # Objectif initial plan: List[str] # Plan actuel etape_courante: Optional[str] # Étape en cours resultats: Annotated[List[str], operator.add] # Résultats accumulés reponse_finale: Optional[str] # Réponse finale

4.3 Noeud Exécuteur

prompt_executeur = ChatPromptTemplate.from_messages([ ("system", """Tu es un exécuteur de tâches. Réalise l'étape demandée en utilisant les résultats précédents. 5. Tree of Thoughts (ToT)

Le Tree of Thoughts (Yao et al., 2023) est une technique avancée qui étend le Chain-of-Thought en explorant plusieurs chemins de raisonnement simultanément, comme un arbre de décisions. Au lieu de suivre un seul fil de pensée, l'agent explore plusieurs branches et évalue chacune pour choisir la meilleure.

Le Tree of Thoughts (ToT) est un framework de raisonnement où le LLM génère plusieurs "pensées" candidates à chaque étape, les évalue, et poursuit uniquement les branches les plus prometteuses. Cela permet une exploration systématique de l'espace des solutions.

5.1 Implémentation simplifiée du Tree of Thoughts

from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field from typing import List llm = ChatOpenAI(model="gpt-4o", temperature=0.7) class Pensee(BaseModel): contenu: str = Field(description="Le raisonnement proposé") score: float = Field(description="Score de confiance entre 0 et 1") class Pensees(BaseModel): pensees: List[Pensee] class Evaluation(BaseModel): score: float = Field(description="Score entre 0 et 1") justification: str # Génère k pensées candidates def generer_pensees(probleme: str, contexte: str, k: int = 3) -> List[str]: prompt = f"""Problème : {probleme} Contexte actuel : {contexte} Génère {k} approches DIFFÉRENTES pour avancer dans la résolution. Chaque approche doit être distincte et explorer un angle différent.""" pensees_llm = llm.with_structured_output(Pensees) result = pensees_llm.invoke(prompt) return [p.contenu for p in result.pensees] # Évalue une pensée def evaluer_pensee(probleme: str, pensee: str) -> float: prompt = f"""Évalue cette approche pour résoudre le problème. Problème : {probleme} Approche : {pensee} Donne un score entre 0 et 1 selon la pertinence et les chances de succès.""" eval_llm = llm.with_structured_output(Evaluation) result = eval_llm.invoke(prompt) return result.score # Tree of Thoughts principal def tree_of_thoughts(probleme: str, profondeur: int = 3, k: int = 3, seuil: float = 0.5) -> str: """Explore un arbre de pensées pour résoudre un problème.""" meilleur_chemin = [] contexte = "Début de la résolution" for niveau in range(profondeur): print(f"\n--- Niveau {niveau + 1} ---") # Générer des pensées candidates pensees = generer_pensees(probleme, contexte, k) # Évaluer chaque pensée scores = [] for i, pensee in enumerate(pensees): score = evaluer_pensee(probleme, pensee) scores.append((pensee, score)) symbole = "✓" if score >= seuil else "✗" print(f" [{symbole}] Score {score:.2f} : {pensee[:60]}...") # Sélectionner la meilleure pensée au-dessus du seuil scores.sort(key=lambda x: x[1], reverse=True) meilleure = scores[0] if meilleure[1] < seuil: print(f" Aucune pensée au-dessus du seuil {seuil}") break meilleur_chemin.append(meilleure[0]) contexte = "\n".join(meilleur_chemin) return contexte # Exécution solution = tree_of_thoughts( "Comment optimiser un pipeline RAG qui retourne des résultats non pertinents ?" ) print(f"\nSolution finale :\n{solution}")
Le Tree of Thoughts multiplie les appels LLM (k appels par niveau, fois la profondeur, plus les évaluations). Pour un arbre de profondeur 3 avec k=3, cela représente au minimum 18 appels LLM. Utilisez-le uniquement pour des problèmes qui justifient ce coût.

6. Self-Reflection et le papier Reflexion

Le papier Reflexion (Shinn et al., 2023) introduit un mécanisme où l'agent critique ses propres résultats et utilise cette critique pour s'améliorer lors des tentatives suivantes. Contrairement à l'apprentissage classique qui modifie les poids du modèle, Reflexion utilise la réflexion verbale stockée en mémoire.

Le cycle Reflexion

  1. Action : l'agent tente de résoudre la tâche
  2. Évaluation : un évaluateur (externe ou le LLM lui-même) juge le résultat
  3. Réflexion : l'agent analyse pourquoi il a échoué ou réussi partiellement
  4. Mémoire : la réflexion est stockée pour guider les tentatives futures
  5. Nouvelle tentative : l'agent réessaie avec le bénéfice de ses réflexions

6.1 Implémentation du pattern Self-Reflection

from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from pydantic import BaseModel, Field llm = ChatOpenAI(model="gpt-4o", temperature=0) class Evaluation(BaseModel): score: int = Field(description="Score de 1 à 10") points_forts: str = Field(description="Ce qui est bien fait") points_faibles: str = Field(description="Ce qui doit être amélioré") suggestion: str = Field(description="Suggestion concrète d'amélioration") class AgentReflexion: def __init__(self, max_iterations: int = 3, seuil_qualite: int = 8): self.max_iterations = max_iterations self.seuil_qualite = seuil_qualite self.reflexions = [] # Mémoire des réflexions def generer(self, tache: str) -> str: """Génère une réponse en tenant compte des réflexions passées.""" reflexions_txt = "" if self.reflexions: reflexions_txt = "\n\nRéflexions des tentatives précédentes :\n" reflexions_txt += "\n".join( f"- Tentative {i+1} : {r}" for i, r in enumerate(self.reflexions) ) prompt = f"""Tâche : {tache} {reflexions_txt} Produis la meilleure réponse possible en tenant compte des réflexions.""" reponse = llm.invoke(prompt) return reponse.content def evaluer(self, tache: str, reponse: str) -> Evaluation: """Évalue la qualité de la réponse.""" eval_llm = llm.with_structured_output(Evaluation) prompt = f"""Évalue cette réponse pour la tâche donnée. Tâche : {tache} Réponse : {reponse} Sois critique et constructif. Score de 1 à 10.""" return eval_llm.invoke(prompt) def reflechir(self, tache: str, reponse: str, evaluation: Evaluation) -> str: """Génère une réflexion sur la tentative.""" prompt = f"""Analyse cette tentative et génère une réflexion utile. Tâche : {tache} Réponse produite : {reponse} Score : {evaluation.score}/10 Points forts : {evaluation.points_forts} Points faibles : {evaluation.points_faibles} Que dois-tu faire différemment la prochaine fois ?""" reflexion = llm.invoke(prompt) return reflexion.content def executer(self, tache: str) -> str: """Boucle principale de réflexion itérative.""" for i in range(self.max_iterations): print(f"\n=== Tentative {i + 1}/{self.max_iterations} ===") # 1. Générer reponse = self.generer(tache) print(f"Réponse générée ({len(reponse)} caractères)") # 2. Évaluer evaluation = self.evaluer(tache, reponse) print(f"Score : {evaluation.score}/10") print(f"Forces : {evaluation.points_forts}") print(f"Faiblesses : {evaluation.points_faibles}") # 3. Vérifier si la qualité est suffisante if evaluation.score >= self.seuil_qualite: print(f"\nQualité suffisante atteinte !") return reponse # 4. Réfléchir pour améliorer reflexion = self.reflechir(tache, reponse, evaluation) self.reflexions.append(reflexion) print(f"Réflexion : {reflexion[:100]}...") print("\nNombre max d'itérations atteint") return reponse # Utilisation agent = AgentReflexion(max_iterations=3, seuil_qualite=8) resultat = agent.executer( "Écris une introduction de 200 mots sur l'impact de l'IA dans l'éducation" )

7. Critique Agent Pattern

Le pattern Critique Agent va plus loin que la self-reflection en utilisant un agent séparé pour évaluer le travail. Cette séparation évite le biais d'auto-évaluation : un agent tend à surévaluer son propre travail.

Pourquoi séparer le critique ?

  • Objectivité : un agent externe n'a pas de biais d'auto-complaisance
  • Spécialisation : le critique peut avoir un prompt optimisé pour l'évaluation
  • Modèle différent : on peut utiliser un modèle plus puissant pour la critique
  • Critères formels : le critique peut vérifier des règles précises

7.1 Implémentation du Critique Agent

from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field from typing import List class CritiqueResult(BaseModel): approuve: bool = Field(description="True si le travail est approuvé") score: int = Field(description="Score de 1 à 10") problemes: List[str] = Field(description="Liste des problèmes identifiés") corrections: List[str] = Field(description="Corrections suggérées") class CritiqueAgent: """Agent critique qui vérifie le travail d'un autre agent.""" def __init__(self, criteres: List[str]): self.llm = ChatOpenAI(model="gpt-4o", temperature=0) self.criteres = criteres def critiquer(self, tache: str, travail: str) -> CritiqueResult: criteres_txt = "\n".join(f"- {c}" for c in self.criteres) prompt = f"""Tu es un critique exigeant. Évalue ce travail selon les critères. Tâche originale : {tache} Travail à évaluer : {travail} Critères d'évaluation : {criteres_txt} Sois rigoureux. N'approuve que si le score est >= 8/10.""" eval_llm = self.llm.with_structured_output(CritiqueResult) return eval_llm.invoke(prompt) class ProducteurAvecCritique: """Agent producteur qui soumet son travail à un critique.""" def __init__(self, critique: CritiqueAgent, max_revisions: int = 3): self.llm = ChatOpenAI(model="gpt-4o", temperature=0.7) self.critique = critique self.max_revisions = max_revisions def produire(self, tache: str) -> str: travail = self.llm.invoke(f"Réalise cette tâche : {tache}").content for revision in range(self.max_revisions): # Soumettre au critique critique = self.critique.critiquer(tache, travail) print(f"Révision {revision + 1}: Score {critique.score}/10") if critique.approuve: print("Travail approuvé par le critique !") return travail # Réviser en tenant compte des corrections corrections_txt = "\n".join(critique.corrections) prompt_revision = f"""Révise ce travail en appliquant les corrections. Travail actuel : {travail} Corrections demandées : {corrections_txt} Produis une version améliorée.""" travail = self.llm.invoke(prompt_revision).content return travail # Utilisation critique = CritiqueAgent(criteres=[ "Exactitude factuelle", "Clarté de l'explication", "Présence d'exemples concrets", "Structure logique", "Complétude (tous les aspects couverts)" ]) producteur = ProducteurAvecCritique(critique) resultat = producteur.produire( "Explique le fonctionnement de l'attention multi-tête dans les Transformers" )

8. Iterative Refinement (Raffinement Itératif)

Le raffinement itératif est un pattern plus léger que Reflexion ou le Critique Agent. L'idée est simple : améliorer progressivement un résultat à travers plusieurs passes, chacune ciblant un aspect spécifique de la qualité.

8.1 Raffinement par passes successives

from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o", temperature=0.3) # Définition des passes de raffinement passes_raffinement = [ { "nom": "Exactitude", "instruction": """Vérifie et corrige l'exactitude factuelle. Corrige toute erreur sans changer le style.""" }, { "nom": "Clarté", "instruction": """Améliore la clarté et la lisibilité. Simplifie les phrases complexes, ajoute des transitions.""" }, { "nom": "Exemples", "instruction": """Ajoute des exemples concrets là où c'est utile. Les exemples doivent illustrer les concepts clés.""" }, { "nom": "Style", "instruction": """Peaufine le style et le ton. Assure une voix cohérente et engageante.""" } ] def raffinement_iteratif(texte_initial: str, passes: list) -> str: """Applique des passes de raffinement successives.""" texte = texte_initial for passe in passes: print(f" Passe : {passe['nom']}...") prompt = f"""{passe['instruction']} Texte à améliorer : {texte} Retourne le texte amélioré (texte complet, pas juste les modifications).""" reponse = llm.invoke(prompt) texte = reponse.content return texte # Exécution texte_brut = llm.invoke( "Écris un paragraphe sur le machine learning" ).content texte_raffine = raffinement_iteratif(texte_brut, passes_raffinement) print(f"\nTexte final :\n{texte_raffine}")
Le raffinement itératif est idéal quand vous connaissez à l'avance les dimensions de qualité à optimiser. Chaque passe est spécialisée, ce qui donne de meilleurs résultats que de demander "améliore tout" en une seule fois. C'est analogue aux passes d'un compilateur qui optimisent chacune un aspect du code.

9. Task Graphs : Représentation des Dépendances

Les Task Graphs (graphes de tâches) permettent de représenter les dépendances entre sous-tâches sous forme de graphe orienté acyclique (DAG). Cela permet d'identifier quelles tâches peuvent être exécutées en parallèle et quelles tâches doivent attendre la complétion d'autres.

9.1 Implémentation d'un exécuteur de Task Graph

import asyncio from collections import defaultdict from typing import Dict, List, Set, Callable, Any class TaskGraph: """Graphe de tâches avec gestion des dépendances et exécution parallèle.""" def __init__(self): self.taches: Dict[str, Callable] = {} self.dependances: Dict[str, List[str]] = defaultdict(list) self.resultats: Dict[str, Any] = {} def ajouter_tache(self, nom: str, fonction: Callable, dependances: List[str] = None): """Ajoute une tâche avec ses dépendances.""" self.taches[nom] = fonction if dependances: self.dependances[nom] = dependances def _taches_pretes(self, completes: Set[str]) -> List[str]: """Retourne les tâches dont toutes les dépendances sont complétées.""" pretes = [] for nom in self.taches: if nom not in completes: deps = self.dependances.get(nom, []) if all(d in completes for d in deps): pretes.append(nom) return pretes async def executer(self) -> Dict[str, Any]: """Exécute le graphe en parallélisant les tâches indépendantes.""" completes: Set[str] = set() while len(completes) < len(self.taches): pretes = self._taches_pretes(completes) if not pretes: raise RuntimeError("Cycle détecté dans le graphe !") print(f"Exécution parallèle : {pretes}") # Exécuter les tâches prêtes en parallèle resultats_batch = await asyncio.gather( *[self._executer_tache(nom) for nom in pretes] ) for nom, resultat in zip(pretes, resultats_batch): self.resultats[nom] = resultat completes.add(nom) return self.resultats async def _executer_tache(self, nom: str) -> Any: """Exécute une tâche individuelle.""" deps_resultats = { d: self.resultats[d] for d in self.dependances.get(nom, []) } return await self.taches[nom](deps_resultats) # Exemple d'utilisation async def recherche_web(deps): await asyncio.sleep(1) # Simule un appel API return "Résultats de recherche web..." async def analyse_docs(deps): await asyncio.sleep(1) return "Analyse des documents..." async def synthese(deps): return f"Synthèse de : {list(deps.values())}" graphe = TaskGraph() graphe.ajouter_tache("recherche", recherche_web) graphe.ajouter_tache("analyse", analyse_docs) graphe.ajouter_tache("synthese", synthese, ["recherche", "analyse"]) resultats = asyncio.run(graphe.executer())
Exécution parallèle : ['recherche', 'analyse'] Exécution parallèle : ['synthese']

10. Replanning : S'adapter en cours d'exécution

Le replanning (replanification) est la capacité d'un agent à modifier son plan en cours d'exécution lorsque les résultats intermédiaires révèlent que le plan initial n'est plus optimal. C'est un aspect essentiel des agents robustes.

Le replanning est le processus par lequel un agent réévalue et modifie son plan d'action en fonction des résultats obtenus lors de l'exécution, des erreurs rencontrées, ou de nouvelles informations découvertes.

10.1 Quand replanifier ?

Déclencheurs de replanification

  • Échec d'une étape : un outil ne retourne pas de résultat ou échoue
  • Résultat inattendu : les données obtenues changent la donne
  • Nouvelle information : une découverte rend certaines étapes obsolètes
  • Dépassement de budget : trop de tokens consommés, il faut raccourcir
  • Blocage : l'agent tourne en rond sans progresser

10.2 Implémentation du replanning

from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field from typing import List, Optional llm = ChatOpenAI(model="gpt-4o", temperature=0) class PlanRevise(BaseModel): doit_replanifier: bool = Field(description="Faut-il replanifier ?") raison: Optional[str] = Field(default=None, description="Raison") nouveau_plan: Optional[List[str]] = Field(default=None) def verifier_besoin_replanification( objectif: str, plan_actuel: List[str], etapes_completees: List[str], dernier_resultat: str ) -> PlanRevise: """Vérifie si le plan doit être modifié.""" prompt = f"""Objectif : {objectif} Plan actuel : {chr(10).join(f' {i+1}. {e}' for i, e in enumerate(plan_actuel))} Étapes déjà complétées : {chr(10).join(f' - {e}' for e in etapes_completees)} Dernier résultat obtenu : {dernier_resultat} Le plan est-il toujours pertinent ? Faut-il replanifier ? Si oui, propose un nouveau plan pour les étapes restantes.""" replan_llm = llm.with_structured_output(PlanRevise) return replan_llm.invoke(prompt) # Dans la boucle d'exécution Plan-and-Execute : def boucle_avec_replanning(objectif: str, plan: List[str]): etapes_completees = [] while plan: etape = plan.pop(0) print(f"Exécution : {etape}") resultat = llm.invoke(f"Exécute : {etape}").content etapes_completees.append(f"{etape} -> {resultat[:100]}") # Vérifier le besoin de replanification if plan: # Seulement s'il reste des étapes verification = verifier_besoin_replanification( objectif, plan, etapes_completees, resultat ) if verification.doit_replanifier: print(f"REPLANIFICATION : {verification.raison}") plan = verification.nouveau_plan print(f"Nouveau plan : {plan}") return etapes_completees
La replanification ajoute un appel LLM après chaque étape. Pour contrôler les coûts, vous pouvez ne vérifier que tous les N étapes, ou uniquement quand un outil échoue. Trouver le bon équilibre entre adaptabilité et efficacité est crucial.

11. Hierarchical Task Network (HTN)

Le Hierarchical Task Network (Réseau de Tâches Hiérarchique) est une approche issue de la planification classique en IA. L'idée est de décomposer les tâches de manière hiérarchique : une tâche de haut niveau est décomposée en sous-tâches, qui sont elles-mêmes décomposables, jusqu'à arriver à des actions primitives directement exécutables.

Niveaux de décomposition HTN

Niveau Type Exemple
0 - Objectif Abstrait "Créer une application web"
1 - Tâches Semi-abstrait "Créer le backend", "Créer le frontend"
2 - Sous-tâches Concret "Créer les routes API", "Créer la page d'accueil"
3 - Actions Primitif "Écrire la fonction get_users()", "Écrire le HTML"

11.1 Implémentation d'un planificateur HTN

from langchain_openai import ChatOpenAI from pydantic import BaseModel, Field from typing import List, Optional llm = ChatOpenAI(model="gpt-4o", temperature=0) class NoeudHTN(BaseModel): id: str = Field(description="Identifiant unique") description: str = Field(description="Description de la tâche") est_primitive: bool = Field(description="True si action exécutable directement") sous_taches: Optional[List['NoeudHTN']] = Field(default=None) class ArbreHTN(BaseModel): racine: NoeudHTN def decomposer_htn(objectif: str, profondeur_max: int = 3) -> NoeudHTN: """Décompose un objectif en arbre HTN.""" htn_llm = llm.with_structured_output(ArbreHTN) prompt = f"""Décompose cet objectif en un arbre hiérarchique de tâches. Objectif : {objectif} Profondeur maximale : {profondeur_max} Règles : - Les feuilles (profondeur max) doivent être des actions primitives - Chaque tâche intermédiaire a 2-4 sous-tâches - Les IDs suivent le format "1", "1.1", "1.1.1" etc.""" arbre = htn_llm.invoke(prompt) return arbre.racine def afficher_htn(noeud: NoeudHTN, indent: int = 0): """Affiche l'arbre HTN de manière lisible.""" prefixe = " " * indent symbole = "[x]" if noeud.est_primitive else "[+]" print(f"{prefixe}{symbole} {noeud.id}: {noeud.description}") if noeud.sous_taches: for st in noeud.sous_taches: afficher_htn(st, indent + 1) def extraire_primitives(noeud: NoeudHTN) -> List[NoeudHTN]: """Extrait toutes les actions primitives en ordre.""" if noeud.est_primitive: return [noeud] primitives = [] if noeud.sous_taches: for st in noeud.sous_taches: primitives.extend(extraire_primitives(st)) return primitives # Utilisation arbre = decomposer_htn("Construire un chatbot RAG pour une entreprise") afficher_htn(arbre) print("\n--- Actions primitives à exécuter ---") primitives = extraire_primitives(arbre) for p in primitives: print(f" {p.id}: {p.description}")

12. Exemple concret : Agent de recherche avec planification

Mettons tout ensemble dans un exemple concret : un agent qui planifie et exécute une recherche complexe en utilisant le pattern Plan-and-Execute avec replanning et raffinement itératif.

12.1 Architecture de l'agent de recherche

from typing import List, Optional, Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, END from langchain_openai import ChatOpenAI from langchain_community.tools.tavily_search import TavilySearchResults from langchain_core.prompts import ChatPromptTemplate from pydantic import BaseModel, Field import operator # --- Outils --- outil_recherche = TavilySearchResults(max_results=3) llm = ChatOpenAI(model="gpt-4o", temperature=0) # --- Modèles --- class EtapeRecherche(BaseModel): query: str = Field(description="La requête de recherche") objectif: str = Field(description="Ce qu'on cherche à savoir") class PlanRecherche(BaseModel): etapes: List[EtapeRecherche] class QualiteEval(BaseModel): score: int = Field(description="Score de 1 à 10") manques: List[str] = Field(description="Informations manquantes") suffisant: bool = Field(description="Les infos sont-elles suffisantes ?") # --- État --- class RechercheState(TypedDict): question: str plan: List[dict] resultats_recherche: Annotated[List[str], operator.add] synthese: Optional[str] evaluation: Optional[dict] iteration: int # --- Noeuds --- def planifier_recherche(state: RechercheState) -> dict: """Planifie les recherches à effectuer.""" question = state["question"] resultats_existants = state.get("resultats_recherche", []) contexte = "" if resultats_existants: contexte = f"\nRésultats déjà obtenus :\n{chr(10).join(resultats_existants[-3:])}" prompt = f"""Planifie des recherches web pour répondre à cette question. Question : {question}{contexte} Génère 3-5 requêtes de recherche complémentaires.""" plan_llm = llm.with_structured_output(PlanRecherche) plan = plan_llm.invoke(prompt) return { "plan": [e.model_dump() for e in plan.etapes], "iteration": state.get("iteration", 0) + 1 } def executer_recherches(state: RechercheState) -> dict: """Exécute toutes les recherches du plan.""" resultats = [] for etape in state["plan"]: try: res = outil_recherche.invoke({"query": etape["query"]}) texte = "\n".join([r["content"] for r in res]) resultats.append(f"[{etape['objectif']}] {texte}") except Exception as e: resultats.append(f"[ERREUR] {etape['query']}: {e}") return {"resultats_recherche": resultats} def synthetiser(state: RechercheState) -> dict: """Synthétise tous les résultats en réponse structurée.""" question = state["question"] resultats = state["resultats_recherche"] prompt = f"""Synthétise ces résultats de recherche en une réponse complète. Question : {question} Résultats : {chr(10).join(resultats)} Rédige une réponse structurée, factuelle et complète.""" reponse = llm.invoke(prompt) return {"synthese": reponse.content} def evaluer_qualite(state: RechercheState) -> dict: """Évalue si la synthèse est suffisamment complète.""" eval_llm = llm.with_structured_output(QualiteEval) prompt = f"""Évalue la qualité de cette réponse. Question : {state['question']} Réponse : {state['synthese']} La réponse couvre-t-elle tous les aspects importants ?""" evaluation = eval_llm.invoke(prompt) return {"evaluation": evaluation.model_dump()} def decider_suite(state: RechercheState) -> str: """Décide si on continue ou si on termine.""" eval_data = state.get("evaluation", {}) iteration = state.get("iteration", 0) if eval_data.get("suffisant", False) or iteration >= 3: return "terminer" return "replanifier" # --- Construction du graphe --- graphe = StateGraph(RechercheState) graphe.add_node("planifier", planifier_recherche) graphe.add_node("rechercher", executer_recherches) graphe.add_node("synthetiser", synthetiser) graphe.add_node("evaluer", evaluer_qualite) graphe.set_entry_point("planifier") graphe.add_edge("planifier", "rechercher") graphe.add_edge("rechercher", "synthetiser") graphe.add_edge("synthetiser", "evaluer") graphe.add_conditional_edges( "evaluer", decider_suite, {"replanifier": "planifier", "terminer": END} ) app = graphe.compile() # Exécution resultat = app.invoke({ "question": "Quels sont les derniers progrès en agents IA autonomes en 2024 ?", "resultats_recherche": [], "iteration": 0 }) print(resultat["synthese"])

12.2 Récapitulatif des patterns de planification

Tableau récapitulatif

Pattern Quand l'utiliser Coût (appels LLM)
Plan-and-Execute Tâches multi-étapes standard Modéré (N+2)
Tree of Thoughts Problèmes avec plusieurs solutions possibles Élevé (k x profondeur x 2)
Self-Reflection Tâches nécessitant de la qualité Modéré (3 x itérations)
Critique Agent Tâches critiques, vérification rigoureuse Modéré (2 x itérations)
Iterative Refinement Tâches créatives, rédaction Faible (N passes)
HTN Projets complexes à décomposer Faible à modéré
Replanning Environnements incertains Variable (1 par vérification)
En pratique, les meilleurs agents combinent plusieurs patterns. Par exemple : Plan-and-Execute pour la structure globale, Self-Reflection sur les étapes critiques, et Replanning quand une étape échoue. Le choix dépend toujours du compromis entre qualité du résultat et coût en tokens/latence.

12.3 Bonnes pratiques de planification

1. Commencer simple : utilisez d'abord un plan linéaire (Plan-and-Execute). N'ajoutez de la complexité (ToT, HTN) que si nécessaire.
2. Limiter la profondeur : un plan de 5-8 étapes est généralement suffisant. Au-delà, décomposez en sous-plans hiérarchiques.
3. Inclure des points de vérification : ajoutez des étapes d'évaluation dans le plan pour détecter les problèmes tôt.
4. Prévoir l'échec : chaque étape devrait avoir un plan B ou au moins un mécanisme de retry avec un prompt différent.
5. Surveiller les coûts : comptez les tokens et les appels API. Fixez un budget maximal et arrêtez si nécessaire.
6. Logger tout : enregistrez le plan, chaque résultat intermédiaire, les replanifications. C'est essentiel pour le débogage et l'amélioration.
Un piège courant est la "surplanification" : l'agent passe plus de temps à planifier et replanifier qu'à exécuter. Fixez toujours un nombre maximal d'itérations de replanification (2-3 maximum) et un timeout global.
