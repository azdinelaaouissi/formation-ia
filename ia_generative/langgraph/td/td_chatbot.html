TD Pratique - Construire un Chatbot avec LangGraph

Deuxieme TD LangGraph -- Concevoir un chatbot conversationnel avec memoire, routage et outils

1. Recapitulatif -- Qu'est-ce qu'un Chatbot ?

Un chatbot est un programme informatique capable de mener une conversation en langage naturel avec un utilisateur. Dans le contexte de LangGraph, un chatbot est modelise comme un graphe oriente avec etat ou chaque message (humain ou IA) est stocke dans un etat partage, permettant au modele de maintenir le contexte de la conversation au fil des echanges.

Fiche d'identite du Chatbot LangGraph

ProprieteValeur
TypeApplication conversationnelle a base de graphe
FrameworkLangGraph (extension de LangChain)
Etat principalListe de messages (HumanMessage, AIMessage, SystemMessage)
PersistanceMemorySaver pour conserver l'historique entre sessions
Noeuds typiquesprocess_message, generate_response, route_message
Cas d'usageAssistance client, FAQ interactive, tuteur virtuel
Imaginez un chatbot LangGraph comme un standardiste telephonique intelligent : il recoit un appel (message), consulte le dossier du client (etat/memoire), decide comment traiter la demande (routage), et fournit une reponse adaptee. La difference avec un simple appel a un LLM, c'est que le standardiste se souvient de tous les echanges precedents grace a l'etat du graphe.

Pourquoi LangGraph pour un chatbot ?

Un simple appel a ChatOpenAI ne conserve aucune memoire. Chaque appel est independant. LangGraph resout ce probleme en gerant un etat persistant qui accumule les messages et permet au modele de repondre en tenant compte de tout l'historique.

Avantage 1 : Memoire native -- l'historique de conversation est gere automatiquement dans l'etat.
Avantage 2 : Routage conditionnel -- le chatbot peut diriger la conversation vers differents noeuds selon le contexte.
Avantage 3 : Integration d'outils -- le chatbot peut appeler des fonctions externes (recherche, calcul, API).

2. Principe de Fonctionnement -- Le StateGraph du Chatbot

Le chatbot LangGraph repose sur un StateGraph dont l'etat central contient la liste des messages echanges. Le flux d'execution passe par deux noeuds principaux :

Noeud 1 : process_message -- Recoit le message de l'utilisateur, l'ajoute a l'etat, et prepare le contexte pour le LLM (ajout du system prompt, troncature si necessaire).
Noeud 2 : generate_response -- Invoque le modele de langage avec l'historique complet des messages et retourne la reponse de l'IA ajoutee a l'etat.

Architecture du graphe simple

Ce schema est la version la plus simple du chatbot. Dans la section 6, nous ajouterons un noeud de routage et un noeud d'outils pour creer un chatbot plus sophistique. Mais il est essentiel de comprendre cette base avant d'aller plus loin.

Code minimal du graphe

from typing import Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END from langgraph.graph.message import add_messages from langchain_openai import ChatOpenAI # Definition de l'etat du chatbot class State(TypedDict): messages: Annotated[list, add_messages] # Initialisation du modele llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) # Noeud unique : generer une reponse def chatbot_node(state: State): response = llm.invoke(state["messages"]) return {"messages": [response]} # Construction du graphe graph_builder = StateGraph(State) graph_builder.add_node("chatbot", chatbot_node) graph_builder.add_edge(START, "chatbot") graph_builder.add_edge("chatbot", END) graph = graph_builder.compile()
Ce graphe minimal contient un seul noeud. C'est le squelette de tout chatbot LangGraph. Chaque amelioration (memoire, routage, outils) s'ajoute a cette base.

3. Les Donnees -- Types de Messages et Etat Conversationnel

Dans LangGraph, la conversation est representee par une liste de messages types. Chaque type de message a un role specifique dans la conversation :

Les trois types de messages fondamentaux

TypeRoleExemple
HumanMessage Message envoye par l'utilisateur HumanMessage(content="Bonjour !")
AIMessage Reponse generee par le modele de langage AIMessage(content="Comment puis-je vous aider ?")
SystemMessage Instructions systeme pour guider le comportement du LLM SystemMessage(content="Tu es un assistant.")
from langchain_core.messages import HumanMessage, AIMessage, SystemMessage # Exemple d'un historique de conversation complet messages = [ SystemMessage(content="Tu es un assistant pedagogique specialise en Python."), HumanMessage(content="Qu'est-ce qu'une liste en Python ?"), AIMessage(content="Une liste est une structure de donnees ordonnee et modifiable..."), HumanMessage(content="Comment ajouter un element a une liste ?"), ] # Chaque message possede des attributs utiles for msg in messages: print(f"{msg.type}: {msg.content[:60]}...")
system: Tu es un assistant pedagogique specialise en Python... human: Qu'est-ce qu'une liste en Python ?... ai: Une liste est une structure de donnees ordonnee et modifiable... human: Comment ajouter un element a une liste ?...

L'etat conversationnel avec add_messages

La fonction add_messages est un reducer qui gere intelligemment l'accumulation des messages. Au lieu de remplacer la liste a chaque noeud, elle ajoute les nouveaux messages a la liste existante.

from typing import Annotated from typing_extensions import TypedDict from langgraph.graph.message import add_messages # L'annotation Annotated[list, add_messages] signifie : # "Cette liste utilise add_messages comme fonction de reduction" class ChatState(TypedDict): messages: Annotated[list, add_messages] # Accumule les messages user_name: str # Donnee supplementaire optionnelle session_id: str # Identifiant de session # Sans add_messages : {"messages": new_list} REMPLACE l'ancienne liste # Avec add_messages : {"messages": new_list} AJOUTE a l'ancienne liste
Sans le reducer add_messages, chaque noeud remplacerait toute la liste de messages. Vous perdriez l'historique de conversation a chaque etape. Utilisez toujours Annotated[list, add_messages] pour le champ messages d'un chatbot.
Pensez au reducer add_messages comme a un cahier de notes : quand un noeud retourne un nouveau message, le reducer l'ecrit a la suite dans le cahier au lieu d'arracher les pages precedentes. C'est ce mecanisme qui donne la "memoire" au chatbot.

Structure complete de l'etat

# Etat enrichi pour un chatbot avance class ChatbotState(TypedDict): messages: Annotated[list, add_messages] # Historique complet user_info: dict # Infos sur l'utilisateur turn_count: int # Nombre de tours de conversation last_tool_used: str # Dernier outil utilise # L'etat circule entre tous les noeuds du graphe # Chaque noeud peut lire et modifier n'importe quel champ

4. Les Parametres -- Configuration du Chatbot

4.1 Temperature

temperature (float, 0.0 a 2.0)

Controle le caractere aleatoire des reponses du modele.

ValeurComportementCas d'usage
0.0Deterministe, toujours la meme reponseFAQ, support technique
0.3 - 0.7Equilibre entre creativite et coherenceChatbot general
1.0 - 2.0Tres creatif, parfois incoherentBrainstorming, ecriture creative
Une temperature de 0.7 est generalement recommandee pour un chatbot conversationnel. Elle offre des reponses variees tout en restant coherentes et pertinentes.

4.2 Selection du modele

model (str)

Le choix du modele impacte directement la qualite, la vitesse et le cout des reponses.

ModeleVitesseQualiteCout
gpt-4o-miniTres rapideBonneFaible
gpt-4oRapideExcellenteMoyen
gpt-4-turboMoyenExcellenteEleve
Pour le developpement et les tests, utilisez gpt-4o-mini (rapide et peu couteux). Passez a gpt-4o en production si la qualite des reponses est critique.

4.3 System Prompt

system_prompt (str)

Le message systeme definit la personnalite, le ton et les regles de comportement du chatbot. C'est le parametre le plus important pour personnaliser votre assistant.

4.4 Persistance de la memoire avec MemorySaver

MemorySaver -- Checkpointer de memoire

Le MemorySaver est un checkpointer qui sauvegarde l'etat du graphe apres chaque etape d'execution. Il permet de reprendre une conversation exactement la ou elle s'est arretee, meme apres un redemarrage du programme.

from langgraph.checkpoint.memory import MemorySaver # Creer le checkpointer memory = MemorySaver() # Compiler le graphe avec la memoire graph = graph_builder.compile(checkpointer=memory) # Utiliser un thread_id pour identifier chaque conversation config = {"configurable": {"thread_id": "conversation_1"}} # Chaque appel avec le meme thread_id reprend la conversation result1 = graph.invoke( {"messages": [HumanMessage(content="Bonjour, je m'appelle Alice.")]}, config ) result2 = graph.invoke( {"messages": [HumanMessage(content="Quel est mon prenom ?")]}, config ) # Le chatbot se souvient de "Alice" grace au MemorySaver
Le thread_id fonctionne comme un numero de dossier dans une armoire : toutes les interactions portant le meme numero sont regroupees dans la meme conversation. Changez le thread_id et vous demarrez une conversation vierge, comme un nouveau dossier.

5. Les Metriques -- Evaluer la Qualite du Chatbot

Un chatbot doit etre evalue selon plusieurs dimensions pour garantir une bonne experience utilisateur. Voici les trois metriques essentielles :

Qualite des reponses

Definition : La pertinence, la coherence et l'exactitude des reponses generees.

Comment mesurer : Evaluation humaine (note de 1 a 5), ou utilisation d'un LLM evaluateur qui juge la qualite de la reponse par rapport a la question posee.

Cible : Score moyen superieur a 4/5 pour un chatbot en production.

Latence (temps de reponse)

Definition : Le temps ecoule entre la question de l'utilisateur et la reponse complete du chatbot.

Comment mesurer : Chronometre autour de graph.invoke() avec time.time().

Cible : Moins de 3 secondes pour une experience utilisateur fluide.

Utilisation de tokens

Definition : Le nombre de tokens consommes par requete (prompt + completion).

Comment mesurer : Inspecter response.usage_metadata sur l'AIMessage retournee.

Cible : Optimiser pour rester sous le budget tout en maintenant la qualite.

import time from langchain_core.messages import HumanMessage def mesurer_performance(graph, message: str, config: dict): """Mesure la latence et l'utilisation de tokens pour une requete.""" start = time.time() result = graph.invoke( {"messages": [HumanMessage(content=message)]}, config ) latence = time.time() - start # Recuperer le dernier message IA ai_msg = result["messages"][-1] print(f"Latence : {latence:.2f} secondes") print(f"Reponse : {ai_msg.content[:100]}...") if hasattr(ai_msg, "usage_metadata") and ai_msg.usage_metadata: tokens = ai_msg.usage_metadata print(f"Tokens prompt : {tokens.get('input_tokens', 'N/A')}") print(f"Tokens reponse : {tokens.get('output_tokens', 'N/A')}") print(f"Tokens total : {tokens.get('total_tokens', 'N/A')}") return result # Exemple d'utilisation config = {"configurable": {"thread_id": "perf_test"}} mesurer_performance(chatbot, "Explique-moi les decorateurs Python.", config)
Latence : 1.23 secondes Reponse : Les decorateurs en Python sont des fonctions qui modifient le comportement d'autres fonctions... Tokens prompt : 156 Tokens reponse : 203 Tokens total : 359

6. Implementation Complete -- Chatbot avec Memoire, Routage et Outils

Nous allons maintenant construire un chatbot complet qui combine trois capacites :

  • Memoire persistante avec MemorySaver pour conserver le contexte
  • Routage conditionnel pour diriger vers differents traitements
  • Integration d'outils pour repondre a des questions factuelles

Diagramme du chatbot complet

Etape 1 : Definir l'etat et les outils

from typing import Annotated, Literal from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END from langgraph.graph.message import add_messages from langgraph.prebuilt import ToolNode from langgraph.checkpoint.memory import MemorySaver from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, SystemMessage from langchain_core.tools import tool # --- Definition de l'etat --- class ChatbotState(TypedDict): messages: Annotated[list, add_messages] # --- Definition des outils --- @tool def rechercher_documentation(query: str) -> str: """Recherche dans la documentation technique. Utilise cet outil quand l'utilisateur pose une question technique.""" # Simulation d'une base de documentation docs = { "python liste": "Une liste Python est une collection ordonnee, modifiable, qui accepte les doublons.", "python dict": "Un dictionnaire Python stocke des paires cle-valeur avec acces O(1).", "langgraph": "LangGraph permet de creer des applications LLM sous forme de graphes orientes.", } for key, val in docs.items(): if key in query.lower(): return val return "Aucun resultat trouve pour cette recherche." @tool def calculer(expression: str) -> str: """Effectue un calcul mathematique. Utilise cet outil pour les questions avec des operations arithmetiques.""" try: result = eval(expression) return f"Le resultat de {expression} est {result}" except Exception as e: return f"Erreur de calcul : {e}" tools = [rechercher_documentation, calculer]

Etape 2 : Creer les noeuds et le routage

# --- Initialisation du LLM avec outils --- llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) llm_with_tools = llm.bind_tools(tools) SYSTEM_PROMPT = ( "Tu es un assistant pedagogique francophone. Tu aides les etudiants " "a comprendre la programmation. Utilise les outils a ta disposition " "quand c'est pertinent pour donner des reponses precises." ) # --- Noeud 1 : preparation du message --- def process_message(state: ChatbotState): messages = state["messages"] # Ajouter le system prompt seulement s'il n'est pas deja present if not messages or messages[0].type != "system": return {"messages": [SystemMessage(content=SYSTEM_PROMPT)]} return {} # --- Noeud 2 : generation de reponse avec outils --- def generate_response(state: ChatbotState): response = llm_with_tools.invoke(state["messages"]) return {"messages": [response]} # --- Fonction de routage conditionnel --- def route_after_generation(state: ChatbotState) -> Literal["tools", "__end__"]: """Si le LLM a demande un outil, on va vers 'tools'. Sinon, on termine la conversation.""" last_message = state["messages"][-1] if hasattr(last_message, "tool_calls") and last_message.tool_calls: return "tools" return "__end__"

Etape 3 : Assembler et compiler le graphe

# --- Construction du graphe complet --- graph_builder = StateGraph(ChatbotState) # Ajout des noeuds graph_builder.add_node("process", process_message) graph_builder.add_node("generate", generate_response) graph_builder.add_node("tools", ToolNode(tools=tools)) # Ajout des aretes graph_builder.add_edge(START, "process") # Entree -> Preparation graph_builder.add_edge("process", "generate") # Preparation -> Generation # Routage conditionnel apres la generation graph_builder.add_conditional_edges( "generate", # Noeud source route_after_generation, # Fonction de routage {"tools": "tools", "__end__": END} # Mapping ) # Apres les outils, retour a la generation pour formuler la reponse graph_builder.add_edge("tools", "generate") # Compilation avec memoire persistante memory = MemorySaver() chatbot = graph_builder.compile(checkpointer=memory) print("Chatbot compile avec succes !")
Chatbot compile avec succes !

Etape 4 : Fonction d'interaction

def chat(user_message: str, thread_id: str = "default"): """Envoie un message au chatbot et affiche la reponse.""" config = {"configurable": {"thread_id": thread_id}} result = chatbot.invoke( {"messages": [HumanMessage(content=user_message)]}, config ) # Recuperer le dernier AIMessage (sans tool_calls) ai_messages = [ m for m in result["messages"] if m.type == "ai" and not getattr(m, "tool_calls", []) ] if ai_messages: print(f"Assistant : {ai_messages[-1].content}") return result # Boucle interactive complete def boucle_chatbot(thread_id: str = "session_1"): """Lance une boucle de chat interactive dans le terminal.""" print("Chatbot LangGraph demarre ! Tapez 'quit' pour quitter.") print("-" * 50) while True: user_input = input("Vous : ") if user_input.lower() in ["quit", "exit", "q"]: print("Au revoir !") break chat(user_input, thread_id) print()

Test 4 : Impact de la temperature sur les reponses

# Comparer les reponses avec differentes temperatures for temp in [0.0, 0.5, 1.0]: llm_temp = ChatOpenAI(model="gpt-4o-mini", temperature=temp) response = llm_temp.invoke([ HumanMessage(content="Decris le soleil en une phrase.") ]) print(f"Temperature {temp} : {response.content}") print()
Temperature 0.0 : Le soleil est une etoile naine jaune situee au centre de notre systeme solaire. Observez le gradient de creativite : a 0.0 la reponse est factuelle et reproductible ; a 0.5 elle est plus imagee ; a 1.0 elle devient poetique et imprevisible. Pour un chatbot de support, utilisez 0.0-0.3. Pour un chatbot creatif, 0.7-1.0.

8. Erreurs Courantes -- Les Six Pieges a Eviter

Erreur 1 : Oublier le reducer add_messages

Symptome : Le chatbot ne se souvient de rien ; chaque reponse est hors contexte.

# FAUX -- pas de reducer, la liste est remplacee a chaque noeud class BadState(TypedDict): messages: list # Pas de reducer ! # CORRECT -- le reducer accumule les messages class GoodState(TypedDict): messages: Annotated[list, add_messages]

Erreur 2 : Ne pas passer le config avec thread_id

Symptome : ValueError car le checkpointer attend un thread_id.

# FAUX -- pas de config quand un checkpointer est actif result = chatbot.invoke({"messages": [HumanMessage(content="Bonjour")]}) # Leve : ValueError: Missing required 'thread_id' # CORRECT -- toujours passer le config avec thread_id config = {"configurable": {"thread_id": "ma_session"}} result = chatbot.invoke({"messages": [HumanMessage(content="Bonjour")]}, config)

Erreur 3 : Oublier de compiler le graphe

Symptome : AttributeError: 'StateGraph' object has no attribute 'invoke'

# FAUX -- appeler invoke sur le builder graph_builder.invoke(...) # StateGraph n'a pas de methode invoke ! # CORRECT -- compiler d'abord, puis appeler invoke graph = graph_builder.compile(checkpointer=memory) graph.invoke(...) # CompiledGraph a la methode invoke

Erreur 4 : System prompt duplique a chaque tour

Symptome : Le system prompt est ajoute a chaque message, consommant des tokens en double.

# FAUX -- ajoute le system prompt a chaque appel sans verifier def bad_process(state): return {"messages": [SystemMessage(content=SYSTEM_PROMPT)]} # CORRECT -- verifier s'il est deja present def good_process(state): if not state["messages"] or state["messages"][0].type != "system": return {"messages": [SystemMessage(content=SYSTEM_PROMPT)]} return {}

Erreur 5 : Outils sans retour au noeud de generation

Symptome : Apres l'execution d'un outil, le chatbot renvoie le ToolMessage brut au lieu d'une reponse en langage naturel.

# FAUX -- les outils envoient directement vers END graph_builder.add_edge("tools", END) # L'utilisateur recoit : "Le resultat de 2+2 est 4" (ToolMessage brut) # CORRECT -- retourner au noeud de generation graph_builder.add_edge("tools", "generate") # Le LLM formule : "Le calcul donne 4 comme resultat."

Erreur 6 : Confondre HumanMessage et string

Symptome : TypeError ou comportement inattendu lors de l'invocation.

# FAUX -- passer une string brute dans la liste result = chatbot.invoke({"messages": ["Bonjour"]}) # CORRECT -- utiliser les objets Message result = chatbot.invoke({"messages": [HumanMessage(content="Bonjour")]})

Note : LangGraph peut parfois convertir les strings automatiquement, mais il est fortement recommande d'utiliser les objets Message explicitement pour eviter les ambiguites.

Schema recapitulatif du flux complet

1. L'utilisateur envoie un HumanMessage avec un thread_id

2. Le noeud process_message ajoute le system prompt si necessaire

3. Le noeud generate_response appelle le LLM avec les outils disponibles

4. Le routeur conditionnel decide : reponse directe (END) ou appel d'outil (tools)

5. Si outil : execution par ToolNode, puis retour a la generation

6. Le MemorySaver sauvegarde l'etat complet pour le prochain message
