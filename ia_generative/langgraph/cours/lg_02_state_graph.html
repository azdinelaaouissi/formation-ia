StateGraph, Noeuds et Edges

Cours 2 — Construire des Graphes avec LangGraph

1. StateGraph — Le constructeur de graphes

StateGraph est la classe principale de LangGraph. C'est un builder (constructeur) qui permet de définir un graphe en y ajoutant des noeuds et des arêtes. Il est paramétré par un type d'état (TypedDict ou Pydantic BaseModel) qui définit la structure des données partagées entre tous les noeuds du graphe.

Le StateGraph n'est pas directement exécutable. Il sert uniquement à construire le graphe. Une fois la construction terminée, on appelle .compile() pour obtenir un CompiledGraph qui peut être invoqué.

from langgraph.graph import StateGraph, START, END from typing_extensions import TypedDict # Définir le type d'état class MonState(TypedDict): question: str reponse: str # Créer le StateGraph avec le type d'état builder = StateGraph(MonState) # À ce stade, le graphe est vide # Il faut ajouter des noeuds et des arêtes avant de compiler

1.1 Signature de StateGraph

StateGraph(state_schema)

  • state_schema (type) : Le type qui définit la structure de l'état. Peut être un TypedDict, un Pydantic BaseModel, ou un dataclass.

Retourne : un objet StateGraph (builder) sur lequel on appelle add_node(), add_edge(), puis compile().

Pensez au StateGraph comme à un plan d'architecte. Il décrit la structure du workflow, mais ne peut pas être « habité » directement. C'est la compilation (.compile()) qui « construit la maison » à partir du plan.

2. Définir le State

Le State est le coeur de tout graphe LangGraph. C'est la structure de données qui circule entre tous les noeuds et qui représente l'état courant du workflow.

2.1 State basique avec TypedDict

La manière la plus simple de définir un état est d'utiliser TypedDict :

from typing_extensions import TypedDict class ChatState(TypedDict): user_input: str # Ce que l'utilisateur a dit ai_response: str # La réponse de l'IA conversation_id: str # Identifiant de la conversation turn_count: int # Nombre de tours de conversation
Avec un TypedDict simple, chaque mise à jour d'un champ remplace la valeur précédente. C'est le comportement par défaut (stratégie LastValue).

2.2 State avec Reducers (Annotated)

Pour des champs qui doivent accumuler des valeurs au lieu de les remplacer, on utilise des reducers via le type Annotated :

Un reducer est une fonction qui définit comment un champ de l'état est mis à jour lorsqu'un noeud retourne une nouvelle valeur pour ce champ. La signature est : (valeur_existante, nouvelle_valeur) -> valeur_finale.
from typing import Annotated from typing_extensions import TypedDict import operator class AgentState(TypedDict): # Champ avec reducer operator.add : les listes sont concaténées messages: Annotated[list, operator.add] # Champ avec reducer operator.add : les entiers sont additionnés total_tokens: Annotated[int, operator.add] # Champ sans reducer : la valeur est remplacée (LastValue) current_step: str

Voici comment les reducers affectent les mises à jour :

# Supposons un état initial : # {"messages": ["bonjour"], "total_tokens": 10, "current_step": "debut"} # Un noeud retourne : # {"messages": ["comment ça va ?"], "total_tokens": 5, "current_step": "milieu"} # Résultat AVEC reducers : # { # "messages": ["bonjour", "comment ça va ?"], ← concaténation (operator.add) # "total_tokens": 15, ← addition (operator.add) # "current_step": "milieu" ← remplacement (LastValue) # }

2.3 Le reducer add_messages

Pour les chatbots, LangGraph fournit un reducer spécialisé add_messages qui gère intelligemment les messages LangChain :

from langgraph.graph.message import add_messages from typing import Annotated from typing_extensions import TypedDict class ChatbotState(TypedDict): # add_messages gère l'ajout et la déduplication des messages messages: Annotated[list, add_messages] user_name: str
Le reducer add_messages est plus intelligent que operator.add pour les listes de messages. Il gère la déduplication (si un message avec le même ID est ajouté, il remplace l'ancien au lieu de le dupliquer) et fonctionne avec les types de messages LangChain (HumanMessage, AIMessage, etc.).

2.4 Tableau des reducers courants

ReducerTypeComportementExemple

3. Noeuds (Nodes)

Un noeud est une fonction Python qui constitue une étape de traitement dans le graphe. Elle reçoit l'état courant complet en paramètre et retourne un dictionnaire partiel contenant uniquement les champs de l'état à mettre à jour.

3.1 Anatomie d'un noeud

def mon_noeud(state: MonState) -> dict: """ Un noeud LangGraph typique. Args: state: L'état complet du graphe (lecture seule) Returns: dict: Un dictionnaire partiel avec les champs à mettre à jour """ # 1. Lire les données de l'état donnees = state["mon_champ"] # 2. Effectuer un traitement resultat = traitement(donnees) # 3. Retourner les mises à jour return {"mon_champ_resultat": resultat}

3.2 Ajouter des noeuds au graphe

add_node(name, function)

  • name (str) : Le nom unique du noeud dans le graphe. Utilisé pour référencer le noeud dans les arêtes.
  • function (callable) : La fonction Python à exécuter. Doit accepter l'état et retourner un dict.
builder = StateGraph(MonState) # Méthode 1 : nom explicite builder.add_node("analyser", analyser_question) # Méthode 2 : le nom est inféré du nom de la fonction builder.add_node(analyser_question) # nom = "analyser_question"

3.3 Exemples de noeuds concrets

Noeud LLM — Appeler un modèle de langage

from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini") def noeud_llm(state: ChatState) -> dict: """Appelle le LLM avec les messages de la conversation.""" messages = state["messages"] reponse = llm.invoke(messages) return {"messages": [reponse]} # Le reducer add_messages concatène

Noeud Outil — Exécuter un outil externe

def noeud_recherche(state: AgentState) -> dict: """Effectue une recherche web basée sur la requête.""" query = state["query"] # Simuler une recherche resultats = [ "Document 1 : LangGraph est un framework...", "Document 2 : Les graphes permettent..." ] return {"documents": resultats, "step": "recherche_terminee"}

Noeud Décision — Préparer le routage

def noeud_evaluation(state: RAGState) -> dict: """Évalue la pertinence des documents récupérés.""" documents = state["documents"] question = state["question"] # Logique d'évaluation score = calculer_pertinence(question, documents) if score > 0.7: decision = "pertinent" else: decision = "non_pertinent" return {"score": score, "decision": decision}
Un noeud ne doit jamais modifier l'état directement. Il doit toujours retourner un nouveau dictionnaire. LangGraph gère la fusion avec l'état existant via les reducers. Modifier l'état directement (ex: state["champ"] = valeur) peut provoquer des comportements imprévisibles.

4. Edges (Arêtes)

Les edges (arêtes) définissent le flux d'exécution entre les noeuds. Ils indiquent à LangGraph quel noeud exécuter après le noeud courant. Il existe des arêtes fixes (toujours le même chemin) et des arêtes conditionnelles (le chemin dépend de l'état).

4.1 Arêtes fixes — add_edge()

add_edge(source, destination)

  • source (str | START) : Le noeud source. Peut être START pour le point d'entrée.
  • destination (str | END) : Le noeud destination. Peut être END pour terminer le graphe.
from langgraph.graph import StateGraph, START, END builder = StateGraph(MonState) # Ajouter les noeuds builder.add_node("etape_1", fonction_etape_1) builder.add_node("etape_2", fonction_etape_2) builder.add_node("etape_3", fonction_etape_3) # Connecter les noeuds avec des arêtes fixes builder.add_edge(START, "etape_1") # Le graphe commence par etape_1 builder.add_edge("etape_1", "etape_2") # etape_1 → etape_2 (toujours) builder.add_edge("etape_2", "etape_3") # etape_2 → etape_3 (toujours) builder.add_edge("etape_3", END) # etape_3 → FIN

4.2 START et END — Les noeuds spéciaux

START et END

  • START : Un noeud virtuel qui représente le point d'entrée du graphe. Il n'exécute aucune fonction. L'arête add_edge(START, "premier_noeud") est obligatoire.
  • END : Un noeud virtuel qui représente la fin du graphe. Quand un noeud est connecté à END, l'exécution s'arrête et l'état final est retourné.

Ces constantes sont importées depuis langgraph.graph.

Oublier de connecter START à un noeud provoquera une erreur à la compilation. De même, chaque chemin du graphe doit éventuellement atteindre END, sinon le graphe ne terminera jamais (sauf en cas de boucle intentionnelle avec condition d'arrêt).

5. Conditional Edges — Arêtes conditionnelles

Les arêtes conditionnelles sont l'une des fonctionnalités les plus puissantes de LangGraph. Elles permettent de router dynamiquement l'exécution en fonction de l'état courant.

5.1 Syntaxe

add_conditional_edges(source, router_function, path_map=None)

  • source (str) : Le noeud après lequel la décision est prise.
  • router_function (callable) : Fonction qui prend l'état et retourne le nom du prochain noeud (ou END).
  • path_map (dict, optionnel) : Mappage entre les valeurs retournées par la router function et les noms de noeuds réels.

5.2 Exemple simple — Routage binaire

from typing import Literal def router(state: MonState) -> Literal["positif", "negatif"]: """Détermine le prochain noeud selon le sentiment.""" if state["sentiment"] == "positif": return "positif" else: return "negatif" # Ajouter l'arête conditionnelle builder.add_conditional_edges( "analyser", # Après ce noeud... router # ...appeler cette fonction pour décider )

5.3 Exemple avec path_map

def router_qualite(state: MonState) -> Literal["bon", "mauvais", "fin"]: """Route selon la qualité du résultat.""" score = state["score"] if score > 0.8: return "bon" elif score > 0.5: return "mauvais" else: return "fin" # Avec un path_map pour mapper les retours aux vrais noms de noeuds builder.add_conditional_edges( "evaluer", router_qualite, { "bon": "generer_reponse", # "bon" → noeud "generer_reponse" "mauvais": "reformuler_question", # "mauvais" → noeud "reformuler_question" "fin": END # "fin" → terminer le graphe } )
L'annotation Literal dans le type de retour de la router function est importante. Elle indique à LangGraph les valeurs possibles et permet de valider le graphe à la compilation. Si la router function retourne une valeur non prévue, une erreur sera levée.

6. Le flux d'exécution

Comprenons en détail comment LangGraph exécute un graphe, étape par étape :

6.1 Cycle d'exécution (Super-step)

LangGraph exécute le graphe par super-steps. Chaque super-step correspond à l'exécution d'un ou plusieurs noeuds en parallèle, suivie de la mise à jour de l'état :

  1. L'exécution commence au noeud START
  2. Les arêtes sortantes de START déterminent le(s) premier(s) noeud(s)
  3. Le(s) noeud(s) s'exécute(nt) et retourne(nt) des mises à jour
  4. L'état est mis à jour via les reducers
  5. Les arêtes sortantes déterminent le(s) prochain(s) noeud(s)
  6. Le cycle continue jusqu'à atteindre END

6.2 Trace d'exécution détaillée

Prenons un exemple concret avec un graphe de 3 noeuds :

from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END class TraceState(TypedDict): valeur: int historique: list[str] def doubler(state: TraceState) -> dict: v = state["valeur"] print(f"[doubler] valeur={v} → {v * 2}") return {"valeur": v * 2, "historique": [f"doublé: {v}→{v*2}"]} def incrementer(state: TraceState) -> dict: v = state["valeur"] print(f"[incrementer] valeur={v} → {v + 1}") return {"valeur": v + 1, "historique": [f"incrémenté: {v}→{v+1}"]} def afficher(state: TraceState) -> dict: print(f"[afficher] Résultat final: {state['valeur']}") print(f"[afficher] Historique: {state['historique']}") return {} builder = StateGraph(TraceState) builder.add_node("doubler", doubler) builder.add_node("incrementer", incrementer) builder.add_node("afficher", afficher) builder.add_edge(START, "doubler") builder.add_edge("doubler", "incrementer") builder.add_edge("incrementer", "afficher") builder.add_edge("afficher", END) graph = builder.compile() resultat = graph.invoke({"valeur": 5, "historique": []})
[doubler] valeur=5 → 10 [incrementer] valeur=10 → 11 [afficher] Résultat final: 11 [afficher] Historique: ['doublé: 5→10', 'incrémenté: 10→11']

7. Compilation

La compilation est l'étape qui transforme un StateGraph (builder) en un CompiledGraph (exécutable). C'est aussi à cette étape que l'on configure les options avancées.

graph.compile(checkpointer=None, interrupt_before=None, interrupt_after=None)

  • checkpointer (BaseCheckpointSaver, optionnel) : Active la persistance de l'état. Chaque super-step est sauvegardé automatiquement.
  • interrupt_before (list[str], optionnel) : Liste de noeuds avant lesquels l'exécution doit s'arrêter (human-in-the-loop).
  • interrupt_after (list[str], optionnel) : Liste de noeuds après lesquels l'exécution doit s'arrêter.

7.1 Compilation simple

# Compilation sans options graph = builder.compile()

7.2 Compilation avec persistance

from langgraph.checkpoint.memory import MemorySaver checkpointer = MemorySaver() graph = builder.compile(checkpointer=checkpointer)

7.3 Compilation avec interruption human-in-the-loop

graph = builder.compile( checkpointer=MemorySaver(), interrupt_before=["executer_action"] # Pause avant ce noeud )
La compilation effectue plusieurs vérifications : existence d'une arête depuis START, pas de noeuds orphelins, cohérence des noms de noeuds dans les arêtes, etc. Si une erreur est détectée, elle est levée à cette étape plutôt qu'à l'exécution — c'est un mécanisme de fail-fast.

8. Invocation — Exécuter le Graphe

8.1 invoke() — Exécution complète

# Invocation simple resultat = graph.invoke({"question": "Qu'est-ce que LangGraph ?"}) print(resultat)

8.2 stream() — Exécution avec streaming

# Streaming par noeud (par défaut : stream_mode="updates") for chunk in graph.stream({"question": "Qu'est-ce que LangGraph ?"}): print(f"Noeud: {list(chunk.keys())[0]}") print(f"Mise à jour: {chunk}") print("---")

8.3 Invocation avec config (persistance)

Quand le graphe est compilé avec un checkpointer, on doit fournir un thread_id dans la configuration pour identifier la conversation :

# Configuration avec thread_id pour la persistance config = {"configurable": {"thread_id": "conversation_1"}} # Premier tour resultat_1 = graph.invoke( {"messages": [("user", "Bonjour !")]}, config ) # Deuxième tour — le graphe se souvient du premier tour resultat_2 = graph.invoke( {"messages": [("user", "De quoi on parlait ?")]}, config )
Grâce au thread_id, LangGraph charge automatiquement l'état sauvegardé du tour précédent. Les messages sont accumulés via le reducer add_messages, créant un historique de conversation persistant.
Si le graphe est compilé avec un checkpointer, le paramètre config avec un thread_id est obligatoire. L'oublier provoquera une erreur.

9. Exemple complet — Classificateur de sentiment

Construisons un graphe complet à 3 noeuds qui classifie le sentiment d'un texte et génère une réponse appropriée en fonction du résultat.

9.1 Architecture du graphe

9.2 Code complet

from typing import Literal from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END # ============================================ # 1. Définition du State # ============================================ class SentimentState(TypedDict): texte: str # Le texte à analyser sentiment: str # "positif" ou "negatif" score: float # Score de confiance reponse: str # La réponse générée # ============================================ # 2. Définition des Noeuds # ============================================ def classifier(state: SentimentState) -> dict: """Noeud 1 : Classifie le sentiment du texte.""" texte = state["texte"].lower() # Mots-clés simples pour la classification mots_positifs = ["super", "génial", "excellent", "bien", "merci", "bravo", "content"] mots_negatifs = ["mauvais", "nul", "terrible", "problème", "erreur", "bug", "triste"] score_pos = sum(1 for mot in mots_positifs if mot in texte) score_neg = sum(1 for mot in mots_negatifs if mot in texte) if score_pos >= score_neg: return {"sentiment": "positif", "score": score_pos / max(score_pos + score_neg, 1)} else: return {"sentiment": "negatif", "score": score_neg / max(score_pos + score_neg, 1)} def repondre_positif(state: SentimentState) -> dict: """Noeud 2a : Génère une réponse pour un sentiment positif.""" return { "reponse": f"Merci pour votre retour positif ! (confiance: {state['score']:.0%}). " f"Nous sommes ravis que vous soyez satisfait." } def repondre_negatif(state: SentimentState) -> dict: """Noeud 2b : Génère une réponse pour un sentiment négatif.""" return { "reponse": f"Nous sommes désolés pour cette expérience (confiance: {state['score']:.0%}). " f"Notre équipe va examiner votre retour." } # ============================================ # 3. Fonction de routage # ============================================ def route_sentiment(state: SentimentState) -> Literal["repondre_positif", "repondre_negatif"]: """Router : dirige vers le noeud approprié selon le sentiment.""" if state["sentiment"] == "positif": return "repondre_positif" else: return "repondre_negatif" # ============================================ # 4. Construction du Graphe # ============================================ builder = StateGraph(SentimentState) # Ajouter les noeuds builder.add_node("classifier", classifier) builder.add_node("repondre_positif", repondre_positif) builder.add_node("repondre_negatif", repondre_negatif) # Arête fixe : START → classifier builder.add_edge(START, "classifier") # Arête conditionnelle : classifier → positif OU négatif builder.add_conditional_edges("classifier", route_sentiment) # Arêtes fixes vers END builder.add_edge("repondre_positif", END) builder.add_edge("repondre_negatif", END) # ============================================ # 5. Compilation et Test # ============================================ graph = builder.compile() # Test 1 : Texte positif resultat1 = graph.invoke({"texte": "Super produit, vraiment excellent !"}) print("Test 1 :", resultat1["reponse"]) # Test 2 : Texte négatif resultat2 = graph.invoke({"texte": "Terrible, le produit a un bug majeur"}) print("Test 2 :", resultat2["reponse"])
Test 1 : Merci pour votre retour positif ! (confiance: 100%). Nous sommes ravis que vous soyez satisfait. Test 2 : Nous sommes désolés pour cette expérience (confiance: 100%). Notre équipe va examiner votre retour.

10.2 Patterns courants

Cheatsheet des patterns

  • Linéaire : START → A → B → C → END
  • Branchement : A → (condition) → B1 | B2 → END
  • Boucle : A → B → (condition) → A (retry) | END
  • Fan-out / Fan-in : A → [B1, B2, B3] → C → END
  • Human-in-the-loop : A → [INTERRUPT] → B → END
