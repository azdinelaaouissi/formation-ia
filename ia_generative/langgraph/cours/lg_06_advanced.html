Fonctionnalités Avancées de LangGraph

Cours 6 — Subgraphs, Persistance, Streaming, Multi-Agents et Déploiement

1. Subgraphs (Sous-graphes)

Un subgraph (sous-graphe) est un graphe LangGraph complet qui est utilisé comme nœud à l'intérieur d'un graphe parent. Cela permet de modulariser des workflows complexes en composants réutilisables et indépendants.

Les subgraphs sont essentiels pour :

  • La modularité — chaque sous-graphe peut être développé et testé séparément
  • La réutilisabilité — un même sous-graphe peut être utilisé dans plusieurs graphes parents
  • L'encapsulation — l'état interne du sous-graphe reste privé
  • La lisibilité — décomposer un workflow complexe en blocs logiques

1.1 Créer et intégrer un subgraph

from langgraph.graph import StateGraph, START, END from typing_extensions import TypedDict from typing import Annotated import operator # --- État du subgraph RAG --- class RAGState(TypedDict): question: str documents: Annotated[list, operator.add] reponse: str def recuperer(state: RAGState) -> dict: docs = vectorstore.similarity_search(state["question"], k=3) return {"documents": [d.page_content for d in docs]} def generer(state: RAGState) -> dict: contexte = "\n".join(state["documents"]) reponse = llm.invoke(f"Contexte: {contexte}\nQuestion: {state['question']}") return {"reponse": reponse.content} # Construire le subgraph rag_builder = StateGraph(RAGState) rag_builder.add_node("recuperer", recuperer) rag_builder.add_node("generer", generer) rag_builder.add_edge(START, "recuperer") rag_builder.add_edge("recuperer", "generer") rag_builder.add_edge("generer", END) rag_subgraph = rag_builder.compile()

1.2 Intégrer le subgraph dans un graphe parent

# --- État du graphe parent --- class ParentState(TypedDict): question: str reponse: str type_requete: str def classifier(state: ParentState) -> dict: # Classifier le type de requête if "document" in state["question"].lower(): return {"type_requete": "rag"} return {"type_requete": "direct"} def reponse_directe(state: ParentState) -> dict: reponse = llm.invoke(state["question"]) return {"reponse": reponse.content} def router(state: ParentState) -> str: return "rag" if state["type_requete"] == "rag" else "direct" # Construire le graphe parent avec le subgraph comme nœud parent_builder = StateGraph(ParentState) parent_builder.add_node("classifier", classifier) parent_builder.add_node("rag", rag_subgraph) # Subgraph compilé comme nœud ! parent_builder.add_node("direct", reponse_directe) parent_builder.add_edge(START, "classifier") parent_builder.add_conditional_edges("classifier", router) parent_builder.add_edge("rag", END) parent_builder.add_edge("direct", END) graph = parent_builder.compile()
Les clés de l'état du subgraph qui portent le même nom que celles du graphe parent sont automatiquement mappées. Si les noms diffèrent, vous devez écrire une fonction wrapper qui traduit les clés entre les deux états. C'est le mécanisme de state mapping.

2. Persistance (Checkpointers)

Un checkpointer est un composant qui sauvegarde l'état du graphe après chaque étape d'exécution. Cela permet de reprendre une exécution interrompue, de mettre en œuvre le human-in-the-loop, et de conserver un historique complet des conversations.

2.1 MemorySaver — Persistance en mémoire

Le checkpointer le plus simple, idéal pour le développement et les tests :

from langgraph.checkpoint.memory import MemorySaver # Créer le checkpointer memory = MemorySaver() # Compiler le graphe avec le checkpointer graph = builder.compile(checkpointer=memory) # Chaque invocation nécessite un thread_id pour identifier la conversation config = {"configurable": {"thread_id": "conversation-1"}} # Première invocation resultat_1 = graph.invoke( {"messages": [HumanMessage(content="Bonjour !")]}, config=config ) # Deuxième invocation — reprend automatiquement l'historique resultat_2 = graph.invoke( {"messages": [HumanMessage(content="Quel était mon premier message ?")]}, config=config ) # L'agent peut voir "Bonjour !" dans l'historique !
MemorySaver stocke tout en RAM. Les données sont perdues quand le processus s'arrête. Ne l'utilisez jamais en production — uniquement pour le développement, les tests et le prototypage.

2.2 SqliteSaver — Persistance locale avec SQLite

from langgraph.checkpoint.sqlite import SqliteSaver # Persistance dans un fichier SQLite with SqliteSaver.from_conn_string("checkpoints.db") as checkpointer: graph = builder.compile(checkpointer=checkpointer) config = {"configurable": {"thread_id": "conv-42"}} resultat = graph.invoke( {"messages": [HumanMessage(content="Bonjour")]}, config=config )

2.3 PostgresSaver — Persistance en production

from langgraph.checkpoint.postgres import PostgresSaver # Connexion à PostgreSQL DB_URI = "postgresql://user:password@localhost:5432/langgraph_db" with PostgresSaver.from_conn_string(DB_URI) as checkpointer: # Créer les tables nécessaires (première fois uniquement) checkpointer.setup() graph = builder.compile(checkpointer=checkpointer) # Utilisation identique aux autres checkpointers config = {"configurable": {"thread_id": "user-123-conv-1"}} resultat = graph.invoke(input_data, config=config)

2.4 Récupérer et inspecter les checkpoints

# Récupérer l'état actuel d'un thread state = graph.get_state(config) print(state.values) # Valeurs actuelles de l'état print(state.next) # Prochain nœud à exécuter (si interrompu) # Lister l'historique complet des checkpoints for checkpoint in graph.get_state_history(config): print(f"Step: {checkpoint.metadata['step']}") print(f"Node: {checkpoint.metadata.get('source', 'N/A')}") print(f"Values: {checkpoint.values}") print("---")
Le thread_id est la clé de voûte de la persistance. Pensez-y comme un identifiant de conversation. Deux utilisateurs différents doivent avoir des thread_id différents pour que leurs historiques restent séparés. En production, utilisez typiquement f"user-{user_id}-conv-{conv_id}".

3. Modes de Streaming

Le streaming permet de recevoir les résultats d'un graphe LangGraph en temps réel, au fur et à mesure que chaque nœud termine son exécution, plutôt que d'attendre la fin complète du graphe.

3.1 Mode values — État complet

Retourne l'état complet après chaque nœud :

# Streaming mode "values" — état complet après chaque nœud for state in graph.stream( {"messages": [HumanMessage(content="Bonjour")]}, config=config, stream_mode="values" ): # state contient l'état COMPLET à ce point print(f"Messages: {len(state['messages'])}") if state["messages"]: print(f"Dernier: {state['messages'][-1].content}")

3.2 Mode updates — Mises à jour partielles

Retourne uniquement les modifications apportées par chaque nœud :

# Streaming mode "updates" — uniquement les changements for update in graph.stream( {"messages": [HumanMessage(content="Bonjour")]}, config=config, stream_mode="updates" ): # update est un dict {nom_du_noeud: {clé: valeur_mise_à_jour}} for node_name, node_output in update.items(): print(f"Nœud '{node_name}' a mis à jour: {list(node_output.keys())}")
Nœud 'classifier' a mis à jour: ['type_requete'] Nœud 'rag_retriever' a mis à jour: ['documents'] Nœud 'rag_generator' a mis à jour: ['reponse']

3.3 Mode debug — Informations détaillées

# Streaming mode "debug" — informations complètes de débogage for event in graph.stream( {"messages": [HumanMessage(content="Bonjour")]}, config=config, stream_mode="debug" ): print(f"Type: {event['type']}") print(f"Timestamp: {event.get('timestamp', 'N/A')}") if event["type"] == "task_result": print(f"Node: {event['payload']['name']}") print(f"Result: {event['payload']['result']}")

3.4 Streaming des tokens LLM

Pour streamer les tokens individuels générés par le LLM :

# astream_events pour les tokens individuels async for event in graph.astream_events( {"messages": [HumanMessage(content="Raconte une histoire")]}, config=config, version="v2" ): if event["event"] == "on_chat_model_stream": token = event["data"]["chunk"].content print(token, end="", flush=True)

4. Tool Calling dans les graphes

Le tool calling (appel d'outils) permet à un nœud LLM de décider dynamiquement d'appeler des fonctions Python pendant l'exécution du graphe. C'est la brique fondamentale pour construire des agents qui interagissent avec le monde extérieur.
from langchain_core.tools import tool from langchain_openai import ChatOpenAI from langgraph.graph import MessagesState, StateGraph, START, END from langgraph.prebuilt import ToolNode # Définir les outils @tool def rechercher_web(query: str) -> str: """Recherche des informations sur le web.""" return f"Résultats pour: {query}..." @tool def calculer(expression: str) -> str: """Calcule une expression mathématique.""" return str(eval(expression)) # Configurer le LLM avec les outils tools = [rechercher_web, calculer] llm = ChatOpenAI(model="gpt-4o").bind_tools(tools) # Créer le nœud d'exécution des outils tool_node = ToolNode(tools) # Nœud agent qui décide d'appeler un outil ou de répondre def agent(state: MessagesState) -> dict: reponse = llm.invoke(state["messages"]) return {"messages": [reponse]} # Router : l'agent a-t-il demandé un outil ? def doit_appeler_outil(state: MessagesState) -> str: dernier_message = state["messages"][-1] if dernier_message.tool_calls: return "tools" return END # Construire le graphe builder = StateGraph(MessagesState) builder.add_node("agent", agent) builder.add_node("tools", tool_node) builder.add_edge(START, "agent") builder.add_conditional_edges("agent", doit_appeler_outil) builder.add_edge("tools", "agent") # Retour à l'agent après l'outil graph = builder.compile()
Ce pattern crée une boucle agent-outil : l'agent réfléchit, appelle un outil si nécessaire, observe le résultat, puis décide de continuer ou de donner sa réponse finale. C'est le pattern ReAct (Reason + Act) implémenté avec LangGraph.
Le ToolNode pré-construit de LangGraph gère automatiquement l'exécution des outils et la conversion des résultats en ToolMessage. Vous n'avez pas besoin de coder cette logique manuellement.

5. Gestion des erreurs et retries

En production, les graphes LangGraph doivent gérer les erreurs de manière robuste. LangGraph offre plusieurs mécanismes : les retry policies (politiques de réessai), les fallbacks (solutions de repli) et la gestion d'erreur dans l'état.

5.1 Retry avec décorateur personnalisé

import time from functools import wraps def retry_node(max_retries: int = 3, delay: float = 1.0): """Décorateur pour ajouter des retries à un nœud.""" def decorator(func): @wraps(func) def wrapper(state): for attempt in range(max_retries): try: return func(state) except Exception as e: if attempt == max_retries - 1: raise time.sleep(delay * (2 ** attempt)) # Backoff exponentiel return wrapper return decorator @retry_node(max_retries=3, delay=1.0) def appel_api_externe(state): reponse = api_client.call(state["query"]) return {"resultat_api": reponse}

5.2 Gestion des erreurs dans l'état

from typing import Annotated, Optional import operator class RobustState(TypedDict): question: str reponse: str erreurs: Annotated[list[str], operator.add] statut: str # "ok", "erreur", "degradé" def noeud_securise(state: RobustState) -> dict: try: reponse = llm.invoke(state["question"]) return { "reponse": reponse.content, "statut": "ok" } except Exception as e: return { "erreurs": [f"Erreur LLM: {str(e)}"], "statut": "erreur", "reponse": "Désolé, une erreur est survenue." } def router_erreur(state: RobustState) -> str: if state["statut"] == "erreur": return "fallback" return "continuer"

5.3 Fallback avec un modèle alternatif

def noeud_fallback(state: RobustState) -> dict: """Modèle de secours moins puissant mais plus fiable.""" try: llm_backup = ChatOpenAI(model="gpt-3.5-turbo", temperature=0) reponse = llm_backup.invoke(state["question"]) return { "reponse": reponse.content, "statut": "dégradé", "erreurs": ["Fallback utilisé: modèle principal indisponible"] } except Exception as e: return { "reponse": "Service temporairement indisponible.", "statut": "erreur", "erreurs": [f"Fallback échoué: {str(e)}"] }
En production, configurez toujours un mécanisme de fallback. Les APIs LLM peuvent avoir des temps de réponse variables, des limites de taux, ou des pannes temporaires. Un bon fallback garantit que votre application reste fonctionnelle.

6. Architectures Multi-Agents

Une architecture multi-agents utilise plusieurs agents LLM spécialisés qui collaborent au sein d'un même graphe. Chaque agent a un rôle défini et les agents communiquent via l'état partagé du graphe.

6.1 Pattern Superviseur

Le superviseur est un agent central qui décide quel agent spécialisé doit travailler :

from langgraph.graph import MessagesState, StateGraph, START, END from typing import Annotated, Literal import operator class MultiAgentState(MessagesState): prochain_agent: str resultats_agents: Annotated[list[dict], operator.add] # Le superviseur décide quel agent appeler def superviseur(state: MultiAgentState) -> dict: system_prompt = """Tu es un superviseur. Analyse la requête et décide quel agent spécialisé doit la traiter : - 'recherche' : pour trouver des informations - 'analyse' : pour analyser des données - 'redaction' : pour rédiger du contenu - 'FINISH' : quand la tâche est terminée""" reponse = llm.invoke([ SystemMessage(content=system_prompt), *state["messages"] ]) return { "messages": [reponse], "prochain_agent": reponse.content.strip() } # Router basé sur la décision du superviseur def router_superviseur(state: MultiAgentState) -> str: agent = state["prochain_agent"] if agent == "FINISH": return END return agent

6.2 Agents spécialisés

def agent_recherche(state: MultiAgentState) -> dict: """Agent spécialisé dans la recherche d'informations.""" prompt = "Tu es un expert en recherche. Trouve les informations pertinentes." reponse = llm_search.invoke([ SystemMessage(content=prompt), *state["messages"] ]) return { "messages": [reponse], "resultats_agents": [{"agent": "recherche", "resultat": reponse.content}] } def agent_analyse(state: MultiAgentState) -> dict: """Agent spécialisé dans l'analyse de données.""" prompt = "Tu es un analyste de données. Analyse les informations fournies." reponse = llm_analyst.invoke([ SystemMessage(content=prompt), *state["messages"] ]) return { "messages": [reponse], "resultats_agents": [{"agent": "analyse", "resultat": reponse.content}] } def agent_redaction(state: MultiAgentState) -> dict: """Agent spécialisé dans la rédaction de contenu.""" prompt = "Tu es un rédacteur expert. Rédige un contenu clair et structuré." reponse = llm_writer.invoke([ SystemMessage(content=prompt), *state["messages"] ]) return { "messages": [reponse], "resultats_agents": [{"agent": "redaction", "resultat": reponse.content}] } # --- Construction du graphe multi-agents --- builder = StateGraph(MultiAgentState) builder.add_node("superviseur", superviseur) builder.add_node("recherche", agent_recherche) builder.add_node("analyse", agent_analyse) builder.add_node("redaction", agent_redaction) builder.add_edge(START, "superviseur") builder.add_conditional_edges("superviseur", router_superviseur) # Chaque agent renvoie au superviseur pour la prochaine décision builder.add_edge("recherche", "superviseur") builder.add_edge("analyse", "superviseur") builder.add_edge("redaction", "superviseur") graph = builder.compile()

6.3 Autres patterns multi-agents

PatternDescriptionCas d'usage
Superviseur Un agent central distribue le travail Tâches complexes nécessitant plusieurs compétences
Séquentiel Les agents travaillent dans un ordre fixe Pipelines : recherche puis analyse puis rédaction
Hiérarchique Superviseurs en cascade, chacun gérant des sous-agents Organisations complexes avec spécialisations imbriquées
Peer-to-Peer Les agents communiquent directement entre eux Débats, revues croisées, consensus
Le pattern superviseur est le plus courant et le plus facile à déboguer. Commencez toujours par celui-ci avant d'envisager des architectures plus complexes. Le superviseur peut utiliser des instructions structurées (structured output) pour garantir des décisions de routing fiables.

7. Déploiement avec LangGraph Cloud

LangGraph Cloud (anciennement LangGraph Platform) est l'infrastructure managée de LangChain pour déployer des graphes LangGraph en production. Elle fournit une API REST, la persistance automatique, le streaming, et un studio de visualisation.

7.1 Structure d'un projet déployable

# Structure de fichiers requise mon_projet/ langgraph.json # Configuration du déploiement mon_agent/ __init__.py agent.py # Le graphe principal tools.py # Les outils state.py # Définition de l'état requirements.txt # Dépendances Python .env # Variables d'environnement

7.2 Fichier langgraph.json

{ "dependencies": ["."], "graphs": { "agent": "./mon_agent/agent.py:graph" }, "env": ".env" }

7.3 Développement local avec LangGraph CLI

7.4 Utiliser l'API du déploiement

from langgraph_sdk import get_client # Se connecter au déploiement client = get_client(url="http://localhost:8123") # Créer un thread (conversation) thread = await client.threads.create() # Lancer une exécution run = await client.runs.create( thread["thread_id"], "agent", # Nom du graphe dans langgraph.json input={"messages": [{"role": "user", "content": "Bonjour !"}]} ) # Streaming via l'API async for chunk in client.runs.stream( thread["thread_id"], "agent", input={"messages": [{"role": "user", "content": "Raconte-moi une histoire"}]}, stream_mode="updates" ): print(chunk)

Avantages de LangGraph Cloud

  • Persistance automatique — PostgreSQL intégré pour les checkpoints
  • Streaming natif — WebSocket et Server-Sent Events
  • LangGraph Studio — Visualisation et débogage interactif
  • Assistants API — Créer plusieurs variantes de votre graphe
  • Cron Jobs — Exécuter des graphes sur un planning
  • Double deployment — Hébergement cloud LangChain ou auto-hébergé

8.2 Checklist de mise en production

1. Persistance — Utiliser PostgresSaver au lieu de MemorySaver
2. Limites — Configurer recursion_limit et des compteurs d'itérations dans l'état
3. Gestion d'erreurs — Try/except dans chaque nœud, fallback pour les appels API

5. Streaming — Utiliser le streaming pour les réponses utilisateur (UX)

9. Agents pré-construits (langgraph.prebuilt)

LangGraph fournit des composants pré-construits pour les cas d'usage courants :

9.1 create_react_agent — Agent ReAct complet

from langgraph.prebuilt import create_react_agent from langchain_openai import ChatOpenAI # Créer un agent ReAct en une seule ligne llm = ChatOpenAI(model="gpt-4o") tools = [rechercher_web, calculer, envoyer_email] agent = create_react_agent( model=llm, tools=tools, checkpointer=MemorySaver() # Optionnel : persistance ) # Utiliser l'agent config = {"configurable": {"thread_id": "test"}} resultat = agent.invoke( {"messages": [HumanMessage(content="Recherche la météo à Paris")]}, config=config )
create_react_agent est parfait pour démarrer rapidement. Il crée automatiquement le graphe agent-outil avec boucle de rétroaction. Pour des besoins plus personnalisés, construisez votre propre graphe comme nous l'avons vu dans les sections précédentes.

9.2 ToolNode — Exécution automatisée des outils

from langgraph.prebuilt import ToolNode, tools_condition # ToolNode exécute automatiquement les tool_calls d'un AIMessage tool_node = ToolNode(tools=[rechercher_web, calculer]) # tools_condition est un router pré-construit # Il retourne "tools" si le dernier message contient des tool_calls # Il retourne END sinon builder = StateGraph(MessagesState) builder.add_node("agent", agent_node) builder.add_node("tools", tool_node) builder.add_edge(START, "agent") builder.add_conditional_edges("agent", tools_condition) builder.add_edge("tools", "agent") graph = builder.compile()

10. Résumé du cours

Points clés à retenir

  • Les subgraphs permettent de modulariser des workflows complexes en composants réutilisables
  • Les checkpointers (MemorySaver, SqliteSaver, PostgresSaver) assurent la persistance de l'état entre les invocations
  • Le streaming offre trois modes (values, updates, debug) adaptés à différents cas d'usage
  • Le tool calling avec ToolNode permet de construire des agents qui interagissent avec le monde extérieur
  • La gestion des erreurs avec retries et fallbacks est indispensable en production
  • Les architectures multi-agents (superviseur, séquentiel, hiérarchique) permettent de résoudre des problèmes complexes
  • LangGraph Cloud fournit une infrastructure complète pour le déploiement en production
  • create_react_agent et ToolNode accélèrent le développement pour les cas courants
Vous disposez maintenant de toutes les briques nécessaires pour construire des applications LLM de niveau production avec LangGraph. La clé du succès est de commencer simple — un graphe linéaire avec MemorySaver — puis d'ajouter progressivement la complexité (subgraphs, multi-agents, PostgreSQL) au fur et à mesure que vos besoins évoluent. Chaque concept de cette série de cours s'emboîte naturellement avec les autres, formant un toolkit complet pour l'orchestration d'agents IA.
