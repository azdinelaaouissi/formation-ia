Gestion de l'État (State Management)

Cours 3 — L'état est le cœur de tout graphe LangGraph

1. Le rôle central de l'état dans LangGraph

L'état (state) dans LangGraph est un objet structuré qui contient toutes les données partagées entre les nœuds du graphe. Chaque nœud lit des valeurs depuis l'état, effectue un traitement, puis retourne un dictionnaire de mises à jour qui sera fusionné dans l'état global.

Dans LangGraph, l'état n'est pas un simple dictionnaire. C'est un schéma typé qui définit :

  • Les clés — quelles données le graphe manipule
  • Les types — quel type Python chaque clé attend
  • Les reducers — comment fusionner les nouvelles valeurs avec les anciennes
Imaginez l'état comme un tableau blanc partagé dans une salle de classe. Chaque nœud est un élève qui peut lire ce qui est écrit, ajouter ses propres notes, et passer le tableau au suivant. Les reducers définissent les règles : est-ce qu'on efface l'ancien texte (remplacement) ou est-ce qu'on ajoute à la suite (accumulation) ?

2. Définir l'état avec TypedDict

La manière la plus courante de définir un état dans LangGraph est d'utiliser TypedDict de la bibliothèque standard Python. C'est un dictionnaire typé qui déclare les clés attendues et leurs types.

from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END # Définition simple de l'état class AgentState(TypedDict): question: str # La question de l'utilisateur contexte: str # Le contexte récupéré reponse: str # La réponse générée score_confiance: float # Score de confiance (0.0 à 1.0)
TypedDict ne fait aucune validation à l'exécution. Il sert uniquement de documentation et d'aide à l'auto-complétion dans votre IDE. Si vous avez besoin d'une validation stricte, utilisez plutôt Pydantic BaseModel (que nous verrons dans la section 7).

2.2 Nœud et mise à jour de l'état

Chaque nœud reçoit l'état complet en entrée et retourne un dictionnaire partiel contenant uniquement les clés à mettre à jour :

def noeud_recherche(state: AgentState) -> dict: # Lire depuis l'état question = state["question"] # Effectuer le traitement contexte = rechercher_documents(question) # Retourner UNIQUEMENT les clés à mettre à jour return {"contexte": contexte} def noeud_generation(state: AgentState) -> dict: question = state["question"] contexte = state["contexte"] reponse = llm.invoke(f"Contexte: {contexte}\nQuestion: {question}") return { "reponse": reponse.content, "score_confiance": 0.85 }
Quand noeud_recherche retourne {"contexte": "..."}, LangGraph fusionne cette valeur dans l'état global. Les autres clés (question, reponse, score_confiance) restent inchangées.

3. Annotated et les Reducers

Un reducer est une fonction qui définit comment une nouvelle valeur retournée par un nœud est fusionnée avec la valeur existante dans l'état. Sans reducer, le comportement par défaut est le remplacement : la nouvelle valeur écrase l'ancienne.

Pour attacher un reducer à un champ, on utilise Annotated de la bibliothèque typing :

from typing import Annotated from typing_extensions import TypedDict import operator class AgentState(TypedDict): # Sans reducer : remplacement (comportement par défaut) question: str # Avec reducer operator.add : accumulation dans une liste messages: Annotated[list, operator.add] # Avec reducer personnalisé compteur: Annotated[int, lambda ancien, nouveau: ancien + nouveau]

3.1 Le reducer operator.add

C'est le reducer le plus utilisé dans LangGraph. Il concatène les listes ensemble au lieu de les remplacer :

from typing import Annotated import operator class ChatState(TypedDict): messages: Annotated[list, operator.add] # Nœud 1 retourne : {"messages": ["Bonjour"]} # État après Nœud 1 : {"messages": ["Bonjour"]} # Nœud 2 retourne : {"messages": ["Comment allez-vous ?"]} # État après Nœud 2 : {"messages": ["Bonjour", "Comment allez-vous ?"]} # operator.add sur des listes = list.__add__ = concaténation
operator.add fonctionne aussi sur les entiers et les chaînes, mais attention au comportement ! Pour des int, c'est une addition arithmétique. Pour des str, c'est une concaténation de texte. Assurez-vous que le type et le reducer correspondent à votre intention.

3.2 Le reducer de remplacement (par défaut)

Sans Annotated, la nouvelle valeur écrase simplement l'ancienne. C'est le comportement le plus intuitif et le plus courant pour les champs scalaires :

class State(TypedDict): reponse: str # Pas de reducer = remplacement # Nœud 1 retourne : {"reponse": "Première version"} # État : {"reponse": "Première version"} # Nœud 2 retourne : {"reponse": "Version améliorée"} # État : {"reponse": "Version améliorée"} (l'ancien est perdu)

3.3 Reducers personnalisés

Vous pouvez écrire votre propre reducer pour n'importe quelle logique de fusion :

def reducer_max_score(ancien: float, nouveau: float) -> float: """Garde toujours le score le plus élevé.""" return max(ancien, nouveau) def reducer_derniers_n(ancien: list, nouveau: list) -> list: """Garde uniquement les 10 derniers éléments.""" combined = ancien + nouveau return combined[-10:] def reducer_union(ancien: set, nouveau: set) -> set: """Fusionne deux ensembles (union).""" return ancien | nouveau class State(TypedDict): meilleur_score: Annotated[float, reducer_max_score] historique: Annotated[list, reducer_derniers_n] sources_vues: Annotated[set, reducer_union]

Règles d'un reducer

Un reducer est une fonction qui prend exactement deux arguments :

  1. ancien — la valeur actuelle dans l'état
  2. nouveau — la valeur retournée par le nœud

Elle retourne la valeur fusionnée qui sera stockée dans l'état. Le reducer doit être déterministe et sans effets de bord.

4. MessagesState — L'état pré-construit pour le chat

MessagesState est un état pré-défini par LangGraph qui contient un champ messages avec un reducer intelligent capable de gérer l'ajout, la suppression et la mise à jour de messages par ID.
from langgraph.graph import MessagesState, StateGraph, START, END from langchain_core.messages import HumanMessage, AIMessage # MessagesState est équivalent à : # class MessagesState(TypedDict): # messages: Annotated[list[AnyMessage], add_messages] class MonState(MessagesState): # On hérite de messages et on ajoute nos propres champs contexte: str score: float

Le reducer add_messages est plus intelligent que operator.add :

Fonctionnalitéoperator.addadd_messages
Ajout de messagesOuiOui
Mise à jour par IDNonOui
Suppression par IDNonOui (avec RemoveMessage)
DéduplicationsNonOui (par ID)
from langchain_core.messages import RemoveMessage def noeud_nettoyage(state: MonState) -> dict: # Supprimer les 5 premiers messages pour libérer le contexte messages_a_supprimer = state["messages"][:5] return { "messages": [RemoveMessage(id=m.id) for m in messages_a_supprimer] }
Utilisez toujours MessagesState quand vous construisez un chatbot ou un agent conversationnel. Il gère automatiquement les subtilités de l'historique de messages, comme la déduplication et la mise à jour partielle. Ne réinventez pas la roue !

5. State Channels (Canaux d'état)

Un canal d'état (state channel) est le mécanisme interne de LangGraph qui gère la lecture, l'écriture et la fusion de chaque champ de l'état. Chaque clé du TypedDict correspond à un canal distinct.

LangGraph utilise trois types de canaux :

1. LastValue Channel — Conserve uniquement la dernière valeur écrite. C'est le canal par défaut pour les champs sans reducer.
2. BinaryOperator Channel — Applique un reducer (fonction binaire) pour fusionner l'ancienne et la nouvelle valeur. Créé automatiquement quand vous utilisez Annotated.
3. Context Channel — Un canal en lecture seule qui est défini une fois à l'invocation du graphe et ne peut pas être modifié par les nœuds.

6. Passer des données entre nœuds

Les nœuds ne communiquent jamais directement entre eux. Toute communication passe par l'état. Voici un exemple complet d'un pipeline RAG où chaque nœud lit et écrit dans l'état partagé :

from typing import Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END import operator class RAGState(TypedDict): question: str documents: Annotated[list, operator.add] reponse: str sources: Annotated[list, operator.add] def recuperer(state: RAGState) -> dict: # Lit 'question', écrit 'documents' et 'sources' docs = vectorstore.similarity_search(state["question"], k=3) return { "documents": [d.page_content for d in docs], "sources": [d.metadata["source"] for d in docs] } def generer(state: RAGState) -> dict: # Lit 'question' et 'documents', écrit 'reponse' contexte = "\n".join(state["documents"]) prompt = f"Contexte:\n{contexte}\n\nQuestion: {state['question']}" reponse = llm.invoke(prompt) return {"reponse": reponse.content} # Construction du graphe builder = StateGraph(RAGState) builder.add_node("recuperer", recuperer) builder.add_node("generer", generer) builder.add_edge(START, "recuperer") builder.add_edge("recuperer", "generer") builder.add_edge("generer", END) graph = builder.compile() # Invocation resultat = graph.invoke({"question": "Qu'est-ce que LangGraph ?"})
{'question': "Qu'est-ce que LangGraph ?", 'documents': ['LangGraph est un framework...', 'Il permet de créer...', 'Basé sur les graphes...'], 'reponse': 'LangGraph est un framework Python qui...', 'sources': ['doc1.pdf', 'doc2.pdf', 'doc3.pdf']}

Principe fondamental

Un nœud n'a aucune connaissance des autres nœuds. Il connaît uniquement la structure de l'état. C'est ce qui rend les nœuds modulaires et réutilisables : vous pouvez réorganiser le graphe sans toucher au code des nœuds.

7. Validation de l'état avec Pydantic

Pour une validation stricte à l'exécution, vous pouvez utiliser un modèle Pydantic au lieu d'un TypedDict :

from pydantic import BaseModel, Field, field_validator from typing import Annotated import operator class AgentStateValide(BaseModel): question: str = Field( description="La question de l'utilisateur", min_length=1, max_length=1000 ) messages: Annotated[list, operator.add] = Field( default_factory=list, description="Historique des messages" ) temperature: float = Field( default=0.7, ge=0.0, le=2.0, description="Température du LLM" ) nb_resultats: int = Field( default=5, ge=1, le=20, description="Nombre de documents à récupérer" ) @field_validator("question") @classmethod def question_non_vide(cls, v): if not v.strip(): raise ValueError("La question ne peut pas être vide") return v.strip()
La validation Pydantic est particulièrement utile en production où les entrées utilisateur sont imprévisibles. En phase de prototypage, TypedDict suffit largement et offre une meilleure performance (pas de validation à chaque écriture).

8. État imbriqué (Nested State)

Pour des applications complexes, vous pouvez structurer votre état avec des sous-objets imbriqués :

from typing_extensions import TypedDict from typing import Annotated import operator class UserInfo(TypedDict): nom: str email: str langue: str class SearchResult(TypedDict): titre: str contenu: str score: float class AppState(TypedDict): # Sous-objet imbriqué utilisateur: UserInfo # Liste d'objets structurés avec accumulation resultats: Annotated[list[SearchResult], operator.add] # Champs simples question: str reponse: str
Quand vous utilisez un état imbriqué sans reducer, la mise à jour d'un sous-champ nécessite de retourner l'objet imbriqué en entier. LangGraph ne fait pas de fusion profonde (deep merge) par défaut.
# CORRECT : retourner l'objet imbriqué complet def mettre_a_jour_langue(state: AppState) -> dict: user = state["utilisateur"].copy() # Copier l'objet existant user["langue"] = "fr" # Modifier le champ voulu return {"utilisateur": user} # Retourner l'objet complet # INCORRECT : retourner uniquement le sous-champ # return {"utilisateur": {"langue": "fr"}} # Écrase nom et email !

8.1 Pattern : Deep Merge avec un reducer personnalisé

Si vous souhaitez une fusion profonde automatique, créez un reducer dédié :

def deep_merge(ancien: dict, nouveau: dict) -> dict: """Fusionne récursivement deux dictionnaires.""" resultat = ancien.copy() for cle, valeur in nouveau.items(): if cle in resultat and isinstance(resultat[cle], dict) and isinstance(valeur, dict): resultat[cle] = deep_merge(resultat[cle], valeur) else: resultat[cle] = valeur return resultat class AppState(TypedDict): utilisateur: Annotated[dict, deep_merge] # Maintenant on peut faire : # return {"utilisateur": {"langue": "fr"}} # Fusionne sans écraser !

9. Schémas d'entrée et de sortie

LangGraph permet de définir des schémas distincts pour l'entrée et la sortie du graphe, ce qui est essentiel pour les APIs et la documentation :

class InputState(TypedDict): # Ce que l'utilisateur fournit question: str class OutputState(TypedDict): # Ce que le graphe retourne reponse: str sources: list[str] class OverallState(TypedDict): # L'état complet interne (superset des deux) question: str documents: list[str] reponse: str sources: list[str] score_interne: float # Pas exposé à l'utilisateur # Spécifier les schémas au moment de la construction builder = StateGraph(OverallState, input=InputState, output=OutputState) # L'invocation ne retournera que les champs de OutputState resultat = graph.invoke({"question": "..."}) # resultat = {"reponse": "...", "sources": [...]} # score_interne et documents ne sont PAS exposés
Les schémas d'entrée/sortie permettent de masquer les détails d'implémentation internes et de fournir une interface propre aux consommateurs du graphe. C'est une bonne pratique pour tout graphe destiné à être utilisé comme API.

10. Patterns courants de gestion d'état

10.1 Pattern Compteur d'itérations

Ce pattern est essentiel pour limiter les boucles dans les graphes cycliques :

class State(TypedDict): messages: Annotated[list, operator.add] iteration: Annotated[int, lambda a, b: a + b] max_iterations: int def agent(state: State) -> dict: # Incrémenter le compteur à chaque passage return { "messages": ["Action effectuée"], "iteration": 1 # Le reducer fait : ancien + 1 } def doit_continuer(state: State) -> str: if state["iteration"] >= state["max_iterations"]: return "fin" return "continuer"

10.2 Pattern Collecteur de résultats

class State(TypedDict): taches: list[str] resultats: Annotated[list[dict], operator.add] erreurs: Annotated[list[str], operator.add] def executer_tache(state: State) -> dict: try: resultat = traiter(state["taches"][0]) return {"resultats": [{"tache": state["taches"][0], "succes": True, "data": resultat}]} except Exception as e: return {"erreurs": [f"Erreur sur {state['taches'][0]}: {e}"]}

10.3 Pattern Drapeaux de contrôle

class State(TypedDict): question: str reponse: str # Drapeaux de contrôle besoin_recherche: bool est_valide: bool langue_detectee: str def classifier(state: State) -> dict: # Analyser la question et définir les drapeaux return { "besoin_recherche": "recherche" in state["question"].lower(), "langue_detectee": detecter_langue(state["question"]) } def router(state: State) -> str: if state["besoin_recherche"]: return "recherche" return "generation_directe"

11. Exemple complet : Agent avec gestion d'état avancée

Voici un exemple qui combine tous les concepts vus dans ce cours :

from typing import Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END import operator # --- Reducer personnalisé --- def garder_meilleur(ancien: float, nouveau: float) -> float: return max(ancien, nouveau) # --- Définition de l'état --- class AgentState(TypedDict): question: str # Remplacement messages: Annotated[list, operator.add] # Accumulation documents: Annotated[list, operator.add] # Accumulation reponse: str # Remplacement meilleur_score: Annotated[float, garder_meilleur] # Garde le max iteration: Annotated[int, lambda a, b: a + b] # Compteur # --- Nœuds --- def rechercher(state: AgentState) -> dict: docs = ["Doc pertinent sur " + state["question"]] return { "documents": docs, "messages": ["Recherche effectuée"], "iteration": 1 } def evaluer(state: AgentState) -> dict: score = 0.75 if len(state["documents"]) > 0 else 0.2 return { "meilleur_score": score, "messages": [f"Score évalué : {score}"] } def generer(state: AgentState) -> dict: return { "reponse": f"Réponse basée sur {len(state['documents'])} documents", "messages": ["Réponse générée"] } def router(state: AgentState) -> str: if state["meilleur_score"] < 0.5 and state["iteration"] < 3: return "rechercher" # Boucle : re-rechercher return "generer" # --- Construction du graphe --- builder = StateGraph(AgentState) builder.add_node("rechercher", rechercher) builder.add_node("evaluer", evaluer) builder.add_node("generer", generer) builder.add_edge(START, "rechercher") builder.add_edge("rechercher", "evaluer") builder.add_conditional_edges("evaluer", router) builder.add_edge("generer", END) graph = builder.compile() # --- Invocation --- resultat = graph.invoke({ "question": "Comment fonctionne LangGraph ?", "meilleur_score": 0.0, "iteration": 0 }) print(resultat["reponse"]) print(f"Itérations : {resultat['iteration']}") print(f"Messages : {resultat['messages']}")
Réponse basée sur 1 documents Itérations : 1 Messages : ['Recherche effectuée', 'Score évalué : 0.75', 'Réponse générée']

12. Résumé du cours

Points clés à retenir

  • L'état est le cœur de LangGraph — toutes les données transitent par lui
  • TypedDict est le moyen standard de définir un schéma d'état
  • Annotated permet d'attacher un reducer à un champ pour contrôler la fusion
  • operator.add est le reducer le plus courant pour accumuler des listes
  • Sans reducer, le comportement est le remplacement de l'ancienne valeur
  • MessagesState fournit un état pré-construit pour les chatbots
  • Les schémas input/output permettent de masquer les détails internes
  • La validation Pydantic est recommandée pour la production
  • Les nœuds communiquent uniquement à travers l'état, jamais directement
Maîtriser la gestion de l'état est la compétence la plus importante dans LangGraph. Un état bien conçu rend votre graphe modulaire, testable et facile à déboguer.
