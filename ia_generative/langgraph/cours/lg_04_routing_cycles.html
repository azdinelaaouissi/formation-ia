Routing Conditionnel et Cycles

Logique de Decision dans LangGraph - Branchement, Boucles et Patterns Avances

1. Routing Conditionnel

Le routing conditionnel est le mecanisme par lequel un graphe LangGraph decide dynamiquement quel noeud executer ensuite, en fonction de l'etat courant. C'est la methode add_conditional_edges() qui rend cela possible.

Contrairement aux aretes normales (add_edge) qui definissent un chemin fixe entre deux noeuds, les aretes conditionnelles evaluent une fonction de routage a chaque execution. Cette fonction recoit l'etat actuel du graphe et retourne le nom du prochain noeud (ou une liste de noeuds) a executer.

1.1 Signature de add_conditional_edges()

Parametres de add_conditional_edges()

  • source (str) : Le noeud source apres lequel la decision est prise
  • path (Callable) : La fonction de routage qui recoit le state et retourne un string (nom du noeud suivant)
  • path_map (dict, optionnel) : Mappage entre les valeurs retournees par la fonction et les noms reels des noeuds
from langgraph.graph import StateGraph, END from typing import TypedDict, Literal class State(TypedDict): question: str category: str response: str # Fonction de routage : decide le prochain noeud def route_by_category(state: State) -> Literal["handle_tech", "handle_billing", "handle_general"]: """Route la question vers le bon gestionnaire.""" category = state["category"] if category == "technique": return "handle_tech" elif category == "facturation": return "handle_billing" else: return "handle_general" # Noeuds du graphe def classify_question(state: State) -> dict: # Logique de classification (LLM ou regles) question = state["question"].lower() if "bug" in question or "erreur" in question: return {"category": "technique"} elif "facture" in question or "prix" in question: return {"category": "facturation"} return {"category": "general"} def handle_tech(state: State) -> dict: return {"response": "Equipe technique contactee pour votre probleme."} def handle_billing(state: State) -> dict: return {"response": "Service facturation notifie."} def handle_general(state: State) -> dict: return {"response": "Reponse generale fournie."} # Construction du graphe builder = StateGraph(State) builder.add_node("classify", classify_question) builder.add_node("handle_tech", handle_tech) builder.add_node("handle_billing", handle_billing) builder.add_node("handle_general", handle_general) builder.set_entry_point("classify") # Arete conditionnelle : apres "classify", appeler route_by_category builder.add_conditional_edges( "classify", route_by_category ) # Tous les gestionnaires menent a END builder.add_edge("handle_tech", END) builder.add_edge("handle_billing", END) builder.add_edge("handle_general", END) graph = builder.compile() # Execution result = graph.invoke({"question": "J'ai un bug dans l'application"}) print(result["response"])
Equipe technique contactee pour votre probleme.

1.2 Diagramme du Routing Conditionnel

La fonction de routage doit toujours retourner un string correspondant a un nom de noeud valide dans le graphe, ou bien END pour terminer l'execution. Utilisez les Literal type hints pour que LangGraph Studio puisse visualiser les chemins possibles.

1.3 Utilisation de path_map

Parfois, les valeurs retournees par votre fonction de routage ne correspondent pas directement aux noms des noeuds. Le parametre path_map permet de faire la traduction :

# La fonction retourne des labels lisibles def route_by_sentiment(state: State) -> str: sentiment = state["sentiment"] if sentiment > 0.7: return "positif" elif sentiment < 0.3: return "negatif" return "neutre" # path_map traduit les retours en noms de noeuds builder.add_conditional_edges( "analyze_sentiment", route_by_sentiment, { "positif": "respond_positive", "negatif": "escalate_to_human", "neutre": "respond_neutral" } )
Le path_map offre une couche d'abstraction entre la logique metier (sentiments) et l'architecture du graphe (noms de noeuds). Cela rend le code plus lisible et maintenable.

2. Patterns de Routing

Le routing conditionnel se decline en plusieurs patterns recurrents dans les applications d'IA. Voici les plus courants avec leur implementation complete.

2.1 Classification Routing (par sujet)

Le pattern le plus courant : classifier une entree utilisateur puis la diriger vers le bon gestionnaire specialise.

from langgraph.graph import StateGraph, END from typing import TypedDict, Literal from langchain_openai import ChatOpenAI class TopicState(TypedDict): question: str topic: str answer: str llm = ChatOpenAI(model="gpt-4o-mini") def classify_topic(state: TopicState) -> dict: """Utilise un LLM pour classifier le sujet.""" prompt = f"""Classifie cette question dans UNE des categories : - science - histoire - programmation Question : {state['question']} Reponds uniquement par le nom de la categorie.""" response = llm.invoke(prompt) topic = response.content.strip().lower() return {"topic": topic} def route_topic(state: TopicState) -> Literal["science_expert", "history_expert", "code_expert"]: topic = state["topic"] routes = { "science": "science_expert", "histoire": "history_expert", "programmation": "code_expert" } return routes.get(topic, "science_expert") def science_expert(state: TopicState) -> dict: response = llm.invoke(f"En tant qu'expert scientifique, reponds : {state['question']}") return {"answer": response.content} def history_expert(state: TopicState) -> dict: response = llm.invoke(f"En tant qu'historien, reponds : {state['question']}") return {"answer": response.content} def code_expert(state: TopicState) -> dict: response = llm.invoke(f"En tant qu'expert en programmation, reponds : {state['question']}") return {"answer": response.content} # Assemblage builder = StateGraph(TopicState) builder.add_node("classify", classify_topic) builder.add_node("science_expert", science_expert) builder.add_node("history_expert", history_expert) builder.add_node("code_expert", code_expert) builder.set_entry_point("classify") builder.add_conditional_edges("classify", route_topic) builder.add_edge("science_expert", END) builder.add_edge("history_expert", END) builder.add_edge("code_expert", END) graph = builder.compile()

2.2 Quality Check Routing (boucle de qualite)

Ce pattern verifie la qualite d'une reponse et decide de la renvoyer pour amelioration ou de la valider.

from typing import Annotated from langgraph.graph import StateGraph, END class QualityState(TypedDict): question: str draft: str quality_score: float attempts: int final_answer: str def generate_draft(state: QualityState) -> dict: """Genere ou regenere une reponse.""" attempts = state.get("attempts", 0) + 1 response = llm.invoke( f"Reponds a cette question de maniere detaillee : {state['question']}" ) return {"draft": response.content, "attempts": attempts} def check_quality(state: QualityState) -> dict: """Evalue la qualite de la reponse.""" prompt = f"""Evalue cette reponse sur une echelle de 0 a 1. Question : {state['question']} Reponse : {state['draft']} Reponds UNIQUEMENT par un nombre decimal.""" result = llm.invoke(prompt) score = float(result.content.strip()) return {"quality_score": score} def finalize(state: QualityState) -> dict: return {"final_answer": state["draft"]} def quality_router(state: QualityState) -> Literal["generate_draft", "finalize"]: """Si qualite insuffisante et tentatives restantes, on regenere.""" if state["quality_score"] >= 0.8: return "finalize" if state.get("attempts", 0) >= 3: return "finalize" # Limite atteinte, on accepte return "generate_draft" # On retente builder = StateGraph(QualityState) builder.add_node("generate_draft", generate_draft) builder.add_node("check_quality", check_quality) builder.add_node("finalize", finalize) builder.set_entry_point("generate_draft") builder.add_edge("generate_draft", "check_quality") builder.add_conditional_edges("check_quality", quality_router) builder.add_edge("finalize", END) graph = builder.compile()
Ce pattern de Quality Check est extremement puissant. Il combine routing conditionnel et cycles pour creer un systeme d'auto-amelioration. C'est un pattern fondamental dans la construction d'agents fiables.

3. Cycles (Boucles)

Un cycle dans LangGraph est une boucle ou un noeud peut etre visite plusieurs fois au cours d'une meme execution. C'est la fonctionnalite qui distingue LangGraph des pipelines lineaires classiques comme les Chains de LangChain.

Les cycles permettent de modeliser des comportements iteratifs : un agent qui reflechit, agit, observe le resultat, et decide de continuer ou de s'arreter. Sans cycles, il serait impossible de construire des agents autonomes capables de raisonnement multi-etapes.

3.2 Implementation d'un Cycle Simple

from langgraph.graph import StateGraph, END from typing import TypedDict, Literal, List class AgentState(TypedDict): task: str plan: str result: str is_complete: bool iterations: int history: List[str] def plan_step(state: AgentState) -> dict: iteration = state.get("iterations", 0) + 1 history = state.get("history", []) if iteration == 1: plan = f"Plan initial pour : {state['task']}" else: plan = f"Plan revise (iteration {iteration}) base sur : {state.get('result', '')}" history.append(f"[Plan {iteration}] {plan}") return {"plan": plan, "iterations": iteration, "history": history} def execute_step(state: AgentState) -> dict: result = f"Execution du plan : {state['plan']}" history = state["history"] + [f"[Exec] {result}"] return {"result": result, "history": history} def check_step(state: AgentState) -> dict: # Simuler une verification (en production, un LLM evaluerait) is_done = state["iterations"] >= 3 return {"is_complete": is_done} def should_continue(state: AgentState) -> Literal["plan", "__end__"]: if state["is_complete"]: return "__end__" return "plan" builder = StateGraph(AgentState) builder.add_node("plan", plan_step) builder.add_node("execute", execute_step) builder.add_node("check", check_step) builder.set_entry_point("plan") builder.add_edge("plan", "execute") builder.add_edge("execute", "check") builder.add_conditional_edges("check", should_continue) graph = builder.compile() result = graph.invoke({"task": "Analyser les donnees de vente"}) print(f"Iterations : {result['iterations']}") for entry in result["history"]: print(entry)
Iterations : 3 [Plan 1] Plan initial pour : Analyser les donnees de vente [Exec] Execution du plan : Plan initial pour : Analyser les donnees de vente [Plan 2] Plan revise (iteration 2) base sur : Execution du plan : Plan initial pour : Analyser les donnees de vente [Exec] Execution du plan : Plan revise (iteration 2)... [Plan 3] Plan revise (iteration 3)... [Exec] Execution du plan : Plan revise (iteration 3)...

4. Max Iterations et recursion_limit

Les cycles sans condition d'arret valide peuvent provoquer des boucles infinies. LangGraph fournit un mecanisme de securite : le parametre recursion_limit. Par defaut, la limite est de 25 etapes.

4.1 Configuration de recursion_limit

# Methode 1 : Dans la configuration d'invocation config = {"recursion_limit": 50} result = graph.invoke( {"task": "Tache complexe"}, config=config ) # Methode 2 : Limiter strictement pour eviter des couts excessifs config = {"recursion_limit": 10} try: result = graph.invoke({"task": "Tache"}, config=config) except GraphRecursionError as e: print(f"Limite de recursion atteinte : {e}") # Gerer gracieusement : retourner le dernier etat valide

4.2 Strategies Anti-Boucle Infinie

Trois lignes de defense contre les boucles infinies

  1. Compteur d'iterations dans le state : Votre logique metier decide quand s'arreter (recommande)
  2. recursion_limit dans config : Filet de securite global au niveau du graphe
  3. Timeouts : Limite de temps d'execution pour les cas extremes
# Approche robuste : combiner compteur interne + recursion_limit class SafeState(TypedDict): task: str iterations: int max_iterations: int # Limite metier result: str def should_continue_safe(state: SafeState) -> Literal["process", "__end__"]: """Double verification avant de continuer.""" if state["iterations"] >= state["max_iterations"]: return "__end__" if "TERMINE" in state.get("result", ""): return "__end__" return "process" # Execution avec double securite result = graph.invoke( {"task": "Tache", "max_iterations": 5, "iterations": 0}, config={"recursion_limit": 20} # Filet de securite )
En production, privilegiez toujours un compteur dans le state plutot que de vous reposer uniquement sur recursion_limit. Le compteur vous donne un controle precis et permet de retourner un resultat partiel, tandis que recursion_limit leve une exception brute.

5. Agent ReAct Pattern

Le pattern ReAct (Reasoning + Acting) est un paradigme d'agent ou le LLM alterne entre reflexion (raisonnement sur la situation), action (appel d'outil), et observation (analyse du resultat). Ce cycle se repete jusqu'a obtenir une reponse finale.

5.2 Implementation Complete du ReAct Agent

from langgraph.graph import StateGraph, END from langgraph.prebuilt import ToolNode from langchain_openai import ChatOpenAI from langchain_core.messages import HumanMessage, AIMessage, ToolMessage from langchain_core.tools import tool from typing import TypedDict, Annotated, Sequence import operator import json # --- Definition des outils --- @tool def search_web(query: str) -> str: """Recherche sur le web pour trouver des informations.""" # Simule une recherche return f"Resultats pour '{query}': LangGraph est un framework de LangChain pour creer des agents a base de graphes." @tool def calculator(expression: str) -> str: """Evalue une expression mathematique.""" try: result = eval(expression) return f"Resultat : {result}" except: return "Erreur dans l'expression" tools = [search_web, calculator] # --- State avec liste de messages --- class AgentState(TypedDict): messages: Annotated[Sequence[any], operator.add] # --- LLM avec outils lies --- llm = ChatOpenAI(model="gpt-4o-mini") llm_with_tools = llm.bind_tools(tools) # --- Noeud agent : le LLM reflechit --- def agent_node(state: AgentState) -> dict: """Le LLM raisonne et decide d'utiliser un outil ou de repondre.""" response = llm_with_tools.invoke(state["messages"]) return {"messages": [response]} # --- Noeud outils : execute les appels d'outils --- tool_node = ToolNode(tools) # --- Router : outil appele ou reponse finale ? --- def should_use_tool(state: AgentState) -> Literal["tools", "__end__"]: last_message = state["messages"][-1] if hasattr(last_message, "tool_calls") and last_message.tool_calls: return "tools" return "__end__" # --- Construction du graphe ReAct --- builder = StateGraph(AgentState) builder.add_node("agent", agent_node) builder.add_node("tools", tool_node) builder.set_entry_point("agent") # Apres l'agent : soit on appelle un outil, soit on termine builder.add_conditional_edges("agent", should_use_tool) # Apres l'outil, on retourne a l'agent (le cycle !) builder.add_edge("tools", "agent") graph = builder.compile() # --- Execution --- result = graph.invoke( {"messages": [HumanMessage(content="Qu'est-ce que LangGraph ?")]}, config={"recursion_limit": 15} ) for msg in result["messages"]: print(f"{msg.__class__.__name__}: {msg.content[:100]}")
HumanMessage: Qu'est-ce que LangGraph ? AIMessage: [tool_call: search_web("LangGraph")] ToolMessage: Resultats pour 'LangGraph': LangGraph est un framework... AIMessage: LangGraph est un framework de LangChain pour creer des agents a base de graphes...

Alternative : create_react_agent() prebuilt

LangGraph fournit une fonction utilitaire pour creer rapidement un agent ReAct sans construire le graphe manuellement :

from langgraph.prebuilt import create_react_agent agent = create_react_agent( model=ChatOpenAI(model="gpt-4o"), tools=[search_web, calculator] ) result = agent.invoke({ "messages": [HumanMessage(content="Calcule 25 * 48 + 13")] })

6. Multi-path Routing (Fan-out / Fan-in)

Le multi-path routing permet d'envoyer l'etat vers plusieurs noeuds simultanement (fan-out), qui s'executent en parallele, puis de fusionner leurs resultats (fan-in). La fonction de routage retourne alors une liste de noeuds au lieu d'un seul.

6.1 Fan-out avec Send API

LangGraph utilise l'objet Send pour le fan-out dynamique. Chaque Send cree une execution parallele avec son propre etat :

from langgraph.graph import StateGraph, END from langgraph.types import Send from typing import TypedDict, List, Annotated import operator class FanOutState(TypedDict): topic: str analyses: Annotated[List[str], operator.add] class AnalysisState(TypedDict): topic: str perspective: str analyses: Annotated[List[str], operator.add] def create_analyses(state: FanOutState) -> list: """Fan-out : creer des analyses paralleles pour chaque perspective.""" perspectives = ["technique", "business", "ethique"] return [ Send("analyze", { "topic": state["topic"], "perspective": p, "analyses": [] }) for p in perspectives ] def analyze(state: AnalysisState) -> dict: """Chaque analyse s'execute en parallele.""" result = f"Analyse {state['perspective']} de '{state['topic']}' : [resultats detailles]" return {"analyses": [result]} def synthesize(state: FanOutState) -> dict: """Fan-in : fusionner toutes les analyses.""" summary = "Synthese de {} analyses :\n".format(len(state["analyses"])) for a in state["analyses"]: summary += f" - {a}\n" return {"analyses": [summary]} builder = StateGraph(FanOutState) builder.add_node("analyze", analyze) builder.add_node("synthesize", synthesize) # Fan-out conditionnel avec Send builder.add_conditional_edges("__start__", create_analyses) # Fan-in : toutes les analyses convergent vers synthesize builder.add_edge("analyze", "synthesize") builder.add_edge("synthesize", END) graph = builder.compile() result = graph.invoke({"topic": "Intelligence Artificielle Generative"}) print(result["analyses"][-1])
Synthese de 3 analyses : - Analyse technique de 'Intelligence Artificielle Generative' : [resultats detailles] - Analyse business de 'Intelligence Artificielle Generative' : [resultats detailles] - Analyse ethique de 'Intelligence Artificielle Generative' : [resultats detailles]

6.2 Fan-out Statique (retourner une liste de noeuds)

# Une fonction de routage peut retourner une liste de noeuds def multi_route(state: State) -> list: """Envoie l'etat a plusieurs noeuds simultanement.""" nodes = ["check_spelling", "check_grammar"] if state.get("needs_fact_check"): nodes.append("fact_check") return nodes builder.add_conditional_edges("draft", multi_route)
Les noeuds retournes s'executent en parallele dans le meme superstep. LangGraph synchronise automatiquement leurs resultats avant de passer au superstep suivant. Les reducers (comme operator.add) fusionnent les mises a jour concurrentes.

7. Error Handling dans les Graphes

La gestion des erreurs dans LangGraph est critique car les agents interagissent avec des services externes (LLMs, APIs, bases de donnees) qui peuvent echouer a tout moment.

7.1 Try/Except dans les Noeuds

from langchain_core.runnables import RunnableConfig class RobustState(TypedDict): query: str result: str error: str retry_count: int def call_api_node(state: RobustState) -> dict: """Noeud avec gestion d'erreur integree.""" retry = state.get("retry_count", 0) try: # Appel a un service externe response = llm.invoke(state["query"]) return { "result": response.content, "error": "", "retry_count": retry } except Exception as e: return { "result": "", "error": str(e), "retry_count": retry + 1 } def error_router(state: RobustState) -> Literal["call_api", "fallback", "__end__"]: """Route en fonction du succes/echec.""" if not state.get("error"): return "__end__" # Succes if state["retry_count"] < 3: return "call_api" # Retry return "fallback" # Abandon, utiliser le fallback def fallback_node(state: RobustState) -> dict: return { "result": "Desole, le service est temporairement indisponible. Veuillez reessayer plus tard.", "error": "" } builder = StateGraph(RobustState) builder.add_node("call_api", call_api_node) builder.add_node("fallback", fallback_node) builder.set_entry_point("call_api") builder.add_conditional_edges("call_api", error_router) builder.add_edge("fallback", END) graph = builder.compile()

7.2 Pattern de Retry avec Backoff Exponentiel

import time import random def call_with_retry(state: RobustState) -> dict: """Retry avec backoff exponentiel.""" retry = state.get("retry_count", 0) if retry > 0: # Backoff exponentiel avec jitter wait_time = min(2 ** retry + random.uniform(0, 1), 30) time.sleep(wait_time) try: result = llm.invoke(state["query"]) return {"result": result.content, "error": "", "retry_count": retry} except Exception as e: return {"error": str(e), "retry_count": retry + 1}
N'utilisez time.sleep() qu'avec moderation dans les noeuds. En production, preferez des mecanismes de retry au niveau du client HTTP (comme tenacity ou les retries integres dans les clients LangChain).

8. Streaming Events

LangGraph offre plusieurs modes de streaming qui permettent de suivre l'execution du graphe en temps reel. Cela est essentiel pour les interfaces utilisateur interactives.

8.1 Les Modes de Streaming

Mode Description Cas d'usage
"values" Emet l'etat complet apres chaque noeud Debug, visualisation d'etat complet
"updates" Emet uniquement les changements (deltas) Performance, UI reactives
"messages" Emet les messages de chat token par token Chatbots, reponses en streaming
"custom" Evenements personnalises emis par les noeuds Suivi de progression, logs

8.2 stream_mode="values"

# Mode "values" : etat complet a chaque etape for state in graph.stream( {"messages": [HumanMessage(content="Bonjour")]}, stream_mode="values" ): # state contient TOUT l'etat du graphe print("=== Etat complet ===") for msg in state["messages"]: print(f" {msg.__class__.__name__}: {msg.content[:80]}") print()
=== Etat complet === HumanMessage: Bonjour === Etat complet === HumanMessage: Bonjour AIMessage: Bonjour ! Comment puis-je vous aider aujourd'hui ?

8.3 stream_mode="updates"

# Mode "updates" : uniquement les deltas for update in graph.stream( {"messages": [HumanMessage(content="Calcule 2+2")]}, stream_mode="updates" ): # update est un dict {nom_noeud: changements} for node_name, changes in update.items(): print(f"Noeud '{node_name}' a produit :") if "messages" in changes: for msg in changes["messages"]: print(f" -> {msg.content[:80]}")
Noeud 'agent' a produit : -> [Appel outil: calculator("2+2")] Noeud 'tools' a produit : -> Resultat : 4 Noeud 'agent' a produit : -> 2 + 2 = 4

8.4 stream_mode="messages"

# Mode "messages" : streaming token par token (ideal pour chatbots) for msg, metadata in graph.stream( {"messages": [HumanMessage(content="Explique la relativite")]}, stream_mode="messages" ): # msg est un chunk de message (AIMessageChunk) if msg.content: print(msg.content, end="", flush=True) print() # Nouvelle ligne a la fin
La relativite, proposee par Albert Einstein, comprend deux theories majeures... (tokens emis un par un en temps reel)

8.5 Combinaison de Modes

# On peut combiner plusieurs modes simultanement for event in graph.stream( {"messages": [HumanMessage(content="Bonjour")]}, stream_mode=["updates", "messages"] ): # Chaque event est un tuple (mode, data) mode, data = event if mode == "updates": print(f"[UPDATE] {data}") elif mode == "messages": msg, meta = data if msg.content: print(f"[MSG] {msg.content}", end="")
Pour les chatbots, utilisez stream_mode="messages" qui offre la meilleure experience utilisateur avec le streaming token par token. Pour le debugging, "values" est ideal car il montre l'etat complet. Pour les tableaux de bord temps reel, "updates" est le plus performant car il n'envoie que les changements.

9. Resume

9.1 Tableau Recapitulatif des Patterns de Routing

Pattern Methode Cas d'usage
Routing Simple add_conditional_edges(source, fn) Classification, aiguillage par categorie
Routing avec Map add_conditional_edges(source, fn, path_map) Traduction labels -> noeuds
Fan-out Statique Retourner une list de noms Execution parallele planifiee

Cycle Arete conditionnelle vers un noeud deja visite Agents, auto-amelioration, retry
ReAct Agent -> Tools -> Agent (cycle) Agents autonomes avec outils

9.2 Checklist de Securite pour les Cycles

Avant de mettre un cycle en production, verifiez :

  • Condition d'arret explicite : Votre logique de routage a-t-elle un chemin vers END ?
  • Compteur d'iterations : Avez-vous un compteur dans le state avec une limite ?
  • recursion_limit configure : Avez-vous defini un recursion_limit raisonnable dans la config ?
  • Gestion d'erreur : Les noeuds dans le cycle gerent-ils les exceptions ?
  • Convergence garantie : La logique du cycle converge-t-elle naturellement vers une solution ?
  • Monitoring : Avez-vous des logs/traces pour observer le comportement en production ?
  • Couts maitrises : Chaque iteration appelle-t-elle un LLM ? Si oui, avez-vous estime le cout maximal ?

Points cles a retenir

  • add_conditional_edges() est la brique fondamentale du routing dans LangGraph
  • Les cycles sont ce qui rend LangGraph unique : ils permettent les agents autonomes
  • Le pattern ReAct (Think-Act-Observe) est le pattern agent le plus repandu
  • Fan-out/Fan-in avec Send permet le traitement parallele
  • Toujours configurer recursion_limit et un compteur interne pour les cycles
  • Le streaming offre 4 modes adaptes a differents cas d'usage
