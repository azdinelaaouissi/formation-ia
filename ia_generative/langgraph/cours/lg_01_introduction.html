Introduction à LangGraph

Cours 1 — Orchestration de Workflows LLM avec des Graphes

1. Qu'est-ce que LangGraph ?

LangGraph est un framework Python développé par l'équipe LangChain qui permet de construire des applications LLM stateful (à état persistant) et multi-acteurs en modélisant les workflows sous forme de graphes orientés. Chaque noeud du graphe représente une étape de traitement, et chaque arête définit le flux d'exécution entre ces étapes.

Contrairement à une simple chaîne linéaire, LangGraph permet de créer des workflows complexes comprenant :

  • Des boucles — un noeud peut revenir en arrière pour itérer
  • Des branchements conditionnels — le flux change selon l'état courant
  • Des exécutions parallèles — plusieurs noeuds s'exécutent simultanément
  • Un état partagé — toutes les étapes lisent et écrivent dans un état commun
  • Une persistance — l'état peut être sauvegardé entre les invocations
LangGraph est la réponse de LangChain au besoin croissant de construire des agents IA sophistiqués qui ne se contentent pas d'une exécution linéaire. Pensez à un agent qui peut réfléchir, agir, observer le résultat, puis décider de continuer ou de recommencer — c'est exactement ce que LangGraph rend possible.

Le nom « LangGraph » vient de la combinaison de Lang (pour LangChain/Language) et Graph (pour la structure de données en graphe). Le framework s'appuie sur la théorie des graphes pour modéliser des workflows d'IA générative de manière déclarative et modulaire.

Caractéristiques principales

  • Basé sur les graphes — Modélisation naturelle des workflows complexes
  • Stateful — État partagé et persistant entre les noeuds
  • Streaming natif — Support du streaming token par token et événement par événement
  • Human-in-the-loop — Capacité de pause et reprise avec intervention humaine
  • Checkpointing — Sauvegarde automatique de l'état pour la reprise
  • Compatible LangChain — S'intègre nativement avec l'écosystème LangChain

2. Pourquoi LangGraph ?

2.1 Les limites des chaînes linéaires

Avec LangChain classique, on construit des chaînes (chains) qui exécutent des étapes de manière séquentielle : Prompt → LLM → Parser → Sortie. Ce modèle fonctionne bien pour des tâches simples, mais il présente des limites fondamentales :

LimiteDescriptionConséquence
Pas de bouclesUne chaîne ne peut pas revenir en arrièreImpossible de ré-essayer ou d'itérer
Pas de branchement dynamiqueLe flux est fixe à la compilationImpossible de router selon le contexte
Pas d'état persistantChaque invocation repart de zéroPas de mémoire conversationnelle native
Pas de parallélismeExécution strictement séquentielleLenteur sur les tâches indépendantes
Pas de human-in-the-loopPas de mécanisme de pauseImpossible d'attendre une validation humaine

2.2 Ce que LangGraph résout

LangGraph adresse toutes ces limitations en introduisant un modèle d'exécution basé sur les graphes. Voici un comparatif visuel entre une chaîne linéaire et un graphe LangGraph :

Avec LangGraph, on passe d'un paradigme linéaire (A → B → C) à un paradigme graphique où les noeuds peuvent se connecter librement, créer des boucles, et s'exécuter conditionnellement.

3. LangGraph vs LangChain

Il est essentiel de comprendre la relation entre LangChain et LangGraph. Ce ne sont pas des concurrents — ils sont complémentaires.

La métaphore du chef d'orchestre

LangChain fournit les instruments : les modèles de langage, les prompts, les parsers, les outils, les retrievers, les vector stores, etc.

LangGraph est le chef d'orchestre : il définit quand, comment et dans quel ordre ces instruments jouent ensemble.

AspectLangChainLangGraph
RôleBibliothèque de composantsFramework d'orchestration
ParadigmeChaînes (LCEL, Runnables)Graphes (StateGraph)
Flux d'exécutionLinéaire, pas de bouclesBoucles, branchements, parallélisme
ÉtatPassé via les argumentsÉtat partagé et persistant (TypedDict)
MémoireVia ConversationBufferMemory (externe)Checkpointing natif (MemorySaver, etc.)
Human-in-the-loopNon natifNatif (interrupt_before, interrupt_after)
StreamingToken par tokenToken + événements + état
Cas d'usage typiqueAppel LLM simple, RAG basiqueAgents, multi-step, workflows complexes
DépendanceAutonomeUtilise les composants LangChain
En pratique, on utilise les deux ensemble. LangChain fournit les ChatModels, les Tools, et les Prompts. LangGraph orchestre leur utilisation dans un graphe. Depuis 2024, l'équipe LangChain recommande LangGraph pour tout workflow non trivial.

3.1 Quand utiliser quoi ?

  • LangChain seul : appels LLM simples, RAG basique avec un seul retriever, chaînes de prompts linéaires
  • LangGraph : agents avec outils, workflows avec boucles de rétroaction, multi-step reasoning, chatbots avec mémoire persistante, pipelines avec validation humaine
  • LangGraph sans LangChain : possible ! LangGraph peut orchestrer n'importe quelle fonction Python, pas seulement des composants LangChain

4. Concepts fondamentaux

LangGraph repose sur quatre concepts fondamentaux issus de la théorie des graphes, adaptés au contexte de l'IA générative :

4.1 State (État)

Le State est une structure de données partagée (typiquement un TypedDict Python) qui contient toutes les informations nécessaires au workflow. Chaque noeud lit l'état courant et retourne une mise à jour partielle de cet état.
from typing_extensions import TypedDict class State(TypedDict): question: str # La question posée par l'utilisateur reponse: str # La réponse générée documents: list[str] # Les documents récupérés score: float # Le score de pertinence

4.2 Node (Noeud)

Un Node est une fonction Python qui reçoit l'état courant en entrée, effectue un traitement (appel LLM, recherche, calcul, etc.), et retourne un dictionnaire contenant les champs de l'état à mettre à jour.
def analyser_question(state: State) -> dict: """Noeud qui analyse la question de l'utilisateur.""" question = state["question"] # Logique d'analyse... return {"score": 0.85}

4.3 Edge (Arête)

Une Edge est une connexion entre deux noeuds qui définit le flux d'exécution. Il existe des arêtes fixes (toujours le même chemin) et des arêtes conditionnelles (le chemin dépend de l'état).

4.4 Graph (Graphe)

Le Graph est le conteneur qui rassemble tous les noeuds et arêtes. On le crée avec StateGraph, on y ajoute des noeuds et des arêtes, puis on le compile pour obtenir un CompiledGraph exécutable.
Retenez cette structure : START → Noeuds → END avec un State qui circule entre tous les noeuds. Chaque noeud est une simple fonction Python. C'est ce modèle mental qui rend LangGraph si puissant et intuitif une fois compris.

6. Premier graphe LangGraph

Construisons notre tout premier graphe, étape par étape. Ce graphe simple prendra un message en entrée, le transformera en majuscules, puis ajoutera un suffixe.

6.1 Étape 1 — Définir le State

Le State est un TypedDict qui décrit les données partagées entre les noeuds du graphe.
from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END # Étape 1 : Définir le State class MonState(TypedDict): message: str # Le message à traiter message_traite: str # Le message après traitement

6.2 Étape 2 — Définir les Noeuds

Chaque noeud est une fonction qui prend le state en paramètre et retourne un dictionnaire avec les champs modifiés.
# Étape 2 : Définir les fonctions des noeuds def mettre_en_majuscules(state: MonState) -> dict: """Noeud 1 : Met le message en majuscules.""" message = state["message"] return {"message_traite": message.upper()} def ajouter_suffixe(state: MonState) -> dict: """Noeud 2 : Ajoute un suffixe au message traité.""" msg = state["message_traite"] return {"message_traite": msg + " [TRAITE]"}

6.3 Étape 3 — Construire le Graphe

On crée un StateGraph, on y ajoute les noeuds avec add_node(), puis on les connecte avec add_edge().
# Étape 3 : Créer et configurer le graphe graph_builder = StateGraph(MonState) # Ajouter les noeuds graph_builder.add_node("majuscules", mettre_en_majuscules) graph_builder.add_node("suffixe", ajouter_suffixe) # Connecter les noeuds avec des edges graph_builder.add_edge(START, "majuscules") # START → majuscules graph_builder.add_edge("majuscules", "suffixe") # majuscules → suffixe graph_builder.add_edge("suffixe", END) # suffixe → END

6.4 Étape 4 — Compiler et Invoquer

La compilation vérifie la structure du graphe et retourne un CompiledGraph. L'invocation exécute le graphe avec un état initial.
# Étape 4 : Compiler le graphe graph = graph_builder.compile() # Étape 5 : Invoquer le graphe avec un état initial resultat = graph.invoke({"message": "bonjour langgraph"}) print(resultat)
{'message': 'bonjour langgraph', 'message_traite': 'BONJOUR LANGGRAPH [TRAITE]'}

6.5 Code complet

Voici le code complet rassemblé en un seul bloc :

from typing_extensions import TypedDict from langgraph.graph import StateGraph, START, END # ---- Définition du State ---- class MonState(TypedDict): message: str message_traite: str # ---- Définition des Noeuds ---- def mettre_en_majuscules(state: MonState) -> dict: """Met le message en majuscules.""" return {"message_traite": state["message"].upper()} def ajouter_suffixe(state: MonState) -> dict: """Ajoute un suffixe au message traité.""" return {"message_traite": state["message_traite"] + " [TRAITE]"} # ---- Construction du Graphe ---- builder = StateGraph(MonState) builder.add_node("majuscules", mettre_en_majuscules) builder.add_node("suffixe", ajouter_suffixe) builder.add_edge(START, "majuscules") builder.add_edge("majuscules", "suffixe") builder.add_edge("suffixe", END) # ---- Compilation et Invocation ---- graph = builder.compile() resultat = graph.invoke({"message": "bonjour langgraph"}) print(f"Entrée : {resultat['message']}") print(f"Sortie : {resultat['message_traite']}")
Entrée : bonjour langgraph Sortie : BONJOUR LANGGRAPH [TRAITE]
Le graphe exécute séquentiellement les deux noeuds. L'état initial {"message": "bonjour langgraph"} est enrichi par chaque noeud : d'abord la mise en majuscules, puis l'ajout du suffixe. Le résultat final contient à la fois le message original et le message traité.
Notez que chaque noeud retourne seulement les champs qu'il modifie (un dictionnaire partiel), pas l'état complet. LangGraph fusionne automatiquement les mises à jour avec l'état existant. C'est le principe des mises à jour partielles.

7. Architecture de LangGraph

Comprendre l'architecture interne de LangGraph aide à mieux l'utiliser. Voici le flux complet depuis la création du graphe jusqu'à son exécution :

7.1 invoke() vs stream()

LangGraph offre deux modes d'exécution :

invoke(input, config=None)

Exécute le graphe complet et retourne l'état final. Bloquant — attend la fin de l'exécution.

resultat = graph.invoke({"message": "test"}) print(resultat) # État complet final

stream(input, config=None, stream_mode="values")

Exécute le graphe et retourne un itérateur sur les états intermédiaires. Utile pour le suivi en temps réel.

for etat in graph.stream({"message": "test"}): print("---") print(etat)
--- {'majuscules': {'message_traite': 'TEST'}} --- {'suffixe': {'message_traite': 'TEST [TRAITE]'}}
Le mode stream est particulièrement utile en développement pour déboguer le flux d'exécution. En production, il permet de fournir un feedback en temps réel à l'utilisateur, montrant quelle étape du workflow est en cours.

8. Cas d'utilisation

LangGraph excelle dans de nombreux scénarios où les chaînes simples montrent leurs limites :

8.1 Chatbots conversationnels

Les chatbots nécessitent un état persistant (historique des messages) et un routage conditionnel (appeler des outils si nécessaire). LangGraph gère nativement la mémoire conversationnelle via les checkpointers.

Flux typique : START → recevoir_message → appeler_llm → (outil_nécessaire ?) → appeler_outil → appeler_llm → END

8.2 Agents avec outils (Tool-calling agents)

Un agent IA qui peut appeler des outils (recherche web, calculatrice, API) a besoin de boucles : il appelle un outil, observe le résultat, décide s'il a besoin d'un autre outil, et ainsi de suite.

Flux typique : START → raisonner → (besoin_outil ?) → exécuter_outil → raisonner → ... → END

8.3 Pipelines RAG avancés

Un pipeline RAG (Retrieval-Augmented Generation) avancé peut inclure de la vérification de pertinence, de la reformulation de requête, et du multi-retrieval.

Flux typique : START → analyser_question → retriever → évaluer_pertinence → (pertinent ?) → générer / reformuler → END

8.4 Raisonnement multi-étapes

Pour des tâches complexes nécessitant une décomposition en sous-tâches, LangGraph permet de modéliser chaque étape de raisonnement comme un noeud, avec des boucles de vérification.

Flux typique : START → décomposer → résoudre_étape → vérifier → (correct ?) → étape_suivante / corriger → assembler → END

8.5 Human-in-the-loop

Certains workflows nécessitent une validation humaine avant de continuer. LangGraph permet de mettre en pause l'exécution, d'attendre l'approbation, puis de reprendre.

Flux typique : START → analyser → proposer_action → [PAUSE : attente validation] → exécuter_action → END

8.6 Systèmes multi-agents

Plusieurs agents IA collaborent pour résoudre un problème. Chaque agent est un sous-graphe dans un graphe principal qui coordonne leurs interactions.

Flux typique : START → dispatcher → agent_recherche | agent_analyse | agent_rédaction → synthétiser → END

Le cas d'usage le plus courant est l'agent avec outils (ReAct pattern). C'est l'exemple canonique de LangGraph et c'est ce que nous construirons progressivement dans les prochains cours.

9.2 Quand utiliser LangGraph ?

Utilisez LangGraph quand vous avez besoin de :

  • Boucles dans votre workflow (retry, itération, réflexion)
  • Branchement conditionnel (router selon le contenu)
  • État persistant entre les invocations (mémoire conversationnelle)
  • Human-in-the-loop (validation humaine avant action)
  • Streaming avancé (par noeud, par token, par événement)
  • Exécution parallèle de sous-tâches

Restez sur LangChain classique (LCEL) quand :

  • Votre workflow est strictement linéaire (A → B → C)
  • Vous n'avez pas besoin de mémoire entre les appels
  • Pas de boucles ni de branchements dynamiques
