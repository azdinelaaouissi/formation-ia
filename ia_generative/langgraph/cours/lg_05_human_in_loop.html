Human-in-the-Loop

L'Humain dans la Boucle - Controle, Approbation et Supervision des Agents IA

1. Pourquoi Human-in-the-Loop ?

Le pattern Human-in-the-Loop (HITL) permet a un humain d'intervenir dans l'execution d'un graphe LangGraph pour approuver, rejeter ou modifier les actions d'un agent IA avant qu'elles ne soient executees. C'est un mecanisme essentiel pour la securite, la fiabilite et la conformite.

Les systemes d'IA, meme les plus avances, commettent des erreurs. Dans certains contextes, ces erreurs peuvent avoir des consequences graves :

  • Actions irreversibles : envoi d'emails, publication sur les reseaux sociaux, modifications en base de donnees
  • Transactions financieres : virements, achats, modifications de contrats
  • Decisions critiques : diagnostics medicaux, decisions juridiques, evaluations de risques
  • Acces a des donnees sensibles : fichiers confidentiels, informations personnelles
  • Actions a fort impact : suppression de comptes, modification d'infrastructures, deploiements

Le HITL cree un point de controle ou l'humain peut examiner ce que l'agent propose de faire et decider de la suite. C'est le principe du "trust but verify" (faire confiance mais verifier).

1.1 Diagramme du Flux HITL

Le HITL n'est pas un aveu d'echec de l'IA. C'est une bonne pratique d'ingenierie. Meme les meilleurs modeles produisent des hallucinations. Un point de validation humain est souvent la difference entre un prototype et un systeme de production fiable.

2. interrupt_before - Pause Avant un Noeud

interrupt_before est un parametre de compilation qui met en pause l'execution du graphe avant qu'un noeud specifie ne s'execute. L'humain peut alors examiner l'etat, approuver, ou modifier avant de reprendre.

2.1 Prerequis : le Checkpointer

Le HITL necessite obligatoirement un checkpointer. Sans persistence, le graphe ne peut pas sauvegarder son etat pendant la pause et le restaurer lors de la reprise.

Sans checkpointer, les interruptions ne fonctionnent pas. Le graphe leve une erreur si vous essayez d'utiliser interrupt_before ou interrupt_after sans avoir configure un checkpointer.

2.2 Exemple Complet avec interrupt_before

from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict class EmailState(TypedDict): to: str subject: str body: str approved: bool sent: bool def draft_email(state: EmailState) -> dict: """L'agent genere un brouillon d'email.""" return { "to": state["to"], "subject": "Re: Votre demande", "body": f"Bonjour,\n\nSuite a votre demande concernant '{state['subject']}', " f"nous avons le plaisir de vous informer que...\n\nCordialement,\nL'equipe" } def send_email(state: EmailState) -> dict: """Envoie l'email (action irreversible!).""" print(f"EMAIL ENVOYE a {state['to']}") print(f" Sujet : {state['subject']}") print(f" Corps : {state['body'][:100]}...") return {"sent": True} # Construction du graphe builder = StateGraph(EmailState) builder.add_node("draft", draft_email) builder.add_node("send", send_email) builder.set_entry_point("draft") builder.add_edge("draft", "send") builder.add_edge("send", END) # Compilation avec interrupt_before sur "send" checkpointer = MemorySaver() graph = builder.compile( checkpointer=checkpointer, interrupt_before=["send"] # PAUSE avant l'envoi ) # --- Etape 1 : Executer jusqu'a la pause --- config = {"configurable": {"thread_id": "email-1"}} result = graph.invoke( {"to": "client@example.com", "subject": "reclamation produit"}, config=config ) # Le graphe est en pause AVANT "send" print("--- GRAPHE EN PAUSE ---") print(f"Brouillon : {result['body']}") print("L'email n'a PAS ete envoye.") # --- Etape 2 : L'humain examine et approuve --- # En production, cela passerait par une UI, un webhook, etc. user_input = input("Approuvez-vous l'envoi ? (oui/non) : ") if user_input.lower() == "oui": # Reprendre l'execution (None = pas de nouvelle entree) result = graph.invoke(None, config=config) print(f"Email envoye : {result['sent']}") else: print("Envoi annule par l'utilisateur.")
--- GRAPHE EN PAUSE --- Brouillon : Bonjour, Suite a votre demande concernant 'reclamation produit', nous avons le plaisir... L'email n'a PAS ete envoye. Approuvez-vous l'envoi ? (oui/non) : oui EMAIL ENVOYE a client@example.com Sujet : Re: Votre demande Corps : Bonjour, Suite a votre demande... Email envoye : True
L'execution se fige juste avant le noeud "send". L'etat est sauvegarde par le checkpointer. Quand l'humain reprend avec graph.invoke(None, config), le graphe repart exactement ou il s'etait arrete.

3. interrupt_after - Pause Apres un Noeud

interrupt_after met en pause l'execution apres qu'un noeud ait fini de s'executer. L'humain peut alors examiner le resultat produit par le noeud et decider comment continuer.

La difference avec interrupt_before est subtile mais importante :

Parametre Quand la pause se produit Ce que l'humain voit
interrupt_before AVANT l'execution du noeud L'etat juste avant le noeud (entree du noeud)
interrupt_after APRES l'execution du noeud L'etat apres le noeud (sortie du noeud)

3.1 Exemple : Revue d'une Analyse avant Publication

from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict class ReportState(TypedDict): data: str analysis: str review_comment: str published: bool def analyze_data(state: ReportState) -> dict: """Genere une analyse a partir des donnees.""" analysis = f"""Analyse des donnees : {state['data']} Conclusions principales : 1. Tendance a la hausse de 15% sur le trimestre 2. Segment B sous-performant (-5%) 3. Recommandation : reallouer le budget du segment B vers A""" return {"analysis": analysis} def publish_report(state: ReportState) -> dict: """Publie le rapport (action visible par les stakeholders).""" print(f"RAPPORT PUBLIE :\n{state['analysis']}") return {"published": True} builder = StateGraph(ReportState) builder.add_node("analyze", analyze_data) builder.add_node("publish", publish_report) builder.set_entry_point("analyze") builder.add_edge("analyze", "publish") builder.add_edge("publish", END) checkpointer = MemorySaver() graph = builder.compile( checkpointer=checkpointer, interrupt_after=["analyze"] # Pause APRES l'analyse ) # Execution : l'analyse s'execute, puis PAUSE config = {"configurable": {"thread_id": "report-42"}} result = graph.invoke({"data": "Ventes Q3 2025"}, config=config) print("--- ANALYSE GENEREE (en attente de revue) ---") print(result["analysis"]) # L'humain examine l'analyse produite et decide de continuer # Reprendre l'execution result = graph.invoke(None, config=config)
--- ANALYSE GENEREE (en attente de revue) --- Analyse des donnees : Ventes Q3 2025 Conclusions principales : 1. Tendance a la hausse de 15% sur le trimestre 2. Segment B sous-performant (-5%) 3. Recommandation : reallouer le budget du segment B vers A RAPPORT PUBLIE : [meme contenu que ci-dessus]

4. Modifier le State Pendant l'Interruption

L'un des aspects les plus puissants du HITL est la capacite de l'humain a modifier l'etat du graphe pendant la pause, avant de reprendre l'execution. Cela permet de corriger des erreurs, d'ajouter des informations, ou de rediriger l'agent.

4.1 Avec update_state()

from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict class ContractState(TypedDict): client_name: str amount: float terms: str approved: bool final_terms: str def draft_contract(state: ContractState) -> dict: terms = f"""CONTRAT DE SERVICE Client : {state['client_name']} Montant : {state['amount']} EUR Conditions : Paiement a 30 jours, livraison sous 2 semaines. Clause de penalite : 1% par jour de retard.""" return {"terms": terms} def finalize_contract(state: ContractState) -> dict: print(f"CONTRAT FINALISE :\n{state['terms']}") return {"final_terms": state["terms"], "approved": True} builder = StateGraph(ContractState) builder.add_node("draft", draft_contract) builder.add_node("finalize", finalize_contract) builder.set_entry_point("draft") builder.add_edge("draft", "finalize") builder.add_edge("finalize", END) checkpointer = MemorySaver() graph = builder.compile( checkpointer=checkpointer, interrupt_before=["finalize"] ) config = {"configurable": {"thread_id": "contract-001"}} result = graph.invoke( {"client_name": "Acme Corp", "amount": 50000}, config=config ) print("--- BROUILLON DU CONTRAT ---") print(result["terms"]) # L'humain veut modifier les termes avant finalisation # Utiliser update_state() pour injecter des modifications graph.update_state( config, { "terms": """CONTRAT DE SERVICE (REVISE) Client : Acme Corp Montant : 50000 EUR Conditions : Paiement a 45 jours, livraison sous 3 semaines. Clause de penalite : 0.5% par jour de retard. Note : Remise de 5% pour paiement anticipe.""" } ) # Reprendre avec l'etat modifie result = graph.invoke(None, config=config) print(f"\nContrat approuve : {result['approved']}")
--- BROUILLON DU CONTRAT --- CONTRAT DE SERVICE Client : Acme Corp Montant : 50000 EUR Conditions : Paiement a 30 jours, livraison sous 2 semaines. Clause de penalite : 1% par jour de retard. CONTRAT FINALISE : CONTRAT DE SERVICE (REVISE) Client : Acme Corp Montant : 50000 EUR Conditions : Paiement a 45 jours, livraison sous 3 semaines. Clause de penalite : 0.5% par jour de retard. Note : Remise de 5% pour paiement anticipe. Contrat approuve : True
La methode update_state() est l'une des fonctionnalites les plus puissantes de LangGraph. Elle permet a l'humain de corriger non seulement le texte genere, mais aussi des valeurs structurees dans le state (scores, flags, listes, etc.). C'est ce qui rend le HITL vraiment interactif plutot que simplement binaire (oui/non).

5. La Fonction interrupt() (Approche Recommandee)

Depuis les versions recentes de LangGraph, la fonction interrupt() est l'approche recommandee pour le HITL. Contrairement a interrupt_before/interrupt_after qui sont statiques (definies a la compilation), interrupt() est dynamique et peut etre appelee a n'importe quel moment dans le code d'un noeud.

interrupt_before/after vs interrupt()

Critere interrupt_before/after interrupt()
Moment de definition A la compilation (statique) A l'execution (dynamique)
Granularite Avant/apres un noeud entier N'importe ou dans un noeud
Donnees echangees Via update_state() externe Valeur de retour directe
Recommandation Pour debug/breakpoints Pour HITL en production

5.1 Syntaxe de interrupt()

from langgraph.types import interrupt, Command from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict class OrderState(TypedDict): order_id: str items: list total: float discount_applied: bool confirmation: str def process_order(state: OrderState) -> dict: """Traite la commande et demande approbation si montant eleve.""" total = state["total"] if total > 1000: # Pause dynamique : montant eleve, demander approbation human_response = interrupt({ "question": f"Commande #{state['order_id']} pour {total} EUR. Approuvez-vous ?", "options": ["approuver", "rejeter", "appliquer_remise"] }) # human_response contient la reponse de l'humain if human_response == "rejeter": return {"confirmation": "Commande rejetee par le superviseur."} elif human_response == "appliquer_remise": return { "total": total * 0.9, "discount_applied": True, "confirmation": f"Remise 10% appliquee. Nouveau total : {total * 0.9} EUR" } return {"confirmation": f"Commande #{state['order_id']} confirmee pour {total} EUR."} builder = StateGraph(OrderState) builder.add_node("process", process_order) builder.set_entry_point("process") builder.add_edge("process", END) checkpointer = MemorySaver() graph = builder.compile(checkpointer=checkpointer) # --- Execution --- config = {"configurable": {"thread_id": "order-789"}} # Etape 1 : le graphe s'execute et s'arrete a interrupt() result = graph.invoke( {"order_id": "789", "items": ["laptop", "souris"], "total": 1500.0}, config=config ) # Le graphe est en pause, montrant la question a l'humain # Etape 2 : L'humain repond via Command(resume=...) result = graph.invoke( Command(resume="appliquer_remise"), config=config ) print(result["confirmation"])
Remise 10% appliquee. Nouveau total : 1350.0 EUR

5.2 Avantages de interrupt()

Pourquoi interrupt() est preferable

  • Conditionnel : L'interruption ne se produit que si necessaire (montant eleve, action risquee)
  • Echange de donnees riche : Vous pouvez passer un objet structure a l'humain et recevoir une reponse typee
  • Plusieurs interruptions : Un meme noeud peut contenir plusieurs appels a interrupt()
  • Resume direct : La valeur retournee par Command(resume=...) est directement disponible dans la variable
  • Inspire de input() : La semantique est similaire a la fonction input() de Python

6. Patterns d'Approbation

En production, il existe plusieurs patterns d'approbation selon le niveau de controle souhaite par l'humain.

6.1 Pattern : Approbation Binaire (Oui/Non)

def risky_action_node(state: dict) -> dict: """Action a risque avec approbation simple.""" action_desc = f"Supprimer {state['file_count']} fichiers du serveur" approval = interrupt({ "type": "binary_approval", "action": action_desc, "risk_level": "ELEVE" }) if approval == "oui": # Executer l'action risquee return {"result": f"{state['file_count']} fichiers supprimes."} else: return {"result": "Action annulee par l'utilisateur."}

6.2 Pattern : Edition et Continuation

def generate_and_review(state: dict) -> dict: """Genere un contenu, permet a l'humain de l'editer.""" # Generer le brouillon draft = llm.invoke(f"Ecris un tweet sur : {state['topic']}").content # Presenter a l'humain pour edition edited = interrupt({ "type": "edit", "original": draft, "instruction": "Editez le texte ou tapez 'ok' pour accepter tel quel." }) # Si l'humain tape "ok", garder l'original final_text = draft if edited == "ok" else edited return {"tweet": final_text}

6.3 Pattern : Fournir des Informations Supplementaires

def clarification_node(state: dict) -> dict: """Demande des precisions a l'humain.""" question = state["question"] # L'agent detecte une ambiguite if "date" not in state or not state["date"]: date_info = interrupt({ "type": "clarification", "question": "Pour quelle date souhaitez-vous cette reservation ?", "context": f"Demande originale : {question}" }) return {"date": date_info} return {}

7. Approbation d'Outils (Tool Approval)

Le Tool Approval est un pattern ou l'agent propose d'utiliser un outil, mais attend l'approbation humaine avant l'execution. C'est critique pour les outils qui ont des effets de bord (envoi d'emails, requetes SQL, appels API externes).
from langgraph.graph import StateGraph, END from langgraph.types import interrupt, Command from langgraph.checkpoint.memory import MemorySaver from langchain_core.messages import HumanMessage, AIMessage, ToolMessage from langchain_core.tools import tool from typing import TypedDict, Annotated, Sequence import operator, json @tool def send_email(to: str, subject: str, body: str) -> str: """Envoie un email (action irreversible).""" return f"Email envoye a {to} avec sujet '{subject}'" @tool def delete_records(table: str, condition: str) -> str: """Supprime des enregistrements en base de donnees.""" return f"Suppression dans {table} WHERE {condition}" # Outils necessitant une approbation DANGEROUS_TOOLS = {"send_email", "delete_records"} class ToolApprovalState(TypedDict): messages: Annotated[Sequence, operator.add] def tool_executor_with_approval(state: ToolApprovalState) -> dict: """Execute les outils AVEC approbation pour les outils dangereux.""" last_message = state["messages"][-1] results = [] for tool_call in last_message.tool_calls: tool_name = tool_call["name"] tool_args = tool_call["args"] if tool_name in DANGEROUS_TOOLS: # Demander approbation pour les outils dangereux approval = interrupt({ "type": "tool_approval", "tool_name": tool_name, "arguments": tool_args, "warning": f"L'agent veut executer '{tool_name}'. Cette action est irreversible." }) if approval != "approuver": results.append(ToolMessage( content=f"Action '{tool_name}' rejetee par l'utilisateur.", tool_call_id=tool_call["id"] )) continue # Executer l'outil (approuve ou non-dangereux) tool_fn = {"send_email": send_email, "delete_records": delete_records}[tool_name] result = tool_fn.invoke(tool_args) results.append(ToolMessage(content=result, tool_call_id=tool_call["id"])) return {"messages": results}
En production, classifiez toujours vos outils par niveau de risque. Les outils en lecture seule (recherche, calcul) peuvent s'executer automatiquement. Les outils avec des effets de bord (ecriture, suppression, envoi) doivent passer par une approbation humaine.

8. Approbation Multi-Etapes

Certains workflows necessitent plusieurs points d'approbation a differentes etapes. Par exemple, un processus de publication qui requiert une revue editoriale, puis une validation juridique, puis une approbation de la direction.

8.1 Pipeline avec Checkpoints Multiples

from langgraph.graph import StateGraph, END from langgraph.types import interrupt, Command from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict, List class PublicationState(TypedDict): content: str editorial_review: str legal_review: str director_approval: bool published: bool rejection_reason: str def ai_write(state: PublicationState) -> dict: """L'IA redige le contenu initial.""" content = f"Article genere par IA sur le sujet demande. Lorem ipsum..." return {"content": content} def editorial_review_node(state: PublicationState) -> dict: """Point de controle editorial.""" review = interrupt({ "step": "revue_editoriale", "content": state["content"], "instruction": "Veuillez relire et corriger le contenu. Retournez le texte corrige ou 'rejeter'." }) if review == "rejeter": return {"rejection_reason": "Rejete en revue editoriale"} return {"content": review, "editorial_review": "approuve"} def legal_review_node(state: PublicationState) -> dict: """Point de controle juridique.""" if state.get("rejection_reason"): return {} # Deja rejete, passer legal = interrupt({ "step": "validation_juridique", "content": state["content"], "instruction": "Y a-t-il des problemes juridiques ? Repondez 'approuve' ou decrivez le probleme." }) if legal != "approuve": return {"rejection_reason": f"Probleme juridique : {legal}"} return {"legal_review": "approuve"} def director_approval_node(state: PublicationState) -> dict: """Approbation finale du directeur.""" if state.get("rejection_reason"): return {} approval = interrupt({ "step": "approbation_direction", "content": state["content"], "reviews": { "editorial": state["editorial_review"], "legal": state["legal_review"] }, "instruction": "Toutes les revues sont approuvees. Publier ? (oui/non)" }) return {"director_approval": approval == "oui"} def publish_node(state: PublicationState) -> dict: if state.get("rejection_reason") or not state.get("director_approval"): return {"published": False} print("ARTICLE PUBLIE AVEC SUCCES") return {"published": True} # Construction du pipeline builder = StateGraph(PublicationState) builder.add_node("write", ai_write) builder.add_node("editorial", editorial_review_node) builder.add_node("legal", legal_review_node) builder.add_node("director", director_approval_node) builder.add_node("publish", publish_node) builder.set_entry_point("write") builder.add_edge("write", "editorial") builder.add_edge("editorial", "legal") builder.add_edge("legal", "director") builder.add_edge("director", "publish") builder.add_edge("publish", END) checkpointer = MemorySaver() graph = builder.compile(checkpointer=checkpointer) # --- Execution progressive --- config = {"configurable": {"thread_id": "pub-2025-001"}} # Etape 1 : Redaction + pause pour revue editoriale result = graph.invoke({}, config=config) # Etape 2 : L'editeur approuve avec corrections result = graph.invoke(Command(resume="Article corrige par l'editeur..."), config=config) # Etape 3 : Le juriste approuve result = graph.invoke(Command(resume="approuve"), config=config) # Etape 4 : Le directeur approuve result = graph.invoke(Command(resume="oui"), config=config) print(f"Publie : {result['published']}")
ARTICLE PUBLIE AVEC SUCCES Publie : True

9. Resume et Replay

Le checkpointer sauvegarde l'etat du graphe a chaque interruption, ce qui permet de reprendre exactement la ou l'execution s'est arretee, meme apres un redemarrage de l'application.

9.1 Reprendre une Execution Interrompue

# L'execution est interrompue (l'utilisateur ferme le navigateur, etc.) # Plus tard, on reprend avec le meme thread_id config = {"configurable": {"thread_id": "session-user-42"}} # Recuperer l'etat sauvegarde state_snapshot = graph.get_state(config) print(f"Dernier noeud execute : {state_snapshot.next}") print(f"Etat actuel : {state_snapshot.values}") # Reprendre l'execution if state_snapshot.next: # Le graphe n'est pas termine, on peut reprendre result = graph.invoke(None, config=config) print("Execution reprise et terminee.") else: print("Le graphe est deja termine.")

9.2 Historique des Checkpoints

9.3 Replay depuis un Checkpoint Specifique

# Revenir a un checkpoint anterieur (time travel) config_replay = { "configurable": { "thread_id": "session-user-42", "checkpoint_id": "ck_abc122" # Revenir avant l'approbation directeur } } # Reprendre depuis ce point avec une decision differente result = graph.invoke( Command(resume="non"), # Cette fois, le directeur refuse config=config_replay ) print(f"Publie : {result['published']}") # False
Le replay (time travel) est une fonctionnalite extremement puissante pour le debugging et les tests. Vous pouvez rejouer n'importe quelle execution depuis n'importe quel point, en changeant les decisions humaines. C'est aussi utile pour les audits : vous pouvez prouver exactement ce qui s'est passe a chaque etape.

10. Resume

10.1 Comparaison des Patterns HITL

Pattern Mecanisme Cas d'usage Complexite
interrupt_before Statique (compilation) Debugging, breakpoints Faible
interrupt_after Statique (compilation) Revue de sortie de noeud Faible
interrupt() Dynamique (runtime) Production HITL, approbations Moyenne
update_state() Modification externe Edition du state pendant pause Moyenne
Tool Approval interrupt() + classification outils Securite des appels d'outils Elevee
Multi-step Plusieurs interrupt() en chaine Workflows reglementaires Elevee

10.2 Quand Utiliser Chaque Approche

Guide de decision

  • Vous debuggez un graphe ? Utilisez interrupt_before/interrupt_after comme breakpoints
  • Action irreversible simple ? Utilisez interrupt() avec approbation binaire
  • Contenu a editer ? Utilisez interrupt() avec retour de texte edite
  • Agent avec outils dangereux ? Implementez le pattern Tool Approval
  • Pipeline reglementaire ? Utilisez le pattern Multi-step avec plusieurs interrupt()
  • Modification externe du state ? Utilisez update_state() avec interrupt_before

Points cles a retenir

  • Le HITL est essentiel pour la securite des agents IA en production
  • Un checkpointer est obligatoire pour toute forme d'interruption
  • La fonction interrupt() avec Command(resume=...) est l'approche recommandee
  • Classifiez vos outils par niveau de risque pour determiner lesquels necessitent une approbation
  • Le replay (time travel) permet de revenir a n'importe quel checkpoint et de prendre une decision differente
  • update_state() permet a l'humain de modifier toutes les valeurs du state, pas seulement d'approuver/rejeter
