Embeddings et VectorStores

Cours 6 - Le Stockage Semantique pour la Recherche par Similarite avec LangChain

1. Rappel : Qu'est-ce qu'un Embedding ?

Un embedding est une representation numerique (vecteur) d'un texte dans un espace mathematique a haute dimension. L'idee fondamentale est que des textes ayant un sens similaire seront representes par des vecteurs proches dans cet espace, tandis que des textes differents seront eloignes.

Un embedding est un vecteur de nombres reels (par exemple, un tableau de 1536 floats) qui capture le sens semantique d'un texte. Des textes semantiquement proches (comme "le chat dort" et "le felin sommeille") auront des vecteurs proches dans l'espace vectoriel, meme si les mots utilises sont completement differents.

Comment fonctionne la similarite ?

La similarite entre deux embeddings se mesure generalement avec la similarite cosinus (cosine similarity). C'est le cosinus de l'angle entre les deux vecteurs :

  • 1.0 = identiques (meme direction)
  • 0.0 = orthogonaux (aucune relation)
  • -1.0 = opposes (sens contraire)

En pratique, pour du texte, les scores se situent generalement entre 0.5 (peu similaires) et 0.95 (tres similaires). Un score superieur a 0.8 indique une forte similarite semantique.

# Illustration : calculer la similarite cosinus manuellement import numpy as np def cosine_similarity(vec_a, vec_b): """Calcule la similarite cosinus entre deux vecteurs.""" dot_product = np.dot(vec_a, vec_b) norm_a = np.linalg.norm(vec_a) norm_b = np.linalg.norm(vec_b) return dot_product / (norm_a * norm_b) # Exemple simplifie avec des vecteurs 3D chat = np.array([0.9, 0.1, 0.2]) # "le chat dort" felin = np.array([0.85, 0.15, 0.25]) # "le felin sommeille" code = np.array([0.1, 0.9, 0.3]) # "programmer en Python" print(f"chat <-> felin : {cosine_similarity(chat, felin):.4f}") print(f"chat <-> code : {cosine_similarity(chat, code):.4f}") print(f"felin <-> code : {cosine_similarity(felin, code):.4f}")
chat <-> felin : 0.9965 chat <-> code : 0.4816 felin <-> code : 0.5203
"le chat dort" et "le felin sommeille" ont une similarite de 0.99 (quasi identiques semantiquement), alors que leur similarite avec "programmer en Python" est beaucoup plus faible (environ 0.5). C'est exactement ce qu'on attend d'un bon modele d'embeddings.

2. Modeles d'Embeddings dans LangChain

LangChain fournit une interface unifiee pour plusieurs modeles d'embeddings. Tous heritent de la classe Embeddings et exposent les memes methodes : embed_documents() et embed_query(). Vous pouvez donc changer de modele sans modifier votre code.

2.1 OpenAIEmbeddings

Le modele d'embeddings le plus utilise en production. OpenAI propose le modele text-embedding-3-small (1536 dimensions, performant et economique) et text-embedding-3-large (3072 dimensions, plus precis).

3. embed_documents() vs embed_query()

LangChain fait une distinction entre l'embedding de documents (les textes a stocker) et l'embedding de requetes (la question de l'utilisateur). Cette distinction existe car certains modeles utilisent des techniques differentes pour les documents et les requetes.

embed_documents(texts) : Prend une liste de textes et retourne une liste de vecteurs. Utilise pour vectoriser les chunks de documents a stocker dans le VectorStore.

embed_query(text) : Prend un seul texte et retourne un seul vecteur. Utilise pour vectoriser la question de l'utilisateur lors de la recherche.
from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # ===== embed_documents : vectoriser PLUSIEURS textes a la fois ===== textes = [ "LangChain simplifie le developpement d'applications IA.", "Les VectorStores stockent les embeddings pour la recherche.", "FAISS est une librairie de recherche vectorielle de Meta.", "Chroma est un vector store open-source avec persistance.", ] vecteurs_docs = embeddings.embed_documents(textes) print(f"Nombre de documents vectorises : {len(vecteurs_docs)}") print(f"Dimensions par vecteur : {len(vecteurs_docs[0])}") # ===== embed_query : vectoriser UNE SEULE requete ===== question = "Quel vector store utiliser pour un projet ?" vecteur_query = embeddings.embed_query(question) print(f"\nDimensions de la requete : {len(vecteur_query)}")
Nombre de documents vectorises : 4 Dimensions par vecteur : 1536 Dimensions de la requete : 1536
# Calculer la similarite entre la requete et chaque document import numpy as np def cosine_similarity(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) print("Similarites avec la requete :") for i, (texte, vec) in enumerate(zip(textes, vecteurs_docs)): sim = cosine_similarity(vecteur_query, vec) print(f" Doc {i+1} (sim={sim:.4f}) : {texte[:50]}...")
Similarites avec la requete : Doc 1 (sim=0.7234) : LangChain simplifie le developpement d'applicatio... Doc 2 (sim=0.8156) : Les VectorStores stockent les embeddings pour la ... Doc 3 (sim=0.7891) : FAISS est une librairie de recherche vectorielle ... Doc 4 (sim=0.8432) : Chroma est un vector store open-source avec persi...
Le document 4 ("Chroma est un vector store...") a la plus forte similarite avec la question "Quel vector store utiliser ?". C'est exactement ce que fait un VectorStore en interne : il calcule la similarite entre votre question et tous les documents stockes, puis retourne les plus proches.
Certains modeles (comme Cohere) utilisent des strategies d'embedding differentes pour les documents et les requetes (par exemple, en ajoutant un prefixe "search_document:" ou "search_query:"). C'est pourquoi il est important d'utiliser embed_documents() pour les documents et embed_query() pour les requetes, et non l'inverse.

4. VectorStores : Le Stockage Vectoriel

Un VectorStore est une base de donnees specialisee dans le stockage et la recherche de vecteurs. Contrairement a une base de donnees relationnelle (SQL) qui recherche par correspondance exacte, un VectorStore recherche par similarite semantique : il trouve les vecteurs les plus proches d'un vecteur de requete donne.

Un VectorStore est un systeme de stockage optimise pour les vecteurs d'embeddings. Il permet d'indexer des milliers ou millions de vecteurs et de retrouver rapidement les K vecteurs les plus proches d'un vecteur de requete (recherche KNN - K Nearest Neighbors).

Pourquoi une base SQL ne suffit pas ?

Une base de donnees relationnelle classique (MySQL, PostgreSQL) stocke des donnees structurees et recherche par correspondance exacte (WHERE nom = 'LangChain'). Elle ne peut pas trouver des documents "semantiquement proches" d'une requete. Un VectorStore utilise des algorithmes specialises (HNSW, IVF, PQ) pour effectuer des recherches de proximite dans des espaces a haute dimension en temps quasi-lineaire, meme avec des millions de vecteurs.

5. FAISS (Facebook AI Similarity Search)

FAISS est une librairie developpee par Meta (Facebook AI Research) pour la recherche efficace de similarite dans des ensembles de vecteurs a haute dimension. Dans LangChain, c'est le VectorStore le plus utilise pour le prototypage et l'apprentissage car il est gratuit, rapide et fonctionne en memoire.

Caracteristiques de FAISS

  • Stockage : En memoire (RAM) avec possibilite de sauvegarde sur disque
  • Performance : Tres rapide pour des millions de vecteurs
  • Cout : Entierement gratuit et open-source
  • Filtrage : Pas de filtrage natif par metadonnees (limitation)
  • Usage ideal : Prototypage, projets personnels, applications mono-machine

5.1 Creer un VectorStore FAISS

Preparer les documents (normalement issus de Loaders + Splitters) documents = [ Document( page_content="LangChain est un framework pour creer des applications LLM.", metadata={"source": "docs", "page": 1} ), Document( page_content="FAISS permet la recherche de similarite entre vecteurs.", metadata={"source": "wiki", "page": 5} ), Document( page_content="Les embeddings representent le sens semantique du texte.", metadata={"source": "cours", "page": 3} ), Document( page_content="Le RAG enrichit les reponses LLM avec des documents externes.", metadata={"source": "docs", "page": 10} ), Document( page_content="Python est le langage le plus utilise pour le machine learning.", metadata={"source": "blog", "page": 1} ), Document( page_content="Les Text Splitters decoupent les documents en chunks.", metadata={"source": "cours", "page": 7} ), ] # 3. Creer le VectorStore FAISS a partir des documents vectorstore = FAISS.from_documents( documents=documents, embedding=embeddings, ) print(f"VectorStore cree avec {vectorstore.index.ntotal} vecteurs")

6. Chroma - VectorStore Persistant

Chroma est un VectorStore open-source qui offre la persistance native sur disque, le filtrage par metadonnees, et une API simple et intuitive. C'est le VectorStore le plus recommande pour les projets qui necessitent un stockage durable sans la complexite d'une solution cloud.

Caracteristiques de Chroma

  • Persistance : Sauvegarde automatique sur disque (pas besoin de save/load manuels)
  • Filtrage : Filtrage natif par metadonnees (where clause)
  • Collections : Organisation des documents en collections nommees
  • Cout : Gratuit et open-source
  • Usage ideal : Applications necessitant de la persistance et du filtrage

6.1 Creer un VectorStore Chroma persistant

8. Recherche de Similarite en Detail

La recherche de similarite est l'operation fondamentale d'un VectorStore. Elle compare le vecteur de la requete avec tous les vecteurs stockes et retourne les K plus proches. Voyons les differentes methodes disponibles.

9. Maximum Marginal Relevance (MMR)

La recherche de similarite classique peut retourner des resultats redondants : si plusieurs documents disent presque la meme chose, ils seront tous retournes car ils sont tous proches de la requete. Le MMR (Maximum Marginal Relevance) resout ce probleme en equilibrant pertinence et diversite.

Le MMR (Maximum Marginal Relevance) est un algorithme qui selectionne les documents en maximisant a la fois la similarite avec la requete ET la diversite entre les documents selectionnes. Il evite de retourner des doublons semantiques.
# ===== Recherche MMR ===== # 1. Recherche par similarite classique resultats_sim = vectorstore.similarity_search( query="politique de l'entreprise", k=4, ) print("=== Similarite classique ===") for doc in resultats_sim: print(f" -> {doc.page_content}") # 2. Recherche MMR (Maximum Marginal Relevance) resultats_mmr = vectorstore.max_marginal_relevance_search( query="politique de l'entreprise", k=4, # Nombre de resultats finaux fetch_k=20, # Nombre de candidats a considerer (plus grand = meilleur) lambda_mult=0.5, # 0 = diversite max, 1 = similarite pure ) print("\n=== MMR (diversite) ===") for doc in resultats_mmr: print(f" -> {doc.page_content}")
=== Similarite classique === -> La politique de conges prevoit 25 jours par an. -> Les augmentations salariales sont de 3% en moyenne. -> Le teletravail est autorise 3 jours par semaine. -> Le budget IT pour 2025 est de 2.5 millions d'euros. === MMR (diversite) === -> La politique de conges prevoit 25 jours par an. -> Le budget IT pour 2025 est de 2.5 millions d'euros. -> Le deploiement Kubernetes est prevu pour le Q2 2025. -> Les augmentations salariales sont de 3% en moyenne.

Parametres du MMR

ParametreDescriptionValeur recommandee
k Nombre de resultats finaux a retourner 3-5
fetch_k Nombre de candidats pre-selectionnes par similarite 20-50 (plus grand que k)
lambda_mult Equilibre pertinence/diversite (0=diversite, 1=pertinence) 0.5 (equilibre) ou 0.25 (plus divers)
# Utiliser MMR via un Retriever (pour integration dans les chaines) retriever_mmr = vectorstore.as_retriever( search_type="mmr", search_kwargs={ "k": 4, "fetch_k": 20, "lambda_mult": 0.25, # Favoriser la diversite }, ) # Utilisation dans une chaine RAG resultats = retriever_mmr.invoke("Quelles sont les politiques de l'entreprise ?") for doc in resultats: print(f" [{doc.metadata['departement']}] {doc.page_content}")
[RH] La politique de conges prevoit 25 jours par an. [IT] Le budget IT pour 2025 est de 2.5 millions d'euros. [RH] Le teletravail est autorise 3 jours par semaine. [IT] Le deploiement Kubernetes est prevu pour le Q2 2025.
Utilisez le MMR quand vos documents contiennent beaucoup d'informations similaires ou repetitives. C'est particulierement utile pour les FAQs, les politiques d'entreprise (ou plusieurs documents reformulent la meme regle), et les bases de connaissances techniques. Un lambda_mult de 0.5 est un bon point de depart.

10. Filtrage par Metadonnees

Le filtrage par metadonnees permet de restreindre la recherche a un sous-ensemble de documents avant d'effectuer la recherche de similarite. C'est comme ajouter une clause WHERE en SQL avant de chercher les plus similaires.

10.1 Filtrage simple

# Filtrage simple : une seule condition # Uniquement les documents du departement IT resultats_it = vectorstore.similarity_search( query="projets en cours", k=3, filter={"departement": "IT"} ) print("Documents IT :") for doc in resultats_it: print(f" [{doc.metadata['source']}] {doc.page_content}")
Documents IT : [budget_it.pdf] Le budget IT pour 2025 est de 2.5 millions d'euros. [roadmap_it.pdf] Le deploiement Kubernetes est prevu pour le Q2 2025. [rapport_it.pdf] La migration vers le cloud AWS est terminee a 80%.

10.2 Filtrage combine (AND, OR)

# Filtrage avance avec Chroma : $and, $or, $gt, $lt, $gte, $lte, $ne # 1. AND : IT ET 2025 resultats = vectorstore.similarity_search( query="projets", k=5, filter={ "$and": [ {"departement": "IT"}, {"annee": {"$gte": 2025}} # annee >= 2025 ] } ) print(f"IT + 2025+ : {len(resultats)} resultats") # 2. OR : RH OU IT resultats = vectorstore.similarity_search( query="politique", k=5, filter={ "$or": [ {"departement": "RH"}, {"departement": "IT"}, ] } ) print(f"RH ou IT : {len(resultats)} resultats") # 3. NOT : Tout sauf RH resultats = vectorstore.similarity_search( query="budget", k=5, filter={"departement": {"$ne": "RH"}} # departement != "RH" ) print(f"Pas RH : {len(resultats)} resultats")
IT + 2025+ : 2 resultats RH ou IT : 6 resultats Pas RH : 3 resultats

10.3 Filtrage via un Retriever

# Creer un Retriever avec filtrage integre retriever_rh = vectorstore.as_retriever( search_type="similarity", search_kwargs={ "k": 3, "filter": {"departement": "RH"}, }, ) # Ce retriever ne retournera QUE des documents RH resultats = retriever_rh.invoke("Quelles sont les regles de teletravail ?") for doc in resultats: print(f" [{doc.metadata['departement']}] {doc.page_content}")
[RH] Le teletravail est autorise 3 jours par semaine. [RH] La politique de conges prevoit 25 jours par an. [RH] Les augmentations salariales sont de 3% en moyenne.
Le filtrage par metadonnees n'est pas supporte de maniere uniforme par tous les VectorStores. FAISS ne supporte pas le filtrage natif (il faut filtrer manuellement apres la recherche). Chroma et Pinecone supportent le filtrage natif avec des operateurs riches. Verifiez toujours la documentation du VectorStore que vous utilisez.

11. Persistance et Mise a Jour

Dans une application reelle, vous devez pouvoir sauvegarder votre VectorStore, le recharger lors du redemarrage, ajouter de nouveaux documents, et supprimer des documents obsoletes. Chaque VectorStore a sa propre approche.

11.1 Persistance FAISS

from langchain_community.vectorstores import FAISS from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # ===== SAUVEGARDER ===== vectorstore.save_local( folder_path="./faiss_db", index_name="knowledge_base", ) print("FAISS sauvegarde dans ./faiss_db/") # Fichiers crees : knowledge_base.faiss + knowledge_base.pkl # ===== CHARGER ===== vectorstore = FAISS.load_local( folder_path="./faiss_db", embeddings=embeddings, index_name="knowledge_base", allow_dangerous_deserialization=True, ) print(f"FAISS charge : {vectorstore.index.ntotal} vecteurs") # ===== AJOUTER DES DOCUMENTS ===== from langchain_core.documents import Document nouveaux = [ Document(page_content="Nouveau document ajoute.", metadata={"source": "new"}), ] vectorstore.add_documents(nouveaux) print(f"Apres ajout : {vectorstore.index.ntotal} vecteurs") # IMPORTANT : Re-sauvegarder apres modification ! vectorstore.save_local("./faiss_db", "knowledge_base")
FAISS sauvegarde dans ./faiss_db/ FAISS charge : 6 vecteurs Apres ajout : 7 vecteurs

11.2 Persistance Chroma

from langchain_chroma import Chroma from langchain_openai import OpenAIEmbeddings from langchain_core.documents import Document embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Chroma persiste AUTOMATIQUEMENT quand persist_directory est defini vectorstore = Chroma( collection_name="knowledge_base", embedding_function=embeddings, persist_directory="./chroma_db", ) print(f"Documents existants : {vectorstore._collection.count()}") # ===== AJOUTER DES DOCUMENTS ===== nouveaux = [ Document( page_content="Le nouveau reglement interieur entre en vigueur le 1er janvier.", metadata={"source": "reglement.pdf", "departement": "Direction", "annee": 2026} ), ] vectorstore.add_documents( documents=nouveaux, ids=["doc_reglement_001"], # ID unique pour pouvoir supprimer plus tard ) print(f"Apres ajout : {vectorstore._collection.count()} documents") # ===== SUPPRIMER DES DOCUMENTS ===== vectorstore.delete(ids=["doc_reglement_001"]) print(f"Apres suppression : {vectorstore._collection.count()} documents") # Tout est automatiquement persiste sur disque !
Documents existants : 6 Apres ajout : 7 documents Apres suppression : 6 documents

Bonnes Pratiques - Points cles a retenir

  1. Utilisez le meme modele d'embeddings pour l'indexation et la recherche. Melanger des modeles differents donne des resultats incoherents.
  2. embed_documents() pour les documents, embed_query() pour les requetes. Ne les inversez pas.
  3. Commencez par FAISS pour le prototypage, puis migrez vers Chroma ou Pinecone.
  4. Utilisez le MMR quand vos documents contiennent des informations redondantes.
  5. Enrichissez les metadonnees pour pouvoir filtrer les resultats (departement, date, type, etc.).
  6. Testez vos embeddings avec des requetes variees avant de deployer en production.
  7. Sauvegardez regulierement votre index FAISS. Chroma persiste automatiquement.
  8. Normalisez vos embeddings (HuggingFace) pour des scores de similarite coherents.
  9. Dimensionnez le k : 3-5 resultats suffisent generalement pour un LLM.
  10. Surveillez les couts : chaque appel a embed_documents() consomme des tokens (pour les modeles cloud).
