Memory dans LangChain - Donner de la Memoire aux LLMs

Cours 4 - Gerer l'historique conversationnel : des strategies classiques a l'approche moderne LCEL

1. Le probleme : les LLMs sont sans memoire

Les LLMs (Large Language Models) sont stateless (sans etat). Chaque appel a l'API est completement independant : le modele ne se "souvient" pas des echanges precedents. Pour creer un chatbot capable de maintenir une conversation coherente, il faut gerer la memoire manuellement.

Quand vous envoyez un message a ChatGPT via l'API, le modele ne recoit que ce message unique. Il n'a aucune connaissance des messages precedents. Voici ce qui se passe sans memoire :

from langchain_openai import ChatOpenAI model = ChatOpenAI(model="gpt-4o") # Premier appel reponse1 = model.invoke("Je m'appelle Alice.") print("Reponse 1:", reponse1.content) # Deuxieme appel - le modele a OUBLIE le premier ! reponse2 = model.invoke("Comment je m'appelle ?") print("Reponse 2:", reponse2.content)
Reponse 1: Bonjour Alice ! Enchante de faire votre connaissance. Comment puis-je vous aider ? Reponse 2: Je suis desole, mais je n'ai pas acces a votre nom. Pourriez-vous me le dire ?
Le modele ne se souvient pas que l'utilisateur s'appelle Alice ! Chaque appel invoke() est independant. C'est le comportement par defaut de tous les LLMs via API.

Pour resoudre ce probleme, il faut stocker l'historique des messages et le re-injecter dans le prompt a chaque nouvel appel. C'est exactement ce que fait la "memoire" dans LangChain.

L'interface web de ChatGPT gere la memoire cote serveur : elle stocke tous vos messages et les envoie a chaque appel. Quand vous utilisez l'API directement, vous devez gerer cela vous-meme. C'est exactement ce que LangChain automatise avec ses classes de memoire.

2. Comment fonctionne la memoire

Le principe est simple : a chaque echange, on stocke le message de l'utilisateur et la reponse de l'IA. Lors du prochain appel, on reinjecte tout l'historique dans le prompt pour que le LLM ait le contexte de la conversation.

Le cycle de la memoire

  1. L'utilisateur envoie un nouveau message
  2. Le systeme recupere l'historique depuis le store de memoire
  3. L'historique est injecte dans le prompt avec le nouveau message
  4. Le LLM recoit le contexte complet et genere une reponse
  5. Le message et la reponse sont stockes dans la memoire
  6. Le cycle recommence au prochain message
La "memoire" des LLMs est en realite une illusion. Le modele ne memorise rien entre les appels. Ce sont les messages precedents qui sont re-envoyes dans le prompt a chaque fois. Plus la conversation est longue, plus le prompt est volumineux (et couteux en tokens). C'est pourquoi differentes strategies de memoire existent pour gerer cette croissance.

3. ConversationBufferMemory

ConversationBufferMemory est le type de memoire le plus simple : il stocke tous les messages de la conversation sans aucune limite ni compression. Chaque message (humain et IA) est conserve integralement.

3.1 Fonctionnement

Cette memoire agit comme un simple tampon (buffer) qui accumule tous les messages. A chaque nouvel echange, l'integralite de l'historique est envoyee au LLM.

from langchain.memory import ConversationBufferMemory from langchain_core.messages import HumanMessage, AIMessage # Creer la memoire memory = ConversationBufferMemory(return_messages=True) # Simuler une conversation memory.save_context( {"input": "Bonjour, je m'appelle Alice."}, {"output": "Bonjour Alice ! Comment puis-je vous aider ?"} ) memory.save_context( {"input": "Quel est le meilleur langage de programmation ?"}, {"output": "Cela depend du contexte. Python est populaire pour l'IA."} ) memory.save_context( {"input": "Et pour le web ?"}, {"output": "JavaScript est incontournable pour le developpement web."} ) # Voir tout l'historique historique = memory.load_memory_variables({}) for msg in historique["history"]: print(f"{msg.type}: {msg.content}")
human: Bonjour, je m'appelle Alice. ai: Bonjour Alice ! Comment puis-je vous aider ? human: Quel est le meilleur langage de programmation ? ai: Cela depend du contexte. Python est populaire pour l'IA. human: Et pour le web ? ai: JavaScript est incontournable pour le developpement web.

3.2 Utilisation dans un chatbot

from langchain.memory import ConversationBufferMemory from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder from langchain_openai import ChatOpenAI from langchain_core.output_parsers import StrOutputParser # 1. Memory memory = ConversationBufferMemory( return_messages=True, memory_key="history" ) # 2. Prompt avec placeholder pour l'historique prompt = ChatPromptTemplate.from_messages([ ("system", "Tu es un assistant amical et utile."), MessagesPlaceholder(variable_name="history"), ("human", "{input}"), ]) # 3. Modele model = ChatOpenAI(model="gpt-4o") # 4. Fonction chatbot def chat(message: str) -> str: # Charger l'historique history = memory.load_memory_variables({})["history"] # Construire et invoquer la chain chain = prompt | model | StrOutputParser() reponse = chain.invoke({"input": message, "history": history}) # Sauvegarder dans la memoire memory.save_context({"input": message}, {"output": reponse}) return reponse # Test de la conversation print(chat("Bonjour, je m'appelle Alice.")) print(chat("Quel est mon prenom ?")) print(chat("Recommande-moi un livre de science-fiction.")) print(chat("Et un autre dans le meme genre ?"))
Bonjour Alice ! Enchantee ! Comment puis-je vous aider aujourd'hui ? Votre prenom est Alice ! Je vous recommande "Dune" de Frank Herbert, un classique de la SF. Dans le meme genre de science-fiction epique, je vous suggere "Fondation" d'Isaac Asimov.
ConversationBufferMemory croit sans limite ! Apres 100 echanges, vous envoyez 200 messages (humain + IA) au LLM a chaque appel. Cela devient tres couteux en tokens et peut depasser la fenetre de contexte du modele (128K tokens pour GPT-4o). Ne l'utilisez que pour des conversations courtes.

4. ConversationBufferWindowMemory

ConversationBufferWindowMemory est une memoire a fenetre glissante. Elle ne conserve que les K derniers echanges (paires humain/IA). Les messages plus anciens sont supprimes definitivement.

4.1 Comparaison K=2 vs K=5

from langchain.memory import ConversationBufferWindowMemory # Fenetre de 2 echanges seulement memory_k2 = ConversationBufferWindowMemory(k=2, return_messages=True) # Fenetre de 5 echanges memory_k5 = ConversationBufferWindowMemory(k=5, return_messages=True) # Simuler 4 echanges dans les deux memoires echanges = [ ("Je m'appelle Alice.", "Bonjour Alice !"), ("J'aime la physique.", "La physique est fascinante !"), ("Je vis a Paris.", "Paris est une belle ville."), ("J'ai 25 ans.", "Vous etes dans la fleur de l'age !"), ] for human, ai in echanges: memory_k2.save_context({"input": human}, {"output": ai}) memory_k5.save_context({"input": human}, {"output": ai}) # Comparer ce que chaque memoire retient print("=== K=2 (2 derniers echanges) ===") for msg in memory_k2.load_memory_variables({})["history"]: print(f" {msg.type}: {msg.content}") print("\n=== K=5 (5 derniers echanges - tout est la) ===") for msg in memory_k5.load_memory_variables({})["history"]: print(f" {msg.type}: {msg.content}")
=== K=2 (2 derniers echanges) === human: Je vis a Paris. ai: Paris est une belle ville. human: J'ai 25 ans. ai: Vous etes dans la fleur de l'age ! === K=5 (5 derniers echanges - tout est la) === human: Je m'appelle Alice. ai: Bonjour Alice ! human: J'aime la physique. ai: La physique est fascinante ! human: Je vis a Paris. ai: Paris est une belle ville. human: J'ai 25 ans. ai: Vous etes dans la fleur de l'age !
Avec K=2, le chatbot a "oublie" le nom de l'utilisateur et son interet pour la physique. Avec K=5, tout est encore accessible. Le choix de K est un compromis entre cout (tokens) et richesse du contexte.

Quand utiliser ConversationBufferWindowMemory ?

  • Conversations longues ou le contexte recent est le plus important
  • Applications a budget token limite
  • Chatbots de support client (les 3-5 derniers echanges suffisent generalement)
  • Quand la taille previsible de la memoire doit etre controlee
Le parametre k represente le nombre de paires humain/IA, pas le nombre de messages. Avec k=3, vous conservez 6 messages (3 humains + 3 IA). Choisissez K en fonction de la fenetre de contexte de votre modele et de votre budget.

5. ConversationSummaryMemory

ConversationSummaryMemory utilise un LLM pour resumer la conversation au fur et a mesure. Au lieu de stocker tous les messages, elle maintient un resume synthetique qui est mis a jour a chaque nouvel echange.

5.1 Comment la compression fonctionne

Bienvenue, en quoi puis-je vous aider ?"} ) memory.save_context( {"input": "Je travaille sur un projet de chatbot avec LangChain."}, {"output": "LangChain est excellent pour ca. Quel type de chatbot ?"} ) memory.save_context( {"input": "Un chatbot RAG pour notre documentation interne."}, {"output": "RAG est parfait pour ca. Avez-vous deja un vector store ?"} ) memory.save_context( {"input": "Oui, j'utilise FAISS avec des embeddings OpenAI."}, {"output": "FAISS est un bon choix. Resume de la conversation : L'utilisatrice Alice est une developpeuse Python basee a Paris. Elle travaille sur un chatbot RAG avec LangChain pour leur documentation interne. Au lieu de stocker 8 messages (4 paires), la memoire ne contient qu'un seul paragraphe resume. La taille du resume croit tres lentement, meme pour de longues conversations. C'est ideal pour les sessions de plusieurs heures.

5.2 Evolution du resume au fil de la conversation

Comment le resume evolue

EchangeContenu du resume
Apres echange 1"Alice est une developpeuse Python a Paris."
Apres echange 2"Alice, dev Python a Paris, travaille sur un chatbot LangChain."
Apres echange 3"Alice, dev Python a Paris, cree un chatbot RAG pour la doc interne."
Apres echange 4"Alice, dev Python a Paris, cree un chatbot RAG avec FAISS + embeddings OpenAI."

Le resume est incrementalement mis a jour : le LLM recoit le resume precedent + les nouveaux messages et produit un resume actualise.

Chaque mise a jour du resume necessite un appel LLM supplementaire. Utilisez un modele rapide et economique (comme gpt-4o-mini) pour la generation des resumes, et reservez le modele puissant pour les reponses. Cout supplementaire : environ 0.001$ par resume avec gpt-4o-mini.

6. ConversationSummaryBufferMemory

ConversationSummaryBufferMemory est un hybride entre BufferMemory et SummaryMemory. Les messages recents sont conserves integralement (comme Buffer), tandis que les messages plus anciens sont resumes (comme Summary). C'est souvent considere comme le meilleur compromis.
from langchain.memory import ConversationSummaryBufferMemory from langchain_openai import ChatOpenAI memory = ConversationSummaryBufferMemory( llm=ChatOpenAI(model="gpt-4o-mini"), max_token_limit=300, # Quand le buffer depasse 300 tokens, resumer les anciens return_messages=True, ) # Les premiers messages sont stockes integralement memory.save_context( {"input": "Bonjour, je suis etudiant en informatique."}, {"output": "Bienvenue ! En quelle annee etes-vous ?"} ) memory.save_context( {"input": "En Master 2, specialite IA."}, {"output": "Excellent choix ! L'IA est un domaine passionnant."} ) # Quand le buffer depasse max_token_limit, les anciens messages sont resumes memory.save_context( {"input": "Je travaille sur mon memoire sur les transformers."}, {"output": "Les transformers sont au coeur de l'IA moderne. Quel aspect etudiez-vous ?"} ) memory.save_context( {"input": "L'attention multi-tete et son optimisation."}, {"output": "L'optimisation de l'attention est un sujet tres actuel. Flash Attention est pertinent."} ) memory.save_context( {"input": "Oui j'etudie Flash Attention 2 et son impact sur l'inference."}, {"output": "FA2 reduit la complexite memoire de O(n2) a O(n). Un gain majeur."} ) # Voir le resultat historique = memory.load_memory_variables({})["history"] for msg in historique: print(f"{msg.type}: {msg.content[:80]}...")
system: L'utilisateur est etudiant en M2 informatique specialite IA, travaillant sur un memoire... human: Oui j'etudie Flash Attention 2 et son impact sur l'inference. ai: FA2 reduit la complexite memoire de O(n2) a O(n). Un gain majeur.
Les anciens echanges sont compresses en un resume systeme, tandis que les messages les plus recents restent intacts. Le LLM a ainsi le contexte global (via le resume) ET les details recents (via les messages complets).
Le parametre max_token_limit controle le seuil a partir duquel les anciens messages commencent a etre resumes. Un bon reglage typique est entre 200 et 500 tokens. Trop bas : trop de resumes (couteux). Trop haut : trop de tokens envoyes au LLM a chaque appel.

7. ConversationTokenBufferMemory

ConversationTokenBufferMemory est similaire a WindowMemory, mais au lieu de limiter par nombre de messages, elle limite par nombre de tokens. Cela permet un controle plus precis du cout et de la taille du contexte.
from langchain.memory import ConversationTokenBufferMemory from langchain_openai import ChatOpenAI memory = ConversationTokenBufferMemory( llm=ChatOpenAI(model="gpt-4o"), # LLM utilise pour compter les tokens max_token_limit=200, # Maximum 200 tokens dans la memoire return_messages=True, ) # Ajouter des messages memory.save_context( {"input": "Explique les reseaux de neurones convolutifs."}, {"output": "Les CNN utilisent des filtres pour detecter des motifs dans les images. Chaque couche detecte des features de plus en plus complexes."} ) memory.save_context( {"input": "Et les RNN ?"}, {"output": "Les RNN traitent des sequences. Ils ont une memoire interne qui permet de tenir compte du contexte precedent."} ) memory.save_context( {"input": "Quelle est la difference avec les transformers ?"}, {"output": "Les transformers utilisent l'attention au lieu de la recurrence. Ils peuvent traiter toute la sequence en parallele."} ) # Les anciens messages sont supprimes quand la limite est depassee historique = memory.load_memory_variables({})["history"] for msg in historique: print(f"{msg.type}: {msg.content}") # Probablement seul le dernier echange reste (les anciens depasses en tokens)
human: Quelle est la difference avec les transformers ? ai: Les transformers utilisent l'attention au lieu de la recurrence. Ils peuvent traiter toute la sequence en parallele.

Quand utiliser ConversationTokenBufferMemory ?

  • Quand vous avez un budget token strict (par exemple, max 500 tokens pour l'historique)
  • Quand les messages ont des longueurs tres variables (certains courts, d'autres tres longs)
  • Pour garantir que l'historique ne depassera jamais une taille donnee
  • Pour controle precis des couts d'API

La difference avec WindowMemory : avec k=3, trois longs messages pourraient consommer 2000 tokens. Avec TokenBufferMemory, vous controlez le total exact.

8. Memory avec LCEL - L'approche moderne

RunnableWithMessageHistory est la facon moderne et recommandee de gerer la memoire dans LangChain v0.3+. Elle s'integre nativement avec LCEL et supporte la gestion de sessions multiples (plusieurs conversations en parallele).
Les classes de memoire legacy (ConversationBufferMemory, etc.) fonctionnent encore mais sont progressivement remplacees par RunnableWithMessageHistory qui s'integre mieux avec LCEL. Pour tout nouveau projet, privilegiez l'approche moderne.

8.1 Les composants cles

ComposantImportRole
RunnableWithMessageHistory langchain_core.runnables.history Wrapper LCEL qui gere automatiquement l'historique
InMemoryChatMessageHistory langchain_core.chat_history Stockage des messages en memoire (RAM)
BaseChatMessageHistory langchain_core.chat_history Classe de base pour les implementations custom

8.2 Chatbot complet avec RunnableWithMessageHistory

Creer le prompt avec un placeholder pour l'historique prompt = ChatPromptTemplate.from_messages([ ("system", "Tu es un assistant pedagogique specialise en IA."), MessagesPlaceholder(variable_name="history"), ("human", "{input}"), ]) # 2. Creer la chain LCEL chain = prompt | ChatOpenAI(model="gpt-4o") | StrOutputParser() # 3. Dictionnaire pour stocker les sessions sessions = {} def get_session_history(session_id: str) -> InMemoryChatMessageHistory: """Retourne ou cree l'historique pour une session donnee.""" if session_id not in sessions: sessions[session_id] = InMemoryChatMessageHistory() return sessions[session_id] # 4. Wrapper avec RunnableWithMessageHistory chatbot = RunnableWithMessageHistory( chain, get_session_history, input_messages_key="input", history_messages_key="history", ) # 5. Utilisation avec des sessions differentes config_alice = {"configurable": {"session_id": "alice_session"}} config_bob = {"configurable": {"session_id": "bob_session"}} # Conversation d'Alice print("Alice:", chatbot.invoke( {"input": "Salut ! Je suis Alice, j'etudie le NLP."}, config=config_alice )) # Conversation de Bob (session separee) print("Bob:", chatbot.invoke( {"input": "Bonjour, je suis Bob, je travaille sur la vision par ordinateur."}, config=config_bob )) # Alice continue sa conversation (sa memoire est intacte) print("Alice:", chatbot.invoke( {"input": "Quel est mon domaine d'etude ?"}, config=config_alice )) # Bob continue (sa memoire est separee) print("Bob:", chatbot.invoke( {"input": "Quel est mon domaine ?"}, config=config_bob ))
Alice: Bonjour Alice ! Le NLP est un domaine fascinant. Qu'etudiez-vous en particulier ? Bob: Salut Bob ! La vision par ordinateur est passionnante. Sur quoi travaillez-vous ? Alice: Votre domaine d'etude est le NLP (Natural Language Processing) ! Bob: Votre domaine est la vision par ordinateur (Computer Vision) !
Chaque session a sa propre memoire isolee. Alice et Bob peuvent converser simultanement sans interference. C'est essentiel pour les applications multi-utilisateurs.

9. Memoire persistante

La memoire persistante survit aux redemarrages de l'application. Contrairement a InMemoryChatMessageHistory qui stocke en RAM (perdue au redemarrage), les solutions persistantes utilisent une base de donnees externe (Redis, PostgreSQL, fichiers, etc.).

9.1 Pourquoi persister la memoire ?

  • In-memory (RAM) : rapide mais volatile. Perdu au redemarrage du serveur, au crash, ou au deploy.
  • Redis : rapide, en memoire mais avec persistence sur disque. Ideal pour le cache et les sessions.
  • PostgreSQL / SQLite : persistant, fiable, interrogeable. Ideal pour l'historique a long terme.
  • Fichiers : simple, pas de dependance externe. Adapt pour le prototypage.

9.2 Avec Redis

from langchain_community.chat_message_histories import RedisChatMessageHistory from langchain_core.runnables.history import RunnableWithMessageHistory # Chaque session est stockee dans Redis avec un TTL optionnel def get_redis_history(session_id: str): return RedisChatMessageHistory( session_id=session_id, url="redis://localhost:6379", ttl=3600, # Expiration apres 1 heure (en secondes) ) chatbot_redis = RunnableWithMessageHistory( chain, get_redis_history, input_messages_key="input", history_messages_key="history", ) # La conversation survit aux redemarrages du serveur ! reponse = chatbot_redis.invoke( {"input": "Bonjour !"}, config={"configurable": {"session_id": "user_123"}} )
Installation : pip install redis langchain-community et un serveur Redis en cours d'execution.

9.6 Implementation custom d'un backend

from langchain_core.chat_history import BaseChatMessageHistory from langchain_core.messages import BaseMessage, HumanMessage, AIMessage import json from typing import List class MonBackendCustom(BaseChatMessageHistory): """Implementation personnalisee de stockage de messages.""" def __init__(self, session_id: str): self.session_id = session_id self.file_path = f"./data/{session_id}.json" @property def messages(self) -> List[BaseMessage]: """Lire les messages depuis le stockage.""" try: with open(self.file_path, "r") as f: data = json.load(f) messages = [] for item in data: if item["type"] == "human": messages.append(HumanMessage(content=item["content"])) else: messages.append(AIMessage(content=item["content"])) return messages except FileNotFoundError: return [] def add_message(self, message: BaseMessage) -> None: """Ajouter un message au stockage.""" messages = self.messages messages.append(message) data = [{"type": m.type, "content": m.content} for m in messages] with open(self.file_path, "w") as f: json.dump(data, f, indent=2) def clear(self) -> None: """Supprimer tous les messages.""" with open(self.file_path, "w") as f: json.dump([], f) # Utilisation avec RunnableWithMessageHistory chatbot_custom = RunnableWithMessageHistory( chain, lambda session_id: MonBackendCustom(session_id), input_messages_key="input", history_messages_key="history", )
Pour creer votre propre backend, heritez de BaseChatMessageHistory et implementez les proprietes/methodes messages, add_message() et clear(). C'est tout ce que LangChain a besoin pour fonctionner.

Tableau comparatif des types de memoire

Type Capacite Cout tokens Cout API Cas d'usage ideal
ConversationBufferMemory Illimitee Croit lineairement Faible (pas d'appel extra) Conversations courtes (<20 echanges)
ConversationBufferWindowMemory K derniers echanges Fixe (plafonne) Faible Chat general, support client
ConversationSummaryMemory Resume compresse Faible et stable Moyen (appel LLM par echange) Conversations tres longues
ConversationSummaryBufferMemory Resume + recents Moderee et stable Moyen Meilleur compromis general
ConversationTokenBufferMemory N tokens max Fixe (plafonne) Faible Budget token strict
RunnableWithMessageHistory Depend du backend Depend de l'implementation Faible Production LCEL, multi-sessions

Points cles a retenir

  • Les LLMs sont stateless : la memoire est geree cote application, pas cote modele
  • La memoire fonctionne en re-injectant l'historique dans le prompt a chaque appel
  • Plus la memoire est grande, plus l'appel est couteux en tokens
  • Choisissez le type de memoire selon : longueur prevue de la conversation, budget token, besoin de precision
  • RunnableWithMessageHistory est l'approche recommandee pour LangChain v0.3+
  • En production, utilisez un backend persistant (Redis, PostgreSQL) pour survivre aux redemarrages
  • La gestion par session_id permet plusieurs conversations simultanees
La memoire est l'un des aspects les plus importants et les plus sous-estimes des applications LLM. Un chatbot sans memoire est frustrant pour l'utilisateur. Un chatbot avec trop de memoire est couteux et lent. Trouvez le bon equilibre pour votre cas d'usage.
