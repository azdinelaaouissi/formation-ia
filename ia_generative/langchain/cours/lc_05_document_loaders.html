Document Loaders et Text Splitters

Cours 5 - Ingerer des Documents dans le Pipeline RAG avec LangChain

1. Le Pipeline d'Ingestion de Documents

Avant qu'un LLM puisse repondre a des questions sur vos documents, ces derniers doivent traverser un pipeline d'ingestion complet. Ce pipeline transforme des fichiers bruts (PDF, CSV, pages web, etc.) en vecteurs stockes dans une base vectorielle, prets a etre recherches semantiquement.

Le pipeline d'ingestion est la sequence d'etapes qui transforme des documents sources en representations vectorielles exploitables par un systeme RAG (Retrieval-Augmented Generation). Il comprend le chargement, le decoupage, la vectorisation et le stockage.
Ce pipeline est la fondation de tout systeme RAG. Chaque etape est cruciale : un mauvais decoupage ruinera la qualite de la recherche, meme avec le meilleur modele d'embeddings au monde. Dans ce cours, nous detaillons les deux premieres etapes : le chargement (Loaders) et le decoupage (Splitters).

Voici les etapes en detail :

  1. Chargement (Loading) : Lire le document source et extraire le texte brut avec ses metadonnees.
  2. Decoupage (Splitting) : Diviser le texte brut en morceaux (chunks) de taille appropriee.
  3. Vectorisation (Embedding) : Convertir chaque chunk en vecteur numerique capturant le sens semantique.
  4. Stockage (Storing) : Sauvegarder les vecteurs dans une base vectorielle pour la recherche par similarite.

2. L'Objet Document de LangChain

Tous les loaders de LangChain retournent des objets Document. C'est la structure de donnees fondamentale qui circule dans tout le pipeline. Comprendre cet objet est essentiel.

Un Document LangChain est un objet contenant deux attributs principaux : page_content (le texte du document) et metadata (un dictionnaire de metadonnees associees comme la source, le numero de page, etc.).
from langchain_core.documents import Document # Creation manuelle d'un Document doc = Document( page_content="LangChain est un framework pour developper des applications basees sur les LLM.", metadata={ "source": "cours_langchain.pdf", "page": 1, "author": "Prof. Martin", "date": "2025-09-15" } ) # Acceder au contenu print(doc.page_content) # "LangChain est un framework pour developper des applications basees sur les LLM." # Acceder aux metadonnees print(doc.metadata) # {"source": "cours_langchain.pdf", "page": 1, "author": "Prof. Martin", "date": "2025-09-15"} # Acceder a une metadonnee specifique print(doc.metadata["source"]) # "cours_langchain.pdf"
LangChain est un framework pour developper des applications basees sur les LLM. {'source': 'cours_langchain.pdf', 'page': 1, 'author': 'Prof. Martin', 'date': '2025-09-15'} cours_langchain.pdf

Pourquoi les metadonnees sont-elles si importantes ?

Les metadonnees permettent de :

  • Filtrer les resultats de recherche (ex: uniquement les documents d'une source specifique)
  • Citer la source exacte dans la reponse generee (page, paragraphe, URL)
  • Tracer l'origine de l'information pour la verification
  • Organiser les documents par categorie, date, auteur, etc.
# Creer une liste de Documents manuellement documents = [ Document( page_content="Python est un langage de programmation polyvalent.", metadata={"source": "wiki", "sujet": "programmation"} ), Document( page_content="LangChain facilite la creation d'applications IA.", metadata={"source": "blog", "sujet": "IA"} ), Document( page_content="Les vecteurs representent le sens semantique du texte.", metadata={"source": "cours", "sujet": "embeddings"} ), ] for doc in documents: print(f"[{doc.metadata['source']}] {doc.page_content[:50]}...")

3. Les Principaux Document Loaders

LangChain fournit des dizaines de loaders specialises pour differents formats de fichiers. Chaque loader gere les specificites de son format (encodage, pagination, structure) et retourne une liste d'objets Document avec des metadonnees appropriees.

Loader Format Package Metadonnees typiques
TextLoader .txt langchain_community source (chemin du fichier)
PyPDFLoader .pdf langchain_community source, page
CSVLoader .csv langchain_community source, row (numero de ligne)
WebBaseLoader URL (web) langchain_community source (URL), title
DirectoryLoader Dossier entier langchain_community source (chemin de chaque fichier)
UnstructuredLoader Multi-format langchain_unstructured source, category, element_type
JSONLoader .json langchain_community source, jq_schema path
Docx2txtLoader .docx langchain_community source

3.1 TextLoader - Fichiers Texte

Le loader le plus simple. Il charge un fichier texte brut et retourne un seul Document contenant tout le contenu du fichier.

from langchain_community.document_loaders import TextLoader # Charger un fichier texte loader = TextLoader("./data/mon_fichier.txt", encoding="utf-8") documents = loader.load() print(f"Nombre de documents : {len(documents)}") print(f"Contenu (100 premiers chars) : {documents[0].page_content[:100]}") print(f"Metadonnees : {documents[0].metadata}")
Nombre de documents : 1 Contenu (100 premiers chars) : LangChain est un framework open-source qui permet de creer des applications basees sur les modeles de Metadonnees : {'source': './data/mon_fichier.txt'}
Le TextLoader charge TOUT le fichier en un seul Document. Pour un fichier de 10 000 lignes, vous obtenez un seul Document enorme. C'est pourquoi le Text Splitter est indispensable apres le chargement. Pensez toujours a specifier l'encodage (encoding="utf-8") pour eviter les erreurs avec les accents.

4. Text Splitters : Pourquoi le Decoupage est Crucial

Apres le chargement, le texte brut est souvent trop long pour etre utilise directement. Un document de 50 pages PDF contient potentiellement des dizaines de milliers de caracteres. Les modeles d'embeddings ont des limites de tokens, et la recherche semantique fonctionne mieux avec des morceaux de taille raisonnable.

Un Text Splitter est un outil qui decoupe un texte long en morceaux plus petits (appeles chunks) tout en essayant de preserver la coherence semantique du contenu. Le choix de la strategie de decoupage a un impact majeur sur la qualite du systeme RAG.

Le dilemme de la taille des chunks

  • Chunks trop grands (ex: 5000 caracteres) : Le sens est dilue, l'embedding capture mal le sujet precis. La recherche retourne des resultats bruyants contenant beaucoup d'informations non pertinentes.
  • Chunks trop petits (ex: 50 caracteres) : Le contexte est perdu, les phrases sont coupees en plein milieu. L'embedding d'une phrase isolee peut etre ambigu.
  • Chunks equilibres (ex: 500-1000 caracteres) : Bon compromis entre precision et contexte. Assez pour capturer une idee complete sans noyer le signal.
Le choix du splitter et de ses parametres est l'un des facteurs les plus determinants dans la qualite d'un systeme RAG. Un bon decoupage peut faire la difference entre un systeme qui repond correctement 80% du temps et un systeme qui atteint 95%. Experimentez toujours avec differents parametres sur vos propres donnees.

5. RecursiveCharacterTextSplitter - Le Splitter Incontournable

Le RecursiveCharacterTextSplitter est le splitter le plus recommande et le plus utilise dans LangChain. Il tente de decouper le texte en respectant la hierarchie naturelle du langage : d'abord par paragraphes, puis par phrases, puis par mots, et enfin par caracteres.

Le RecursiveCharacterTextSplitter fonctionne en essayant recursivement de decouper le texte avec une liste ordonnee de separateurs : ["\n\n", "\n", " ", ""]. Il commence par essayer le premier separateur (double saut de ligne = paragraphe), et si les chunks resultants sont encore trop grands, il passe au separateur suivant, et ainsi de suite.

Parametres essentiels

ParametreDescriptionValeur par defaut
chunk_size Taille maximale de chaque chunk (en caracteres) 1000
chunk_overlap Nombre de caracteres de chevauchement entre chunks consecutifs 200
separators Liste ordonnee de separateurs a essayer recursivement ["\n\n", "\n", " ", ""]
length_function Fonction pour mesurer la longueur d'un chunk len (nombre de caracteres)
is_separator_regex Indique si les separateurs sont des expressions regulieres False

Stocker les vecteurs dans un VectorStore 5. Creer un Retriever pour la recherche semantique 6. 7. TokenTextSplitter - Decoupage par Tokens

Les LLM ne comptent pas en caracteres mais en tokens. Un token est une unite de texte (mot, sous-mot, ponctuation) telle que traitee par le modele. Le TokenTextSplitter decoupe en se basant sur le nombre de tokens, ce qui est plus precis pour respecter les limites de contexte des modeles.

# pip install tiktoken from langchain_text_splitters import TokenTextSplitter # Decouper par nombre de tokens (pas de caracteres) splitter = TokenTextSplitter( encoding_name="cl100k_base", # Encodage utilise par GPT-4 et GPT-3.5 chunk_size=100, # 100 tokens par chunk chunk_overlap=20, # 20 tokens de chevauchement ) texte = """Les systemes de Retrieval-Augmented Generation (RAG) combinent la puissance des modeles de langage avec la precision des bases de connaissances. En recuperant des documents pertinents avant de generer une reponse, le RAG reduit considerablement les hallucinations et permet au modele de fournir des informations precises et a jour. Cette approche est devenue un standard dans l'industrie pour les applications d'IA d'entreprise, car elle permet de controler les sources d'information utilisees.""" chunks = splitter.split_text(texte) for i, chunk in enumerate(chunks): print(f"Chunk {i+1} : {chunk[:80]}...")
Chunk 1 : Les systemes de Retrieval-Augmented Generation (RAG) combinent la puissance des mo... Chunk 2 : le RAG reduit considerablement les hallucinations et permet au modele de fournir ...
Utilisez le TokenTextSplitter quand vous devez garantir que chaque chunk respecte une limite stricte de tokens (par exemple, pour un modele avec une fenetre de contexte limitee). L'encodage cl100k_base est celui utilise par les modeles OpenAI (GPT-4, GPT-3.5-turbo). Pour d'autres modeles, verifiez quel tokenizer ils utilisent.
# Combiner RecursiveCharacterTextSplitter avec un compteur de tokens from langchain_text_splitters import RecursiveCharacterTextSplitter import tiktoken # Creer la fonction de comptage de tokens encoding = tiktoken.get_encoding("cl100k_base") def token_length(text): return len(encoding.encode(text)) # Utiliser le RecursiveCharacterTextSplitter avec comptage de tokens splitter = RecursiveCharacterTextSplitter( chunk_size=256, # 256 tokens (pas caracteres !) chunk_overlap=30, length_function=token_length, # Fonction de mesure personnalisee ) # Maintenant les chunks font max 256 tokens chacun chunks = splitter.split_text(texte) for i, chunk in enumerate(chunks): print(f"Chunk {i+1} : {token_length(chunk)} tokens, {len(chunk)} caracteres")
Chunk 1 : 98 tokens, 487 caracteres Chunk 2 : 42 tokens, 175 caracteres

8. Splitters Specialises

LangChain fournit des splitters adaptes a des formats specifiques. Ces splitters comprennent la structure du document et creent des chunks semantiquement plus coherents en exploitant les en-tetes, les balises, ou la syntaxe du langage de programmation.

8.1 MarkdownHeaderTextSplitter

Decoupe un document Markdown en se basant sur les niveaux de titres (#, ##, ###). Chaque section devient un chunk avec les titres parents dans les metadonnees.

from langchain_text_splitters import MarkdownHeaderTextSplitter markdown_text = """# Introduction a LangChain LangChain est un framework pour les applications LLM. ## Document Loaders Les Document Loaders chargent des fichiers dans des objets Document. Chaque page devient un Document distinct. ## Text Splitters Les Text Splitters decoupent les documents en chunks. ### RecursiveCharacterTextSplitter Le splitter le plus recommande pour un usage general. 8.3 CodeTextSplitter (RecursiveCharacterTextSplitter.from_language)

Pour le code source, LangChain fournit des splitters qui comprennent la syntaxe des langages de programmation. Ils evitent de couper au milieu d'une fonction ou d'une classe.

9. Gestion des Metadonnees

Les metadonnees sont un element crucial du pipeline RAG. Elles permettent non seulement de tracer l'origine de chaque chunk, mais aussi de filtrer les resultats de recherche et d'enrichir les reponses generees par le LLM.

9.1 Ajouter des metadonnees personnalisees

from langchain_community.document_loaders import TextLoader from langchain_text_splitters import RecursiveCharacterTextSplitter from datetime import datetime # 1. Enrichir les metadonnees AVANT le split for doc in docs: doc.metadata["categorie"] = "ressources_humaines" doc.metadata["departement"] = "RH" doc.metadata["date_ingestion"] = datetime.now().isoformat() doc.metadata["version"] = "2.1" doc.metadata["confidentialite"] = "interne" # 3. Decouper - les metadonnees sont propagees a TOUS les chunks splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50) chunks = splitter.split_documents(docs) print(f"Nombre de chunks : {len(chunks)}") print(f"\nMetadonnees du chunk 0 :") for key, value in chunks[0].metadata.items(): print(f" {key} : {value}")

9.2 Ajouter des metadonnees apres le split (par chunk)

Les metadonnees doivent etre des types simples (str, int, float, bool) pour etre compatibles avec la plupart des vector stores. Evitez les objets complexes, les listes ou les dictionnaires imbriques dans les metadonnees. FAISS, Chroma et Pinecone ont chacun leurs propres contraintes.

10. Strategies de Chunking

Le choix de la strategie de chunking depend de votre cas d'usage, du type de documents, et de la precision souhaitee. Voici les recommandations basees sur l'experience pratique et les bonnes pratiques de la communaute.

10.1 Tailles recommandees par cas d'usage

Cas d'usage chunk_size chunk_overlap Justification
FAQ / Questions courtes 200-300 30-50 Reponses precises et concises
Documentation technique 500-800 50-100 Contexte suffisant pour comprendre
Articles / Blog 800-1200 100-200 Paragraphes complets et coherents
Code source 1000-2000 100-200 Fonctions/classes entieres
Documents juridiques 500-1000 100-150 Clauses et articles complets
Conversations / Chat 300-500 50-80 Echanges individuels

10.2 La regle du chevauchement

Regle pratique pour le chunk_overlap

En general, le chunk_overlap devrait representer environ 10% a 20% du chunk_size. Par exemple :

  • chunk_size=500 -> chunk_overlap=50 a 100
  • chunk_size=1000 -> chunk_overlap=100 a 200
  • chunk_size=200 -> chunk_overlap=20 a 40

Un overlap trop grand cree de la redondance inutile et augmente le nombre de chunks. Un overlap trop petit risque de perdre le contexte aux frontieres.

10.3 Chunking hierarchique

Le chunking hierarchique consiste a combiner plusieurs niveaux de decoupage : d'abord par structure (chapitres, sections), puis par taille. C'est la strategie la plus avancee.

from langchain_text_splitters import ( MarkdownHeaderTextSplitter, RecursiveCharacterTextSplitter ) markdown_doc = """# Chapitre 1 : Les Fondamentaux ## 1.1 Introduction Paragraphe tres long sur l'introduction qui depasse la taille maximale... (imaginez 2000 caracteres ici) ## 1.2 Concepts Cles Un autre paragraphe tres long sur les concepts cles... (imaginez 1500 caracteres ici) # Chapitre 2 : Mise en Pratique ## 2.1 Installation Instructions detaillees d'installation tres longues... (imaginez 1800 caracteres ici)""" # Etape 1 : Decouper par en-tetes Markdown headers_to_split = [ ("#", "chapitre"), ("##", "section"), ] md_splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split) md_chunks = md_splitter.split_text(markdown_doc) # Etape 2 : Re-decouper les sections trop longues text_splitter = RecursiveCharacterTextSplitter( chunk_size=500, chunk_overlap=50, ) final_chunks = text_splitter.split_documents(md_chunks) print(f"Apres Markdown split : {len(md_chunks)} sections") print(f"Apres re-split : {len(final_chunks)} chunks finaux") for chunk in final_chunks: print(f" [{chunk.metadata}] {chunk.page_content[:50]}...")

Comparaison des Splitters

Splitter Strategie Recommande pour Qualite
RecursiveCharacterTextSplitter Recursif (paragraphes, phrases, mots) Usage general - LE splitter par defaut Excellente
CharacterTextSplitter Un seul separateur Texte a structure previsible Correcte
TokenTextSplitter Compte les tokens Respect strict des limites de tokens Bonne

Points cles a retenir

  1. Choisissez le loader adapte au format de vos documents.
  2. Le RecursiveCharacterTextSplitter est le choix par defaut pour 90% des cas.
  3. Commencez avec chunk_size=800 et chunk_overlap=100, puis ajustez.
  4. L'overlap doit representer 10% a 20% du chunk_size.
  5. Enrichissez les metadonnees avant le split : elles sont propagees aux chunks.
  6. Pour le Markdown/HTML, combinez les splitters specialises avec le RecursiveCharacter.
  7. Utilisez lazy_load() pour les gros documents afin de preserver la memoire.
  8. Testez toujours vos parametres sur un echantillon de vos vrais documents.
