Retrievers - Retrouver l'Information Pertinente

Cours 7 - LangChain : Les interfaces de recherche documentaire au coeur du RAG

1. Qu'est-ce qu'un Retriever ?

Un Retriever est une interface LangChain qui, pour une requête textuelle non structurée donnée, retourne une liste de Documents pertinents. C'est le composant central de toute architecture RAG (Retrieval-Augmented Generation) : il fait le pont entre la question de l'utilisateur et la base de connaissances.

L'interface d'un Retriever est délibérément simple : elle expose une unique méthode principale, invoke(query) (anciennement get_relevant_documents(query)), qui accepte une chaîne de caractères et retourne une liste d'objets Document.

Retriever vs VectorStore : quelle différence ?

Il est essentiel de comprendre la différence entre ces deux concepts :

  • VectorStore : un système de stockage qui stocke des documents sous forme de vecteurs (embeddings) et permet des recherches par similarité. Exemples : Chroma, FAISS, Pinecone.
  • Retriever : une interface générique qui retourne des documents pour une requête. Un Retriever n'a pas besoin de stocker les documents lui-même. Un VectorStore peut être wrappé dans un Retriever via la méthode .as_retriever().

Autrement dit, tout VectorStore peut devenir un Retriever, mais un Retriever peut aussi être un algorithme BM25, une API externe, un filtre sur des métadonnées, ou une combinaison de plusieurs sources.

L'interface Runnable

Depuis LangChain v0.2+, les Retrievers implémentent l'interface Runnable, ce qui signifie qu'on peut les utiliser avec invoke(), ainvoke() (asynchrone), batch(), et les composer dans des chaînes LCEL.

# Interface de base d'un Retriever from langchain_core.retrievers import BaseRetriever from langchain_core.documents import Document from typing import List class MonRetriever(BaseRetriever): """Un retriever personnalisé minimaliste.""" def _get_relevant_documents(self, query: str) -> List[Document]: """Logique de recherche personnalisée.""" # Implanter ici la logique de recherche return [ Document(page_content="Contenu pertinent...", metadata={"source": "ma_source"}) ] # Utilisation retriever = MonRetriever() docs = retriever.invoke("ma question") print(docs)
[Document(page_content='Contenu pertinent...', metadata={'source': 'ma_source'})]
Dans la grande majorité des cas, vous n'aurez pas à créer un Retriever de zéro. LangChain fournit de nombreux retrievers prêts à l'emploi. Le plus courant est le VectorStoreRetriever, que nous allons étudier en détail.

2. VectorStoreRetriever

Le VectorStoreRetriever est le retriever le plus utilisé dans LangChain. Il enveloppe un VectorStore et expose l'interface standard du Retriever. On le crée via la méthode vectorstore.as_retriever().

2.1 Création basique avec as_retriever()

# Préparation : créer un VectorStore avec Chroma from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings from langchain_core.documents import Document # Documents d'exemple documents = [ Document(page_content="Le RAG combine recherche et génération pour des réponses précises.", metadata={"source": "cours_rag", "annee": 2024}), Document(page_content="Les embeddings transforment du texte en vecteurs numériques.", metadata={"source": "cours_embeddings", "annee": 2024}), Document(page_content="LangChain est un framework pour construire des applications LLM.", metadata={"source": "cours_langchain", "annee": 2023}), Document(page_content="FAISS est une bibliothèque de recherche vectorielle rapide.", metadata={"source": "cours_vectordb", "annee": 2023}), Document(page_content="Les transformers utilisent l'attention pour traiter les séquences.", metadata={"source": "cours_ml", "annee": 2022}), Document(page_content="La similarité cosinus mesure la proximité entre deux vecteurs.", metadata={"source": "cours_maths", "annee": 2022}), ] # Créer le VectorStore vectorstore = Chroma.from_documents(documents, OpenAIEmbeddings()) # Transformer en Retriever retriever = vectorstore.as_retriever() # Utiliser le Retriever resultats = retriever.invoke("Qu'est-ce que le RAG ?") for doc in resultats: print(f"[{doc.metadata['source']}] {doc.page_content[:60]}...")

2.2 Le paramètre search_type

Le paramètre search_type définit l'algorithme de recherche utilisé par le retriever. LangChain supporte trois types :

search_type Description Quand l'utiliser
"similarity" Recherche par similarité cosinus standard. Retourne les k documents les plus proches. Cas général, par défaut
"mmr" Maximal Marginal Relevance : optimise à la fois la pertinence ET la diversité des résultats. Quand vous voulez éviter la redondance
"similarity_score_threshold" Comme "similarity", mais filtre les documents dont le score est inférieur au seuil. Quand la qualité minimale est critique

2.3 Recherche par similarité (par défaut)

# search_type="similarity" (par défaut) retriever_sim = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 3} # Retourner les 3 documents les plus similaires ) docs = retriever_sim.invoke("Comment fonctionne la recherche vectorielle ?") print(f"Nombre de documents retournés : {len(docs)}") for i, doc in enumerate(docs, 1): print(f"\nDoc {i} [{doc.metadata['source']}] :") print(f" {doc.page_content}")
Nombre de documents retournés : 3 Doc 1 [cours_vectordb] : FAISS est une bibliothèque de recherche vectorielle rapide. Doc 2 [cours_embeddings] : Les embeddings transforment du texte en vecteurs numériques. Doc 3 [cours_maths] : La similarité cosinus mesure la proximité entre deux vecteurs.

2.4 Recherche MMR (Maximal Marginal Relevance)

Principe du MMR

L'algorithme MMR résout un problème fréquent : quand les documents sont très similaires entre eux, la recherche par similarité standard peut retourner des résultats redondants. MMR sélectionne itérativement des documents qui sont à la fois pertinents par rapport à la requête ET différents des documents déjà sélectionnés.

Paramètres search_kwargs pour MMR

  • k (int, défaut 4) : Nombre de documents à retourner dans le résultat final.
  • fetch_k (int, défaut 20) : Nombre de documents candidats récupérés avant l'application de l'algorithme MMR.
  • lambda_mult (float, défaut 0.5) : Paramètre entre 0 et 1. 0 = diversité maximale, 1 = similarité maximale (aucune diversification).
# search_type="mmr" - Maximal Marginal Relevance retriever_mmr = vectorstore.as_retriever( search_type="mmr", search_kwargs={ "k": 3, # 3 documents finaux "fetch_k": 10, # Chercher parmi 10 candidats "lambda_mult": 0.25 # Favoriser la diversité } ) docs_mmr = retriever_mmr.invoke("Expliquez les vecteurs et embeddings") for i, doc in enumerate(docs_mmr, 1): print(f"Doc {i} [{doc.metadata['source']}] : {doc.page_content}")
Doc 1 [cours_embeddings] : Les embeddings transforment du texte en vecteurs numériques. Doc 2 [cours_rag] : Le RAG combine recherche et génération pour des réponses précises. Doc 3 [cours_ml] : Les transformers utilisent l'attention pour traiter les séquences.
Remarquez que MMR a sélectionné des documents plus diversifiés : au lieu de retourner trois documents parlant tous de vecteurs, il a inclus des documents sur le RAG et les transformers, offrant une vue plus large et plus riche.

2.5 Recherche avec seuil de score

# search_type="similarity_score_threshold" retriever_seuil = vectorstore.as_retriever( search_type="similarity_score_threshold", search_kwargs={ "score_threshold": 0.75, # Score minimum de similarité (entre 0 et 1) "k": 5 # Nombre maximum de documents } ) # Question très spécifique : seuls les documents très pertinents passent docs_seuil = retriever_seuil.invoke("Qu'est-ce que FAISS ?") print(f"Documents au-dessus du seuil 0.75 : {len(docs_seuil)}") for doc in docs_seuil: print(f" - {doc.page_content}") print() # Question vague : moins de documents passent le seuil docs_vague = retriever_seuil.invoke("Parlez-moi de la cuisine italienne") print(f"Documents au-dessus du seuil 0.75 : {len(docs_vague)}")
Documents au-dessus du seuil 0.75 : 2 - FAISS est une bibliothèque de recherche vectorielle rapide. - Les embeddings transforment du texte en vecteurs numériques. Documents au-dessus du seuil 0.75 : 0
Le seuil de score est particulièrement utile en production : plutôt que de toujours retourner k documents (dont certains pourraient être hors sujet), le seuil garantit un niveau minimum de pertinence. Si aucun document ne dépasse le seuil, la liste est vide, ce qui permet de gérer proprement les cas où la base ne contient pas d'information pertinente.

2.6 Filtrage par métadonnées

# Combiner recherche sémantique + filtre sur les métadonnées retriever_filtre = vectorstore.as_retriever( search_type="similarity", search_kwargs={ "k": 3, "filter": {"annee": 2024} # Uniquement les documents de 2024 } ) docs_2024 = retriever_filtre.invoke("technologie IA") for doc in docs_2024: print(f"[{doc.metadata['annee']}] {doc.page_content}")

3. MultiQueryRetriever

Le MultiQueryRetriever utilise un LLM pour générer automatiquement plusieurs variantes de la requête initiale de l'utilisateur, puis exécute chaque variante indépendamment et fusionne les résultats. Cette technique améliore le recall (rappel) en capturant des documents qui auraient été manqués par une seule formulation de la requête.

Pourquoi plusieurs requêtes ?

La recherche par embeddings dépend fortement de la formulation de la question. Par exemple :

  • "Qu'est-ce que le RAG ?" pourrait manquer un document qui parle de "retrieval augmented generation" sans abréviation.
  • "Avantages du RAG" captera des documents différents de "Comment fonctionne le RAG ?"

En générant 3-5 variantes, on maximise les chances de trouver tous les documents pertinents.

from langchain.retrievers.multi_query import MultiQueryRetriever from langchain_openai import ChatOpenAI import logging # Activer le logging pour voir les requêtes générées logging.basicConfig() logging.getLogger("langchain.retrievers.multi_query").setLevel(logging.INFO) # Créer le LLM llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Créer le retriever de base base_retriever = vectorstore.as_retriever(search_kwargs={"k": 3}) # Créer le MultiQueryRetriever multi_retriever = MultiQueryRetriever.from_llm( retriever=base_retriever, llm=llm ) # Lancer la recherche question = "Quels sont les avantages du RAG ?" docs = multi_retriever.invoke(question) print(f"Documents trouvés : {len(docs)}") for doc in docs: print(f" - {doc.page_content[:80]}...")

Pourquoi utiliser le Retrieval-Augmented Generation dans les applications IA ? 3. En quoi le RAG améliore-t-il les réponses des modèles de langage ? Le MultiQueryRetriever a généré 3 variantes de la question, chacune capturant un angle différent. Les résultats sont automatiquement dédupliqués et fusionnés, donnant 5 documents uniques au lieu de 3.

3.1 Personnaliser le prompt de génération

from langchain_core.prompts import PromptTemplate from langchain_core.output_parsers import LineListOutputParser # Prompt personnalisé pour générer les variantes en français prompt_template = PromptTemplate( input_variables=["question"], template="""Vous êtes un assistant spécialisé en IA. Le MultiQueryRetriever fait un appel LLM supplémentaire à chaque invocation pour générer les variantes. Cela ajoute de la latence et du coût. Utilisez un modèle rapide et peu coûteux comme gpt-4o-mini pour cette étape.

4. ContextualCompressionRetriever

Le ContextualCompressionRetriever résout un problème classique du RAG : les documents retournés contiennent souvent beaucoup d'informations non pertinentes par rapport à la question posée. Ce retriever compresse (réduit) les documents récupérés pour ne garder que les passages directement pertinents.

Le problème

Imaginons qu'un chunk de 500 mots est retourné par le retriever, mais que seule une phrase de 20 mots répond vraiment à la question. Envoyer les 500 mots au LLM :

  • Augmente le coût (plus de tokens en entrée)
  • Dilue l'information pertinente
  • Peut confondre le LLM avec des informations contradictoires

Le ContextualCompressionRetriever résout cela en extrayant ou filtrant les passages pertinents avant de les envoyer au LLM.

4.1 Avec LLMChainExtractor

Le LLMChainExtractor utilise un LLM pour extraire uniquement les passages pertinents de chaque document :

from langchain.retrievers import ContextualCompressionRetriever from langchain.retrievers.document_compressors import LLMChainExtractor from langchain_openai import ChatOpenAI # LLM pour la compression llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # Créer le compresseur compressor = LLMChainExtractor.from_llm(llm) # Retriever de base base_retriever = vectorstore.as_retriever(search_kwargs={"k": 4}) # Retriever avec compression compression_retriever = ContextualCompressionRetriever( base_compressor=compressor, base_retriever=base_retriever ) # Comparer les résultats AVANT et APRÈS compression question = "Qu'est-ce qu'un embedding ?" # Sans compression docs_bruts = base_retriever.invoke(question) print("=== SANS compression ===") for doc in docs_bruts: print(f" [{len(doc.page_content)} cars] {doc.page_content[:80]}...") print() # Avec compression docs_compresses = compression_retriever.invoke(question) print("=== AVEC compression ===") for doc in docs_compresses: print(f" [{len(doc.page_content)} cars] {doc.page_content}")

Le compresseur a éliminé les documents sur FAISS et le RAG qui ne répondaient pas directement à "Qu'est-ce qu'un embedding ?". Seuls les 2 documents véritablement pertinents sont conservés.

4.2 Avec EmbeddingsFilter

L'EmbeddingsFilter est une alternative plus rapide et moins coûteuse que le LLMChainExtractor. Il utilise les embeddings pour filtrer les documents dont la similarité est inférieure à un seuil :

from langchain.retrievers.document_compressors import EmbeddingsFilter from langchain_openai import OpenAIEmbeddings # Filtre basé sur les embeddings embeddings = OpenAIEmbeddings() embeddings_filter = EmbeddingsFilter( embeddings=embeddings, similarity_threshold=0.76 # Score minimum de similarité ) # Retriever avec filtre embeddings compression_retriever_emb = ContextualCompressionRetriever( base_compressor=embeddings_filter, base_retriever=base_retriever ) docs_filtres = compression_retriever_emb.invoke("Qu'est-ce qu'un embedding ?") print(f"Documents après filtre : {len(docs_filtres)}") for doc in docs_filtres: print(f" - {doc.page_content}")
Documents après filtre : 2 - Les embeddings transforment du texte en vecteurs numériques. - La similarité cosinus mesure la proximité entre deux vecteurs.

from langchain.retrievers.document_compressors import DocumentCompressorPipeline from langchain_community.document_transformers import EmbeddingsRedundantFilter # Combiner plusieurs étapes de compression redundant_filter = EmbeddingsRedundantFilter(embeddings=embeddings) pipeline_compressor = DocumentCompressorPipeline( transformers=[ redundant_filter, # 1. Supprimer les doublons embeddings_filter, # 2. En production, préférez EmbeddingsFilter à LLMChainExtractor : il est beaucoup plus rapide (pas d'appel LLM) et moins coûteux. Réservez le LLMChainExtractor aux cas où vous avez besoin d'une extraction fine de passages à l'intérieur d'un même document.

5. SelfQueryRetriever

Le SelfQueryRetriever est un retriever intelligent qui utilise un LLM pour analyser la requête de l'utilisateur et en extraire automatiquement :
1. Une requête sémantique (pour la recherche vectorielle)
2. Des filtres structurés sur les métadonnées (année, source, catégorie, etc.)

Exemple concret

Un utilisateur demande : "Documents sur le machine learning écrits après 2023"

Le SelfQueryRetriever décompose cette requête en :

  • Recherche sémantique : "machine learning"
  • Filtre métadonnées : annee > 2023

Sans ce retriever, le VectorStore standard ne comprend pas "après 2023" comme un filtre - il l'inclurait dans la recherche sémantique, ce qui donnerait des résultats imprécis.

from langchain.retrievers.self_query.base import SelfQueryRetriever from langchain.chains.query_constructor.schema import AttributeInfo from langchain_openai import ChatOpenAI # Définir les champs de métadonnées filtrables metadata_field_info = [ AttributeInfo( name="source", description="Le nom du cours d'origine du document", type="string", ), AttributeInfo( name="annee", description="L'année de publication du document", type="integer", ), ] # Description du contenu de la collection document_content_description = "Notes de cours sur l'intelligence artificielle et le machine learning" # Créer le SelfQueryRetriever llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) self_query_retriever = SelfQueryRetriever.from_llm( llm=llm, vectorstore=vectorstore, document_contents=document_content_description, metadata_field_info=metadata_field_info, verbose=True ) # Test 1 : Requête avec filtre implicite sur l'année print("=== Test 1 : Filtre par année ===") docs = self_query_retriever.invoke("Documents sur le machine learning écrits après 2023") for doc in docs: print(f" [{doc.metadata['annee']}] {doc.page_content}") print() # Test 2 : Requête avec filtre sur la source print("=== Test 2 : Filtre par source ===") docs2 = self_query_retriever.invoke("Informations du cours sur les embeddings") for doc in docs2: print(f" [{doc.metadata['source']}] {doc.page_content}")
=== Test 1 : Filtre par année === Requête générée : query='machine learning' filter=Comparison(comparator=<Comparator.GT>, attribute='annee', value=2023) [2024] Le RAG combine recherche et génération pour des réponses précises. [2024] Les embeddings transforment du texte en vecteurs numériques. === Test 2 : Filtre par source === Requête générée : query='embeddings' filter=Comparison(comparator=<Comparator.EQ>, attribute='source', value='cours_embeddings') [cours_embeddings] Les embeddings transforment du texte en vecteurs numériques.
Le SelfQueryRetriever est extrêmement puissant pour les bases documentaires avec des métadonnées riches (dates, auteurs, catégories, types). Il permet aux utilisateurs d'exprimer des filtres en langage naturel sans connaître la structure des métadonnées.
Le SelfQueryRetriever nécessite un VectorStore qui supporte le filtrage par métadonnées. Chroma, Pinecone, Weaviate et Qdrant le supportent. Vérifiez la compatibilité de votre VectorStore avant d'utiliser ce retriever.

6. ParentDocumentRetriever

Le ParentDocumentRetriever résout un dilemme fondamental du chunking : les petits chunks sont meilleurs pour la recherche (plus précis), mais les grands chunks sont meilleurs pour le contexte (plus complets). Ce retriever stocke les petits chunks dans le VectorStore pour la recherche, mais retourne les documents parents (plus grands) pour la génération.
from langchain.retrievers import ParentDocumentRetriever from langchain.storage import InMemoryStore from langchain_text_splitters import RecursiveCharacterTextSplitter from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings # Documents longs (simulés) long_docs = [ Document( page_content="""Le Retrieval-Augmented Generation (RAG) est une technique qui combine la recherche d'information avec la génération de texte. Le processus se décompose en plusieurs étapes : d'abord, la requête de l'utilisateur est utilisée pour chercher des documents pertinents dans une base de connaissances. Ensuite, ces documents sont passés en contexte au LLM qui génère une réponse fondée sur ces informations. L'avantage principal du RAG est qu'il permet au modèle d'accéder à des informations à jour sans nécessiter un réentraînement coûteux.""", metadata={"source": "guide_rag.pdf"} ), ] # Splitter pour les petits chunks (recherche) child_splitter = RecursiveCharacterTextSplitter(chunk_size=200, chunk_overlap=20) # Splitter pour les parents (optionnel, sinon le document entier est le parent) parent_splitter = RecursiveCharacterTextSplitter(chunk_size=800, chunk_overlap=50) # Stockages vectorstore_parent = Chroma( collection_name="child_chunks", embedding_function=OpenAIEmbeddings() ) docstore = InMemoryStore() # Stocke les documents parents # Créer le ParentDocumentRetriever parent_retriever = ParentDocumentRetriever( vectorstore=vectorstore_parent, docstore=docstore, child_splitter=child_splitter, parent_splitter=parent_splitter, # Optionnel ) # Ajouter les documents parent_retriever.add_documents(long_docs) # Vérifier ce qui est stocké print(f"Chunks dans le VectorStore : {len(list(vectorstore_parent.get()['ids']))}") print(f"Parents dans le DocStore : {len(list(docstore.yield_keys()))}") # Rechercher - retourne le document parent complet docs = parent_retriever.invoke("Qu'est-ce que le RAG ?") print(f"\nDocument parent retourné ({len(docs[0].page_content)} caractères) :") print(docs[0].page_content[:200] + "...")

7. EnsembleRetriever

L'EnsembleRetriever combine les résultats de plusieurs retrievers en utilisant l'algorithme Reciprocal Rank Fusion (RRF). Il permet de fusionner des approches de recherche complémentaires, typiquement la recherche sémantique (vectorielle) et la recherche par mots-clés (BM25), pour obtenir le meilleur des deux mondes.

Reciprocal Rank Fusion (RRF)

L'algorithme RRF fusionne les classements de plusieurs retrievers. Pour chaque document, son score RRF est calculé comme :

score(d) = ∑ (1 / (k + rang(d, retriever_i)) * poids_i)

Où k est une constante (généralement 60), et rang(d, retriever_i) est le rang du document dans la liste du retriever i. Les poids permettent de favoriser un retriever par rapport à un autre.

from langchain.retrievers import EnsembleRetriever from langchain_community.retrievers import BM25Retriever from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings # Documents d'exemple textes = [ "Le RAG combine recherche et génération pour des réponses précises.", "Les embeddings transforment du texte en vecteurs numériques.", "FAISS est une bibliothèque de recherche vectorielle créée par Meta.", "LangChain est un framework Python pour les applications LLM.", "Les transformers sont basés sur le mécanisme d'attention.", "La tokenisation est la première étape du traitement du langage.", ] # Retriever 1 : BM25 (mots-clés) bm25_retriever = BM25Retriever.from_texts(textes) bm25_retriever.k = 3 # Retriever 2 : Vectoriel (sémantique) vs = Chroma.from_texts(textes, OpenAIEmbeddings()) vector_retriever = vs.as_retriever(search_kwargs={"k": 3}) # EnsembleRetriever avec poids ensemble_retriever = EnsembleRetriever( retrievers=[bm25_retriever, vector_retriever], weights=[0.4, 0.6] # 40% BM25, 60% vectoriel ) # Recherche hybride question = "Comment fonctionne FAISS pour la recherche vectorielle ?" print("=== Résultats BM25 seul ===") for doc in bm25_retriever.invoke(question): print(f" - {doc.page_content}") print("\n=== Résultats vectoriel seul ===") for doc in vector_retriever.invoke(question): print(f" - {doc.page_content}") print("\n=== Résultats Ensemble (hybride) ===") for doc in ensemble_retriever.invoke(question): print(f" - {doc.page_content}")

L'EnsembleRetriever a fusionné les résultats des deux approches : FAISS arrive en tête (trouvé par les deux retrievers), et des résultats complémentaires de chaque approche sont inclus.

8. BM25Retriever

BM25 (Best Matching 25, aussi appelé Okapi BM25) est un algorithme de recherche par mots-clés (sparse retrieval). Contrairement à la recherche vectorielle qui compare des embeddings sémantiques, BM25 se base sur la fréquence des termes (TF-IDF amélioré) pour classer les documents.

Recherche sparse vs dense

Caractéristique BM25 (Sparse) Vectoriel (Dense)

9.2 Intégration dans une chaîne RAG (LCEL)

from langchain_core.prompts import ChatPromptTemplate from langchain_core.runnables import RunnablePassthrough from langchain_core.output_parsers import StrOutputParser from langchain_openai import ChatOpenAI # N'importe quel retriever peut être utilisé ici retriever = ensemble_retriever # ou tout autre retriever # Prompt RAG template = """Répondez à la question en vous basant sur le contexte suivant : Contexte : {context} Question : {question} Réponse :""" prompt = ChatPromptTemplate.from_template(template) # Fonction pour formater les documents def format_docs(docs): return "\n\n".join(doc.page_content for doc in docs) # Chaîne RAG complète llm = ChatOpenAI(model="gpt-4o-mini") chain = ( {"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | llm | StrOutputParser() ) # Exécuter reponse = chain.invoke("Comment fonctionne la recherche vectorielle ?") print(reponse)
La recherche vectorielle fonctionne en transformant du texte en vecteurs numériques grâce aux embeddings. Les documents sont stockés sous forme de vecteurs dans des bibliothèques spécialisées comme FAISS (créée par Meta). Lorsqu'une requête est reçue, elle est également transformée en vecteur, et la similarité cosinus est utilisée pour mesurer la proximité entre le vecteur de la requête et ceux des documents.

10. Résumé - Arbre de décision

Quel Retriever choisir ?

Suivez cet arbre de décision pour choisir le bon retriever :

Résumé des points clés

  • Un Retriever est une interface générique qui retourne des documents pour une requête donnée.
  • Le VectorStoreRetriever (via as_retriever()) est le point de départ pour tout projet RAG.
  • Utilisez search_type="mmr" pour diversifier les résultats et éviter la redondance.
  • Le MultiQueryRetriever améliore le recall en générant plusieurs variantes de la requête.
  • Le ContextualCompressionRetriever réduit le bruit en ne gardant que l'information pertinente.
  • Le SelfQueryRetriever transforme le langage naturel en filtres structurés sur les métadonnées.
  • Le ParentDocumentRetriever offre le meilleur des deux mondes : précision de recherche + richesse de contexte.
  • L'EnsembleRetriever combine plusieurs approches pour une recherche hybride optimale.
  • En production, la recherche hybride BM25 + vectorielle est la meilleure pratique reconnue.
  • Tous les retrievers sont compatibles avec LCEL et peuvent être intégrés dans des chaînes RAG.
Maîtriser les différents types de retrievers est essentiel pour construire des systèmes RAG performants. Le choix du bon retriever peut faire la différence entre un système qui donne des réponses médiocres et un système qui impressionne par la pertinence de ses réponses.
