TD Pratique - Memory avec LangChain

LangChain v0.3 -- Systemes de memoire conversationnelle -- ConversationBufferMemory, WindowMemory, SummaryMemory, TokenBufferMemory

1. Recapitulatif

La Memory (memoire) dans LangChain est un mecanisme qui permet a un chatbot ou une chain de se souvenir des echanges precedents. Sans memoire, chaque appel au LLM est independant : le modele ne sait pas ce qui a ete dit avant. La memoire stocke l'historique des messages et le reinjecte automatiquement dans le prompt a chaque nouvel echange.

Analogie

Imaginez une conversation avec un ami. Si votre ami oubliait tout apres chaque phrase, la discussion serait impossible. La Memory joue le role du cerveau : elle retient ce qui a ete dit et utilise ce contexte pour formuler des reponses coherentes.

  • Buffer = se souvenir de TOUT mot pour mot (enregistrement audio complet)
  • Window = se souvenir des N derniers echanges (memoire a court terme)
  • Summary = se souvenir d'un resume (notes de cours)
  • TokenBuffer = se souvenir de ce qui tient dans un budget (carnet a pages limitees)

En LangChain v0.3, l'approche moderne est d'utiliser RunnableWithMessageHistory avec un ChatMessageHistory. Les anciennes classes comme ConversationChain sont toujours fonctionnelles mais considerees comme legacy. Ce TD couvre les deux approches.

2. Principe de Fonctionnement

La memoire dans LangChain suit un cycle precis a chaque echange conversationnel :

Etape 1 -- Chargement : Recuperer l'historique des messages precedents depuis le stockage.
Etape 2 -- Injection : Inserer l'historique dans le prompt via un MessagesPlaceholder.
Etape 3 -- Generation : Le LLM recoit le prompt complet (systeme + historique + question) et genere une reponse.
Etape 4 -- Sauvegarde : La question ET la reponse sont sauvegardees dans la memoire.
Le point crucial : le LLM lui-meme n'a aucune memoire intrinseque. C'est LangChain qui stocke l'historique et le reinjecte a chaque appel. Le LLM recoit a chaque fois un prompt complet contenant tout le contexte. Plus la conversation est longue, plus le prompt grandit et plus l'appel coute cher.

3. Les Donnees : Types de Memoire

LangChain propose quatre types principaux de memoire, chacun avec une strategie differente pour gerer l'historique conversationnel.

3.1 ConversationBufferMemory

ConversationBufferMemory stocke l'integralite de la conversation mot pour mot. Rien n'est oublie, rien n'est resume. Chaque message est conserve dans une liste ordonnee.
from langchain.memory import ConversationBufferMemory memory = ConversationBufferMemory(memory_key="history", return_messages=True) # Simuler des echanges memory.save_context( {"input": "Bonjour, je m'appelle Alice"}, {"output": "Bonjour Alice ! Comment puis-je vous aider ?"} ) memory.save_context( {"input": "Quel temps fait-il a Paris ?"}, {"output": "Je n'ai pas acces a la meteo en temps reel."} ) print(memory.load_memory_variables({}))
{'history': [ HumanMessage(content="Bonjour, je m'appelle Alice"), AIMessage(content="Bonjour Alice ! Comment puis-je vous aider ?"), HumanMessage(content="Quel temps fait-il a Paris ?"), AIMessage(content="Je n'ai pas acces a la meteo en temps reel.") ]}
Tous les messages sont conserves. La memoire grandit lineairement avec le nombre d'echanges. Ideal pour les conversations courtes (moins de 20 echanges).

3.2 ConversationBufferWindowMemory

ConversationBufferWindowMemory conserve uniquement les k derniers echanges. Les echanges plus anciens sont automatiquement supprimes (fenetre glissante).
from langchain.memory import ConversationBufferWindowMemory memory_window = ConversationBufferWindowMemory(k=2, memory_key="history", return_messages=True) memory_window.save_context({"input": "Je m'appelle Alice"}, {"output": "Bonjour Alice !"}) memory_window.save_context({"input": "J'habite a Lyon"}, {"output": "Lyon est une belle ville !"}) memory_window.save_context({"input": "Quel est mon prenom ?"}, {"output": "Desole, je n'ai pas cette information."}) # Le premier echange (prenom) a ete supprime ! print(memory_window.load_memory_variables({}))
{'history': [ HumanMessage(content="J'habite a Lyon"), AIMessage(content="Lyon est une belle ville !"), HumanMessage(content="Quel est mon prenom ?"), AIMessage(content="Desole, je n'ai pas cette information.") ]}
Avec k=2, le premier echange (ou Alice donne son prenom) est supprime. Le LLM ne peut plus se souvenir du prenom. C'est le compromis de la fenetre glissante.

3.3 ConversationSummaryMemory

ConversationSummaryMemory utilise un LLM pour generer un resume de la conversation. Le resume est mis a jour apres chaque echange, conservant les informations cles en moins de tokens.
from langchain.memory import ConversationSummaryMemory from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) memory_summary = ConversationSummaryMemory(llm=llm, memory_key="history", return_messages=True) memory_summary.save_context( {"input": "Je m'appelle Alice et j'ai 25 ans"}, {"output": "Bonjour Alice ! Vous avez 25 ans, c'est note !"} ) memory_summary.save_context( {"input": "Je suis etudiante en informatique a Lyon"}, {"output": "L'informatique a Lyon, excellent choix !"} ) memory_summary.save_context( {"input": "J'adore le machine learning et Python"}, {"output": "Python est le langage ideal pour le ML !"} ) print(memory_summary.load_memory_variables({}))
{'history': [ SystemMessage(content="L'utilisatrice s'appelle Alice, 25 ans. Elle est etudiante en informatique a Lyon. Elle s'interesse au machine learning et programme en Python.") ]}
Au lieu de 6 messages, la memoire contient un seul resume. Toutes les informations cles sont preservees en beaucoup moins de tokens. Le cout : un appel LLM supplementaire apres chaque echange.

3.4 ConversationTokenBufferMemory

ConversationTokenBufferMemory conserve les messages recents tant que le nombre total de tokens ne depasse pas max_token_limit. Les anciens sont supprimes en premier.
from langchain.memory import ConversationTokenBufferMemory from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini") memory_token = ConversationTokenBufferMemory(llm=llm, max_token_limit=100, memory_key="history", return_messages=True) memory_token.save_context( {"input": "Bonjour, je m'appelle Alice et je suis etudiante"}, {"output": "Bonjour Alice ! Bienvenue !"} ) memory_token.save_context( {"input": "Peux-tu m'expliquer les reseaux de neurones ?"}, {"output": "Les reseaux de neurones sont des modeles inspires du cerveau humain."} ) print(memory_token.load_memory_variables({}))
{'history': [ HumanMessage(content="Peux-tu m'expliquer les reseaux de neurones ?"), AIMessage(content="Les reseaux de neurones sont des modeles inspires du cerveau humain.") ]}
La ConversationTokenBufferMemory est souvent le meilleur choix en production : elle garantit un budget de tokens fixe, contrairement a la WindowMemory qui raisonne en nombre d'echanges (un echange court et un long comptent pareil).

4. Les Parametres

4.1 memory_key

memory_key (str) -- defaut : "history"

Cle sous laquelle l'historique est stocke. Doit correspondre exactement au variable_name du MessagesPlaceholder dans le prompt.

from langchain.memory import ConversationBufferMemory from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder # La cle doit correspondre entre memory et prompt memory = ConversationBufferMemory(memory_key="chat_history", return_messages=True) prompt = ChatPromptTemplate.from_messages([ ("system", "Tu es un assistant utile."), MessagesPlaceholder(variable_name="chat_history"), # meme cle ! ("human", "{input}") ]) memory.save_context({"input": "Salut !"}, {"output": "Bonjour !"}) print("Cle :", list(memory.load_memory_variables({}).keys()))
Cle : ['chat_history']

4.2 return_messages

return_messages (bool) -- defaut : False

Si True, retourne une liste d'objets Message (pour les ChatModels). Si False, retourne une chaine de texte brut.

# return_messages=False (defaut) : chaine texte mem_str = ConversationBufferMemory(memory_key="history") mem_str.save_context({"input": "Salut"}, {"output": "Bonjour !"}) print(repr(mem_str.load_memory_variables({}))) # return_messages=True : liste de Messages mem_msg = ConversationBufferMemory(memory_key="history", return_messages=True) mem_msg.save_context({"input": "Salut"}, {"output": "Bonjour !"}) print(mem_msg.load_memory_variables({}))
{'history': 'Human: Salut\nAI: Bonjour !'} {'history': [HumanMessage(content='Salut'), AIMessage(content='Bonjour !')]}
Avec les ChatModels (GPT-4, Claude), utilisez toujours return_messages=True pour que l'historique soit compatible avec MessagesPlaceholder.

4.3 k (WindowMemory)

k (int) -- defaut : 5

Nombre d'echanges (paires question-reponse) a conserver. Avec k=3, la memoire conserve au maximum 6 messages (3 paires).

for k_val in [1, 3]: mem = ConversationBufferWindowMemory(k=k_val, return_messages=True) mem.save_context({"input": "Msg 1"}, {"output": "Rep 1"}) mem.save_context({"input": "Msg 2"}, {"output": "Rep 2"}) mem.save_context({"input": "Msg 3"}, {"output": "Rep 3"}) msgs = mem.load_memory_variables({})["history"] print(f"k={k_val} : {len(msgs)} messages")
k=1 : 2 messages k=3 : 6 messages

4.4 max_token_limit (TokenBufferMemory)

max_token_limit (int) -- defaut : 2000

Nombre maximum de tokens en memoire. Les anciens messages sont supprimes quand la limite est atteinte. En production, une valeur typique est entre 500 et 4000 selon le modele.

4.5 human_prefix et ai_prefix

human_prefix / ai_prefix (str) -- defaut : "Human" / "AI"

Personnalise les prefixes dans le format texte (return_messages=False).

memory_custom = ConversationBufferMemory(human_prefix="Etudiant", ai_prefix="Professeur") memory_custom.save_context( {"input": "Qu'est-ce qu'un tenseur ?"}, {"output": "Un tenseur est une generalisation des matrices."} ) print(memory_custom.load_memory_variables({})["history"])
Etudiant: Qu'est-ce qu'un tenseur ? Professeur: Un tenseur est une generalisation des matrices.

5. Les Metriques

5.1 Consommation de Tokens

Nombre de tokens utilises pour l'historique a chaque appel. Impacte directement le cout et la latence.

import tiktoken def compter_tokens(messages, model="gpt-4o-mini"): encoding = tiktoken.encoding_for_model(model) return sum(len(encoding.encode(m.content)) for m in messages) # Comparer Buffer vs Window apres 5 echanges echanges = [ ("Bonjour, je suis Alice", "Bonjour Alice !"), ("J'etudie l'informatique", "C'est passionnant !"), ("J'aime Python", "Python est populaire en IA."), ("Qu'est-ce que le deep learning ?", "C'est une branche du ML."), ("Et le NLP ?", "Le NLP traite du langage naturel."), ] mem_buf = ConversationBufferMemory(return_messages=True) mem_win = ConversationBufferWindowMemory(k=2, return_messages=True) for i, o in echanges: mem_buf.save_context({"input": i}, {"output": o}) mem_win.save_context({"input": i}, {"output": o}) print(f"Buffer : {compter_tokens(mem_buf.load_memory_variables({})['history'])} tokens") print(f"Window : {compter_tokens(mem_win.load_memory_variables({})['history'])} tokens")
Buffer : 112 tokens Window : 38 tokens

5.2 Retention d'Information

Capacite a conserver les informations cles mentionnees plus tot. Mesurable en verifiant si des faits specifiques sont toujours accessibles apres N echanges.

5.3 Latence et Cout

Buffer/Window : latence negligeable. Summary : +1 appel LLM par echange (latence et cout supplementaires). TokenBuffer : negligeable. Plus l'historique est long, plus chaque appel coute cher en tokens d'entree.

6. Implementation Complete

6.1 Approche moderne : RunnableWithMessageHistory

RunnableWithMessageHistory est la methode recommandee en LangChain v0.3. Elle encapsule une chain LCEL et gere automatiquement le chargement et la sauvegarde de l'historique.
Etape 1 : Installer les dependances.
# pip install langchain langchain-openai langchain-community python-dotenv import os from dotenv import load_dotenv load_dotenv() # Charge OPENAI_API_KEY depuis .env
Etape 2 : Creer la chain LCEL (prompt + model + parser).
from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder from langchain_core.output_parsers import StrOutputParser prompt = ChatPromptTemplate.from_messages([ ("system", "Tu es un assistant pedagogique bienveillant. Reponds en francais."), MessagesPlaceholder(variable_name="history"), ("human", "{input}") ]) model = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) chain = prompt | model | StrOutputParser()
Etape 3 : Configurer les sessions et encapsuler la chain.
from langchain_core.runnables.history import RunnableWithMessageHistory from langchain_community.chat_message_histories import ChatMessageHistory store = {} # Stockage des historiques par session (en RAM) def get_session_history(session_id: str): if session_id not in store: store[session_id] = ChatMessageHistory() return store[session_id] chain_with_history = RunnableWithMessageHistory( chain, get_session_history, input_messages_key="input", history_messages_key="history" )
Etape 4 : Utiliser la chain avec memoire.
config = {"configurable": {"session_id": "etudiant_01"}} # Premier echange r1 = chain_with_history.invoke( {"input": "Bonjour ! Je m'appelle Alice et j'etudie le machine learning."}, config=config ) print("Bot :", r1) # Deuxieme echange -- le bot se souvient ! r2 = chain_with_history.invoke( {"input": "Quel est mon prenom et qu'est-ce que j'etudie ?"}, config=config ) print("Bot :", r2)
Bot : Bonjour Alice ! Le machine learning est fascinant. Comment puis-je t'aider ? Bot : Tu t'appelles Alice et tu etudies le machine learning !

6.2 Approche legacy : ConversationChain

from langchain.chains import ConversationChain from langchain.memory import ConversationBufferMemory llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) memory = ConversationBufferMemory(return_messages=True) conversation = ConversationChain(llm=llm, memory=memory, verbose=True) r1 = conversation.predict(input="Je m'appelle Bob, j'ai 30 ans.") print("Bot :", r1) r2 = conversation.predict(input="Quel est mon age ?") print("Bot :", r2)
Bot : Bonjour Bob ! Vous avez 30 ans, c'est note. Bot : Vous avez 30 ans, Bob !

6.3 Sessions multiples

# Chaque utilisateur a son propre session_id config_a = {"configurable": {"session_id": "user_alice"}} config_b = {"configurable": {"session_id": "user_bob"}} chain_with_history.invoke({"input": "Je suis Alice, j'aime Python."}, config=config_a) chain_with_history.invoke({"input": "Je suis Bob, je prefere JavaScript."}, config=config_b) r_a = chain_with_history.invoke({"input": "Quel langage j'aime ?"}, config=config_a) r_b = chain_with_history.invoke({"input": "Quel langage j'aime ?"}, config=config_b) print("Alice :", r_a) print("Bob :", r_b)
Alice : Vous aimez Python ! Bob : Vous preferez JavaScript !
L'isolation par session est fondamentale en production. En production reelle, remplacez le dictionnaire store par Redis (RedisChatMessageHistory) ou PostgreSQL.

7. Experimentation

Comparons les quatre types de memoire sur la meme conversation de 5 echanges.

from langchain.memory import ( ConversationBufferMemory, ConversationBufferWindowMemory, ConversationSummaryMemory, ConversationTokenBufferMemory ) llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) conversation = [ ("Je m'appelle Alice, j'ai 25 ans, j'habite a Lyon.", "Bonjour Alice !"), ("Je suis en master informatique.", "Excellent choix !"), ("Mon langage prefere est Python.", "Python est tres populaire."), ("J'ai un chat nomme Moustache.", "Quel joli nom !"), ("Je fais un stage chez Google.", "Impressionnant !"), ] memories = { "Buffer": ConversationBufferMemory(return_messages=True, memory_key="history"), "Window(k=2)": ConversationBufferWindowMemory(k=2, return_messages=True, memory_key="history"), "Summary": ConversationSummaryMemory(llm=llm, return_messages=True, memory_key="history"), "TokenBuf(100)": ConversationTokenBufferMemory(llm=llm, max_token_limit=100, return_messages=True, memory_key="history"), } for inp, out in conversation: for mem in memories.values(): mem.save_context({"input": inp}, {"output": out})

7.2 Test de retention

infos = [("prenom","Alice"), ("age","25"), ("ville","Lyon"), ("etudes","informatique"), ("langage","Python"), ("animal","Moustache"), ("stage","Google")] print(f"{'Info':<12} {'Buffer':<10} {'Window':<10} {'Summary':<10} {'TokenBuf':<10}") print("-" * 52) for info, mot in infos: res = [] for mem in memories.values(): h = mem.load_memory_variables({})["history"] contenu = " ".join([m.content for m in h]) res.append("oui" if mot in contenu else "NON") print(f"{info:<12} {res[0]:<10} {res[1]:<10} {res[2]:<10} {res[3]:<10}")
Info Buffer Window Summary TokenBuf ---------------------------------------------------- prenom oui NON oui NON age oui NON oui NON ville oui NON oui NON etudes oui NON oui NON langage oui NON oui NON animal oui oui oui oui stage oui oui oui oui

Analyse des resultats

  • Buffer : Retention parfaite (7/7), mais le plus gros volume de tokens.
  • Window (k=2) : Seuls les 2 derniers echanges retenus (2/7). Infos anciennes perdues.
  • Summary : Excellent rapport retention/tokens (7/7). Le resume conserve tout.
  • TokenBuffer (100) : Similaire a Window (2/7). Anciens messages supprimes.
Le choix depend du cas d'usage. Support client (nom, commande) : SummaryMemory. Chatbot decontracte (contexte recent) : WindowMemory. Prototypage rapide : BufferMemory.

8. Erreurs Courantes

Erreur 1 : Oublier de vider la memoire entre les tests

L'historique des tests precedents s'accumule et pollue les nouvelles conversations. Solution : appeler memory.clear() avant de recommencer.
memory = ConversationBufferMemory(return_messages=True) memory.save_context({"input": "test 1"}, {"output": "reponse 1"}) memory.save_context({"input": "test 2"}, {"output": "reponse 2"}) print(len(memory.load_memory_variables({})["history"])) # 4 memory.clear() # Nettoyer ! memory.save_context({"input": "test 2"}, {"output": "reponse 2"}) print(len(memory.load_memory_variables({})["history"])) # 2

Erreur 2 : memory_key differente du MessagesPlaceholder

Si les cles ne correspondent pas, le LLM ne recoit jamais l'historique. Solution : verifier que memory_key == variable_name.
# ERREUR memory = ConversationBufferMemory(memory_key="history", return_messages=True) MessagesPlaceholder(variable_name="chat_history") # "history" != "chat_history" ! # SOLUTION : meme cle partout memory = ConversationBufferMemory(memory_key="chat_history", return_messages=True) MessagesPlaceholder(variable_name="chat_history") # OK !

Erreur 3 : Oublier return_messages=True

Sans return_messages=True, la memoire retourne du texte brut. Or MessagesPlaceholder attend une liste de Messages. Solution : toujours activer return_messages=True pour les ChatModels.

Erreur 4 : Depassement de la fenetre de contexte

Avec BufferMemory, l'historique croit indefiniment. Apres 1000 echanges, le prompt depasse la limite du modele. Solution : utiliser TokenBufferMemory ou WindowMemory en production.

Erreur 5 : Oublier le session_id

RunnableWithMessageHistory necessite un session_id dans le config. Sans lui, LangChain leve une erreur.
# ERREUR : config manquant # chain_with_history.invoke({"input": "Bonjour"}) # SOLUTION config = {"configurable": {"session_id": "user_42"}} chain_with_history.invoke({"input": "Bonjour"}, config=config)

Erreur 6 : Partager une session entre utilisateurs

Si tous les utilisateurs partagent le meme session_id, leurs historiques se melangent. L'utilisateur B voit les messages de A. Solution : un session_id unique (UUID) par utilisateur authentifie.
# ERREUR : meme session pour tous config_shared = {"configurable": {"session_id": "shared"}} # Fuite de donnees ! # SOLUTION : sessions isolees config_alice = {"configurable": {"session_id": "user_alice_abc123"}} config_bob = {"configurable": {"session_id": "user_bob_def456"}}

Bonnes pratiques

  1. Toujours utiliser return_messages=True avec les ChatModels
  2. Verifier que memory_key correspond au MessagesPlaceholder
  3. Utiliser WindowMemory ou TokenBufferMemory en production
  4. Isoler les sessions avec un session_id unique par utilisateur
  5. Appeler memory.clear() entre les tests
  6. Preferer RunnableWithMessageHistory a ConversationChain
  7. Prevoir une marge dans la fenetre de contexte (prompt + reponse)
  8. Utiliser un backend persistant (Redis, PostgreSQL) en production

Arbre de decision

Votre conversation depasse-t-elle 20 echanges ?

  • Non : ConversationBufferMemory (simple et complet)
  • Oui : Les infos du debut sont-elles critiques ?
    • Non : ConversationBufferWindowMemory (k=5 a 10)
    • Oui : Le cout d'appels LLM extra est acceptable ?
      • Oui : ConversationSummaryMemory
      • Non : ConversationTokenBufferMemory (2000-4000 tokens)
