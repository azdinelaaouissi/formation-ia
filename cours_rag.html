C'est quoi le RAG ?

Retrieval Augmented Generation - La definition

Definition simple

RAG (Retrieval Augmented Generation) est une technique qui permet a un modele d'IA (comme ChatGPT ou Claude) d'aller chercher des informations dans vos documents avant de repondre a une question.


Au lieu de compter uniquement sur ce que l'IA a appris pendant son entrainement, on lui donne acces a une base de connaissances externe (vos fichiers, votre base de donnees, vos PDFs, etc.).

En 3 mots : Chercher + Lire + Repondre

Le RAG fonctionne en 3 etapes simples :

  1. Retrieval (Recherche) : L'IA cherche les informations pertinentes dans vos documents
  2. Augmented (Augmentation) : L'IA enrichit sa reponse avec ces informations trouvees
  3. Generation (Generation) : L'IA genere une reponse naturelle et precise
A retenir

Le RAG ne modifie pas l'IA elle-meme. Il lui donne simplement acces a des documents supplementaires au moment de la question. C'est comme donner un livre a quelqu'un avant de lui poser une question.

Pourquoi le RAG ?

Les problemes que le RAG resout

Les limites d'une IA classique (sans RAG)

Les modeles de langage (LLM) comme ChatGPT ont des limitations importantes :

  • Connaissances figees : L'IA ne connait que ce qu'elle a appris pendant l'entrainement (date de coupure)
  • Pas vos donnees : Elle ne connait pas vos documents internes, vos emails, votre base de donnees
  • Hallucinations : Quand elle ne sait pas, elle invente des reponses qui semblent vraies
  • Pas de sources : Impossible de verifier d'ou vient l'information

Sans RAG

  • Reponses generiques
  • Pas acces a vos donnees
  • Risque d'hallucinations
  • Informations potentiellement obsoletes
  • Impossible de citer les sources

Avec RAG

  • Reponses precises et contextuelles
  • Acces a vos documents
  • Reponses basees sur des faits
  • Toujours a jour (vos documents)
  • Sources citees et verifiables
Cas d'usage populaires

Chatbot pour votre entreprise, assistant medical, FAQ intelligente, recherche dans des documents juridiques, assistant code base sur votre documentation, support client automatise, etc.

Les composants du RAG

Les pieces du puzzle

Les 5 composants essentiels

Un systeme RAG est compose de 5 elements principaux qui travaillent ensemble :

1. Les Documents Sources

Vos donnees brutes : PDFs, fichiers texte, pages web, bases de donnees, emails, etc.


Ce sont les informations que vous voulez rendre accessibles a l'IA.

2. Le Modele d'Embedding

Un modele special qui transforme le texte en vecteurs numeriques (des listes de nombres).


Exemples : text-embedding-ada-002 (OpenAI), all-MiniLM-L6-v2 (gratuit)

3. La Base Vectorielle

Une base de donnees speciale qui stocke les vecteurs et permet de chercher par similarite semantique.


Exemples : ChromaDB, Pinecone, FAISS, Weaviate

4. Le Retriever

Le composant qui cherche et recupere les documents les plus pertinents par rapport a la question.


Il utilise la base vectorielle pour trouver les "voisins les plus proches".

5. Le LLM (Large Language Model)

Le cerveau final : le modele d'IA qui recoit la question + les documents trouves et genere la reponse finale en langage naturel.

Exemples : GPT-4, Claude, Llama, Mistral, etc.

5

Les Embeddings

Le concept le plus important a comprendre

C'est quoi un Embedding ?

Un embedding, c'est la transformation d'un texte en une liste de nombres (un vecteur). Ces nombres representent le sens du texte dans un espace mathematique.


Pourquoi ? Parce qu'un ordinateur ne comprend pas les mots. Il comprend les nombres. Les embeddings permettent de comparer le sens de deux textes en comparant leurs nombres.

Exemple concret

exemple_embedding.py 
# Le texte "Le chat dort" est transforme en vecteur :
"Le chat dort"  →  [0.23, -0.45, 0.89, 0.12, -0.67, ...]  # 1536 nombres

# "Le felin sommeille" (meme sens) donne un vecteur SIMILAIRE :
"Le felin sommeille"  →  [0.22, -0.44, 0.87, 0.13, -0.65, ...]

# "La voiture roule" (sens different) donne un vecteur DIFFERENT :
"La voiture roule"  →  [-0.56, 0.78, 0.11, -0.34, 0.92, ...]

La similarite cosinus

Pour mesurer si deux textes sont proches en sens, on calcule la similarite cosinus entre leurs vecteurs :

  • 1.0 = textes identiques en sens
  • 0.8 - 0.9 = textes tres similaires
  • 0.5 = textes vaguement lies
  • 0.0 = aucun rapport
similarite.py 
from sklearn.metrics.pairwise import cosine_similarity

# Similarite entre "chat dort" et "felin sommeille"
similarity = cosine_similarity(vec1, vec2)
# Resultat : 0.92 (tres similaire !)

# Similarite entre "chat dort" et "voiture roule"
similarity = cosine_similarity(vec1, vec3)
# Resultat : 0.15 (pas du tout similaire)
Point cle

Les embeddings capturent le SENS, pas juste les mots. "Docteur" et "Medecin" auront des vecteurs proches, meme si les mots sont differents. C'est ce qui rend la recherche semantique bien plus puissante que la recherche par mots-cles classique.

La Base de Donnees Vectorielle

Ou sont stockes les embeddings

C'est quoi une base vectorielle ?

C'est une base de donnees specialisee pour stocker et rechercher des vecteurs (embeddings). Contrairement a une base SQL classique qui cherche par correspondance exacte, elle cherche par similarite.

Analogie : La bibliotheque intelligente

Imaginez une bibliotheque ou les livres ne sont pas ranges par ordre alphabetique, mais par theme et sens. Les livres sur la cuisine sont tous proches les uns des autres. Un livre sur la "patisserie francaise" sera a cote de "desserts italiens" car ils sont thematiquement proches.


Quand vous cherchez "comment faire un gateau ?", le bibliothecaire (le retriever) ne cherche pas le mot exact "gateau" - il va dans la zone cuisine/patisserie et vous ramene les livres les plus pertinents.

Comparaison des bases vectorielles populaires

Outil Type Difficulte Prix Ideal pour
ChromaDB Local Facile Gratuit Debutants, prototypage
FAISS Local Moyen Gratuit Performance, gros volumes
Pinecone Cloud Facile Freemium Production, scalabilite
Weaviate Les deux Moyen Gratuit Fonctionnalites avancees
Qdrant Les deux Moyen Gratuit Rust, performance
Recommandation debutant

Commence avec ChromaDB. C'est le plus simple a installer et utiliser. On l'utilisera dans notre projet pratique. pip install chromadb et c'est parti !

Le Chunking (Decoupage)

Comment decouper vos documents

Pourquoi decouper ?

Les documents sont souvent trop longs pour etre traites en une seule fois. On les decoupe en morceaux (chunks) plus petits pour :

  • Que chaque morceau soit assez specifique pour etre pertinent
  • Respecter les limites de tokens du modele d'embedding
  • Avoir une recherche plus precise (trouver le bon paragraphe, pas tout le livre)

Les strategies de chunking

1. Chunking par taille fixe (le plus simple)

On decoupe le texte tous les N caracteres (ex: tous les 500 caracteres).

  • Avantage : Simple a implementer
  • Inconvenient : Peut couper une phrase en plein milieu
chunking_fixe.py 
def chunk_by_size(text, chunk_size=500, overlap=50):
    chunks = []
    for i in range(0, len(text), chunk_size - overlap):
        chunks.append(text[i:i + chunk_size])
    return chunks
2. Chunking par separateur (recommande)

On decoupe a des points naturels : fins de paragraphes, titres, etc.

  • Avantage : Respecte la structure du texte
  • Inconvenient : Taille variable des chunks
chunking_separateur.py 
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ". ", " "]
)

chunks = splitter.split_text(document_text)
3. Chunking semantique (avance)

On decoupe en fonction du sens du texte. Chaque chunk contient une idee complete.

  • Avantage : Meilleure qualite de retrieval
  • Inconvenient : Plus complexe, plus lent

L'overlap (chevauchement)

L'overlap est une zone de chevauchement entre deux chunks consecutifs. Il permet d'eviter de perdre du contexte aux frontieres.

overlap_illustration.txt 
Chunk 1 : [============================================]
Chunk 2 :                                   [============================================]
                                   ^^^^^^^^^^
                                   = Overlap  

Sans overlap : on risque de couper une info entre 2 chunks
Avec overlap : l'info est presente dans les 2 chunks
Bonnes pratiques

chunk_size : 200-1000 caracteres (500 est un bon defaut).
overlap : 10-20% de la taille du chunk.
Trop petit = perte de contexte. Trop grand = bruit et lenteur.

Le Workflow Complet du RAG

Vue d'ensemble de A a Z

Les 2 grandes phases

Un systeme RAG fonctionne en 2 phases distinctes :

Phase 1 : Indexation (Offline)

Preparation des documents. Se fait une seule fois (ou quand les documents changent).

  1. Charger les documents
  2. Decouper en chunks
  3. Generer les embeddings
  4. Stocker dans la base vectorielle

Phase 2 : Requete (Online)

Repondre aux questions. Se fait a chaque question de l'utilisateur.

  1. Recevoir la question
  2. Convertir en embedding
  3. Chercher les chunks similaires
  4. Envoyer question + chunks au LLM
  5. Generer la reponse
9

Phase d'Indexation (Detail)

Preparer vos documents pour le RAG

Etape par etape avec du code

Voici le code complet pour indexer des documents :

indexation.py 
# ===== ETAPE 1 : Installer les dependances =====
# pip install langchain chromadb sentence-transformers

# ===== ETAPE 2 : Charger les documents =====
from langchain.document_loaders import TextLoader, PyPDFLoader

# Charger un fichier texte
loader = TextLoader("mon_document.txt")
documents = loader.load()

# Ou charger un PDF
loader = PyPDFLoader("mon_fichier.pdf")
documents = loader.load()

# ===== ETAPE 3 : Decouper en chunks =====
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # Taille max d'un chunk
    chunk_overlap=50,     # Chevauchement entre chunks
    separators=["\n\n", "\n", ". ", " "]
)
chunks = splitter.split_documents(documents)
print(f"Nombre de chunks : {len(chunks)}")

# ===== ETAPE 4 : Creer les embeddings et stocker =====
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma

# Modele d'embedding gratuit (tourne en local)
embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2"
)

# Creer la base vectorielle ChromaDB
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./ma_base_vectorielle"
)

print("Indexation terminee !")
Astuce

L'indexation se fait une seule fois. Ensuite, la base vectorielle est sauvegardee sur disque (persist_directory). Au prochain demarrage, il suffit de la recharger sans tout re-indexer.

Phase de Recherche (Retrieval)

Trouver les bons documents

Comment ca marche ?

Quand l'utilisateur pose une question :

  1. La question est transformee en embedding (vecteur)
  2. Ce vecteur est compare aux vecteurs stockes dans la base
  3. Les K chunks les plus similaires sont retournes (Top-K)

Code de la recherche

recherche.py 
# Charger la base existante
vectorstore = Chroma(
    persist_directory="./ma_base_vectorielle",
    embedding_function=embeddings
)

# Creer un retriever (chercheur)
retriever = vectorstore.as_retriever(
    search_type="similarity",    # Recherche par similarite
    search_kwargs={"k": 4}       # Retourner les 4 meilleurs chunks
)

# Rechercher
question = "Quelle est la posologie du paracetamol ?"
docs = retriever.get_relevant_documents(question)

# Afficher les resultats
for i, doc in enumerate(docs):
    print(f"--- Chunk {i+1} ---")
    print(doc.page_content)
    print()

Le parametre K (Top-K)

Le K determine combien de chunks sont retournes :

Valeur de K Avantage Inconvenient
K = 2-3 Rapide, moins de bruit Peut manquer des infos
K = 4-5 (recommande) Bon equilibre -
K = 8-10 Plus de contexte Plus de bruit, plus lent, plus cher

Phase de Generation

Construire le prompt et generer la reponse

Le Prompt Template

C'est ici que la magie opere. On construit un prompt qui contient :

  • Les instructions pour l'IA
  • Le contexte (les chunks trouves)
  • La question de l'utilisateur
prompt_template.py 
prompt_template = """
Tu es un assistant specialise. Reponds a la question en te basant
UNIQUEMENT sur le contexte fourni ci-dessous. Si tu ne trouves pas
la reponse dans le contexte, dis "Je n'ai pas trouve cette information
dans les documents disponibles."

CONTEXTE :
{context}

QUESTION : {question}

REPONSE :
"""

Code complet de la generation

generation.py 
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

# 1. Le modele LLM
llm = ChatOpenAI(
    model_name="gpt-4",
    temperature=0    # 0 = reponses deterministes
)

# 2. Le template du prompt
prompt = PromptTemplate(
    template=prompt_template,
    input_variables=["context", "question"]
)

# 3. La chaine RAG complete
rag_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",              # Methode simple
    retriever=retriever,
    chain_type_kwargs={"prompt": prompt},
    return_source_documents=True     # Retourner les sources
)

# 4. Poser une question
result = rag_chain({"query": "Quelle est la posologie du paracetamol ?"})

print("Reponse :", result["result"])
print("Sources :", [doc.metadata for doc in result["source_documents"]])
Les chain types

stuff : Met tous les chunks dans le prompt (simple, fonctionne pour la plupart des cas).
map_reduce : Traite chaque chunk separement puis combine (pour de gros volumes).
refine : Ameliore la reponse iterativement avec chaque chunk.
Pour debuter, stuff suffit amplement.

Patterns Avances

Pour aller plus loin

Les differentes variantes du RAG

1. Naive RAG (le basique)

C'est le RAG simple que nous avons vu : Indexation → Retrieval → Generation.

Quand l'utiliser : Projets simples, prototypage, documents de taille moderee.

2. Advanced RAG (pre et post-retrieval)

Ajoute des etapes d'amelioration :

  • Pre-retrieval : Reformulation de la question (query rewriting), expansion de la requete
  • Post-retrieval : Re-ranking des resultats, filtrage, compression du contexte

Quand l'utiliser : Quand le Naive RAG ne donne pas assez de precision.

3. Modular RAG (composants modulaires)

Architecture flexible ou chaque composant est independant et interchangeable :

  • Routing : diriger la question vers la bonne source
  • Multi-source : chercher dans plusieurs bases
  • Feedback loop : ameliorer les resultats automatiquement

Quand l'utiliser : Systemes complexes en production.

4. RAG avec Re-Ranking

Apres la recherche initiale, un modele de re-ranking reordonne les resultats pour mettre les plus pertinents en premier.

reranking.py 
# Exemple avec un re-ranker
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

# Recuperer 20 chunks, puis garder les 5 meilleurs
initial_docs = retriever.get_relevant_documents(query, k=20)
scores = reranker.predict([
    (query, doc.page_content) for doc in initial_docs
])
# Trier par score et garder le top 5
5. Hybrid Search (Vectoriel + Mots-cles)

Combine la recherche semantique (vecteurs) avec la recherche par mots-cles (BM25) pour de meilleurs resultats.

  • Recherche vectorielle : trouve les documents par sens
  • BM25 : trouve les documents par mots-cles exacts
  • Fusion : combine les deux resultats

Erreurs Courantes

Les pieges a eviter

Chunk trop gros

Si vos chunks sont trop grands (ex: 5000 caracteres), la recherche sera imprecise. L'IA recevra trop de bruit.


Solution : Gardez des chunks entre 200-1000 caracteres.

Chunk trop petit

Si vos chunks sont trop petits (ex: 50 caracteres), ils manqueront de contexte. L'IA ne pourra pas comprendre.


Solution : Utilisez l'overlap et assurez-vous que chaque chunk est comprehensible seul.

Pas de metadata

Stocker les chunks sans metadata (source, page, date) rend impossible le tracage des sources.


Solution : Ajoutez toujours source, page, date dans les metadata.

Mauvais prompt

Un prompt qui ne dit pas "base-toi sur le contexte" laisse l'IA inventer des reponses.


Solution : Instruisez clairement l'IA de se baser sur le contexte fourni.

K trop eleve

Recuperer 20 chunks alors que 4 suffisent ajoute du bruit et augmente les couts.


Solution : Commencez avec K=4, ajustez si necessaire.

Embedding mal choisi

Utiliser un embedding anglais pour des documents francais reduit la qualite.


Solution : Choisissez un modele multilingue ou adapte a votre langue.

RAG vs Fine-tuning

Quelle approche choisir ?

Comparaison detaillee

Critere RAG Fine-tuning
Principe Donne des documents a l'IA au moment de la question Re-entraine l'IA sur vos donnees
Mise a jour Facile - ajouter des documents Difficile - re-entrainer le modele
Cout Faible - base vectorielle Eleve - GPU, temps
Precision Haute - cite les sources Variable - peut halluciner
Transparence Oui - on voit les sources Non - boite noire
Complexite Simple Complexe
Ideal pour Questions-reponses sur documents, FAQ, support Changer le style/comportement de l'IA
En resume

RAG = "Donner un livre a l'IA" (rapide, pas cher, mis a jour facilement).
Fine-tuning = "Renvoyer l'IA a l'ecole" (long, couteux, mais modifie l'IA en profondeur).

Dans 90% des cas, le RAG est la meilleure solution. On peut aussi combiner les deux.
