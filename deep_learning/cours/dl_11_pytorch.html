1. Pourquoi PyTorch ?

PyTorch est devenu le framework de Deep Learning le plus utilise en recherche et de plus en plus adopte en production. Developpe par Facebook AI Research (FAIR), il se distingue par son approche intuitive et pythonique.

1.1 Historique

2002
Torch est cree en Lua par Ronan Collobert et al. au IDIAP. Framework precurseur mais langage peu populaire.
2015
TensorFlow est publie par Google Brain. Graphe statique (define-and-run). Domine pendant plusieurs annees.
2016
PyTorch 0.1 est lance par FAIR. Approche dynamique (define-by-run) qui seduit immediatement les chercheurs.
2018
PyTorch 1.0 fusionne avec Caffe2. Ajout de TorchScript pour la production. Adoption massive en recherche.
2022
PyTorch 2.0 avec torch.compile, acceleration significative, graphes dynamiques compiles.
2023
PyTorch rejoint la Linux Foundation. Il devient le standard de facto pour la recherche ET la production.

1.2 PyTorch vs TensorFlow

PyTorch

  • Graphe dynamique (eager mode)
  • Debug avec pdb/breakpoints
  • API pythonique et intuitive
  • Dominant en recherche (>80% des papiers)
  • TorchScript pour la production
  • Ecosysteme : HuggingFace, Lightning

TensorFlow

  • Graphe statique (graph mode, eager depuis v2)
  • TensorBoard pour la visualisation
  • Keras comme API haut niveau
  • Fort en production/mobile (TF Lite, TF.js)
  • SavedModel pour le deploiement
  • Ecosysteme : TFX, TF Serving
Le graphe dynamique (define-by-run) signifie que le graphe de calcul est construit a la volee lors de l'execution du code Python. Chaque appel forward cree un nouveau graphe. Cela permet d'utiliser des conditions, des boucles et du debugging Python natif.
En 2025, le debat PyTorch vs TensorFlow est largement resolu : PyTorch domine la recherche et gagne du terrain en production. Je recommande PyTorch comme premier framework a apprendre. TensorFlow reste pertinent pour les deploiements mobiles et web.

2. Tensors - La brique fondamentale

Les tensors sont la structure de donnees centrale de PyTorch. Ce sont des tableaux multidimensionnels similaires aux arrays NumPy, mais avec le support GPU et la differentiation automatique.

Toutes les operations entre tensors doivent se faire sur le meme device. Tenter de multiplier un tensor CPU avec un tensor GPU produit une erreur. Utilisez .to(device) de maniere coherente.

3. Autograd - Differentiation automatique

L'autograd est le moteur de differentiation automatique de PyTorch. Il calcule automatiquement les gradients necessaires a la retropropagation, eliminant le besoin de calculs manuels de derivees.

Autograd est un systeme de differentiation automatique qui enregistre toutes les operations effectuees sur des tensors avec requires_grad=True, construisant un graphe de calcul dynamique. Lors de l'appel a .backward(), il parcourt ce graphe en sens inverse pour calculer tous les gradients via la regle de la chaine.

3.1 Bases de l'autograd

# --- requires_grad active le tracking --- x = torch.tensor([2.0, 3.0], requires_grad=True) y = torch.tensor([4.0, 5.0], requires_grad=True) # Operations trackes automatiquement z = x * y + x ** 2 print(f"z = {z}") print(f"z.grad_fn = {z.grad_fn}") # Montre l'operation qui a cree z # Calculer les gradients loss = z.sum() loss.backward() # dz/dx = y + 2x print(f"x.grad = {x.grad}") # [4+4, 5+6] = [8, 11] print(f"y.grad = {y.grad}") # [2, 3] = x
z = tensor([12., 24.], grad_fn=<AddBackward0>) z.grad_fn = <AddBackward0> x.grad = tensor([ 8., 11.]) y.grad = tensor([2., 3.])

3.2 Le graphe de calcul

Analogie : le fil d'Ariane

Imaginez que chaque operation laisse un fil invisible reliant le resultat a ses operandes. Quand on appelle .backward(), PyTorch remonte ces fils en sens inverse, calculant le gradient a chaque noeud. C'est exactement la regle de la chaine du calcul differentiel, automatisee.

3.3 Controle du gradient

# --- Desactiver le calcul de gradient --- # Methode 1 : torch.no_grad() (pour l'inference) with torch.no_grad(): pred = model(input_data) # Pas de graphe construit # Methode 2 : detach() (couper du graphe) z = x * y z_detached = z.detach() # Copie sans historique de gradient # Methode 3 : requires_grad_(False) x.requires_grad_(False) # In-place, stoppe le tracking # --- Remettre les gradients a zero --- x = torch.tensor([2.0], requires_grad=True) y = x ** 2 y.backward() print(f"Gradient apres 1er backward: {x.grad}") y = x ** 2 y.backward() print(f"Gradient ACCUMULE (probleme!): {x.grad}") # Solution : remettre a zero x.grad.zero_() y = x ** 2 y.backward() print(f"Gradient apres zero_grad: {x.grad}")
Gradient apres 1er backward: tensor([4.]) Gradient ACCUMULE (probleme!): tensor([8.]) Gradient apres zero_grad: tensor([4.])
Les gradients s'accumulent par defaut dans PyTorch ! Il faut imperativement appeler optimizer.zero_grad() ou tensor.grad.zero_() avant chaque nouveau calcul de gradient. Oublier cette etape est l'une des erreurs les plus frequentes des debutants.

4. nn.Module - Definir des modeles

Tous les modeles PyTorch heritent de nn.Module. Cette classe de base fournit une interface standard pour definir, entrainer et deployer des reseaux de neurones.

4.1 Structure de base

import torch.nn as nn class SimpleClassifier(nn.Module): def __init__(self, input_dim, hidden_dim, num_classes): super().__init__() # Toujours appeler super().__init__() ! # Definir les couches dans __init__ self.layer1 = nn.Linear(input_dim, hidden_dim) self.relu = nn.ReLU() self.dropout = nn.Dropout(0.3) self.layer2 = nn.Linear(hidden_dim, hidden_dim) self.layer3 = nn.Linear(hidden_dim, num_classes) def forward(self, x): """Definir le forward pass.""" x = self.relu(self.layer1(x)) x = self.dropout(x) x = self.relu(self.layer2(x)) x = self.layer3(x) # Pas de softmax ici (inclu dans CrossEntropyLoss) return x # Instanciation model = SimpleClassifier(input_dim=784, hidden_dim=256, num_classes=10) # Tester avec une entree aleatoire x = torch.randn(32, 784) # Batch de 32, 784 features output = model(x) print(f"Output shape: {output.shape}") print(f"Nombre de parametres: {sum(p.numel() for p in model.parameters())}")

4.3 CNN avec nn.Module

class CNN(nn.Module): def __init__(self, num_classes=10): super().__init__() self.features = nn.Sequential( nn.Conv2d(1, 32, kernel_size=3, padding=1), nn.BatchNorm2d(32), nn.ReLU(), nn.MaxPool2d(2), # 28x28 -> 14x14 nn.Conv2d(32, 64, kernel_size=3, padding=1), nn.BatchNorm2d(64), nn.ReLU(), nn.MaxPool2d(2), # 14x14 -> 7x7 ) self.classifier = nn.Sequential( nn.Flatten(), # [B, 64, 7, 7] -> [B, 3136] nn.Linear(64 * 7 * 7, 128), nn.ReLU(), nn.Dropout(0.5), nn.Linear(128, num_classes) ) def forward(self, x): x = self.features(x) x = self.classifier(x) return x model = CNN() x = torch.randn(8, 1, 28, 28) print(f"Output: {model(x).shape}")
Output: torch.Size([8, 10])
La convention PyTorch est de separer les features (couches convolutives) du classifier (couches denses). Cela facilite le transfer learning : on peut geler les features et ne re-entrainer que le classifier.

5. Datasets et DataLoaders

PyTorch fournit un systeme elegant pour charger et pre-traiter les donnees via les classes Dataset et DataLoader.

5.1 torch.utils.data.Dataset

Un Dataset est un objet qui represente un jeu de donnees. Il doit implementer deux methodes : __len__() qui retourne le nombre d'echantillons, et __getitem__(idx) qui retourne l'echantillon a l'index donne.
from torch.utils.data import Dataset, DataLoader class CustomDataset(Dataset): def __init__(self, data, labels, transform=None): self.data = data self.labels = labels self.transform = transform def __len__(self): return len(self.data) def __getitem__(self, idx): sample = self.data[idx] label = self.labels[idx] if self.transform: sample = self.transform(sample) return sample, label # Exemple avec des donnees synthetiques X = torch.randn(1000, 784) y = torch.randint(0, 10, (1000,)) dataset = CustomDataset(X, y) print(f"Taille du dataset: {len(dataset)}") sample, label = dataset[0] print(f"Shape d'un echantillon: {sample.shape}, Label: {label}")
Taille du dataset: 1000 Shape d'un echantillon: torch.Size([784]), Label: 7

6. Transforms et data augmentation

Les transforms permettent de pre-traiter les donnees et d'appliquer des augmentations qui ameliorent la generalisation du modele.

6.1 Transforms courants

from torchvision import transforms # Pipeline de transforms pour l'entrainement (avec augmentation) train_transform = transforms.Compose([ transforms.RandomResizedCrop(224, scale=(0.8, 1.0)), transforms.RandomHorizontalFlip(p=0.5), transforms.RandomRotation(degrees=15), transforms.ColorJitter( brightness=0.2, contrast=0.2, saturation=0.2, hue=0.1 ), transforms.RandomGrayscale(p=0.1), transforms.ToTensor(), # PIL Image -> Tensor [0, 1] transforms.Normalize( mean=[0.485, 0.456, 0.406], # ImageNet mean std=[0.229, 0.224, 0.225] # ImageNet std ) ]) # Pipeline pour la validation (PAS d'augmentation) val_transform = transforms.Compose([ transforms.Resize(256), transforms.CenterCrop(224), transforms.ToTensor(), transforms.Normalize( mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225] ) ])
L'augmentation de donnees ne doit etre appliquee que sur les donnees d'entrainement, jamais sur la validation ou le test. Le pipeline de validation ne contient que le redimensionnement, le crop et la normalisation.

6.2 Custom transform

class AddGaussianNoise: """Transform personnalise qui ajoute du bruit gaussien.""" def __init__(self, mean=0.0, std=0.1): self.mean = mean self.std = std def __call__(self, tensor): noise = torch.randn_like(tensor) * self.std + self.mean return tensor + noise # Utilisation dans un Compose transform = transforms.Compose([ transforms.ToTensor(), AddGaussianNoise(std=0.05), transforms.Normalize([0.5], [0.5]) ])

# --- Classification multi-classe avec CrossEntropyLoss --- criterion = nn.CrossEntropyLoss() # Le modele produit des logits BRUTS (pas de softmax!) logits = torch.randn(4, 10) # Batch de 4, 10 classes targets = torch.tensor([3, 7, 1, 9]) # Classes cibles (indices) loss = criterion(logits, targets) print(f"CrossEntropy Loss: {loss.item():.4f}") # --- Regression avec MSELoss --- mse = nn.MSELoss() predictions = torch.tensor([2.5, 3.2, 4.1]) ground_truth = torch.tensor([2.0, 3.0, 4.0]) loss_mse = mse(predictions, ground_truth) print(f"MSE Loss: {loss_mse.item():.4f}") # --- Classification binaire avec BCEWithLogitsLoss --- bce = nn.BCEWithLogitsLoss() logits_bin = torch.tensor([0.8, -0.5, 1.2, -0.3]) targets_bin = torch.tensor([1.0, 0.0, 1.0, 0.0]) loss_bce = bce(logits_bin, targets_bin) print(f"BCE Loss: {loss_bce.item():.4f}")
CrossEntropy Loss: 2.4876 MSE Loss: 0.1000 BCE Loss: 0.4613
L'erreur la plus frequente est d'appliquer un softmax avant CrossEntropyLoss. Cette loss integre deja le log_softmax en interne pour des raisons de stabilite numerique. Si vous ajoutez un softmax dans votre forward, vous obtiendrez un double softmax et de mauvais resultats.

8. Boucle d'entrainement complete

Voici la boucle d'entrainement standard en PyTorch, expliquee etape par etape. C'est le patron de conception le plus important a maitriser.

8.1 Les 5 etapes sacrees

1. optimizer.zero_grad() - Remettre les gradients a zero
2. output = model(input) - Forward pass
3. loss = criterion(output, target) - Calculer la perte
4. loss.backward() - Calculer les gradients (backpropagation)
5. optimizer.step() - Mettre a jour les parametres

8.2 Implementation complete avec validation et early stopping

import torch import torch.nn as nn from torch.utils.data import DataLoader, random_split from torchvision import datasets, transforms # ============================================ # CONFIGURATION # ============================================ device = torch.device("cuda" if torch.cuda.is_available() else "cpu") EPOCHS = 20 BATCH_SIZE = 128 LR = 1e-3 PATIENCE = 5 # Early stopping # ============================================ # DONNEES # ============================================ transform = transforms.Compose([ transforms.ToTensor(), transforms.Normalize((0.1307,), (0.3081,)) ]) full_train = datasets.MNIST("./data", train=True, download=True, transform=transform) test_dataset = datasets.MNIST("./data", train=False, transform=transform) # Split train/validation (90/10) train_size = int(0.9 * len(full_train)) val_size = len(full_train) - train_size train_dataset, val_dataset = random_split(full_train, [train_size, val_size]) train_loader = DataLoader(train_dataset, batch_size=BATCH_SIZE, shuffle=True) val_loader = DataLoader(val_dataset, batch_size=BATCH_SIZE, shuffle=False) test_loader = DataLoader(test_dataset, batch_size=BATCH_SIZE, shuffle=False) # ============================================ # MODELE # ============================================ model = CNN(num_classes=10).to(device) criterion = nn.CrossEntropyLoss() optimizer = torch.optim.Adam(model.parameters(), lr=LR) scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau( optimizer, mode="min", patience=3, factor=0.5 )

8.3 Les fonctions train et evaluate

def train_one_epoch(model, loader, criterion, optimizer, device): """Entraine le modele pendant une epoch.""" model.train() # Active dropout, batchnorm en mode train running_loss = 0.0 correct = 0 total = 0 for data, targets in loader: data, targets = data.to(device), targets.to(device) # Les 5 etapes sacrees optimizer.zero_grad() # 1. Reset gradients outputs = model(data) # 2. Forward pass loss = criterion(outputs, targets) # 3. Calculer la loss loss.backward() # 4. Backpropagation optimizer.step() # 5. Mise a jour des poids # Statistiques running_loss += loss.item() * data.size(0) _, predicted = outputs.max(1) total += targets.size(0) correct += predicted.eq(targets).sum().item() epoch_loss = running_loss / total epoch_acc = 100.0 * correct / total return epoch_loss, epoch_acc @torch.no_grad() def evaluate(model, loader, criterion, device): """Evalue le modele sur un dataset.""" model.eval() # Desactive dropout, batchnorm en mode eval running_loss = 0.0 correct = 0 total = 0 for data, targets in loader: data, targets = data.to(device), targets.to(device) outputs = model(data) loss = criterion(outputs, targets) running_loss += loss.item() * data.size(0) _, predicted = outputs.max(1) total += targets.size(0) correct += predicted.eq(targets).sum().item() epoch_loss = running_loss / total epoch_acc = 100.0 * correct / total return epoch_loss, epoch_acc

8.4 Boucle principale avec early stopping

# ============================================ # BOUCLE D'ENTRAINEMENT # ============================================ best_val_loss = float("inf") patience_counter = 0 for epoch in range(EPOCHS): # Entrainement train_loss, train_acc = train_one_epoch( model, train_loader, criterion, optimizer, device ) # Validation val_loss, val_acc = evaluate(model, val_loader, criterion, device) # Learning rate scheduling scheduler.step(val_loss) # Affichage current_lr = optimizer.param_groups[0]["lr"] print(f"Epoch {epoch+1:02d}/{EPOCHS} | " f"Train Loss: {train_loss:.4f} Acc: {train_acc:.2f}% | " f"Val Loss: {val_loss:.4f} Acc: {val_acc:.2f}% | " f"LR: {current_lr:.2e}") # Early stopping if val_loss < best_val_loss: best_val_loss = val_loss patience_counter = 0 # Sauvegarder le meilleur modele torch.save(model.state_dict(), "best_model.pth") print(" -> Meilleur modele sauvegarde!") else: patience_counter += 1 if patience_counter >= PATIENCE: print(f"Early stopping apres {epoch+1} epochs.") break # Test final avec le meilleur modele model.load_state_dict(torch.load("best_model.pth")) test_loss, test_acc = evaluate(model, test_loader, criterion, device) print(f"\nTest Final | Loss: {test_loss:.4f} | Accuracy: {test_acc:.2f}%")

La combinaison early stopping + ReduceLROnPlateau + sauvegarde du meilleur modele est un patron de conception robuste qui previent le surapprentissage et garantit que vous conservez le meilleur modele rencontre pendant l'entrainement.

9. Sauvegarder et charger des modeles

Savoir correctement sauvegarder et charger des modeles est essentiel pour la reproductibilite et le deploiement.

9.1 Sauvegarder le state_dict (recommande)

# --- Methode recommandee : sauvegarder le state_dict --- # Sauvegarde torch.save(model.state_dict(), "model_weights.pth") # Chargement model = CNN(num_classes=10) # Recreer l'architecture model.load_state_dict(torch.load("model_weights.pth")) model.eval() # Mettre en mode evaluation # --- Sauvegarder un checkpoint complet --- checkpoint = { "epoch": epoch, "model_state_dict": model.state_dict(), "optimizer_state_dict": optimizer.state_dict(), "scheduler_state_dict": scheduler.state_dict(), "best_val_loss": best_val_loss, "train_loss": train_loss, } torch.save(checkpoint, "checkpoint.pth") # Reprendre l'entrainement a partir du checkpoint checkpoint = torch.load("checkpoint.pth") model.load_state_dict(checkpoint["model_state_dict"]) optimizer.load_state_dict(checkpoint["optimizer_state_dict"]) scheduler.load_state_dict(checkpoint["scheduler_state_dict"]) start_epoch = checkpoint["epoch"] + 1 best_val_loss = checkpoint["best_val_loss"] print(f"Reprise a l'epoch {start_epoch}")

9.2 Sauvegarder le modele complet

# --- Methode alternative : sauvegarder le modele entier --- # Plus simple mais moins flexible (lie au code source) torch.save(model, "model_complet.pth") model = torch.load("model_complet.pth") model.eval()
La methode torch.save(model, ...) utilise pickle et sauvegarde une reference au code source de la classe. Si vous modifiez la classe ou deplacez le fichier, le chargement echouera. Privilegiez toujours state_dict() pour la production.

9.3 TorchScript pour l'export

# --- TorchScript : export independant de Python --- # Methode 1 : torch.jit.trace (pour les modeles sans branchement) model.eval() example_input = torch.randn(1, 1, 28, 28) traced_model = torch.jit.trace(model, example_input) traced_model.save("model_traced.pt") # Methode 2 : torch.jit.script (pour les modeles avec controle de flux) scripted_model = torch.jit.script(model) scripted_model.save("model_scripted.pt") # Chargement (meme sans le code Python original!) loaded = torch.jit.load("model_traced.pt") output = loaded(example_input) print(f"Output TorchScript: {output.shape}")

10. Bonnes pratiques

Ce chapitre regroupe les conseils essentiels pour un entrainement efficace et fiable.

10.1 Entrainement GPU efficace

# --- Mixed Precision Training (AMP) --- # Reduit la memoire et accelere l'entrainement de 2-3x from torch.cuda.amp import autocast, GradScaler scaler = GradScaler() for data, targets in train_loader: data, targets = data.to(device), targets.to(device) optimizer.zero_grad() # Forward pass en mixed precision with autocast(): outputs = model(data) loss = criterion(outputs, targets) # Backward pass avec scaling scaler.scale(loss).backward() scaler.step(optimizer) scaler.update()

10.2 Gradient Accumulation

Quand la memoire GPU est limitee

Si votre batch size ideal est 256 mais votre GPU ne supporte que 64, vous pouvez accumuler les gradients sur 4 mini-batches pour simuler un batch de 256.

# --- Gradient Accumulation --- ACCUMULATION_STEPS = 4 # Effectif batch_size = 64 * 4 = 256 for batch_idx, (data, targets) in enumerate(train_loader): data, targets = data.to(device), targets.to(device) outputs = model(data) loss = criterion(outputs, targets) / ACCUMULATION_STEPS # Normaliser! loss.backward() # Gradients s'accumulent if (batch_idx + 1) % ACCUMULATION_STEPS == 0: optimizer.step() # Mise a jour apres N batches optimizer.zero_grad() # Reset apres la mise a jour

10.3 Reproductibilite

import random import numpy as np def set_seed(seed=42): """Fixer toutes les graines pour la reproductibilite.""" random.seed(seed) np.random.seed(seed) torch.manual_seed(seed) torch.cuda.manual_seed_all(seed) # Pour un determinisme total (peut ralentir) torch.backends.cudnn.deterministic = True torch.backends.cudnn.benchmark = False set_seed(42)

10.4 Gradient Clipping

# Prevenir l'explosion de gradients (essentiel pour RNNs et Transformers) loss.backward() torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step()

10.5 Checklist de debugging

Quand le modele n'apprend pas :

  1. Verifier les shapes : print(x.shape) a chaque etape du forward
  2. Overfit un seul batch : le modele doit atteindre 100% accuracy sur 1 batch
  3. Verifier la loss initiale : pour 10 classes, la loss initiale doit etre ~ln(10) = 2.3
  4. Verifier les gradients : aucun parametre ne doit avoir grad = None ou grad = 0
  5. Verifier le learning rate : commencer avec 1e-3 (Adam) ou 1e-2 (SGD)
  6. Verifier la normalisation : les donnees doivent etre normalisees
  7. Verifier train/eval : model.train() pour l'entrainement, model.eval() pour l'evaluation
  8. Verifier zero_grad : les gradients doivent etre remis a zero a chaque iteration
# --- Overfit un seul batch (test de sante) --- single_batch = next(iter(train_loader)) data, targets = single_batch[0].to(device), single_batch[1].to(device) model.train() for i in range(100): optimizer.zero_grad() output = model(data) loss = criterion(output, targets) loss.backward() optimizer.step() if (i + 1) % 20 == 0: _, pred = output.max(1) acc = pred.eq(targets).float().mean().item() * 100 print(f"Step {i+1} | Loss: {loss.item():.4f} | Acc: {acc:.1f}%")

Si votre modele ne peut pas overfit un seul batch en 100 iterations, il y a un bug dans votre code (shape incorrecte, loss mal configuree, zero_grad manquant, etc.). C'est le premier test a faire systematiquement.

10.6 Resume des patterns essentiels

Les 10 regles d'or de PyTorch

  1. model.train() pendant l'entrainement, model.eval() et torch.no_grad() pendant l'evaluation
  2. optimizer.zero_grad() avant chaque forward pass
  3. Pas de softmax dans le modele si vous utilisez CrossEntropyLoss
  4. Toujours normaliser les donnees d'entree
  5. Utiliser pin_memory=True dans le DataLoader pour l'entrainement GPU
  6. Sauvegarder le state_dict, pas le modele entier
  7. Utiliser Mixed Precision pour accelerer et reduire la memoire
  8. Tester d'abord en overfittant un seul batch
  9. Fixer les graines aleatoires pour la reproductibilite
  10. Utiliser gradient clipping pour les architectures recurrentes et Transformers
