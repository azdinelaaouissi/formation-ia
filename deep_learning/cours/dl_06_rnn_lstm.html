1. Pourquoi les sequences ?

Jusqu'a present, nous avons traite des donnees statiques : une image, un vecteur de features. Mais une grande partie des donnees du monde reel sont sequentielles : elles ont un ordre temporel ou positionnel qui porte une information essentielle.

1.2 Limitations des MLP pour les sequences

Pourquoi ne pas simplement aplatir une sequence et la donner a un MLP classique ? Trois raisons fondamentales :

Probleme 1 : Taille fixe

Un MLP exige un vecteur d'entree de taille fixe. Or, les sequences ont des longueurs variables : une phrase peut avoir 5 mots ou 50 mots. Il faudrait padding et troncature, ce qui est sous-optimal.

Probleme 2 : Pas de partage de parametres

Un MLP traite chaque position d'entree avec des poids differents. Il ne peut pas apprendre qu'un pattern "je suis" a la meme signification au debut ou a la fin d'une phrase. Les poids ne sont pas partages dans le temps.

Probleme 3 : Pas de memoire

Un MLP traite chaque entree de maniere independante. Il ne conserve aucune "memoire" des elements precedents. Pour comprendre "il" dans "Jean mange une pomme. Il la trouve delicieuse", il faut se souvenir de "Jean" et "pomme".

Analogie : lire un livre

Un MLP, c'est comme si vous lisiez chaque page d'un livre independamment, sans vous souvenir des pages precedentes. Vous ne pourriez jamais comprendre l'intrigue. Un RNN, c'est comme un lecteur humain qui maintient en memoire le contexte au fur et a mesure de la lecture, accumulant les informations pour comprendre le sens global.

2. RNN - Reseaux Recurrents

Un Reseau de Neurones Recurrent (RNN) est un reseau qui traite les elements d'une sequence un par un, en maintenant un etat cache (hidden state) qui sert de "memoire" et qui est mis a jour a chaque pas de temps. Le meme jeu de poids est partage a travers tous les pas de temps.

2.1 Architecture et equations

A chaque pas de temps t, le RNN recoit deux entrees : l'element courant x_t et l'etat cache precedent h_{t-1}. Il produit un nouvel etat cache h_t :

Equations du RNN vanilla

h_t = tanh(W_hh * h_{t-1} + W_xh * x_t + b_h)

y_t = W_hy * h_t + b_y

Ou :

  • x_t : entree au temps t (vecteur de dimension d_input)
  • h_t : etat cache au temps t (vecteur de dimension d_hidden)
  • W_xh : matrice de poids entree -> cache (d_hidden x d_input)
  • W_hh : matrice de poids cache -> cache (d_hidden x d_hidden)
  • W_hy : matrice de poids cache -> sortie (d_output x d_hidden)
  • b_h, b_y : biais

2.3 Implementation d'un RNN from scratch

import torch import torch.nn as nn class RNNFromScratch(nn.Module): """RNN vanilla implemente manuellement pour comprendre le mecanisme.""" def __init__(self, input_size, hidden_size, output_size): super().__init__() self.hidden_size = hidden_size # Poids partages dans le temps self.W_xh = nn.Linear(input_size, hidden_size) # entree -> cache self.W_hh = nn.Linear(hidden_size, hidden_size) # cache -> cache self.W_hy = nn.Linear(hidden_size, output_size) # cache -> sortie def forward(self, x): """ x : (batch_size, seq_len, input_size) Retourne : outputs (batch, seq_len, output_size), h_final """ batch_size, seq_len, _ = x.shape h = torch.zeros(batch_size, self.hidden_size).to(x.device) outputs = [] for t in range(seq_len): # Equation fondamentale du RNN h = torch.tanh(self.W_xh(x[:, t, :]) + self.W_hh(h)) y = self.W_hy(h) outputs.append(y.unsqueeze(1)) outputs = torch.cat(outputs, dim=1) return outputs, h # Test rnn = RNNFromScratch(input_size=10, hidden_size=32, output_size=5) x = torch.randn(4, 20, 10) # batch=4, seq_len=20, features=10 out, h_final = rnn(x) print(f"Entree : {x.shape}") print(f"Sorties : {out.shape}") print(f"h_final : {h_final.shape}")
Entree : torch.Size([4, 20, 10]) Sorties : torch.Size([4, 20, 5]) h_final : torch.Size([4, 32])
Cette implementation manuelle est pedagogique. En pratique, on utilise nn.RNN de PyTorch qui est optimisee avec des operations CUDA et du calcul parallele. Mais comprendre la boucle explicite est essentiel pour apprehender les architectures plus complexes.

3. Entrainement des RNN

3.1 BPTT - Backpropagation Through Time

Le BPTT (Backpropagation Through Time) est l'algorithme d'entrainement des RNN. On "deplie" le reseau dans le temps et on applique la retropropagation standard sur le graphe deplie. Les gradients se propagent a rebours dans le temps, de t=T jusqu'a t=0.

Le probleme fondamental est que les gradients doivent traverser de nombreuses multiplications matricielles (une par pas de temps). Selon les valeurs propres de W_hh, deux scenarios problematiques emergent :

3.2 Vanishing Gradient (gradient qui s'evanouit)

Le probleme

Si les valeurs propres de W_hh sont inferieures a 1, les gradients decroissent exponentiellement au fur et a mesure qu'ils remontent dans le temps. Apres quelques dizaines de pas, le gradient est si proche de zero que les premiers elements de la sequence n'ont aucune influence sur la mise a jour des poids.

Mathematiquement : si on note le gradient du loss L par rapport a h_t :

dL/dh_1 = dL/dh_T * (W_hh)^(T-1) * produit des derivees de tanh

Puisque la derivee de tanh est dans [0, 1] et que les valeurs propres de W_hh sont typiquement < 1, ce produit converge vers 0 exponentiellement.

Le RNN vanilla ne peut pas capturer les dependances a long terme. Dans la phrase "Le chat, que j'ai adopte il y a trois ans dans un refuge pres de Lyon, dort sur le canape", le RNN a du mal a lier "chat" au verbe "dort" a cause des dizaines de mots entre les deux.

3.3 Exploding Gradient (gradient qui explose)

A l'inverse, si les valeurs propres de W_hh sont superieures a 1, les gradients croissent exponentiellement. Les poids subissent des mises a jour enormes qui destabilisent completement l'entrainement.

# Solution : Gradient Clipping # Limite la norme du gradient a une valeur maximale optimizer.zero_grad() loss.backward() # Clip le gradient a une norme maximale de 1.0 torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step()
Le gradient clipping est une rustine necessaire mais pas une solution au vanishing gradient. C'est pour resoudre ce probleme fondamental que le LSTM a ete invente.

4. LSTM en detail

Le LSTM (Long Short-Term Memory), propose par Hochreiter et Schmidhuber en 1997, est une variante de RNN specialement concue pour capturer les dependances a long terme. Sa structure introduit un cell state (etat de cellule) qui agit comme un "tapis roulant d'information", ainsi que trois portes (gates) qui controlent le flux d'information.

4.1 Architecture de la cellule LSTM

4.2 Les trois portes en detail

1. Forget Gate (Porte d'oubli) - "Que doit-on oublier ?"

f_t = sigmoid(W_f * [h_{t-1}, x_t] + b_f)

Produit un vecteur de valeurs entre 0 et 1. Une valeur proche de 0 signifie "oublie cette information du cell state", une valeur proche de 1 signifie "conserve cette information".

2. Input Gate (Porte d'entree) - "Que doit-on memoriser ?"

i_t = sigmoid(W_i * [h_{t-1}, x_t] + b_i)

c_tilde_t = tanh(W_c * [h_{t-1}, x_t] + b_c)

L'input gate decide quelles valeurs mettre a jour. Les candidats c_tilde sont les nouvelles informations potentielles. Le produit i_t * c_tilde determine ce qui sera effectivement ajoute au cell state.

3. Output Gate (Porte de sortie) - "Que doit-on communiquer ?"

o_t = sigmoid(W_o * [h_{t-1}, x_t] + b_o)

h_t = o_t * tanh(C_t)

L'output gate determine quelle partie du cell state mis a jour sera exposee comme hidden state (sortie de la cellule). Le cell state est d'abord passe par tanh (valeurs entre -1 et 1), puis filtre par la porte de sortie.

4.3 Mise a jour du Cell State

L'equation cle

C_t = f_t * C_{t-1} + i_t * c_tilde_t

C'est cette equation qui resout le vanishing gradient. Le cell state est mis a jour par des operations additives, pas multiplicatives. L'information peut circuler le long du cell state sans etre multipliee de maniere repetee, preservant ainsi les gradients sur de longues sequences.

Analogie : le carnet de notes

Imaginez que le cell state est un carnet de notes que vous tenez en lisant un long roman :

  • Forget gate : vous rayez les informations devenues obsoletes ("le personnage A est vivant" -> raye quand il meurt)
  • Input gate : vous ecrivez de nouvelles informations importantes ("nouveau personnage B introduit")
  • Output gate : quand on vous pose une question, vous ne lisez que les notes pertinentes a la question, pas tout le carnet

5. GRU - Gated Recurrent Unit

Le GRU (Gated Recurrent Unit), propose par Cho et al. en 2014, est une version simplifiee du LSTM. Il fusionne le cell state et le hidden state en un seul etat, et n'utilise que deux portes au lieu de trois : la porte de reset et la porte de mise a jour.

5.1 Architecture du GRU

5.2 Les deux portes du GRU

Reset Gate (r_t) - "Combien du passe ignorer ?"

r_t = sigmoid(W_r * [h_{t-1}, x_t] + b_r)

Quand r_t est proche de 0, le candidat h_tilde est calcule en ignorant presque completement l'etat cache precedent. Cela permet au reseau de "repartir de zero" si necessaire.

Update Gate (z_t) - "Combien du nouveau garder ?"

z_t = sigmoid(W_z * [h_{t-1}, x_t] + b_z)

Joue un double role : elle combine les fonctions du forget gate et de l'input gate du LSTM. Quand z_t est proche de 1, on adopte le candidat (nouvelle info). Quand z_t est proche de 0, on conserve l'ancien h_{t-1}.

5.3 LSTM vs GRU

CritereLSTMGRU
Nombre de portes3 (forget, input, output)2 (reset, update)

Parametres~4x plus que RNN vanilla~3x plus que RNN vanilla
VitessePlus lentPlus rapide (~20-30%)

En pratique, la difference de performance entre LSTM et GRU est souvent marginale. Le GRU est prefere quand la vitesse d'entrainement est critique ou quand on a peu de donnees. Le LSTM reste le choix par defaut pour les sequences tres longues ou les taches exigeant une grande capacite memorielle.

6. Architectures avancees

6.1 Bidirectional RNN

Un RNN bidirectionnel traite la sequence dans les deux sens : de gauche a droite et de droite a gauche. Les deux hidden states sont concatenes, ce qui donne au reseau acces au contexte passe et futur a chaque position.
# RNN Bidirectionnel en PyTorch bilstm = nn.LSTM( input_size=100, hidden_size=128, num_layers=2, bidirectional=True, # Active le mode bidirectionnel batch_first=True, dropout=0.3 ) x = torch.randn(8, 50, 100) # batch=8, seq=50, features=100 output, (h_n, c_n) = bilstm(x) print(f"Output : {output.shape}") # (8, 50, 256) - 128*2 car bidirectionnel print(f"h_n : {h_n.shape}") # (4, 8, 128) - 2 layers * 2 directions print(f"c_n : {c_n.shape}")
Output : torch.Size([8, 50, 256]) h_n : torch.Size([4, 8, 128]) c_n : torch.Size([4, 8, 128])

6.2 Deep RNN (Stacked LSTM)

On peut empiler plusieurs couches de LSTM. La sortie de la premiere couche sert d'entree a la deuxieme, et ainsi de suite. Cela permet au reseau d'apprendre des representations hierarchiques des sequences.

Nombre de couches recommande

  • 1 couche : suffisant pour la plupart des taches simples
  • 2 couches : standard pour les taches NLP moderees
  • 3-4 couches : traduction automatique, synthese vocale
  • 6+ couches : rare pour les RNN, plus courant avec les Transformers

Au-dela de 3-4 couches, le rendement decroissant s'installe rapidement pour les RNN. Le dropout entre les couches est essentiel pour eviter l'overfitting.

6.3 Preview : le mecanisme d'attention

Le mecanisme d'attention (Bahdanau et al., 2015) est une innovation majeure qui permet au decodeur d'un reseau sequence-to-sequence de "regarder" toutes les positions de la sequence d'entree a chaque pas de decodage, plutot que de se fier uniquement au dernier hidden state.

L'attention a resolu le goulot d'etranglement de l'encodeur : toute l'information de la sequence source n'a plus besoin d'etre compressee dans un seul vecteur. C'est cette idee qui, poussee a son extreme, donnera naissance aux Transformers (cours 7).

7. Applications pratiques

7.1 Classification de texte

L'une des applications les plus courantes des RNN/LSTM est la classification de texte : analyse de sentiment, detection de spam, categorisation de documents.

import torch import torch.nn as nn class TextClassifier(nn.Module): """Classifieur de texte base sur un LSTM bidirectionnel.""" def __init__(self, vocab_size, embed_dim, hidden_dim, num_classes, num_layers=2, dropout=0.3): super().__init__() # Couche d'embedding : transforme les indices de mots en vecteurs denses self.embedding = nn.Embedding(vocab_size, embed_dim, padding_idx=0) # LSTM bidirectionnel self.lstm = nn.LSTM( input_size=embed_dim, hidden_size=hidden_dim, num_layers=num_layers, bidirectional=True, batch_first=True, dropout=dropout ) # Classifieur self.classifier = nn.Sequential( nn.Dropout(dropout), nn.Linear(hidden_dim * 2, 128), # *2 car bidirectionnel nn.ReLU(), nn.Dropout(dropout), nn.Linear(128, num_classes) ) def forward(self, x): # x : (batch, seq_len) - indices de mots embedded = self.embedding(x) # (batch, seq_len, embed_dim) lstm_out, (h_n, _) = self.lstm(embedded) # lstm_out: (batch, seq, hidden*2) # Concatener les derniers hidden states des deux directions # h_n[-2] : derniere couche, direction forward # h_n[-1] : derniere couche, direction backward hidden = torch.cat([h_n[-2], h_n[-1]], dim=1) # (batch, hidden*2) return self.classifier(hidden) # Instanciation model = TextClassifier( vocab_size=20000, embed_dim=128, hidden_dim=256, num_classes=3 # positif, negatif, neutre ) # Test avec un batch de phrases tokenisees fake_input = torch.randint(0, 20000, (16, 100)) # 16 phrases de 100 tokens logits = model(fake_input) print(f"Logits : {logits.shape}")
Logits : torch.Size([16, 3])

7.2 Generation de texte

Les RNN/LSTM peuvent generer du texte caractere par caractere ou mot par mot. A chaque pas, le modele predit le prochain token en se basant sur l'historique genere jusqu'ici.

class CharGenerator(nn.Module): """Generateur de texte caractere par caractere.""" def __init__(self, vocab_size, embed_dim, hidden_dim, num_layers=2): super().__init__() self.hidden_dim = hidden_dim self.num_layers = num_layers self.embedding = nn.Embedding(vocab_size, embed_dim) self.lstm = nn.LSTM(embed_dim, hidden_dim, num_layers, batch_first=True, dropout=0.2) self.fc = nn.Linear(hidden_dim, vocab_size) def forward(self, x, hidden=None): embedded = self.embedding(x) output, hidden = self.lstm(embedded, hidden) logits = self.fc(output) return logits, hidden def generate(self, start_token, max_len=200, temperature=0.8): """Generation autoregressive avec temperature.""" self.eval() hidden = None current = start_token.unsqueeze(0).unsqueeze(0) # (1, 1) generated = [start_token.item()] with torch.no_grad(): for _ in range(max_len): logits, hidden = self.forward(current, hidden) # Appliquer la temperature logits = logits[0, -1] / temperature probs = torch.softmax(logits, dim=-1) # Echantillonner le prochain caractere next_token = torch.multinomial(probs, 1) generated.append(next_token.item()) current = next_token.unsqueeze(0) return generated
La temperature controle la creativite de la generation. T=1.0 donne la distribution originale. T<1.0 rend le modele plus conservateur (choisit les tokens les plus probables). T>1.0 le rend plus "creatif" mais aussi plus incoherent. En pratique, T entre 0.7 et 0.9 donne les meilleurs resultats.

8.2 Padding et Packing pour les sequences de longueurs variables

En pratique, les sequences d'un batch n'ont pas toutes la meme longueur. PyTorch fournit des utilitaires pour gerer cela efficacement :

from torch.nn.utils.rnn import pad_sequence, pack_padded_sequence, pad_packed_sequence # Sequences de longueurs differentes seq1 = torch.randn(10, 64) # longueur 10 seq2 = torch.randn(7, 64) # longueur 7 seq3 = torch.randn(15, 64) # longueur 15 seq4 = torch.randn(4, 64) # longueur 4 # Etape 1 : Padding (ajouter des zeros pour egaliser les longueurs) sequences = [seq3, seq1, seq2, seq4] # Triees par longueur decroissante lengths = torch.tensor([15, 10, 7, 4]) padded = pad_sequence(sequences, batch_first=True) # (4, 15, 64) print(f"Padded batch : {padded.shape}") # Etape 2 : Packing (compresser pour ignorer les zeros lors du calcul) packed = pack_padded_sequence(padded, lengths, batch_first=True, enforce_sorted=True) # Etape 3 : Passer par le LSTM lstm = nn.LSTM(64, 128, batch_first=True) packed_output, (h_n, c_n) = lstm(packed) # Etape 4 : Decompresser (unpack) output, output_lengths = pad_packed_sequence(packed_output, batch_first=True) print(f"Output apres unpack : {output.shape}") print(f"Longueurs reelles : {output_lengths}")
Padded batch : torch.Size([4, 15, 64]) Output apres unpack : torch.Size([4, 15, 128]) Longueurs reelles : tensor([15, 10, 7, 4])
Le packing est important pour la performance ET la correction. Sans packing, le LSTM traite les tokens de padding comme de vrais tokens, ce qui pollue les hidden states et degrade les performances. Avec le packing, PyTorch saute automatiquement les positions paddees.

8.3 Collate function personnalisee

from torch.utils.data import Dataset, DataLoader class TextDataset(Dataset): def __init__(self, texts, labels): self.texts = texts # Liste de tenseurs de longueurs variables self.labels = labels # Liste de labels def __len__(self): return len(self.texts) def __getitem__(self, idx): return self.texts[idx], self.labels[idx] def collate_fn(batch): """Fonction de collation pour gerer les longueurs variables.""" texts, labels = zip(*batch) # Trier par longueur decroissante (requis pour pack_padded_sequence) lengths = torch.tensor([len(t) for t in texts]) sorted_indices = lengths.argsort(descending=True) sorted_texts = [texts[i] for i in sorted_indices] sorted_labels = torch.tensor([labels[i] for i in sorted_indices]) sorted_lengths = lengths[sorted_indices] # Padding padded = pad_sequence(sorted_texts, batch_first=True, padding_value=0) return padded, sorted_labels, sorted_lengths # Utilisation # loader = DataLoader(dataset, batch_size=32, collate_fn=collate_fn)

9.1 Preparation des donnees

import torch import torch.nn as nn import numpy as np from torch.utils.data import Dataset, DataLoader # ============================================ # Generation d'une serie temporelle synthetique # ============================================ np.random.seed(42) t = np.arange(0, 1000, 0.1) # Signal = tendance + saisonnalite + bruit signal = 0.005 * t + np.sin(0.1 * t) + 0.5 * np.sin(0.03 * t) + np.random.normal(0, 0.1, len(t)) # Normalisation mean_val = signal.mean() std_val = signal.std() signal_norm = (signal - mean_val) / std_val # Creation des fenetres glissantes def create_sequences(data, seq_length, pred_length=1): """Cree des paires (sequence, cible) avec une fenetre glissante.""" xs, ys = [], [] for i in range(len(data) - seq_length - pred_length + 1): x = data[i : i + seq_length] y = data[i + seq_length : i + seq_length + pred_length] xs.append(x) ys.append(y) return np.array(xs), np.array(ys) SEQ_LEN = 50 PRED_LEN = 1 X, Y = create_sequences(signal_norm, SEQ_LEN, PRED_LEN) # Split train/val/test n = len(X) train_size = int(n * 0.7) val_size = int(n * 0.15) X_train, Y_train = X[:train_size], Y[:train_size] X_val, Y_val = X[train_size:train_size+val_size], Y[train_size:train_size+val_size] X_test, Y_test = X[train_size+val_size:], Y[train_size+val_size:] # Conversion en tenseurs X_train = torch.FloatTensor(X_train).unsqueeze(-1) # (N, 50, 1) Y_train = torch.FloatTensor(Y_train) X_val = torch.FloatTensor(X_val).unsqueeze(-1) Y_val = torch.FloatTensor(Y_val) print(f"Train : {X_train.shape} -> {Y_train.shape}") print(f"Val : {X_val.shape} -> {Y_val.shape}")

9.2 Modele LSTM pour series temporelles

class TimeSeriesLSTM(nn.Module): """Modele LSTM pour la prediction de series temporelles.""" def __init__(self, input_size=1, hidden_size=64, num_layers=2, output_size=1, dropout=0.2): super().__init__() self.lstm = nn.LSTM( input_size=input_size, hidden_size=hidden_size, num_layers=num_layers, batch_first=True, dropout=dropout ) self.fc = nn.Sequential( nn.Linear(hidden_size, 32), nn.ReLU(), nn.Linear(32, output_size) ) def forward(self, x): # x : (batch, seq_len, input_size) lstm_out, _ = self.lstm(x) # Prendre uniquement le dernier hidden state last_output = lstm_out[:, -1, :] # (batch, hidden_size) prediction = self.fc(last_output) return prediction

9.3 Boucle d'entrainement

# ============================================ # Entrainement complet # ============================================ device = torch.device('cuda' if torch.cuda.is_available() else 'cpu') model = TimeSeriesLSTM(input_size=1, hidden_size=64, num_layers=2).to(device) criterion = nn.MSELoss() optimizer = torch.optim.Adam(model.parameters(), lr=1e-3) scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau( optimizer, mode='min', patience=5, factor=0.5 ) # DataLoaders train_dataset = torch.utils.data.TensorDataset(X_train, Y_train) val_dataset = torch.utils.data.TensorDataset(X_val, Y_val) train_loader = DataLoader(train_dataset, batch_size=64, shuffle=True) val_loader = DataLoader(val_dataset, batch_size=64) EPOCHS = 30 best_val_loss = float('inf') for epoch in range(EPOCHS): # === Phase d'entrainement === model.train() train_loss = 0.0 for batch_x, batch_y in train_loader: batch_x = batch_x.to(device) batch_y = batch_y.to(device) optimizer.zero_grad() pred = model(batch_x) loss = criterion(pred, batch_y) loss.backward() torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step() train_loss += loss.item() train_loss /= len(train_loader) # === Phase de validation === model.eval() val_loss = 0.0 with torch.no_grad(): for batch_x, batch_y in val_loader: batch_x = batch_x.to(device) batch_y = batch_y.to(device) pred = model(batch_x) val_loss += criterion(pred, batch_y).item() val_loss /= len(val_loader) scheduler.step(val_loss) # Sauvegarde du meilleur modele if val_loss < best_val_loss: best_val_loss = val_loss torch.save(model.state_dict(), 'best_model.pt') if (epoch + 1) % 5 == 0: print(f"Epoch {epoch+1}/{EPOCHS} - Train: {train_loss:.6f} - Val: {val_loss:.6f}")

Notez trois pratiques essentielles dans cette boucle : (1) le gradient clipping pour stabiliser l'entrainement du LSTM, (2) le scheduler ReduceLROnPlateau qui reduit le learning rate quand la validation stagne, et (3) la sauvegarde du meilleur modele basee sur la loss de validation, pas la loss d'entrainement.

9.4 Metriques pour les series temporelles

def evaluate_timeseries(model, test_loader, device): """Evalue le modele avec plusieurs metriques.""" model.eval() all_preds = [] all_targets = [] with torch.no_grad(): for batch_x, batch_y in test_loader: pred = model(batch_x.to(device)) all_preds.append(pred.cpu()) all_targets.append(batch_y) preds = torch.cat(all_preds) targets = torch.cat(all_targets) # MSE mse = torch.mean((preds - targets) ** 2).item() # MAE mae = torch.mean(torch.abs(preds - targets)).item() # RMSE rmse = mse ** 0.5 print(f"MSE : {mse:.6f}") print(f"MAE : {mae:.6f}") print(f"RMSE : {rmse:.6f}") return preds, targets

10. Recapitulatif et transition vers les Transformers

10.1 Synthese des architectures recurrentes

ArchitecturePortesAvantagesInconvenients
RNN vanilla0Simple, rapideVanishing gradient, memoire courte
LSTM3Memoire longue, tres etudieLent, beaucoup de parametres
GRU2Plus rapide que LSTM, efficaceMemoire parfois insuffisante
BiLSTM3 x2Contexte bidirectionnelPas pour la generation

10.2 Quand utiliser quoi ?

Guide de decision

  • Series temporelles simples : LSTM ou GRU unidirectionnel
  • Classification de texte : BiLSTM avec attention
  • Generation de texte : LSTM unidirectionnel (autoregressive)
  • NER / Tagging : BiLSTM-CRF
  • Traduction : Encoder BiLSTM + Decoder LSTM + Attention
  • Sequences tres longues (>500 tokens) : preferer les Transformers

10.3 Les limites des RNN

Malgre leurs succes, les RNN souffrent de limitations fondamentales qui ont motive le developpement des Transformers :

Traitement sequentiel : un RNN traite les tokens un par un. Impossible de paralleliser le calcul sur une sequence. Sur un GPU moderne, les Transformers sont 10-100x plus rapides a entrainer.
Memoire limitee en pratique : meme le LSTM peine sur des sequences de plus de 500-1000 tokens. Le mecanisme d'attention des Transformers n'a pas cette limitation (mais a un cout quadratique en memoire).
Difficulte a capturer les dependances tres longues : malgre le cell state, l'information se degrade progressivement. Les Transformers, avec leur attention directe entre toutes les positions, resolvent ce probleme elegamment.

10.4 Vers les Transformers

La grande idee

Et si on se debarrassait completement de la recurrence ? Et si, au lieu de traiter les tokens un par un, on permettait a chaque token de "regarder" directement tous les autres tokens de la sequence ? C'est exactement ce que fait le mecanisme de Self-Attention, coeur des Transformers.

Le papier "Attention Is All You Need" (Vaswani et al., 2017) a montre qu'un modele base uniquement sur l'attention, sans aucune recurrence ni convolution, pouvait surpasser les LSTM sur la traduction automatique tout en etant bien plus rapide a entrainer. C'est cette revolution que nous etudierons au prochain cours.

Analogie : la reunion d'equipe

Le RNN, c'est comme un telephone arabe : chaque personne chuchote le message a la suivante. L'information se degrade inevitablement. Le Transformer, c'est comme une reunion de travail ou tout le monde est dans la meme salle : chacun peut directement s'adresser a n'importe qui, sans intermediaire. L'information circule sans perte.

Les RNN ne sont pas "morts" pour autant. Ils restent pertinents pour le deploiement sur appareils a ressources limitees, pour les series temporelles courtes, et pour les applications ou la latence d'inference est critique (un RNN peut generer token par token avec un etat cache constant). Les Transformers, eux, brillent quand on dispose de ressources de calcul importantes et de grandes quantites de donnees. Rendez-vous au cours 7 pour decouvrir cette architecture revolutionnaire.
