1. Du prototype a la production

La majorite des projets de Deep Learning echouent lors du passage en production. Un modele qui fonctionne dans un notebook Jupyter est tres different d'un modele qui sert des millions de requetes en temps reel. Ce dernier cours couvre toutes les etapes necessaires pour deployer un modele de maniere fiable et efficace.

Analogie : le prototype de voiture

Entrainer un modele de DL, c'est comme construire un prototype de voiture dans un garage. Le prototype roule, mais il n'est pas pret pour la production en serie : il faut l'optimiser pour le cout, la fiabilite, la maintenance, la securite et la scalabilite. Le passage du prototype a la production est un metier a part entiere.

1.1 Les defis du deploiement

Les 5 defis majeurs

  • Latence : Un modele en production doit repondre en quelques millisecondes (10-100ms typiquement). Un ResNet-152 prend ~50ms sur GPU mais ~500ms sur CPU.
  • Memoire : GPT-3 necessite ~350 Go de VRAM en FP16. Meme un modele "moyen" peut necessiter plusieurs Go.
  • Cout : Les GPU en cloud coutent $1-30/heure. Optimiser le modele reduit directement les couts d'inference.
  • Scalabilite : Servir 10 requetes/s est trivial. Servir 10 000 requetes/s necessite une architecture robuste.
  • Reproductibilite : Le modele doit produire les memes resultats en production qu'en developpement.

2. ONNX - Open Neural Network Exchange

ONNX est un format ouvert pour representer les modeles de Machine Learning. Il permet d'exporter un modele entraine dans un framework (PyTorch) et de l'executer dans un autre runtime optimise (ONNX Runtime, TensorRT, etc.).

ONNX (Open Neural Network Exchange) est un standard ouvert qui definit un ensemble commun d'operateurs et un format de fichier pour representer les modeles de Deep Learning, independamment du framework d'entrainement utilise.

2.1 Pourquoi ONNX ?

  • Portabilite : un modele ONNX peut tourner sur PyTorch, TensorFlow, Caffe2, ou des runtimes specialises
  • Performance : ONNX Runtime applique des optimisations de graphe (fusion d'operateurs, elimination de noeuds inutiles)
  • Deploiement multi-plateforme : CPU, GPU, mobile, edge, navigateur web
  • Acceleration hardware : support TensorRT (NVIDIA), OpenVINO (Intel), CoreML (Apple)

2.2 Export PyTorch vers ONNX

3. TorchScript

TorchScript est le propre systeme de PyTorch pour exporter des modeles en un format serializeable et optimisable, executable sans Python.

3.1 torch.jit.trace vs torch.jit.script

jit.trace

Execute le modele avec une entree exemple et enregistre les operations. Ideal pour les modeles sans branchement conditionnel.

jit.script

Analyse le code Python et le compile. Supporte les conditions if/else et les boucles. Plus flexible mais plus restrictif sur la syntaxe.

import torch # --- Methode 1 : torch.jit.trace --- model.eval() example = torch.randn(1, 3, 224, 224) traced_model = torch.jit.trace(model, example) traced_model.save("model_traced.pt") # Verification output_original = model(example) output_traced = traced_model(example) print(f"Difference max: {(output_original - output_traced).abs().max().item()}") # --- Methode 2 : torch.jit.script --- class ConditionalModel(nn.Module): def __init__(self): super().__init__() self.linear = nn.Linear(10, 5) def forward(self, x, use_relu: bool = True): x = self.linear(x) if use_relu: x = torch.relu(x) return x cond_model = ConditionalModel() scripted_model = torch.jit.script(cond_model) scripted_model.save("model_scripted.pt") # Chargement sans le code Python original loaded = torch.jit.load("model_traced.pt") output = loaded(example) print(f"Inference TorchScript: {output.shape}")
Difference max: 0.0 Inference TorchScript: torch.Size([1, 10])
torch.jit.trace ne capture pas les branchements conditionnels. Si votre modele contient des if/else qui dependent des donnees, le trace ne suivra qu'un seul chemin. Utilisez torch.jit.script pour ces cas.

4. Quantization

La quantization reduit la precision numerique des poids et activations (de FP32 a INT8, par exemple), reduisant la taille du modele et accelerant l'inference.

La quantization est le processus de conversion des poids et activations d'un modele de nombres a virgule flottante (FP32, FP16) vers des entiers de precision inferieure (INT8, INT4), avec une perte minimale de precision.

5. Pruning (elagage)

Le pruning consiste a supprimer les connexions (poids) les moins importantes du reseau, reduisant sa taille et son cout de calcul.

Le pruning est une technique d'optimisation qui met a zero les poids ayant la plus faible magnitude dans le reseau, creant un modele "creux" (sparse) qui necessite moins de calculs et de memoire pour l'inference.

5.1 Types de pruning

Unstructured Pruning

Supprime des poids individuels. Haute sparsity possible mais necessite du hardware specialise pour accelerer reellement.

Structured Pruning

Supprime des filtres ou canaux entiers. Acceleration reelle sur hardware standard mais sparsity plus limitee.

5.2 Implementation avec torch.nn.utils.prune

import torch import torch.nn as nn import torch.nn.utils.prune as prune # Modele exemple model = nn.Sequential( nn.Linear(784, 256), nn.ReLU(), nn.Linear(256, 128), nn.ReLU(), nn.Linear(128, 10) ) # --- Unstructured Pruning : supprimer 30% des poids les plus petits --- for name, module in model.named_modules(): if isinstance(module, nn.Linear): prune.l1_unstructured(module, name="weight", amount=0.3) # Verifier la sparsity def compute_sparsity(model): total = 0 zeros = 0 for p in model.parameters(): total += p.numel() zeros += (p == 0).sum().item() return zeros / total * 100 print(f"Sparsity: {compute_sparsity(model):.1f}%") # --- Rendre le pruning permanent --- for name, module in model.named_modules(): if isinstance(module, nn.Linear): prune.remove(module, "weight") # --- Structured Pruning : supprimer 20% des filtres --- conv_model = nn.Conv2d(64, 128, 3, padding=1) prune.ln_structured(conv_model, name="weight", amount=0.2, n=1, dim=0) print(f"Filtres restants effectifs: {128 - int(128 * 0.2)} / 128")
Sparsity: 30.0% Filtres restants effectifs: 102 / 128

6. Knowledge Distillation

La knowledge distillation transfere les connaissances d'un grand modele (teacher) vers un petit modele (student), obtenant un modele compact et performant.

La knowledge distillation (Hinton et al., 2015) entraine un modele student a imiter les sorties (soft labels) d'un modele teacher plus grand, plutot que de s'entrainer uniquement sur les labels durs du dataset.

6.1 L'intuition

Analogie : l'apprenti et le maitre

Imaginez un maitre artisan (teacher) qui transmet son savoir a un apprenti (student). L'apprenti n'apprend pas seulement les reponses correctes, mais aussi les nuances : "cette piece ressemble aussi un peu a celle-la". Les soft probabilities du teacher contiennent plus d'information que les simples labels durs.

6.2 Temperature et soft labels

Le role de la temperature T

On divise les logits par une temperature T avant d'appliquer le softmax :

p_i = exp(z_i / T) / sum(exp(z_j / T))

  • T = 1 : softmax classique (sharp)
  • T > 1 : distribution plus douce, revele les relations entre classes
  • Typiquement T = 3 a 20 pour la distillation

Avec T eleve, la probabilite de la classe "8" quand la vraie classe est "3" revele que les deux chiffres se ressemblent visuellement. Cette information est perdue avec les labels durs.

6.3 Loss de distillation

import torch import torch.nn as nn import torch.nn.functional as F class DistillationLoss(nn.Module): def __init__(self, temperature=4.0, alpha=0.7): super().__init__() self.temperature = temperature self.alpha = alpha # Poids de la distillation vs hard labels self.ce_loss = nn.CrossEntropyLoss() def forward(self, student_logits, teacher_logits, targets): # 1. Soft loss (KL divergence entre les distributions douces) soft_teacher = F.softmax(teacher_logits / self.temperature, dim=1) soft_student = F.log_softmax(student_logits / self.temperature, dim=1) soft_loss = F.kl_div(soft_student, soft_teacher, reduction="batchmean") soft_loss = soft_loss * (self.temperature ** 2) # Compensation du scaling # 2. Hard loss (cross-entropy classique avec les vrais labels) hard_loss = self.ce_loss(student_logits, targets) # 3. Combinaison ponderee total_loss = self.alpha * soft_loss + (1 - self.alpha) * hard_loss return total_loss

6.4 Boucle d'entrainement de distillation

# Teacher : grand modele pre-entraine (fige) teacher_model = BigResNet().to(device) teacher_model.load_state_dict(torch.load("teacher_weights.pth")) teacher_model.eval() # Student : petit modele a entrainer student_model = SmallNet().to(device) # Configuration distill_criterion = DistillationLoss(temperature=4.0, alpha=0.7) optimizer = torch.optim.Adam(student_model.parameters(), lr=1e-3) # Entrainement for epoch in range(20): student_model.train() for data, targets in train_loader: data, targets = data.to(device), targets.to(device) # Forward du teacher (sans gradient!) with torch.no_grad(): teacher_logits = teacher_model(data) # Forward du student student_logits = student_model(data) # Loss de distillation loss = distill_criterion(student_logits, teacher_logits, targets) optimizer.zero_grad() loss.backward() optimizer.step() # Evaluation _, acc = evaluate(student_model, val_loader, nn.CrossEntropyLoss(), device) print(f"Epoch {epoch+1} | Student Acc: {acc:.2f}%")
Epoch 1 | Student Acc: 96.45% Epoch 5 | Student Acc: 98.12% Epoch 10 | Student Acc: 98.67% Epoch 20 | Student Acc: 98.91%
Le student distille atteint souvent des performances proches du teacher tout en etant 5 a 10 fois plus petit et rapide. C'est la technique utilisee par exemple pour creer DistilBERT (60% plus petit, 97% des performances de BERT).

7. Serving - Mettre le modele en ligne

Une fois le modele optimise, il faut le rendre accessible via une API pour que les applications puissent l'utiliser.

7.1 FastAPI + PyTorch (solution simple et flexible)

# api.py - Serveur FastAPI pour l'inference from fastapi import FastAPI, UploadFile, File from fastapi.responses import JSONResponse import torch import torch.nn.functional as F from torchvision import transforms from PIL import Image import io app = FastAPI(title="Image Classification API") # Charger le modele au demarrage device = torch.device("cuda" if torch.cuda.is_available() else "cpu") model = torch.jit.load("model_traced.pt", map_location=device) model.eval() # Preprocessing preprocess = transforms.Compose([ transforms.Resize(256), transforms.CenterCrop(224), transforms.ToTensor(), transforms.Normalize( mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225] ) ]) CLASSES = ["chat", "chien", "oiseau", "poisson", "fleur", "voiture", "avion", "maison", "arbre", "personne"] @app.post("/predict") async def predict(file: UploadFile = File(...)): """Endpoint de prediction d'image.""" # Lire et pre-traiter l'image image_bytes = await file.read() image = Image.open(io.BytesIO(image_bytes)).convert("RGB") input_tensor = preprocess(image).unsqueeze(0).to(device) # Inference with torch.no_grad(): logits = model(input_tensor) probs = F.softmax(logits, dim=1) confidence, predicted = probs.max(1) return JSONResponse({ "class": CLASSES[predicted.item()], "confidence": round(confidence.item(), 4), "all_probabilities": { cls: round(p, 4) for cls, p in zip(CLASSES, probs[0].tolist()) } }) @app.get("/health") async def health(): return {"status": "ok", "device": str(device)} # Lancer avec : uvicorn api:app --host 0.0.0.0 --port 8000

9.3 CI/CD pour le Machine Learning

Pipeline CI/CD typique pour le ML

  1. Tests unitaires : verifier les shapes, les types, les outputs du modele
  2. Tests de qualite des donnees : verifier les distributions, les valeurs manquantes
  3. Entrainement automatise : declenche par un push ou un schedule
  4. Evaluation automatique : comparer les metriques avec le modele en production
  5. Validation : tests A/B, canary deployment
  6. Deploiement : mise en production si les metriques sont satisfaisantes
  7. Monitoring : alertes si les performances degradent (data drift, model drift)

Le Deep Learning est un domaine en evolution rapide. Les fondamentaux couverts dans cette formation resteront pertinents, mais les architectures et techniques specifiques evolueront. L'apprentissage continu est essentiel : suivez les publications des conferences majeures (NeurIPS, ICML, ICLR, CVPR, ACL) et experimentez constamment. Le meilleur apprentissage passe par la pratique et la reimplementation des papiers.
