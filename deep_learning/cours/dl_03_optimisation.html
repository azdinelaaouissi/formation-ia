Cours 3/12 - SGD, Adam, learning rate scheduling et regularisation

1. Le probleme d'optimisation en Deep Learning

Entrainer un reseau de neurones revient a minimiser une fonction de cout (loss) L(θ) par rapport aux parametres θ du modele. Ce probleme est non-convexe, de tres haute dimension, et presente de nombreux defis specifiques.

En Deep Learning, la surface de la loss (le "paysage de la loss") est extremement complexe. Contrairement a la regression lineaire ou la fonction de cout est convexe avec un unique minimum global, les reseaux de neurones presentent :

  • Des minima locaux : points ou le gradient est nul et la courbure est positive dans toutes les directions
  • Des saddle points (points de selle) : points ou le gradient est nul mais la courbure est positive dans certaines directions et negative dans d'autres
  • Des plateaux : regions ou le gradient est tres proche de zero sur de grandes etendues
  • Des ravines : zones ou la loss change rapidement dans une direction mais lentement dans une autre
En haute dimension, les saddle points sont beaucoup plus frequents que les minima locaux. Pour un reseau avec des millions de parametres, la probabilite d'avoir un vrai minimum local (courbure positive dans TOUTES les directions) est astronomiquement faible. Le vrai probleme en pratique, ce sont les saddle points et les plateaux.

Bonne nouvelle

Des recherches recentes montrent que pour les grands reseaux, la plupart des minima locaux ont des valeurs de loss tres proches du minimum global. En pratique, trouver un "bon" minimum local suffit generalement a obtenir d'excellentes performances de generalisation.

1.1 Formulation mathematique

L'objectif est de trouver les parametres optimaux θ* :

θ* = argmin L(θ) θ ou L(θ) = (1/N) ∑ l(f(x_i; θ), y_i) pour i = 1 a N avec : - f(x_i; θ) : la prediction du modele - y_i : la vraie etiquette - l : la fonction de perte individuelle (ex: cross-entropy) - N : le nombre d'exemples d'entrainement

2. Descente de Gradient (Gradient Descent)

La descente de gradient est un algorithme iteratif d'optimisation qui met a jour les parametres dans la direction opposee au gradient de la fonction de cout : θ ← θ - η ∇L(θ), ou η est le learning rate (taux d'apprentissage).

2.1 Batch Gradient Descent (GD complet)

Le gradient est calcule sur l'ensemble du jeu d'entrainement a chaque mise a jour :

θ_{t+1} = θ_t - η * (1/N) ∑ ∇ l(f(x_i; θ_t), y_i)
  • Avantage : gradient tres precis, convergence stable
  • Inconvenient : extremement lent pour de grands datasets, ne tient pas en memoire GPU

2.2 Stochastic Gradient Descent (SGD)

Le gradient est calcule sur un seul exemple choisi aleatoirement :

θ_{t+1} = θ_t - η * ∇ l(f(x_i; θ_t), y_i) pour un seul i aleatoire
  • Avantage : mises a jour tres rapides, peut s'echapper des minima locaux grace au bruit
  • Inconvenient : gradient tres bruite, convergence oscillante

2.3 Mini-Batch Gradient Descent

Compromis entre les deux : le gradient est calcule sur un mini-batch de B exemples :

θ_{t+1} = θ_t - η * (1/B) ∑ ∇ l(f(x_j; θ_t), y_j) pour j dans le mini-batch
En pratique, quand on parle de "SGD" en Deep Learning, on fait presque toujours reference au mini-batch SGD. La taille typique du batch est de 32, 64, 128 ou 256. Le choix depend de la memoire GPU disponible et de la nature du probleme.

3. Momentum et Nesterov Accelerated Gradient

Analogie : la balle qui roule

Imaginez une balle qui roule sur le paysage de la loss. Sans momentum, la balle se deplace uniquement en fonction de la pente locale : si la pente change de direction, la balle change immediatement. Avec momentum, la balle a une inertie : elle accumule de la vitesse dans une direction et ne change pas de cap instantanement. Cela lui permet de traverser les petites bosses (bruit du SGD) et d'accelerer dans les vallees etroites.

3.1 SGD avec Momentum

Le momentum introduit un terme de vitesse v_t qui accumule les gradients passes avec un facteur d'attenuation β (typiquement 0.9) :
v_t = β * v_{t-1} + ∇L(θ_t) θ_{t+1} = θ_t - η * v_t ou : - v_t : le vecteur de vitesse (momentum) - β : le coefficient de momentum (0.9 typiquement) - η : le learning rate
Le momentum lisse les oscillations du SGD et accelere la convergence, surtout dans les directions ou le gradient est consistant. Il permet de traverser les plateaux et les saddle points plus rapidement.

3.2 Nesterov Accelerated Gradient (NAG)

L'idee de Nesterov est de calculer le gradient non pas a la position actuelle, mais a la position anticipee apres l'application du momentum :

v_t = β * v_{t-1} + ∇L(θ_t - η * β * v_{t-1}) θ_{t+1} = θ_t - η * v_t Intuition : "regarde devant toi avant de corriger ta trajectoire"
Nesterov est presque toujours preferable au momentum classique. En PyTorch, il suffit d'ajouter nesterov=True dans l'optimiseur SGD. En pratique, le gain est modeste mais systematique.
# SGD avec momentum classique optimizer_m = optim.SGD(model.parameters(), lr=0.01, momentum=0.9) # SGD avec Nesterov momentum optimizer_n = optim.SGD(model.parameters(), lr=0.01, momentum=0.9, nesterov=True)

4. Optimiseurs adaptatifs

Les optimiseurs adaptatifs ajustent le learning rate individuellement pour chaque parametre en fonction de l'historique de ses gradients. Cela est crucial quand differents parametres necessitent des taux d'apprentissage differents.

4.1 AdaGrad (Adaptive Gradient)

G_t = G_{t-1} + (∇L(θ_t))^2 (accumulation des gradients carres) θ_{t+1} = θ_t - (η / √(G_t + ε)) * ∇L(θ_t) ε = 1e-8 (pour eviter la division par zero)
  • Avantage : adapte le LR par parametre; bon pour les donnees eparses
  • Inconvenient : G_t croit indefiniment, le LR tend vers 0 et l'apprentissage s'arrete

4.2 RMSProp (Root Mean Square Propagation)

Propose par Geoffrey Hinton, RMSProp resout le probleme d'AdaGrad en utilisant une moyenne mobile exponentielle des gradients carres :

E[g^2]_t = β * E[g^2]_{t-1} + (1 - β) * (∇L(θ_t))^2 θ_{t+1} = θ_t - (η / √(E[g^2]_t + ε)) * ∇L(θ_t) β = 0.99 typiquement
Le learning rate ne decroit plus indefiniment, il s'adapte a la courbure locale recente de la loss.

4.3 Adam (Adaptive Moment Estimation)

Adam combine les avantages du momentum (premier moment) et de RMSProp (second moment). C'est l'optimiseur le plus utilise en Deep Learning.
# Formules d'Adam : m_t = β_1 * m_{t-1} + (1 - β_1) * g_t # Premier moment (moyenne des gradients) v_t = β_2 * v_{t-1} + (1 - β_2) * g_t^2 # Second moment (variance des gradients) # Correction du biais (crucial au debut de l'entrainement) m_hat_t = m_t / (1 - β_1^t) v_hat_t = v_t / (1 - β_2^t) # Mise a jour θ_{t+1} = θ_t - η * m_hat_t / (√(v_hat_t) + ε) # Hyperparametres par defaut : β_1 = 0.9, β_2 = 0.999, ε = 1e-8
La correction du biais (bias correction) est essentielle. Au debut (t petit), m_t et v_t sont biaises vers zero car ils sont initialises a zero. La correction divise par (1 - β^t) qui vaut presque 1 pour t grand mais corrige significativement pour t petit.

4.4 AdamW (Adam avec Weight Decay decouple)

Adam standard applique la regularisation L2 dans le gradient, ce qui interfere avec le mecanisme adaptatif. AdamW decouple le weight decay de l'optimisation :

# Adam standard (weight decay mal applique) : g_t = ∇L(θ_t) + λ * θ_t # L2 dans le gradient # AdamW (weight decay decouple) : θ_{t+1} = θ_t - η * (m_hat_t / (√(v_hat_t) + ε)) - η * λ * θ_t
AdamW est aujourd'hui considere comme le meilleur choix par defaut pour la plupart des taches de Deep Learning. C'est l'optimiseur utilise pour entrainer GPT, BERT, ViT, et la quasi-totalite des modeles modernes.

4.5 Comparaison en PyTorch

import torch.optim as optim # SGD classique opt_sgd = optim.SGD(model.parameters(), lr=0.01) # SGD avec momentum et Nesterov opt_sgd_m = optim.SGD(model.parameters(), lr=0.01, momentum=0.9, nesterov=True) # RMSProp opt_rms = optim.RMSprop(model.parameters(), lr=0.001, alpha=0.99) # Adam opt_adam = optim.Adam(model.parameters(), lr=0.001, betas=(0.9, 0.999)) # AdamW (le prefere en pratique) opt_adamw = optim.AdamW(model.parameters(), lr=0.001, weight_decay=0.01)

5. Learning Rate Scheduling

Un learning rate scheduler ajuste le learning rate au cours de l'entrainement selon une strategie predefined. L'idee est de commencer avec un LR eleve pour explorer rapidement, puis de le reduire pour affiner la convergence.

5.1 StepLR

Reduit le LR d'un facteur γ tous les step_size epochs :

# LR divise par 10 toutes les 30 epochs scheduler = optim.lr_scheduler.StepLR(optimizer, step_size=30, gamma=0.1) # lr = lr_initial * gamma^(epoch // step_size)

5.2 CosineAnnealingLR

Fait suivre au LR une courbe cosinus, redescendant de η_max a η_min :

# Cosine annealing sur T_max epochs scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=100, eta_min=1e-6) # lr_t = eta_min + 0.5 * (eta_max - eta_min) * (1 + cos(pi * t / T_max))

5.3 OneCycleLR

Strategie proposee par Leslie Smith : le LR monte d'abord (warmup) puis descend, formant un seul cycle :

# OneCycleLR avec warmup integre scheduler = optim.lr_scheduler.OneCycleLR( optimizer, max_lr=0.01, total_steps=1000, pct_start=0.3, # 30% du temps pour le warmup anneal_strategy='cos' )

5.4 ReduceLROnPlateau

Reduit le LR quand une metrique stagne pendant un certain nombre d'epochs :

# Reduction quand la validation loss stagne scheduler = optim.lr_scheduler.ReduceLROnPlateau( optimizer, mode='min', # reduire quand la metrique diminue factor=0.1, # LR *= 0.1 patience=10, # attendre 10 epochs sans amelioration min_lr=1e-7 ) # Utilisation (noter: step() prend la metrique en argument) scheduler.step(val_loss)

5.5 Warmup

Le warmup consiste a demarrer avec un LR tres bas et l'augmenter lineairement pendant les premieres iterations. C'est essentiel pour stabiliser le debut de l'entrainement, surtout avec Adam et les grands batch sizes.

# Warmup lineaire + cosine decay (utilise par les Transformers) def get_lr_lambda(current_step, warmup_steps, total_steps): if current_step < warmup_steps: return current_step / warmup_steps progress = (current_step - warmup_steps) / (total_steps - warmup_steps) return 0.5 * (1.0 + math.cos(math.pi * progress)) scheduler = optim.lr_scheduler.LambdaLR( optimizer, lr_lambda=lambda step: get_lr_lambda(step, 1000, 50000) )
En pratique, la combinaison AdamW + CosineAnnealing avec warmup est devenue la norme pour les Transformers et le NLP. Pour la vision par ordinateur avec des CNN, SGD + Momentum + StepLR donne souvent les meilleurs resultats en generalisation.

6. Regularisation

La regularisation designe l'ensemble des techniques qui empechent le modele de surapprendre (overfitting) les donnees d'entrainement. Elle penalise la complexite du modele pour ameliorer sa capacite de generalisation.

6.1 Regularisation L2 (Ridge / Weight Decay)

Ajoute un terme de penalite proportionnel a la norme L2 des poids :

L_reg(θ) = L(θ) + (λ/2) * ||θ||_2^2 ∇L_reg = ∇L(θ) + λ * θ En PyTorch, le weight decay est integre a l'optimiseur : optimizer = optim.Adam(model.parameters(), lr=0.001, weight_decay=0.01)
Pousse les poids vers zero, empechant les grandes valeurs. Les features peu utiles recoivent des poids proches de zero.

6.2 Regularisation L1 (Lasso)

L_reg(θ) = L(θ) + λ * ||θ||_1 # Pas d'option native dans les optimiseurs PyTorch # On l'ajoute manuellement : l1_lambda = 0.001 l1_norm = sum(p.abs().sum() for p in model.parameters()) loss = loss + l1_lambda * l1_norm
Produit des poids exactement nuls (parcimonie / sparsity). Utile pour la selection de features.

6.3 Dropout

Le Dropout desactive aleatoirement une fraction p des neurones pendant l'entrainement. A chaque forward pass, un masque binaire aleatoire est applique. A l'inference, tous les neurones sont actifs mais les sorties sont multipliees par (1-p) pour compenser.
import torch.nn as nn # Modele avec Dropout model = nn.Sequential( nn.Linear(784, 512), nn.ReLU(), nn.Dropout(p=0.5), # 50% des neurones desactives nn.Linear(512, 256), nn.ReLU(), nn.Dropout(p=0.3), # 30% des neurones desactives nn.Linear(256, 10) ) # IMPORTANT : passer en mode eval pour l'inference ! model.eval() # Desactive le dropout model.train() # Reactive le dropout
Oublier de passer le modele en mode model.eval() avant l'inference est une erreur classique. Le dropout sera toujours actif et les predictions seront aleatoires et sous-optimales.

6.4 Data Augmentation

Bien que ce ne soit pas de la regularisation au sens strict, la data augmentation est la technique la plus efficace contre le surapprentissage :

from torchvision import transforms train_transform = transforms.Compose([ transforms.RandomHorizontalFlip(p=0.5), transforms.RandomRotation(degrees=15), transforms.RandomCrop(32, padding=4), transforms.ColorJitter(brightness=0.2, contrast=0.2), transforms.ToTensor(), transforms.Normalize((0.5,), (0.5,)) ])

7. Batch Normalization en detail

La Batch Normalization (BatchNorm) normalise les activations de chaque couche en soustrayant la moyenne et divisant par l'ecart-type du mini-batch, puis applique une transformation affine apprise. Proposee par Ioffe & Szegedy en 2015.

7.1 Le probleme : Internal Covariate Shift

Lorsqu'on empile de nombreuses couches, la distribution des activations de chaque couche change a chaque mise a jour des parametres des couches precedentes. Les couches suivantes doivent constamment s'adapter a des distributions changeantes, ce qui ralentit et destabilise l'entrainement.

7.2 Formules de BatchNorm

# Pour un mini-batch B = {x_1, ..., x_B} : # 1. Calculer la moyenne du mini-batch μ_B = (1/B) ∑ x_i # 2. Calculer la variance du mini-batch σ^2_B = (1/B) ∑ (x_i - μ_B)^2 # 3. Normaliser x_hat_i = (x_i - μ_B) / √(σ^2_B + ε) # 4. Transformation affine apprise (scale & shift) y_i = γ * x_hat_i + β # γ et β sont des parametres appris par le reseau # Ils permettent au reseau de "defaire" la normalisation si necessaire

7.3 Train vs Eval

Mode entrainement vs inference

  • Entrainement : μ et σ sont calcules sur le mini-batch courant. Des moyennes mobiles exponentielles (running_mean, running_var) sont mises a jour.
  • Inference : μ et σ utilisent les statistiques accumulees (running_mean, running_var) car on peut avoir un seul exemple (pas de batch).
import torch.nn as nn # BatchNorm dans un MLP model = nn.Sequential( nn.Linear(784, 256), nn.BatchNorm1d(256), # BN pour tenseurs 1D (apres Linear) nn.ReLU(), nn.Linear(256, 128), nn.BatchNorm1d(128), nn.ReLU(), nn.Linear(128, 10) ) # BatchNorm dans un CNN conv_block = nn.Sequential( nn.Conv2d(3, 64, kernel_size=3, padding=1), nn.BatchNorm2d(64), # BN pour tenseurs 2D (apres Conv2d) nn.ReLU() )

7.4 Layer Normalization

Alternative a BatchNorm qui normalise sur les features plutot que sur le batch. Utilisee dans les Transformers car elle ne depend pas de la taille du batch :

# LayerNorm normalise sur la derniere dimension layer_norm = nn.LayerNorm(normalized_shape=256) # Comparaison # BatchNorm : normalise sur le batch (dimension 0) # LayerNorm : normalise sur les features (derniere dimension)
BatchNorm est a utiliser avec les CNN et les MLP. LayerNorm est a utiliser avec les Transformers et les RNN. La regle est simple : si votre batch size est variable ou petit, preferez LayerNorm.

8. Initialisation des poids

L'initialisation des poids est critique pour le bon demarrage de l'entrainement. Une mauvaise initialisation peut mener a des gradients qui explosent ou disparaissent des les premieres iterations.

8.1 Pourquoi pas zeros ?

Initialiser tous les poids a zero est catastrophique. Tous les neurones d'une couche calculent la meme chose, recoivent le meme gradient, et restent identiques a jamais. C'est le probleme de symetrie : le reseau ne peut pas apprendre des representations differentes.

8.2 Xavier / Glorot (2010)

L'initialisation de Xavier (pour les activations lineaires, sigmoide, tanh) initialise les poids selon : W ~ U(-a, a) ou N(0, σ^2) avec a = √(6 / (n_in + n_out)) et σ^2 = 2 / (n_in + n_out). L'objectif est de maintenir la variance des activations constante a travers les couches.

8.3 He / Kaiming (2015)

L'initialisation de He (pour ReLU et ses variantes) compense le fait que ReLU annule la moitie des activations : W ~ N(0, σ^2) avec σ^2 = 2 / n_in. C'est l'initialisation par defaut recommandee pour les reseaux utilisant ReLU.
import torch.nn as nn import torch.nn.init as init # Initialisation Xavier (pour tanh/sigmoid) init.xavier_uniform_(layer.weight) init.xavier_normal_(layer.weight) # Initialisation He/Kaiming (pour ReLU) init.kaiming_uniform_(layer.weight, nonlinearity='relu') init.kaiming_normal_(layer.weight, nonlinearity='relu') # Initialiser un modele complet def init_weights(m): if isinstance(m, nn.Linear): init.kaiming_normal_(m.weight, nonlinearity='relu') if m.bias is not None: init.zeros_(m.bias) elif isinstance(m, nn.Conv2d): init.kaiming_normal_(m.weight, mode='fan_out', nonlinearity='relu') model.apply(init_weights)

En pratique, PyTorch utilise deja Kaiming Uniform par defaut pour les couches nn.Linear et nn.Conv2d. Vous n'avez generalement pas besoin de reinitialiser manuellement, sauf si vous utilisez des activations non standard.

9. Problemes de gradient

9.1 Vanishing Gradients (gradients qui disparaissent)

Le vanishing gradient se produit quand les gradients deviennent exponentiellement petits en remontant les couches lors de la retropropagation. Les couches proches de l'entree ne recoivent presque plus de signal de gradient et n'apprennent plus.

Causes principales :

  • Fonctions d'activation saturantes (sigmoid, tanh) dont les derivees sont proches de 0 aux extremes
  • Reseaux tres profonds sans techniques de stabilisation
  • Mauvaise initialisation des poids

Solutions :

  • Utiliser ReLU ou ses variantes (LeakyReLU, ELU, GELU)
  • Batch Normalization
  • Connexions residuelles (skip connections)
  • Initialisation He/Kaiming

9.2 Exploding Gradients (gradients qui explosent)

L'exploding gradient est le probleme inverse : les gradients deviennent exponentiellement grands, causant des mises a jour de poids enormes et une divergence de l'entrainement. Tres frequent dans les RNN.

Solutions :

  • Gradient Clipping : technique la plus directe
  • Batch Normalization
  • Reduction du learning rate
  • Meilleure initialisation

9.3 Gradient Clipping

# Gradient Clipping par norme (le plus courant) # Limite la norme globale du gradient a max_norm torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) # Gradient Clipping par valeur # Clipe chaque composante du gradient dans [-clip_value, clip_value] torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=0.5) # Utilisation dans la boucle d'entrainement : optimizer.zero_grad() loss.backward() torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) # AVANT step() optimizer.step()

9.4 Skip Connections (introduction)

Les connexions residuelles permettent au gradient de "sauter" des couches, resolvant elegamment le vanishing gradient :

# Bloc residuel simple class ResidualBlock(nn.Module): def __init__(self, dim): super().__init__() self.block = nn.Sequential( nn.Linear(dim, dim), nn.BatchNorm1d(dim), nn.ReLU(), nn.Linear(dim, dim), nn.BatchNorm1d(dim) ) self.relu = nn.ReLU() def forward(self, x): residual = x # Sauvegarder l'entree out = self.block(x) # Transformation out = out + residual # Skip connection ! return self.relu(out)

Pourquoi les skip connections fonctionnent ?

Si F(x) est la transformation du bloc, la sortie est y = F(x) + x. Le gradient est ∇y = ∇F(x) + 1. Meme si ∇F(x) tend vers zero, le gradient est au minimum 1 grace au terme identite. Le gradient ne peut plus disparaitre !

10. Recapitulatif : boucle d'entrainement complete en PyTorch

Voici un exemple complet qui integre toutes les notions vues dans ce cours : optimiseur, scheduler, regularisation, gradient clipping, et bonnes pratiques.

import torch import torch.nn as nn import torch.optim as optim from torch.utils.data import DataLoader from torchvision import datasets, transforms # ============================================ # 1. CONFIGURATION # ============================================ device = torch.device('cuda' if torch.cuda.is_available() else 'cpu') EPOCHS = 50 BATCH_SIZE = 128 LR = 0.001 WEIGHT_DECAY = 0.01 # ============================================ # 2. DONNEES avec augmentation # ============================================ train_transform = transforms.Compose([ transforms.RandomHorizontalFlip(), transforms.RandomCrop(28, padding=4), transforms.ToTensor(), transforms.Normalize((0.1307,), (0.3081,)) ]) test_transform = transforms.Compose([ transforms.ToTensor(), transforms.Normalize((0.1307,), (0.3081,)) ]) train_dataset = datasets.MNIST('./data', train=True, download=True, transform=train_transform) test_dataset = datasets.MNIST('./data', train=False, transform=test_transform) train_loader = DataLoader(train_dataset, batch_size=BATCH_SIZE, shuffle=True) test_loader = DataLoader(test_dataset, batch_size=BATCH_SIZE) # ============================================ # 3. MODELE avec BatchNorm et Dropout # ============================================ class MLP(nn.Module): def __init__(self): super().__init__() self.flatten = nn.Flatten() self.network = nn.Sequential( nn.Linear(784, 512), nn.BatchNorm1d(512), nn.ReLU(), nn.Dropout(0.3), nn.Linear(512, 256), nn.BatchNorm1d(256), nn.ReLU(), nn.Dropout(0.2), nn.Linear(256, 10) ) def forward(self, x): x = self.flatten(x) return self.network(x) model = MLP().to(device) # ============================================ # 4. OPTIMISEUR + SCHEDULER # ============================================ optimizer = optim.AdamW(model.parameters(), lr=LR, weight_decay=WEIGHT_DECAY) scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=EPOCHS) criterion = nn.CrossEntropyLoss() # ============================================ # 5. BOUCLE D'ENTRAINEMENT # ============================================ best_val_acc = 0.0 for epoch in range(EPOCHS): # --- Phase d'entrainement --- model.train() train_loss = 0.0 correct = 0 total = 0 for batch_x, batch_y in train_loader: batch_x, batch_y = batch_x.to(device), batch_y.to(device) optimizer.zero_grad() outputs = model(batch_x) loss = criterion(outputs, batch_y) loss.backward() # Gradient clipping torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step() train_loss += loss.item() * batch_x.size(0) _, predicted = outputs.max(1) total += batch_y.size(0) correct += predicted.eq(batch_y).sum().item() train_loss /= total train_acc = 100. * correct / total # --- Phase de validation --- model.eval() val_loss = 0.0 correct = 0 total = 0 with torch.no_grad(): for batch_x, batch_y in test_loader: batch_x, batch_y = batch_x.to(device), batch_y.to(device) outputs = model(batch_x) loss = criterion(outputs, batch_y) val_loss += loss.item() * batch_x.size(0) _, predicted = outputs.max(1) total += batch_y.size(0) correct += predicted.eq(batch_y).sum().item() val_loss /= total val_acc = 100. * correct / total # Mise a jour du scheduler scheduler.step() # Sauvegarde du meilleur modele if val_acc > best_val_acc: best_val_acc = val_acc torch.save(model.state_dict(), 'best_model.pth') current_lr = optimizer.param_groups[0]['lr'] print(f'Epoch {epoch+1}/{EPOCHS} | ' f'Train Loss: {train_loss:.4f} | Train Acc: {train_acc:.2f}% | ' f'Val Loss: {val_loss:.4f} | Val Acc: {val_acc:.2f}% | ' f'LR: {current_lr:.6f}') print(f'\nMeilleure accuracy en validation : {best_val_acc:.2f}%')

Checklist d'entrainement

  1. Choisir l'optimiseur : AdamW en general, SGD+Momentum pour les CNN en vision
  2. Choisir le LR initial : 0.001 pour Adam, 0.01-0.1 pour SGD
  3. Ajouter un scheduler : CosineAnnealing ou OneCycleLR
  4. Ajouter un warmup pour les grands modeles
  5. Regulariser : weight decay (0.01), Dropout (0.1-0.5)
  6. Utiliser BatchNorm apres chaque couche (avant l'activation)
  7. Gradient clipping si instabilite (max_norm=1.0)
  8. Sauvegarder le meilleur modele sur la validation
  9. Toujours mettre model.eval() avant l'inference
  10. Utiliser torch.no_grad() pendant la validation
L'optimisation est un art autant qu'une science. Les hyperparametres par defaut d'Adam (lr=0.001, betas=(0.9, 0.999)) fonctionnent remarquablement bien dans la majorite des cas. Commencez par les valeurs par defaut et ajustez uniquement si necessaire. La recherche d'hyperparametres systematique (grid search, random search, Bayesian optimization) sera abordee dans un cours ulterieur.
