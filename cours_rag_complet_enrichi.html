Web Scraping avec Trafilatura

Source : Stephane Robert - Recuperer du contenu web pour le RAG

Pourquoi le scraping dans un RAG ?

Souvent, vos donnees ne sont pas dans des fichiers locaux mais sur des pages web : documentation en ligne, articles de blog, wikis internes, FAQ. Le scraping permet d'extraire ce contenu automatiquement.

Trafilatura : L'outil ideal

Trafilatura est une bibliotheque Python qui extrait le contenu textuel utile d'une page web en ignorant automatiquement les menus, publicites, footers, et code HTML inutile.

Installation 
pip install trafilatura
scraper.py - Scraping avec Trafilatura 
import trafilatura

def scrape_url(url):
    """Extraire le texte propre d'une page web."""

    # 1. Telecharger la page HTML
    downloaded = trafilatura.fetch_url(url)

    # 2. Extraire le contenu textuel (sans menus, pubs, etc.)
    text = trafilatura.extract(
        downloaded,
        include_comments=False,    # Pas de commentaires
        include_tables=True,       # Garder les tableaux
        include_links=False,       # Pas de liens URL
        output_format="txt"        # Format texte brut
    )

    return text

# Exemple d'utilisation
url = "https://fr.wikipedia.org/wiki/Nutrition"
content = scrape_url(url)
print(content[:500])  # Les 500 premiers caracteres

Scraper plusieurs pages

multi_scraper.py 
import trafilatura
from langchain.schema import Document

def scrape_multiple(urls):
    """Scraper une liste d'URLs et retourner des Documents."""
    documents = []

    for url in urls:
        print(f"Scraping: {url}")
        downloaded = trafilatura.fetch_url(url)
        text = trafilatura.extract(downloaded)

        if text:
            doc = Document(
                page_content=text,
                metadata={"source": url, "type": "web"}
            )
            documents.append(doc)

    print(f"\n{len(documents)} pages scrapees.")
    return documents

# Utilisation
urls = [
    "https://example.com/article1",
    "https://example.com/article2",
]
docs = scrape_multiple(urls)
# Ensuite : chunking -> embedding -> ChromaDB

Nettoyage du texte

Source : Stephane Robert / Blent.ai - L'etape oubliee mais cruciale
Statistique cle

Selon une etude de 2024, 42% des echecs de pipeline RAG sont dus a un mauvais nettoyage des donnees. Le preprocessing est souvent la cause #1 de mauvais resultats.

Pourquoi nettoyer ?

Le texte brut (surtout apres scraping) contient du bruit qui pollue les embeddings :

  • Espaces multiples, tabulations, retours a la ligne excessifs
  • Caracteres speciaux residuels (HTML, Unicode)
  • Headers, footers, menus repetes
  • Liens URL inutiles dans le texte
  • Numeros de page, copyrights

Fonctions de nettoyage

cleaner.py - Nettoyage du texte pour RAG 
import re

def clean_text(text):
    """
    Nettoie le texte brut pour le RAG.
    Supprime le bruit tout en gardant le contenu utile.
    """

    # 1. Supprimer les balises HTML residuelles
    text = re.sub(r"<[^>]+>", "", text)

    # 2. Supprimer les URLs
    text = re.sub(r"https?://\S+", "", text)

    # 3. Supprimer les caracteres speciaux inutiles
    text = re.sub(r"[^\w\s.,;:!?()'\"-]", " ", text)

    # 4. Normaliser les espaces (multiples -> un seul)
    text = re.sub(r" +", " ", text)

    # 5. Normaliser les sauts de ligne (3+ -> 2)
    text = re.sub(r"\n{3,}", "\n\n", text)

    # 6. Supprimer les espaces en debut/fin
    text = text.strip()

    return text


def clean_documents(documents):
    """Nettoie une liste de Documents LangChain."""
    for doc in documents:
        doc.page_content = clean_text(doc.page_content)
    print(f"  {len(documents)} documents nettoyes.")
    return documents
Ordre du pipeline

L'ordre correct est : Charger → Nettoyer → Chunker → Embedder. Le nettoyage DOIT se faire AVANT le chunking, sinon le bruit se retrouve dans les chunks et pollue la recherche.

Pipeline de preprocessing complet

Les 6 etapes avant le RAG (source : Stephane Robert)

Le vrai pipeline complet (pas juste "charger et chunker")

Les sources web revelent que le pipeline avant l'indexation est plus riche que ce qu'on avait :

1

Collecte des donnees

Scraping web (Trafilatura), PDF (PyPDF), CSV, BDD, API, fichiers locaux

2

Nettoyage textuel (NOUVEAU)

Supprimer HTML, URLs, caracteres speciaux, normaliser espaces. 42% des echecs RAG sont lies a cette etape.

3

Filtrage de contenu (NOUVEAU)

Supprimer les documents vides, trop courts, dupliques, ou hors sujet. Filtrer par qualite.

4

Chunking intelligent

RecursiveCharacterTextSplitter avec 300-800 tokens, 10-20% overlap. Adapter la taille au contenu.

5

Enrichissement metadata (NOUVEAU)

Ajouter source, date, auteur, categorie, langue a chaque chunk. Permet le filtrage au retrieval.

6

Embedding + Stockage

Vectoriser avec le modele choisi et stocker dans la base vectorielle.

Lecon cle de Blent.ai

"La qualite du chunking et la pertinence des documents impactent directement les performances du systeme, bien plus que le choix du LLM. Le goulot d'etranglement est la vectorisation initiale, pas la generation."

Pinecone vs ChromaDB

Source : Blent.ai - Quand utiliser une base cloud ?

ChromaDB (Local)

  • Gratuit, open source
  • Tourne sur ton PC
  • Simple : pip install chromadb
  • Ideal pour prototypage
  • Limite : 1 machine seule
  • Pas de monitoring integre
VS

Pinecone (Cloud)

  • Freemium, heberge
  • Tourne dans le cloud AWS
  • API + dashboard web
  • Ideal pour production
  • Scalable : millions de vecteurs
  • Monitoring et analytics
pinecone_exemple.py - Source : Blent.ai 
# pip install langchain-pinecone pinecone-client
from pinecone import Pinecone, ServerlessSpec
from langchain_pinecone import PineconeVectorStore

# 1. Connexion a Pinecone
pc = Pinecone(api_key="VOTRE_CLE_API")

# 2. Creer un index (si premiere fois)
pc.create_index(
    name="mon-rag",
    dimension=384,              # Doit correspondre a votre embedding
    spec=ServerlessSpec(
        cloud="aws",
        region="us-east-1"
    )
)

# 3. Stocker les embeddings
index = pc.Index("mon-rag")
vectorstore = PineconeVectorStore(index=index, embedding=embeddings)
vectorstore.add_documents(chunks)

# 4. Recherche (identique a ChromaDB !)
results = vectorstore.similarity_search("ma question", k=3)

Ollama (le plus simple)

Installe et lance des modeles en 2 commandes. Ideal debutants.

HuggingFace Pipeline

Charge le modele directement dans Python. Plus de controle.

Serveur d'inference optimise. Pour la production a haute performance.

Source : Stephane Robert

mistral_huggingface.py - Source : Blent.ai 
# Charger Mistral 7B directement avec HuggingFace
from langchain_huggingface import HuggingFacePipeline
from transformers import AutoModelForCausalLM, AutoTokenizer, pipeline

# 1. Charger le modele et le tokenizer
model_id = "mistralai/Mistral-7B-Instruct-v0.3"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id)

# 2. Creer le pipeline de generation
pipe = pipeline(
    "text-generation",
    model=model,
    tokenizer=tokenizer,
    max_new_tokens=4096,    # Longueur max de la reponse
    do_sample=False,          # Pas d'echantillonnage (deterministe)
    return_full_text=False   # Ne pas repeter le prompt
)

# 3. Wrapper LangChain
llm = HuggingFacePipeline(pipeline=pipe)

# Utilisation identique a Ollama ou OpenAI !
response = llm.invoke("Quelle est la dose de paracetamol ?")
Point cle de Blent.ai

"Les vecteurs stockes de facon permanente eliminent la re-vectorisation repetee, permettant de modifier le LLM independamment sans regenerer les embeddings." Autrement dit : tu peux switcher de Llama a Mistral sans retoucher la base vectorielle.

Checklist RAG en production

Source : Stephane Robert - De prototype a production

Donnees

  • Nettoyage automatise du texte
  • Pipeline de mise a jour incrementale
  • Detection des doublons
  • Validation de la qualite des chunks
  • Metadata enrichies (source, date, auteur)

Retrieval

  • Re-ranking active
  • Recherche hybride (Dense + Sparse)
  • Seuil de score minimum
  • Tests de pertinence reguliers
  • Monitoring des requetes sans resultats

Generation

  • Prompt template teste et valide
  • Temperature = 0 pour le factuel
  • Guardrails anti-hallucination
  • Fallback "je ne sais pas"
  • Citation des sources systematique

Operations

  • Evaluation RAGAS automatisee
  • LangSmith pour le monitoring
  • Logs des questions/reponses
  • Alertes sur latence anormale
  • Feedback utilisateur collecte

LangSmith : Monitoring du RAG

Source : LBKE - Surveiller et optimiser

C'est quoi LangSmith ?

LangSmith est l'outil de monitoring et debugging officiel de LangChain. Il permet de voir en temps reel :

  • Chaque etape du pipeline (retrieval, prompt, generation)
  • Le temps de chaque etape (latence)
  • Les tokens consommes et le cout
  • Les chunks recuperes pour chaque question
  • Les erreurs et les requetes echouees

RAG Agentique : Le code

Source : LBKE - create_agent avec outils

La difference avec le RAG classique en code

Dans un RAG classique, le retrieval est toujours execute. Dans un RAG agentique, l'agent DECIDE s'il a besoin de chercher, et quels outils utiliser.

agentic_rag.py - Source : LBKE 
# RAG Agentique : l'agent choisit ses outils
import requests
from langchain.agents import create_agent
from langchain.tools import tool

# 1. Definir un outil de recherche web
@tool
def fetch_url(url: str) -> str:
    """Recuperer le contenu textuel d'une URL."""
    response = requests.get(url, timeout=10.0)
    return response.text

# 2. Definir un outil de recherche vectorielle
@tool
def search_docs(query: str) -> str:
    """Chercher dans la base de documents indexee."""
    results = vectorstore.similarity_search(query, k=3)
    return "\n".join([doc.page_content for doc in results])

# 3. Creer l'agent avec ses outils
agent = create_agent(
    model="claude-sonnet-4-5-20250929",   # Le LLM
    tools=[fetch_url, search_docs],         # Les outils disponibles
    system_prompt="""Tu es un assistant. Utilise search_docs pour
    chercher dans la base documentaire. Si tu ne trouves pas assez
    d'info, utilise fetch_url pour chercher sur le web."""
)

# 4. L'agent decide seul quoi faire !
response = agent.invoke("Effets du paracetamol chez les diabetiques ?")
# L'agent va automatiquement :
# - Chercher d'abord dans search_docs
# - Si pas assez d'info, utiliser fetch_url
# - Combiner les resultats et repondre
Le decorateur @tool

Le @tool de LangChain transforme une simple fonction Python en un outil utilisable par l'agent. La docstring (texte entre triples guillemets) est lue par l'agent pour comprendre quand et comment utiliser l'outil.

LangGraph : Pipelines avances

Source : LBKE - Le framework pour agents complexes

C'est quoi LangGraph ?

LangGraph est le framework de LangChain pour creer des pipelines complexes avec des boucles et des conditions. Contrairement a LangChain classique (lineaire), LangGraph permet :

  • Des boucles : l'agent peut revenir en arriere et re-chercher
  • Des branches conditionnelles : "si le score < 0.7, chercher sur le web"
  • Un etat partage : les etapes partagent des informations
  • Persistence : sauvegarder l'etat entre les sessions
langgraph_rag.py - Pipeline conditionnel 
# pip install langgraph
from langgraph.graph import StateGraph, END

# Definir l'etat partage
class RAGState:
    question: str
    documents: list
    answer: str
    score: float

# Definir les noeuds (etapes)
def retrieve(state):
    """Chercher les documents pertinents."""
    docs = retriever.invoke(state.question)
    return {"documents": docs}

def evaluate(state):
    """Evaluer la pertinence des resultats."""
    score = compute_relevance(state.question, state.documents)
    return {"score": score}

def generate(state):
    """Generer la reponse."""
    answer = llm.invoke(build_prompt(state.question, state.documents))
    return {"answer": answer}

def web_search(state):
    """Fallback : chercher sur le web."""
    ...

# Construire le graphe
graph = StateGraph(RAGState)
graph.add_node("retrieve", retrieve)
graph.add_node("evaluate", evaluate)
graph.add_node("generate", generate)
graph.add_node("web_search", web_search)

# Definir les transitions conditionnelles
graph.add_edge("retrieve", "evaluate")
graph.add_conditional_edges("evaluate", {
    "good": "generate",       # Score > 0.7 -> generer
    "bad": "web_search",      # Score < 0.7 -> web search
})
graph.add_edge("web_search", "generate")
graph.add_edge("generate", END)

Glossaire complet du RAG

Tous les termes techniques en un coup d'oeil
Terme Definition
RAGRetrieval Augmented Generation - Technique qui combine recherche documentaire et generation de texte par IA
LLMLarge Language Model - Modele de langage (GPT-4, Claude, Llama, Mistral)
EmbeddingRepresentation vectorielle (liste de nombres) qui capture le sens d'un texte
VecteurListe de nombres representant un texte dans un espace mathematique
ChunkMorceau de document decoupe pour l'indexation (200-1000 caracteres)
OverlapZone de chevauchement entre deux chunks consecutifs (10-20% de la taille)
Similarite cosinusMesure de proximite entre deux vecteurs (0 = different, 1 = identique)
Base vectorielleBase de donnees specialisee pour stocker et rechercher des vecteurs (ChromaDB, FAISS, Pinecone)
RetrieverComposant qui cherche les documents les plus pertinents dans la base vectorielle
Top-KLe nombre de resultats retournes par le retriever (generalement 3-5)
Prompt templateModele de texte avec des variables {context} et {question} a remplir
TemperatureParametre du LLM : 0 = precis/deterministe, 1 = creatif/aleatoire
TokenUnite de texte traitee par le LLM (environ 1 mot = 1.3 tokens)
HallucinationQuand l'IA invente une reponse qui semble vraie mais qui est fausse
Re-rankingReordonner les resultats de recherche par pertinence reelle (Cross-Encoder)
BM25Algorithme de recherche par mots-cles classique (non-semantique)
Dense retrievalRecherche par vecteurs (embeddings) = recherche semantique
Sparse retrievalRecherche par mots-cles (BM25, TF-IDF)
Hybrid searchCombinaison de dense + sparse pour de meilleurs resultats
Knowledge GraphGraphe de connaissances : entites + relations (utilise dans Graph RAG)
RAGASFramework d'evaluation du RAG avec 4 metriques (precision, recall, faithfulness, relevancy)
LangChainFramework Python pour construire des applications LLM et RAG
LangGraphExtension de LangChain pour pipelines avec boucles et conditions
LangSmithOutil de monitoring et debugging pour LangChain
OllamaOutil pour executer des LLM en local facilement (llama, mistral)
TrafilaturaBibliotheque Python pour extraire le texte propre d'une page web
Agentic RAGRAG ou un agent autonome planifie, decide et itere ses recherches
