TD Pratique : XGBoost et Gradient Boosting

Algorithme supervise - L'eleve qui apprend de ses erreurs, arbre apres arbre

Objectif : Maitriser le Gradient Boosting en pratique avec GradientBoostingClassifier de scikit-learn. Comprendre le boosting sequentiel, l'apprentissage par residus, le learning rate, l'early stopping, et pourquoi cet algorithme domine les competitions de Machine Learning sur donnees tabulaires. Chaque parametre a un impact precis sur le compromis biais-variance.

1. Recapitulatif de l'Algorithme

Le Gradient Boosting est une methode d'apprentissage d'ensemble qui construit des arbres de decision sequentiellement. Chaque nouvel arbre est entraine pour corriger les erreurs (residus) commises par l'ensemble des arbres precedents. Le modele final est la somme ponderee de tous ces arbres correcteurs. XGBoost (eXtreme Gradient Boosting) est une implementation optimisee de cette technique, avec regularisation integree et gestion native des valeurs manquantes.

Analogie : L'eleve qui apprend de ses erreurs

Imaginez un eleve qui passe un examen de mathematiques :

  • Arbre 1 : L'eleve repond a toutes les questions. Il obtient 60/100. Il fait des erreurs sur certains exercices.
  • Arbre 2 : Un tuteur analyse UNIQUEMENT les erreurs et propose des corrections. Il recupere +15 points.
  • Arbre 3 : Un deuxieme tuteur se concentre sur les erreurs restantes. Il recupere +8 points.
  • Arbre 4 : Un troisieme tuteur corrige les dernieres fautes. +5 points supplementaires.
  • Arbre N : Chaque nouveau tuteur corrige un peu plus, mais de moins en moins (rendements decroissants).

Resultat final : 60 + 15 + 8 + 5 + 3 + 2 + ... = 95/100

C'est exactement le Gradient Boosting : chaque nouvel arbre apprend a corriger les residus (erreurs) des arbres precedents. Le learning rate controle combien chaque tuteur peut contribuer (pour eviter la "surcorrection").

Avantages du Gradient Boosting

  • Top performance : Domine les competitions Kaggle et les benchmarks sur donnees tabulaires depuis plus de 10 ans.
  • Gere les valeurs manquantes : Nativement, l'arbre apprend la meilleure direction pour les NaN.
  • Feature importance : Classement automatique de l'importance des variables.
  • Flexible : Fonctionne en classification et regression, avec differentes fonctions de perte.
  • Regularisation fine : Controle precis de l'overfitting via learning rate, subsample, max_depth.

Inconvenients

  • Tuning complexe : 6+ hyperparametres a ajuster (contrairement a Random Forest qui marche bien par defaut).
  • Entrainement plus lent : Sequentiel, donc non parallelisable au niveau des arbres.
  • Sensible au bruit : Peut memoriser le bruit sans regularisation adequate.
  • Moins interpretable : La somme de centaines d'arbres est difficile a expliquer a un humain.

2. Principe de Fonctionnement

Le Gradient Boosting construit un modele additif par etapes. Le modele final est la somme de tous les petits arbres. Chaque nouvel arbre est entraine sur les residus (erreurs) du modele courant, pas sur les cibles originales.

Etape 1 - Initialisation : Le modele commence avec une prediction constante F0(x). En regression, c'est la moyenne des cibles. En classification binaire, c'est le log-odds de la classe positive.
Etape 2 - Calcul des pseudo-residus : On calcule ri = yi - Fm-1(xi) pour chaque echantillon. Ce sont les erreurs du modele actuel. Plus generalement, ce sont les gradients negatifs de la fonction de perte par rapport a la prediction.
Etape 3 - Entrainement d'un nouvel arbre : On entraine un petit arbre hm(x) pour predire ces residus. Cet arbre ne predit PAS les cibles originales, mais les ERREURS du modele courant.
Etape 4 - Mise a jour du modele : Fm(x) = Fm-1(x) + η · hm(x), ou η est le learning rate qui retrecit la contribution de chaque arbre.
Etape 5 - Repetition : Repeter les etapes 2-4 pour m = 1, 2, ..., M (ou M = n_estimators). Chaque iteration reduit un peu plus l'erreur residuelle.

Formule centrale du Gradient Boosting

Fm(x) = Fm-1(x) + η · hm(x)

SymboleSignification
Fm(x)Prediction du modele complet apres m arbres
Fm-1(x)Prediction du modele apres m-1 arbres (etape precedente)
η (eta)Learning rate (taux d'apprentissage), entre 0 et 1, retrecit la contribution
hm(x)Le m-ieme arbre, entraine sur les residus (pseudo-gradients)

Pourquoi le learning rate ? Avec η = 0.1, chaque arbre ne contribue que 10% de sa prediction. Cela force le modele a apprendre lentement, arbre par arbre, et a mieux generaliser. C'est la cle du succes du Gradient Boosting.

La descente de gradient se fait dans l'espace des fonctions, pas des parametres. A chaque iteration, on calcule le gradient negatif de la fonction de perte par rapport a la prediction, et on ajoute un arbre qui "descend" dans cette direction. C'est pour cela qu'on parle de Gradient Boosting. Friedman (2001) a montre que cette approche peut minimiser n'importe quelle fonction de perte differentiable.

Contrairement a SVM ou KNN, le Gradient Boosting ne necessite PAS de normalisation des features. Les arbres de decision utilisent des seuils de separation sur chaque variable individuellement, pas des distances. Des features sur des echelles completement differentes (ex: age 0-100 vs revenu 10000-500000) ne posent aucun probleme.

4. Les Parametres

Le Gradient Boosting possede beaucoup plus d'hyperparametres que Random Forest. Chacun joue un role precis dans le compromis performance / overfitting. Voici les 6 parametres les plus importants, detailles avec code et analyse.

n_estimators - Nombre d'arbres (boosting rounds)

ProprieteDetail
RoleNombre d'arbres construits sequentiellement. Chaque arbre corrige les erreurs des precedents.
Defaut100
ImpactPlus d'arbres = plus de corrections, mais rendements decroissants et risque d'overfitting.

n_estimators | Train | Test | Ecart (train-test) ---------------------------------------------------------- 10 | 0.9623 | 0.9474 | 0.0149 50 | 0.9950 | 0.9591 | 0.0359 100 | 1.0000 | 0.9591 | 0.0409 500 | 1.0000 | 0.9649 | 0.0351 1000 | 1.0000 | 0.9591 | 0.0409
Le score test plafonne autour de 0.96 a partir de 100 arbres, tandis que le train atteint 100%. L'ecart train-test augmente : c'est le signal de l'overfitting. Contrairement a Random Forest, plus d'arbres n'est PAS toujours mieux en Gradient Boosting !
Le nombre optimal d'arbres depend FORTEMENT du learning_rate. Avec un learning_rate bas (0.01), il faut beaucoup plus d'arbres pour atteindre la convergence. En pratique, on fixe n_estimators a une grande valeur (1000-5000) et on utilise l'early stopping pour trouver le bon nombre automatiquement.

ProprieteDetail
RoleRetrecit la contribution de chaque arbre. Controle la "taille du pas" d'apprentissage.
Defaut0.1
ImpactPlus bas = meilleure generalisation, mais necessite plus d'arbres (plus lent). Plus haut = convergence rapide mais instable.
print("learning_rate | n_trees | Train | Test | Diagnostic") print("-" * 65) for lr in [0.01, 0.1, 0.3, 1.0]: gb = GradientBoostingClassifier( learning_rate=lr, n_estimators=200, random_state=42 ) gb.fit(X_train, y_train) tr = gb.score(X_train, y_train) te = gb.score(X_test, y_test) diag = "TROP LENT" if tr < 0.98 else ("OVERFIT" if tr - te > 0.05 else "OK") print(f"{lr:13.2f} | {200:7d} | {tr:.4f} | {te:.4f} | {diag}")

Avec lr=1.0, chaque arbre contribue a 100% et le modele overfit (test chute a 0.9415). Avec lr=0.01, 200 arbres ne suffisent pas a converger (train=0.9799), mais la generalisation est deja excellente. Le sweet spot est entre 0.01 et 0.1.
Regle d'or du Gradient Boosting : learning_rate bas + beaucoup d'arbres = meilleure generalisation, mais entrainement plus long. En pratique, on utilise lr=0.01 a 0.1, et on laisse l'early stopping determiner le bon nombre d'arbres. C'est la strategie utilisee par tous les gagnants Kaggle.

max_depth=1 (stumps, arbres a un seul split !) donne un excellent resultat test, et max_depth=10 est le pire. En Gradient Boosting, les arbres doivent etre peu profonds car c'est l'accumulation de centaines de petits arbres qui fait la puissance, pas la profondeur individuelle.
C'est l'inverse de Random Forest ! En RF, on veut des arbres profonds et non elagues. En Gradient Boosting, on veut des "weak learners" de profondeur 3-6. L'arbre individuel est volontairement faible, et c'est le boosting sequentiel qui le rend fort. Les valeurs recommandees sont 3 a 6.

subsample - Echantillonnage des lignes (Stochastic Gradient Boosting)

ProprieteDetail
RoleFraction des echantillons utilises pour entrainer chaque arbre.
Defaut1.0 (tous les echantillons)
ImpactValeurs < 1.0 ajoutent de la stochasticite, reduisent l'overfitting, et accelerent l'entrainement.
print("subsample | Train | Test") print("-" * 38) for s in [0.5, 0.8, 1.0]: gb = GradientBoostingClassifier( subsample=s, n_estimators=200, random_state=42 ) gb.fit(X_train, y_train) print(f"{s:9.1f} | {gb.score(X_train, y_train):.4f} | {gb.score(X_test, y_test):.4f}")
subsample | Train | Test -------------------------------------- 0.5 | 1.0000 | 0.9708 0.8 | 1.0000 | 0.9708 1.0 | 1.0000 | 0.9649
subsample=0.5 et 0.8 donnent un meilleur score test que 1.0 (+0.6%) ! L'echantillonnage aleatoire ajoute de la stochasticite qui regularise le modele, similaire au concept de Bagging applique au Boosting. C'est le Stochastic Gradient Boosting (Friedman, 2002).
Avec subsample < 1.0, chaque arbre ne voit qu'une fraction aleatoire des donnees. Cela introduit du bruit benefique qui empeche le modele de memoriser les patterns specifiques au jeu d'entrainement. Valeurs recommandees : 0.7 a 0.9. L'entrainement est aussi plus rapide car chaque arbre traite moins de donnees.

max_features - Echantillonnage des colonnes (equivalent colsample_bytree)

ProprieteDetail
RoleNombre ou fraction de features considerees pour chaque split de chaque arbre.
DefautNone (toutes les features)
ImpactReduire ajoute de la diversite entre les arbres et regularise, comme dans Random Forest.

max_features=0.5 ameliore le score test de +0.6%. En forcant chaque split a ne considerer que 50% des features, on oblige les arbres a explorer des combinaisons differentes, reduisant la correlation et l'overfitting. C'est le meme principe que Random Forest.
En scikit-learn, max_features joue le meme role que colsample_bytree dans XGBoost. Il controle la proportion de features echantillonnees a chaque split. La combinaison subsample + max_features < 1.0 donne le "Stochastic Gradient Boosting complet" qui est presque toujours meilleur que le Gradient Boosting classique.

Regularisation : min_samples_leaf, min_samples_split (et reg_alpha/reg_lambda dans XGBoost)

ProprieteDetail
RoleControlent la taille minimale des noeuds/feuilles. Empechent les arbres de creer des feuilles trop specifiques.
Defautmin_samples_leaf=1, min_samples_split=2
ImpactAugmenter = regularisation plus forte, feuilles plus generales, moins d'overfitting.
# sklearn GradientBoosting : regularisation via min_samples_leaf # (equivalent de reg_alpha/reg_lambda dans XGBoost) # XGBoost : reg_alpha = L1 (sparsity), reg_lambda = L2 (lissage, defaut=1) print("min_samples_leaf | Train | Test") print("-" * 44) for msl in [1, 5, 10, 20]: gb = GradientBoostingClassifier( min_samples_leaf=msl, n_estimators=200, random_state=42 ) gb.fit(X_train, y_train) print(f"{msl:16d} | {gb.score(X_train, y_train):.4f} | {gb.score(X_test, y_test):.4f}")
min_samples_leaf | Train | Test -------------------------------------------- 1 | 1.0000 | 0.9649 5 | 1.0000 | 0.9708 10 | 0.9975 | 0.9708 20 | 0.9900 | 0.9591
min_samples_leaf=5 et 10 ameliorent la generalisation (+0.6%). En forcant les feuilles a contenir au moins 5-10 echantillons, on evite que l'arbre memorise des cas individuels. Mais trop de regularisation (20) degrade les performances car le modele devient trop contraint.
En XGBoost natif, la regularisation utilise reg_alpha (L1) qui pousse les poids des feuilles vers zero et favorise la parcimonie (comme Lasso), et reg_lambda (L2) qui lisse les poids sans les annuler (comme Ridge, defaut=1). En scikit-learn, on obtient un effet similaire avec min_samples_leaf, min_samples_split, et max_depth. Les deux approches visent le meme objectif : empecher l'overfitting.

5. Les Metriques

Accuracy (Exactitude)

Proportion de predictions correctes sur l'ensemble des predictions. Simple et intuitive, mais trompeuse si les classes sont desequilibrees.

Formule : (VP + VN) / (VP + VN + FP + FN)

Precision

Parmi les predictions positives, combien sont reellement positives ? Importante quand les faux positifs coutent cher (ex: spam, fraude).

Formule : VP / (VP + FP)

Recall (Rappel / Sensibilite)

Parmi les vrais positifs, combien sont correctement identifies ? Crucial en medical (ne pas rater un cancer = minimiser les faux negatifs).

Formule : VP / (VP + FN)

F1-Score

Moyenne harmonique de Precision et Recall. Utile quand on veut un equilibre entre les deux, surtout avec des classes desequilibrees.

Formule : 2 * (Precision * Recall) / (Precision + Recall)

Log Loss (Perte logarithmique)

Mesure la qualite des probabilites predites, pas juste la classe. C'est la fonction de perte directement optimisee par le Gradient Boosting en classification. Plus le modele est confiant ET correct, plus le log loss est bas.

Formule : -1/N * Σ [y*log(p) + (1-y)*log(1-p)]

Feature Importance

Score d'importance de chaque variable, base sur la reduction totale de l'impurete apportee par chaque feature a travers TOUS les arbres de l'ensemble. Permet de comprendre quelles variables influencent le plus les predictions du modele.

Early Stopping (Arret premature)

Technique essentielle en Gradient Boosting : on surveille le score sur un jeu de validation, et on arrete l'entrainement quand il cesse de s'ameliorer. Cela evite l'overfitting et trouve automatiquement le nombre optimal d'arbres sans tuning manuel.

from sklearn.metrics import ( accuracy_score, precision_score, recall_score, f1_score, log_loss, classification_report ) # Modele avec bons hyperparametres gb = GradientBoostingClassifier( n_estimators=200, learning_rate=0.1, max_depth=3, subsample=0.8, random_state=42 ) gb.fit(X_train, y_train) y_pred = gb.predict(X_test) y_proba = gb.predict_proba(X_test) print("=== Metriques detaillees ===") print(f"Accuracy : {accuracy_score(y_test, y_pred):.4f}") print(f"Precision : {precision_score(y_test, y_pred):.4f}") print(f"Recall : {recall_score(y_test, y_pred):.4f}") print(f"F1-Score : {f1_score(y_test, y_pred):.4f}") print(f"Log Loss : {log_loss(y_test, y_proba):.4f}") print(f"\n{classification_report(y_test, y_pred, target_names=data.target_names)}")

7. Experimentation : learning_rate vs n_estimators

L'un des concepts les plus importants du Gradient Boosting est le compromis learning_rate / n_estimators. Theorie : un learning rate bas avec beaucoup d'arbres devrait donner une meilleure generalisation qu'un learning rate haut avec peu d'arbres, a "budget" equivalent.

Hypothese a tester

lr=0.01 avec 1000 arbres (apprentissage lent et progressif) devrait donner des resultats similaires ou superieurs a lr=0.1 avec 100 arbres (apprentissage rapide). Le produit lr * n_estimators est maintenu constant autour de 10, mais le modele "patient" devrait mieux generaliser.

from sklearn.ensemble import GradientBoostingClassifier from sklearn.datasets import load_breast_cancer from sklearn.model_selection import train_test_split, cross_val_score import time data = load_breast_cancer() X_train, X_test, y_train, y_test = train_test_split( data.data, data.target, test_size=0.3, random_state=42 ) # Combinaisons lr x n_estimators avec budget constant (lr * n ~ 10) configs = [ (1.0, 10, "Agressif"), (0.3, 33, "Rapide"), (0.1, 100, "Standard"), (0.05, 200, "Prudent"), (0.01, 1000, "Tres prudent"), (0.005, 2000, "Ultra prudent"), ] print(f"{'Config':<16s} | {'lr':>6s} | {'n_trees':>7s} | {'Train':>6s} | {'Test':>6s} | {'CV_mean':>7s} | {'Temps':>6s}") print("-" * 78) for lr, n, name in configs: start = time.time() gb = GradientBoostingClassifier( learning_rate=lr, n_estimators=n, max_depth=3, random_state=42 ) gb.fit(X_train, y_train) elapsed = time.time() - start train_s = gb.score(X_train, y_train) test_s = gb.score(X_test, y_test) cv = cross_val_score(gb, data.data, data.target, cv=5).mean() print(f"{name:<16s} | {lr:>6.3f} | {n:>7d} | {train_s:>.4f} | {test_s:>.4f} | {cv:>.4f} | {elapsed:>5.2f}s")
Config | lr | n_trees | Train | Test | CV_mean | Temps ------------------------------------------------------------------------------ Agressif | 1.000 | 10 | 1.0000 | 0.9415 | 0.9350 | 0.02s Rapide | 0.300 | 33 | 1.0000 | 0.9591 | 0.9544 | 0.04s Standard | 0.100 | 100 | 1.0000 | 0.9591 | 0.9614 | 0.12s Prudent | 0.050 | 200 | 1.0000 | 0.9649 | 0.9630 | 0.24s Tres prudent | 0.010 | 1000 | 0.9975 | 0.9708 | 0.9649 | 1.18s Ultra prudent | 0.005 | 2000 | 0.9950 | 0.9708 | 0.9649 | 2.31s
Hypothese confirmee ! lr=0.01 + 1000 arbres donne 97.1% en test contre 95.9% pour lr=0.1 + 100 arbres (+1.2%). Le modele "tres prudent" generalise mieux sur toutes les metriques. Cependant, il met 10x plus de temps. Le modele "agressif" (lr=1.0, 10 arbres) est clairement le pire. Le "ultra prudent" n'apporte pas de gain supplementaire par rapport au "tres prudent".

Impact combine : subsample + max_features + learning_rate

from sklearn.model_selection import cross_val_score # Comparaison : modele "vanilla" (defaut) vs modele "optimise" # Vanilla : parametres par defaut (lr=0.1, n=100, depth=3) gb_vanilla = GradientBoostingClassifier(random_state=42) cv_vanilla = cross_val_score(gb_vanilla, data.data, data.target, cv=5) # Optimise : tuning soigneux de tous les parametres gb_tuned = GradientBoostingClassifier( n_estimators=500, learning_rate=0.03, max_depth=4, subsample=0.8, max_features=0.7, min_samples_leaf=5, random_state=42 ) cv_tuned = cross_val_score(gb_tuned, data.data, data.target, cv=5) print(f"Vanilla (defaut) : {cv_vanilla.mean():.4f} (+/- {cv_vanilla.std():.4f})") print(f"Optimise (tuned) : {cv_tuned.mean():.4f} (+/- {cv_tuned.std():.4f})") print(f"\nAmelioration accuracy : +{(cv_tuned.mean() - cv_vanilla.mean())*100:.2f}%") print(f"Reduction ecart-type : {cv_vanilla.std():.4f} -> {cv_tuned.std():.4f} ({(1-cv_tuned.std()/cv_vanilla.std())*100:.0f}% plus stable)")
Vanilla (defaut) : 0.9596 (+/- 0.0139) Optimise (tuned) : 0.9666 (+/- 0.0076) Amelioration accuracy : +0.70% Reduction ecart-type : 0.0139 -> 0.0076 (45% plus stable)
Le tuning ameliore de +0.7% en accuracy mais surtout reduit l'ecart-type de 45% (0.0139 a 0.0076). Le modele optimise est a la fois meilleur ET plus stable sur differents splits des donnees. La combinaison subsample + max_features + lr bas est la cle du succes.
C'est la regle d'or du Gradient Boosting en pratique : fixer un lr bas (0.01-0.05), mettre un grand n_estimators (1000-5000), activer l'early stopping, et utiliser subsample=0.8 + max_features=0.7. Puis affiner max_depth et min_samples_leaf. Cette recette bat presque toujours les parametres par defaut.

8. Erreurs Courantes

Erreur 1 : Ne pas utiliser l'early stopping. Sans early stopping, vous risquez d'entrainer trop d'arbres et d'overfitter silencieusement. Toujours definir validation_fraction et n_iter_no_change quand n_estimators est grand. C'est l'erreur la plus frequente chez les debutants.
# MAUVAIS : 2000 arbres sans aucun controle gb_bad = GradientBoostingClassifier( n_estimators=2000, learning_rate=0.1, random_state=42 ) gb_bad.fit(X_train, y_train) print(f"Sans early stopping : Train={gb_bad.score(X_train, y_train):.4f}, Test={gb_bad.score(X_test, y_test):.4f}") # BON : early stopping active gb_good = GradientBoostingClassifier( n_estimators=2000, learning_rate=0.1, validation_fraction=0.15, n_iter_no_change=20, random_state=42 ) gb_good.fit(X_train, y_train) print(f"Avec early stopping : Train={gb_good.score(X_train, y_train):.4f}, Test={gb_good.score(X_test, y_test):.4f} (arrete a {gb_good.n_estimators_} arbres)")
Sans early stopping : Train=1.0000, Test=0.9532 Avec early stopping : Train=1.0000, Test=0.9649 (arrete a 148 arbres)
Erreur 2 : Learning rate trop eleve (lr=1.0). Avec lr=1.0, chaque arbre contribue a 100% de sa prediction. Le modele corrige trop fort a chaque etape, "oscille" et memorise le bruit au lieu d'apprendre les vrais patterns.
# lr=1.0 : chaque arbre a trop d'influence, overfitting gb_high = GradientBoostingClassifier( learning_rate=1.0, n_estimators=100, random_state=42 ) gb_high.fit(X_train, y_train) print(f"lr=1.0 : Train={gb_high.score(X_train, y_train):.4f}, Test={gb_high.score(X_test, y_test):.4f} -- OVERFIT") # lr=0.05 : apprentissage progressif et stable gb_low = GradientBoostingClassifier( learning_rate=0.05, n_estimators=500, random_state=42 ) gb_low.fit(X_train, y_train) print(f"lr=0.05 : Train={gb_low.score(X_train, y_train):.4f}, Test={gb_low.score(X_test, y_test):.4f} -- STABLE")
lr=1.0 : Train=1.0000, Test=0.9415 -- OVERFIT lr=0.05 : Train=1.0000, Test=0.9708 -- STABLE
Erreur 3 : Oublier de tuner les hyperparametres. Contrairement a Random Forest qui donne d'excellents resultats par defaut, le Gradient Boosting necessite un tuning soigneux. Les parametres par defaut (lr=0.1, n=100, depth=3) sont un bon point de depart, mais rarement optimaux.
Erreur 4 : max_depth trop grand. En Gradient Boosting, les arbres doivent etre des "weak learners". Un max_depth de 10 ou 15 cree des arbres trop complexes qui captent le bruit a chaque etape de boosting. Le bruit s'accumule ! Restez entre 3 et 6.
# max_depth=15 : arbres trop profonds et complexes gb_deep = GradientBoostingClassifier( max_depth=15, n_estimators=200, random_state=42 ) gb_deep.fit(X_train, y_train) print(f"max_depth=15 : Train={gb_deep.score(X_train, y_train):.4f}, Test={gb_deep.score(X_test, y_test):.4f}") # max_depth=3 : arbres "faibles" mais le boosting les rend forts gb_shallow = GradientBoostingClassifier( max_depth=3, n_estimators=200, random_state=42 ) gb_shallow.fit(X_train, y_train) print(f"max_depth=3 : Train={gb_shallow.score(X_train, y_train):.4f}, Test={gb_shallow.score(X_test, y_test):.4f}")
max_depth=15 : Train=1.0000, Test=0.9474 max_depth=3 : Train=1.0000, Test=0.9649
Erreur 5 : Confondre les strategies Random Forest et Gradient Boosting. En RF : arbres profonds, plus d'arbres = toujours mieux, peu de tuning. En GB : arbres peu profonds, trop d'arbres = overfitting, tuning indispensable. Les deux sont des methodes d'ensemble, mais avec des strategies completement opposees.
Erreur 6 : Ne pas utiliser subsample et max_features. Garder subsample=1.0 et max_features=None (defauts) prive le modele de toute regularisation stochastique. Le Stochastic Gradient Boosting (subsample < 1.0 + max_features < 1.0) est presque toujours superieur au Gradient Boosting classique.
# Sans regularisation stochastique (defauts) gb_no_stoch = GradientBoostingClassifier( n_estimators=200, subsample=1.0, max_features=None, random_state=42 ) gb_no_stoch.fit(X_train, y_train) # Avec regularisation stochastique (recommande) gb_stoch = GradientBoostingClassifier( n_estimators=200, subsample=0.8, max_features=0.7, random_state=42 ) gb_stoch.fit(X_train, y_train) print(f"Sans stochastique : Test={gb_no_stoch.score(X_test, y_test):.4f}") print(f"Avec stochastique : Test={gb_stoch.score(X_test, y_test):.4f}") print(f"Amelioration : +{(gb_stoch.score(X_test, y_test) - gb_no_stoch.score(X_test, y_test))*100:.2f}%")
Sans stochastique : Test=0.9649 Avec stochastique : Test=0.9708 Amelioration : +0.58%

Points cles a retenir

  • Boosting = sequentiel : chaque arbre corrige les erreurs (residus) du modele precedent, contrairement au Bagging qui est parallele.
  • Formule centrale : Fm(x) = Fm-1(x) + η * hm(x) -- chaque arbre ajoute une petite correction ponderee par le learning rate.
  • Learning rate bas + beaucoup d'arbres = meilleure generalisation, mais entrainement plus long. C'est le compromis fondamental.
  • TOUJOURS utiliser l'early stopping (n_iter_no_change) pour eviter l'overfitting et trouver le bon nombre d'arbres automatiquement.
  • Arbres peu profonds (max_depth 3-6) : le boosting transforme des "weak learners" en un modele puissant.
  • Stochastic Gradient Boosting (subsample + max_features < 1.0) ameliore presque toujours les resultats.
  • Pas de normalisation necessaire : les arbres de decision sont invariants aux echelles des variables.
  • Feature importance integree : permet de comprendre et interpreter les decisions du modele.
  • Le tuning est essentiel : contrairement a Random Forest, les parametres par defaut sont rarement optimaux.
