TD Pratique : Regression Logistique

Apprentissage supervise -- Classification binaire et multi-classes

Partie 1 : Recapitulatif de l'Algorithme

La regression logistique est un algorithme d'apprentissage supervise utilise pour la classification. Malgre son nom qui contient le mot "regression", il ne predit pas une valeur continue : il estime la probabilite qu'une observation appartienne a une classe donnee (entre 0 et 1) grace a la fonction sigmoide, puis attribue une etiquette en fonction d'un seuil de decision. C'est l'un des algorithmes les plus utilises en machine learning pour la classification binaire.

Analogie simple

Imaginez un medecin qui doit decider si une tumeur est maligne ou benigne. Il dispose de plusieurs mesures (rayon, texture, perimetre...). Plutot que de dire directement "maligne" ou "benigne", il evalue d'abord une probabilite : "il y a 87% de chances que cette tumeur soit maligne". Si cette probabilite depasse un certain seuil (par defaut 50%), il classe la tumeur comme maligne. C'est exactement ce que fait la regression logistique : elle calcule d'abord une probabilite, puis prend une decision binaire.

Avantages

  • Simple et rapide : Entrainement tres rapide meme sur de grands jeux de donnees
  • Interpretable : Chaque coefficient a un sens clair (impact de la feature sur la probabilite)
  • Probabilites calibrees : Fournit directement des probabilites, pas juste des etiquettes de classe
  • Peu de risque de surapprentissage : Grace a la regularisation integree (parametre C)
  • Bonne baseline : Excellent premier modele a tester avant des algorithmes plus complexes
  • Pas de parametres a deviner : Pas de nombre de voisins, de profondeur d'arbre, etc.

Inconvenients

  • Frontiere lineaire : Ne peut capturer que des frontieres de decision lineaires entre les classes
  • Sensible aux features correlees : La multicolinearite peut rendre les coefficients instables et peu fiables
  • Hypothese de linearite : Suppose une relation lineaire entre les features et le log-odds de la variable cible
  • Performances limitees : Sur des problemes complexes (images, texte brut, patterns non lineaires), d'autres algorithmes font mieux
  • Sensible aux outliers : Les valeurs extremes influencent la position de la frontiere de decision

Partie 2 : Principe de Fonctionnement

Etape par etape

Etape 1 -- Combinaison lineaire : Le modele calcule un score brut z = b0 + b1*x1 + b2*x2 + ... + bn*xn, exactement comme une regression lineaire. Ce score peut aller de -infini a +infini.
Etape 2 -- Fonction sigmoide : Le score z est passe dans la fonction sigmoide sigma(z) = 1 / (1 + e^(-z)) qui transforme n'importe quel nombre reel en une valeur entre 0 et 1. Cette valeur represente la probabilite d'appartenir a la classe positive.
Etape 3 -- Seuil de decision : Si la probabilite est superieure au seuil (par defaut 0.5), on predit la classe 1 (positive). Sinon, on predit la classe 0 (negative). Ce seuil est ajustable selon le contexte metier.
Etape 4 -- Optimisation : Pendant l'entrainement, l'algorithme ajuste les poids (b0, b1, ...) pour minimiser la fonction de cout Log Loss, qui penalise fortement les predictions confiantes mais fausses.
Etape 5 -- Regularisation : Un terme de regularisation (L1, L2 ou Elastic Net) est ajoute a la fonction de cout pour empecher le surapprentissage en contraignant la taille des coefficients.

La fonction sigmoide : le coeur de l'algorithme

Les formules essentielles

1. Score lineaire (combinaison lineaire)

z = b0 + b1*x1 + b2*x2 + ... + bn*xn

Identique a la regression lineaire. b0 est le biais (intercept), b1..bn sont les poids (coefficients). Le score z peut valoir n'importe quel nombre reel, de -infini a +infini.

2. Fonction sigmoide (activation)

sigma(z) = 1 / (1 + e^(-z))

Transforme le score z en une probabilite entre 0 et 1. Proprietes cles :

  • Quand z est tres negatif (z = -10), sigma(z) tend vers 0
  • Quand z est tres positif (z = +10), sigma(z) tend vers 1
  • Quand z = 0, sigma(z) = 0.5 exactement (point d'inflexion)
  • La fonction est continue, derivable, et strictement croissante

3. Regle de decision

y_pred = 1 si sigma(z) >= seuil, sinon y_pred = 0

Par defaut, le seuil est 0.5. Mais on peut l'ajuster : un seuil plus bas (ex : 0.3) augmente le recall, un seuil plus haut (ex : 0.7) augmente la precision. Le choix depend du cout relatif des faux positifs vs faux negatifs.

4. Fonction de cout : Log Loss (entropie croisee binaire)

J = -(1/N) * SUM[ y_i * log(p_i) + (1 - y_i) * log(1 - p_i) ]

Cette fonction penalise fortement les predictions confiantes mais fausses. Si le modele predit P=0.99 pour un exemple qui est en realite de classe 0, la penalite est enorme (log(0.01) = -4.6). Si le modele predit P=0.99 pour un vrai positif, la penalite est minime (log(0.99) = -0.01). C'est cette asymetrie qui guide l'optimisation.

Le concept de seuil de decision

Le seuil de decision (threshold) est la valeur de probabilite a partir de laquelle on decide de classer une observation comme "positive". Par defaut, sklearn utilise 0.5 dans la methode predict(), mais ce n'est pas toujours optimal :

  • Seuil = 0.3 : "Dans le doute, je classe positif." On detecte plus de vrais positifs (recall eleve) mais on fait aussi plus de faux positifs (precision basse). Utile en medecine ou manquer un cancer est potentiellement mortel.
  • Seuil = 0.5 : Equilibre standard entre precision et recall. Bon point de depart.
  • Seuil = 0.7 : "Je ne classe positif que si je suis tres confiant." Precision elevee mais recall bas. Utile pour le filtrage de spam ou un faux positif est tres genant (email important perdu).

Pour ajuster le seuil avec sklearn, on utilise predict_proba() au lieu de predict(), puis on applique le seuil manuellement.

La regression logistique ne trace pas une courbe dans l'espace des features : elle trace une frontiere de decision lineaire (un hyperplan). La sigmoide est utilisee pour convertir la distance signee a cette frontiere en une probabilite. Un point loin du cote positif obtient P proche de 1, un point loin du cote negatif obtient P proche de 0. C'est une distinction importante a retenir.

Partie 3 : Les Donnees

Types de donnees acceptes

  • Features numeriques : Directement utilisables. La standardisation est fortement recommandee car les solvers convergent plus vite sur des donnees centrees-reduites.
  • Features categorielles : Doivent etre encodees (One-Hot Encoding pour les variables nominales, Ordinal Encoding pour les variables ordinales) car le modele travaille avec des nombres.
  • Features binaires : Parfaitement adaptees, pas de transformation necessaire (0/1).
  • Variable cible : Doit etre categorielle. Classes discretes : 0/1 pour binaire, ou 0/1/2/... pour multi-classes. Ne supporte pas les variables continues comme cible.

Preparation des donnees

  • Standardisation (essentielle) : Utilisez StandardScaler. Les solvers comme lbfgs et saga convergent beaucoup plus vite sur des donnees centrees-reduites (moyenne 0, ecart-type 1). De plus, le parametre C de regularisation n'a de sens que si les features sont sur la meme echelle.
  • Gestion des valeurs manquantes : La regression logistique ne gere pas les NaN. Il faut imputer les valeurs manquantes avant l'entrainement (SimpleImputer avec strategie moyenne, mediane, ou KNNImputer pour plus de precision).
  • Multicolinearite : Des features tres correlees rendent les coefficients instables et difficiles a interpreter. La regularisation L2 attenue ce probleme, et L1 peut eliminer les features redondantes en mettant leurs coefficients a zero.
  • Outliers : Les valeurs extremes peuvent deplacer la frontiere de decision. Envisagez de les detecter avec IQR ou Z-score et de les traiter (suppression, winsorisation, ou transformation logarithmique).

Gestion du desequilibre de classes

Dans beaucoup de problemes reels, une classe est beaucoup plus frequente que l'autre (ex : 95% benin, 5% malin). Le modele peut alors toujours predire la classe majoritaire et obtenir 95% d'accuracy sans rien apprendre d'utile. C'est le piege de l'accuracy. Solutions :

  • class_weight='balanced' : Sklearn ajuste automatiquement les poids des classes inversement proportionnels a leur frequence. Poids = n_total / (n_classes * n_par_classe).
  • Sur-echantillonnage (SMOTE) : Creer des exemples synthetiques de la classe minoritaire avec la librairie imbalanced-learn.
  • Sous-echantillonnage : Reduire aleatoirement les exemples de la classe majoritaire pour equilibrer.
  • Ajuster le seuil : Baisser le seuil de decision (ex : 0.3 au lieu de 0.5) pour favoriser la detection de la classe minoritaire.
  • Combiner : En pratique, combiner class_weight='balanced' avec un seuil ajuste donne souvent les meilleurs resultats.

Partie 4 : Les Parametres

Chaque parametre de LogisticRegression a un impact precis sur le comportement du modele. Comprenez-les un par un avec des exemples concrets et du code executable.

C -- Force inverse de la regularisation

Role : Controle l'equilibre entre ajustement aux donnees d'entrainement et generalisation. C est l'inverse de la force de regularisation lambda : C = 1/lambda. Un C petit signifie une regularisation forte (modele simple), un C grand signifie une regularisation faible (modele qui colle aux donnees).

Valeur par defaut : C = 1.0

Impact : C'est le parametre le plus important a regler. Un C trop petit cause du sous-apprentissage (underfitting), un C trop grand cause du surapprentissage (overfitting). L'ecart entre l'accuracy train et test revele le probleme.

Avec C=0.01 (regularisation forte), le modele est contraint et generalise correctement mais est un peu sous-optimal (test accuracy plus basse). Avec C=1.0, le bon equilibre est atteint avec le meilleur test accuracy. Avec C=100, le modele colle trop aux donnees d'entrainement : l'ecart train-test double (signe de surapprentissage naissant).
En pratique, on ne teste jamais seulement 3 valeurs. Utilisez GridSearchCV avec C = [0.001, 0.01, 0.1, 1, 10, 100] pour trouver l'optimum automatiquement. Vous pouvez aussi utiliser LogisticRegressionCV qui fait cette recherche en interne de maniere plus efficace. C est le parametre numero un a optimiser.

penalty -- Type de regularisation

Role : Definit quel type de penalite est applique aux coefficients du modele pour eviter le surapprentissage. La regularisation ajoute un terme a la fonction de cout qui penalise les grands coefficients.

Valeur par defaut : 'l2'

Options :

  • 'l2' (Ridge) : Penalise la somme des carres des coefficients : lambda * SUM(bi^2). Reduit les coefficients vers zero mais ne les met jamais exactement a zero. Ideal quand toutes les features sont potentiellement utiles.
  • 'l1' (Lasso) : Penalise la somme des valeurs absolues des coefficients : lambda * SUM(|bi|). Peut mettre certains coefficients a exactement zero, agissant comme un selecteur de features automatique.
  • 'elasticnet' : Combinaison de L1 et L2 (necessite solver='saga' et le parametre l1_ratio pour doser le melange).
  • None : Aucune regularisation. Rarement recommande car risque de surapprentissage eleve.

Impact : L1 pour la selection de features, L2 pour la stabilite des coefficients. L1 necessite le solver 'liblinear' ou 'saga'.

import numpy as np from sklearn.linear_model import LogisticRegression from sklearn.datasets import load_breast_cancer from sklearn.preprocessing import StandardScaler X, y = load_breast_cancer(return_X_y=True) scaler = StandardScaler() X_s = scaler.fit_transform(X) feature_names = load_breast_cancer().feature_names # L1 : selection de features (met des coefficients a zero) model_l1 = LogisticRegression(penalty='l1', C=0.1, solver='liblinear', max_iter=1000) model_l1.fit(X_s, y) # L2 : garde tous les coefficients non nuls model_l2 = LogisticRegression(penalty='l2', C=0.1, solver='lbfgs', max_iter=1000) model_l2.fit(X_s, y) print("=== Features selectionnees par L1 (coef != 0) ===") for name, coef in zip(feature_names, model_l1.coef_[0]): if coef != 0: print(f" {name:28s} : {coef:+.4f}") print(f"\n=== Comparaison ===") print(f"L1 : {sum(model_l1.coef_[0] != 0):2d}/30 features gardees (les autres = 0)") print(f"L2 : {sum(model_l2.coef_[0] != 0):2d}/30 features gardees (aucune = 0)")

L1 a automatiquement selectionne 11 features sur 30, eliminant les 19 features jugees non informatives. L2 garde tous les coefficients non nuls mais plus petits. L1 est ideal pour comprendre quelles features comptent reellement et reduire la dimensionnalite.
La regularisation L1 est tres utile quand vous avez beaucoup de features et que vous suspectez que seules quelques-unes sont pertinentes. C'est une technique de selection de features integree au modele, sans etape supplementaire. Attention : L1 necessite le solver 'liblinear' ou 'saga'. L'elasticnet combine les deux avantages mais necessite 'saga' et un parametre supplementaire l1_ratio.

solver -- Algorithme d'optimisation

Role : Determine l'algorithme utilise pour trouver les coefficients optimaux qui minimisent la fonction de cout (Log Loss + regularisation).

Valeur par defaut : 'lbfgs'

Impact : Le choix du solver affecte la vitesse de convergence, la compatibilite avec les types de penalite, et la scalabilite sur de grands datasets. Les resultats finaux sont generalement tres proches.

max_iter -- Nombre maximum d'iterations

Role : Definit le nombre maximum d'iterations pour que le solver converge vers la solution optimale. Si le solver n'a pas converge en max_iter iterations, sklearn affiche un ConvergenceWarning.

Valeur par defaut : 100

Impact : La premiere solution n'est pas d'augmenter max_iter mais de standardiser les donnees. Des features sur la meme echelle permettent au solver de converger en beaucoup moins d'iterations.

import warnings from sklearn.linear_model import LogisticRegression from sklearn.datasets import load_breast_cancer from sklearn.preprocessing import StandardScaler X, y = load_breast_cancer(return_X_y=True) # Test 1 : SANS standardisation, max_iter=100 par defaut with warnings.catch_warnings(record=True) as w: warnings.simplefilter("always") model = LogisticRegression(max_iter=100) model.fit(X, y) if w: print("[1] Sans scaler, max_iter=100 : ConvergenceWarning !") # Test 2 : AVEC standardisation, max_iter=100 X_s = StandardScaler().fit_transform(X) model2 = LogisticRegression(max_iter=100) model2.fit(X_s, y) print(f"[2] Avec scaler, max_iter=100 : OK en {model2.n_iter_[0]} iterations") # Test 3 : Sans scaler mais max_iter tres grand model3 = LogisticRegression(max_iter=10000) model3.fit(X, y) print(f"[3] Sans scaler, max_iter=10000 : OK en {model3.n_iter_[0]} iterations") print(f"\nConclusion : standardiser = {model3.n_iter_[0] // model2.n_iter_[0]}x moins d'iterations !")
[1] Sans scaler, max_iter=100 : ConvergenceWarning ! [2] Avec scaler, max_iter=100 : OK en 16 iterations [3] Sans scaler, max_iter=10000 : OK en 1547 iterations Conclusion : standardiser = 96x moins d'iterations !
Standardiser les donnees est 96 fois plus efficace qu'augmenter max_iter. Avec StandardScaler, le modele converge en seulement 16 iterations au lieu de 1547. C'est la premiere chose a faire en cas de ConvergenceWarning.
Si vous obtenez un ConvergenceWarning, verifiez d'abord que vos donnees sont standardisees. Si c'est deja le cas, augmentez max_iter a 1000 ou 5000. Ne mettez jamais max_iter a un million -- c'est un signe que quelque chose ne va pas dans la preparation de vos donnees ou le choix du solver.

class_weight -- Poids des classes

Role : Attribue un poids a chaque classe pour gerer le desequilibre. Avec 'balanced', le poids de chaque classe est inversement proportionnel a sa frequence : poids_i = n_total / (n_classes * n_classe_i). Les exemples de la classe rare pesent plus lourd dans la fonction de cout.

Valeur par defaut : None (toutes les classes ont le meme poids = 1)

Impact : Avec 'balanced', le modele est force a mieux detecter la classe minoritaire, au prix d'une legere baisse d'accuracy globale. Le recall de la classe rare augmente significativement.

import numpy as np from sklearn.linear_model import LogisticRegression from sklearn.datasets import make_classification from sklearn.model_selection import train_test_split from sklearn.metrics import classification_report # Creer un dataset TRES DESEQUILIBRE : 95% classe 0, 5% classe 1 X, y = make_classification(n_samples=2000, n_features=20, weights=[0.95, 0.05], random_state=42, flip_y=0.02) X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.3, random_state=42) print(f"Distribution test : Classe 0 = {sum(y_test==0)}, Classe 1 = {sum(y_test==1)}\n") # SANS class_weight model_normal = LogisticRegression(max_iter=1000) model_normal.fit(X_train, y_train) print("--- SANS class_weight (defaut) ---") print(classification_report(y_test, model_normal.predict(X_test), digits=3)) # AVEC class_weight='balanced' model_balanced = LogisticRegression(class_weight='balanced', max_iter=1000) model_balanced.fit(X_train, y_train) print("--- AVEC class_weight='balanced' ---") print(classification_report(y_test, model_balanced.predict(X_test), digits=3))

Sans balancement, le recall de la classe 1 (minoritaire) est catastrophique : seulement 10%. Le modele rate 90% des vrais positifs ! Avec 'balanced', le recall monte a 63.3%, soit 6 fois mieux. L'accuracy globale baisse legerement (de 95.7% a 94.4%), mais le modele detecte enfin la classe rare.

Partie 5 : Les Metriques d'Evaluation

La regression logistique est un classificateur : ses metriques sont differentes de celles de la regression lineaire. Voici les metriques essentielles en classification, avec leurs formules, leur utilite et leurs pieges.

Accuracy (Exactitude)

Formule : Accuracy = (VP + VN) / (VP + VN + FP + FN)

Proportion de predictions correctes parmi toutes les predictions. Simple, intuitive, et la plus connue. Mais attention au piege majeur :

Si 95% des exemples sont de classe 0, un modele qui predit toujours 0 obtient 95% d'accuracy sans rien apprendre. Ne jamais se fier uniquement a l'accuracy en cas de desequilibre de classes. C'est l'erreur la plus frequente en examen.

Precision

Formule : Precision = VP / (VP + FP)

Parmi toutes les predictions "positives", combien sont reellement positives ? La precision mesure la fiabilite des predictions positives. Une precision haute signifie peu de faux positifs.

Quand la privilegier : Quand un faux positif est couteux ou dangereux. Exemples :

  • Filtrage de spam : un email important classe comme spam est tres genant
  • Justice : condamner un innocent est une erreur grave
  • Recommandation : afficher un produit non pertinent agace l'utilisateur

Recall (Rappel / Sensibilite)

Formule : Recall = VP / (VP + FN)

Parmi tous les vrais positifs reels, combien ont ete correctement detectes ? Le recall mesure la capacite du modele a ne pas rater de positifs. Un recall haut signifie peu de faux negatifs.

Quand le privilegier : Quand un faux negatif est dangereux ou couteux. Exemples :

  • Detection de cancer : rater un cancer est potentiellement mortel
  • Detection de fraude : laisser passer une fraude coute de l'argent
  • Detection d'intrusion : rater une attaque informatique compromet la securite

F1-Score

Formule : F1 = 2 * (Precision * Recall) / (Precision + Recall)

Moyenne harmonique de la precision et du recall. Le F1-score est eleve uniquement si les deux metriques sont elevees. Pourquoi la moyenne harmonique et pas arithmetique ?

  • Moyenne arithmetique de P=1.0 et R=0.1 : (1.0+0.1)/2 = 0.55 (trop optimiste)
  • Moyenne harmonique de P=1.0 et R=0.1 : 2*1.0*0.1/(1.0+0.1) = 0.18 (realiste)

La moyenne harmonique penalise fortement les desequilibres. Le F1 est la metrique de reference pour les classes desequilibrees.

ROC-AUC (Area Under the ROC Curve)

La courbe ROC trace le Taux de Vrais Positifs (TVP = Recall) en ordonnee contre le Taux de Faux Positifs (TFP = FP/(FP+VN)) en abscisse, pour differents seuils de decision de 0 a 1.

L'AUC (aire sous la courbe) resume la qualite du modele en un seul chiffre entre 0 et 1. Interpretation :

  • AUC = 0.5 : le modele ne fait pas mieux que le hasard (diagonale)
  • AUC = 0.7-0.8 : modele acceptable mais ameliorable
  • AUC = 0.8-0.9 : bon modele
  • AUC = 0.9-1.0 : excellent modele

L'avantage du ROC-AUC est qu'il est independant du seuil de decision : il evalue la capacite du modele a separer les classes sur l'ensemble des seuils possibles. C'est la meilleure metrique pour comparer deux modeles de classification.

Matrice de Confusion

Tableau 2x2 qui resume toutes les predictions du modele en 4 categories :

Predit Negatif (0)Predit Positif (1)
Reel Negatif (0)Vrais Negatifs (VN) -- CorrectFaux Positifs (FP) -- Erreur de type I
Reel Positif (1)Faux Negatifs (FN) -- Erreur de type IIVrais Positifs (VP) -- Correct

La diagonale principale (VN et VP) represente les predictions correctes. Les elements hors diagonale sont les erreurs. L'objectif est de maximiser la diagonale. Toutes les autres metriques (accuracy, precision, recall, F1) se calculent a partir de ces 4 valeurs.

Log Loss (Perte Logarithmique)

Formule : Log Loss = -(1/N) * SUM[ y_i * log(p_i) + (1 - y_i) * log(1 - p_i) ]

C'est la fonction de cout que la regression logistique minimise pendant l'entrainement. Contrairement aux autres metriques, un Log Loss plus bas est meilleur. Elle penalise particulierement les predictions confiantes mais fausses :

  • Predire P=0.99 quand classe reelle = 0 : penalite = -log(0.01) = 4.6 (enorme)
  • Predire P=0.51 quand classe reelle = 0 : penalite = -log(0.49) = 0.71 (moderee)
  • Predire P=0.99 quand classe reelle = 1 : penalite = -log(0.99) = 0.01 (minime)

Reperes : Log Loss = 0 (parfait, impossible), Log Loss = 0.693 (equivalent hasard 50/50), Log Loss < 0.3 (generalement bon modele).

Partie 7 : Experimentation -- Impact du Seuil de Decision

Le seuil de decision est le parametre le plus sous-estime en classification. Par defaut, sklearn utilise 0.5 dans predict(), mais selon le contexte metier, ajuster ce seuil peut drastiquement changer les performances. Explorons l'impact concret de 3 seuils differents sur le compromis precision/recall.

Le compromis Precision / Recall

C'est un principe fondamental en classification : on ne peut pas maximiser les deux en meme temps.

  • Augmenter le seuil (ex : 0.7) : Moins de predictions positives, donc moins de faux positifs (precision augmente), mais plus de vrais positifs rates (recall diminue).
  • Baisser le seuil (ex : 0.3) : Plus de predictions positives, donc plus de vrais positifs detectes (recall augmente), mais aussi plus de faux positifs (precision diminue).

Le choix optimal depend toujours du contexte metier : quel type d'erreur est le plus couteux ?

Retenez : si rater un positif est grave (cancer, fraude), baissez le seuil. Si un faux positif est grave (spam, justice), montez le seuil.

Partie 8 : Erreurs Courantes

Erreur 1 : Oublier de standardiser les donnees.
La regression logistique utilise des methodes d'optimisation iteratives (gradient) qui convergent beaucoup plus lentement si les features ont des echelles tres differentes. Une feature "age" (0-100) et une feature "salaire" (20000-200000) ont des echelles incompatibles. Sans StandardScaler, vous obtiendrez soit un ConvergenceWarning, soit un modele sous-optimal dont les coefficients ne sont pas comparables entre eux.
Erreur 2 : Utiliser l'accuracy comme seule metrique sur des classes desequilibrees.
C'est le piege le plus frequent et le plus dangereux. Si 95% des exemples sont de classe 0, un modele trivial qui predit toujours "classe 0" obtient 95% d'accuracy. Mais il n'a rien appris et son recall pour la classe 1 est 0%. Utilisez toujours le F1-score, le recall de la classe minoritaire, la matrice de confusion, et le ROC-AUC pour evaluer reellement votre modele.
Erreur 3 : Confondre "regression logistique" avec un algorithme de regression.
Malgre son nom, la regression logistique est un algorithme de classification, pas de regression. Le mot "regression" fait reference a la technique mathematique sous-jacente (regression sur les log-odds de la probabilite). La sortie est une probabilite d'appartenance a une classe (entre 0 et 1), pas une valeur continue. Pour predire une valeur continue, utilisez LinearRegression.
Erreur 4 : Utiliser un solver incompatible avec la penalite choisie.
Toutes les combinaisons solver/penalty ne sont pas valides. 'lbfgs' ne supporte pas 'l1'. 'elasticnet' ne fonctionne qu'avec 'saga'. 'liblinear' ne supporte pas 'multinomial' en multi_class. Sklearn leve une erreur explicite, mais mieux vaut connaitre les compatibilites. Regle : L1 -> liblinear ou saga. L2 -> lbfgs (defaut). Elasticnet -> saga uniquement.
Erreur 5 : Appliquer fit_transform sur les donnees de test.
Le scaler doit etre ajuste (fit) uniquement sur les donnees d'entrainement, puis applique (transform) aux donnees de test. Ecrire scaler.fit_transform(X_test) recalcule la moyenne et l'ecart-type sur le test set, ce qui constitue du data leakage : le modele "voit" indirectement les statistiques du test set. La bonne pratique est : scaler.fit_transform(X_train) puis scaler.transform(X_test).
Erreur 6 : Ignorer la multicolinearite entre les features.
Si deux features sont tres correlees (ex : rayon et perimetre d'un cercle -- l'un est proportionnel a l'autre), les coefficients deviennent instables et tres sensibles aux variations dans les donnees. Cela rend l'interpretation des coefficients peu fiable. La regularisation L2 attenue ce probleme, et L1 peut eliminer automatiquement l'une des features redondantes. Verifiez toujours la matrice de correlation avant d'interpreter les coefficients du modele.
L'erreur la plus frequente en examen est l'erreur 2. Les etudiants regardent uniquement l'accuracy, voient 97%, et concluent que leur modele est excellent. Puis on leur montre que la matrice de confusion revele un recall de 0% pour la classe rare. Prenez toujours l'habitude de calculer au minimum : matrice de confusion, precision, recall et F1-score de chaque classe.

Les 7 commandements de la regression logistique

  1. Tes donnees tu standardiseras -- StandardScaler avant tout. Le solver convergera vite et C aura du sens.
  2. Le parametre C tu optimiseras -- Avec GridSearchCV sur [0.001, 0.01, 0.1, 1, 10, 100]. C'est le levier principal.
  3. La bonne penalite tu choisiras -- L2 par defaut pour la stabilite. L1 si tu veux selectionner des features.
  4. Les metriques appropriees tu calculeras -- Jamais l'accuracy seule. F1, recall, matrice de confusion, ROC-AUC.
  5. Le seuil tu ajusteras -- Selon le contexte metier. Bas si FN grave, haut si FP grave.
  6. Le desequilibre tu gereras -- class_weight='balanced' ou SMOTE si les classes sont desequilibrees.
  7. Les coefficients tu interpreteras -- Signe = direction d'influence. Amplitude = force d'influence. Mais attention a la multicolinearite.
